              <p>I have a function where I validate emails via a <code>regex</code>. I was wondering if this is the best regex to use or if there's a better way of doing it.</p>

        &lt;script&gt;
        function validateEmail(){
            var email = document.forms["form"]["Email"].value;
            var regex = /^([0-9a-zA-Z]([-_\\.]*[0-9a-zA-Z]+)*)@([0-9a-zA-Z]([-_\\.]*[0-9a-zA-Z]+)*)[\\.]([a-zA-Z]{2,9})$/;

            if(!regex.test(email)){
                alert("Enter a valid email");
                return false;
                }
        }

            <td class="answercell"> 
             <div class="post-text" itemprop="text"> 
              <p>I disagree with rolfl's assertion that, "There is no practical way to validate an e-mail address by regex alone."</p> 
              <p>He is correct—as illustrated by the somewhat infamous SO answer he linked to—that it's impractical to validate <em>any RFC-5322-compliant email addresses</em> because, to quote the HTML5 spec:</p> 
              <blockquote> 
               <p>RFC-5322 … defines a syntax for e-mail addresses that is simultaneously too strict (before the '@' character), too vague (after the '@' character), and too lax (allowing comments, whitespace characters, and quoted strings in manners unfamiliar to most users) to be of practical use here.</p> 
              </blockquote> 
              <p>...not to mention that there are now literally thousands of TLDs.</p> 
              <p>But it's absolutely practical to come up with a very good compromise, which is exactly what the writers of the HTML5 spec did. To that end, <a href="http://www.w3.org/TR/html5/forms.html#valid-e-mail-address" rel="noreferrer">the spec provides this PCRE</a>:</p> 
              <pre><code>/^[a-zA-Z0-9.!#$%&amp;'*+\/=?^_`{|}~-]+@[a-zA-Z0-9](?:[a-zA-Z0-9-]{0,61}[a-zA-Z0-9])?(?:\.[a-zA-Z0-9](?:[a-zA-Z0-9-]{0,61}[a-zA-Z0-9])?)*$/
</code></pre> 
              <p>This regular expression will yield false positives (rejections) for <em>technically</em> valid email addresses, including ones with spaces and comments (yes, there is a syntax for putting comments in an email address), but <em>practically</em> all email addresses in use in the real world will pass.</p> 
              <p>As rolfl suggests, the only <em>perfect</em> way to validate an email address is to send an email to it and see if it's received—not all syntactically valid email addresses actually exist, after all—but that doesn't mean we can't use a <em>pretty good</em> validation to give the user useful feedback in the client, and for that we can look to the very smart people who did the work for us when they wrote the HTML5 spec.</p> 
              <p>As to your second question:</p> 
              <blockquote> 
               <p>Also is there a difference between this method and just setting the <code>type="email"</code> in the HTML form?</p> 
              </blockquote> 
              <p>Sure. Using <code>type="email"</code> makes the browser do the validation for you. That's obvious, I know, but it has a number of implications, some good and some not-necessarily-good. Among them:</p> 
              <ol> 
               <li>The browser might not do it. It might not support automatic validations, or the user might have disabled it. This limitation could be overcome with a polyfill, however.</li> 
               <li>In browsers that support it, it just works, and you don't have to write any code, which is pretty nice.</li> 
               <li>Currently there's no widely-supported way to customize the way browsers show HTML5 validation errors, so you can't control how it'll look to the user.</li> 
              </ol> 
              <p>I'm sure there are others, but you get the idea.</p> 
              <p>My personal recommendation is that unless you have a good reason not to, use <code>type="email"</code> along with a polyfill (if you have users with older browsers). Very smart people wrote the HTML5 spec, and very smart people wrote the browsers that implement it. Unless you know something that they don't, use the solutions they've already built.</p>

              <p>Confirming @Jordan, the Javascript regex provided <a href="http://www.w3.org/TR/html5/forms.html#valid-e-mail-address" rel="nofollow">here</a> is practical, but needs to escape the forward slash / specified between the enclosures /..... / in order to work:</p> 
              <p>Original:</p> 
              <pre><code>/^[a-zA-Z0-9.!#$%&amp;'*+/=?^_`{|}~-]+@[a-zA-Z0-9](?:[a-zA-Z0-9-]{0,61}[a-zA-Z0-9])?(?:\.[a-zA-Z0-9](?:[a-zA-Z0-9-]{0,61}[a-zA-Z0-9])?)*$/
</code></pre> 
              <p>With escape:</p> 
              <pre><code>/^[a-zA-Z0-9.!#$%&amp;'*+\/=?^_`{|}~-]+@[a-zA-Z0-9](?:[a-zA-Z0-9-]{0,61}[a-zA-Z0-9])?(?:\.[a-zA-Z0-9](?:[a-zA-Z0-9-]{0,61}[a-zA-Z0-9])?)*$/
</code></pre>
