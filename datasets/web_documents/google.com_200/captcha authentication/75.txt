     <div class="section"> 
      <h1>autologin 0.1.4</h1> 
      <div id="download-button"> 
       <a class="button green" style="float:right;" href="https://pypi.python.org/packages/87/ca/52896c16ac0988456837348b7605a7a4db5695735765931601846a9f4c17/autologin-0.1.4.tar.gz">Download<br> <span style="font-size: 75%">autologin-0.1.4.tar.gz</span> </a> 
      </div> 
      <p style="font-style: italic">A utility for finding login links, forms and autologging into websites with a set of valid credentials.</p> 
      <a href="https://pypi.python.org/pypi/autologin" rel="nofollow"><img src="https://img.shields.io/pypi/v/autologin.svg"> </a> 
      <a href="http://travis-ci.org/TeamHG-Memex/autologin" rel="nofollow"><img src="https://img.shields.io/travis/TeamHG-Memex/autologin/master.svg"> </a> 
      <a href="https://codecov.io/github/TeamHG-Memex/autologin?branch=master" rel="nofollow"><img src="https://codecov.io/github/TeamHG-Memex/autologin/coverage.svg?branch=master"></a> 
      <p>Autologin is a library that makes it easier for web spiders to <strong>crawl websites that require login</strong>. Provide it with credentials and a URL or the html source of a page (normally the homepage), and it will attempt to login for you. Cookies are returned to be used by your spider.</p> 
      <p>The goal of Autologin is to make it easier for web spiders to crawl websites that require authentication <strong>without having to re-write login code for each website.</strong></p> 
      <p>Autologin can be used as a library, on the command line, or as a service. You can make use of Autologin without generating http requests, so you can drop it right into your spider without worrying about impacting rate limits.</p> 
      <p>If you are using <a href="scrapy.readthedocs.org" rel="nofollow">Scrapy</a> for crawling, check out <a href="https://github.com/TeamHG-Memex/autologin-middleware" rel="nofollow">autologin-middleware</a>, which is a scrapy middleware that uses autologin http-api to maintain a logged-in state for a scrapy spider.</p> 
      <p>Autologin works on Python 2.7 and 3.3+.</p> 
      <div> 
       <p>Note</p> 
       <p>The library is in the alpha stage. API can still change, especially around the keychain UI.</p> 
      </div>

      <div id="features"> 
       <h2><a href="#id2" rel="nofollow">Features</a></h2> 
       <ul> 
        <li>Automatically find login forms and fields</li> 
        <li>Obtain authenticated cookies</li> 
        <li>Obtain form requests to submit from your own spider</li> 
        <li>Extract links to login pages</li> 
        <li>Use as a library with or without making http requests</li> 
        <li>Command line client</li> 
        <li>Web service</li> 
        <li>UI for managing login credentials</li> 
        <li>Captcha support</li> 
       </ul> 
      </div> 
      <div id="quickstart"> 
       <h2><a href="#id3" rel="nofollow">Quickstart</a></h2> 
       <p>Don’t like reading documentation?</p> 
       <pre>from autologin import AutoLogin

url = 'https://reddit.com'
username = 'foo'
password = 'bar'
al = AutoLogin()
cookies = al.auth_cookies_from_url(url, username, password)
</pre> 
       <p>You now have a <a href="https://docs.python.org/2/library/cookielib.html" rel="nofollow">cookiejar</a> that you can use in your spider. Don’t want a cookiejar?</p> 
       <pre>cookies.__dict__
</pre> 
       <p>You now have a dictionary.</p> 
      </div> 
      <div id="installation"> 
       <h2><a href="#id4" rel="nofollow">Installation</a></h2> 
       <p>Install the latest release from PyPI:</p> 
       <pre>$ pip install -U autologin
</pre> 
       <p>or the version with the latest changes from Github:</p> 
       <pre>$ pip install git+https://github.com/TeamHG-Memex/autologin.git
</pre> 
       <p>Autologin depends on <a href="https://github.com/TeamHG-Memex/Formasaurus" rel="nofollow">Formasaurus</a> for field and form classification, which has quite a lot of dependencies. These packages may require extra steps to install, so the command above may fail. In this case install dependencies manually, one by one (follow their install instructions).</p> 
       <p>A recent <tt>pip</tt> is recommended (update it with <tt>pip install pip <span class="pre">-U</span></tt>). On Ubuntu, the following packages are required:</p> 
       <pre>$ apt-get install build-essential libssl-dev libffi-dev \
                  libxml2-dev libxslt1-dev \
                  python-dev  # or python3-dev for python 3
</pre> 
      </div> 
      <div id="auth-cookies-from-url"> 
       <h2><a href="#id5" rel="nofollow">Auth cookies from URL</a></h2> 
       <p>This method makes an http request to the URL, extracts the login form (if there is one), fills the fields and submits the form. It then return any cookies it has picked up:</p> 
       <pre>cookies = al.auth_cookies_from_url(url, username, password)
</pre> 
       <p>Note that it returns all cookies, they may be session cookies rather than authenticated cookies.</p> 
       <p>This call is blocking, and uses Crochet to run the Twisted reactor and a Scrapy spider in a separate thread. If you have a Scrapy spider (or use Twisted in some other way), use the HTTP API, or the non-blocking API (it’s not documented, see <tt>http_api.AutologinAPI._login</tt>).</p> 
       <p>There are also optional arguments for <tt>AutoLogin.auth_cookies_from_url</tt>:</p> 
       <ul> 
        <li><p><tt>settings</tt> is a dictionary with Scrapy settings to override. Useful settings to pass include:</p> 
         <blockquote> 
          <ul> 
           <li><tt>HTTP_PROXY</tt>, <tt>HTTPS_PROXY</tt> set proxies to use for all requests.</li> 
           <li><tt>SPLASH_URL</tt> if set, <a href="http://splash.readthedocs.org" rel="nofollow">Splash</a> will be used to make all requests. Use it if your crawler also uses splash and the session is tied to IP and User-Agent, or for Tor sites.</li> 
           <li><tt>USER_AGENT</tt> overrides default User-Agent.</li> 
          </ul> 
         </blockquote> </li> 
        <li><p><tt>extra_js</tt> (experimental) is a string with an extra JS script that should be executed on the login page before making a POST request. For example, it can be used to accept cookie use. It is supported only when <tt>SPLASH_URL</tt> is also given in <tt>settings</tt>.</p> </li> 
       </ul> 
       <p>An example of using this options:</p> 
       <pre>cookies = al.auth_cookies_from_url(
    url, username, password,
    extra_js='document.getElementById("accept-cookies").click();',
    settings={
        'SPLASH_URL': 'http://127.0.0.1:8050',
        'USER_AGENT': 'Mozilla/2.02 [fr] (WinNT; I)',
    })
</pre> 
      </div> 
      <div id="login-request"> 
       <h2><a href="#id6" rel="nofollow">Login request</a></h2> 
       <p>This method extracts the login form (if there is one), fills the fields and returns a dictionary with the form url and args for your spider to submit. No http requests are made:</p> 
       <pre>&gt;&gt;&gt; al.login_request(html_source, username, password, base_url=None)
{'body': 'login=admin&amp;password=secret',
 'headers': {b'Content-Type': b'application/x-www-form-urlencoded'},
 'method': 'POST',
 'url': '/login'}
</pre> 
       <p>Relative form action will be resolved against the <tt>base_url</tt>.</p> 
      </div> 
      <div id="command-line"> 
       <h2><a href="#id7" rel="nofollow">Command Line</a></h2> 
       <pre>$ autologin
usage: autologin [-h] [--splash-url SPLASH_URL] [--http-proxy HTTP_PROXY]
                 [--https-proxy HTTPS_PROXY] [--extra-js EXTRA_JS]
                 [--show-in-browser]
                 username password url
</pre> 
      </div> 
      <div id="http-api"> 
       <h2><a href="#id8" rel="nofollow">HTTP API</a></h2> 
       <p>You can start the autologin HTTP API with:</p> 
       <pre>$ autologin-http-api
</pre> 
       <p>and use <tt><span class="pre">/login-cookies</span></tt> endpoint. Make a POST request with JSON body. The following arguments are supported:</p> 
       <ul> 
        <li><tt>url</tt> (required): url of the site where we would like to login</li> 
        <li><tt>username</tt> (optional): if not provided, it will be fetched from the login keychain</li> 
        <li><tt>password</tt> (optional): if not provided, it will be fetched from the login keychain</li> 
        <li><tt>extra_js</tt> (optional, experimental) is a string with an extra JS script that should be executed on the login page before making a POST request. For example, it can be used to accept cookie use. It is supported only when <tt>SPLASH_URL</tt> is also given in <tt>settings</tt>.</li> 
        <li><tt>settings</tt> (optional) - a dictionary with Scrapy settings to override, useful values are described above.</li> 
       </ul> 
       <p>If <tt>username</tt> and <tt>password</tt> are not provided, autologin tries to find them in the login keychain. If no matching credentials are found (they are matched by domain, not by precise url), then human is expected to eventually provide them in the keychain UI, or mark domain as “skipped”.</p> 
       <p>Response is JSON with the following fields:</p> 
       <ul> 
        <li><p><tt>status</tt>, which can take the following values:</p> 
         <blockquote> 
          <ul> 
           <li><tt>error</tt> status means an error occurred, <tt>error</tt> field has more info</li> 
           <li><tt>skipped</tt> means that domain is marked as “skipped” in keychain UI</li> 
           <li><tt>pending</tt> means there is an item in keychain UI (or it was just created), and no credentials have been entered yet</li> 
           <li><tt>solved</tt> means that login was successful and cookies were obtained</li> 
          </ul> 
         </blockquote> </li> 
        <li><p><tt>error</tt> - human-readable explanation of the error.</p> </li> 
        <li><p><tt>response</tt> - last response received by autologin (can be None in some cases). This is a dict with <tt>cookies</tt>, <tt>headers</tt>, and either a <tt>text</tt> or <tt>body_b64</tt> fields (depending on response content type).</p> </li> 
        <li><p><tt>cookies</tt> - a list of dictionaries in <tt>Cookie.__dict__</tt> format. Present only if status is <tt>solved</tt>.</p> </li> 
        <li><p><tt>start_url</tt> - a url that was reached after successful login.</p> </li> 
       </ul> 
      </div> 
      <div id="proxy-support"> 
       <h2><a href="#id9" rel="nofollow">Proxy support</a></h2> 
       <p>Proxies can be specified via <tt>HTTP_PROXY</tt> and <tt>HTTPS_PROOXY</tt> keys in <tt>settings</tt> argument. Username and password can be specified as part of the proxy url (the format is <tt><span class="pre">protocol://username:password@url</span></tt>).</p> 
       <p>If you are using proxy with Splash, it is assumed that you want to have Splash make requests via given proxy, and not make a request <strong>to</strong> Splash via proxy. <tt>HTTP_PROXY</tt> is always used for Splash.</p> 
      </div> 
      <div id="captcha-support"> 
       <h2><a href="#id10" rel="nofollow">Captcha support</a></h2> 
       <p>There is experimental captcha support: if the login form contains a captcha, we will try to solve it using an external service (DeathByCaptcha), and will submit it as part of login request. This does not affect API in any way, you only have to provide environment variables with your DeathByCaptcha account details: <tt>DEATHBYCAPTCHA_USERNAME</tt> and <tt>DEATHBYCAPTCHA_PASSWORD</tt>. This applies to all APIs: <tt><span class="pre">autologin-http-api</span></tt>, <tt>autologin</tt>, and the Python API.</p> 
       <p>You also need to install the <tt>decaptcha</tt> library:</p> 
       <pre>pip install git+https://github.com/TeamHG-Memex/decaptcha.git
</pre> 
       <p>Support is still experimental, new Google ReCaptcha/NoCaptcha are not supported. Also, it currently works only with splash (when <tt>SPLASH_URL</tt> is passed in <tt>settings</tt>).</p> 
      </div> 
      <div id="keychain-ui"> 
       <h2><a href="#id11" rel="nofollow">Keychain UI</a></h2> 
       <p>Start keychain UI with:</p> 
       <pre>$ autologin-server
</pre> 
       <p>Note that both <tt><span class="pre">autologin-server</span></tt> and <tt><span class="pre">autologin-http-api</span></tt> are not protected by any authentication.</p> 
       <p>Keychain UI stores credentials in an sqlite database. It is located near the library itself by default, which is not always good, especially if you want to persist the data between updates or do not have write permissions for that folder. You can configure database location and <tt>secret_key</tt> used by the flask app by creating an <tt>/etc/autologin.cfg</tt> or <tt><span class="pre">~/.autologin.cfg</span></tt> file (should be the same user under which autologin services are running). Here is an example config that changes default secret_key and specifies a different database path (both items are optional):</p> 
       <pre>[autologin]
secret_key = 8a0b923820dcc509a6f75849b
db = /var/autologin/db.sqlite
</pre> 
      </div>
