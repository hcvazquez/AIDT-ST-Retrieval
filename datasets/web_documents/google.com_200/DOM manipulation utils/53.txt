        <div class="post-body__content">
         <p>Ever marvelled at the magic of <a href="http://mootools.net/">Mootools</a>? Ever wondered how <a href="http://dojotoolkit.org/">Dojo</a> does it? Ever been curious about <a href="http://jquery.com">jQuery's</a> gymnastics? In this tutorial, we’re going to sneak behind the scenes and try our hand at building a super-simple version of your favorite library.</p> 
         <p>
          <!--more--></p> 
         <p>We use JavaScript libraries nearly every day. When you’re just getting started, having something like jQuery is fantastic, mainly because of the DOM. First, the DOM can be rather rough to wrangle for a beginner; it’s a pretty poor excuse for an API. Secondly, it’s not even consistent across all browsers.</p> 
         <blockquote class="pullquote">
          <p>We’re wrapping the elements in an object because we want to be able to create methods for the object.</p>
         </blockquote> 
         <p>In this tutorial, we’re going to take a (decidedly shallow) stab at building one of these libraries from scratch. Yes, it’ll be fun, but before you get too excited, let me clarify a few points:</p> 
         <ul> 
          <li>This won’t be a completely full-featured library. Oh, we’ve got a solid set of methods to write, but it’s no jQuery. We’ll do enough to give you a good feeling for the kind of problems that you’ll run into when building libraries.</li> 
          <li>We aren’t going for complete browser compatibility across the board here. What we’re writing today should work on Internet Explorer 8+, Firefox 5+, Opera 10+, Chrome, and Safari.</li> 
          <li>We aren’t going to cover every possible use of our library. For example, our <code>append</code> and <code>prepend</code> methods will only work if you pass them an instance of our library; they won’t work with raw DOM nodes or nodelists.</li> 
         </ul> 
         <blockquote> 
          <p>One more thing: while we won’t be writing tests for this library, I did that when first developing this. You can get the library and tests <a href="https://github.com/andrew8088/dome">on Github</a>.</p> 
         </blockquote> 
         <hr> 
         <h2> <span> Step 1: </span> Creating the Library Boilerplate</h2> 
         <p>We’ll start with some wrapper code, which will contain our whole library. It’s your typical <a href="http://benalman.com/news/2010/11/immediately-invoked-function-expression/">immediately invoked function expression (IIFE)</a>.</p> 
         <pre class="brush: js noskimlinks noskimwords">window.dome = (function () {
	function Dome (els) {
		
	}
	
	var dome = {
		get: function (selector) {
		
		}	
	};
	
	return dome;
}());</pre> 
         <p>As you can see, we’re calling our library Dome, because it’s primarily a DOM library. Yes, it’s lame.</p> 
         <p>We’ve got a couple of things going on here. First, we have a function; it will eventually be a constructor function for the instances of our library; those objects will wrap our selected or created elements.</p> 
         <p>Then, we have our <code>dome</code> object, which is our actual library object; as you can see, it’s returned at the end there. It’s got an empty <code>get</code> function, which we’ll use to select elements from the page. So, let’s fill that in now.</p> 
         <hr> 
         <h2> <span> Step 2: </span> Getting Elements</h2> 
         <p>The <code>dome.get</code> function will take one parameter, but it could be a number of things. If it’s a string, we’ll assume it’s a CSS selector; but we can also take a single DOM Node, or a NodeList.</p> 
         <pre class="brush: js noskimlinks noskimwords">get: function (selector) {
	var els;
	if (typeof selector === "string") {
    	els = document.querySelectorAll(selector);
	} else if (selector.length) {
		els = selector;
	} else {
		els = [selector];
	}
	return new Dome(els);
}</pre> 
         <p>We’re using <code>document.querySelectorAll</code> to simplify the finding of elements: of course, this does limit our browser support, but for this case, that’s okay. If <code>selector</code> is not a string, we’ll check for a <code>length</code> property. If it exists, we’ll know we have a <code>NodeList</code>; otherwise, we have a single element and we’ll put that in an array. That’s because we need an array to pass to our call to <code>Dome</code> at the bottom there; as you can see, we’re returning a new <code>Dome</code> object. So let’s go back to that empty <code>Dome</code> function and fill it in.</p> 
         <hr> 
         <h2> <span> Step 3: </span> Creating <code>Dome</code> Instances</h2> 
         <p>Here’s that <code>Dome</code> function:</p> 
         <pre class="brush: js noskimlinks noskimwords">function Dome (els) {
	for(var i = 0; i &lt; els.length; i++ ) {
		this[i] = els[i];
	}
	this.length = els.length;
}</pre> 
         <blockquote class="pullquote">
          <p>I really recommend you dig around inside a few of your favourite libraries.</p>
         </blockquote> 
         <p>This is really simple: we just iterate over the elements we selected and stick them onto the new object with numeric indices. Then, we add a <code>length</code> property.</p> 
         <p>But what’s the point here? Why not just return the elements? We’re wrapping the elements in an object because we want to be able to create methods for the object; these are the methods that will allow us to interact with those elements. This is actually a boiled-down version of the way jQuery does it.</p> 
         <p>So, now that we have our <code>Dome</code> object being returned, let’s add some methods to its prototype. I’m going to put those methods right under the <code>Dome</code> function.</p> 
         <hr> 
         <h2> <span> Step 4: </span> Adding a Few Utilities</h2> 
         <p>The first functions we’re going to write are simple utility functions. Since our <code>Dome</code> objects could wrap more than one DOM element, we’re going to need to loop over every element in pretty much every method; so, these utilities will be handy.</p> 
         <p>Let’s start with a <code>map</code> function:</p> 
         <pre class="brush: js noskimlinks noskimwords">Dome.prototype.map = function (callback) {
	var results = [], i = 0;
	for ( ; i &lt; this.length; i++) {
		results.push(callback.call(this, this[i], i));
	}
	return results;
};</pre> 
         <p>Of course, the <code>map</code> function takes a single parameter, a callback function. We’ll loop over the items in the array, collecting whatever is returned from the callback in the <code>results</code> array. Notice how we’re calling that callback function:</p> 
         <pre class="brush: js noskimlinks noskimwords">callback.call(this, this[i], i));</pre> 
         <p>By doing it this way, the function will be called in the context of our <code>Dome</code> instance, and it will receive two parameters: the current element, and the index number.</p> 
         <p>We also want a <code>forEach</code> function. This is actually really simple:</p> 
         <pre class="brush: js noskimlinks noskimwords">Dome.prototype.forEach(callback) {
	this.map(callback);
	return this;
};</pre> 
         <p>Since the only difference between <code>map</code> and <code>forEach</code> is that <code>map</code> needs to return something, we can just pass our callback to <code>this.map</code> and ignore the returned array; instead, we’ll return <code>this</code> to make our library chainable. We’ll be using <code>forEach</code> quite a bit. So, notice that when we return our <code>this.forEach</code> call from a function, we’re actually returning <code>this</code>. For example, these methods actually return the same thing:</p> 
         <pre class="brush: js noskimlinks noskimwords">Dome.prototype.someMethod1 = function (callback) {
	this.forEach(callback);
	return this;
};

Dome.prototype.someMethod2 = function (callback) {
	return this.forEach(callback);
};</pre> 
         <p>One more: <code>mapOne</code>. It’s easy to see what this function does, but the real question is, why do we need it? This requires a bit of what you could call “library philosophy.”</p> 
         <h3>A Short “Philosophical” Detour</h3> 
         <blockquote class="pullquote">
          <p>Firstly, the DOM can be rather rough to wrangle for a beginner; it’s a pretty poor excuse for an API.</p>
         </blockquote> 
         <p>If building a library were just about writing the code, it wouldn’t be too difficult a job. But as I worked on this project, I found the tougher part was deciding how certain methods should work.</p> 
         <p>Soon, we’re going to build a <code>text</code> method that returns the text of our selected elements. If our <code>Dome</code> object wraps several DOM node (<code>dome.get("li")</code>, for example), what should this return? If you do something similar in jQuery (<code>$("li").text()</code>), you’ll get a single string with the text of all the elements concatenated together. Is this useful? I don’t think so, but I’m not sure what a better return value would be.</p> 
         <p>For this project, I’ll return the text of multiple elements as an array, unless there’s only one item in the array; then we’ll just return the text string, not an array with a single item. I think you’ll most often be getting the text of a single element, so we optimize for that case. However, if you’re getting the text of multiple elements, we’ll return something you can work with.</p> 
         <h3>Back to Coding</h3> 
         <p>So, the <code>mapOne</code> method will simply run <code>map</code>, and then either return the array, or the single item that was in the array. If you’re still not sure how this is useful, stick around: you’ll see!</p> 
         <pre class="brush: js noskimlinks noskimwords">Dome.prototype.mapOne = function (callback) {
	var m = this.map(callback);
	return m.length &gt; 1 ? m : m[0];
};</pre> 
         <hr> 
         <h2> <span> Step 5: </span> Working with Text and HTML</h2> 
         <p>Next, let’s add that <code>text</code> method. Just like jQuery, we can pass it a string and set the element’s text, or use no parameters to get the text back.</p> 
         <pre class="brush: js noskimlinks noskimwords">Dome.prototype.text = function (text) {
	if (typeof text !== "undefined") {
		return this.forEach(function (el) {
			el.innerText = text;
		});
	} else {
		return this.mapOne(function (el) {
			return el.innerText;
		});
	}
};</pre> 
         <p>As you might expect, we need to check for a value in <code>text</code> to see if we’re setting or getting. Note that just <code>if (text)</code> wouldn’t work, because an empty string is a false value.</p> 
         <p>If we’re setting, we’ll do a <code>forEach</code> over the elements and set their <code>innerText</code> property to the <code>text</code>. If we’re getting, we’ll return the elements’ <code>innerText</code> property. Note our use of the <code>mapOne</code> method: if we’re working with multiple elements, this will return an array; otherwise, it will be just the string.</p> 
         <p>The <code>html</code> method will do pretty much the same thing as <code>text</code>, except that it will use the <code>innerHTML</code> property, instead of <code>innerText</code>.</p> 
         <pre class="brush: js noskimlinks noskimwords">Dome.prototype.html = function (html) {
	if (typeof html !== "undefined") {
		this.forEach(function (el) {
			el.innerHTML = html;
		});
		return this;
	} else {
		return this.mapOne(function (el) {
			return el.innerHTML;
		});
	}
};</pre> 
         <p>Like I said: almost identical.</p> 
         <hr> 
         <h2> <span> Step 6: </span> Hacking Classes</h2> 
         <p>Next up, we want to be able to add and remove classes; so let’s write the <code>addClass</code> and <code>removeClass</code> methods.</p> 
         <p>Our <code>addClass</code> method will take either a string or an array of class names. To make this work, we need to check the type of that parameter. If it’s an array, we’ll loop over it and create a string of class names. Otherwise, we’ll just add a single space to the front of the class name, so it doesn’t mess with the existing classes on the element. Then, we just loop over the elements and append the new classes to the <code>className</code> property.</p> 
         <pre class="brush: js noskimlinks noskimwords">Dome.prototype.addClass = function (classes) {
	var className = "";
	if (typeof classes !== "string") {
		for (var i = 0; i &lt; classes.length; i++) {
			className += " " + classes[i];
		}
	} else {
		className = " " + classes;
	}
	return this.forEach(function (el) {
		el.className += className;
	});
};</pre> 
         <p>Pretty straightforward, eh?</p> 
         <p>Now, what about removing classes? To keep it simple, we’ll only allow removing one class at a time.</p> 
         <pre class="brush: js noskimlinks noskimwords">Dome.prototype.removeClass = function (clazz) {
	return this.forEach(function (el) {
		var cs = el.className.split(" "), i;

		while ( (i = cs.indexOf(clazz)) &gt; -1) { 
			cs = cs.slice(0, i).concat(cs.slice(++i));
		}
		el.className = cs.join(" ");
	}); 
};</pre> 
         <p>On every element, we’ll split the <code>el.className</code> into an array. Then, we use a while loop to slice out the offending class until <code>cs.indexOf(clazz)</code> returns -1. We do this to cover the edge case where the same classes has been added to an element more than once: we need to make sure it’s really gone. Once we’re sure we’ve cut out every instance of the class, we join the array with spaces and set it on <code>el.className</code>.</p> 
         <hr> 
         <h2> <span> Step 7: </span> Fixing an IE Bug</h2> 
         <p>The worst browser we’re dealing is IE8. In our little library, there’s only one IE bug that we need to deal with; thankfully, it’s pretty simple. IE8 doesn’t support the <code>Array</code> method <code>indexOf</code>; we use it in <code>removeClass</code>, so let’s polyfill it:</p> 
         <pre class="brush: js noskimlinks noskimwords">if (typeof Array.prototype.indexOf !== "function") {
	Array.prototype.indexOf = function (item) {
		for(var i = 0; i &lt; this.length; i++) {
			if (this[i] === item) {
				return i;
			}
		}
		return -1;
	};
}</pre> 
         <p>It’s pretty simple, and it’s not a full implementation (doesn’t support the second parameter), but it will work for our purposes.</p> 
         <hr> 
         <h2> <span> Step 8: </span> Adjusting Attributes</h2> 
         <p>Now, we want an <code>attr</code> function. This’ll be easy, because it’s practically identical to our <code>text</code> or <code>html</code> methods. Like those methods, we’ll be able to both get and set attributes: we’ll take an attribute name and value to set, and just an attribute name to get.</p> 
         <pre class="brush: js noskimlinks noskimwords">Dome.prototype.attr = function (attr, val) {
	if (typeof val !== "undefined") {
		return this.forEach(function(el) {
			el.setAttribute(attr, val);
		});
	} else {
		return this.mapOne(function (el) {
			return el.getAttribute(attr);
		});
	}
};</pre> 
         <p>If the <code>val</code> has a value, we’ll loop through the elements and set the selected attribute with that value, using the element’s <code>setAttribute</code> method. Otherwise, we’ll use <code>mapOne</code> to return that attribute via the <code>getAttribute</code> method.</p> 
         <hr> 
         <h2> <span> Step 9: </span> Creating Elements</h2> 
         <p>We should be able to create new elements, like any good library can. Of course, this would be no good as a method on a <code>Dome</code> instance, so let’s put it right on our <code>dome</code> object.</p> 
         <pre class="brush: js noskimlinks noskimwords">var dome = {
	// get method here
	create: function (tagName, attrs) {

	}
};</pre> 
         <p>As you can see, we’ll take two parameters: the name of the element, and an object of attributes. Most of the attributes be applied via our <code>attr</code> method, but two will get special treatment. We’ll use the <code>addClass</code> method for the <code>className</code> property, and the <code>text</code> method for the <code>text</code> property. Of course, we’ll need to create the element and the <code>Dome</code> object first. Here’s all that in action:</p> 
         <pre class="brush: js noskimlinks noskimwords">create: function (tagName, attrs) {
	var el = new Dome([document.createElement(tagName)]);
		if (attrs) {
			if (attrs.className) {
				el.addClass(attrs.className);
				delete attrs.className;
			}
		if (attrs.text) {
			el.text(attrs.text);
			delete attrs.text;
		}
		for (var key in attrs) {
			if (attrs.hasOwnProperty(key)) {
				el.attr(key, attrs[key]);
			}
		}
	}
	return el;
}</pre> 
         <p>As you can see, we create the element and send it right into a new <code>Dome</code> object. Then, we deal with the attributes. Notice that we have to delete the <code>className</code> and <code>text</code> attributes after working with them. This keeps them from being applied as attributes when we loop over the rest of the keys in <code>attrs</code>. Of course, we end by returning the new <code>Dome</code> object.</p> 
         <p>But now that we’re creating new elements, we’ll want to insert them into the DOM, right?</p> 
         <hr> 
         <h2> <span> Step 10: </span> Appending and Prepending Elements</h2> 
         <p>Next up, we’ll write <code>append</code> and <code>prepend</code> methods, Now, these are actually a bit tricky functions to write, mainly because of the multiple use cases. Here’s what we want to be able to do:</p> 
         <pre class="brush: js noskimlinks noskimwords">dome1.append(dome2);
dome1.prepend(dome2);</pre> 
         <blockquote class="pullquote">
          <p>The worst browser we’re dealing is IE8.</p>
         </blockquote> 
         <p>The use cases are as these: we might want to append or prepend</p> 
         <ul> 
          <li>one new element to one or more existing elements.</li> 
          <li>multiple new elements to one or more existing element.</li> 
          <li>one existing element to one or more existing elements.</li> 
          <li>multiple existing elements to one or more existing elements.</li> 
         </ul> 
         <p>Note: I’m using “new” to mean elements not yet in the DOM; existing elements are already in the DOM.</p> 
         <p>Let’s step though it now:</p> 
         <pre class="brush: js noskimlinks noskimwords">Dome.prototype.append = function (els) {
	this.forEach(function (parEl, i) {
		els.forEach(function (childEl) {
		
		});
	});
};</pre> 
         <p>We expect that <code>els</code> parameter to be a <code>Dome</code> object. A complete DOM library would accept this as a node or nodelist, but we won’t do that. We have to loop over each of our elements, and then inside that, we loop over each of the elements we want to append.</p> 
         <p>If we’re appending the <code>els</code> to more than one element, we need to clone them. However, we don’t want to clone the nodes the first time they’re appended, only subsequent times. So we’ll do this:</p> 
         <pre class="brush: js noskimlinks noskimwords">if (i &gt; 0) {
	childEl = childEl.cloneNode(true);
}</pre> 
         <p>That <code>i</code> comes from the outer <code>forEach</code> loop: it’s the index of the current parent element. If we aren’t appending to the first parent element, we’ll clone the node. This way, the actual node will go in the first parent node, and every other parent will get a copy. This works well, because the <code>Dome</code> object that was passed in as an argument will only have the original (uncloned) nodes. So, if we’re only appending a single element to a single element, all the nodes involved will be part of their respective <code>Dome</code> objects.</p> 
         <p>Finally, we’ll actually append the element:</p> 
         <pre class="brush: js noskimlinks noskimwords">parEl.appendChild(childEl);</pre> 
         <p>So, altogether, this is what we have:</p> 
         <pre class="brush: js noskimlinks noskimwords">Dome.prototype.append = function (els) {
	return this.forEach(function (parEl, i) {
		els.forEach(function (childEl) {
			if (i &gt; 0) {
				childEl = childEl.cloneNode(true); 
			}
			parEl.appendChild(childEl);
		}); 
	}); 
};</pre> 
         <h3>The <code>prepend</code> Method</h3> 
         <p>We want to cover the same cases for the <code>prepend</code> method, so the method is pretty very similar:</p> 
         <pre class="brush: js noskimlinks noskimwords">Dome.prototype.prepend = function (els) {
	return this.forEach(function (parEl, i) {
		for (var j = els.length -1; j &gt; -1; j--) {
			childEl = (i &gt; 0) ? els[j].cloneNode(true) : els[j];
			parEl.insertBefore(childEl, parEl.firstChild);
		}
	}); 
};</pre> 
         <p>The different when prepending is that if you sequentially prepend a list of elements to another element, they’ll end up in reverse order. Since we can’t <code>forEach</code> backwards, I’m going through the loop backwards with a <code>for</code> loop. Again, we’ll clone the node if this isn’t the first parent we’re appending to.</p> 
         <hr> 
         <h2> <span> Step 11: </span> Removing Nodes</h2> 
         <p>For our last node manipulation method, we want to be able to remove nodes from the DOM. Easy, really:</p> 
         <pre class="brush: js noskimlinks noskimwords">Dome.prototype.remove = function () {
	return this.forEach(function (el) {
		return el.parentNode.removeChild(el);
	});
};</pre> 
         <p>Just iterate through the nodes and call the <code>removeChild</code> method on each element’s <code>parentNode</code>. The beauty here (all thanks to the DOM) is that this <code>Dome</code> object will still work fine; we can use any method we want on it, including appending or prepending it back into the DOM. Nice, eh?</p> 
         <hr> 
         <h2> <span> Step 12: </span> Working with Events</h2> 
         <p>Last, but certainly not least, we’re going to write a few functions for event handlers.</p> 
         <p>As you probably know, IE8 uses the old IE events, so we’ll have to check for that. Also, we’ll throw in the DOM 0 events, just ‘cause we can.</p> 
         <p>Check out the method, and then we’ll discuss it:</p> 
         <pre class="brush: js noskimlinks noskimwords">Dome.prototype.on = (function () {
	if (document.addEventListener) {
		return function (evt, fn) {
			return this.forEach(function (el) {
				el.addEventListener(evt, fn, false);
			});
		};
	} else if (document.attachEvent)  {
		return function (evt, fn) {
			return this.forEach(function (el) {
				el.attachEvent("on" + evt, fn);
			});
		};
	} else {
		return function (evt, fn) {
			return this.forEach(function (el) {
				el["on" + evt] = fn;
			});
		};
	}
}());</pre> 
         <p>Here, we have an <a href="http://benalman.com/news/2010/11/immediately-invoked-function-expression/">IIFE</a>, and inside it we’re doing feature checking. If <code>document.addEventListener</code> exists, we’ll use that; otherwise, we’ll check for <code>document.attachEvent</code> or fall back to DOM 0 events. Notice how we’re returning the final function from the IIFE: that’s what will end up being assigned to <code>Dome.prototype.on</code>. When doing feature detection, it’s really handy to be able to assign the appropriate function like this, instead of checking for the features each time the function is run.</p> 
         <p>The <code>off</code> function, which unhooks event handlers, is pretty much identical:</p> 
         <pre class="brush: js noskimlinks noskimwords">Dome.prototype.off = (function () {
	if (document.removeEventListener) {
		return function (evt, fn) {
			return this.forEach(function (el) {
				el.removeEventListener(evt, fn, false);
			});
		};
	} else if (document.detachEvent)  {
		return function (evt, fn) {
			return this.forEach(function (el) {
				el.detachEvent("on" + evt, fn);
			});
		};
	} else {
		return function (evt, fn) {
			return this.forEach(function (el) {
				el["on" + evt] = null;
			});
		};
	}
}());</pre> 
         <hr> 
         <h2>That’s It!</h2> 
         <p>I hope you give our little library a try, and maybe even extend it a bit. Like I mentioned earlier, I have it up on <a href="https://github.com/andrew8088/dome">Github</a>, along with the Jasmine test suite for the code we’re written above. Feel free to fork it, play around, and send a pull request.</p> 
         <blockquote>
          <p>Let me clarify again: the point of this tutorial isn’t to suggest that you should always be writing your own libraries.</p>
         </blockquote> 
         <p>There are dedicated teams of people working together to make the big, established libraries as good as possible. The point here was to give a small peek into what might go on inside a library; I hope you’ve picked up a few tips here.</p> 
         <p>I really recommend you dig around inside a few of your favourite libraries. You’ll find that they aren’t so cryptic as you might have thought, and you’ll probably learn a lot. Here are a few great places to start:</p>
