             <div class="post-text" itemprop="text"> 
              <p>There are two quicksort partition methods mentioned in Cormen:</p> 
              <pre><code>Hoare-Partition(A, p, r)
x = A[p]
i = p - 1
j = r + 1
while true
    repeat
        j = j - 1
    until A[j] &lt;= x
    repeat
        i = i + 1
    until A[i] &gt;= x
    if i &lt; j
        swap( A[i], A[j] )
    else
        return j
</code></pre> 
              <p>and:</p> 
              <pre><code>Lomuto-Partition(A, p, r)
x = A[r]
i = p - 1
for j = p to r - 1
    if A[j] &lt;= x
        i = i + 1
        swap( A[i], A[j] )
swap( A[i +1], A[r] )
return i + 1
</code></pre> 
              <p>Disregarding the method of choosing the pivot, in what situations is one preferable to the other? I know for instance that Lomuto preforms relatively poorly when there is a high percentage of duplicate values ( i.e. where say more than 2/3rds the array is the same value ), where as Hoare performs just fine in that situation.</p> 
              <p>What other special cases make one partition method significant better than the other?</p>

              <h1>Pedagogical Dimension</h1> 
              <p>Due to its simplicity Lomuto's partitioning method might be easier to implement. There is a nice anecdote in Jon Bentley's <em>Programming Pearl</em> on Sorting:</p> 
              <blockquote> 
               <p>“Most discussions of Quicksort use a partitioning scheme based on two approaching indices [...] [i.e. Hoare's]. Although the basic idea of that scheme is straightforward, I have always found the details tricky - I once spent the better part of two days chasing down a bug hiding in a short partitioning loop. A reader of a preliminary draft complained that the standard two-index method is in fact simpler than Lomuto's and sketched some code to make his point; I stopped looking after I found two bugs.”</p> 
              </blockquote> 
              <h1>Performance Dimension</h1> 
              <p>For practical use, ease of implementation might be sacrificed for the sake of efficiency. On a theoretical basis, we can determine the number of element comparisons and swaps to compare performance. Additionally, actual running time will be influenced by other factors, such as caching performance and branch mispredictions.</p> 
              <p>As shown below, the algorithms behave very similar on random permutations <strong>except for the number of swaps</strong>. There Lomuto needs <em>thrice</em> as many as Hoare!</p> 
              <h2>Number of Comparisons</h2> 
              <p>Both methods can be implemented using $n-1$ comparisons to partition an array of length $n$. This is essentially optimal, since we need to compare every element to the pivot for deciding where to put it.</p> 
              <h2>Number of Swaps</h2> 
              <p>The number of swaps is random for both algorithms, depending on the elements in the array. If we assume <em>random permutations</em>, i.e. all elements are distinct and every permutation of the elements is equally likely, we can analyze the <em>expected</em> number of swaps.</p> 
              <p>As only relative order counts, we assume that the elements are the numbers $1,\ldots,n$. That makes the discussion below easier since the rank of an element and its value coincide.</p> 
              <h3>Lomuto's Method</h3> 
              <p>The index variable $j$ scans the whole array and whenever we find an element $A[j]$ smaller than pivot $x$, we do a swap. Among the elements $1,\ldots,n$, exactly $x-1$ ones are smaller than $x$, so we get $x-1$ swaps if the pivot is $x$.</p> 
              <p>The overall expectation then results by averaging over all pivots. Each value in $\{1,\ldots,n\}$ is equally likely to become pivot (namely with prob. $\frac1n$), so we have</p> 
              <p>$$ \frac1n \sum_{x=1}^n (x-1) = \frac n2 - \frac12\;. $$ </p> 
              <p>swaps on average to partition an array of length $n$ with Lomuto's method.</p> 
              <h3>Hoare's Method</h3> 
              <p>Here, the analysis is slightly more tricky: Even fixing pivot $x$, the number of swaps remains random. </p> 
              <p>More precisely: The indices $i$ and $j$ run towards each other until they cross, which <em>always</em> happens at $x$ (by correctness of Hoare's partitioning algorithm!). This effectively divides the array into two parts: A left part which is scanned by $i$ and a right part scanned by $j$.</p> 
              <p>Now, a swap is done exactly for every <em>pair</em> of “misplaced” elements, i.e. a large element (larger than $x$, thus belonging in the right partition) which is currently located in the left part and a small element located in the right part. Note that this pair forming always works out, i.e. there the number of small elements initially in the right part equals the number of large elements in the left part.</p> 
              <p>One can show that the number of these pairs is <a href="http://en.wikipedia.org/wiki/Hypergeometric_distribution">hypergeometrically</a> $\mathrm{Hyp}(n-1,n-x,x-1)$ distributed: For the $n-x$ large elements we randomly draw their positions in the array and have $x-1$ positions in the left part. Accordingly, the expected number of pairs is $(n-x)(x-1)/(n-1)$ given that the pivot is $x$.</p> 
              <p>Finally, we average again over all pivot values to obtain the overall expected number of swaps for Hoare's partitioning:</p> 
              <p>$$ \frac1n \sum_{x=1}^n \frac{(n-x)(x-1)}{n-1} = \frac n6 - \frac13\;. $$</p> 
              <p>(A more detailed description can be found in <a href="https://kluedo.ub.uni-kl.de/frontdoor/index/index/docId/3463">my master's thesis</a>, page 29.)</p> 
              <h2>Memory Access Pattern</h2> 
              <p>Both algorithms use two pointers into the array that scan it <em>sequentially</em>. Therefore both behave almost optimal w.r.t. caching.</p> 
              <h2>Equal Elements and Already Sorted Lists</h2> 
              <p>As already mentioned by Wandering Logic, the performance of the algorithms differs more drastically for lists that are not random permutations.</p> 
              <p>On an array that is already sorted, Hoare's method <em>never</em> swaps, as there are no misplaced pairs (see above), whereas Lomuto's method still does its roughly $n/2$ swaps!</p> 
              <p>The presence of equal elements requires special care in Quicksort. (I stepped into this trap myself; see <a href="https://kluedo.ub.uni-kl.de/frontdoor/index/index/docId/3463">my master's thesis</a>, page 36, for a “Tale on Premature Optimization”) Consider as extreme example an array which filled with $0$s. On such an array, Hoare's method performs a swap for <em>every</em> pair of elements - which is the worst case for Hoare's partitioning - but $i$ and $j$ always meet in the middle of the array. Thus, we have optimal partitioning and the total running time remains in $\mathcal O(n\log n)$.</p> 
              <p>Lomuto's method behaves much more stupidly on the all $0$ array: The comparison <code>A[j] &lt;= x</code> will always be true, so we do a swap for <em>every single element</em>! But even worse: After the loop, we always have $i=n$, so we observe the worst case partitioning, making the overall performance degrade to $\Theta(n^2)$!</p> 
              <h1>Conclusion</h1> 
              <p>Lomuto's method is simple and easier to implement, but should not be used for implementing a library sorting method. </p>
