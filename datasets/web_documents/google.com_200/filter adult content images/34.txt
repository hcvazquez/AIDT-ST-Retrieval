          <h2><a href="https://cloud.google.com/blog/big-data/2016/08/filtering-inappropriate-content-with-the-cloud-vision-api.html">Filtering inappropriate content with the Cloud Vision API</a></h2> 
          <div>
           <small>Wednesday, August 17, 2016</small>
          </div> 
          <div> 
           <p><em>Posted by <a href="https://twitter.com/SRobTweets">Sara Robinson</a>, Developer Advocate</em></p> 
           <p>You may know the Cloud Vision API for its face, object, and landmark detection, but you might not know that the Vision API can also detect inappropriate content in images using the same machine learning models that power <a href="https://support.google.com/websearch/answer/510?co=GENIE.Platform%3DDesktop&amp;hl=en">Google SafeSearch</a>. Since <a href="https://cloudplatform.googleblog.com/2016/04/Google-Cloud-and-Autodesk-enable-10x-improvement-in-media-rendering-efficiency.html">we announced the Google Cloud Vision API GA</a> in April, we’ve seen over 100 million requests for SafeSearch detection.</p> 
           <p>Any application with user uploaded images can make use of inappropriate content detection. To get started with this feature, just add <code>SAFE_SEARCH_DETECTION</code> to your API request:</p> 
           <pre>
<code>     <span style="font-family:courier new,courier,monospace">{ "image": { "content": "base64ImageString" }, "features": [ { "type": "SAFE_SEARCH_DETECTION" }, // More feature detection types…. ] }</span></code></pre> 
           <p>Instead of reviewing all user uploaded images manually, the Vision API’s SafeSearch detection feature flags inappropriate images automatically and returns very few “false positives” (images flagged as inappropriate with no explicit content).</p> 
           <p>Let’s see how this API method looks for and categorizes inappropriate content (using appropriate images, of course). For example, let’s run SafeSearch detection on this picture of avocado toast:</p> 
           <center> 
            <p><img alt="" src="https://cloud.google.com/blog/big-data/2016/08/images/147135382326160/inappropriate-content-detection-3.png"></p> 
           </center> I get the following response: 
           <pre>
<code>  <span style="font-family:courier new,courier,monospace">"safeSearchAnnotation" : { "spoof" : "VERY_UNLIKELY", "medical" : "UNLIKELY", "adult" : "VERY_UNLIKELY", "violence" : "VERY_UNLIKELY" }</span>
</code></pre> 
           <p>For each of the four inappropriate content types (spoof, medical, adult, and violence) the API returns one of five possible likelihood values: “VERY_UNLIKELY”, “UNLIKELY”, “POSSIBLE”, “LIKELY”, or “VERY_LIKELY”. Using these likelihood values, you can decide which subset of images require manual review for your app. For example, you might want to manually review only images that the API identifies as “POSSIBLE” and above.</p> 
           <p>Unsurprisingly, the API response does not give my avocado toast high chances of containing spoof, medical, adult, or violent content. Among the four labels, medical has the highest score due to the close-up nature of the image. If any of these four categories indicated a higher likelihood, it would be easy to flag the image.</p> 
           <p>The Vision API SafeSearch detection feature uses a deep neural network model specifically trained to classify inappropriate content in images. It computes a score on a 0 to 1 scale for each vertical, and based on that score applies the corresponding likelihood as a string.</p> 
           <p>What exactly is the API looking for in each SafeSearch category? I chatted with Nikola Todorovic, an engineer on the SafeSearch models, to find out the details. Let’s take a closer look at each part of the API response:</p> 
           <h3>Adult</h3> 
           <p>The adult content classifier is trained to separate pornographic and non-pornographic images. For example, photos of people in swimsuits without nudity are labeled by the API as UNLIKELY or VERY_UNLIKELY. On the other hand, sexual intercourse, nudity and adult content in cartoon images, like hentai, typically gets LIKELY or VERY_LIKELY labels. This classification can be useful in the following scenarios:</p> 
           <ul> 
            <li>When you want to be absolutely certain you’ve removed all inappropriate content (high recall)</li> 
            <li>When you want to be sure to only remove offensive images (high precision)</li> 
           </ul> 
           <p>In situations where you want to avoid false positives and flag images that you are very sure contain pornographic content, you would remove only images that the API identifies as VERY_LIKELY. If you aren’t as concerned with false positives and want to be on the safe side, you would flag and review all images that return POSSIBLE and above. Ultimately, your use case will determine how you handle the classifier’s response.</p> 
           <p>The classifier is trained not to flag pictures that contain nudity in a medical, scientific, educational, or artistic context. In addition, the classifier is quite effective in correctly classifying swimsuit and lingerie photos as non-pornographic; despite the skin showing, Cloud Vision is able to detect that the “strategic” parts are covered.</p> 
           <p>The SafeSearch engineering team has built a debug tool to analyze image classifications and better understand the deep neural networks powering the detection model. Using the internal neural network activations and backpropagation gradients, this tool imposes a heatmap on the image, indicating the parts that the classifier identified as possibly explicit. In the following heatmap, the final classification (VERY_UNLIKELY) is mostly influenced by the highlighted swimsuit area:</p> 
           <div></div> 
           <center> 
            <p><img alt="" src="https://cloud.google.com/blog/big-data/2016/08/images/147135382326160/inappropriate-content-detection-10.png"><br> <a href="http://www.shutterstock.com/pic-136980398/stock-photo-couple-running-on-a-sandy-beach.html?src=UTRs1-VdVyt0N02FAkpycQ-1-35">Image Source</a></p> 
           </center> 
           <h3>Violence</h3> 
           <p>Images flagged as violent include pictures depicting killing, shooting, or blood and gore. Let’s take a look at a few examples.</p> 
           <p>Simple images of knives or guns won’t be labeled as violent. For the following image, the API returns UNLIKELY:</p> 
           <div></div> 
           <center>
            <img alt="" src="https://cloud.google.com/blog/big-data/2016/08/images/147135382326160/inappropriate-content-detection-14.png">
            <br> 
            <a href="https://www.flickr.com/photos/capcase/4968878587/in/photolist-8z5Nut-a71ee6-7hiQAm-ktck1-6FFgJ-9DSG3B-h88McJ-8zbNa9-nPPotB-8zbP6L-iQAB3M-9SQ5v2-7QcVF-9z3tT1-atj6hR-nu3ozP-5pviTW-552f2S-41hR3v-aNdj7v-daPEj4-6u8aWX-eMWeFM-avVuHc-cEGucb-57fp7K-KDJXT-KDK1i-66ZrE-57fpdM-dV7HR9-2aDzxt-kgf8py-8AxewJ-kgfcAj-9cyi2R-bEQxyv-bXmFeY-9cykfv-a1Qa2K-9eEPRM-a2JR1F-a2JQPF-kgdr3T-a2JQeR-8HbrLk-a2JRfc-7ryCgD-a2MGWE-a2JRrv">Image source</a>
           </center> 
           <p>This image on the other hand, is flagged as POSSIBLE:</p> 
           <center>
            <img alt="" src="https://cloud.google.com/blog/big-data/2016/08/images/147135382326160/inappropriate-content-detection-11.png">
            <br> 
            <a href="http://www.shutterstock.com/pic-72854470/stock-photo-young-soldiers-with-guns.html?src=Pig4-noRo3nnfqiQCeayaA-1-15">Image source</a>
           </center> 
           <p>Running the image above through the heatmap debugging tool, the following heatmap shows us which parts of the image cause the API to return POSSIBLE for violence:</p> 
           <div></div> 
           <center> 
            <p><img alt="" src="https://cloud.google.com/blog/big-data/2016/08/images/147135382326160/inappropriate-content-detection-12.png"></p> 
           </center> This indicates that the model has detected soldiers in the photo. In addition, the pointed rifle will also impact the classification. 
           <h3>Medical</h3> 
           <p>Images flagged as medical content contain explicit images of surgery, diseases, or body parts. A before-and-after picture of hand or face surgery, or a picture depicting an operating room with no close-ups of the patient would not be flagged by the API. The medical classifier looks primarily for graphic photographs of open wounds, genital close-ups, and egregious disease symptoms.</p> 
           <p>Although the following image clearly depicts a medical procedure, it is labeled as UNLIKELY by the Vision API since there’s nothing inappropriate or shocking about it:</p> 
           <center>
            <img alt="" src="https://cloud.google.com/blog/big-data/2016/08/images/147135382326160/inappropriate-content-detection-15.png">
            <br> 
            <a href="http://www.shutterstock.com/pic-56319280.html">Image source</a>
           </center> 
           <h3>Spoof</h3> 
           <p>The spoof detection classifier primarily looks for memes, which are indicated by the presence of text, typical meme faces, and backgrounds.</p> 
           <p>The following meme returns a spoof likelihood of VERY_LIKELY:</p> 
           <center> 
            <p><img alt="" src="https://cloud.google.com/blog/big-data/2016/08/images/147135382326160/inappropriate-content-detection-18.png"><br> <a href="https://commons.wikimedia.org/wiki/File:Land-law-meme.jpg">Image source</a></p> 
           </center> Looking at the heatmap, we can see that the spoof classifier clearly identified the combination of the face and text as the spoofed part of this image: 
           <div></div> 
           <center> 
            <p><img alt="" src="https://cloud.google.com/blog/big-data/2016/08/images/147135382326160/inappropriate-content-detection-17.png"></p> 
           </center> 
           <h3>Use cases for SafeSearch detection</h3> 
           <p>Since the Vision API became generally available, we’ve seen SafeSearch detection used across a variety of industries for applications with user-generated content. Messaging apps are a primary use case for safe content detection, as they often monitor content as it comes in. Rather than reviewing all content manually, the Vision API lets you automate the content approval process and significantly reduces the number of images requiring manual review. We’ve also seen SafeSearch detection used in crowdsourced digital marketing campaigns where brands ask consumers to upload content.</p> 
           <h3>What’s next?</h3> 
           <p>Try SafeSearch detection directly in the browser by uploading a picture to the Vision API demo <a href="https://cloud.google.com/vision/">here</a>. Dive into the <a href="https://cloud.google.com/vision/reference/rest/v1/images/annotate#safesearchannotation">API documentation</a> for SafeSearch detection or use the <a href="http://stackoverflow.com/questions/tagged/google-cloud-vision">google-cloud-vision</a> tag on StackOverflow to ask questions. To start building your own apps with the Vision API, check out <a href="https://github.com/GoogleCloudPlatform/cloud-vision">this GitHub repo</a> for samples in your favorite language.</p>
