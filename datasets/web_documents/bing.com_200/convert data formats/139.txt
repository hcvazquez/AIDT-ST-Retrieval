          <h2>Java SE 8 Date and Time</h2> 
          <p><em>by Ben Evans and Richard Warburton</em></p> 
          <p><strong>Why do we need a new date and time library?</strong></p> 
          <p>A long-standing bugbear of Java developers has been the inadequate support for the date and time use cases of ordinary developers.</p> 
          <p>For example, the existing classes (such as <tt>java.util.Date</tt> and <tt>SimpleDateFormatter</tt>) aren’t thread-safe, leading to potential concurrency issues for users—not something the average developer would expect to deal with when writing date-handling code.</p> 
          <p>Some of the date and time classes also exhibit quite poor API design. For example, years in <tt>java.util.Date</tt> start at 1900, months start at 1, and days start at 0—not very intuitive.</p> 
          <p>These issues, and several others, have led to the popularity of third-party date and time libraries, such as Joda-Time.</p> 
          <p>In order to address these problems and provide better support in the JDK core, a new date and time API, which is free of these problems, has been designed for Java SE 8.</p> 
          <p>The project has been led jointly by the author of Joda-Time (Stephen Colebourne) and Oracle, under JSR 310, and will appear in the new Java SE 8 package <tt>java.time</tt>.</p> 
          <h3>Core Ideas</h3> 
          <p>The new API is driven by three core ideas:&nbsp;</p> 
          <ul> 
           <li><strong>Immutable-value classes.</strong> One of the serious weaknesses of the existing formatters in Java is that they aren’t thread-safe. This puts the burden on developers to use them in a thread-safe manner and to think about concurrency problems in their day-to-day development of date-handling code. The new API avoids this issue by ensuring that all its core classes are immutable and represent well-defined values.</li> 
           <li><p><strong>Domain-driven design.</strong> The new API models its domain very precisely with classes that represent different use cases for <tt>Date</tt> and <tt>Time</tt> closely. This differs from previous Java libraries that were quite poor in that regard. For example, <tt>java.util.Date</tt> represents an instant on the timeline—a wrapper around the number of milli-seconds since the UNIX epoch—but if you call <tt>toString()</tt>, the result suggests that it has a time zone, causing confusion among developers.</p> <p>This emphasis on domain-driven design offers long-term benefits around clarity and understandability, but you might need to think through your application’s domain model of dates when porting from previous APIs to Java SE 8.</p></li> 
           <li><p><strong>Separation of chronologies.</strong> The new API allows people to work with different calendaring systems in order to support the needs of users in some areas of the world, such as Japan or Thailand, that don’t necessarily follow ISO-8601. It does so without imposing additional burden on the majority of developers, who need to work only with the standard chronology.&nbsp;</p></li> 
          </ul> 
          <h2>LocalDate and LocalTime</h2> 
          <p>The first classes you will probably encounter when using the new API are <tt>LocalDate</tt> and <tt>LocalTime</tt>. They are local in the sense that they represent date and time from the context of the observer, such as a calendar on a desk or a clock on your wall. There is also a composite class called <tt>LocalDateTime</tt>, which is a pairing of <tt>LocalDate</tt> and <tt>LocalTime</tt>.&nbsp;</p> 
          <!-- C44v0 --> 
          <div class="c44 c44v0">
           <div class="c44w1">
            <p>The existing classes aren’t thread-safe, leading to potential concurrency issues for users—<em>not something the average developer would expect.</em></p>
           </div>
          </div> 
          <p>
           <!-- /C44v0 --></p> 
          <p>Time zones, which disambiguate the contexts of different observers, are put to one side here; you should use these local classes when you don’t need that context. A desktop JavaFX application might be one of those times. These classes can even be used for representing time on a distributed system that has consistent time zones.</p> 
          <h2>Creating Objects</h2> 
          <p>All the core classes in the new API are constructed by fluent factory methods. When constructing a value by its constituent fields, the factory is called <tt>of</tt>; when converting from another type, the factory is called <tt>from</tt>. There are also parse methods that take strings as parameters. See Listing 1.&nbsp;</p> 
          <pre>
LocalDateTime timePoint = LocalDateTime.now(
    );     // The current date and time
LocalDate.of(2012, Month.DECEMBER, 12); // from values
LocalDate.ofEpochDay(150);  // middle of 1970
LocalTime.of(17, 18); // the train I took home today
LocalTime.parse("10:15:30"); // From a String
</pre> 
          <strong><br> Listing 1&nbsp;</strong> 
          <p><br> Standard Java getter conventions are used in order to obtain values from Java SE 8 classes, as shown in Listing 2.&nbsp;</p> 
          <pre>
LocalDate theDate = timePoint.toLocalDate();
Month month = timePoint.getMonth();
int day = timePoint.getDayOfMonth();
timePoint.getSecond();
</pre> 
          <strong><br> Listing 2&nbsp;</strong> 
          <p><br> You can also alter the object values in order to perform calculations. Because all core classes are immutable in the new API, these methods are called <tt>with</tt> and return new objects, rather than using setters (see Listing 3). There are also methods for calculations based on the different fields.&nbsp;</p> 
          <pre>
// Set the value, returning a new object
LocalDateTime thePast = timePoint.withDayOfMonth(
    10).withYear(2010);

/* You can use direct manipulation methods, 
    or pass a value and field pair */
LocalDateTime yetAnother = thePast.plusWeeks(
    3).plus(3, ChronoUnit.WEEKS);
</pre> 
          <strong><br> Listing 3&nbsp;</strong> 
          <p><br> The new API also has the concept of an <em>adjuster</em>—a block of code that can be used to wrap up common processing logic. You can either write a <tt>WithAdjuster</tt>, which is used to set one or more fields, or a <tt>PlusAdjuster</tt>, which is used to add or subtract some fields. Value classes can also act as adjusters, in which case they update the values of the fields they represent. Built-in adjusters are defined by the new API, but you can write your own adjusters if you have specific business logic that you wish to reuse. See Listing 4.&nbsp;</p> 
          <pre>
import static java.time.temporal.TemporalAdjusters.*;

LocalDateTime timePoint = ...
foo = timePoint.with(lastDayOfMonth());
bar = timePoint.with(previousOrSame(ChronoUnit.WEDNESDAY));

// Using value classes as adjusters
timePoint.with(LocalTime.now());&nbsp;</pre> 
          <strong><br> Listing 4&nbsp;</strong> 
          <h2><br> Truncation</h2> 
          <p>The new API supports different precision time points by offering types to represent a date, a time, and date with time, but obviously there are notions of precision that are more fine-grained than this.</p> 
          <p>The <tt>truncatedTo</tt> method exists to support such use cases, and it allows you to truncate a value to a field, as shown in Listing 5.&nbsp;</p> 
          <pre>
LocalTime truncatedTime = time.truncatedTo(ChronoUnit.SECONDS);
</pre> 
          <strong><br> Listing 5&nbsp;</strong> 
          <h2><br> Time Zones</h2> 
          <p>The local classes that we looked at previously abstract away the complexity introduced by time zones. A time zone is a set of rules, corresponding to a region in which the standard time is the same. There are about 40 of them. Time zones are defined by their offset from Coordinated Universal Time (UTC). They move roughly in sync, but by a specified difference.</p> 
          <p>Time zones can be referred to by two identifiers: abbreviated, for example, “PLT,” and longer, for example, “Asia/Karachi.” When designing your application, you should consider what scenarios are appropriate for using time zones and when offsets are appropriate.&nbsp;</p> 
          <ul> 
           <li><p><tt>ZoneId</tt> is an identifier for a region (see Listing 6). Each <tt>ZoneId</tt> corresponds to some rules that define the time zone for that location. When designing your software, if you consider throwing around a string such as “PLT” or “Asia/Karachi,” you should use this domain class instead. An example use case would be storing users’ preferences for their time zone.</p> <pre>
// You can specify the zone id when creating a zoned date time
ZoneId id = ZoneId.of("Europe/Paris");
ZonedDateTime zoned = ZonedDateTime.of(dateTime, id);
assertEquals(id, ZoneId.from(zoned));
</pre> <p><strong><br> Listing 6</strong></p></li> 
           <li><p><tt>ZoneOffset</tt> is the period of time representing a difference between Greenwich/UTC and a time zone. This can be resolved for a specific <tt>ZoneId</tt> at a specific moment in time, as shown in Listing 7.&nbsp;&nbsp;</p> <pre>
ZoneOffset offset = ZoneOffset.of("+2:00");
</pre> <p><strong><br> Listing 7</strong></p></li> 
          </ul> 
          <h2>Time Zone Classes&nbsp;</h2> 
          <ul> 
           <li><p><tt>ZonedDateTime</tt> is a date and time with a fully qualified time zone (see Listing 8). This can resolve an offset at any point in time. The rule of thumb is that if you want to represent a date and time without relying on the context of a specific server, you should use <tt>ZonedDateTime</tt>.&nbsp;&nbsp;</p> <pre>
ZonedDateTime.parse("2007-12-03T10:15:30+01:00[Europe/Paris]");
</pre> <p><strong><br> Listing 8</strong></p></li> 
           <li><p><tt>OffsetDateTime</tt> is a date and time with a resolved offset. This is useful for serializing data into a database and also should be used as the serialization format for logging time stamps if you have servers in different time zones.</p></li> 
           <li><p><tt>OffsetTime</tt> is a time with a resolved offset, as shown in Listing 9.&nbsp;</p> <pre>
OffsetTime time = OffsetTime.now();
// changes offset, while keeping the same point on the timeline
OffsetTime sameTimeDifferentOffset = time.withOffsetSameInstant(
    offset);
// changes the offset, and updates the point on the timeline
OffsetTime changeTimeWithNewOffset = time.withOffsetSameLocal(
    offset);
// Can also create new object with altered fields as before
changeTimeWithNewOffset
 .withHour(3)
 .plusSeconds(2);
</pre> <p><strong><br> Listing 9 </strong></p></li> 
          </ul> 
          <p>There is an existing time zone class in Java—<tt>java.util.TimeZone</tt>—but it isn’t used by Java SE 8 be-cause all JSR 310 classes are immutable and time zone is mutable.</p> 
          <h2>Periods</h2> 
          <p>A <tt>Period</tt> represents a value such as “3 months and 1 day,” which is a distance on the timeline. This is in contrast to the other classes we’ve looked at so far, which have been points on the timeline. See Listing 10.&nbsp;</p> 
          <pre>
// 3 years, 2 months, 1 day
Period period = Period.of(3, 2, 1);

// You can modify the values of dates using periods
LocalDate newDate = oldDate.plus(period);
ZonedDateTime newDateTime = oldDateTime.minus(period);
// Components of a Period are represented by ChronoUnit values
assertEquals(1, period.get(ChronoUnit.DAYS));&nbsp;</pre> 
          <p><strong><br> Listing 10</strong></p> 
          <!-- C44v0 --> 
          <div class="c44 c44v0">
           <div class="c44w1">
            <p><em>Java SE 8 will ship with a new date and time API in java.time</em> that offers greatly improved safety and functionality for developers. The new API models the domain well, with a good selection of classes for modeling a wide variety of developer use cases.</p>
           </div>
          </div> 
          <!-- /C44v0 --> 
          <h2>Durations</h2> 
          <p>A <tt>Duration</tt> is a distance on the timeline measured in terms of time, and it fulfills a similar purpose to <tt>Period</tt>, but with different precision, as shown in Listing 11.&nbsp;</p> 
          <pre>
// A duration of 3 seconds and 5 nanoseconds
Duration duration = Duration.ofSeconds(3, 5);
Duration oneDay = Duration.between(today, yesterday);
</pre> 
          <strong><br> Listing 11&nbsp;</strong> 
          <p><br> It’s possible to perform normal plus, minus, and “with” operations on a <tt>Duration</tt> instance and also to modify the value of a date or time using the <tt>Duration</tt>.</p> 
          <h2>Chronologies</h2> 
          <p>In order to support the needs of developers using non-ISO calendaring systems, Java SE 8 introduces the concept of a <tt>Chronology</tt>, which represents a calendaring system and acts as a factory for time points within the&nbsp;calendaring system. There are also interfaces that correspond to core time point classes, but are parameterized by</p> 
          <pre>
Chronology:
ChronoLocalDate
ChronoLocalDateTime
ChronoZonedDateTime
</pre> 
          <p><br> These classes are there purely for developers who are working on highly internationalized applications that need to take into account local calendaring systems, and they shouldn’t be used by developers without these requirements. Some calendaring systems don’t even have a concept of a month or a week and calculations would need to be performed via the very generic field API.</p>

          <h2>The Rest of the API</h2> 
          <p>Java SE 8 also has classes for some other common use cases. There is the <tt>MonthDay</tt> class, which contains a pair of <tt>Month</tt> and <tt>Day</tt> and is useful for representing birthdays. The <tt>YearMonth</tt> class covers the credit card start date and expiration date use cases and scenarios in which people have a date with no specified day.</p> 
          <p>JDBC in Java SE 8 will support these new types, but there will be no public JDBC API changes. The existing generic <tt>setObject</tt> and <tt>getObject</tt> methods will be sufficient.</p>

          <h2>Conclusion</h2> 
          <p>Java SE 8 will ship with a new date and time API in <tt>java.time</tt> that offers greatly improved safety and functionality for developers. The new API models the domain well, with a good selection of classes for modeling a wide variety of developer use cases.</p>
