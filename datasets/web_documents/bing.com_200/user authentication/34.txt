        <p>Traditional Java security mechanisms didn't provide the infrastructure required to support traditional authentication and authorization; security in J2SE is based on public key cryptography and code signing. In other words, authentication was based on the idea that <em>code</em> is executing in the JVM and not a principal making a request for a resource. And authorization was based on the notion that the code attempts to use a computing resource. The Java Authentication and Authorization Service (JAAS) was designed to address these shortcomings.</p> 
        <p>The JAAS augments the existing code-based access controls with user-based access controls and authentication capabilities. This enables you to grant permissions not just based on what code is running but <em>who</em> is running it. This article:</p> 
        <p>&nbsp;</p> 
        <ul> 
         <li>Discusses the shortcomings of code-based authentication</li> 
         <li>Provides an overview of JAAS</li> 
         <li>Presents a code-intensive example of authentication</li> 
         <li>Presents a code-intensive example of authorization</li> 
         <li>Offers a flavor of the effort involved in developing authentication and authorizations services</li> 
         <li>Provides sample code that you can adapt for your own applications</li> 
         <li>Provides a tutorial to get started with JAAS</li> 
        </ul> 
        <h2>Motivation</h2> 
        <p>Traditional operating systems (such as UNIX) perform authentication on a principal or entity through some sort of challenge-response mechanism. A user ID and password combination is the most salient of these. This technique is also used for protected web resources using the HTTP basic authentication scheme. The challenge, however, can be made more complex: for example, the information can be encrypted or rely on possession of specific information such as mother's maiden name or an answer to a user-selected question. The response then must be valid based on the type of the challenge.</p> 
        <p>In addition, most operating systems base authorization on an entity or principal and a list of resources authorized for use by that entity or principal. For example, when a user tries to read/write a file, the authorization mechanism verifies that the current executing principal is authorized for the resource.</p> 
        <h2>Overview of JAAS</h2> 
        <p>JAAS consists of two parts: <em>authentication</em> and <em>authorization</em>. In other words, it can be used for both, authentication and authorization:</p> 
        <ul> 
         <li>For authenticating users to securely determine who is executing Java code, regardless of whether the code is a stand-alone Java application, an applet, an enterprise JavaBean, or a servlet.</li> 
         <li>For authorization of users to make sure they have the right permissions required to perform their actions.</li> 
        </ul> 
        <p>Authentication is based on the <a href="#">Pluggable Authentication Modules (PAMs)</a> with a framework to be used for both clients and servers. The authorization part is an extension of the existing authorization schemes in J2SE using policy files. Performing authentication in a pluggable fashion enables Java applications to be independent of the underlying authentication mechanism. This has the advantage that new or revised authentication mechanisms can be plugged in without modifying the application itself.</p> 
        <p>Authorization is an extension of the existing policy-based mechanism which is used to specify what permissions an application (or executing code) can and cannot do. It is based on protection domains. In other words, this mechanism grants permissions based on where the code is coming from and not based on who is executing the code. With JAAS permissions or access control can be based not only on what code is running <em>but who is running it</em>.</p>

        <h2>Using JAAS for Authentication</h2> 
        <p>Clients interact with JAAS through a <code>LoginContext</code> object, which provides a way to develop applications independent of the underlying authentication technology. The <code>LoginContext</code> class, which is part of the <code>javax.security.auth.login</code> package, describes the methods used to authenticate subjects. A subject is an identity in a system that you want to authenticate and assign access rights to. A subject can be a human user, a process, or a machine and it is represented by the <code>javax.security.auth.Subject</code> class. Since a subject may interact with multiple authorities (one password for online banking and another for an email system), a <code>java.security.Principal</code> is used to represent the identity in those interactions. In other words, the <code>Principal</code> interface is an abstract notion that can be used to represent an entity, a company, or a login ID. A <code>Subject</code> may contain multiple <code>Principle</code>s. An example class implementing the <code>Principal</code> interface will be shown later.</p> 
        <p>The <code>LoginContext</code> object invokes the <code>LoginModule</code>s that are responsible for implementing and performing the authentication. The <code>LoginModule</code> interface, which is part of the <code>javax.security.auth.spi</code> package, must be implemented by authentication technology providers and can be specified by applications to provide a specific type of authentication. The <code>LoginContext</code> reads the <code>Configuration</code> and instantiates the specified <code>LoginModule</code>s.</p> 
        <p>A <code>Configuration</code> is used to specify the authentication technology, or <code>LoginModule</code>, to be used with a particular application. As a result, different <code>LoginModule</code>s can be plugged in under an application without any modifications to the application's code itself.</p> 
        <p>Code Sample 1 shows a sample JAAS client. I have used the <code>LoginContext</code> that will invoke the <code>LoginModule</code>s specified in the configuration to perform the authentication, and have initialized the <code>LoginContext</code> with a name "WeatherLogin", and callback handler "MyCallbackHandler" whose implementation is shown in Code Sample 2. The name will be used as the index in the Configuration file to determine which <code>LoginModule</code> should be used. This will become clearer when you see the configuration file (shown in Code Sample 5) later. The callback handler is passed to the underlying LoginModule so that they can communicate and interact with the user to prompt for a username/password, for example, through a textual or graphical user interface. Once the <code>LoginContext</code> has been initialized, the <code>login</code> method is called to login.</p> 
        <p><strong>Code Sample 1</strong>: MyClient.java</p>

        <p>A JAAS-based application implements the <code>CallbackHandler</code> interface so that it can interact with users to enter specific authentication data, such as user names or passwords, or to display error and warning messages. The underlying security service may request different types of information via passing individual callbacks to the callback handler. Based on the callbacks passed, the callback handler decides how to retrieve and display information. For example, if the underlying service needs a username and a password to authenticate a user, it uses a <code>NameCallback</code> and <code>PasswordCallback</code>. Other callbacks, which are part of the <code>javax.security.auth.callback</code> class, include:</p> 
        <ul> 
         <li><code>ChoiceCallback</code> (display a list of choices)</li> 
         <li><code>ConfirmationCallback</code> (ask for YES/NO, OK/CANCEL)</li> 
         <li><code>LanguageCallback</code> (the <code>Locale</code> used for localizing text</li> 
         <li><code>TextInputCallback</code> (retrieve generic text information)</li> 
         <li><code>TextOutputCallback</code> (display information, warning, and error messages)</li> 
        </ul> 
        <p>Implementing the <code>CallbackHandler</code> interface means that you need to provide implementation to the <code>handle</code> method to retrieve or display the information requested in the provided callbacks. A Sample implementation is shown in Code Sample 2. Note that here I am using the <code>NameCallback</code> to interact with the user.</p>

    for(int i=0;i&lt;callbacks.length;i++) {
      if(callbacks[i] instanceof NameCallback) {

        throw(new UnsupportedCallbackException(callbacks[i], "Callback handler not support"));
      }
    }
  }

        <p>Now, let's see a sample implementation of a <code>LoginModule</code>. Note that in reality application developers do not need to implement <code>LoginModule</code>s themselves; they can use login modules and plug them into their applications. For example, Sun Microsystems ships several <code>LoginModule</code>s including: <code>JndiLoginModule</code>, <code>KeyStoreLoginModule</code>, <code>Krb5LoginModule</code>, <code>NTLoginModule</code>, <code>UNIXLoginModule</code>. If you'd like to learn how to use any of these login modules, please refer to the documentation in the For More Information section at the end of this article.</p> 
        <p>For the curious, however, Code Sample 3 shows a sample implementation of a <code>LoginModule</code>. This example is very simple as it recognizes only one authentication string and one <code>Principal</code> "SunnyDay", both of which are hard-coded. To login, the system displays "What is the weather like today?", if the answer is "Sunny", the user is logged in. Note how the <code>MyCallbackHandler</code> is being used in the implementation of the <code>login</code> method. In addition to the <code>login</code> method, you must provide implementation for four methods: <code>initialize</code>, <code>commit</code>, <code>abort</code>, and <code>logout</code>. These methods will be used by the <code>LoginContext</code> in the following order:</p> 
        <p>&nbsp;</p> 
        <ul> 
         <li><code>initialize</code>: The purpose of this method is to initialize this <code>LoginModule</code> with the relevant information. The <code>Subject</code> passed in this method is used to store the <code>Principal</code>s and <code>Credential</code>s if login succeeds. Note that this method takes a <code>CallbackHanlder</code> that can be used for entering authentication information. In this example, I do not use the <code>CallbackHandler</code>. The <code>CallbackHandler</code> is useful as it decouples the services provider from the specific input device being used.</li> 
         <li><code>login</code>: Asks the <code>LoginModule</code> to authenticate the <code>Subject</code>. Note that the <code>Principal</code> has not been assigned yet.</li> 
         <li><code>commit</code>: This method is called if the <code>LoginContext</code>'s overall authentication succeeded.</li> 
         <li><code>abort</code>: Informs the <code>LoginModule</code> that some other providers or modules have failed to authenticate the subject. The whole login should fail.</li> 
         <li><code>logout</code>: Logout the <code>Subject</code> by removing the <code>Principal</code>s and <code>Credential</code>s from the <code>Subject</code>.</li> 
        </ul> 
        <br> 
        <br> 
        <p><strong>Code Sample 3</strong>: WeatherLoginModule.java</p> 
        <p><code> 
          <!-- BEGIN VCD7 CODE SAMPLE COMPONENT  --> </code></p>

           <td> <pre>
import java.io.*;
import java.util.*;
import java.security.Principal;
import javax.security.auth.Subject;
import javax.security.auth.callback.*;
import javax.security.auth.spi.LoginModule;
import javax.security.auth.login.LoginException;


public class WeatherLoginModule implements LoginModule {
  private Subject subject;
  private ExamplePrincipal entity;
  private CallbackHandler callbackhandler;
  private static final int NOT = 0;
  private static final int OK = 1;
  private static final int COMMIT = 2;
  private int status;

  public void initialize(Subject subject, CallbackHandler//
   callbackhandler, Map state, Map options) {
    status = NOT;
    entity = null;
    this.subject = subject;
    this.callbackhandler = callbackhandler;
  }

  public boolean login() throws LoginException {

    if(callbackhandler == null) {
      throw new LoginException("No callback handler is available");
    }
    Callback callbacks[] = new Callback[1];
    callbacks[0] = new NameCallback("What is the weather like today?");
    String name = null;
    try {
      callbackhandler.handle(callbacks);
      name = ((NameCallback)callbacks[0]).getName();
    } catch(java.io.IOException ioe) {
      throw new LoginException(ioe.toString());
    } catch(UnsupportedCallbackException ce) {
      throw new LoginException("Error: "+ce.getCallback().toString());
    }
    if(name.equals("Sunny")) {
      entity = new ExamplePrincipal("SunnyDay");
      status = OK;
      return true;
    } else {
      return false;
    }
  }

  public boolean commit() throws LoginException {
    if(status == NOT) {
      return false;
    }
    if(subject == null) {
      return false;
    }
    Set entities = subject.getPrincipals();
    if(!entities.contains(entity)) {
      entities.add(entity);
    }
    status = COMMIT;
    return true;
  }

  public boolean abort() throws LoginException {
    if((subject != null) &amp;&amp; (entity != null)) {
      Set entities = subject.getPrincipals();
      if(entities.contains(entity)) {
        entities.remove(entity);
      }
    }
    subject = null;
    entity = null;
    status = NOT;
    return true;
  }

  public boolean logout() throws LoginException {
    subject.getPrincipals().remove(entity);
    status = NOT;
    subject = null;
    return true;
  }
}

        <h3>Using JAAS for Authorization</h3> 
        <p>The JAAS authorization extends the code-centric Java security architecture that uses a security policy to specify what access rights are granted to executing code. For example, in the security policy in Code Sample 6, permissions are granted to all the code in the current directory; it does not matter whether the code is signed or unsigned, or who is running the code. JAAS extends this with <em>user-centric</em> access controls. Permissions can be granted based not only on what code is running but based on who is running it. Permissions can be granted in the policy file to specific principals as you will see later</p> 
        <p>In order to use JAAS authorization:</p> 
        <ol> 
         <li>The user must first be authenticated as I have done above.</li> 
         <li>The doAs (or doAsPrivileged) method of the <code>Subject</code> class must be called, which in turn invokes the run method that contains the code to be executed as the specified subject.</li> 
        </ol> 
        <p>Based on this, the files <code>ExamplePrincipal.java</code>, <code>WeatherLoginModule.java</code>, and <code>example.conf</code> remain the same.</p> 
        <p>In order to enable the client to authorize users, once the login is successful (user has been authenticated), the authenticated subject is obtained using <code>Subject subject = ctx.getSubject()</code>. The <code>Subject.doAsPrivileged()</code> is then called passing it the authenticated subject and a privileged action, and a <code>null</code> <code>AccessControlContext</code>. These changes to <code>MyClient.java</code> are highlighted in Code Sample 7. Note that once the authorized action is performed, I log the user out by calling <code>ctx.logout()</code></p>

        <h2>Conclusion</h2> 
        <p>The JAAS is a set of APIs that enable services to authenticate and enforce access controls upon users. JAAS authentication is performed in a pluggable fashion enabling Java applications to remain independent of the underlying authentication mechanism, and JAAS authorization augments the existing code-centric Java security architecture with user-based access controls and authentication capabilities.</p> 
        <p>This article presented a fast track tutorial on JAAS. The examples shown throughout demonstrate how to use JAAS for authentication and authorization. Feel free to adapt the code for your own applications. Note that in this example the application and the action are deployed in the same codebase, as an exercise revise the security policy to place the action in its own codebase.</p>
