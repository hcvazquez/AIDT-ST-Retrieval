         <p>Customers frequently ask us how they can use NGINX&nbsp;Plus and NGINX to secure protected resources or applications by authenticating the users who request them. Today we’re announcing a reference implementation of such an authentication system, and making it available in the NGINX, Inc. repository on <a target="_blank" href="https://github.com/nginxinc/nginx-ldap-auth">GitHub</a>. In this post we describe how the implementation works, how to install it, and how to use it as a model for your own authentication system.</p> 
         <p>The solution takes advantage of the <a target="_blank" href="http://nginx.org/en/docs/http/ngx_http_auth_request_module.html">ngx_http_auth_request_module</a> module in NGINX&nbsp;Plus and NGINX, which forwards authentication requests to an external service. In the reference implementation, that service is a daemon we call <em>ldap?auth</em>. It’s written in Python and communicates with a Lightweight Directory Access Protocol (LDAP) authentication server &nbsp;– <a href="http://www.openldap.org/" target="_blank">OpenLDAP</a> by default, but we have tested the ldap?auth daemon against default configurations of Microsoft® Windows® Server Active Directory as well (both the&nbsp;2003 and&nbsp;2012 versions).</p> 
         <p>The ldap?auth daemon serves as a model for your own “connector” app, which you might write in other languages, deploy with different authentication systems, or both. The NGINX <a href="http://nginx.com/services/">Professional Services</a> team is available to assist with such adaptations.</p> 
         <p><strong>Note:</strong> For ease of reading, the rest of this article refers to NGINX&nbsp;Plus, but the reference implementation also works with the open source NGINX software. The prerequisite http_auth_request module is included in both <a target="_blank" href="http://cs.nginx.com/repo_setup">NGINX&nbsp;Plus packages</a> and <a target="_blank" href="http://nginx.org/en/linux_packages.html">prebuilt NGINX binaries</a>.</p> 
         <h2 id="ldap-auth-flow">How Authentication Works in the Reference Implementation</h2> 
         <p>To perform authentication, the http_auth_request module makes an HTTP subrequest to the ldap?auth daemon, which acts as intermediary and interprets the subrequest for the LDAP server&nbsp;– it uses HTTP for communication with NGINX&nbsp;Plus and the appropriate API for communication with the LDAP server.</p> 
         <p>We assume that if you’re interested in the reference implementation, you already have an application or other resources you want to protect by requiring authentication. To make it easier to test the reference implementation, however, we’re providing a sample backend daemon, also written in Python, which listens on port&nbsp;9000. It can stand in for an actual HTTP application during testing, by prompting for user credentials and creating a cookie based on them.</p> 
         <p><img src="https://cdn-1.wp.nginx.com/wp-content/uploads/2016/02/ldap-auth-components.jpg" alt="The NGINX Plus reference implementation for LDAP authentication includes the ldap-auth daemon and a sample backend daemon" width="509" height="271" class="aligncenter size-full wp-image-26491" style="border:2px solid #666666; padding:2px; margin:2px;" srcset="https://cdn-1.wp.nginx.com/wp-content/uploads/2016/02/ldap-auth-components.jpg 509w, https://cdn-1.wp.nginx.com/wp-content/uploads/2016/02/ldap-auth-components-300x160.jpg 300w, https://cdn-1.wp.nginx.com/wp-content/uploads/2016/02/ldap-auth-components-150x80.jpg 150w, https://cdn-1.wp.nginx.com/wp-content/uploads/2016/02/ldap-auth-components-320x170.jpg 320w" sizes="(max-width: 509px) 100vw, 509px"></p> 
         <p>Here’s a step?by?step description of the authentication process in the reference implementation. The details are determined by settings in the <span style="white-space: nowrap; font-weight: bold;">nginx-ldap-auth.conf</span> configuration file; see <a href="#ldap-auth-configure">Configuring the Reference Implementation</a> below. The flow chart following the steps summarizes the process.</p> 
         <ol> 
          <li> <p>A client sends an HTTP request for a protected resource hosted on a server for which NGINX&nbsp;Plus is acting as reverse proxy.</p></li> 
          <li> <p>NGINX&nbsp;Plus (specifically, the http_auth_request module) forwards the request to the ldap?auth daemon, which responds with HTTP code&nbsp;<code>401</code> because no credentials were provided.</p> </li> 
          <li> <p>NGINX&nbsp;Plus forwards the request to <strong>http://backend/login</strong>, which corresponds to the backend daemon. It writes the original request URI to the <span style="white-space: nowrap;"><code>X-Target</code></span> header of the forwarded request.</p> </li> 
          <li> <p>The backend daemon sends the client a login form (the form is defined in the Python code for the daemon). As configured by the <a target="_blank" href="http://nginx.org/en/docs/http/ngx_http_core_module.html#error_page"><code>error_page</code></a> directive, NGINX sets the HTTP code on the login form to&nbsp;<code>200</code>.</p> </li> 
          <li> <p>The user fills in the Username and Password fields on the form and clicks the Login button. Per the code in the form, the client generates an HTTP <code>POST</code> request directed to <strong>/login</strong>, which NGINX&nbsp;Plus forwards to the backend daemon.</p> </li> 
          <li> <p>The backend daemon constructs a string of the format <code><em>username</em>:<em>password</em></code>, applies Base64 encoding, generates a cookie called <strong>nginxauth</strong> with its value set to the encoded string, and sends the cookie to the client. It sets the <code>httponly</code> flag to prevent use of JavaScript to read or manipulate the cookie (protecting against the <a target="_blank" href="http://en.wikipedia.org/wiki/Cross-site_scripting">cross?site scripting [XSS]</a> vulnerability).</p> </li> 
          <li> <p>The client retransmits its original request (from Step&nbsp;1), this time including the cookie in the <code>Cookie</code> field of the HTTP header. NGINX&nbsp;Plus forwards the request to the ldap?auth daemon (as in Step&nbsp;2).</p> </li> 
          <li> <p>The ldap?auth daemon decodes the cookie, and sends the username and password to the LDAP server in an authentication request.</p> </li> 
          <li> <p>The next action depends on whether the LDAP server successfully authenticates the user:</p> 
           <ul> 
            <li> <p>If authentication succeeds, the ldap?auth daemon sends HTTP code <code>200</code> to NGINX&nbsp;Plus. NGINX&nbsp;Plus requests the resource from the backend daemon. In the reference implementation, the backend daemon returns the following text:</p> <pre><code class="config">   Hello, world! Requested URL: <em>URL</em></code></pre> <p>The <span style="white-space: nowrap; font-weight: bold;">nginx-ldap-auth.conf</span> file includes directives for caching the results of the authentication attempt; to disable caching, see <a href="#ldap-auth-caching">Caching</a> below.</p> </li> 
            <li> <p>If authentication fails, the ldap?auth daemon sends HTTP code <code>403</code> to NGINX&nbsp;Plus. NGINX&nbsp;Plus forwards the request to the backend daemon again (as in Step&nbsp;3), and the process repeats.</p> </li> 
           </ul> </li> 
         </ol> 
         <p><img src="https://cdn-1.wp.nginx.com/wp-content/uploads/2016/02/ldap-auth-flowchart.jpg" alt="In the NGINX Plus reference implementation for LDAP authentication, the ldap-auth daemon is the intermediary between NGINX Plus and the LDAP server" width="720" height="540" class="aligncenter size-full wp-image-26506" style="border:2px solid #666666; padding:2px; margin:2px;" srcset="https://cdn-1.wp.nginx.com/wp-content/uploads/2016/02/ldap-auth-flowchart.jpg 720w, https://cdn-1.wp.nginx.com/wp-content/uploads/2016/02/ldap-auth-flowchart-300x225.jpg 300w, https://cdn-1.wp.nginx.com/wp-content/uploads/2016/02/ldap-auth-flowchart-150x113.jpg 150w, https://cdn-1.wp.nginx.com/wp-content/uploads/2016/02/ldap-auth-flowchart-640x480.jpg 640w, https://cdn-1.wp.nginx.com/wp-content/uploads/2016/02/ldap-auth-flowchart-320x240.jpg 320w" sizes="(max-width: 720px) 100vw, 720px"></p> 
         <h2>Installing the Components</h2> 
         <p>The NGINX&nbsp;Plus configuration file distributed with the reference implementation, <span style="white-space: nowrap; font-weight: bold;">nginx-ldap-auth.conf</span>, configures all components other than the LDAP server (that is, NGINX&nbsp;Plus, the client, the ldap?auth daemon, and the backend daemon) to run on the same host, which is adequate for testing purposes. The LDAP server can also run on that host during testing.</p> 
         <p>In an actual deployment, the backend application and authentication server typically each run on a separate host, with NGINX&nbsp;Plus on a third host. The ldap-auth daemon does not consume many resources in most situations, so it can run on the NGINX&nbsp;Plus host or another host of your choice.</p> 
         <ol> 
          <li> <p>Create a clone of the <a target="_blank" href="https://github.com/nginxinc/nginx-ldap-auth">GitHub repository</a>.</p> </li> 
          <li> <p>If NGINX&nbsp;Plus is not already running, install it according to the <a target="_blank" href="https://cs.nginx.com/repo_setup">instructions</a> for your operating system.</p> </li> 
          <li> <p>If an LDAP server is not already running, install and configure one. By default the ldap?auth daemon communicates with OpenLDAP, but Microsoft Windows Active Directory&nbsp;2003 and&nbsp;2012 are also supported.</p> <p>If you are using the LDAP server only to test the reference implementation, you can use the <a target="_blank" href="https://github.com/osixia/docker-openldap">OpenLDAP server Docker image</a> that is available on GitHub, or you can set up a server in a virtual environment using instructions such as <a target="_blank" href="https://www.digitalocean.com/community/tutorials/how-to-install-and-configure-a-basic-ldap-server-on-an-ubuntu-12-04-vps">How To Install and Configure a Basic LDAP Server on an Ubuntu&nbsp;12.04&nbsp;VPS</a>.</p> <p>Make note of the values you set for the Base&nbsp;DN, Bind&nbsp;DN, and Bind password. You will put them in the NGINX configuration file in <a href="#ldap-auth-configure">Configuring the Reference Implementation</a>.</p> </li> 
          <li> <p>On the host where the ldap?auth daemon is to run, install the following additional software. We recommend using the versions that are distributed with the operating system, instead of downloading the software from an open source repository.</p> 
           <ul> 
            <li>Python version&nbsp;2. Version&nbsp;3 is not supported.</li> 
            <li>The Python LDAP module, <strong>python?ldap</strong> (created by the <a target="_blank" href="http://python-ldap.org">python-ldap.org</a> open source project).</li> 
           </ul> </li> 
          <li> <p>Copy the following files from your repository clone to the indicated hosts:</p> 
           <ul> 
            <li><strong>nginx-ldap-auth.conf</strong>&nbsp;– NGINX&nbsp;Plus configuration file that includes the minimal set of directives for testing the reference implementation. Install on the NGINX&nbsp;Plus host (in the <strong>/etc/nginx/conf.d</strong> directory if using the conventional configuration scheme). To avoid configuration conflicts, remember to move or rename any default configuration files installed with NGINX&nbsp;Plus.<p></p> </li> 
            <li><strong>nginx-ldap-auth-daemon.py</strong>&nbsp;– Python code for the ldap?auth daemon. Install on the host of your choice.<p></p> </li> 
            <li><strong>nginx-ldap-auth-daemon-ctl.sh</strong>&nbsp;– Sample shell script for starting and stopping the daemon. Install on the same host as the ldap?auth daemon.<p></p> </li> 
            <li><strong>backend-sample-app.py</strong>&nbsp;– Python code for the daemon that during testing stands in for a backend application server. Install on the host of your choice.<p></p> </li> 
           </ul> </li> 
          <li> <p>Modify the NGINX&nbsp;Plus configuration file as described in <a href="#ldap-auth-configure">Configuring the Reference Implementation</a> below. After making your changes, run the <span style="white-space: nowrap;"><code>nginx</code> <code>-t</code></span> command to verify that the file is syntactically valid.</p> <p></p><pre><code class="terminal">root# <span style="color:#66ff99; font-weight: bold;">nginx -t</span><br>
nginx: the configuration file /etc/nginx/nginx.conf syntax is ok<br>
nginx: configuration file /etc/nginx/nginx.conf test is successful</code></pre> </li> 
          <li> <p>Start NGINX&nbsp;Plus. If NGINX&nbsp;Plus is already running, run the following command to reload the configuration file:</p> <p></p><pre><code class="terminal">root# <span style="color:#66ff99; font-weight: bold;">nginx -s reload</span></code></pre> </li> 
          <li> <p>Run the following commands on the appropriate hosts to start the ldap?auth daemon and the backend daemon.</p> <p></p><pre><code class="terminal">root# <span style="color:#66ff99; font-weight: bold;">nginx-ldap-auth-daemon-ctl.sh start</span><br>
root# <span style="color:#66ff99; font-weight: bold;">python backend-sample-app.py</span></code></pre> </li> 
          <li> <p>Use a web browser to access <span style="white-space: nowrap; font-weight: bold;">http://<em>nginx-server-address</em>:8081</span>. Verify that the browser presents the authentication form. After you fill out the form and submit it, verify that the server returns the expected response to valid credentials. As noted above, the backend daemon returns the following text:</p> <p></p><pre><code class="config">   Hello, world! Requested URL: <em>URL</em></code></pre> </li> 
         </ol> 
         <h2 id="ldap-auth-configure">Configuring the Reference Implementation</h2> 
         <p>Make the following changes in the <span style="white-space: nowrap; font-weight: bold;">nginx-ldap-auth.conf</span> file. Some are required and some optional, as indicated.</p> 
         <h3>LDAP Server Settings</h3> 
         <p>As implemented in <span style="white-space: nowrap; font-weight: bold;">nginx-ldap-auth-daemon.py</span>, the ldap?auth daemon communicates with an OpenLDAP server, passing in parameters to specify which user account to authenticate. To eliminate the need to modify the Python code, the <span style="white-space: nowrap; font-weight: bold;">nginx-ldap-auth.conf</span> file contains <a target="_blank" href="http://nginx.org/en/docs/http/ngx_http_proxy_module.html#proxy_set_header"><code>proxy_set_header</code></a> directives that set values in the HTTP header that are then used to set the parameters. The following table maps the parameters and headers.</p> 
         <table class="nginx-blog"> 
          <tbody> 
          </tbody>
          <colgroup>
           <col width="30%"> 
          </colgroup>
          <tbody>
           <tr> 
            <th>LDAP Parameter</th> 
            <th>HTTP Header</th> 
           </tr> 
           <tr> 
            <td><code>basedn</code></td> 
            <td><code>X-Ldap-BaseDN</code></td> 
           </tr> 
           <tr> 
            <td><code>binddn</code></td> 
            <td><code>X-Ldap-BindDN</code></td> 
           </tr> 
           <tr> 
            <td><code>bindpasswd</code></td> 
            <td><code>X-Ldap-BindPass</code></td> 
           </tr> 
           <tr> 
            <td><code>cookiename</code></td> 
            <td><code>X-CookieName</code></td> 
           </tr> 
           <tr> 
            <td><code>realm</code></td> 
            <td><code>X-Ldap-Realm</code></td> 
           </tr> 
           <tr> 
            <td><code>template</code></td> 
            <td><code>X-Ldap-Template</code></td>

         <ul> 
          <li> <p>(Required) In the following directives, replace the values in bold with the correct values for your LDAP server deployment. Note in particular that the <span style="white-space: nowrap; font-weight: bold;">nginx-ldap-auth.conf</span> file uses the well?known port for LDAPS,&nbsp;636. If you change the port to&nbsp;389 (the well?known port for LDAP) or another LDAP port, remember also to change the protocol name from <code>ldaps</code> to <code>ldap</code>.</p> <p></p><pre><code class="config"># URL and port for connecting to the LDAP server<br>
proxy_set_header X-Ldap-URL "<strong>ldaps</strong>://<strong>example.com</strong>:<strong>636</strong>";<p></p>
<p># Base DN<br> proxy_set_header X-Ldap-BaseDN "<strong>cn=Users,dc=test,dc=local</strong>";</p>
<p># Bind DN<br> proxy_set_header X-Ldap-BindDN "<strong>cn=root,dc=test,dc=local</strong>";</p>
<p># Bind password<br> proxy_set_header X-Ldap-BindPass "<strong>secret</strong>";</p></code></pre> </li> 
          <li> <p>(Required if using Active Directory instead of OpenLDAP) Uncomment the following directive as shown:</p> <p></p><pre><code class="config">proxy_set_header X-Ldap-Template "(SAMAccountName=%(username)s)";</code></pre> </li> 
          <li> <p>(Optional) The reference implementation uses cookie?based authentication. If you are using HTTP basic authentication instead, comment out the following directives as shown:</p> <p></p><pre><code class="config"><strong>#</strong>proxy_set_header X-CookieName "nginxauth";<br>
<strong>#</strong>proxy_set_header Cookie nginxauth=$cookie_nginxauth;</code></pre> </li> 
          <li> <p>(Optional) If you want to change the value for the <code>template</code> parameter that the ldap?auth daemon passes to the OpenLDAP server by default, uncomment the following directive as shown, and change the value:</p> <p></p><pre><code class="config">proxy_set_header X-Ldap-Template "<strong>(cn=%(username)s)</strong>";</code></pre> </li> 
          <li> <p>(Optional) If you want to change the realm name from the default value (<strong>Restricted</strong>), uncomment and change the following directive:</p> <p></p><pre><code class="config">proxy_set_header X-Ldap-Realm "<strong>Restricted</strong>";</code></pre> </li> 
         </ul> 
         <h3>IP Address for Backend Daemon</h3> 
         <p>If the backend daemon is not running on the same host as NGINX&nbsp;Plus, change the IP address for it in the <a target="_blank" href="http://nginx.org/en/docs/http/ngx_http_upstream_module.html#upstream"><code>upstream</code></a> configuration block:</p> 
         <pre><code class="config">upstream backend {<br>
    server <strong>127.0.0.1</strong>:9000;<br>
 }</code></pre> 
         <h3>IP Address for ldap?auth Daemon</h3> 
         <p>If the ldap?auth daemon is not running on the same host as NGINX&nbsp;Plus, change the IP address in this <a target="_blank" href="http://nginx.org/en/docs/http/ngx_http_proxy_module.html#proxy_pass"><code>proxy_pass</code></a> directive:</p> 
         <pre><code class="config">location = /auth-proxy {<br>
    proxy_pass http://<strong>127.0.0.1</strong>:8888;<br>
    # ...<br>
}</code></pre> 
         <h3>IP Address and Port on Which NGINX Listens</h3> 
         <p>If the client is not running on the same host as NGINX&nbsp;Plus, change the IP address in this <a target="_blank" href="http://nginx.org/en/docs/http/ngx_http_core_module.html#listen"><code>listen</code></a> directive (or remove the address completely to accept traffic from any client). You can also change the port on which NGINX listens from&nbsp;8081 if you wish:</p> 
         <pre><code class="config">server {<br>
    listen <strong>127.0.0.1</strong>:<strong>8081</strong>;<br>
    # ...<br>
 }</code></pre> 
         <h3 id="ldap-auth-caching">Caching</h3> 
         <p>The <span style="white-space: nowrap; font-weight: bold;">nginx-ldap-auth.conf</span> file enables caching of both data and credentials. Optionally, you can change the following settings:</p> 
         <ul> 
          <li> <p>The <a target="_blank" href="http://nginx.org/en/docs/http/ngx_http_proxy_module.html#proxy_cache_path"><code>proxy_cache_path</code></a> directive in the <code>http</code> configuration block creates a local disk directory called <strong>cache</strong>, and allocates 10&nbsp;MB in shared memory for a zone called <strong>auth_cache</strong>, where metadata is stored.</p> <pre><code class="config">proxy_cache_path <strong>cache/</strong> keys_zone=<strong>auth_cache</strong>:<strong>10m</strong>;</code></pre> <p>If you change the name of the shared memory zone, you must also change it in the <a target="_blank" href="http://nginx.org/en/docs/http/ngx_http_proxy_module.html#proxy_cache"><code>proxy_cache</code></a> directive (in the <a target="_blank" href="http://nginx.org/en/docs/http/ngx_http_core_module.html#location"><code>location</code></a> block that directs traffic to the ldap?auth daemon).</p> <p></p><pre><code class="config">location = /auth-proxy {<br>
    proxy_cache <strong>auth_cache</strong>;<br>
    # ...<br>
}</code></pre> </li> 
          <li> <p>The <a target="_blank" href="http://nginx.org/en/docs/http/ngx_http_proxy_module.html#proxy_cache_valid"><code>proxy_cache_valid</code></a> directive (in the same <code>location</code> block as <code>proxy_cache</code>) specifies that cached responses marked with HTTP code <code>200</code> or <code>403</code> are valid for 10&nbsp;minutes.</p> <p></p><pre><code class="config">location = /auth-proxy {<br>
    proxy_cache_valid 200 403 <strong>10m</strong>;<br>
    # ...<br>
}</code></pre> </li> 
         </ul> 
         <p>To disable caching, comment out these three directives plus the <code>proxy_cache_key</code> directive.</p> 
         <h2>Customizing the Authentication System</h2> 
         <p>As mentioned above, you can use the ldap?auth daemon as a model for your own application that accepts requests from the http_auth_request module. If writing an app in Python to communicate with a different (non?LDAP) type of authentication server, write a new authentication?handler class to replace <code>LDAPAuthHandler</code> in the <span style="white-space: nowrap; font-weight: bold;">nginx-ldap-auth-daemon.py</span> script.</p> 
         <h2>A Note about Security</h2> 
         <p>The backend daemon uses Base64 encoding on the username and password in the cookie. Base64 is a very weak form of scrambling, rendering the credentials vulnerable to extraction and misuse. For authentication to serve any real purpose, you need to use more sophisticated encryption in your backend application.</p>
