  <p>The <font face="monospaced,courier" size="-1"><b>HelloGraphics</b></font> example in <a href="Introduction.html">Chapter 1</a> offers a simple example of how to write graphical programs, but does not explain the details behind the methods it contains. The purpose of this chapter is to give you a working knowledge of the facilities available in the <font face="monospaced,courier" size="-1"><b>acm.graphics</b></font> package and how to use them effectively. </p>
  <p><font face="monospaced,courier" size="-1"><b>&nbsp;&nbsp;</b></font>The class structure of <font face="monospaced,courier" size="-1"><b>acm.graphics</b></font> package appears in <a href="#FigGraphicsClasses">Figure 2-1</a>. Conceptually, <font face="monospaced,courier" size="-1"><b>GObject</b></font> represents the universal class of graphical objects that can be displayed. When you use <font face="monospaced,courier" size="-1"><b>acm.graphics</b></font>, you assemble a picture by constructing various

  <h2 clear="all"><a name="Coordinates"></a>2.1 The acm.graphics model</h2> 
  <p>When you create a picture using the <font face="monospaced,courier" size="-1"><b>acm.graphics</b></font> package, you do so by arranging graphical objects at various positions on a background called a <b>canvas.</b> The underlying model is similar to that of a collage in which an artist creates a composition by taking various objects and assembling them on a background canvas. In the world of the collage artist, those objects might be geometrical shapes, words clipped from newspapers, lines formed from bits of string, or images taken from magazines. In the <font face="monospaced,courier" size="-1"><b>acm.graphics</b></font> package, there are counterparts for each of these graphical objects. </p>
  <h4>The ?felt board? metaphor</h4> 
  <p>Another metaphor that often helps students understand the conceptual model of the <font face="monospaced,courier" size="-1"><b>acm.graphics</b></font> package is that of a felt board?the sort one might find in an elementary school classroom. A child creates pictures by taking shapes of colored felt and sticking them onto a large felt board that serves as the background canvas for the picture as a whole. The pieces stay where the child puts them because felt fibers interlock tightly enough for the pieces to stick together. The left side of <a href="#FigFeltBoard">Figure 2-2</a> shows a physical felt board with a red rectangle and a green oval attached. The right side of the figure is the virtual equivalent in the <font face="monospaced,courier" size="-1"><b>acm.graphics</b></font> world. To create the picture, you would need to create two graphical objects?a red rectangle and a green oval?and add them to the graphical canvas that forms the background. </p>

  <p><font face="monospaced,courier" size="-1"><b>&nbsp;&nbsp;</b></font>In this tutorial, the order in which objects are layered on the canvas will be called the <b>stacking order</b>. (In more mathematical descriptions, this ordering is often called <b><i>z</i>-ordering</b>, because the <i>z</i>-axis is the one that projects outward from the screen.) Whenever a new object is added to a canvas, it appears at the front of the stack. Graphical objects are always drawn from back to front so that the frontmost objects overwrite those that are further back. </p>

  <h4>The coordinate system</h4> 
  <p>The <font face="monospaced,courier" size="-1"><b>acm.graphics</b></font> package uses the same basic coordinate system that traditional Java programs do. Coordinate values are expressed in terms of pixels, which are the individual dots that cover the face of the screen. Each pixel in a graphics window is identified by its <i>x</i> and <i>y</i> coordinates, with <i>x</i> values increasing as you move rightward across the window and <i>y</i> values increasing as you move down from the top. The point (0,&nbsp;0)?which is called the <b>origin</b>?is in the upper left corner of the window. This coordinate system is illustrated by the diagram in <a href="#FigCoordinateSystem">Figure 2-4</a>, which shows only the red rectangle from the <font face="monospaced,courier" size="-1"><b>FeltBoard.java</b></font> program. The location of that rectangle is (100,&nbsp;50), which means that its upper left corner is 100 pixels to the right and 50 pixels down from the origin of the graphics window. </p>
  <p><a name="FigCoordinateSystem"></a><font face="sans-serif,helvetica,arial,geneva"><b>Figure 2-4. The Java coordinate system</b></font></p>
  <table bgcolor="#FFFFFF" border="1" cellspacing="0" cellpadding="4" width="750">

  <p><font face="monospaced,courier" size="-1"><b>&nbsp;&nbsp;</b></font>The only difference between the coordinate systems used in the <font face="monospaced,courier" size="-1"><b>acm.graphics</b></font> package and Java?s <font face="monospaced,courier" size="-1"><b>Graphics</b></font> class is that the <font face="monospaced,courier" size="-1"><b>acm.graphics</b></font> package uses 
   <nobr>
    <font face="monospaced,courier" size="-1"><b>double</b></font>s
   </nobr> to represent coordinate values instead of 
   <nobr>
    <font face="monospaced,courier" size="-1"><b>int</b></font>s
   </nobr>. This change makes it easier to create figures whose locations and dimensions are produced by mathematical calculations in which the results are typically not whole numbers. As a simple example, the dimensions of the red rectangle in Figure 2-4 are proportional to the <i>golden ratio</i>, which Greek mathematicians believed gave rise to the most pleasing aesthetic effect. The golden ratio is approximately equal to 1.618 and is usually denoted in mathematics by the symbol f. Because the <font face="monospaced,courier" size="-1"><b>acm.graphics</b></font> package uses 
   <nobr>
    <font face="monospaced,courier" size="-1"><b>double</b></font>s
   </nobr> to specify coordinates and dimensions, the code to generate the rectangle looks like this: </p>

  <p>In the integer-based Java model, it would be necessary to include explicit code to convert the height parameter to an <font face="monospaced,courier" size="-1"><b>int</b></font>. In addition to adding complexity to the code, forcing students to convert coordinates to integers can introduce rounding errors that distort the geometry of the displayed figures. </p>
  <p><font face="monospaced,courier" size="-1"><b>&nbsp;&nbsp;</b></font>Judging from the experience of the instructors who tested the <font face="monospaced,courier" size="-1"><b>acm.graphics</b></font> package while it was in development, the change from 
   <nobr>
    <font face="monospaced,courier" size="-1"><b>int</b></font>s
   </nobr> to 
   <nobr>
    <font face="monospaced,courier" size="-1"><b>double</b></font>s
   </nobr> causes no confusion but instead represents an important conceptual simplification. The only aspect of Java?s coordinate system that students find problematic is the fact that the origin is in a different place from what they know from traditional Cartesian geometry. Fortunately, it doesn?t take too long to become familiar with the Java model. <a name="GeometricalTypes"></a></p>
  <h4>The GPoint, GDimension, and GRectangle classes</h4> 
  <p>Although it is usually possible to specify individual values for coordinate values, it is often convenient to encapsulate an <i>x</i> and a <i>y</i> coordinate as a point, a <i>width</i> and a <i>height</i> value as a composite indication of the dimensions of an object, or all four values as the bounding rectangle for a figure. Because the coordinates are stored as 
   <nobr>
    <font face="monospaced,courier" size="-1"><b>double</b></font>s
   </nobr> in the <font face="monospaced,courier" size="-1"><b>acm.graphics</b></font> package, using Java?s integer-based <font face="monospaced,courier" size="-1"><b>Point</b></font>, <font face="monospaced,courier" size="-1"><b>Dimension</b></font>, and <font face="monospaced,courier" size="-1"><b>Rectangle</b></font> classes would entail a loss of precision. To avoid this problem the <font face="monospaced,courier" size="-1"><b>acm.graphics</b></font> package exports the classes <font face="monospaced,courier" size="-1"><b>GPoint</b></font>, <font face="monospaced,courier" size="-1"><b>GDimension</b></font>, and <font face="monospaced,courier" size="-1"><b>GRectangle</b></font>, which have the same semantics as their standard counterparts except for the fact that their coordinates are 
   <nobr>
    <font face="monospaced,courier" size="-1"><b>double</b></font>s
   </nobr>. </p>
  <p><font face="monospaced,courier" size="-1"><b>&nbsp;&nbsp;</b></font>As an example, the declaration </p>

The advantage of encapsulating these values into objects is that they can then be passed from one method to another using a single variable. </p>
  <h4>The GMath class</h4> 
  <p>Computing the coordinates of a graphical design can sometimes require the use of simple trigonometric functions. Although functions like <font face="monospaced,courier" size="-1"><b>sin</b></font> and <font face="monospaced,courier" size="-1"><b>cos</b></font> are defined in Java?s standard <font face="monospaced,courier" size="-1"><b>Math</b></font> class, students find them confusing in graphical applications because of inconsistencies in the way angles are represented. In Java?s graphics libraries, angles are measured in degrees; in the <font face="monospaced,courier" size="-1"><b>Math</b></font> class, angles must be given in radians. To minimize the confusion associated with this inconsistency of representation, the <font face="monospaced,courier" size="-1"><b>acm.graphics</b></font> package includes a class called <font face="monospaced,courier" size="-1"><b>GMath</b></font>, which exports the methods shown in <a href="#FigGMathMethods">Figure 2-5</a>. Most of these methods are simply degree-based versions of the standard trigonometric functions, but the <font face="monospaced,courier" size="-1"><b>distance</b></font>, <font face="monospaced,courier" size="-1"><b>angle</b></font>, and <font face="monospaced,courier" size="-1"><b>round</b></font> methods are also worth noting. </p>

  <h2 clear="all"><a name="GCanvas"></a>2.2 The GCanvas class</h2> 
  <p>In the <font face="monospaced,courier" size="-1"><b>acm.graphics</b></font> model, pictures are created by adding graphical objects?each of which is an instance of the <font face="monospaced,courier" size="-1"><b>GObject</b></font> class hierarchy described in <a href="#GObject">section 2.3</a>?to a background canvas. That background?the analogue of the felt board in the physical world?is provided by the <font face="monospaced,courier" size="-1"><b>GCanvas</b></font> class. The <font face="monospaced,courier" size="-1"><b>GCanvas</b></font> class is a lightweight component and can be added to any Java container in either the <font face="monospaced,courier" size="-1"><b>java.awt</b></font> or <font face="monospaced,courier" size="-1"><b>javax.swing</b></font> packages, which makes it possible to use the graphics facilities in any Java application. For the most part, however, students in introductory courses won?t use the <font face="monospaced,courier" size="-1"><b>GCanvas</b></font> class directly but will instead use the <font face="monospaced,courier" size="-1"><b>GraphicsProgram</b></font> class, which automatically creates a <font face="monospaced,courier" size="-1"><b>GCanvas</b></font> and installs it in the program window, as illustrated in several preceding examples. The <font face="monospaced,courier" size="-1"><b>GraphicsProgram</b></font> class forwards operations such as <font face="monospaced,courier" size="-1"><b>add</b></font> and <font face="monospaced,courier" size="-1"><b>remove</b></font> to the embedded <font face="monospaced,courier" size="-1"><b>GCanvas</b></font> so that students don?t need to be aware of the underlying implementation details. </p>
  <p><font face="monospaced,courier" size="-1"><b>&nbsp;&nbsp;</b></font>The most important methods supported by the <font face="monospaced,courier" size="-1"><b>GCanvas</b></font> class are shown in <a href="#FigGCanvasMethods">Figure 2-6</a>. Many of these methods are concerned with adding and removing graphical objects. These methods are easy to understand, particularly if you keep in mind that a <font face="monospaced,courier" size="-1"><b>GCanvas</b></font> is conceptually a container for <font face="monospaced,courier" size="-1"><b>GObject</b></font> values. The container metaphor explains the functionality provided by the <font face="monospaced,courier" size="-1"><b>add</b></font>, <font face="monospaced,courier" size="-1"><b>remove</b></font>, and <font face="monospaced,courier" size="-1"><b>removeAll</b></font> methods in Figure 2-6, which are analogous to the identically named methods in <font face="monospaced,courier" size="-1"><b>JComponent</b></font> and <font face="monospaced,courier" size="-1"><b>Container</b></font>. </p>

  <h2 clear="all"><a name="GObject"></a>2.3 The GObject class</h2> 
  <p>The <font face="monospaced,courier" size="-1"><b>GObject</b></font> class represents the universe of graphical objects that can be displayed on a <font face="monospaced,courier" size="-1"><b>GCanvas</b></font>. The <font face="monospaced,courier" size="-1"><b>GObject</b></font> class itself is abstract, which means that programs never create instances of the <font face="monospaced,courier" size="-1"><b>GObject</b></font> class directly. Instead, programs create instances of one of the <font face="monospaced,courier" size="-1"><b>GObject</b></font> subclasses that represent specific graphical objects such as rectangles, ovals, and lines. The most important such classes are the ones that appear at the bottom of the class diagram from <a href="#FigShapeClasses">Figure 2-1</a>, which are collectively called the <b>shape classes.</b> The shape classes are described in detail in <a href="#ShapeClasses">section 2.4</a>. Before going into those details, however, it makes sense to begin by describing the characteristics that are common to the <font face="monospaced,courier" size="-1"><b>GObject</b></font> class as a whole. </p>

  <h4>Determining the location and size of a GObject</h4> 
  <p>The first several methods in Figure 2-7 make it possible to determine the location and size of any <font face="monospaced,courier" size="-1"><b>GObject</b></font>. The <font face="monospaced,courier" size="-1"><b>getX</b></font>, <font face="monospaced,courier" size="-1"><b>getY</b></font>, <font face="monospaced,courier" size="-1"><b>getWidth</b></font>, and <font face="monospaced,courier" size="-1"><b>getHeight</b></font> methods return these coordinate values individually, and the <font face="monospaced,courier" size="-1"><b>getLocation</b></font>, <font face="monospaced,courier" size="-1"><b>getSize</b></font>, and <font face="monospaced,courier" size="-1"><b>getBounds</b></font> methods return composite values that encapsulate that information in a single object, as described in <a href="#GeometricalTypes">section 2.1</a>. </p>

  <h4>The GFillable, GResizable, and GScalable interfaces</h4> 
  <p>You have probably noticed that several of the examples you?ve already seen in this tutorial include methods that do not appear in the list in <a href="#FigGObjectMethods">Figure 2-7</a>. For example, the <font face="monospaced,courier" size="-1"><b>FeltBoard</b></font> program includes calls to a <font face="monospaced,courier" size="-1"><b>setFilled</b></font> method to mark the rectangle and oval as filled rather than outlined. Looking at Figure 2-7, it appears that the <font face="monospaced,courier" size="-1"><b>GObject</b></font> class does not include a <font face="monospaced,courier" size="-1"><b>setFilled</b></font> method, which is indeed the case. </p>
  <p><font face="monospaced,courier" size="-1"><b>&nbsp;&nbsp;</b></font>As the caption for Figure 2-7 makes clear, the methods listed in that table are the ones that are common to <i>every</i> <font face="monospaced,courier" size="-1"><b>GObject</b></font> subclass. While it is always possible to set the location of a graphical object, it is only possible to fill that object if the idea of ?filling? makes sense for that class. Filling is easily defined for geometrical shapes such as ovals, rectangles, polygons, and arcs, but it is not clear what it might mean to fill a line, an image, or a label. Since there are subclasses that cannot give a meaningful interpretation to <font face="monospaced,courier" size="-1"><b>setFilled</b></font>, that method is not defined at the <font face="monospaced,courier" size="-1"><b>GObject</b></font> level but is instead implemented only for those subclasses for which filling is defined. </p>
  <p><font face="monospaced,courier" size="-1"><b>&nbsp;&nbsp;</b></font>At the same time, it is important to define the <font face="monospaced,courier" size="-1"><b>setFilled</b></font> method so that it works the same way for any class that implements it. If <font face="monospaced,courier" size="-1"><b>setFilled</b></font>, for example, worked differently in the <font face="monospaced,courier" size="-1"><b>GRect</b></font> and <font face="monospaced,courier" size="-1"><b>GOval</b></font> classes, trying to keep track of the different styles would inevitably cause confusion.   <h2 clear="all"><a name="ShapeClasses"></a>2.4 Descriptions of the individual shape classes</h2> 
  <p>So far, this tutorial has looked only at methods that apply to all 
   <nobr>
    <font face="monospaced,courier" size="-1"><b>GObject</b></font>s
   </nobr>, along with a few interfaces that define methods shared by some subset of the <font face="monospaced,courier" size="-1"><b>GObject</b></font> hierarchy. The most important classes in that hierarchy are the shape classes that appear at the bottom of <a href="#FigGraphicsClasses">Figure 2-1</a>. The sections that follow provide additional background on each of the shape classes and include several simple examples that illustrate their use. </p>
  <p><font face="monospaced,courier" size="-1"><b>&nbsp;&nbsp;</b></font>As you go through the descriptions of the individual shape classes, you are likely to conclude that some of them are designed in ways that are less than ideal for introductory students. In the abstract, this conclusion is almost certainly correct. For practical reasons that look beyond the introductory course, the Java Task Force decided to implement the shape classes so that they match their counterparts in Java?s standard <font face="monospaced,courier" size="-1"><b>Graphics</b></font> class. In particular, the set of shape classes corresponds precisely to the facilities that the <font face="monospaced,courier" size="-1"><b>Graphics</b></font> class offers for drawing geometrical shapes, text strings, and images. Moreover, the constructors for each class take the same parameters and have the same semantics as the corresponding method in the <font face="monospaced,courier" size="-1"><b>Graphics</b></font> class. Thus, the <font face="monospaced,courier" size="-1"><b>GArc</b></font> constructor?which is arguably the most counterintuitive in many ways?has the structure it does, not because we thought that structure was perfect, but because that is the structure used by the <font face="monospaced,courier" size="-1"><b>drawArc</b></font> method in the <font face="monospaced,courier" size="-1"><b>Graphics</b></font> class. By keeping the semantics consistent with its Java counterpart, the <font face="monospaced,courier" size="-1"><b>acm.graphics</b></font> package makes it easier for students to move on to the standard packages as they learn more about programming. <a name="GRect"></a></p>

  <ul> 
   <!--&#149;-->
   <li>Calling <font face="monospaced,courier" size="-1"><b>setLocation(x,&nbsp;y)</b></font> or <font face="monospaced,courier" size="-1"><b>move(dx,&nbsp;dy)</b></font> on a <font face="monospaced,courier" size="-1"><b>GLine</b></font> object moves the line without changing its length or orientation.    <li>The <font face="monospaced,courier" size="-1"><b>GLine</b></font> class implements <font face="monospaced,courier" size="-1"><b>GScalable</b></font>?which expands or contracts the line relative to its starting point?but not <font face="monospaced,courier" size="-1"><b>GFillable</b></font> or <font face="monospaced,courier" size="-1"><b>GResizable</b></font>. 
    <!--&#149;--></li>
   <li>From a mathematical perspective, a line has no thickness and therefore does not actually any points.     <!--&#149;--></li>

  </ul>

  <h4>The <font face="monospaced,courier" size="-1"><b><font size="+1">GLabel</font></b></font> class</h4> 
  <p>The <a href="../javadoc/student#acm/graphics/GLabel.html"><font face="monospaced,courier" size="-1"><b>GLabel</b></font></a> class is used to display text strings on the canvas. The <font face="monospaced,courier" size="-1"><b>GLabel</b></font> class is different from the other shape classes because the operations one wants to perform on strings are different from those that are appropriate for geometrical figures. The most important thing to realize is that the position of a <font face="monospaced,courier" size="-1"><b>GLabel</b></font> is not defined by the upper left corner, but by the starting point of the <b>baseline,</b> which is the imaginary line on which the characters sit. The origin and baseline properties of the <font face="monospaced,courier" size="-1"><b>GLabel</b></font> class are illustrated in the following diagram: </p>

  <p><font face="monospaced,courier" size="-1"><b>&nbsp;&nbsp;</b></font>Several of the methods in Figure 2-14 are included to assist in the process of positioning the <font face="monospaced,courier" size="-1"><b>GLabel</b></font>. The most important methods for controlling the position, however, are not specific to the <font face="monospaced,courier" size="-1"><b>GLabel</b></font> class but are simply the <font face="monospaced,courier" size="-1"><b>getWidth</b></font> and <font face="monospaced,courier" size="-1"><b>getHeight</b></font> methods inherited from <font face="monospaced,courier" size="-1"><b>GObject</b></font>. As they do for all graphical objects, these methods return the dimensions of the rectangle that bounds the figure. The <font face="monospaced,courier" size="-1"><b>getWidth</b></font> method returns the horizontal extent of the label and will change if you change either the font or the internal string. The <font face="monospaced,courier" size="-1"><b>getHeight</b></font> method returns the height of a single line of text in the current font, which is defined to be the distance between the baselines of successive lines. The <font face="monospaced,courier" size="-1"><b>getAscent</b></font> and <font face="monospaced,courier" size="-1"><b>getDescent</b></font> methods return the maximum distance the current font extends above and below the baseline, respectively. </p>
  <p><font face="monospaced,courier" size="-1"><b>&nbsp;&nbsp;</b></font>Now that you know about these methods, you are finally in a position to understand the details of how the code in the <font face="monospaced,courier" size="-1"><b>HelloGraphics</b></font> example from <a href="Introduction.html">Chapter 1</a> centers the label on the canvas. The relevant lines look like this: </p>
  <p></p>

  <p>The calculation of the <i>x</i> coordinate moves rightward half the width of the canvas as a whole, but subtracts away half the width of the <font face="monospaced,courier" size="-1"><b>GLabel</b></font> to reach its origin point. The calculation of the <i>y</i> coordinate is similar, but this time the offset is half the distance the font extends above the baseline, which is given by <font face="monospaced,courier" size="-1"><b>getAscent</b></font>. The plus sign in the calculation of the <i>y</i> coordinate value may initially seem confusing, but the confusion disappears when you remember that <i>y</i> values increase as you move down the screen. </p>
  <p><font face="monospaced,courier" size="-1"><b>&nbsp;&nbsp;</b></font>If you work with a lot of text on the canvas, you will probably discover at some point that the strategy of using <font face="monospaced,courier" size="-1"><b>getAscent</b></font> to center a <font face="monospaced,courier" size="-1"><b>GLabel</b></font> vertically doesn?t quite work. Most labels that you display on the canvas will appear to be a few pixels too low. The reason for this behavior is that <font face="monospaced,courier" size="-1"><b>getAscent</b></font> returns the <i>maximum</i> ascent of the font and not the distance the text of this particular label happens to rise above the baseline. For most fonts, the parentheses and diacritical marks extend above the tops of the uppercase letters and therefore make the font ascent larger than it seems to be in practice. If you are a stickler for aesthetics, you may need to adjust the vertical centering by a pixel or two to have things look precisely right. <a name="GImage"></a></p>
  <h4>The <font face="monospaced,courier" size="-1"><b><font size="+1">GImage</font></b></font> class</h4> 
  <p>The <a href="../javadoc/student#acm/graphics/GImage.html"><font face="monospaced,courier" size="-1"><b>GImage</b></font></a> class is used to display an image on the canvas. That image must be presented in one of the standard formats for image data, of which the most common are the GIF (Graphics Interchange Format) and JPEG (Joint Photographic Experts Group) formats. To create an image, you will need to use an image processing tool (such as Adobe Photoshop?) or download the image from the web. Unless they are specifically marked as being in public domain, images you find on the web are usually subject to copyright protection, which means that you must ensure that any use you make of those images fits under the <a href="http://fairuse.stanford.edu/">standard ?fair use? guidelines</a>. </p>
  <p><font face="monospaced,courier" size="-1"><b>&nbsp;&nbsp;</b></font>Once you have created the image file, you need to put it in a place where your Java application can find it. The standard <font face="monospaced,courier" size="-1"><b>GImage</b></font> constructor takes the name of the image and then looks for the image data in the following places: </p>
  <ol> 
   <!--1.-->
   <li>Inside the JAR file for the application, looking first for an image resource in the top-level package and then for one in a package named <font face="monospaced,courier" size="-1"><b>images</b></font>. 
    <!--2.--></li>
   <li>Inside the directory that contains the application, looking for an image file with the appropriate name. 
    <!--3.--></li>
   <li>Inside a subdirectory called <font face="monospaced,courier" size="-1"><b>images</b></font> of the application directory. </li>
  </ol> 
  <p>If the image is found in any of those places, it is loaded automatically to create a Java <font face="monospaced,courier" size="-1"><b>Image</b></font> object. If none of these places contains the image, the <font face="monospaced,courier" size="-1"><b>GImage</b></font> constructor generates a runtime error. A more extensive discussion of the search strategy for images appears in the documentation for the <a href="../javadoc/student#acm/util/MediaTools.html"><font face="monospaced,courier" size="-1"><b>MediaTools</b></font></a> class in the <font face="monospaced,courier" size="-1"><b>acm.util</b></font> package. </p>

  <h4>The <font face="monospaced,courier" size="-1"><b><font size="+1">GPolygon</font></b></font> class</h4> 
  <p>The shape that represents the greatest deviation from the traditional Java model is the <a href="../javadoc/student#acm/graphics/GPolygon.html"><font face="monospaced,courier" size="-1"><b>GPolygon</b></font></a> class, which is used to draw closed polygonal shapes. Unlike the other shape classes, the constructor for the <font face="monospaced,courier" size="-1"><b>GPolygon</b></font> does not create a fully formed object but instead creates a polygon with no vertices. Starting with that empty polygon, you can then add additional vertices using any of three different methods?<font face="monospaced,courier" size="-1"><b>addVertex</b></font>, <font face="monospaced,courier" size="-1"><b>addEdge</b></font>, and <font face="monospaced,courier" size="-1"><b>addPolarEdge</b></font>?that are defined as part of the <font face="monospaced,courier" size="-1"><b>GPolygon</b></font> class. </p>
  <p><font face="monospaced,courier" size="-1"><b>&nbsp;&nbsp;</b></font>These methods are easiest to illustrate by example. Before doing so, however, it is important to note that the coordinates of each vertex are not expressed in the global coordinate space of the canvas but are instead defined in relation to a point chosen to represent the <b>origin</b> of the polygon. The origin need not be one of the vertices and is typically chosen to be the center, particularly for regular polygonal shapes. Setting the location of a <font face="monospaced,courier" size="-1"><b>GPolygon</b></font> corresponds to specifying the location of its origin, and the polygon is then drawn relative to that point. The advantage of this strategy is that moving a polygon requires changing the coordinates of just one point and does not require adjusting the coordinates of each vertex. </p>
  <p><font face="monospaced,courier" size="-1"><b>&nbsp;&nbsp;</b></font>The simplest method to explain is <font face="monospaced,courier" size="-1"><b>addVertex(x,&nbsp;y)</b></font>, which adds a vertex at the point (<font face="monospaced,courier" size="-1"><b>x</b></font>,&nbsp;<font face="monospaced,courier" size="-1"><b>y</b></font>) relative to the location of the polygon. Suppose, for example, that you wanted to draw a diamond shape, 80 pixels high and 60 pixels wide, that looks like this: </p>

  <h2 clear="all"><a name="GCompound"></a>2.5 The GCompound class</h2> 
  <p>The shape classes described in <a href="#ShapeClasses">Section 2.4</a> are the basic building blocks that allow you to build more complicated structures. In a sense, these classes represent the ?atoms? of the <font face="monospaced,courier" size="-1"><b>acm.graphics</b></font> world. In the <font face="monospaced,courier" size="-1"><b>acm.graphics</b></font> package, this facility is provided by the <a href="../javadoc/student#acm/graphics/GCompound.html"><font face="monospaced,courier" size="-1"><b>GCompound</b></font></a> class. </p>

  <h4>The <font face="monospaced,courier" size="-1"><b><font size="+1">GCompound</font></b></font> coordinate system</h4> 
  <p>The general paradigm for using <font face="monospaced,courier" size="-1"><b>GCompound</b></font> is to create an empty instance of the class and then to add other graphical objects to it. This strategy means that you can add a compound object to a canvas and the move all its elements as a unit simply by setting the location of the compound.   <h4>Recentering objects using <font face="monospaced,courier" size="-1"><b><font size="+1">GCompound</font></b></font></h4> 
  <p>The fact that the <font face="monospaced,courier" size="-1"><b>GCompound</b></font> class maintains its own coordinate system has an additional advantage that may not immediately spring to mind: it allows you to change the reference point for a single graphical object, usually to make the interpretation of that reference point more closely correspond to the way that object might behave in the real world. As an example, consider how you might represent a bouncing ball on the canvas. The obvious solution is to use a <font face="monospaced,courier" size="-1"><b>GOval</b></font> with equal width and height so that it appears as a circle. The only problem with that strategy is that the reference point of a <font face="monospaced,courier" size="-1"><b>GOval</b></font> is in the upper left corner. If you want to perform any physical calculations involving the ball, it would be far better if the location of the ball were defined to be its center. </p>
  <p><font face="monospaced,courier" size="-1"><b>&nbsp;&nbsp;</b></font>The simplest way to accomplish this shift in the reference point from the corner to the center is to use the <font face="monospaced,courier" size="-1"><b>GCompound</b></font> class. If you add a <font face="monospaced,courier" size="-1"><b>GOval</b></font> with radius <i>r</i> so that its location in the coordinate system of the compound is at (?<i>r</i>,&nbsp;?<i>r</i>), then the <font face="monospaced,courier" size="-1"><b>GCompound</b></font> will display itself on the canvas as a circle centered at the location of the <font face="monospaced,courier" size="-1"><b>GCompound</b></font> object. The code in <a href="#FigGBallCode">Figure 2-18</a> shows how to use this strategy to create a new <font face="monospaced,courier" size="-1"><b>GBall</b></font> class whose location represents the center of the ball. You?ll have a chance to see this class in action in <a href="AnimationAndInteractivity.html">Chapter 3</a> which includes code to animate a <font face="monospaced,courier" size="-1"><b>GBall</b></font> object so that it bounces around inside the boundaries of the canvas. </p>

  <h2 clear="all"><a name="GPen"></a>2.6 The GPen and GTurtle classes</h2> 
  <p>The remaining two classes in the <font face="monospaced,courier" size="-1"><b>acm.graphics</b></font> package are the <font face="monospaced,courier" size="-1"><b>GPen</b></font> and <font face="monospaced,courier" size="-1"><b>GTurtle</b></font> classes, which don?t really fit into the shape class framework. Their purpose is to provide students with a simple mechanism for drawing figures using a paradigm that is more aligned with the pen-on-paper model that the felt-board model used in the rest of the package. </p>
  <h4>The <font face="monospaced,courier" size="-1"><b><font size="+1">GPen</font></b></font> class</h4> 
  <p>The <font face="monospaced,courier" size="-1"><b>GPen</b></font> class models a pen that remembers its current location on the <font face="monospaced,courier" size="-1"><b>GCanvas</b></font> on which it is installed. The most important methods for <font face="monospaced,courier" size="-1"><b>GPen</b></font> are <font face="monospaced,courier" size="-1"><b>setLocation</b></font> (or <font face="monospaced,courier" size="-1"><b>move</b></font> to specify relative motion) and <font face="monospaced,courier" size="-1"><b>drawLine</b></font>. The former corresponds to picking up the pen and moving it to a new location; the latter represents motion with the pen against the canvas, thereby drawing a line. Each subsequent line begins where the last one ended, which makes it very easy to draw connected figures. The <font face="monospaced,courier" size="-1"><b>GPen</b></font> object also remembers the path it has drawn, making it possible to redraw the path when repaint requests occur. The most important methods for the <font face="monospaced,courier" size="-1"><b>GPen</b></font> class are shown in <a href="#FigGPenMethods">Figure 2-19</a>. </p>

  <p><font face="monospaced,courier" size="-1"><b>&nbsp;&nbsp;</b></font>Both the <font face="monospaced,courier" size="-1"><b>GPen</b></font> class and the <font face="monospaced,courier" size="-1"><b>GTurtle</b></font> class described in the following section are often used to create animated displays. To provide clients with some control over the speed of the animation, both classes include a <font face="monospaced,courier" size="-1"><b>setSpeed</b></font> method, which takes a number between 0.0 and 1.0 as its argument. Calling <font face="monospaced,courier" size="-1"><b>setSpeed(0.0)</b></font> means that the animation crawls along at a very slow pace; calling <font face="monospaced,courier" size="-1"><b>setSpeed(1.0)</b></font> makes it proceed as fast as the system allows. Intermediate values are interpreted so as to provide a smoothly varying speed of operation. Thus, if the speed value is associated with a scrollbar whose ends represent the values 0.0 and 1.0, adjusting the scrollbar will cause the animation to speed up or slow down in a way that seems reasonably natural to users. </p>
  <h4>The <font face="monospaced,courier" size="-1"><b><font size="+1">GTurtle</font></b></font> class</h4> 
  <p>The <font face="monospaced,courier" size="-1"><b>GTurtle</b></font> class is similar to <font face="monospaced,courier" size="-1"><b>GPen</b></font> but uses a ?turtle graphics? model derived from the Project Logo turtle described in Seymour Papert?s book <i>Mindstorms</i>. In the turtle graphics world, the conceptual model is that of a turtle moving on a large piece of paper. A <font face="monospaced,courier" size="-1"><b>GTurtle</b></font> object maintains its current location just as a <font face="monospaced,courier" size="-1"><b>GPen</b></font> does, but also maintains a current direction. </p>
  <p><font face="monospaced,courier" size="-1"><b>&nbsp;&nbsp;</b></font>The most common methods in the <font face="monospaced,courier" size="-1"><b>GTurtle</b></font> class are shown in <a href="#FigGTurtleMethods">Figure 2-21</a>. The path is created by a pen located at the center of the turtle. If the pen is down, calls to <font face="monospaced,courier" size="-1"><b>forward</b></font> generate a line; if the pen is up, such calls simply move the turtle without drawing a line. </p>

  <p><font face="monospaced,courier" size="-1"><b>&nbsp;&nbsp;</b></font>Although <font face="monospaced,courier" size="-1"><b>GTurtle</b></font> and <font face="monospaced,courier" size="-1"><b>GPen</b></font> have similar capabilities, they are likely to be used in different ways. The <font face="monospaced,courier" size="-1"><b>GTurtle</b></font> class is designed to be used at the very beginning of a course and must be both simple and evocative as intuitive model. The <font face="monospaced,courier" size="-1"><b>GTurtle</b></font> therefore has somewhat different defaults than its <font face="monospaced,courier" size="-1"><b>GPen</b></font> counterpart does. The image of the <font face="monospaced,courier" size="-1"><b>GTurtle</b></font>, for example, is initially visible, while the <font face="monospaced,courier" size="-1"><b>GPen</b></font> image is hidden. Moreover, the <font face="monospaced,courier" size="-1"><b>GTurtle</b></font> does not actually draw lines until the pen is lowered. The <font face="monospaced,courier" size="-1"><b>GPen</b></font> offers no option; the pen is always down. These defaults make the <font face="monospaced,courier" size="-1"><b>GTurtle</b></font> consistent with the Logo model, in which students learn to move the turtle first and then start drawing pictures with it. </p>
  <h2 clear="all"><a name="GObjectTrace"></a>2.7 The GObjectTrace demonstration program</h2> 
  <p>In order for students to get a good sense of how the <font face="monospaced,courier" size="-1"><b>acm.graphics</b></font> model works, they need to see it in action. Although writing graphical programs is essential to gaining an understanding, there is a sense in which the resulting programs have lost some immediacy. Students need to understand that whenever they call a method in a graphical object?or, in the language of object-oriented programming, send it a message?that object will respond by repainting itself on the canvas. In a running application, those repaint requests are typically coalesced into a single operation, which makes it difficult to see how an object responds to the individual messages. </p>
  <p><font face="monospaced,courier" size="-1"><b>&nbsp;&nbsp;</b></font>To make it easier for students to see precisely how graphical objects work and what effect each type of message has on that object, the <font face="monospaced,courier" size="-1"><b>demos</b></font> section of the Java Task Force web site contains a program called <font face="monospaced,courier" size="-1"><b>GObjectTrace</b></font>, which you can find at the following URL: </p>
  <p></p>
  <center> 
   <a href="http://www.acm.org/jtf/demos/classroom/GObjectTrace.html" <font face="monospaced,courier" size="-1"><b>http://www.acm.org/jtf/demos/classroom/GObjectTrace.html</b></a> 
  </center> 
  <p>This applet brings up a program window that has a <font face="monospaced,courier" size="-1"><b>GCanvas</b></font> in the upper portion and an interactive console along the bottom. This console is running a stripped-down version of a Java interpreter that includes definitions for the entire <font face="monospaced,courier" size="-1"><b>GObject</b></font> hierarchy. By typing Java statements into this window, you can create arbitrary objects, add them to the canvas, and then send those objects messages that are immediately reflected in the canvas display. </p>
