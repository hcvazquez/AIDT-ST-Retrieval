      <h1> <a href="#language-babel"></a>language-babel</h1> 
      <p><a href="https://travis-ci.org/gandm/language-babel"><img src="https://i.github-camo.com/e93c7aeefc2b306dfd585a8e4810089e3ab909c3/68747470733a2f2f7472617669732d63692e6f72672f67616e646d2f6c616e67756167652d626162656c2e7376673f6272616e63683d6d6173746572" alt="Build Status" data-canonical-src="https://travis-ci.org/gandm/language-babel.svg?branch=master"></a> <a href="https://ci.appveyor.com/project/gandm/language-babel/branch/master"><img src="https://i.github-camo.com/f44448dfa7cf513d500639df1e1e000ba0accbc7/68747470733a2f2f63692e6170707665796f722e636f6d2f6170692f70726f6a656374732f7374617475732f6566333271726a3869706c6e62716d322f6272616e63682f6d61737465723f7376673d74727565" alt="Build status" data-canonical-src="https://ci.appveyor.com/api/projects/status/ef32qrj8iplnbqm2/branch/master?svg=true"></a> <a href="https://david-dm.org/gandm/language-babel"><img src="https://i.github-camo.com/d982025a08dc13b69d3f5f6084b005dd42016a10/68747470733a2f2f64617669642d646d2e6f72672f67616e646d2f6c616e67756167652d626162656c2e737667" alt="Build Dependencies" data-canonical-src="https://david-dm.org/gandm/language-babel.svg"></a></p> 
      <p>Language grammar for all versions of JavaScript including ES2016 and ESNext, JSX syntax as used by <a href="http://facebook.github.io/react/index.html">Facebook React</a>, <a href="https://github.com/atom/etch">Atom's etch</a> and others, as well as optional typed JavaScript using <a href="http://flowtype.org/">Facebook flow</a>. This package also supports highlighting of <a href="https://github.com/gandm/language-babel#graphql-code-highlighting">GraphQL</a> language constructs when inside certain JavaScript template strings. For <code>.graphql</code> and <code>.gql</code> file support please see <a href="https://atom.io/packages/language-graphql">language-graphql</a> . The colour of syntax is determined by the theme in use.</p> 
      <p>The package also provides</p> 
      <ul> 
       <li> <a href="https://github.com/gandm/language-babel#automatic-indenting-of-jsx">automatic indentation</a> of JSX.</li> 
       <li> <a href="https://github.com/gandm/language-babel#auto-completion-of-jsx-tags-elements-and-attributes">JSX tag closure and JSX html element completion suggestions</a>.</li> 
       <li>Context aware <a href="https://github.com/gandm/language-babel#commenting-out-jsx-elements">commenting out of JSX elements</a>.</li> 
       <li>Additional <a href="https://github.com/gandm/language-babel#automatic-insertion-between-jsx-tags">newline insertion between JSX tag pairs</a>.</li> 
       <li>Babel <a href="https://github.com/gandm/language-babel#interface-to-babel-v6--v5">transpilation on file saves</a> (optional).</li> 
       <li>Babel transpile of all files in a directory or directories.</li> 
       <li>Babel transpiled <a href="https://github.com/gandm/language-babel#interface-to-babel-v6--v5">code preview</a>.</li> 
       <li>Support for <a href="https://github.com/styled-components/styled-components">styled-components</a> CSS inside tagged template strings including auto-completion.</li> 
       <li>Support for <a href="https://github.com/gandm/language-babel#graphql-code-highlighting">GraphQL code highlighting</a> </li> 
       <li> <a href="https://github.com/gandm/language-babel#javascript-tagged-template-literal-grammar-extensions">Support for adding other language grammars to highlight code inside template literal strings</a>.</li> 
      </ul> 
      <p>By default the language-babel package will detect file types <code>.js</code>,<code>.babel</code>,<code>.jsx</code>, <code>.es</code>, <code>.es6</code>, <code>.mjs</code> and <code>.flow</code>. Use the standard ATOM interface to enable it for other file types. This provides a grammar that scopes the file in order to colour the text in a meaningful way. If other JavaScript grammars are enabled these may take precedence over language-babel. Look at the bottom right status bar indicator to determine the language grammar of a file being edited. language-babel will be shown as either <code>Babel</code> or <code>Babel ES6 JavaScript</code>. Clicking the name will allow the grammar for a file to be changed.</p> 
      <p>language-babel provides <a href="http://babeljs.io/">Babel</a> V6 &amp; V5 transpiler support. If you only require grammar/syntax highlighting ensure that the package settings <code>Transpile On Save</code> and <code>Allow Local Override</code> are both off.</p> 
      <h2> <a href="#installation"></a>Installation</h2> 
      <p>Install via ATOM or by using <code>apm install language-babel</code>. If you only need to use the provided grammar read no further!</p> 
      <h2> <a href="#auto-completion-of-jsx-tags-elements-and-attributes"></a>Auto Completion of JSX tags, elements and attributes</h2> 
      <p>JSX tag closures are provided as auto-complete options. In addition, common HTML elements and their associated properties are displayed as auto-complete lists. Those supported by language-babel are described <a href="http://facebook.github.io/react/docs/tags-and-attributes.html">here</a></p> 
      <p><img src="https://i.github-camo.com/095381eda99433819d2c90d61a4e23f26147ec72/68747470733a2f2f636c6f75642e67697468756275736572636f6e74656e742e636f6d2f6173736574732f323331333233372f31323335323334382f32313833343862362d626237642d313165352d393234352d6264306431343637643731642e676966" alt="autoclose" data-canonical-src="https://cloud.githubusercontent.com/assets/2313237/12352348/218348b6-bb7d-11e5-9245-bd0d1467d71d.gif"></p> 
      <h2> <a href="#commenting-out-jsx-elements"></a>Commenting out JSX elements</h2> 
      <p>JSX elements cannot be commented out by using standard <code>//</code> or <code>/* */</code> commenting. Normally <code>{/* */}</code> is used instead. language-babel changes the Atom toggle comments behaviour when inside a JSX block to support this behaviour. Nested elements within JSX that require a // form of commenting will be detected automatically.</p> 
      <p><img src="https://i.github-camo.com/4e8823004c3edd2f000b117b7ba4f6e42b896f54/68747470733a2f2f636c6f75642e67697468756275736572636f6e74656e742e636f6d2f6173736574732f323331333233372f31323434313735322f34643637326265362d626634322d313165352d386532302d3333613936613831646236362e676966" alt="autoclose" data-canonical-src="https://cloud.githubusercontent.com/assets/2313237/12441752/4d672be6-bf42-11e5-8e20-33a96a81db66.gif"></p> 
      <h2> <a href="#automatic-insertion-between-jsx-tags"></a>Automatic insertion between JSX tags</h2> 
      <p>When a newline is inserted between a matched open/close pair of JSX tags, language-babel inserts an additional line and positions the cursor. If Auto Indent is also turned on then the cursor is correctly indented.</p> 
      <p><img src="https://i.github-camo.com/184202b38837e63282916fa9aa285fd00f0bb626/68747470733a2f2f636c6f75642e67697468756275736572636f6e74656e742e636f6d2f6173736574732f323331333233372f31353733363936312f31636261613738302d323839622d313165362d396364332d3438663734396139316639382e676966" alt="autoclose" data-canonical-src="https://cloud.githubusercontent.com/assets/2313237/15736961/1cbaa780-289b-11e6-9cd3-48f749a91f98.gif"></p> 
      <h2> <a href="#automatic-indenting-of-jsx"></a>Automatic Indenting of JSX</h2> 
      <p>The package setting <code>Auto Indent JSX</code> if enabled will auto-indent any JSX code typed or moved across using suitable Atom defaults. There are also three commands - <code>language-babel:toggle-auto-indent-jsx</code>, <code>language-babel:auto-indent-jsx-on</code> and <code>language-babel:auto-indent-jsx-off</code> that override the default behaviour. These can be mapped to keyboard shortcuts if needed.</p> 
      <p>Auto-Indenting will also attempt to read the <code>.eslintrc</code> file associated with an edited file's project for the presence of four properties whose defaults are shown below. These rules, which are part of the <a href="https://github.com/yannickcr/eslint-plugin-react">ESLint-plugin-React</a> EsLint plugin, are then used to determine the alignment and tab/spaces spacing of JSX elements. For more information on the options for these rules see <a href="https://github.com/yannickcr/eslint-plugin-react/blob/master/docs/rules/jsx-closing-bracket-location.md">Closing bracket</a>, <a href="https://github.com/yannickcr/eslint-plugin-react/blob/master/docs/rules/jsx-indent.md">Indent</a> and <a href="https://github.com/yannickcr/eslint-plugin-react/blob/master/docs/rules/jsx-indent.md">Indent Props</a>.</p> 
      <p>Please note that no attempt is currently made to read eslint settings in any other file. e.g. <code>package.json</code>, <code>eslint.js</code>, <code>extends...</code>, etc.</p>

      <h2> <a href="#interface-to-babel-v6--v5"></a>Interface to Babel V6 &amp; V5</h2> 
      <p>language-babel fully supports the Babel JavaScript transpiler versions 5 and 6.</p> 
      <p>Options in the language-babel package settings and/or in <code>.languagebabel</code> project based JSON files allow for Babel validations to be carried out on a file saves using <code>.babelrc</code> options. A file tree context menu - <code>Language-Babel</code> - is also provided that allows whole directories to be transpiled obeying any <code>.babelrc</code> and <code>.languagebabel</code> settings. Even if using a workflow such as gulp, webpack, etc, this can be very useful. Additional options allow the output from Babel (transpiled code and maps ) to be output to other directories.</p> 
      <p>It is also possible to preview any source file as Babel would output it.</p>

      <h4> <a href="#transpiling"></a>Transpiling</h4> 
      <p>This package works by using the concept of a project folder which we assume contains a project or even nested projects any of which may contain a Babel project. In a Babel project, we expect to see one or more <code>.babelrc</code> files, <code>node_modules</code> folders at the root's of the project containing an optional <code>babel-core</code> (either v5 or v6) and other babel plugins/presets as determined by the project's <code>package.json</code> file. In addition, we may expect to see one or more <code>.languagebabel</code> files in the project. Projects are either implicit (an Atom project folder) or explicit (denoted by a <code>.languagebabel</code> property of <code>"projectRoot": true</code>). If no <code>babel-core</code> is found in the project then a version will be provided by the package but this will be a Babel Version 6 instance. Plugins and presets will not be provided by the package.</p> 
      <p>A trivial example project that shows examples of using <code>.languagebabel</code> and <code>.babelrc</code> files may be found <a href="https://github.com/gandm/example-language-babel">here</a>.</p> 
      <p><img src="https://i.github-camo.com/2357da73a5418181b05086046291a4870e05f6c4/68747470733a2f2f636c6f75642e67697468756275736572636f6e74656e742e636f6d2f6173736574732f323331333233372f31313134353732302f31386266306635322d386130302d313165352d383266302d3366343734616565666362372e676966" alt="example" data-canonical-src="https://cloud.githubusercontent.com/assets/2313237/11145720/18bf0f52-8a00-11e5-82f0-3f474aeefcb7.gif"></p> 
      <p>Multiple projects may be open at any time inside Atom and <code>language-babel</code> must allow the use of differing <code>babel-core</code> versions and associated plugins when transpiling. It does this by using background tasks - one per Babel project. When a <code>language-babel</code> grammar enabled file is saved the package settings and optionally any <code>.languagebabel</code> configuration files are read to determine if the file should be transpiled and what to do with the output. These settings and <code>.languagebabel</code> options are described below.</p> 
      <p>It is very easy to ensure that language-babel does not transpile files that are not needed. Just turn off the global package setting <code>Transpile On Save</code> and turn on <code>Allow Local Override</code>. Apart from grammar highlighting only projects having a <code>.languagebabel</code> in the file path will then be affected by this package. Further granularity is easy too.</p> 
      <p>If <code>.babelrc</code> and/or <code>package.json</code> files contain Babel properties that are environment specific these environments should be created before Atom is started. In particular, Babel assumes a <code>development</code> environment by default, whereas Atom assumes a <code>production</code> environment by default.</p> 
      <p>e.g.</p> 
      <p>Windows</p> 
      <pre><code>set NODE_ENV="development"
atom
</code></pre> 
      <p>OSX/Unix</p> 
      <pre><code>NODE_ENV="development"
atom
</code></pre>

       <li> <h4> <a href="#javascript-tagged-template-literal-grammar-extensions"></a>JavaScript Tagged Template Literal Grammar Extensions</h4> <p>This package setting allows language-babel to include third party grammars to highlight code inside template literal strings. These may be tagged template literals, <a href="https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Template_literals">described here</a>, or where no appropriate function tag exists, another form of tag marking that signifies the templates nature. e.g. A comment string.</p> <p>For example you may wish to highlight templates prefixed with <code>/* @html */`&lt;div&gt;&lt;/div&gt;`</code> as HTML, and maybe <code>sql`select * from table foo`</code> as SQL. The latter assumes a tagged function named sql exists in the code.</p> <p>In order to do this, you need to find a language package that supports the highlighting of the template code. This language package should then be installed into Atom. You then need to find the scope name for that grammar. This can be done in a number of ways, but if you view the grammars JSON/CSON file and look for the <code>scopeName</code> property field, this indicates the scope name for that grammar.</p> <p>If we use the Atom provided languages <a href="https://atom.io/packages/language-css">language-css</a>, <a href="https://atom.io/packages/language-html">language-html</a> and <a href="https://atom.io/packages/language-sql">language-sql</a> in our example above to highlight the template code, then this field would look like.</p> <p><code>"(?:css\.(?:[a-z])+)":source.css, /* @html */:text.html.basic, sql:source.sql</code></p> <p>In other words, the package settings for this field is an array of strings, with each string in the form of <code>template-prefix:grammar-scopename#optional-include</code>.</p> <p>where:</p> 
        <ul> 
         <li> <code>template-prefix</code> is a literal string or an Oniguruma regular expression ( Oniguruma is the regular expression engine for TextMate grammars used by Atom) that comes before the opening back-tick of a template. A literal string may contain any characters except a comma but including colons. A regular expression is denoted by being enclosed by double quote marks <code>"RegExp Here"</code>. <strong>Most importantly, a regular expression should create no capturing groups.</strong> </li> 
         <li> <code>:</code>The last colon in the string signifies the start of the embedded grammars scopeName.</li> 
         <li> <code>grammar-scopename</code> is the scopeName of the grammar used to highlight the template.</li> 
         <li> <code>#optional-include</code> if present, will use that include block in the grammars repository.</li> 
        </ul> <p><strong>Please note:</strong> As language-babel passes off highlighting of the template to another grammar, it cannot highlight any interpolated code. Actually, that's not strictly correct, as it does highlight interpolated JavaScript code but only at the top most rule stack of the embedded grammar, but this is unlikely to be useful with most grammars. Also note that there is a delay of 10 seconds between the last character being entered into this field and a new extension grammar being generated.</p> </li> 
      </ul> 
      <h2> <a href="#languagebabel-configuration"></a>.languagebabel Configuration</h2> 
      <p><code>.languagebabel</code> JSON configuration files can exist in any directory of the path that contains a source file to be compiled. The properties in this file override the global package settings above. If <code>.languagebabel</code> files are present, they read and merged starting in the source files directory up to the project root directory. Properties defined closest the source file take precedence.</p> 
      <p>To use this option please enable the <code>Allow Local Override</code> package setting.</p> 
      <p>A <code>.languagebabel</code> file may contain one or more of the following properties.</p> 
      <pre><code>{
  "babelMapsPath":                    "relPath",
  "babelMapsAddUrl":                  true|false,
  "babelSourcePath":                  "relPath",
  "babelTranspilePath":               "relPath",
  "createMap":                        true|false,
  "createTargetDirectories":          true|false,
  "createTranspiledCode":             true|false,
  "disableWhenNoBabelrcFileInPath":   true|false,
  "keepFileExtension":                true|false,
  "projectRoot":                      true|false,  
  "suppressSourcePathMessages":       true|false,
  "suppressTranspileOnSaveMessages":  true|false,
  "transpileOnSave":                  true|false
}
</code></pre> 
      <h2> <a href="#graphql-code-highlighting"></a>GraphQL Code Highlighting</h2> 
      <p>language-babel supports highlighting of GraphQL code within JavaScript files. For highlighting <code>.graphql</code> and <code>.gql</code> files see its sister grammar - <a href="https://atom.io/packages/language-graphql">language-graphql</a>.</p> 
      <p>Inside JavaScript, GraphQL enclosed in back-ticks, a.k.a. Quasi or template strings are highlighted. Other GraphQL structures, notably types, are supported by the Flow component of this package.</p> 
      <p>Strings that have one of three prefixes/tags are parsed by the grammar to highlight the code enclosed.</p> 
      <pre><code> Relay.QL`This is how Relay uses template strings`

 graphql`This is how Relay will use template strings`

 gql`This is how Apollo for GraphQL uses template strings`

 /* GraphQL */`For cases where no template tag function is available`
</code></pre> 
      <p>An example of using the third method for highlighting code using <code>/* GraphQL */</code></p> 
      <pre><code> var { graphql, buildSchema } = require('graphql');

 var schema = buildSchema(/* GraphQL */`
   type Query {
     hello: String
   }
 `);
</code></pre>
