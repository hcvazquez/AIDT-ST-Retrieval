       <h1 class="entry-title">JavaScript Mergesort: Top-Down vs Bottom-Up</h1> 
       <h2 class="entry-meta"> by <a href="https://www.akawebdesign.com/author/arthurakay/" title="Posts by Arthur Kay" rel="author">Arthur Kay</a>&nbsp;•&nbsp;<a href="https://www.akawebdesign.com/2012/04/13/javascript-mergesort-top-down-vs-bottom-up/"><time class="published updated" datetime="2012-04-13">April 13, 2012</time></a>&nbsp;•&nbsp;<a href="https://www.akawebdesign.com/2012/04/13/javascript-mergesort-top-down-vs-bottom-up/#comments">1 Comment</a> </h2> 
      </header> 
      <div class="entry-content"> 
       <p>In my continuing series on JavaScript algorithms, I thought it might be fun to examine two methods for implementing “mergesort”.</p> 
       <p>Mergesort is essentially a “divide and conquer” technique, where the algorithm breaks an array into smaller pieces. Each of the small pieces is sorted and then recursively merged back together.</p> 
       <p>Mergesort is an attractive option for sorting large arrays because it is fast… and we’ll look at its efficiency in a moment. However, it also has a disadvantage: the algorithm requires more memory than Selection Sort and Insertion Sort, so for systems in which memory usage must be kept low mergesort may not be a good option.</p> 
       <h2>Memory Use</h2> 
       <p>Mergesort works by copying all values of an array into a temporary array. The original array is then recursively broken into smaller chunks to sort… but the temporary array is used throughout the process as a placeholder.</p> 
       <p>Thus, sorting an array of size N requires the system have available memory of 2N (the original array, plus a placeholder of the same size). If you have a considerable amount of data to be sorted, memory usage quickly balloons.</p> 
       <p>By contrast, Selection Sort and Insertion Sort use very small amounts of memory – each of these algorithms requires only a temporary placeholder for the array <em>index</em> currently being moved.</p> 
       <h2>Measuring Sort Efficiency</h2> 
       <p>In a <a href="https://www.akawebdesign.com/2011/12/07/javascript-binary-search-vs-linear-search/">previous post</a> on algorithms, I described the concept of Big-O notation as follows:</p> 
       <p>When we study the structure of algorithms, we represent the growth rate using <a href="http://en.wikipedia.org/wiki/Big_O_notation" target="_blank">Big-O Notation</a>. In short, Big-O Notation is a mathematical forumla that demonstrates how a function performs as its inputs (or memory requirements) change.</p> 
       <p>In practice, Big-O Notation represents the mathematical limit (i.e. upper-bound, or ceiling) for performance given some input. It is the worst-case scenario for the algorithm.</p> 
       <p>—</p> 
       <p>Mergesort guarantees an array will be sorted in time proportional to <em>N * log(N)</em> – which is far better than the worst-case scenarios for Selection and Insertion sorts.</p>

       <p><em>*These results are not actual times… just the maximum number of iterations in the algorithm</em></p> 
       <p>In reality, the efficiency of mergesort is closer to <em>O( 6N * log(N) )</em> (due to the number of array accesses for copy/move/compare actions) – so on smaller datasets, Selection and Insertion sort would actually run faster! But as the datasets grow in size, Mergesort quickly becomes a faster solution.</p> 
       <p>In theory, you could write a mergesort algorithm that utilized one of the other sorting methods only on the smallest chunks… thereby using the most efficient parts of each algorithm. I won’t get into that in this post, but it would be a cool exercise.</p> 
       <h2>Implementations of Mergesort</h2> 
       <p>There are two methods for implementing a mergesort algorithm: a top-down approach or a bottom-up approach.</p> 
       <h3>Top-Down Method</h3> 
       <p>The top-down approach to mergesort is perhaps the most obvious. Given an array of size N, the algorithm recursively breaks the array in half and then merges the results together.</p> 
       <pre><code>
function topDownMergeSort(inputArray) {
    var sort = function (array, low, high) {
        if (high &lt;= low) { return; }

        var mid = Math.floor(low + (high - low) / 2);

        //recursively sort the array by breaking it in half
        sort(array, low, mid);
        sort(array, mid + 1, high);

        //merge() is defined elsewhere
        ArraySort.merge(array, temp, low, mid, high);
    };

    var temp = []; //allocate space just once

    sort(inputArray, 0, inputArray.length - 1);

    return inputArray;
}
</code></pre> 
       <p>The concept is pretty straight-forward - and is easily represented as a tree.</p> 
       <p><img src="https://www.akawebdesign.com/wp-content/uploads/2012/04/tree-300x223.jpg" alt="" title="tree" width="300" height="223"></p> 
       <p>Using the tree as a visual aid, the root represents the entire array. It is split recursively into 2 halves; when the halves can no longer be split, the pieces are individually sorted. Then the sub-parts are merged back together, moving upwards through the tree until the algorithm is back at the root (at which point the array is completely sorted).</p> 
       <h3>Bottom-Up Method</h3> 
       <p>The bottom-up approach is perhaps a little less clear at first.</p> 
       <p>Rather than breaking the overall array into distinct pieces, bottum-up mergesort loops over the array using intervals of varying sizes. Each interval is sorted and merged together; subsequent loops over the array have larger intervals, effectively merging our previously sorted (smaller) intervals together.</p> 
       <pre><code>
function bottomUpMergeSort(inputArray) {
    var length = inputArray.length,
        size = 1,
        temp = []; //allocate space just once

    for (size; size &lt; length; size = size*2) {
        var low = 0;

        for(low; low &lt; length-size; low += size*2) {
            var mid = low + size - 1,
                high = Math.min(low + (size*2 - 1), length -1);

            ArraySort.merge(inputArray, temp, low, mid, high);
        }
    }

    return inputArray;
}
</code></pre> 
       <p>I personally feel that visualizing bottom-up mergesort is difficult without the help of an example like this:</p> 
       <p><img src="https://www.akawebdesign.com/wp-content/uploads/2012/04/bottomup.gif" alt="" title="bottomup" width="185" height="179"></p> 
       <h3>Results</h3> 
       <p><a href="https://www.akawebdesign.com/ExtJS/algorithms/mergesorts.html">I graphed the expected/actual results</a> for these two algorithms using ExtJS. The graphs display the number of items in the array (X axis) and the number of iterations needed to sort the array (Y axis).</p> 
       <p>Remember: both top-down and bottom-up mergesorts have the same estimated (worst-case) runtime. However, my analysis seems to indicate that bottom-up mergesort runs faster in practice! I am admittedly a bit unsure as to <em>why</em> that is the case... so perhaps some discussion on the topic is in order.</p> 
       <p><strong>UPDATE:</strong> I asked the <a href="http://stackoverflow.com/questions/10153393/mergesort-is-bottom-up-faster-than-top-down" target="_blank">StackOverflow</a> community to look at my analysis, and they seem to (generally) agree with me. It was correctly pointed out that I'm measuring "iterations", and not "speed" or "array accesses" - so my results are in fact valid. While my analysis isn't exactly "complete" in that sense, some people agreed that the bottom-up method should be somewhat faster as it doesn't have the recursive method calls (which lead to computing overhead).</p>
