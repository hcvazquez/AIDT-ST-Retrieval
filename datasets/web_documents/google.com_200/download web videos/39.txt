       <p><a class="headerAnchor" name="Introduction"></a></p> 
       <h1>Introduction</h1> 
       <p>Vidyo.io™ is a platform as a service (PaaS) that enables developers to provide high-quality, real-time video communication capabilities. The service includes the VidyoClient SDK, which provides the APIs for integrating such communication capabilities into a variety of applications and workflows.</p> 
       <p>Some of the features of <a href="http://Vidyo.io">Vidyo.io</a> include:</p> 
       <ul> 
        <li>Multi-party audio and video conferencing</li> 
        <li>Group chat</li> 
        <li>Application sharing</li> 
       </ul> 
       <p>The VidyoClient SDK offers the same APIs on all supported platforms, providing a fast learning curve and enabling rapid development on all device types.</p> 
       <p>Connecting to the <a href="http://Vidyo.io">Vidyo.io</a> cloud is done by passing a token from your application through the VidyoClient SDK to the <a href="http://Vidyio.io">Vidyio.io</a> cloud. The token identifies both your application and the user.</p> 
       <p>To learn more about how to get the most out of <a href="http://Vidyo.io">Vidyo.io</a>, we invite you to review the <a href="#KeyConcepts" title="Key Concepts">Key Concepts</a> of our platform.</p>

       <p><a class="headerAnchor" name="GettingStarted1"></a></p> 
       <h2 class="docs-header">Getting Started</h2> 
       <p>To start enjoying <a href="http://Vidyo.io">Vidyo.io</a>, your main objective is to write your client application and integrate it with your business flow.</p> 
       <p>Here’s the easiest way to understand <a href="http://Vidyo.io">Vidyo.io</a> and get started with using it:</p> 
       <ol> 
        <li>Launch the <a href="#GettingStarted2" title="Getting Started">demo app</a>, and tinker with it.</li> 
        <li>Pick your platform of choice and follow the Quick Start Guide in the <a href="#SupportedPlatforms" title="Supported Platforms">Supported Platforms</a> section above.</li> 
       </ol> 
       <p>To dig deeper into <a href="http://Vidyo.io">Vidyo.io</a>, we suggest you check out its <a href="#KeyConcepts" title="Key Concepts">Key Concepts</a>.</p>

       <h2 class="docs-header">Need Help?</h2>

       <h1 class="docs-header">Key Concepts</h1> 
       <p><a href="http://Vidyo.io">Vidyo.io</a> comprises a high-quality video cloud service coupled with a VidyoClient SDK. The VidyoClient SDK offers the same APIs on all supported platforms, offering a fast learning curve and enabling rapid development on all device types.</p> 
       <p>When developing your application, you integrate the VidyoClient SDK with your application on any of the platforms necessary. Connecting to the <a href="http://Vidyo.io">Vidyo.io</a> cloud is done by passing a token from your application through the VidyoClient SDK to the <a href="http://Vidyio.io">Vidyio.io</a> cloud, identifying both your application and the user.</p> 
       <p><img src="/images/documentation/4-1-16-8/overviewStructure.jpg" alt=""></p> 
       <p>The illustration above details the components you will be interacting with when using <a href="http://Vidyo.io">Vidyo.io</a>:</p> 
       <ol> 
        <li><strong style="word-wrap: break-word;"><a href="http://Vidyo.io">Vidyo.io</a> Cloud Service</strong> - This is the backend service that the VidyoClient SDK and ultimately your own application interacts with. It enables support for multiparty conferences, smart routing of video, resource allocation and management, and many other functions.</li> 
        <li><strong style="word-wrap: break-word;">VideoClient SDK</strong> - This is the client-side SDK that your application calls when it needs to add video support. The VidyoClient SDK shares the same APIs across the multiple <a href="#SupportedPlatforms" title="Supported Platforms">platforms</a>.</li> 
        <li><strong style="word-wrap: break-word;">Your Application</strong> - This is the application that you are developing. It has a lot to do with your own service and a bit to do with video communications, which is where it interacts with the VidyoClient SDK. To make it simpler, we’ve created a reference application that you can use as a starting point for your own app.</li> 
       </ol> 
       <p>The rest of this document will cover some important concepts that you need to understand in order to make the best use of the <a href="http://Vidyo.io">Vidyo.io</a> platform.</p> 
       <hr> 
       <p><a class="headerAnchor" name="VidyoClient"></a></p> 
       <h2 class="docs-header">The VidyoClient</h2> 
       <p>There are two important concepts in <a href="http://Vidyo.io">Vidyo.io</a> that you need to know about:</p> 
       <ol> 
        <li>Tokens</li> 
        <li>Resources</li> 
       </ol> 
       <p>To make things simple, <strong style="word-wrap: break-word;">tokens</strong> identify a device to the <a href="http://Vidyo.io">Vidyo.io</a> service and <strong style="word-wrap: break-word;">resources</strong> are meeting points for your users. You need to generate these access tokens when you connect a new device via the VidyoClient SDK to the <a href="http://Vidyo.io">Vidyo.io</a> cloud. And you create a resource simply by calling it by name - resources don’t need to be provisioned ahead of time and can be created ad hoc as you go along.</p> 
       <p>Check out more about <a href="#Tokens" title="Tokens">tokens</a> and <a href="#Resources" title="Resources">resources</a>.</p>

       <h2 class="docs-header">Tokens</h2> 
       <p>To connect to <a href="http://Vidyo.io">Vidyo.io</a>, the VidyoClient SDK needs to pass a token. A token is a short-lived authentication credential that grants access to the <a href="http://Vidyo.io">Vidyo.io</a> service on behalf of the developer to a specific user. When an endpoint requests access to the service, your application backend should generate a token and pass it on to the client application.</p> 
       <blockquote> 
        <p><strong style="word-wrap: break-word;">Important</strong>: Never generate a token in the client application itself - this exposes your <strong style="word-wrap: break-word;">DeveloperKey</strong> which can then be used by other developers.</p> 
       </blockquote> 
       <h3 class="docs-header">How to Generate a Token</h3> 
       <p>For each application, you must create a <strong style="word-wrap: break-word;">DeveloperKey</strong> and <strong style="word-wrap: break-word;">ApplicationID</strong> in the <a href="/dashboard#tab_apikey">API Key</a> section of the <a href="http://Vidyo.io">Vidyo.io</a> site and securely store it in your application backend. Then, each time you need to grant access to the <a href="http://Vidyo.io">Vidyo.io</a> service, you must generate a <strong style="word-wrap: break-word;">Token</strong> from the <strong style="word-wrap: break-word;">DeveloperKey</strong> and <strong style="word-wrap: break-word;">ApplicationID</strong> as described below and pass it to their endpoint application or the <strong style="word-wrap: break-word;">VidyoConnector</strong> application.</p> 
       <p><img src="/images/documentation/4-1-16-8/overviewToken.jpg" alt=""></p> 
       <p>Tokens can be rather long (typically 200 characters or more), so plan accordingly.</p> 
       <blockquote> 
        <p>DeveloperKey and ApplicationID are “static” in your application. You will also need to provide a user name and an expiration time whenever you need to generate a token for a user.</p> 
       </blockquote> 
       <h4>Developer Key</h4> 
       <p>A DeveloperKey is a shared secret between the <a href="http://Vidyo.io">Vidyo.io</a> service and the developer’s backend application. It is used to generate access tokens on the customer backend and should never be sent to the endpoint itself. Access to the DeveloperKey should be restricted since every token that is signed by it will be used for access and billing.</p> 
       <h4>Application ID</h4> 
       <p>The ApplicationID identifies your application when connecting to the <a href="http://Vidyo.io">Vidyo.io</a> platform. You can automatically provision any user (for example, user1@ApplicationID) when generating a token.</p> 
       <h4>User Name</h4> 
       <p>A token is associated to a specific user in <a href="http://Vidyo.io">Vidyo.io</a>. To that end, when a token is generated, a user name needs to be provided. The user name is an open-ended alphanumeric string of your choosing which will be provisioned on the fly.</p> 
       <blockquote> 
        <p><a href="http://Vidyo.io">Vidyo.io</a> will refer to users in Participant and Chat related APIs as user@applicationId.</p> 
       </blockquote> 
       <h4>Expiration</h4> 
       <p>Each token has a lifetime. Its lifetime is determined when the token is generated by specifying the number of seconds until it expires.</p> 
       <p>Expiration ensures that access to the system is secure and isn’t open to abuse.</p>

       <p><a class="headerAnchor" name="Resources"></a></p> 
       <h2 class="docs-header">Resources</h2> 
       <p>Resources are strings of text denoting meeting points in your application. They are in an ad-hoc fashion simply by directing a user to a resource. If the resource doesn’t exist, then it is created. If the resource already exists, the user will join the other participants who are already “connected” to the resource.</p> 
       <h3 class="docs-header">Naming Resources</h3> 
       <p>The following examples show how you can use the resourceId. Think about the requirements of your application, and then decide how to name the resources needed to connect your video sessions.</p> 
       <ul> 
        <li>You can think of resources as meeting rooms. You can connect participants to a meeting room with a resourceId of “Blue” or “Red”, for example.</li> 
        <li>You can decide that resources are private meeting rooms of users, and name your resourceIds accordingly, such as “John room” or <a href="mailto:%22john@example.com">"john@example.com</a>".</li> 
        <li>You can create random rooms by designating random resourceIds, such as “3e24d54ff8”.</li> 
        <li>You can even generate 1:1 calls with them, such as “Alice-Bob” if Alice calls Bob.</li> 
       </ul> 
       <p>A resource operates in <a href="http://Vidyo.io">Vidyo.io</a> as a collection of users joining in a video session together. These users are referred to as participants. You can get notified on participant updates or send and receive messages with participants connected to the same resourceId.</p>

       <p><a class="headerAnchor" name="DeviceManagement"></a></p> 
       <h2 class="docs-header">Device Management</h2> 
       <p>The VidyoConnector will pick the default camera, microphone, and speaker. However, you can override this using one of two ways of manipulating devices: cycling and selecting.</p> 
       <h3 class="docs-header">Cycling</h3> 
       <p>With cycling, the VidyoClient SDK creates a button that the user can press to cycle the camera/microphone/speaker until the right one is found. This is similar to how you switch between front and back cameras in your smartphone’s built-in camera app. This provides an easy-to-use UI where the user can attempt the various combinations until reaching the one desired.</p> 
       <ul class="nav nav-tabs">

       <h2 class="docs-header">Screen Sharing</h2> 
       <p>Windows and monitors can be shared in a session taking place on a resourceId. Passing <em>null</em> ("{}" in JavaScript) into <strong style="word-wrap: break-word;">SelectWindow / SelectMonitor</strong> will stop the sharing.</p> 
       <blockquote> 
        <p>The <code style=" word-wrap: break-word;">RegisterLocalWindowShareEventListener</code> should only be called when the application is ready to share and it should be unregistered after the share was selected.</p> 
       </blockquote> 
       <p>In the case of WebRTC, an extension is required to enable window sharing. It can be downloaded from the link provided in <code style=" word-wrap: break-word;">status.downloadPathWebRTCExtension;</code> inside the <code style=" word-wrap: break-word;">VidyoClient.js?onload=</code> event. See <a href="#Initializing" title="Initializing">Initializing</a> section. The browser will pop up it’s own share selection dialog once <code style=" word-wrap: break-word;">RegisterLocalWindowShareEventListener</code> is called and the result of the user selection will be the available in the <code style=" word-wrap: break-word;">onAdded</code> event.</p> 
       <blockquote> 
        <p>Screen sharing does not currently work in the mobile SDKs (iOS and Android).</p> 
       </blockquote>

       <p><a class="headerAnchor" name="Layout"></a></p> 
       <h2 class="docs-header">Video Layout</h2> 
       <p>While VidyoConnector provides a default layout, you can also customize it if you want. The default compositing renderer is used when you call <em>ConnectorConstruct</em> with the viewId of your choice. If you’d rather create your own layout, you can instead pass <em>Null</em> to <em>ConnectorConstruct</em> as the viewId and show/hide sources into particular views using the following APIs:</p> 
       <ul> 
        <li>AssignViewToLocalCamera, AssignViewToRemoteCamera, AssignViewToRemoteWindowShare: assigns a view to a local camera, remote camera, or remote window share</li> 
        <li>ShowViewAt: display the view in the UI that is assigned in AssignViewTo*<br> - It is not necessary to call ShowViewAt when using JavaScript API</li> 
        <li>HideView: hide a local camera, remote camera, or remote window share from a view</li> 
       </ul> 
       <p>Other considerations when using a custom layout:</p> 
       <ul> 
        <li>AssignViewTo*, ShowViewAt, and HideView should be called on the main/UI thread of application.</li> 
        <li>When replacing one camera or window share with another, it is not necessary to call HideView on the view being replaced. Only call AssignViewTo* followed by ShowViewAt.</li> 
       </ul> 
       <blockquote> 
        <p>Compositing and custom layouts are mutually exclusive.</p> 
       </blockquote>

       <p>While using the custom layout above, you can register to access raw frames from both local and remote sources.</p> 
       <p>Register a listener for raw <code style=" word-wrap: break-word;">VdyoVideoFrame</code> or <code style=" word-wrap: break-word;">VidyoAudioFrame</code> events using the following API:</p> 
       <ul> 
        <li><code style=" word-wrap: break-word;">RegisterLocalCameraFrameListener()</code>, <code style=" word-wrap: break-word;">RegisterLocalWindowShareFrameListener()</code>, <code style=" word-wrap: break-word;">RegisterLocalMonitorFrameListener()</code>, <code style=" word-wrap: break-word;">RegisterLocalMicrophoneFrameListener()</code>, <code style=" word-wrap: break-word;">RegisterRemoteCameraFrameListener()</code>, <code style=" word-wrap: break-word;">RegisterRemoteWindowShareFrameListener()</code>, <code style=" word-wrap: break-word;">RegisterRemoteMicrophoneFrameListener()</code></li>

void OnLocalCameraFrame(VidyoConnector* c, VidyoLocalCamera* localCamera, const VidyoVideoFrame *videoFrame)
{
	std::stringstream filePathSS;
	std::string filePath;
	std::ofstream *fileStream;

	/* Create file path. Each resolution will have it's own file since the raw YCbCr does not have header describing the width/heigh and the stream size might change due to bandwidth/CPU */
	filePathSS &lt;&lt; "C:\\tmp\\" &lt;&lt; LmiStringCStr(VidyoLocalCameraGetName(localCamera)) &lt;&lt; "-" &lt;&lt; VidyoVideoFrameGetWidth(videoFrame) &lt;&lt; "x" &lt;&lt; VidyoVideoFrameGetHeight(videoFrame) &lt;&lt; ".yuv";
	filePath = filePathSS.str();

	if (fileMap.find(filePath) == fileMap.end()) {
		/* Open new file or append to existing file */
		fileMap[filePath] = new std::ofstream(filePath, std::ofstream::binary | std::ofstream::out | std::ofstream::app);
	}

	fileStream = fileMap[filePath];
	
	/* Decide if the copy should contain the padding or if it needs to be stripped out */
	if (copyWidthPadding) {
		/* Write data pointer and size of video frame payload to file */	
		fileStream-&gt;write(videoFrame-&gt;data, length:videoFrame-&gt;size);
	} else {
		/* copy the data stripping out the padding and offset */
		/* Data pointer to video frame payload */
		char* data = (char *)videoFrame-&gt;data;
		/* Data pointer to Chroma plane of the YCbCr frame */
		char* dataY = data + videoFrame-&gt;offsetY;
		/* width in bytes of the plane*/
		LmiSizeT widthY = videoFrame-&gt;widthY;
		/* Iterate over the entire plane and save the data to a file without padding */
		for (int i = 0; i &lt; videoFrame-&gt;heightY;i++) {
			/* offset by pitch to move to the next line */
			fileStream-&gt;write(dataY, widthY);
			dataY += videoFrame-&gt;pitchY;
		}
		/* Data pointer to Chroma plane of the YCbCr frame */
		char* dataCb = data + videoFrame-&gt;sizeY + videoFrame-&gt;offsetCb;
		/* width in bytes of the plane*/
		LmiSizeT widthCb = videoFrame-&gt;widthCb;
		/* Iterate over the entire plane and save the data to a file without padding */
		for (int i = 0; i &lt; videoFrame-&gt;heightCb; i++) {
			/* offset by pitch to move to the next line */
			fileStream-&gt;write(dataCb, widthCb);
			dataCb += videoFrame-&gt;pitchCb;
		}
		/* Data pointer to Chroma plane of the YCbCr frame */
		char* dataCr = data + videoFrame-&gt;sizeY + videoFrame-&gt;sizeCb + videoFrame-&gt;offsetCr;
		/* width in bytes of the plane*/
		LmiSizeT widthCr = videoFrame-&gt;widthCr;
		/* Iterate over the entire plane and save the data to a file without padding */
		for (int i = 0; i &lt; videoFrame-&gt;heightCr; i++) {
			/* offset by pitch to move to the next line */
			fileStream-&gt;write(dataCr, widthCr);
			dataCr += videoFrame-&gt;pitchCr;
		}
	}
}

       <p><a class="headerAnchor" name="MediaBridge"></a></p> 
       <h1 class="docs-header">Media Bridge</h1> 
       <p>As you develop applications that leverage <a href="http://Vidyo.io">Vidyo.io</a>, you may want to include a feature for recording and distributing recordings for private or public viewing. You may also want to include SIP-enabled participants in your calls. By integrating the Media Bridge Docker image with your Vidyo application, you can record and distribute your Vidyo conferences and include incorporate SIP-enabled video conferencing systems.</p> 
       <p><a class="headerAnchor" name="WhatsDocker"></a></p> 
       <h2 class="docs-header">What’s Docker?</h2> 
       <p>Docker is a robust software container platform you install on a dedicated server and use to run the Media Bridge image. The Media Bridge image provides record/playback functionality you might want to integrate with your Vidyo.io-based application. Docker provides the thorough resources and documentation you need to integrate basic or extensive Media Bridge features. See the Docker Overview <a href="https://www.docker.com/what-docker">here</a>.</p> 
       <hr> 
       <p><a class="headerAnchor" name="WhatsRecorder"></a></p> 
       <h2 class="docs-header">What’s Media Bridge?</h2> 
       <p>Media Bridge is a downloadable Docker image that provides the following:</p> 
       <ul> 
        <li>Recorder for the archiving of Vidyo.io-based video conferences</li> 
        <li>Streamer for the distribution of Vidyo.io-based video conferences</li> 
        <li>Gateway for SIP-enabled systems</li> 
       </ul> 
       <p>From Docker, you pull the Media Bridge image, create a config file, create a container, drop in your config file, then start recording.</p> 
       <hr> 
       <p><a class="headerAnchor" name="BeforeYouBegin"></a></p> 
       <h2 class="docs-header">Before You Begin</h2> 
       <p>Before you begin, ensure that your Linux server is <a href="https://docs.docker.com/engine/installation/linux/ubuntu/">configured with a running Docker Engine</a>.</p> 
       <h3 class="docs-header">What you will need</h3> 
       <ul> 
        <li>A Linux server running [Ubuntu 14.04][Ubuntu1404] or newer.</li> 
        <li>The <a href="https://docs.docker.com/engine/installation/linux/ubuntu/#install-from-a-package">Docker Engine</a> package installed and running on your Ubuntu Linux server.</li> 
        <li>The complete <a href="https://docs.docker.com/">Docker Documentation set</a>.</li> 
        <li>Access tokens to <a href="http://vidyo.io">vidyo.io</a> (see <a href="https://support.vidyo.io/hc/en-us/articles/115003127747-How-to-generate-a-token">How to generate a Token</a>).</li> 
       </ul> 
       <blockquote> 
        <p>Server platforms other than Ubuntu Linux 14.04+ have not been tested; they might not run smoothly or predictably in your deployment.</p> 
       </blockquote> 
       <hr> 
       <p><a class="headerAnchor" name="RecorderSetup"></a></p> 
       <h2 class="docs-header">Set-up</h2> 
       <p>To set up your server with Media Bridge for managing video conference recordings, follow these steps:</p> 
       <ol> 
        <li> <p>Install the <code style=" word-wrap: break-word;">Media Bridge</code> docker image:</p> <pre><code style=" word-wrap: break-word;">docker pull vidyo/mediabridge
</code></pre> </li> 
        <li> <p>Create a recording configuration file (see <a href="#RecordingConfigFile">Recording Configuration File</a>).</p> </li> 
        <li> <p>Create a container from the <code style=" word-wrap: break-word;">Media Bridge</code> image:</p> <pre><code style=" word-wrap: break-word;">docker run -d --name my_mediabridge vidyo/mediabridge
</code></pre> </li> 
        <li> <p>Copy your config file to the container:</p> <pre><code style=" word-wrap: break-word;">docker cp config my_mediabridge:/opt/vidyo
</code></pre> </li> 
       </ol> 
       <hr> 
       <p><a class="headerAnchor" name="StartRecording"></a></p> 
       <h2 class="docs-header">Start a Recording</h2> 
       <p>To start the container, enter:</p> 
       <pre><code style=" word-wrap: break-word;">docker exec -d my_mediabridge /opt/vidyo/connect
</code></pre> 
       <hr> 
       <p><a class="headerAnchor" name="MonitorRecording"></a></p> 
       <h2 class="docs-header">Monitor a Recording</h2> 
       <p>To monitor the recording, enter:</p> 
       <pre><code style=" word-wrap: break-word;">docker exec my_mediabridge pgrep -x call
</code></pre> 
       <blockquote> 
        <p>If a recording is active, the command returns the process ID; otherwise, it returns nothing.</p> 
       </blockquote> 
       <hr> 
       <p><a class="headerAnchor" name="StopRecording"></a></p> 
       <h2 class="docs-header">Stop a Recording</h2> 
       <p>The recording stops automatically when the last participant leaves the room.</p> 
       <p>To force the recording to stop with a <code style=" word-wrap: break-word;">SIGINT</code> call, enter:</p> 
       <pre><code style=" word-wrap: break-word;">docker exec my_mediabridge pkill -x --signal INT call
</code></pre> 
       <hr> 
       <p><a class="headerAnchor" name="RetrieveRecording"></a></p> 
       <h2 class="docs-header">Retrieve a Recording</h2> 
       <p>By default, the recording is saved to a file named <code style=" word-wrap: break-word;">recording.flv</code> at this path:</p> 
       <pre><code style=" word-wrap: break-word;">/opt/vidyo folder
</code></pre> 
       <p>To retrieve the recording, enter:</p> 
       <pre><code style=" word-wrap: break-word;">docker cp my_mediabridge:/opt/vidyo/recording.flv $HOME
</code></pre> 
       <p>To override this, use the <a href="#destination"><strong style="word-wrap: break-word;">destination</strong></a> option described in the <strong style="word-wrap: break-word;">Recording Configuration File Options</strong> table below.</p> 
       <hr> 
       <p>&lt;a class=“headerAnchor” name=“SIPGatewayCalls”"&gt;</p> 
       <h2 class="docs-header">Include SIP-enabled systems</h2> 
       <p>To make a SIP outbound call, set the destination key in the config file to the full SIP URI:</p> 
       <pre><code style=" word-wrap: break-word;">destination=sip:mary@5.6.7.8
</code></pre> 
       <p>To receive an incoming SIP call, leave the destination key empty:</p> 
       <pre><code style=" word-wrap: break-word;">destination=     
</code></pre> 
       <p>To start the Docker container for SIP calls, use the <code style=" word-wrap: break-word;">-p</code> parameter to map host ports to the container for both media and signaling. For example, if the <code style=" word-wrap: break-word;">mediaPorts</code> key in the config file is set to <code style=" word-wrap: break-word;">50000-50100</code>, the Docker command is:</p> 
       <pre><code style=" word-wrap: break-word;">docker run -d --name sipCall -p 50000-50100:50000-50100/udp -p 5060:5060/udp vidyo/mediabridge
</code></pre> 
       <blockquote> 
        <p>You must map SIP signaling UDP port <code style=" word-wrap: break-word;">5060</code> from the host to the container.</p> 
       </blockquote> 
       <hr> 
       <p><a class="headerAnchor" name="RecorderTroubleshooting"></a></p> 
       <h2 class="docs-header">Troubleshooting</h2> 
       <p>If you encounter a problem, retrieve the log file <code style=" word-wrap: break-word;">gw.log</code> from the container at this path and use it to troubleshoot:</p> 
       <pre><code style=" word-wrap: break-word;">/var/log
</code></pre> 
       <blockquote> 
        <p>To enable full logging, add <code style=" word-wrap: break-word;">log=full|debug|info,|warn|error|none</code>. The default is <code style=" word-wrap: break-word;">debug</code>.</p> 
       </blockquote> 
       <hr>

       <p><a class="headerAnchor" name="RecordingConfigFile"></a></p> 
       <h2 class="docs-header">Recording Configuration File</h2> 
       <p>The recording configuration file is a plain text file with key-value pairs. Name this file <code style=" word-wrap: break-word;">config</code> and place it at this path:</p> 
       <pre><code style=" word-wrap: break-word;">/opt/vidyo
</code></pre> 
       <p>Each key-value pair must be on a separate line.</p> 
       <p>The key is separated from the value with an equal (=) sign only; do not use spaces in key-value pairs.</p> 
       <p>Use a # before comments; text appearing on a line after a # is ignored at runtime.</p> 
       <p>The following table describes the options available for the recording configuration file.</p> 
       <div class="table-responsive">
        <table class="table table-striped table-bordered table-condensed"> 
         <thead> 
          <tr> 
           <th>Option</th> 
           <th>Description</th> 
          </tr> 
         </thead> 
         <tbody> 
          <tr> 
           <td><strong style="word-wrap: break-word;">width</strong></td> 
           <td>The width of the video stream.</td> 
          </tr> 
          <tr> 
           <td><strong style="word-wrap: break-word;">height</strong></td> 
           <td>The height of the video stream.</td> 
          </tr> 
          <tr> 
           <td><strong style="word-wrap: break-word;">fps</strong></td> 
           <td>The frame rate of the video stream in frames per second.</td> 
          </tr> 
          <tr> 
           <td><strong style="word-wrap: break-word;">kbps</strong></td> 
           <td>The bandwidth of the recording in kilo-bits per second.</td> 
          </tr> 
          <tr> 
           <td><strong style="word-wrap: break-word;">layout</strong></td> 
           <td>Set to <code style=" word-wrap: break-word;">1</code> for a basic layout, set to <code style=" word-wrap: break-word;">2</code> for a presenter layout, where the current speaker occupies a larger tile than the rest, set to <code style=" word-wrap: break-word;">3</code> for a film strip layout, where the current speaker occupies most of the screen and other participants appear in a film strip at the bottom.</td> 
          </tr> 
          <tr> 
           <td><strong style="word-wrap: break-word;">maxParticipants</strong></td> 
           <td>The maximum number of participants to be rendered in the video stream. Minimum is <code style=" word-wrap: break-word;">1</code>, maximum is <code style=" word-wrap: break-word;">8</code>.</td> 
          </tr> 
          <tr> 
           <td><strong style="word-wrap: break-word;">overlay</strong></td> 
           <td>When set to <code style=" word-wrap: break-word;">1</code>, Media Bridge displays each participant’s name as an overlay in the video stream.</td> 
          </tr> 
          <tr> 
           <td><strong style="word-wrap: break-word;">presentationAllowed</strong></td> 
           <td>When set to <code style=" word-wrap: break-word;">0</code>, Media Bridge ignores presentations during the conference. When set to <code style=" word-wrap: break-word;">1</code>, Media Bridge records presentations instead of the main video streams.</td> 
          </tr> 
          <tr> 
           <td><strong style="word-wrap: break-word;">presWidth</strong></td> 
           <td>The maximum width of the presentation stream.</td> 
          </tr> 
          <tr> 
           <td><strong style="word-wrap: break-word;">presHeight</strong></td> 
           <td>The maximum height of the presentation stream.</td> 
          </tr> 
          <tr> 
           <td><strong style="word-wrap: break-word;">presFps</strong></td> 
           <td>The frame rate of the presentation stream in frames per second.</td> 
          </tr> 
          <tr> 
           <td><strong style="word-wrap: break-word;">presKbps</strong></td> 
           <td>The bandwidth used for the presentation stream in kilo-bits per second.</td> 
          </tr> 
          <tr> 
           <td><a class="headerAnchor" name="destination"></a><strong style="word-wrap: break-word;">destination</strong></td> 
           <td>This can be a file name, a folder name (preceded by <code style=" word-wrap: break-word;">flv:</code>) or an RTMP URI (preceded by <code style=" word-wrap: break-word;">rtmp:</code>). When you set this field to a folder name, Media Bridge randomly generates the file name. This directsÂ the recording to a central storage location you mountÂ inside the docker container (use the <code style=" word-wrap: break-word;">-v</code> option). When you set this field to an RTMP URI, Media Bridge streams the recording to that destination. No permanent record is stored.</td> 
          </tr> 
          <tr> 
           <td><strong style="word-wrap: break-word;">resourceId</strong></td> 
           <td>The <a href="http://vidyo.io">vidyo.io</a> resource to connect to and record.</td> 
          </tr> 
          <tr> 
           <td><strong style="word-wrap: break-word;">token</strong></td> 
           <td>The access token to the <a href="http://vidyo.io">vidyo.io</a> system.</td> 
          </tr> 
          <tr> 
           <td><strong style="word-wrap: break-word;">host</strong></td> 
           <td>The <a href="http://vidyo.io">vidyo.io</a> host name to connect to. Make sure that this host name can be resolved inside your container. This can be done using the <code style=" word-wrap: break-word;">--add-host</code> parameter when creating the container.</td> 
          </tr> 
          <tr> 
           <td><strong style="word-wrap: break-word;">mediaPorts</strong></td> 
           <td>The ports used to send and receive media from the SIP endpoint (e.g. <code style=" word-wrap: break-word;">mediaPorts=50000-50100</code>). When you start the Docker container, map these ports to the same port numbers on the host via the <code style=" word-wrap: break-word;">-p</code> parameter.</td> 
          </tr> 
          <tr> 
           <td><strong style="word-wrap: break-word;">publicIp</strong></td> 
           <td>The IP address of the host. It must be reachable by the SIP endpoint participating in the call (e.g. <code style=" word-wrap: break-word;">publicIp=1.2.3.4</code>).</td> 
          </tr> 
         </tbody> 
        </table>
       </div>
