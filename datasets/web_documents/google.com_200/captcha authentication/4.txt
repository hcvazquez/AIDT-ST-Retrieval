       <h1 itemprop="name" class="devsite-page-title"> Authentication and Authorization in the Google Data Protocol </h1> 
       <nav class="devsite-page-nav-embedded devsite-nav"></nav> 
       <div class="devsite-article-body clearfix
            " itemprop="articleBody"> 
        <p class="warning"><b>Warning</b>: Most newer Google APIs are not Google Data APIs. The Google Data APIs documentation applies only to the older APIs that are listed in the <a href="https://developers.google.com/gdata/docs/directory">Google Data APIs directory</a>. For information about a specific new API, see that API's documentation. For information about authorizing requests with a newer API, see <a href="https://developers.google.com/accounts/">Google Accounts Authentication and Authorization</a>.</p> 
        <p>Third-party applications often require limited access to a user's Google Account for certain types of activity. To ensure that user data is not abused, all requests for access must be approved by the account holder. Access control has two components, authentication and authorization.</p> 
        <p>Authentication services allow users to sign in to your application using a Google Account. Some services also allow users to sign in using another account, such as an OpenID login.</p> 
        <p>Authorization services let users provide your application with access to the data they have stored in Google applications. Google takes privacy seriously, and any application that requires access to a user's data must be authorized by the user.</p> 
        <p>Authentication and authorization services are often referred to collectively as <i>auth</i>. </p>
        <p>Authentication and authorization for Google APIs allow third-party applications to get limited access to a user's Google accounts for certain types of activities. This document introduces the available auth mechanisms and describes what each one provides for your application.</p> 
        <ul> 
         <li><a href="#OAuth2">OAuth 2.0</a> is a new, simplified authorization protocol for all Google APIs. OAuth 2.0 relies on SSL for security instead of requiring your application to do cryptographic signing directly. This protocol allows your application to request access to data associated with a user's Google Account. </li> 
         <li><a href="#OAuth">OAuth 1.0</a> provides authorization for all Google APIs.</li> 
         <li><a href="https://developers.google.com/accounts/docs/OpenID#oauth">Hybrid protocol</a> provides both authentication and authorization for web applications, allowing users to log in and authorize access to their data in a single step. This protocol uses OpenID to provide authentication services, and OAuth to provide authorization to Google APIs.</li> 
         <li><a href="https://developers.google.com/accounts/docs/OpenID">OpenID</a> authenticates users with their Google Accounts. This allows users to log in to your web site without having to sign up for a new account.</li> 
        </ul> 
        <p><a href="https://developers.google.com/accounts/docs/AuthSub">AuthSub</a> and <a href="https://developers.google.com/accounts/docs/AuthForInstalledApps">ClientLogin</a> are Google's proprietary authorization APIs, available as an alternative to OAuth for most Google APIs. If possible, you should avoid using these services. If you already have applications that use them, you should migrate to the recommended options described above.</p> 
        <p>If your application is a gadget (for use in iGoogle or other OpenSocial containers), see the section <a href="https://developers.google.com/gdata/docs/auth/overview#Gadgets">Authentication for Gadgets</a>.</p> 
        <p class="note"><strong>Note</strong>: This document is intended to provide an overview of each authentication method. For detailed information on each method, see the full <a href="https://developers.google.com/accounts/Authentication">Google Account Authentication APIs</a> documentation.</p> 
        <p>See also the <a href="http://groups.google.com/group/Google-Accounts-API">Google Accounts API Group</a> for discussion on using the Google Accounts APIs.</p>

        <a name="OAuth"></a>
        <h2>OAuth - authorization for web and installed applications</h2> 
        <p>Many Google services allow third-party access to user-generated data, such as Calendar or Documents data, as long as access is authorized by the user. This feature allows users to share and exchange data between their Google applications and third-party applications for a variety of purposes.</p> 
        <p>Google supports two versions of OAuth for getting authorized access to a user's Google data: OAuth 1.0 and OAuth 2.0, both offering access to both web applications and installed applications.</p> 
        <h3>OAuth 2.0 for web and installed applications</h3> 
        <p>Web applications or installed applications can use the new, simplified OAuth 2.0 protocol to authorize access to data associated with a Google account. For details and examples of how to implement OAuth 2.0 with Google, see our <a href="https://developers.google.com/accounts/docs/OAuth2">documentation on OAuth 2.0</a>.</p> 
        <h3>OAuth 1.0 for web applications</h3> 
        <p>Web applications that need authorized access to data associated with a Google Account or a Google Apps Account can use Google's implementation of the OAuth API. For complete information about implementing OAuth for web-based applications, including examples, see the <a href="https://developers.google.com/accounts/docs/OAuth">OAuth for Web Apps</a> guide, or see the <a href="#OAuthWeb">overview</a> in this document.</p> 
        <h3>OAuth 1.0 for installed applications</h3> 
        <p>Applications installed on users' computers and mobile devices can use OAuth to authorize access to data associated with a Google Account. For complete information about implementing OAuth for installed applications, see the <a href="https://developers.google.com/accounts/docs/OAuthForInstalledApps">OAuth for Installed Applications</a> guide, or see the <a href="#OAuthInstalled">overview</a> in this document.</p> 
        <h2 id="OAuthWeb">Using OAuth with web applications</h2> 
        <p>All Google Data APIs support <a href="http://oauth.net/" target="_blank">OAuth</a>, an open standard for authorizing the use of data in web applications. All web applications that make OAuth requests must upload a security certificate and register with Google. See <a href="https://developers.google.com/accounts/docs/RegistrationForWebAppsAuto">Registration for Web-Based Applications</a> for more information.</p> 
        <p>The <a href="https://developers.google.com/gdata/docs/client-libraries">Google Data APIs Client Libraries</a> provide methods to help you use OAuth in your web application. Specifically, there are methods for constructing the request token, authorizing the request token, and exchanging the authorized request token for an access token. The libraries also handle the necessary signing algorithms when making requests to a Google Data service. For extensive examples of how to use OAuth with the Google Data API Client Libraries,see <a href="https://developers.google.com/gdata/oauth">Using OAuth with the Google Data APIs Client Libraries</a>.</p> 
        <h3 id="AuthProcess">The OAuth authorization process</h3> 
        <p>The OAuth authorization process involves a series of interactions between your web application, Google's authorization servers, and the end user.</p> 
        <p>At a basic level, the process is as follows:</p> 
        <ol> 
         <li>Your application requests access and gets an unauthorized request token from Google's authorization server.</li> 
         <li>Google asks the user to grant you access to the required data.</li> 
         <li>Your application gets an authorized request token from the authorization server.</li> 
         <li>You exchange the authorized request token for an access token.</li> 
         <li>You use the access token to request data from Google's service access servers.</li> 
        </ol> 
        <p>When your application initially requests access to a user's data, Google issues an unauthorized request token to your application.</p> 
        <p>If the user is not already logged in, Google prompts the user to log in. Google then displays an authorization page that allows the user to see what Google service data your application is requesting access to.</p> 
        <p>If the user approves your application's access request, Google issues an authorized request token. Each request token is valid for only one hour. Only an authorized request token can be exchanged for an access token, and this exchange can be done only once per authorized request token.</p> 
        <p>By default, access tokens are long-lived. Each access token is specific to the user account specified in the original request for authorization, and grants access only to the services specified in that request. Your application should store the access token securely, because it's required for all access to a user's data.</p> 
        <h3 id="ReadyOauth">Getting ready for OAuth</h3> 
        <p>Before you can set up your application to use the Google Authorization service with OAuth, you must complete the following tasks.</p> 
        <a name="prepRegister"></a> 
        <h3>Deciding whether to register your web application</h3> 
        <p>To provide your users with additional assurances of the security of their data, you can choose to register your web application with Google and sign your requests with the registered security certificate. Some Google Data API feeds are available to registered applications only. See the documentation for the Google Data API you're interested in to determine if that API only works with registered applications.</p> 
        <p>Your application must sign each OAuth request it makes. If you choose to use a RSA-SHA1 signature to sign your requests, you must upload a security certificate as part of the registration process.</p> 
        <p>Alternatively, you can use a HMAC-SHA1 signature to sign your requests. No certificate is required for HMAC-SHA1 signatures. Instead, Google generates an OAuth <code>consumer secret</code> value, which is displayed on your domain's registration page after you have registered.</p> 
        <p>For more information on the registration process, see <a href="https://developers.google.com/accounts/docs/RegistrationForWebAppsAuto">Registration for Web Applications</a>.</p> 
        <a name="prepScope"></a> 
        <h3>Determining the scope of the data your application will access</h3> 
        <p>Each Google service sets limits on the access it allows through the Google Data APIs. This access is expressed as a scope value. Some services provide a variety of scope values, to allow a user to choose which applications should have access to which data. For information about the available scope values for the Google service you want to access, see the documentation for that service.</p> 
        <p>In general, you should request a token for the narrowest scope that includes the data you need. For example, if your application requires access to the user's "All Calendars" feed, you should request a token for the scope <code>http://www.google.com/calendar/feeds/default/allcalendars/full</code>.</p> 
        <a name="prepToken"></a> 
        <h3>Setting up a mechanism to manage OAuth tokens</h3> 
        <p>When you obtain an OAuth access token for a user's data, you must use that access token for all future interactions with the specified Google service on behalf of the user.</p> 
        <p>Your application should manage token storage securely, including tracking the Google service for which each token is valid. If you require access to more than one Google service, you can obtain multiple access tokens, but no more than ten access tokens per user and application can be outstanding at any time. </p> 
        <p> If your application supports multiple user accounts, you must keep track of which account each token is associated with. Each OAuth token is specific to the user who authorized access. Your application must be able to associate a token with the correct user. One way to manage this is to issue a cookie to the user before making the token request. After the user grants access to the requested data, Google sends an authorized request token and redirects the user to your application. You can then use your application's cookie to associate the token with the correct user. </p> 
        <a name="prepServiceCall"></a> 
        <h3>Setting up a mechanism to request access to a Google service</h3> 
        <p>Each request to a Google service must be signed, and must include a valid OAuth access token. In general, each request is made in the form of a HTTP GET request, with the access token and signature included in the header. Requests that write new data should use HTTP POST.</p> 
        <p>For more information on the proper request format for each <a href="http://code.google.com/apis/gdata/">Google Data API</a>, refer to the documentation for that API.</p> 
        <a name="prepOpenID"></a> 
        <h3>Implementing OpenID (optional)</h3> 
        <p>If you're implementing OpenID for user authentication, consider using the <a href="https://developers.google.com/accounts/docs/OpenID#oauth">hybrid protocol</a> to combine the two processes. With <a href="https://developers.google.com/accounts/docs/OpenID#oauth">OpenID+OAuth</a>, the tasks of getting a request token and authorizing it are handled as part of the OpenID request with OAuth extensions. As with <code>OAuthGetRequestToken</code>, these extensions are used to identify the Google services to be accessed. A successful response to the OpenID request contains an authorized request token. Once this token is received, use <code>OAuthGetAccessToken</code> to exchange it for an access token.</p> 
        <h3><a name="WorkingOauth" id="WorkingOauth"></a>Working with OAuth tokens</h3> 
        <p>To use OAuth, your application must generate well-formed, signed token request calls, and handle the responses, for the following sequence:</p> 
        <ol>
         <li>Get an unauthorized request token (<a href="https://developers.google.com/accounts/docs/OAuth_ref#RequestToken">OAuthGetRequestToken</a>) </li>
         <li>Authorize the request token (<a href="https://developers.google.com/accounts/docs/OAuth_ref#GetAuth">OAuthAuthorizeToken</a>) </li>
         <li>Exchange the authorized request token for an access token (<a href="https://developers.google.com/accounts/docs/OAuth_ref#AccessToken">OAuthGetAccessToken</a>)</li>
        </ol>
        <p></p> 
        <p>All OAuth requests must be signed, whether or not your application is registered. For further information, see <a href="https://developers.google.com/accounts/docs/OAuth_ref#SigningOAuth">Signing OAuth Requests</a>. </p> 
        <p>You can experiment with requesting and receiving authorization tokens in the <a href="https://developers.google.com/accounts/docs/OAuth_ref#TestingDebugging">OAuth Playground</a>.</p> 
        <p>For detailed documentation, see the <a href="https://developers.google.com/accounts/docs/OAuth_ref">OAuth API Reference</a>.</p> 
        <a name="tokensCallback"></a> 
        <h3>Setting a callback URL</h3> 
        <p>You can specify a value for <code>oauth_callback</code> in an <code>OAuthGetRequestToken</code> request, to determine where Google redirects the user after they authorize your access request. The callback URL can include query parameters. The redirect will include the same query parameters, as well as the authorized request token, which your application must be able to parse.</p> 
        <p>For example, when supporting multiple languages, you can include a query parameter that identifies the version of the application that a user is viewing. An <code>oauth_callback</code> value of "http://www.yoursite.com/Retrievetoken?Lang=de would result in the redirect "http://www.yoursite.com/Retrievetoken?Lang=de&amp;oauth_token=DQAADKEDE". Parsing the token and the language parameter ensures that the user is redirected back to the correct version of the site.</p> 
        <p>If the <code>oauth_callback</code> parameter is not included, Google will direct the user to a web page that displays a verification number (<a href="https://developers.google.com/accounts/images/OauthUX_nocallback.png">see example</a>), after authorizing your access request. The user must manually return to your application and enter the verification number before you can obtain an authorized request token.</p> 
        <a name="tokensIdentifying"></a> 
        <h3>Identifying your application to users</h3> 
        <p>Google normally displays the name of an application when requesting access consent from the user (<a href="https://developers.google.com/accounts/images/InstalledOauthUX.png">see example</a>).</p> 
        <p>If your application is not registered, use the <code><a href="https://developers.google.com/accounts/docs/OAuth_ref#Parameters">xoauth_displayname</a></code> parameter in your <code>OAuthGetRequestToken</code> request to specify the name of your application. If that parameter is not specified, Google displays the domain name of the URL provided by the <code>oauth_callback</code> parameter. If no callback URL is provided, Google displays the string "anonymous".</p> 
        <p>Do not set this parameter if your application is registered. By default, Google shows the display name specified during registration. If you set a display name in your <code>OAuthGetRequestToken</code> request, Google will use this instead of your registered display name, and will include a message that the identity of your application cannot be verified.</p> 
        <p class="note"><strong>Note:</strong> To set the <code>xoauth_displayname</code> parameter in the <a href="https://developers.google.com/accounts/docs/OAuth_ref#TestingDebugging">OAuth Playground</a>, check the "Advanced" box before fetching the request token.</p> 
        <a name="tokensGADomains"></a> 
        <h3>Working with Google Apps domains</h3> 
        <p>If your application is designed for users on a hosted Google Accounts domain, consider using the <a href="https://developers.google.com/accounts/docs/OAuth_ref#Parameters3">hd</a> parameter when authorizing a token. For more information on the <em>hd</em> parameter, see <a href="https://developers.google.com/accounts/docs/OAuth_ref#hd">Handling Users with Multiple Accounts</a>.</p> 
        <h3>More information about OAuth</h3> 
        <p>For detailed information on Google's implementation of OAuth, including how to register your application and construct the necessary OAuth parameters, see these additional resources:</p> 
        <ul> 
         <li><a href="//code.google.com/apis/gdata/oauth.html">Using OAuth with the Google Data APIs Client Libraries</a></li> 
         <li>Article: <a href="//code.google.com/apis/gdata/articles/oauth.html">Using OAuth with the Google Data APIs</a>, including a description of the OAuth Playground, an interactive tool for trying out OAuth.</li>

Installed applications should follow the instructions for an unregistered application.</p> 
        <p>You can experiment with requesting and receiving authorization tokens in the <a href="https://developers.google.com/accounts/docs/OAuth_ref#TestingDebugging">OAuth Playground</a>.</p> 
        <p>For detailed documentation, see the <a href="https://developers.google.com/accounts/docs/OAuth_ref">OAuth API Reference</a>.</p> 
        <a name="tokensIdentifying"></a> 
        <h3>Identifying your application to users</h3> 
        <p>Google normally displays the name of an application when requesting access consent from the user (<a href="https://developers.google.com/accounts/images/InstalledOauthUX.png">see example</a>).</p> 
        <p>Use the <code><a href="https://developers.google.com/accounts/docs/OAuth_ref#Parameters">xoauth_displayname</a></code> parameter in your <code>OAuthGetRequestToken</code> request to specify the name of your application. If that parameter is not specified, Google displays the domain name of the URL provided by the <code>oauth_callback</code> parameter. If no callback URL is provided, Google displays the string "anonymous".</p> 
        <p class="note"><strong>Note:</strong> To set the <code>xoauth_displayname</code> parameter in the <a href="https://developers.google.com/accounts/docs/OAuth_ref#TestingDebugging">OAuth Playground</a>, check the "Advanced" box before fetching the request token.</p> 
        <a name="tokensLaunchingBrowser"></a> 
        <h3>Launching a web browser</h3> 
        <p>As part of the OAuth authorization process, your application must make a <code>OAuthAuthorizeToken</code> request. The user must then log in to a Google web page and authorize your application's access request.</p> 
        <ul> 
         <li>AutoDetect mode should be used for most applications</li> 
         <li>Device mode should be used for applications which cannot launch a full web browser.</li> 
         <li>Development mode should be used during early development only.</li> 
        </ul> 
        <h4>AutoDetect mode</h4> 
        <p>If possible, your application should launch a browser window and make an <code>OAuthAuthorizeToken</code> request to open the Google page. When Google returns the authorized token, your application should detect this and regain focus from the web browser.</p> 
        <p>This mode requires that you provide a callback URL to which the user is redirected after they authorize your access request. This URL must be provided as the <code>oauth_callback</code> parameter of the <code>OAuthGetRequestToken</code> request, and as the <code>verifier</code> parameter of the <code>OAuthGetAccessToken</code> request. </p>
        <p>To improve the user experience, your application should attempt to automatically detect when the user is redirected to this URL, and immediately bring itself to the foreground and make a <code>OAuthGetAccessToken</code> request to complete the OAuth process.</p> 
        <p>For more information and best practices, see <a href="http://sites.google.com/site/oauthgoog/oauth-practices/auto-detecting-approval">Auto-Detecting Approval</a>.</p> 
        <p>If your application cannot automatically detect when the user is redirected to the callback URL, or cannot bring itself to the foreground, the callback URL should display a page that explains how to bring your application to the foreground and how to initiate the <code>OAuthGetAccessToken</code> request from within your application.</p> 
        <h4>Device mode</h4> 
        <p>If your application cannot launch a full web browser, it is also possible for rich-client devices to <a href="http://sites.google.com/site/oauthgoog/UXFedLogin/nobrowser/input-capable-devices">authorize without a web browser</a>.</p> 
        <p>This mode allows a developer to set up a website where a user can authorize the request for access. After authorization, the user is given a code generated by Google, and redirected to the developer's site. This site should explain to the user how to input the code into their device to complete the authorization process.</p> 
        <h4>Development mode</h4> 
        <p>This mode is recommended for use during the early development of an application only.</p> 
        <p>As in the AutoDetect mode, your application must launch a browser, and the user must authorize your request. However, instead of creating a webpage for the callback URL, you can set the value of the <code>oauth_callback</code> parameter to <code>"oob"</code> (out of band).</p> 
        <p>In that case, after the user authorizes your request, Google directs the user to a Google Accounts page that displays a verification number (see <a href="http://code.google.com/apis/accounts/images/OauthUX_nocallback.png">example</a>).</p> 
        <p>The user must return to your application and enter the verification number, before you can make a <code>OAuthGetAccessToken</code> request.</p> 
        <h3>More information about OAuth</h3> 
        <p>For detailed information on Google's implementation of OAuth, including how to register your application and construct the necessary OAuth parameters, see these additional resources:</p> 
        <ul> 
         <li><a href="//code.google.com/apis/gdata/oauth.html">Using OAuth with the Google Data APIs Client Libraries</a></li> 
         <li>Article: <a href="//code.google.com/apis/gdata/articles/oauth.html">Using OAuth with the Google Data APIs</a>, including a description of the OAuth Playground, an interactive tool for trying out OAuth.</li> 
         <li><a href="https://developers.google.com/accounts/docs/OAuthForInstalledApps">OAuth for Installed Applications</a> (full documentation)</li> 
         <li><a href="//code.google.com/apis/gdata/docs/auth/authsub.html#Registered">Generating keys and certificates</a></li> 
         <li><a href="http://oauth.net/" target="_blank">OAuth specification</a></li> 
        </ul>

        <h2 id="authsub">Using AuthSub for authorization</h2> 
        <p>AuthSub is a Google proprietary authorization API, available as an alternative to OAuth for most Google APIs. You should avoid using AuthSub if possible. If you already have applications that use AuthSub, you should migrate to OAuth or the hybrid protocol.</p> 
        <h2 id="AuthProcess">The AuthSub authorization process</h2> 
        <p>Authorization with AuthSub involves a sequence of interactions between three entities: the web application, Google services, and the user. This diagram illustrates the sequence:</p> 
        <p align="center"><img src="https://developers.google.com/accounts/images/AuthsubDiagram.png" alt="Authorization process"></p> 
        <ol class="doublespace"> 
         <li>When the web application needs to access a user's Google service, it makes an AuthSub call to Google's Authorization Proxy service.</li> 
         <li>The Authorization service responds by serving up an Access Request page. This Google-managed page prompts the user to grant/deny access to their Google service. The user may first be asked to log into their account.</li> 
         <li>The user decides whether to grant or deny access to the web application. If the user denies access, they are directed to a Google page rather than back to the web application.</li> 
         <li>If the user grants access, the Authorization service redirects the user back to the web application. The redirect contains an authorization token good for one use; it can be exchanged for a long-lived token.</li> 
         <li>The web application contacts the Google service with a request, using the authorization token to act as an agent for the user.</li> 
         <li>If the Google service recognizes the token, it supplies the requested data.</li> 
        </ol> 
        <h2 id="WorkingAuthSub">Working with AuthSub</h2> 
        <p>Incorporating AuthSub into your web application requires these tasks:</p> 
        <ol class="listhead"> 
         <li>Decide whether or not to register your web application. <p>Registered web applications have the advantage of being recognized by Google; the standard caveat displayed to users on the Google login page is modified or omitted. In addition, registered web applications are identified with a descriptive name rather than merely the calling URL. Keep in mind that some Google services allow only limited access to unregistered web applications. If you choose to register, use this automated <a href="https://developers.google.com/accounts/docs/RegistrationForWebAppsAuto">registration process</a>. </p> <p>If you register, you can also provide a security certificate and keys. Registered web applications with a certificate on file are able to acquire secure tokens for use when requesting data from a Google service. (They can also use non-secure tokens if necessary.)</p> </li> 
         <li>Decide what type of tokens to use and how to manage them. <p>An authorization token received from Google is intended to be used for all future interactions with the specified Google service for that user. The type of token you choose to use--single-use or session--depends on the type of interactions your web application will have with a Google service. For example, a single-use token may be sufficient if the interaction is a one-time or rare event.</p> <p>If you choose to get session tokens and use them regularly to access the Google service, your web application will need to manage token storage, including tracking the user and Google service the token is valid for. Google Accounts is not set up to manage large numbers of tokens, and in fact does not allow more than ten valid tokens (per user, per web application) to be outstanding at any one time. This limitation allows a web application to get multiple tokens to cover different services, if necessary; it does not support getting a new token each time the web application needs to access a Google service. If you decide to store session tokens, the tokens should be treated as securely as any other sensitive information stored on the server. </p> <p>Alternatively, you can choose to get fresh tokens on a regular basis as long as you set up a token revocation mechanism. Your application would need to revoke the existing token before requesting another. In this scenario, the user would be required to log in and grant access each time a new token is requested.</p> </li> 
         <li>Determine the scope required by the Google service to be accessed. <p>Each Google service determines how much and what type of access it will allow. This access is expressed as a scope value. A service's scope may be a simple URL identifying the entire service, or it may specify more restricted access. Some services severely limit access, such as allowing read-only access to limited information. To get the allowed scopes for the Google service you want to access, refer to the documentation for that service. You should request the most tightly scoped token possible. For example, if attempting to access Gmail's Atom feed feature, use the scope "http://www.google.com/calendar/feeds/", not "http://www.google.com/calendar/". Google services are much more restrictive with large-scope requests.<a name="authsub_mechanism" id="authsub_mechanism"></a></p> </li> 
         <li>Set up a mechanism to request and receive an authorization token. <p>The mechanism must generate a well-formed <a href="#AuthSubRequest">AuthSubRequest</a> call, including specifying the appropriate <em>next</em> and <em>scope</em> URL values (determined in step 3). If you are using secure tokens and/or are managing session tokens, the request must include values for these variables as well.</p> <p>The next URL can include query parameters. For example, when supporting multiple languages, include a query parameter that identifies the version of the web application the user is viewing. A <em>next</em> value of <code>http://www.yoursite.com/Retrievetoken?Lang=de</code> would result in the redirect <code>http://www.yoursite.com/Retrievetoken?Lang=de&amp;token=DQAADKEDE</code>. Parsing the token and the Lang parameter ensures that the user is redirected back to the correct version of the site.</p> <p>In certain circumstances, using the <em>hd</em> parameter can help streamline your users' experience when they are asked to grant access on the Google Accounts site. In most cases, the process is a simple matter of logging into their account and choosing whether or not to grant access. Users who have more than one Google Account (a regular gmail account and/or one or more Google Apps <a href="http://www.google.com/a/help/intl/en/index.html">hosted accounts</a>), however, may need to follow the extra "universal login" process to to identify which account they wish to have accessed. If your application is designed for one particular managed domain, you can eliminate these extra steps by using this parameter to identify that domain. You can also use the <em>hd</em> parameter if your application accesses services that are not available on hosted accounts--setting the value to 'default' will limit authorization to regular accounts only. When the <em>hd</em> value is set, Google will automatically select the correct account for authorization.</p> <p>The token mechanism must be equipped to parse the redirect received from Google, which contains the single-use token, and take action with it. Because authorization tokens are specific to a user, your application must be able to associate a token with its user. The preferred option is to issue a cookie to the user before making the token request. Then, when Google redirects the user back to your site with an appended token, your application can read the cookie and associate the token with the correct user identification.</p> </li> 
         <li>Set up mechanisms to request session tokens and store or revoke them, if relevant. <p>Depending on what token management decisions you made in step 2, you may need to create mechanisms to get and revoke session tokens (<a href="#AuthSubSessionToken">AuthSubSessionToken</a> and <a href="#AuthSubRevokeToken">AuthSubRevokeToken</a>). To test session and revocation mechanisms, use <a href="#AuthSubTokenInfo">AuthSubTokenInfo</a>, which indicates whether a given token is valid or not. If storing tokens, the application must track both the user ID and the service (scope) covered by the token.</p> </li> 
         <li id="ServiceCall">Set up a mechanism to request access to a Google service. <p>Refer to documentation for the Google service for information on the proper request format. All requests to a Google service must include a valid authorization token. In general, a request to a Google service is in the form of an HTTP GET (or POST if writing new data), with the token referenced in the request header. </p> <p>The request header should take the following form:</p> <pre>Authorization: AuthSub token="token"</pre> <p>where <em>token</em> is the authorization token, single-use or session, received from Google in response to an AuthSub request. If the token is secure, it must be accompanied by a digital signature. See "<a href="#signingrequests">Signing requests</a>" for instructions and examples.</p> <p>The example below illustrates a request header for a call to the Google Calendar feed service. This request contains a non-secure token: </p> <pre>GET /calendar/feeds/default/private/full HTTP/1.1
Content-Type: application/x-www-form-urlencoded
Authorization: AuthSub token="GD32CMCL25aZ-v____8B"
User-Agent: Java/1.5.0_06
Host: www.google.com
Accept: text/html, image/gif, image/jpeg, *; q=.2, */*; q=.2
Connection: keep-alive</pre> </li>
        </ol> 
        <h3>More information about AuthSub</h3> 
        <p>For information on AuthSub, including registering your application with Google and a detailed explanation of exchanging a one-time token for a session token, see these additional resources:</p> 
        <ul> 
         <li><a href="https://developers.google.com/accounts/docs/AuthSub">AuthSub Authentication for Web Applications</a> (full documentation)</li>

        </ul>

        <h2>Using ClientLogin for authorization</h2> 
        <p>ClientLogin is a Google proprietary authorization API, available as an alternative to OAuth for most Google APIs. You should avoid using ClientLogin if possible. If you already have applications that use ClientLogin, you should migrate to OAuth or the hybrid protocol.</p> 
        <h2 id="ClientLogin">Authentication for Installed Applications: ClientLogin</h2> 
        <p>ClientLogin allows your users to log into their Google account from inside your application. The application then contacts Google with the login data and requests access to a specified Google Data API. Once the login information has been successfully authenticated, Google returns a token, which your application will reference each time it requests access to the user's account, such as to get or post data. The token remains valid for a set length of time, defined by whichever Google service you're working with.</p> 
        <p class="note"><strong>Note</strong>: The <a href="https://developers.google.com/gdata/docs/client-libraries">Google Data APIs Client Libraries</a> provide methods to help you use ClientLogin in your installed applications. Specifically, there are methods for acquiring an authentication token, handling CAPTCHA challenges, recalling the auth token for later use, and sending the correct <code>Authorization</code> header with every request. If you are working with one of the libraries, see <a href="https://developers.google.com/gdata/clientlogin">Using ClientLogin with the Google Data APIs Client Libraries</a>.</p> 
        <h2 id="AuthProcess">The ClientLogin authorization process</h2> 
        <p>Authorization with ClientLogin involves a sequence of interactions between three entities: the installed application, Google services, and the user. This diagram illustrates the sequence:</p> 
        <p align="center"><img src="https://developers.google.com/accounts/images/ClientLoginDiagram.png" alt="Authorization process"></p> 
        <ol class="doublespace"> 
         <li>When the third-party application needs to access a user's Google service, it retrieves the user's login name and password.</li> 
         <li>The third-party application then makes a ClientLogin call to Google's Authorization service.</li> 
         <li>If the Google Authorization service decides additional vetting is necessary, it returns failure response with a CAPTCHA token and challenge, in the form of a URL for a CAPTCHA image. </li> 
         <li>If a CAPTCHA challenge is received, the third-party application displays the CAPTCHA image for the user and solicits an answer from the user.</li> 
         <li>If requested, the user submits an answer to the CAPTCHA challenge.</li> 
         <li>The third-party application makes a new ClientLogin call, this time including the CAPTCHA answer and token (received with the failure response).</li> 
         <li>On a successful login attempt (with or without CAPTCHA challenge), the Google Authorization service returns a token to the application.</li> 
         <li>The application contacts the Google service with a request for data access, referencing the token received from the Google Authorization service.</li> 
         <li>If the Google service recognizes the token, it supplies the requested data access.</li> 
        </ol> 
        <h2 id="Using">Using ClientLogin</h2> 
        <p>Use this interface in your installed application to programmatically access a user's Google account. After collecting login information from the user, call ClientLogin to request access to the user's account. Once the login information has been successfully authenticated, Google returns a token, which your application will reference each time it requests access to the user's account. The token remains valid for a set length of time, which is defined by whichever Google service you're working with.</p> 
        <p>Incorporating ClientLogin into your application requires these tasks:</p> 
        <ol class="listhead"> 
         <li>Create a UI element to capture login data from the user. <p>The UI needs to solicit a user name (email address including domain) and password. The UI should also be capable of displaying a CAPTCHA image using the URL received from Google, if one is required, and soliciting a correct answer from the user. Ideally, your UI will include a link to Google Accounts login page ("https://www.google.com/accounts/Login") in the event that the user needs to sign up for a new account or do other account maintenance.</p> </li> 
         <li>Write code to generate a well-formed HTTPS POST <code>ClientLogin</code> request and transmit it. <p>This code needs to contain logic to handle a CAPTCHA challenge and include both the <code>logintoken</code> and <code>logincaptcha</code> parameters. The application should also be able to detect when the user omits required information--or repeats incorrect data after a login failure--and display an error without sending a superfluous request.</p> </li> 
         <li>Handle responses from Google. <p>There are four possible responses to a login request:</p> 
          <ul> 
           <li class="sspace">success (an HTTP 200)</li> 
           <li class="sspace">failure (an HTTP 403) with an explanatory error code</li> 
           <li class="sspace">invalid request, generally resulting from a malformed request</li> 
           <li>failure with a CAPTCHA challenge</li> 
          </ul> <p>A success response contains an authorization token labeled "Auth". This token must be included in all subsequent requests to the Google service for this account. Authorization tokens should be closely guarded and should not be given to any other application, as they represent access to the user's account. The time limit on the token varies depending on which service issued it.</p> <p>A failure response includes one or more error codes and a URL with the error message that can be displayed for the user. Please note that <code>ClientLogin</code> does not differentiate between a failure due to an incorrect password or one due to an unrecognized user name (for example, if the user has not yet signed up for an account). Your application needs to handle all possible error messages as appropriate. </p><p>A failure response with a CAPTCHA challenge means that Google has decided, for whatever reason, that additional security measures should be taken. This response is accompanied by a CAPTCHA image URL and a token representing the specific CAPTCHA challenge.</p> </li> 
         <li>Handle a CAPTCHA challenge from Google. <p>To handle the challenge, the application must display the CAPTCHA image and solicit an answer from the user. To display the CAPTCHA image, use the value of <code>CaptchaUrl</code> returned with the failure response, prefixing it with the Google Accounts URL: "http://www.google.com/accounts/". Once the user provides an answer, the application should resend the login request, this time including the CAPTCHA token (<code>logintoken</code>) and the user's answer (<code>logincaptcha</code>). Google validates the user's answer before authorizing access to the account. </p> <p>There is an alternative for developers who do not want to manage the process's of getting and transmitting a user CAPTCHA response. In response to a CAPTCHA challenge, the application can direct the user to the Google hosted page: "https://www.google.com/accounts/DisplayUnlockCaptcha". Once the user has successfully responded to the challenge, the Google server trusts the computer in use. The application can then resend the original login request to obtain the authorization token.</p> </li> 
         <p class="note"><strong>Note:</strong> Google does not validate the login attempt prior to issuing a CAPTCHA challenge. This means a login attempt could fail even after a CAPTCHA challenge. </p> 
        </ol> 
        <p style="font-size: x-small;">* CAPTCHA is a trademark of Carnegie Mellon University</p>

        <h2 id="Gadgets">Authentication for Gadgets</h2> 
        <h3 id="OAuthProxy">OAuth Proxy</h3> 
        <p>If you're building a gadget using the standard <a href="https://developers.google.com/gadgets">Gadgets API</a>, you can leverage a feature of the gadget platform called the OAuth Proxy to access the Google Data APIs. OAuth (<a href="#OAuth">described above</a>) is an authentication standard that allows users to access their private data in a gadget hosting service such as iGoogle, MySpace, or Orkut, or share their data with another website or gadget. The OAuth Proxy is designed to make development easier for gadget developers by hiding many of OAuth's authentication details. The Proxy is based on an open-source project called <a href="http://incubator.apache.org/shindig/">Shindig</a>, which is an implementation of the gadget specification.</p> 
        <p class="note"><strong>Note</strong>: The OAuth Proxy is only supported for gadgets that use the <code>gadgets.*</code> API and run in OpenSocial containers. It is not supported for the <a href="https://developers.google.com/gadgets/docs/legacy/dev_guide">legacy gadgets API</a>.</p> 
        <h4 id="GadgetAuthenticationFlow">Authentication flow</h4> 
        <p>Your gadget must obtain an OAuth token before it can access a user's data. The OAuth Proxy manages the authentication flow for you. The first time a user installs your gadget, the following process takes place:</p> 
        <ol> 
         <li>Your gadget loads for the first time and attempts to access the user's data using one of the Google Data APIs.</li> 
         <li>The request fails because the user hasn't granted access. The response object contains a URL (in <code>response.oauthApprovalUrl</code>) for the OAuth approval page. Your gadget should provide a method to launch a new window with that URL.</li> 
         <li>On the approval page, the user chooses to grant or deny access to your gadget. If successful, the user is taken to the <code>oauth_callback</code> page you specify. For the best user experience, use <code>http://oauth.gmodules.com/gadgets/oauthcallback</code>.</li> 
         <li>Next, the user closes the popup window. To notify your gadget that the user has approved, you can use a <a href="http://code.google.com/p/gdata-samples/source/browse/trunk/gadgets/popup.js">popup handler</a> to detect the approval window closing. Alternatively, your gadget can display a link (e.g. "<a>I've approved access</a>") for the user to click after this window closes.</li> 
         <li>Your gadget attempts to access the Google Data API a second time by re-requesting the user's data. This attempt is successful.</li> 
         <li>Your gadget is authenticated and can begin operating normally.</li> 
        </ol> 
        <h4>Gadget setup</h4> 
        <p>To access one or more of the Google Data APIs, you first need to tell your gadget to use OAuth as the authentication method. Add an <code>&lt;OAuth&gt;</code> element in the <code>&lt;ModulePrefs&gt;</code> section of your gadget's XML:</p> 
        <pre class="prettyprint">
&lt;ModulePrefs&gt;
...
&lt;OAuth&gt;
  &lt;Service name="google"&gt;
    &lt;Access url="https://www.google.com/accounts/OAuthGetAccessToken" method="GET" /&gt;
    &lt;Request url="https://www.google.com/accounts/OAuthGetRequestToken?
                  scope=http://www.blogger.com/feeds/%20http://www.google.com/calendar/feeds/" method="GET" /&gt;
    &lt;Authorization url="https://www.google.com/accounts/OAuthAuthorizeToken?
                        oauth_callback=http://oauth.gmodules.com/gadgets/oauthcallback" /&gt;
  &lt;/Service&gt;
&lt;/OAuth&gt;
...
&lt;/ModulePrefs&gt;
</pre> 
        <p>In this section, only change the following query parameters:</p> 
        <dl> 
         <dt>
          <code>scope</code>
         </dt> 
         <dd>
          A required parameter in the Request URL. Your gadget can access data from the 
          <code>scope</code>(s) used in this parameter. In this example, the gadget can access Blogger and Calendar data. A gadget can request data for a single scope or multiple scopes, as this example does.
         </dd> 
         <dt>
          <code>oauth_callback</code>
         </dt> 
         <dd>
          An optional parameter in the Authorization URL. The OAuth approval page redirects to this URL after the user has approved access to data. You should set this parameter to 
          <code>http://oauth.gmodules.com/gadgets/oauthcallback</code> to provide the best user experience when users install your gadget. That page provides a snippet of JavaScript that automatically closes the popup window. Alternatively, you can set this parameter to point to your own "approved" page, or you can simply leave the parameter blank.
         </dd> 
        </dl> 
        <h4>Accessing an authenticated Google Data API feed</h4> 
        <p>Once your gadget has authenticated the user, accessing a Google Data API is straightforward with the Google Data APIs JavaScript client library. For information on how to use the library in the OAuth Proxy, see <a href="https://developers.google.com/gdata/client-js#AuthenticationOAuthProxy">Using the JavaScript Client Library</a>.</p>
