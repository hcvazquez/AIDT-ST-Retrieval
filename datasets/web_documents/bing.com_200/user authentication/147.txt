In this tutorial, we'll be discussing token-based authentication systems and how they differ from traditional login systems.&nbsp;At the end of this tutorial, you'll see a&nbsp;fully working demo written in AngularJS and NodeJS.<br></p>

         <h2>Traditional Authentication Systems<br> </h2>
         <p>Before proceeding with a token-based authentication system, let's have a look at a traditional authentication system first.<br></p>
         <ol> 
          <li>The user provides a&nbsp;<strong>username </strong>and <strong>password</strong>&nbsp;in the&nbsp;login form and clicks&nbsp;<strong>Log&nbsp;In</strong>.</li> 
          <li>After the request is made, validate the user on the backend by querying in the database. If the request&nbsp;is valid, create a session by using the user information fetched from the&nbsp;database, and then return the session information in the response header in order to store the session ID in the browser.<br> </li> 
          <li>Provide the session information for accessing restricted endpoints in the application.<br> </li> 
          <li>If the session&nbsp;information is valid, let the user access specified end points, and respond with the&nbsp;rendered HTML content.</li> 
         </ol>
         <figure class="post_image">
          <img alt="" src="https://cms-assets.tutsplus.com/uploads/users/487/posts/22543/image/traditional-authentication-system-png.png">
         </figure>
         <p>Everything is fine until this point. The web application works well, and it is able to&nbsp;authenticate users so that they may&nbsp;access restricted endpoints; however, what happens&nbsp;when you want to develop another client, say for&nbsp;Android, for your application? Will you be able to use the current application to authenticate mobile clients and to serve restricted content? As it currently stands, no. There are two main reasons for this:</p>
         <ol> 
          <li>Sessions and cookies do not make sense for mobile applications. You cannot share sessions or cookies created on the server-side with mobile clients.</li> 
          <li>In the current application, the rendered HTML is&nbsp;returned. In a mobile client, you need something like JSON or XML to be included as the response.<br> </li> 
         </ol>
         <p>In this case, you need a client-independent application.<br></p>
         <h2>Token-Based Authentication<br> </h2>
         <p>In token-based authentication, cookies and sessions will not be used. A token will be used for authenticating a&nbsp;user for each request to the&nbsp;server. Let's redesign the&nbsp;first scenario with token-based authentication.<br></p>
         <p>It will use the following flow of control:</p>
         <ol> 
          <li>The user provides a&nbsp;<strong>username</strong> and <strong>password</strong> in the login form and clicks <strong>Log&nbsp;In</strong>.</li> 
          <li>After a request is&nbsp;made, validate the user on the backend by querying in the database. If the request&nbsp;is valid, create a token by using the user information fetched from the database, and then return that information in the response header so that we can&nbsp;store the&nbsp;token browser in&nbsp;local storage.<br> </li> 
          <li>Provide token information in every request header for accessing restricted endpoints in the&nbsp;application.<strong><br></strong> </li> 
          <li>If the&nbsp;token fetched from the request header information is valid, let the user access the specified end point, and respond with JSON or XML.<br> </li> 
         </ol>
         <p>In this case, we have no returned session or cookie, and we have not returned any HTML content. That means that&nbsp;we can use this architecture for any client for a specific application. You can see the architecture schema below:</p>
         <figure class="post_image">
          <img alt="" src="https://cms-assets.tutsplus.com/uploads/users/487/posts/22543/image/token-based-authentication-system-png.png">
         </figure>
         <p>So, what is this JWT?</p>
         <h2>JWT</h2>
         <p>JWT stands for&nbsp;<strong>JSON Web Token</strong>&nbsp;and&nbsp;is a token format used in authorization headers. This token helps you to design communication&nbsp;between two systems in a secure way. Let's rephrase JWT as the "bearer&nbsp;token" for the purposes&nbsp;of this tutorial. A bearer token consists of three parts: header, payload, and signature.</p>
         <ul> 
          <li>The header&nbsp;is the part of the token that&nbsp;keeps the token&nbsp;type and encryption method, which is also encrypted with base-64.</li> 
          <li>The payload includes the information.&nbsp;You can put any kind of data like user info, product info and so on, all of&nbsp;which is stored with base-64 encryption.</li> 
          <li>The signature consists of combinations of the header, payload, and secret key. The secret key must be kept securely on the server-side.</li> 
         </ul>
         <p>You can see the JWT schema and an example token below;</p>
         <figure class="post_image">
          <img alt="" src="https://cms-assets.tutsplus.com/uploads/users/487/posts/22543/image/jwt-schema-png.png">
         </figure>

         <h2>A Practical Example</h2>
         <p>After covering some basic information about token-based authentication, we can now proceed with a practical example. Take a look at the following schema, after which we'll analyze it in more detail:</p>
         <figure class="post_image">
          <img alt="" src="https://cms-assets.tutsplus.com/uploads/users/487/posts/22543/image/real-life-architecture-png.png">
         </figure>
         <ol> 
          <li>The requests are made by several clients such as a web application, a mobile client, etc., to the API for a specific purpose.</li> 
          <li>The requests are made to a service like <code><a href="https://api.yourexampleapp.com">https://api.yourexampleapp.com</a></code>. If lots of people use&nbsp;the application, multiple servers may be required to serve the requested operation.<br> </li> 
          <li>Here, the&nbsp;load balancer is used for balancing requests to best suit the application servers at the back-end. When you make a request to <code><a href="https://api.yourexampleapp.com">https://api.yourexampleapp.com</a></code>, first the load balancer will handle a request, and then it will redirect the client to a&nbsp;specific server.<br> </li> 
          <li>There is one application, and this application is deployed to several servers (server-1, server-2, ..., server-n). Whenever a request is made to <code><a href="https://api.yourexampleapp.com">https://api.yourexampleapp.com</a></code>, the back-end application will intercept the request header and extract token information from the authorization header. A database query will be made by using this token. If this token is&nbsp;valid and has the&nbsp;required permission to access the requested endpoint, it will continue. If not,&nbsp;it will return a 403 response code (which indicates a forbidden status).<br> </li> 
         </ol>
         <h2>Advantages<br> </h2>
         <p>Token-based authentication comes with several advantages that solve serious problems. Some of them are as follows:<br></p>
         <ul> 
          <li> <strong>Client Independent Services.</strong>&nbsp;In token-based authentication, a token is&nbsp;transferred via request headers, instead of keeping the authentication information in sessions or cookies. This means there is no state. You can send a request to the server from any type of client that can make HTTP requests.</li> 
          <li> <strong>CDN.</strong>&nbsp;In most current web applications, views are rendered on the back-end and HTML content is returned to the browser. Front-end logic depends on back-end code. There is no need to make such a dependency. This comes with several problems. For example,&nbsp;if you are working with a design agency that implements your front-end HTML, CSS, and JavaScript, you need to take that front-end code and migrate it into your back-end code in order to do some rendering or populating operations. After some time, your rendered HTML content will differ greatly&nbsp;from what the code agency implemented. In token-based authentication, you can develop a front-end project separately from the back-end code. Your back-end code will return a JSON&nbsp;response instead of rendered HTML, and you can put the&nbsp;minified, gzipped version of the front-end code&nbsp;into the CDN. When you go to your web page, HTML content will be served from the CDN, and page content will be populated by API services using the token in the authorization headers<br> </li> 
          <li> <strong>No Cookie-Session (or&nbsp;No CSRF).</strong>&nbsp;CSRF is a major problem in modern web security because&nbsp;it doesn't check&nbsp;whether a&nbsp;request source is trusted or not. To solve this&nbsp;problem, a token pool is used for sending that token on every form post. In token-based authentication, a token is used in authorization headers, and CSRF does not include that information.<br> </li> 
          <li> <strong>Persistent Token Store.</strong>&nbsp;When a session read, write, or&nbsp;delete operation is made in the application, it will make a file operation in the operating system's <code class="inline">temp</code> folder, at least for the first time. Let's say that you have multiple servers and a session is created on the first server. When you make another request and your request drops in another server, session information will not exist and will get an "unauthorized" response. I know, you can solve that with a&nbsp;sticky session. However, in token-based authentication, this case is solved naturally. There is no sticky session problem, because the request token is intercepted on every request on any server.<br> </li> 
         </ul>
         <p>Those are the most common advantages of token-based authentication and communication.&nbsp;That's the end of the theoretical and architectural talk about token-based authentication. Time for a practical example.</p>
         <h2>An Example Application<br> </h2>
         <p>You will see two applications to&nbsp;demonstrate token-based authentication:<br></p>
         <ol> 
          <li>token-based-auth-backend</li> 
          <li>token-based-auth-frontend<br> </li> 
         </ol>
         <p>In the back-end project, there will be service implementations, and service results will be in JSON format. There is no view returned in services. In the front-end project, there will be an AngularJS project for front-end HTML and then the front-end app will be populated&nbsp;by AngularJS&nbsp;services to make requests to the back-end services.<br></p>
         <h3>token-based-auth-backend<br> </h3>
         <p>In the back-end project, there are three main files:<br></p>
         <ul> 
          <li> <code class="inline">package.json</code>&nbsp;is for dependency management.</li> 
          <li> <code class="inline">models\User.js</code>&nbsp;contains a User model that will be used for making database operations about users.<br> </li> 
          <li> <code class="inline">server.js</code>&nbsp;is for project bootstrapping and&nbsp;request handling.<br> </li> 
         </ul>
         <p>That's it! This project is very simple, so that you can&nbsp;understand the main concept easily without doing a deep dive.</p>
         <pre class="brush: javascript noskimlinks noskimwords">{
    "name": "angular-restful-auth",
	"version": "0.0.1",
	"dependencies": {
		"express": "4.x",
		"body-parser": "~1.0.0",
		"morgan": "latest",
		"mongoose": "3.8.8",
		"jsonwebtoken": "0.4.0"
	},
	"engines": {
		"node": "&gt;=0.10.0"
	}
}</pre>
         <p><code class="inline">package.json</code>&nbsp;contains dependencies for the project: <code class="inline">express</code>&nbsp;for MVC, <code class="inline">body-parser</code><strong>&nbsp;</strong>for simulating post request handling in NodeJS, <code class="inline">morgan</code><strong>&nbsp;</strong>for request logging, <code class="inline">mongoose</code><strong> </strong>for our ORM&nbsp;framework to connect to MongoDB, and&nbsp;<code class="inline">jsonwebtoken</code>&nbsp;for creating JWT tokens by using our User model. There is also an attribute called <code class="inline">engines</code>&nbsp;that says that this project is made by using NodeJS version &gt;= 0.10.0. This is useful for PaaS services like Heroku. We will also cover that topic in another section.</p>
         <pre class="brush: javascript noskimlinks noskimwords">var mongoose     = require('mongoose');
var Schema       = mongoose.Scema;

var UserSchema   = new Schema({
    email: String,
    password: String,
    token: String
});

module.exports = mongoose.model('User', UserSchema);</pre>
         <p>We said that we would generate a token by using the&nbsp;user model payload. This model helps us to make user operations on MongoDB. In <code class="inline">User.js</code>, the user-schema is defined and the User model is&nbsp;created by using a mongoose model. This model is ready for database operations.</p>
         <p>Our dependencies are defined, and our user model is&nbsp;defined, so now let's combine all those to construct a service for handling specific requests.</p>
         <pre class="brush: javascript noskimlinks noskimwords">// Required Modules
var express    = require("express");
var morgan     = require("morgan");
var bodyParser = require("body-parser");
var jwt        = require("jsonwebtoken");
var mongoose   = require("mongoose");
var app        = express();</pre>
         <p>In NodeJS, you can include a module in your project by using <code class="inline">require</code>. First, we need to import&nbsp;the necessary modules into the project:</p>
         <pre class="brush: javascript noskimlinks noskimwords">var port = process.env.PORT || 3001;
var User     = require('./models/User');

// Connect to DB
mongoose.connect(process.env.MONGO_URL);</pre>
         <p>Our service will serve through a specific port. If any port variable is defined in the system environment variables, you can use that, or we have defined port <code class="inline">3001</code>. After that, the User model is included, and the database&nbsp;connection is&nbsp;established in order to do some user operations. Do not forget to define an environment variable<code class="inline">MONGO_URL</code>for the database connection URL.</p>
         <pre class="brush: javascript noskimlinks noskimwords">app.use(bodyParser.urlencoded({ extended: true }));
app.use(bodyParser.json());
app.use(morgan("dev"));
app.use(function(req, res, next) {
    res.setHeader('Access-Control-Allow-Origin', '*');
    res.setHeader('Access-Control-Allow-Methods', 'GET, POST');
    res.setHeader('Access-Control-Allow-Headers', 'X-Requested-With,content-type, Authorization');
    next();
});</pre>
         <p>In the above section, we've made&nbsp;some configurations for simulating an HTTP request handling in NodeJS by using Express. We are allowing requests to come from different domains in order to develop a client-independent system. If you do not allow this, you will trigger a CORS (Cross Origin Request Sharing) error in the&nbsp;web browser.</p>
         <ul> 
          <li> <code class="inline">Access-Control-Allow-Origin</code>&nbsp;allowed for all domains.</li> 
          <li>You can send&nbsp;<code class="inline">POST</code> and <code class="inline">GET</code> requests to this service.<br> </li> 
          <li> <code class="inline">X-Requested-With</code>&nbsp;and <code class="inline">content-type</code>&nbsp;headers are allowed.<br> </li> 
         </ul>
         <pre class="brush: javascript noskimlinks noskimwords">app.post('/authenticate', function(req, res) {
    User.findOne({email: req.body.email, password: req.body.password}, function(err, user) {
        if (err) {
            res.json({
                type: false,
                data: "Error occured: " + err
            });
        } else {
            if (user) {
               res.json({
                    type: true,
                    data: user,
                    token: user.token
                }); 
            } else {
                res.json({
                    type: false,
                    data: "Incorrect email/password"
                });    
            }
        }
    });
});</pre>
         <p>We have imported all of the&nbsp;required modules and defined our&nbsp;configuration, so now it's&nbsp;time to define request handlers. In the above code, whenever you make a <code class="inline">POST</code><strong>&nbsp;</strong>request to <code class="inline">/authenticate</code>&nbsp;with username&nbsp;and password, you will get a <code class="inline">JWT</code>&nbsp;token. First, the database&nbsp;query is processed by using a username and password. If a user exists, the user data will be returned with its token. But, what if there is no such user matching the username and/or password?</p>
         <pre class="brush: javascript noskimlinks noskimwords">app.post('/signin', function(req, res) {
    User.findOne({email: req.body.email, password: req.body.password}, function(err, user) {
        if (err) {
            res.json({
                type: false,
                data: "Error occured: " + err
            });
        } else {
            if (user) {
                res.json({
                    type: false,
                    data: "User already exists!"
                });
            } else {
                var userModel = new User();
                userModel.email = req.body.email;
                userModel.password = req.body.password;
                userModel.save(function(err, user) {
                    user.token = jwt.sign(user, process.env.JWT_SECRET);
                    user.save(function(err, user1) {
                        res.json({
                            type: true,
                            data: user1,
                            token: user1.token
                        });
                    });
                })
            }
        }
    });
});</pre>
         <p>When you make a <code class="inline">POST</code>&nbsp;request to <code class="inline">/signin</code> with username and&nbsp;password, a new user will be created by using posted user information. On the&nbsp;<code class="inline">19th</code> line, you can see that a&nbsp;new JSON token is generated by using the&nbsp;<code class="inline">jsonwebtoken</code>&nbsp;module, which has been assigned to the&nbsp;<code class="inline">jwt</code> variable. The authentication part is OK. What if we try to access a restricted endpoint? How can we manage to access that endpoint?</p>
         <pre class="brush: javascript noskimlinks noskimwords">app.get('/me', ensureAuthorized, function(req, res) {
    User.findOne({token: req.token}, function(err, user) {
        if (err) {
            res.json({
                type: false,
                data: "Error occured: " + err
            });
        } else {
            res.json({
                type: true,
                data: user
            });
        }
    });
});</pre>
         <p>When you make a&nbsp;<code class="inline">GET</code>&nbsp;request to&nbsp;<code class="inline">/me</code>, you will get the current user info, but in order to continue with the requested endpoint, the&nbsp;<code class="inline">ensureAuthorized</code> function will be executed.</p>
         <pre class="brush: javascript noskimlinks noskimwords">function ensureAuthorized(req, res, next) {
    var bearerToken;
    var bearerHeader = req.headers["authorization"];
    if (typeof bearerHeader !== 'undefined') {
        var bearer = bearerHeader.split(" ");
        bearerToken = bearer[1];
        req.token = bearerToken;
        next();
    } else {
        res.send(403);
    }
}</pre>
         <p>In this function, request headers are intercepted and the&nbsp;<code class="inline">authorization</code>&nbsp;header is extracted. If a&nbsp;bearer token exists in this header, that token is assigned to&nbsp;<code class="inline">req.token</code>&nbsp;in order to be&nbsp;used throughout the&nbsp;request, and the request can be continued by using&nbsp;<code class="inline">next()</code>. If a token does not exist, you will get a 403 (Forbidden) response. Let's go back to the&nbsp;handler&nbsp;<code class="inline">/me</code>, and use&nbsp;<code class="inline">req.token</code>&nbsp;to fetch user data with this token.&nbsp;Whenever you create a&nbsp;new user, a token is generated and saved in the user model in DB. Those tokens are unique.</p>
         <p>We have only three handlers for this&nbsp;simple project. After that, you will see;<br></p>
         <pre class="brush: javascript noskimlinks noskimwords">process.on('uncaughtException', function(err) {
    console.log(err);
});</pre>
         <p>The NodeJS&nbsp;app may crash if an error occurs. With the above code, that crash is prevented and an error log is printed in the console. And finally, we can start the server by using the&nbsp;following code snippet.&nbsp;</p>
         <pre class="brush: javascript noskimlinks noskimwords">// Start Server
app.listen(port, function () {
    console.log( "Express server listening on port " + port);
});</pre>
         <p>To sum up:</p>
         <ul> 
          <li>Modules are imported.</li> 
          <li>Configurations are made.<br> </li> 
          <li>Request handlers are defined.<br> </li> 
          <li>A middleware is defined in order to intercept restricted endpoints.<br> </li> 
          <li>The server is started.<br> </li> 
         </ul>
         <p>We are done with the back-end service. So that it can&nbsp;be used by multiple clients, you can deploy this simple server application to your servers, or maybe you can deploy in Heroku. There is a file called&nbsp;<code class="inline">Procfile</code>&nbsp;in the project's root folder. Let's deploy our service in Heroku.<br></p>
         <h4>Heroku Deployment<br> </h4>
         <p>You can clone the back-end project from this <a href="http://github.com/cubuzoa/token-based-auth-backend" target="_self">GitHub repository</a>.<br></p>
         <p>I will not be discussing&nbsp;how to create an app in Heroku; you can refer <a href="http://devcenter.heroku.com/articles/creating-apps" rel="external" target="_blank">to this article</a> for creating a Heroku app if you have not done this before. After you create your Heroku app, you can add a destination to your current project by using the following command:<br></p>
         <pre class="brush: bash noskimlinks noskimwords">git remote add heroku &lt;your_heroku_git_url&gt;</pre>
         <p>Now you have cloned a project and added a destination. After&nbsp;<code class="inline">git add</code>&nbsp;and&nbsp;<code class="inline">git commit</code>,&nbsp;you can push&nbsp;your code to Heroku by performing&nbsp;<code class="inline">git push heroku master</code>. When you successfully push a project,&nbsp;Heroku&nbsp;will perform the&nbsp;<code class="inline">npm install</code>&nbsp;command to download dependencies&nbsp;into the&nbsp;<code class="inline">temp</code> folder on Heroku. After that,&nbsp;it will start your application and&nbsp;you can access your service by using the HTTP protocol.</p>
         <h3>token-based-auth-frontend<br> </h3>
         <p>In the front-end project, you will see an AngularJS project. Here, I'll only&nbsp;mention the main sections in the front-end project, because AngularJS is not something that can be covered within a single&nbsp;tutorial.</p>
         <p>You can clone the project from this <a href="https://github.com/cubuzoa/token-based-auth-frontend" target="_self">GitHub repository</a>.&nbsp;In this project, you will see the following folder structure:</p>

         <p><code class="inline">ngStorage.js</code>&nbsp;is a library for AngularJS to manipulate local storage operations. Also, there is a main layout&nbsp;<code class="inline">index.html</code>&nbsp;and partials that extend the main layout under the&nbsp;<code class="inline">partials</code>&nbsp;folder.&nbsp;<code class="inline">controllers.js</code>&nbsp;is&nbsp;for defining our controller actions in the&nbsp;front-end.&nbsp;<code class="inline">services.js</code>&nbsp;is&nbsp;for&nbsp;making service&nbsp;requests to our service that I mentioned in the previous project.          <p>In the main layout HTML file, all of the required JavaScript files are included for AngularJS-related libraries, as well as&nbsp;our custom controller, service, and app file.</p>
         <pre class="brush: javascript noskimlinks noskimwords">'use strict';

/* Controllers */

angular.module('angularRestfulAuth')
    .controller('HomeCtrl', ['$rootScope', '$scope', '$location', '$localStorage', 'Main', function($rootScope, $scope, $location, $localStorage, Main) {

        $scope.signin = function() {
            var formData = {
                email: $scope.email,
                password: $scope.password
            }

            Main.signin(formData, function(res) {
                if (res.type == false) {
                    alert(res.data)    
                } else {
                    $localStorage.token = res.data.token;
                    window.location = "/";    
                }
            }, function() {
                $rootScope.error = 'Failed to signin';

        $scope.signup = function() {
            var formData = {

            Main.save(formData, function(res) {

                    alert(res.data)
                } else {
                    $localStorage.token = res.data.token;

         <p>In the above code, the&nbsp;<code class="inline">HomeCtrl</code>&nbsp;controller is defined and some required modules are&nbsp;injected like&nbsp;<code class="inline">$rootScope</code>&nbsp;and&nbsp;<code class="inline">$scope</code>. Dependency injection is one of the strongest properties of AngularJS.&nbsp;<code class="inline">$scope</code>&nbsp;is the bridge variable between controllers and views in AngularJS that means&nbsp;you can use&nbsp;<code class="inline">test</code>&nbsp;in view if you defined it in a specified controller like&nbsp;<code class="inline">$scope.test=....</code>&nbsp;</p>
         <p>In this&nbsp;controller, some utility functions are defined, such as:</p>
         <ul> 
          <li> <code class="inline">signin</code>&nbsp;to set up a sign-in button on the sign-in form</li> 
          <li> <code class="inline">signup</code>&nbsp;for sign-up form handling</li> 
          <li> <code class="inline">me</code>&nbsp;for&nbsp;assigning the Me button in the layout</li> 
         </ul>
         <p>In the main layout, in the main menu list, you can see the&nbsp;<code class="inline">data-ng-controller</code>&nbsp;attribute with a value&nbsp;<code class="inline">HomeCtrl</code>. That means that&nbsp;this menu <code class="inline">dom</code> element&nbsp;can&nbsp;share scope with&nbsp;<code class="inline">HomeCtrl</code>. When you click the sign-up button in the form, the sign-up function in the controller file will be executed, and in this function, the sign-up service is used from the <code class="inline">Main</code> service that is&nbsp;already injected in this controller.&nbsp;</p>
         <p>The main structure is&nbsp;<code class="inline">view -&gt; controller -&gt; service</code>. This service makes simple Ajax requests to the back-end in order to get specific data.</p>
         <pre class="brush: javascript noskimlinks noskimwords">'use strict';

angular.module('angularRestfulAuth')
    .factory('Main', ['$http', '$localStorage', function($http, $localStorage){
        var baseUrl = "your_service_url";
        function changeUser(user) {
            angular.extend(currentUser, user);
        }

        function urlBase64Decode(str) {
            var output = str.replace('-', '+').replace('_', '/');
            switch (output.length % 4) {
                case 0:
                    break;
                case 2:
                    output += '==';
                    break;
                case 3:
                    output += '=';
                    break;
                default:
                    throw 'Illegal base64url string!';
            }
            return window.atob(output);
        }

        function getUserFromToken() {
            var token = $localStorage.token;
            var user = {};
            if (typeof token !== 'undefined') {
                var encoded = token.split('.')[1];
                user = JSON.parse(urlBase64Decode(encoded));
            }
            return user;
        }

        var currentUser = getUserFromToken();

        return {
            save: function(data, success, error) {
                $http.post(baseUrl + '/signin', data).success(success).error(error)
            },
            signin: function(data, success, error) {
                $http.post(baseUrl + '/authenticate', data).success(success).error(error)
            },
            me: function(success, error) {
                $http.get(baseUrl + '/me').success(success).error(error)
            },
            logout: function(success) {
                changeUser({});
                delete $localStorage.token;
                success();
            }
        };
    }
]);</pre>
         <p>In the above code, you can see service functions like making requests for authenticating. In controller.js, you may have already realised that there are functions like&nbsp;<code class="inline">Main.me</code>. This&nbsp;<code class="inline">Main</code>&nbsp;service&nbsp;has been injected in the controller, and in the controller, the services belonging to this service are called directly.&nbsp;</p>
         <p>These&nbsp;functions are simply Ajax requests to our service which we deployed together. Do not forget to put the service URL in&nbsp;<code class="inline">baseUrl</code>&nbsp;in the above code. When you deploy your service to Heroku, you will get a service URL like&nbsp;<code class="inline">appname.herokuapp.com</code>. In the above code, you&nbsp;will set&nbsp;<code class="inline">var baseUrl = "appname.herokuapp.com"</code>.&nbsp;</p>
         <p>In the&nbsp;sign-up or sign-in part of the application, the bearer token responds to the&nbsp;request and this token is saved to local storage. Whenever you make a request to a service in the back-end, you need to put this token in the headers. You can do this by using AngularJS interceptors.</p>
         <pre class="brush: javascript noskimlinks noskimwords">$httpProvider.interceptors.push(['$q', '$location', '$localStorage', function($q, $location, $localStorage) {
            return {
                'request': function (config) {
                    config.headers = config.headers || {};
                    if ($localStorage.token) {
                        config.headers.Authorization = 'Bearer ' + $localStorage.token;
                    }
                    return config;
                },
                'responseError': function(response) {
                    if(response.status === 401 || response.status === 403) {
                        $location.path('/signin');
                    }
                    return $q.reject(response);
                }
            };
        }]);</pre>
         <p>In the above code, every request is intercepted and an authorization header and value are put&nbsp;in the headers.</p>
         <p>In the front-end project, we have some partial pages like&nbsp;<code class="inline">signin</code>,&nbsp;<code class="inline">signup</code>,&nbsp;<code class="inline">profile&nbsp;details</code>, and&nbsp;<code class="inline">vb</code>. These partial&nbsp;pages are related with specific controllers. You can see that relation in&nbsp;<code class="inline">app.js</code>:<br></p>
         <pre class="brush: javascript noskimlinks noskimwords">angular.module('angularRestfulAuth', [
    'ngStorage',
    'ngRoute'
])
.config(['$routeProvider', '$httpProvider', function ($routeProvider, $httpProvider) {

    $routeProvider.
        when('/', {
            templateUrl: 'partials/home.html',
            controller: 'HomeCtrl'
        }).
        when('/signin', {
            templateUrl: 'partials/signin.html',
            controller: 'HomeCtrl'
        }).
        when('/signup', {
            templateUrl: 'partials/signup.html',
            controller: 'HomeCtrl'
        }).
        when('/me', {
            templateUrl: 'partials/me.html',
            controller: 'HomeCtrl'
        }).
        otherwise({
            redirectTo: '/'
        });</pre>
         <p>As you can easily understand in the above code, when you go to <code class="inline">/</code>, the&nbsp;<code class="inline">home.html</code>&nbsp;page will be rendered.&nbsp;Another example: if you go to&nbsp;<code class="inline">/signup</code>,&nbsp;<code class="inline">signup.html</code>&nbsp;will be rendered. This&nbsp;rendering&nbsp;operation will be done in the browser, not on the server-side.</p>
         <h2>Conclusion</h2>
         <p>You can see how everything we discussed in this tutorial works into practice&nbsp;by checking out this&nbsp;<a href="http://token-based-auth.herokuapp.com/" target="_self">working demo</a>.<br></p>
         <p>Token-based authentication systems help you to construct an authentication/authorization system&nbsp;while you are developing client-independent&nbsp;services. By using this technology, you will just focus on your services (or APIs).&nbsp;</p>
         <p>The authentication/authorization part will be handled by the token-based authentication system as a layer in front of your services. You can access and use services from any client like web browsers, Android, iOS, or a desktop client.</p>
         <p>And if you're looking for ready-made solutions, check out the&nbsp;<a href="https://codecanyon.net/tags/authentication" target="_self">authentication scripts and apps</a>&nbsp;on Envato Market.<br></p>
