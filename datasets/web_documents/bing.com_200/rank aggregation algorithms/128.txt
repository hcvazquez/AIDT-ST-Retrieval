             <div class="post-text" itemprop="text"> 
              <p>As part of my graduate thesis (area: psychology) I have gathered preference data. The data includes approximately 50000 heads-up comparison between elementX and elementX. I have a total of 15 elements. The participant would be shown 36 comparisons in random order, of which he/she would then choose the preferred element. I need to be able to calculate a meaningful "interval" scale-value for each element in order to see if the actual preferences of each element can predict theoretically generated scale values using regression analysis.</p> 
              <p>The Data: 14 elements, forming a 14x14 matrix table 91 unique possible comparisons ((14*14-14)/2) = 50000 comparison (NOT evenly distributed across all comparisons)</p> 
              <p>The problem: Since showing the participant all 91 possible comparison would be too time consuming I've just presented each participant 36 comparisons. Moreover, each participant was not eligible for each element, the eligible elements was determined if the participant owned the element.</p> 
              <p>For example: ParticipantX owns element; 1,5,6,8 and 10. And would therefore only be shown comparisons that includes those elements.</p> 
              <p>Furthermore, as owning some elements were considered more rare, I made a priority list for each element, so that element1 &gt; element2 &gt; .... &gt; element14</p> 
              <p>For example: If the participant would be eligible for 50 possible comparisons (but restricted by the 36 comparison-cap) the comparisons shown to the participant would be based on the rarity order.</p> 
              <p>I'm afraid I destroyed the data by making this rarity order, since it resulted in some some elements being compared unequally to some elements; For example: of 1000 comparisons that element 1 was included in, 30% was with element3, 10% with element4 etc...</p> 
              <p>My questions: What method would you suggest would be the best for calculating "scale" values for each element? Currently I've calculated the basic probability that a given element has been choosen over another element. For example; element1 being chosen 95% of times to be preferred over any other element. However, I would prefer a more refined method to calculate the element values.</p> 
              <p>Thanks for your time, Chris, Finland.</p>

             <div class="post-text" itemprop="text"> 
              <p>Hopefully this isn't too late, but your data is still valuable using the Bradley-Terry model. This paper is a nice reference <a href="http://sites.stat.psu.edu/~dhunter/papers/bt.pdf" rel="noreferrer">MM Algorithms for generalized Bradley-Terry models</a>. Equation 4 is the one you want.</p> 
              <p>Brief explanation as I guess you are not a mathematition, and I'm being nice as I'd really like a copy of your data to test my implementation of this algorithm!</p> 
              <h1>Explanation of Bradley-Terry Model</h1> 
              <p>Basically Bradley-Terry assigns a value to each object, which is the answer you want - e.g. if you are asking "Which of these cars looks sexier", the value would measure sexiness. Then it says if you have two cars with sexinesses $s_i$ and $s_j$, the probability of someone choosing car $i$ over car $j$ is:</p> 
              <p>$$P(o_i\succ o_j)=\frac{e^{s_i}}{e^{s_i}-e^{s_j}}$$</p> 
              <p>That actually only depends on $s_j-s_i$ as follows:</p> 
              <p><img src="https://i.stack.imgur.com/WGYMM.png" alt="bt function"></p> 
              <p>So if they have the same sexiness the chances are 50-50 (random guessing), and if one has a higher sexiness it is more likely to be chosen.</p> 
              <p>This model is mostly plucked out of the air for its plausibility and nice mathematical properties. I don't know how realistic it really is. It is widely use though.</p> 
              <h2>Maximum Likelihood</h2> 
              <p>BT then uses something called <a href="https://en.wikipedia.org/wiki/Maximum_likelihood" rel="noreferrer">Maximum Likelihood</a> to estimate the values of $s_i$. Put in words, it finds the sexinesses of all the cars that would make the choices that you measured ("Dave says car A is sexier than car B" etc.) as likely as possible, according to our choice model above. So if people choose car A over car B a lot, then giving $s_A$ a lower sexiness than $s_B$ would mean that their choices (A &gt; B) are very unlikely. So those sexiness values are not a maximum likelihood estimate.</p> 
              <p>To actually find all the $s_i$'s you can use a generic optimisation method (e.g. <code>fminsearch</code> in matlab). Or there is an iterative algorithm that they claim is guaranteed to converge on a "unique maximum" (which I assume means global?).</p> 
              <p>That algorithm is equation 4 in the linked paper (plus normalisation).</p> 
              <h2>Algorithm.</h2> 
              <p>Basically you count the number of times each object is beaten by every other. Store that in $w_{ij}$. The total number of comparisons between any two objects is $N_{ij}=w_{ij}+w_{ji}$. Define new numbers $\gamma_i=e^{s_i}$. Then start with all $\gamma_i=0$ and iteratively update, looping repeatedly through the $i$'s. </p> 
              <p>$$\gamma_i\leftarrow W_{i}\left[\sum_{j\neq i}\frac{N_{ij}}{\gamma_{i}+\gamma_{j}}\right]^{-1}$$</p> 
              <p>You don't need to complete a whole loop before updating each $i$. By which I mean you don't need to store old values of $\gamma_i$. As soon as a new value is calculated you can use it in subsequent updates.</p> 
              <p>$W_i$ is the total number of wins by $i$: $\sum_{j}w_{ij}=W_{i}$</p> 
              <p>You also need to normalise $\gamma$ after each loop over it:</p> 
              <p>$$\gamma \leftarrow \frac{\gamma}{\mathrm{mean}(\gamma)}$$</p> 
              <h2>Regularisation.</h2> 
              <p>You probably won't need this since your data set is so large and dense. In some cases you need to regularise by adding "phantom" comparisons of every object with an extra virtual one. Apparently the standard method is to count $\lambda$ wins for the real objects vs virtual, and $\lambda$ losses. I've found $\lambda$=0.1 to work fine.</p> 
              <p>Please can you send me your data and expected results if you still have it? :-)</p>
