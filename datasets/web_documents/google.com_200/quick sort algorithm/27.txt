       <div class="entry-content"> 
        <p>I migrated from Octopress back to WordPress and all of my code snippets exploded, I’m in the process of fixing those, please bear with me. This post also has huge improvements detailed in the comments section <del>that makes me think this post warrants a revisit, I’ll link to it from here if so</del>. <strong>I’ve written a <a href="https://oli.me.uk/2014/12/17/revisiting-searching-javascript-arrays-with-a-binary-search/">new post</a>!</strong> This one includes benchmarks and generative testing (that all passes!). So you can be sure that it works and that any changes made do actually make it faster. This post is therefore <strong>deprecated</strong>. The repository for the new implementation can be found at <a href="https://github.com/Olical/binary-search">Olical/binary-search</a>.</p> 
        <hr> 
        <p>We are asking browsers to do more and more as they become more capable. I’m not sure if that’s a good thing or not, but that’s a heated flame war that should be saved for another day.</p> 
        <p>There may come a time in your browser (ab)use in which you need to search a ridiculous amount or values within a cat gif processing platform. Or maybe you need to search through all of the elements in a bloated DOM.</p> 
        <p>You will probably find that <code>indexOf</code> doesn’t quite cut it in those situations. In fact, if you <em>do</em> happen survive the inevitable violent explosion caused by your CPU trying to take the easy way out, you might want to try a JavaScript search method that is better suited for that much data.</p> 
        <p>This is where you have an excuse to suggest a binary search and blow everyone else’s minds.</p> 
        <p>
         <!-- more --></p> 
        <h2>What is it</h2> 
        <p>A binary search searches by splitting your array into smaller and smaller chunks until it finds your desired value. Unlike the normal <code>indexOf</code> which searches from left to right in a simple iteration. <a href="http://en.wikipedia.org/wiki/Binary_search_algorithm">The binary search Wikipedia article</a> explains it best (as always). There are a couple of downsides; It will be slower with smaller data sets (this needs proving) and the array you are searching <strong>needs to be sorted</strong>.</p> 
        <p>Because a binary search is <a href="http://en.wikipedia.org/wiki/Logarithmic_time#Logarithmic_time">O(log n)</a>, and not O(n) like <code>indexOf</code>, it’s great for large sets of data. I set up a little <a href="http://jsperf.com/binaryindexof-and-indexof">jsPerf for my version</a> of the JavaScript implementation; it is searching <strong>100,000</strong> numbers.</p> 
        <p>The source for my <a href="https://gist.github.com/Wolfy87/5734530">JavaScript binary search implementation</a> is currently held in a gist. The really cool thing about this is that it’s only 138 bytes when minified, that’s tiny enough to fit inside a tweet.</p>

/**
 * Performs a binary search on the host array. This method can either be
 * injected into Array.prototype or called with a specified scope like this:
 * binaryIndexOf.call(someArray, searchElement);
 *
 * @param {*} searchElement The item to search for within the array.
 * @return {Number} The index of the element which defaults to -1 when not found.
 */
function binaryIndexOf(searchElement) {
    'use strict';

    var minIndex = 0;
    var maxIndex = this.length - 1;
    var currentIndex;
    var currentElement;

    while (minIndex &lt;= maxIndex) {
        currentIndex = (minIndex + maxIndex) / 2 | 0;
        currentElement = this[currentIndex];

        if (currentElement &lt; searchElement) {
            minIndex = currentIndex + 1;
        }
        else if (currentElement &gt; searchElement) {
            maxIndex = currentIndex - 1;
        }
        else {
            return currentIndex;
        }
    }

    return -1;

        <p>Let me know what you think and if you have any improvements or suggestions. I hope this can help to destroy a bottleneck or two (or 100,000).</p> 
        <h2>Edit: Where should you stick it?</h2> 
        <p><a href="http://www.reddit.com/r/javascript/comments/1fx4od/searching_javascript_arrays_with_a_binary_search/caeo5is">doomslice over on reddit</a> suggested that I return the <a href="http://en.wikipedia.org/wiki/Two%27s_complement">twos compliment</a> in place of -1. As far as I can tell, this means returning the negative version of the last place that was checked. So if you wanted to insert a value and wanted to know where you should put it, you could run the function and use the returned number to splice the value into the array.</p> 
        <p>This way you can add items without ruining the order. You don’t exactly want to go re-sorting potentially thousands of values every time you add something. Here’s the modified function I came up with, as well as a working example. It demonstrates finding where to insert an element and then inserting it.</p> 
        <p><iframe width="100%" height="300" src="http://jsfiddle.net/Wolfy87/5dcWN/embedded/" allowfullscreen frameborder="0"></iframe></p> 
        <p>As you can see, because I use the bitwise NOT operator (<code>~</code>, also suggested by doomsice!) on the index during the splice, it will always insert the element in the right place, even if there is an identical value already there! Pretty cool. You do have to be careful though, now you are going to need to check for <code>index &lt; 0</code>, and not <code>index === -1</code> when you are looking for existence of a value.</p>

           <div id="dsq-comment-message-91" class="dsq-comment-message">
            <p>Should be ~currentIndex not ~maxIndex.</p> 
            <p>Example:<br> var a = [1, 2, 3];<br> a.binaryIndexOf(0);</p> 
            <p>With your current code, the above will give you 0. It should be -1 since 0 wasn’t in the array and needs to be inserted into the 0th index. </p> 
            <p>a.binaryIndexOf(1.5); will give you -1. It should be -2 since 1.5 should be inserted into the array at index 1.</p> 
            <p>The magnitude of the returned missing index should always be 1 greater than the actual insertion index otherwise you can’t decide between if something is missing in the array or needs to be inserted as the first element in the array.</p> 
            <p>In other words, if the missing element should be inserted at index 0 you should return -1. If it should be inserted at index 1, you need to return -2 and so on. If you don’t do this you confuse the case of “the array is missing the element which belongs at index 0” and “the array found the element at index 0”</p>
