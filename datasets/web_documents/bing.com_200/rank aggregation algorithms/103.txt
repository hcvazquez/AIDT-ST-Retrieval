                    <div id="yWiner"></div>
                    <span class="rendered_qtext">There are a number of dimensions you can look at to give you a sense of what will be a reasonable algorithm to start with, namely:<br>
                     <ul>
                      <li>Number of training examples</li>
                      <li>Dimensionality of the feature space</li>
                      <li>Do I expect the problem to be linearly separable?</li>
                      <li>Are features independent?</li>
                      <li>Are features expected to linearly dependent with the target variable? *EDIT: see my <span class="qlink_container"><a href="http://comment" rel="noopener nofollow" target="_blank" onclick="return Q.openUrl(this, 12553792);" class="external_link">comment</a></span> on what I mean by this</li>
                      <li>Is overfitting expected to be a problem?</li>
                      <li>What are the system's requirement in terms of speed/performance/memory usage...?</li>
                      <li>...</li>
                     </ul><br>This list may seem a bit daunting because there are many issues that are not straightforward to answer. The good news though is, that as many problems in life, you can address this question by following the <span class="qlink_container"><a href="/topic/Occams-Razor">Occam's Razor</a></span> principle: use the least complicated algorithm that can address your needs and only go for something more complicated if strictly necessary. <br><br><b><u>Logistic Regression</u></b><br><br>As a general rule of thumb, I would recommend to start with <span class="qlink_container"><a href="/topic/Logistic-Regression">Logistic Regression</a></span>. Logistic regression is a pretty well-behaved classification algorithm that can be trained as long as you expect your features to be roughly linear and the problem to be linearly separable. You can do some feature engineering to turn most non-linear features into linear pretty easily. It is also pretty robust to noise and you can avoid overfitting and even do feature selection by using l2 or l1 regularization. Logistic regression can also be used in Big Data scenarios since it is pretty efficient and can be distributed using, for example, ADMM (see <span class="qlink_container"><a href="https://web.stanford.edu/~boyd/papers/admm/logreg-l1/logreg.html" rel="noopener nofollow" target="_blank" onclick="return Q.openUrl(this, 12553792);" class="external_link" data-qt-tooltip="stanford.edu">logreg</a></span>). A final advantage of LR is that the output can be interpreted as a probability. This is something that comes as a nice side effect since you can use it, for example, for ranking instead of classification. <br><br>Even in a case where you would not expect Logistic Regression to work 100%, do yourself a favor and run a simple l2-regularized LR to come up with a baseline before you go into using "fancier" approaches.<br><br>Ok, so now that you have set your baseline with Logistic Regression, what should be your next step. I would basically recommend two possible directions: (1) SVM's, or (2) Tree Ensembles. If I knew nothing about your problem, I would definitely go for (2), but I will start with describing why SVM's might be something worth considering. <br><br><b><u>Support Vector Machines</u></b><br><br><span class="qlink_container"><a href="/topic/Support-Vector-Machines">Support Vector Machines</a></span> (SVMs) use a different loss function (Hinge) from LR. They are also interpreted differently (maximum-margin). However, in practice, an SVM with a linear kernel is not very different from a Logistic Regression (If you are curious, you can see how Andrew Ng derives SVMs from Logistic Regression in his <span class="qlink_container"><a href="https://d396qusza40orc.cloudfront.net/ml/docs/slides/Lecture12.pdf" rel="noopener nofollow" target="_blank" onclick="return Q.openUrl(this, 12553792);" class="external_link" data-qt-tooltip="d396qusza40orc.cloudfront.net">Coursera Machine Learning Course</a></span>). The main reason you would want to use an SVM instead of a Logistic Regression is because your problem might not be linearly separable. In that case, you will have to use an SVM with a non linear kernel (e.g. RBF). The truth is that a Logistic Regression can also be used with a different kernel, but at that point you might be better off going for SVMs for practical reasons. Another related reason to use SVMs is if you are in a highly dimensional space. For example, SVMs have been reported to work better for text classification. <br><br>Unfortunately, the major downside of SVMs is that they can be painfully inefficient to train. So, I would not recommend them for any problem where you have many training examples. I would actually go even further and say that I would not recommend SVMs for most "industry scale" applications. Anything beyond a toy/lab problem might be better approached with a different algorithm.<br><br><b><u>Tree Ensembles</u></b><br><br>This gets me to the third family of algorithms: Tree Ensembles. This basically covers two distinct algorithms: <span class="qlink_container"><a href="/topic/Random-Forests-algorithm">Random Forests (algorithm)</a></span> and <span class="qlink_container"><a href="/topic/Gradient-Boosted-Trees">Gradient Boosted Trees</a></span>. I will talk about the differences later, but for now let me treat them as one for the purpose of comparing them to Logistic Regression.<br><br>Tree Ensembles have different advantages over LR. One main advantage is that they do not expect linear features or even features that interact linearly. Something I did not mention in LR is that it can hardly handle categorical (binary) features. Tree Ensembles, because they are nothing more than a bunch of Decision Trees combined, can handle this very well. The other main advantage is that, because of how they are constructed (using bagging or boosting) these algorithms handle very well high dimensional spaces as well as large number of training examples.<br><br>As for the difference between Random Forests (RF) and Gradient Boosted Decision Trees (GBDT), I won't go into many details, but one easy way to understand it is that GBDTs will usually perform better, but they are harder to get right. More concretely, GBDTs have more hyper-parameters to tune and are also more prone to overfitting. RFs can almost work "out of the box" and that is one reason why they are very popular.<br><br><b><u>Deep Learning</u></b><br><br>Last but not least, this answer would not be complete without at least a minor reference to <span class="qlink_container"><a href="/topic/Deep-Learning">Deep Learning</a></span>. I would definitely not recommend this approach as a general-purpose technique for classification. But, you might probably have heard how well these methods perform in some cases such as image classification. If you have gone through the previous steps and still feel you can squeeze something out of your problem, you might want to use a Deep Learning approach. The truth is that if you use an open source implementation such as <span class="qlink_container"><a href="http://deeplearning.net/software/theano/" rel="noopener nofollow" target="_blank" onclick="return Q.openUrl(this, 12553792);" class="external_link" data-qt-tooltip="deeplearning.net">Theano</a></span>, you can get an idea of how some of these approaches perform in your dataset pretty quickly. <br><br><b><u>Summary</u></b><br><br>So, recapping, start with something simple like Logistic Regression to set a baseline and only make it more complicated if you need to. At that point, tree ensembles, and in particular Random Forests since they are easy to tune, might be the right way to go. If you feel there is still room for improvement, try GBDT or get even fancier and go for Deep Learning.<br><br>You can also take a look at the <span class="qlink_container"><a href="https://www.kaggle.com/competitions" rel="noopener nofollow" target="_blank" onclick="return Q.openUrl(this, 12553792);" class="external_link" data-qt-tooltip="kaggle.com">Kaggle Competitions</a></span>. If you search for the keyword "classification" and select those that are completed, you will get a good sense of what people used to win competitions that might be similar to your problem at hand. At that point you will probably realize that using an ensemble is always likely to make things better. The only problem with ensembles, of course, is that they require to maintain all the independent methods working in parallel. That might be your final step to get as fancy as it gets.</span>

                    <span class="rendered_qtext">There were already great answers given by <span class="qlink_container"><a href="/profile/Edwin-Chen-1">Edwin Chen</a></span> and <span class="qlink_container"><a href="/profile/Xavier-Amatriain">Xavier Amatriain</a></span>. If I had one thing to add it would be the capability of online-learning which is especially useful for out-of-core learning -- working with data that doesn't fit into memory. <br><br>For example, stochastic gradient descent-based classifiers are great for online learning where you can update your classifier on the fly when new data arrives (e.g., using perceptrons, ADALINE, or better: logistic regression). But they are also useful if you want to process large datasets by reading them in incrementally from disk instance by instance or in mini-batches.<br><br>There is also the "lazy vs. eager learning" discussion:<br>Let's consider the k-nearest neighbor classifier as a popular example of lazy learners. They may be useful for medium to large low-dimensional datasets where you want to classify some samples without an expensive learning step. However, the disadvantage is that it can easily become computationally expensive if you want to classify a large number of instances (since you don't have a simple function like in logistic regression) and have to keep all your training data around to classify new samples.</span>

                    <span class="rendered_qtext"><b>Instance based learning techniques (e.g. nearest neighbor)</b>
                     <ul>
                      <li>Take up a lot of memory to run (storing all the instances)</li>
                      <li>Work well for a small number of dimensions, but not a high number of dimensions</li>
                     </ul><br><b>Decision trees and divide-and-conquer algorithms</b><br>
                     <ul>
                      <li>May overfit data </li>
                      <li>May get stuck in local minima so need ensembles to help reduce the variance</li>
                     </ul><br><b>SVM</b><br>
                     <ul>
                      <li>Picking/finding the right kernel can be a challenge</li>
                      <li>Results/output are incomprehensible</li>
                      <li>No standardized way for dealing with multi-class problems; fundamentally a binary classifier</li>
                     </ul><br><b>Neural networks</b><br>
                     <ul>
                      <li>Picking the correct topology is difficult</li>
                      <li>Training takes a long time/requires a lot of data</li>
                      <li>Output/issues are incomprehensible</li>
                     </ul><br><b>Logistic regression</b><br>
                     <ul>
                      <li>Limited expressive power</li>
                      <li>Fundamentally a binary classifier</li>
                      <li>Hard to make incremental </li>
                     </ul><br><b>Naive Bayes</b><br>
                     <ul>
                      <li>Very simple representation doesn't allow for rich hypotheses</li>
                      <li>Assumption of independence of attributes is too constraining (although, see <span class="qlink_container"><a href="http://web.cs.ucdavis.edu/~vemuri/classes/ecs271/Bayesian.pdf" rel="noopener nofollow" target="_blank" onclick="return Q.openUrl(this, 9546852);" class="external_link" data-qt-tooltip="ucdavis.edu">Domingos' paper on why it's not so bad</a></span>)</li>
                     </ul><br><b>Bayes networks/Graphical models</b><br>
                     <ul>
                      <li>Determining the topology of dependence is difficult</li>
                     </ul></span>
