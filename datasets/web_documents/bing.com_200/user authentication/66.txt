   <h1 id="firstHeading" class="firstHeading" lang="en">Security Support Provider Interface</h1> 
   <div id="bodyContent" class="mw-body-content"> 
    <div id="siteSub" class="noprint">
     From Wikipedia, the free encyclopedia
    </div> 
    <div id="contentSub"></div> 
    <div id="jump-to-nav" class="mw-jump">
      Jump to: 
     <a href="#mw-head">navigation</a>,

    <div id="mw-content-text" lang="en" dir="ltr" class="mw-content-ltr">
     <div class="mw-parser-output">
      <p><b>Security Support Provider Interface</b> (<b>SSPI</b>) is a <a href="/wiki/Win32" class="mw-redirect" title="Win32">Win32</a> <a href="/wiki/Application_programming_interface" title="Application programming interface">API</a> used by Microsoft Windows systems to perform a variety of security-related operations such as <a href="/wiki/Authentication" title="Authentication">authentication</a>.</p> 
      <p>SSPI functions as a common interface to several Security Support Providers (SSPs):<sup id="cite_ref-1" class="reference"><a href="#cite_note-1">[1]</a></sup> A Security Support Provider is a <a href="/wiki/Dynamic-link_library" title="Dynamic-link library">dynamic-link library</a> (DLL) that makes one or more security packages available to applications.</p> 
      <p></p>

      <h2><span class="mw-headline" id="Windows_SSPs">Windows SSPs</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Security_Support_Provider_Interface&amp;action=edit&amp;section=1" title="Edit section: Windows SSPs">edit</a><span class="mw-editsection-bracket">]</span></span></h2> 
      <p>The following SSPs are installed with Windows:</p> 
      <ul> 
       <li><a href="/wiki/NTLMSSP" title="NTLMSSP">NTLM</a> (Introduced in <a href="/wiki/Windows_NT_3.51" title="Windows NT 3.51">Windows NT 3.51</a>) (msv1_0.dll) - Provides <a href="/wiki/NTLM" class="mw-redirect" title="NTLM">NTLM</a> challenge/response authentication for client-server domains prior to <a href="/wiki/Windows_2000" title="Windows 2000">Windows 2000</a> and for non-domain authentication (<a href="/wiki/Server_Message_Block" title="Server Message Block">SMB</a>/CIFS).<sup id="cite_ref-2" class="reference"><a href="#cite_note-2">[2]</a></sup></li> 
       <li><a href="/wiki/Kerberos_(protocol)" title="Kerberos (protocol)">Kerberos</a> (Introduced in <a href="/wiki/Windows_2000" title="Windows 2000">Windows 2000</a> and updated in <a href="/wiki/Windows_Vista" title="Windows Vista">Windows Vista</a> to support <a href="/wiki/Advanced_Encryption_Standard" title="Advanced Encryption Standard">AES</a>) <sup id="cite_ref-3" class="reference"><a href="#cite_note-3">[3]</a></sup> (kerberos.dll) - Preferred for mutual client-server domain authentication in Windows 2000 and later.<sup id="cite_ref-4" class="reference"><a href="#cite_note-4">[4]</a></sup></li> 
       <li><a href="/wiki/SPNEGO" title="SPNEGO">Negotiate</a> (Introduced in Windows 2000) (secur32.dll) - Selects Kerberos and if not available, NTLM protocol. Negotiate SSP provides <a href="/wiki/Single_sign-on" title="Single sign-on">single sign-on</a> capability, sometimes referred to as <a href="/wiki/Integrated_Windows_Authentication" title="Integrated Windows Authentication">Integrated Windows Authentication</a> (especially in the context of IIS).<sup id="cite_ref-5" class="reference"><a href="#cite_note-5">[5]</a></sup> On <a href="/wiki/Windows_7" title="Windows 7">Windows 7</a> and later, NEGOExts is introduced which negotiates the use of installed custom SSPs which are supported on the client and server for authentication.</li> 
       <li>Secure Channel (aka SChannel) - Introduced in <a href="/wiki/Windows_2000" title="Windows 2000">Windows 2000</a> and updated in <a href="/wiki/Windows_Vista" title="Windows Vista">Windows Vista</a> to support stronger AES encryption and <a href="/wiki/Elliptic_curve_cryptography" title="Elliptic curve cryptography">ECC</a> <sup id="cite_ref-6" class="reference"><a href="#cite_note-6">[6]</a></sup> This provider uses SSL/TLS records to encrypt data payloads. (schannel.dll)</li> 
       <li><a href="/wiki/Private_Communications_Technology" title="Private Communications Technology">PCT</a> (obsolete) and Microsoft's implementation of <a href="/wiki/TLS/SSL" class="mw-redirect" title="TLS/SSL">TLS/SSL</a> - <a href="/wiki/Public_key_cryptography" class="mw-redirect" title="Public key cryptography">Public key cryptography</a> SSP that provides encryption and secure communication for authenticating clients and servers over the internet.<sup id="cite_ref-7" class="reference"><a href="#cite_note-7">[7]</a></sup> Updated in Windows 7 to support TLS 1.2.</li> 
       <li><a href="/wiki/Digest_access_authentication" title="Digest access authentication">Digest SSP</a> (Introduced in <a href="/wiki/Windows_XP" title="Windows XP">Windows XP</a>) (wdigest.dll) - Provides challenge/response based HTTP and <a href="/wiki/Simple_Authentication_and_Security_Layer" title="Simple Authentication and Security Layer">SASL</a> authentication between Windows and non-Windows systems where Kerberos is not available.<sup id="cite_ref-8" class="reference"><a href="#cite_note-8">[8]</a></sup></li> 
       <li>Credential (CredSSP) (Introduced in <a href="/wiki/Windows_Vista" title="Windows Vista">Windows Vista</a> and available on Windows XP SP3) (credssp.dll) - Provides SSO and <a href="/wiki/Network_Level_Authentication" title="Network Level Authentication">Network Level Authentication</a> for <a href="/wiki/Remote_Desktop_Services" title="Remote Desktop Services">Remote Desktop Services</a>.<sup id="cite_ref-9" class="reference"><a href="#cite_note-9">[9]</a></sup></li> 
       <li>Distributed Password Authentication (DPA) - (Introduced in Windows 2000) (msapsspc.dll) - Provides internet authentication using <a href="/wiki/Public_key_certificate" title="Public key certificate">digital certificates</a>.<sup id="cite_ref-10" class="reference"><a href="#cite_note-10">[10]</a></sup></li> 
       <li>Public Key Cryptography User-to-User (PKU2U) (Introduced in <a href="/wiki/Windows_7" title="Windows 7">Windows 7</a>) (pku2u.dll) - Provides peer-to-peer authentication using digital certificates between systems that are not part of a domain.</li> 
      </ul> 
      <h2><span class="mw-headline" id="Comparison">Comparison</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Security_Support_Provider_Interface&amp;action=edit&amp;section=2" title="Edit section: Comparison">edit</a><span class="mw-editsection-bracket">]</span></span></h2> 
      <p>SSPI is a proprietary variant of <a href="/wiki/Generic_Security_Services_Application_Program_Interface" title="Generic Security Services Application Program Interface">GSSAPI</a> with extensions and very Windows-specific data types. It shipped with <a href="/wiki/Windows_NT_3.51" title="Windows NT 3.51">Windows NT 3.51</a> and <a href="/wiki/Windows_95" title="Windows 95">Windows 95</a> with the <a href="/wiki/NTLM" class="mw-redirect" title="NTLM">NT LAN Manager</a> <a href="/wiki/NTLMSSP" title="NTLMSSP">Security Support Provider</a> (<a href="/wiki/NTLMSSP" title="NTLMSSP">NTLMSSP</a>). For Windows 2000, an implementation of Kerberos 5 was added, using token formats conforming to the official protocol standard <a class="external mw-magiclink-rfc" rel="nofollow" href="//tools.ietf.org/html/rfc1964">RFC 1964</a> (The Kerberos 5 GSSAPI mechanism) and providing wire-level interoperability with Kerberos 5 implementations from other vendors.</p> 
      <p>The tokens generated and accepted by the SSPI are mostly compatible with the GSS-API so an SSPI client on Windows may be able to authenticate with a GSS-API server on Unix depending on the specific circumstances. One significant shortcoming of SSPI is its lack of <abbr title="Channel binding is a way to cryptographically bind end-to-end authentication at the application layer to a secure channel at a lower layer. This cryptographic binding is a way to eliminate [[man-in-the-middle attack]]s in that secure channel. It is particularly useful to applications that intend to rely on TLS or IPsec for session/transport security. Channel bindings also stimulate the development of APIs for IPsec and an unauthenticated mode of IPsec.">channel bindings</abbr>, which makes some GSSAPI interoperability impossible.</p> 
      <p>Another fundamental difference between the <a href="/wiki/Internet_Engineering_Task_Force" title="Internet Engineering Task Force">IETF</a>-defined GSSAPI and Microsoft's SSPI is the concept of "<a href="/wiki/Access_token#Types_of_tokens" title="Access token">impersonation</a>". In this model, a server can switch to and operate with the <i>full</i> privileges of the authenticated client, so that the operating system performs all <a href="/wiki/Access_control" title="Access control">access control</a> checks, e.g. when opening new files. Whether these are less privileges or more privileges than that of the original service account depends entirely on which client connects/authenticates. In the traditional (GSSAPI) model, when a server runs under a service account, it cannot elevate its privileges, and has to perform access control in a client-specific and application-specific fashion. The obvious negative security implications of the impersonation concept are prevented in Windows Vista by restricting impersonation to selected service accounts.<sup id="cite_ref-11" class="reference"><a href="#cite_note-11">[11]</a></sup> Impersonation can be implemented in a Unix/Linux model using the seteuid or related system calls. While this means an unprivileged process cannot elevate its privileges, it also means that to take advantage of impersonation the process must run as root (or another process with the CAP_SETUID capability).</p>
