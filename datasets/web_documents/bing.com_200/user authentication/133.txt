       <h1 id="authenticating-users" data-magellan-target="authenticating-users">Authenticating users</h1> 
       <p class="intro">Application security is very important so Pusher provides a mechanism for authenticating a user’s access to a channel at the point of subscription.<br> To do so you must expose an authentication endpoint from your own servers that is called by the Pusher WebSocket libraries. Your server can then respond to either allow or deny the subscription.</p> 
       <p>This can be used both to restrict access to <a href="/docs/private_channels">private channels</a>, and in the case of <a href="/docs/presence_channels">presence channels</a> notify subscribers of who else is also subscribed via <a href="/docs/client_api_guide/client_presence_events">presence events</a>.</p> 
       <ul class="toc"> 
        <li><a href="#authEndpoint">Setting the channel authentication endpoint</a></li> 
        <li><a href="#authTransport">Setting the channel authentication transport (JS only)</a></li> 
        <li><a href="#implementing_endpoints">Implementing authentication endpoints</a> 
         <ul> 
          <li><a href="#implementing_private_endpoints">Private channel</a></li> 
          <li><a href="#implementing_presence_endpoints">Presence channel</a></li> 
          <li><a href="#jsonp_auth_endpoints">JSONp authentication endpoints (JS only)</a></li> 
         </ul> </li> 
        <li><a href="#batching-auth-requests">Batching auth requests</a></li> 
        <li><a href="#authentication_process">The authentication process</a></li> 
       </ul> 
       <h2 id="authEndpoint"><a href="#authEndpoint">Setting the Channel Authentication endpoint</a></h2> 
       <p>The destination of the authentication request can be configured.</p> 
       <article data-lang="js" data-group-id="auth_endpoint"> 
        <div data-lang="js" class="highlighter-rouge">
         <pre class="highlight"><code>new Pusher('app_key', { authEndpoint: '/pusher_auth.php' });</code></pre> 
        </div> 
        <p>The default value for this is: <code class="highlighter-rouge">/pusher/auth</code></p> 
        <p>If the authentication endpoint is protected by a CSRF filter, then you can pass in a CSRF token via the <code class="highlighter-rouge">auth</code> hash under <code class="highlighter-rouge">headers</code>.</p> 
        <div data-lang="js" class="highlighter-rouge">
         <pre class="highlight"><code>var pusher = new Pusher('app_key', {
  authEndpoint: '/pusher_auth.php',
  auth: {
    headers: {
      'X-CSRF-Token': "SOME_CSRF_TOKEN"
    }
  }
});</code></pre> 
        </div> 
        <p>Note that you should change the name of the CSRF token key to the convention you prefer.</p> 
       </article> 
       <article data-lang="ruby"> 
        <p>As an example, in Rails, you can inject the CSRF token into Javacript like this using ERB</p> 
        <div data-lang="erb" class="highlighter-rouge">
         <pre class="highlight"><code>&lt;script&gt;
  var pusher = new Pusher('app_key', {
    authEndpoint: '/pusher/auth',
    auth: {
      headers: {
        'X-CSRF-Token': "&lt;%= form_authenticity_token %&gt;"
      }
    }
  });
&lt;/script&gt;</code></pre> 
        </div> 
       </article> 
       <article data-lang="ios" data-group-id="auth_endpoint"> 
        <p>In order to connect to a private or presence channel using libPusher, you first need to configure your server authorisation URL.</p> 
        <div data-lang="objective-c" class="highlighter-rouge">
         <pre class="highlight"><code>_client.authorizationURL = [NSURL URLWithString:@"http://www.yourserver.com/authorise"];</code></pre> 
        </div> 
        <p>When you attempt to connect to a private or presence channel, libPusher will make a form-encoded POST request to the above URL, passing along the <code class="highlighter-rouge">socket_id</code> and <code class="highlighter-rouge">channel_name</code> as parameters. Prior to sending the request, the Pusher delegate will be notified, passing in the NSMutableURLRequest instance that will be sent.</p> 
        <p>Its up to you to configure the request to handle whatever authentication mechanism you are using. In this example, we set a custom header with a token which the server will use to authenticate the user before proceeding with authorisation.</p> 
        <div data-lang="objective-c" class="highlighter-rouge">
         <pre class="highlight"><code>- (void)pusher:(PTPusher *)pusher willAuthorizeChannel:(PTPusherChannel *)channel withRequest:(NSMutableURLRequest *)request
{
  [request setValue:@"some-authentication-token" forHTTPHeaderField:@"X-MyCustom-AuthTokenHeader"];
}</code></pre> 
        </div> 
       </article> 
       <article data-lang="java" data-group-id="auth_endpoint"> 
        <div data-lang="java" class="highlighter-rouge">
         <pre class="highlight"><code>HttpAuthorizer authorizer = new HttpAuthorizer("http://example.com/some_auth_endpoint");
PusherOptions options = new PusherOptions().setAuthorizer(authorizer);
Pusher pusher = new Pusher(YOUR_APP_KEY, options);</code></pre> 
        </div> 
       </article> 
       <h2 id="authTransport"><a href="#authTransport">Setting the Channel Authentication transport</a></h2> 
       <p class="alert alert--information">Channel authentication transport is only applicable to the Pusher JavaScript library.</p> 
       <p>The authentication transport can be configured by passing an option to the <code class="highlighter-rouge">Pusher</code> constructor.</p> 
       <div data-lang="js" class="highlighter-rouge">
        <pre class="highlight"><code>new Pusher('app_key', { authTransport: 'jsonp' });</code></pre> 
       </div> 
       <p>The default value for this is: <code class="highlighter-rouge">ajax</code>.</p> 
       <h3 id="ajax_authentication"><a href="#ajax_authentication">‘ajax’ authentication transport</a></h3> 
       <p>The HTTP <code class="highlighter-rouge">POST</code> request that is made to the authentication endpoint when a subscription takes place contains the following request parameters:</p> 
       <ul> 
        <li><code class="highlighter-rouge">socket_id</code> 
         <ul> 
          <li>A unique identifier for the specific client connection to Pusher</li> 
         </ul> </li> 
        <li><code class="highlighter-rouge">channel_name</code> 
         <ul> 
          <li>The name of the channel being subscribed to</li> 
         </ul> </li> 
       </ul> 
       <h3 id="jsonp_authentication"><a href="#jsonp_authentication">‘jsonp’ authentication transport</a></h3> 
       <p><a href="http://en.wikipedia.org/wiki/JSON#JSONP">JSONp</a> authentication can be used when the authentication endpoint is on a different domain to the web application. An additional parameter called <code class="highlighter-rouge">callback</code> will also be passed identifying the JavaScript function to be called from the script generated by the authentication endpoint.</p> 
       <p><em>Note: To use JSONp support, you must use Pusher’s client version 1.6 or above.</em></p> 
       <p>As described above you need to set the <code class="highlighter-rouge">authTransport</code> Pusher constructor option to <code class="highlighter-rouge">jsonp</code> and optionally pass <code class="highlighter-rouge">authEndpoint</code> value.</p> 
       <p>When using JSONp the authentication the following parameters will be passed as query parameters (a <code class="highlighter-rouge">GET</code> request):</p> 
       <ul> 
        <li><code class="highlighter-rouge">socket_id</code> 
         <ul> 
          <li>A unique identifier for the specific client connection to Pusher</li> 
         </ul> </li> 
        <li><code class="highlighter-rouge">channel_name</code> 
         <ul> 
          <li>The name of the channel being subscribed to</li> 
         </ul> </li> 
        <li><code class="highlighter-rouge">callback</code> 
         <ul> 
          <li>The name of the function to be called in the script to be generated by the authentication endpoint</li> 
         </ul> </li> 
       </ul> 
       <p>As discussed, the duty of the server is to check the user has permission to subscribe to the supplied channel. If the user does have permission then an authentication HTTP response must be returned in a JSON format with an authentication signature. We have a number of <a href="/docs/rest_libraries">server libraries</a> that make doing this really simple.</p> 
       <p><em>Note: If you don’t want to use one of the existing libraries, or there isn’t one in the technology you want to use, please see <a href="/docs/auth_signatures">authenticating signatures</a> or <a href="/support">get in touch</a>.</em></p> 
       <h2 id="implementing_endpoints"><a href="#implementing_endpoints">Implementing authentication endpoints</a></h2> 
       <p>The following sections show how this is achieved for the different types of channels.</p> 
       <h3 id="general-notes"><a href="#general-notes">General notes</a></h3> 
       <p>In all cases, the format of the response is very similar:</p> 
       <ul> 
        <li><strong>Unsuccessful</strong> responses from an authentication endpoint should serve a <code class="highlighter-rouge">403 Forbidden</code> HTTP status.</li> 
        <li><strong>Successful</strong> responses from an authentication endpoint should carry a <code class="highlighter-rouge">200 OK</code> HTTP status and a body of the form</li> 
       </ul> 
       <div class="highlighter-rouge">
        <pre class="highlight"><code><span class="w"> </span><span class="p">{</span><span class="w"> </span><span class="nt">"auth"</span><span class="p">:</span><span class="w"> </span><span class="err">$AUTHORIZATION_STRING</span><span class="w"> </span><span class="p">}</span><span class="w"> </span></code></pre> 
       </div> 
       <p><em>Details of <code class="highlighter-rouge">$AUTHORIZATION_STRING</code> described below.</em></p> 
       <h3 id="types-of-channel"><a href="#types-of-channel">Types of channel</a></h3> 
       <ul> 
        <li><strong>Private channels</strong> - used to control access to a source of information</li> 
        <li><strong>Presence channels</strong> - used to control access, but also allows info about the user to be distributed to other users.</li> 
       </ul>

       <h3 id="jsonp_auth_endpoints"><a href="#jsonp_auth_endpoints">JSONp Authentication endpoints</a></h3> 
       <p class="alert alert--information">JSONp auth endpoints are only applicable to the Pusher JavaScript library.</p> 
       <p>As discussed above the <code class="highlighter-rouge">authTransport</code> should be set to <code class="highlighter-rouge">jsonp</code> in order to use JSONp authentication. If JSONp authentication is being used then it is highly likely that the endpoint is on a different domain so the <code class="highlighter-rouge">authEndpoint</code> should also be set.</p> 
       <div data-lang="html" class="highlighter-rouge">
        <pre class="highlight"><code>&lt;script src="//js.pusher.com/4.1/pusher.min.js"&gt;&lt;/script&gt;

    authTransport: 'jsonp',

&lt;/script&gt;</code></pre>

       <p>Then you would subscribe to a private or presence channel, as needed.</p> 
       <div data-lang="html" class="highlighter-rouge">
        <pre class="highlight"><code>&lt;script&gt;

    alert(data.greeting);
  });

       <p>As demonstrated in the sequence diagram the server would then receive a call. Authentication works in much the same way as it does with an AJAX request. The function to be called is identified by the <code class="highlighter-rouge">callback</code> parameter in the query string. Because the JavaScript code returned in the response is executed by the browser, the response needs to have a content type of <code class="highlighter-rouge">application/javascript</code>.</p>

       <h2 id="batching-auth-requests"><a href="#batching-auth-requests">Batching auth requests (aka multi-auth)</a></h2> 
       <p>Currently, pusher-js itself does not support authenticating multiple channels in one HTTP request. However, thanks to <a href="https://github.com/dirkbonhomme/">Dirk Bonhomme</a> you can use the <a href="https://github.com/dirkbonhomme/pusher-js-auth">pusher-js-auth</a> plugin that buffers subscription requests and sends auth requests to your endpoint in batches.</p> 
       <h2 id="authentication_process"><a href="#authentication_process">The Authentication Process</a></h2> 
       <p>The basic mechanism for doing this is broadly the same for both scenarios. The following sequence diagram shows the signing process:</p> 
       <p><img alt="Auth Process" src="/assets/docs/private_channel_auth_process-7937849b2eb38e34e43823f54e046b5d1bed03a8a1da1e05cd7ffedcdd94795c.png"></p> 
       <ol> 
        <li>When a new instance of the <code class="highlighter-rouge">Pusher</code> object is created a new WebSocket object is created.</li> 
        <li>The WebSocket object connects to the Pusher WebSocket endpoint.</li> 
        <li> <p>Once the connection has been established a universally unique <code class="highlighter-rouge">socket_id</code> is returned to the Pusher JavaScript library.</p> </li> 
        <li>A subscription is made to a <a href="/docs/private_channels"><code class="highlighter-rouge">private-</code></a> or <a href="/docs/presence"><code class="highlighter-rouge">presence-</code></a> channel.</li> 
        <li>The <code class="highlighter-rouge">private-</code> or <code class="highlighter-rouge">presence-</code> prefix identifies the channel as requiring authentication so a request is made to an authentication endpoint via either AJAX or JSONp. For more information see <a href="#authTransport">channel authentication transports</a>.</li> 
        <li>If successful your application returns an authorisation string (see <a href="/docs/auth_signatures">generating the authentication signature</a>) to the Pusher JavaScript library signed with your Pusher secret.</li> 
        <li>The <code class="highlighter-rouge">channel_name</code> and authentication signature is sent to Pusher over the WebSocket, which completes the authorisation if the authorisation string has been correctly signed.</li> 
       </ol> 
       <p>For example, in both the <code class="highlighter-rouge">subscribe</code> calls below an authentication request will take place.</p> 
       <div data-lang="js" class="highlighter-rouge">
        <pre class="highlight"><code>var pusher = new Pusher('APP_KEY');
var privateChannel = pusher.subscribe('private-channel');
var presenceChannel = pusher.subscribe('presence-channel');</code></pre> 
       </div> 
       <p>The purpose of the authentication callback is so that you can check that the current user of your application has permission to access the channel that they are trying to subscribe to. How the user is authenticated depends on the decisions you make and the system you are developing.</p>
