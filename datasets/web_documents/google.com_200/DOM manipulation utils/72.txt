      <h1 id="page-title" class="page-title post-page__page-title___2KPKG">Unit and Integration Testing Front-end Javascript</h1>
      <div id="page-body" class="page-body">
       <div class="post-page__post-date___3HkPG">
        2016/03/09
       </div>
       <span class="markdown__markdown-render___9BIqe"><p>tl;dr= Reference/starter project can be found at <a href="https://github.com/bguiz/front-end-js-testing" title="Front-end Javascript Testing starter/ reference project">github.com/bguiz/front-end-js-testing</a></p> <h2 id="testing-nodejs-vs-testing-front-end">Testing NodeJs vs Testing Front-end</h2> <p>When developing a NodeJs application, testing it can be quite straight forward, in the sense that you simply need to to interact one thing: your <em>Javascript modules</em>.</p> <p>This is fairly easy to do if your test runner or framework is also written in NodeJs - all you have got to do is <code>require()</code> the necessary modules in, anything from an individual function to the entire application, and run your tests against them.</p> <p>It is not quite so simple with front-end Javascript, because there is an <em>additional</em> thing that you have to deal with: <em>the browser</em>.</p> <p>It does not suffice to point your test runner or framework at your Javascript modules and expect them to run. Inevitably, they’ll need access to <code>window</code>, <code>document</code>, a <code>DOMElement</code> of some variety, or any number of other things that only a browser would give you access to.</p> <h2 id="browser-based-test-runners">Browser-based Test Runners</h2> <p>The solution to the problem above is to use test runners which have browser capabilities. That is, the test runner is able to start and stop browsers that are installed on the local system (or even on remote ones), and run test cases within the browser while they are open.</p> <p>This way, you can test your front-end Javascript code in an <em>actual browser</em> environment.</p> <p>In any type of software testing, there are two paradigms: there are unit tests, and there are integration tests. Different browser-based test runners lend themselves more naturally to each of these paradigms, and we’ll take a look at those next.</p> <h2 id="unit-testing-using-karma">Unit Testing using Karma</h2> <p>Unit testing is sometimes referred to as white-box testing.</p> <p>It is where you do not concern yourself with the overall application, but rather the individual bits and pieces used to build the application. The focus here is on testing individual methods and functions, and getting very specific on on how each of them perform in isolation. Testing each unit of software, hence the name <em>unit testing</em>.</p> <p><a href="https://karma-runner.github.io/">Karma</a> is a browser-based test runner that is very well suited to doing unit testing.</p> <h2 id="integration-testing-using-webdriverio">Integration testing using WebdriverIO</h2> <p>Integration testing is sometimes referred to as black-box testing.</p> <p>It is where you do not concern yourself with the individual bits and pieces used to build the application, but rather the overall application. The focus here is on testing the application as a whole, similar to how an end-user might interact with your application. Testing the how the entire software works together, hence the name <em>integration testing</em>.</p> <p><a href="https://webdriver.io/">Webdriver.io</a> is a browser-based test runner that us very well suited to doing integration testing.</p> <h2 id="starter-reference-project">Starter/ Reference Project</h2> <p>I have put together a trivial front-end Javascript application. It has two buttons, and pressing each one of them results in a different piece of text getting put on the screen. (When I said trivial, I meant <em>trivial</em>).</p> <p>The intent was, of course, not the application itself, but rather to expose the mechanics of testing it, and to that end, it is sufficient.</p> <p>The complete project, including the setup and config for both Karma and WebdriverIO, and sample unit tests and sample integration tests are available at: <a href="https://github.com/bguiz/front-end-js-testing" title="Front-end Javascript Testing starter/ reference project">github.com/bguiz/front-end-js-testing</a></p> <h3 id="asynchronous-assertions-using-chai">Asynchronous Assertions using Chai</h3> <p>The first thing to note is that in the Javascript world, nearly everything is asynchronous. There quite likely are a mix of callback functions, promises, generator functions, and <code>async</code> functions in most Javascript applications.</p> <p>For a primer on asynchronous Javascript, check out: <a href="/2016/01/04/async-ascendance-in-javascript">Async Ascendance in Javascript</a></p> <p>This means that our tests need to take that into account.</p> <p>For this, in the starter project, we will extend our assertions library, <code>chai</code>, with <code>chai-as-promised</code>, so that we can use <code>expect(somePromise).to.eventually.equal(someValue)</code>.</p> <h3 id="nodejs-test-runner-within-a-browser-based-test-runner">NodeJs Test Runner within a Browser-based Test Runner</h3> <p>Just because we are using browser-based test runners, does not mean that we need to learn a complete new structure for writing our test cases.</p> <p>Test runner libraries such as Mocha and Jasmine can be included within both Karma and WebdriverIO as a framework, and therefore you can continue writing test cases using syntax that is already familiar.</p> <p>For this, in the starter project, we will be including Mocha as a framework within both Karma and WebdriverIO.</p> <h3 id="front-end-framework-agnostic">Front-end Framework Agnostic</h3> <p>It is worth pointing out at this juncture that the set up for this project is intentionally agnostic to the Javascript framework that the application may have been written with.</p> <p>BackboneJs, AngularJs, EmberJs, ReactJs, et cetera, and all of their accompanying library and tool stacks quite often come bundled with their own testing utilities. Where this is not the case, the community of developers usually has a consensus around their preferred testing utilities.</p> <p>This project however chooses to work with testing utilities and tools that are not tied to any one particular framework. In theory, they would work just as well with vanilla Javascript applications, as they would with Javascript application built using one of these front-end frameworks.</p> <h3 id="set-up-for-the-system-under-test">Set up for the System Under Test</h3> <p>The front-end Javascript application we will be testing in this case in the system under test.</p> <pre class="hljs bash lang-bash">npm install --no-progress
npm run global-install
npm run build-dev</pre><p>In the directory of the starter project, run the commands above. This sets up both the local and global dependencies, and uses Webpack to bundle the Javascript sources, and uses <code>http-server</code> to serve static files on <code>localhost</code>.</p> <p>This step does not have anything to do with testing, just to do with the system under test. By all means, use any other set of tools to get this done, as there are a myriad of different way to do this.</p> <h3 id="set-up-and-run-karma-unit-testing">Set up and run Karma Unit Testing</h3> <p>Look in <a href="https://github.com/bguiz/front-end-js-testing/blob/develop/package.json"><code>package.json</code></a>, in the <code>devDependencies</code> section, and you’ll find these:</p> <pre class="hljs undefined lang-undefined"><span class="hljs-string">"karma"</span>: <span class="hljs-string">"^0.13.8"</span>,
<span class="hljs-string">"karma-chrome-launcher"</span>: <span class="hljs-string">"^0.2.0"</span>,
<span class="hljs-string">"karma-mocha"</span>: <span class="hljs-string">"^0.2.0"</span>,
<span class="hljs-string">"karma-sinon-chai"</span>: <span class="hljs-string">"^1.0.0"</span>,
<span class="hljs-string">"karma-webpack"</span>: <span class="hljs-string">"^1.7.0"</span>,</pre><p>These have already been installed as part of the previous step.</p> <p>Next look in <a href="https://github.com/bguiz/front-end-js-testing/blob/develop/karma.conf.js"><code>karma.conf.js</code></a>, and you’ll find this:</p> <pre class="hljs undefined lang-undefined">files: [
  <span class="hljs-symbol">'test</span>/unit<span class="hljs-comment">/**/</span>*.spec.js'
],</pre><p>This tells Karma to run any <code>*.spec.js</code> files found in <code>test/unit</code> or its subdirectories.</p> <p>Finally let us look in this folder, and we see <code>test/unit/sample-unit.spec.js</code>, which contains a couple of unit tests for <code>src/function-to-test.js</code>.</p> <p>In order to run these unit tests, we can enter the following command:</p> <pre class="hljs bash lang-bash">npm run unit-test-background</pre><p>Take a look in the <code>scripts</code> section of <a href="https://github.com/bguiz/front-end-js-testing/blob/develop/package.json"><code>package.json</code></a> to see exactly what this does.</p> <p>A new browser window should open, run through the tests once, and then idle. Switch back to the command prompt, and you will see the test output.</p> <p>Try modifying the test file, <code>test/unit/sample-unit.spec.js</code> by inserting a new test case, or modifying one that is already there. When you save the file, the tests should re-run, giving you potentially different results.</p> <p>If we wish for the tests to just run through once, and exit right away, instead of idling, and waiting for some changes, we can run the following command instead:</p> <pre class="hljs bash lang-bash">npm run unit-test</pre><h2 id="set-up-and-run-webdriverio-integration-testing">Set up and run WebdriverIO Integration Testing</h2> <p>Take a look in the <code>scripts</code> section of <a href="https://github.com/bguiz/front-end-js-testing/blob/develop/package.json"><code>package.json</code></a>, and you’ll find the following lines:</p> <pre class="hljs undefined lang-undefined"><span class="hljs-string">"global-install"</span>: <span class="hljs-string">"npm i --global selenium-standalone@5.0.0 http-server@0.9.0 &amp;&amp; selenium-standalone install"</span>,
<span class="hljs-string">"start-selenium-server"</span>: <span class="hljs-string">"selenium-standalone start"</span>,</pre><p>Look in <a href="https://github.com/bguiz/front-end-js-testing/blob/develop/package.json"><code>package.json</code></a>, in the <code>devDependencies</code> section, and you’ll find these:</p> <pre class="hljs undefined lang-undefined"><span class="hljs-string">"wdio-mocha-framework"</span>: <span class="hljs-string">"^0.2.11"</span>,
<span class="hljs-string">"webdriverio"</span>: <span class="hljs-string">"^3.4.0"</span>,</pre><p>These commands did various global and local installations. WebdriverIO uses a well known 3rd-party browser testing protocol, the Selenium protocol, in order to do its integration testing. This requires it to have a Selenium server installed and running before running the tests.</p> <p>To start the Selenium server now, we run:</p> <pre class="hljs bash lang-bash">npm run start-selenium-server</pre><p>Leave that process running, while we examine the project further.</p> <p>Open up <a href="https://github.com/bguiz/front-end-js-testing/blob/develop/wdio.conf.js"><code>wdio.conf.js</code></a>, and you will see:</p> <pre class="hljs undefined lang-undefined">specs: [
    <span class="hljs-string">'./test/integration/**/*.spec.js'</span>
],</pre><p>This tells WebdriverIO to run any <code>*.spec.js</code> files found in <code>test/integration</code> or its subdirectories.</p> <p>Finally let us look in this folder, and we see <code>test/integration/sample-integration.spec.js</code>, which contains a couple of integration tests for the application as a whole.</p> <p>In order to run these integration tests, we can enter the following command:</p> <pre class="hljs bash lang-bash">npm run integration-test</pre><p>You should see, similar to when we ran Karma unit tests previously, the browser window open while running through the tests. The results of the tests will be visible in the terminal as well.</p> <p>The difference that you’ll notice, from when running the unit tests, is that we actually see the system under test appearing in the browser, and the browser automatically whirring through each of the steps. Usually this is slow enough for us to see each of the steps briefly, but faster than any human would be able to execute by hand.</p> <h2 id="walkthrough">Walkthrough</h2> <p>We have just gone through the basics of setting up, configuring, and running both unit tests and integration tests for system under test - in this case also known as “the world’s most rudimentary front-end Javascript application”.</p> <p>We have discussed neither the configuration, nor the syntax of the test cases, in detail. That is perhaps something for a follow up post!</p> </span>
