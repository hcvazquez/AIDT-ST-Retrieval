overkill in some cases. CAPTCHAs help in differentiating between
real users and automated bots. In this article, Anand Raman uses
CAPTCHAs as weak authentication mechanisms for J2EE Web
applications. He begins with a quick introduction to both the J2EE
Web application security model and CAPTCHAs. He then builds on
these concepts to implement a JAAS (Java Authentication and
Authorization Service) login module using CAPTCHAs and integrates
it with an application server's existing security infrastructure.
The artifacts are based on standard J2EE security mechanisms.
Hence, the module can be reused on any J2EE application or across
different application servers with some minor modifications.">

       <p>First, let's quickly look at the J2EE Web application security model.</p>
       <h2>J2EE security model</h2>
       <p>Security has been one of the focus areas guiding Java's development. Needless to say, J2EE has adopted the same principles and provides a robust framework for building secure applications. Security in J2EE applications is a vast topic that I can't cover in detail here. Several excellent resources cover this topic in detail. I strongly recommend that teams and developers spend time familiarizing themselves with these concepts. For this article, I present a 20,000-foot view of some of the key concepts.</p>
       <h4>Key concepts</h4>
       <p>Security in J2EE applications can be enforced by using either a <em>declarative approach</em>or a <em>programmatic approach</em>.</p>
       <p>As the name suggests, when using the declarative approach, an application developer defines an application's security constraints outside the application code. These declarations are made in deployment descriptors (<code>web.xml</code>, <code>ejb-jar.xml</code>), and the container's runtime environment guarantees their enforcement. Declarative security allows developers to:</p>
       <aside class="nativo-promo tablet desktop" id=""> 
       </aside>
       <ul>
        <li>Restrict access to resources only to principals with given roles (e.g., "/admin/*" is restricted to the "administrator" role)</li>
        <li>Restrict access to URLs only on certain protocols (e.g., "/customer/*" is only accessible via HTTPS)</li>
        <li>Restrict access to servlets only to principals with given roles (e.g., <code>SiteShutdownServlet</code> is restricted to the "god" role)</li>
        <li>Restrict individual methods on EJB (Enterprise JavaBeans) components only to principals with given roles (e.g., <code>PurchasingEJB</code>is restricted to the "customer" role)</li>
        <li>Automatically redirect users to login page when they request a restricted resource, but have not yet logged into the system</li>
       </ul>
       <p>On the other hand, the programmatic approach provides mechanisms for calling and querying the security infrastructure. The developer must enforce these constraints. The programmatic approach helps to:</p>
       <ul>
        <li>Retrieve the principal associated with the current authenticated user: <code>HttpServletRequest.getUserPrincipal</code>or <code>EJBContext.getCallerPrincipal</code></li>
        <li>Query if the user has been assigned a specific role: <code>HttpServletRequest.isUserInRole(String role)</code>or <code>EJBContext.isCallerInRole(String role)</code></li>
       </ul>
       <p>Both these approaches have their own limitations and hence complement each other.</p>
       <h4>Declarative security for Web applications</h4>
       <p>Declarative security for Web applications is passive in nature. This means that only on initial access to the protected resource by an unauthenticated client, is the user forwarded to the login page. If the user is already authenticated and has the necessary authorization (role), he or she is allowed to access the resource.</p>
       <p>One of the most common approaches to declarative security in Web applications is to use <em>form-based authentication.</em> The Web application deployment descriptor <code>web.xml</code> declares all the necessary elements required for such configuration in two sections.</p>
       <p>The first section is common to the entire Web application. It identifies:</p>
       <ul>
        <li>The login method that needs to be used (<code>auth-method</code>). J2EE supports the following authentication mechanisms BASIC, DIGEST, FORM, or CERT.</li>
        <li>The login and error page for form-based authentication (<code>form-login-config</code>).</li>
        <li>Superset of all roles that can be used in the application (<code>security-role</code>).</li>
       </ul>

        <li>The transport guarantees associated with accessing a resource (<code>user-data-constraint</code>).</li>

       <p>So far, I have covered the key concepts of the J2EE security model. Let's now look at ways to extend a container's security infrastructure.</p>
       <h4>Extending a container's security infrastructure</h4>
       <p><em>JAAS</em>(Java Authentication and Authorization Services) implements a version of <em>Pluggable Authentication Modules (PAM)</em> for Java applications. It allows parallel development of authentication technologies and enterprise applications. Application developers can choose from these options and configure them for their applications. JAAS has several benefits since it allows parallel development of authentication technologies and promotes its reuse across different applications.</p>
       <p>JAAS brings the same value to the application server landscape. However, JAAS has not achieved the same success in the J2EE space. Until recently, application servers exposed custom APIs for extending out-of-the-box security infrastructure. But things are changing. Application servers now provide adaptors for integrating custom JAAS login modules to a security infrastructure. This integration is still application-server-specific and the complexity varies widely.</p>
       <p>Tomcat provides a relatively easy and straightforward integration with JAAS. The custom login modules are configured using configuration files (Tomcat realm configuration and the standard JAAS configuration). When the server needs to call the login module, it routes all the calls through the <code>org.apache.catalina.realm.JAASRealm</code>adaptor. For more details on integrating JAAS authentication with Tomcat, refer to <a href="#resources" rel="nofollow">Resources</a>.</p>
       <p>In this article, we implement a JAAS login module and later integrate it with the Tomcat server to provide a J2EE security solution.</p>
       <h2>Solution</h2>
       <p>Before proceeding to the implementation aspects, let's understand our solution's goals and the approach.</p>
       <p>Our goals are as follows:</p>
       <ul>
        <li>Provide a <strong>weak authentication mechanism</strong> for Web applications. Weak authentication means that the authentication module or the application doesn't differentiate one remote user from another. Once authenticated, all the users have similar roles.</li>
        <li>Forgo a unique identifier for every user of the system (login name). This guarantees anonymity of the remote user.</li>
        <li>The authentication mechanism should differentiate between computers and humans to prevent automated spam-bots from misusing the available resources. We will use CAPTCHAs for this test.</li>
        <li>The authentication mechanism should be based on the J2EE security model. We will avoid other approaches that do not conform to this model.</li>
       </ul>
       <p>Given the above goals, obviously, we need to certify only that every session belongs to a human user. Application servers maintain sessions to preserve a user's state. When unauthenticated users access a protected resource, J2EE declarative security diverts them to the login page. The login page generates a unique CAPTCHA and associates it with the user's session. The login page then presents the CAPTCHA as an image and challenges the user to identify the image. The login page also has a hidden input field that contains the current session ID.</p>
       <p>The user enters the challenge in the password field and submits the form. On form submission, the login module retrieves the session ID and the user's response. It then checks the response with the associated CAPTCHA. If these values match, the login module authenticates the user and assigns him the role "anonymous."</p>
       <p>After authentication, the remote user can access all the protected resources constrained by the role "anonymous."</p>
       <p>Figure 4 illustrates our approach.</p>
