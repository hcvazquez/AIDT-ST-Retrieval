     <h2 class="adjustedAnchor" id="whatIsUML">What is a UML Diagram?</h2> 
     <p>UML is a way of visualizing a software program using a collection of diagrams. The notation has evolved from the work of Grady Booch, James Rumbaugh, Ivar Jacobson, and the Rational Software Corporation to be used for object-oriented design, but it has since been extended to cover a wider variety of software engineering projects. Today, UML is accepted by the Object Management Group (OMG) as the standard for modeling software development.</p> 
     <h2 class="adjustedAnchor" id="UMLMeaning">What is Meant by UML?</h2> 
     <p>UML stands for Unified Modeling Language. UML 2.0 helped extend the original UML specification to cover a wider portion of software development efforts including agile practices.</p> 
     <ul class="vertList"> 
      <li>Improved integration between structural models like class diagrams and behavior models like activity diagrams.</li> 
      <li>Added the ability to define a hierarchy and decompose a software system into components and sub-components.</li> 
      <li>The original UML specified nine diagrams; UML 2.x brings that number up to 13. The four new diagrams are called: communication diagram, composite structure diagram, interaction overview diagram, and timing diagram. It also renamed statechart diagrams to state machine diagrams, also known as state diagrams.</li> 
     </ul> 
     <h2 class="adjustedAnchor" id="UMLTutorial">UML Diagram Tutorial</h2> 
     <p>The key to making a UML diagram is connecting shapes that represent an object or class with other shapes to illustrate relationships and the flow of information and data. To learn more about creating UML diagrams:</p> 
     <ul class="vertList">

      <li><a href="/uml-diagram/uml-diagram-tips.htm">Browse Tips for Better UML Diagrams</a></li>

     <h2 class="adjustedAnchor" id="UMLTypes">Types of UML Diagrams</h2> 
     <p>The current UML standards call for 13 different types of diagrams: class, activity, object, use case, sequence, package, state, component, communication, composite structure, interaction overview, timing, and deployment. </p> 
     <p>These diagrams are organized into two distinct groups: structural diagrams and behavioral or interaction diagrams. </p> 
     <p>Structural UML diagrams </p> 
     <ul class="vertList"> 
      <li>Class diagram</li> 
      <li>Package diagram</li> 
      <li>Object diagram</li> 
      <li>Component diagram</li> 
      <li>Composite structure diagram</li> 
      <li>Deployment diagram</li> 
     </ul> 
     <p>Behavioral UML diagrams </p> 
     <ul class="vertList"> 
      <li>Activity diagram</li> 
      <li>Sequence diagram</li> 
      <li>Use case diagram</li> 
      <li>State diagram</li> 
      <li>Communication diagram</li> 
      <li>Interaction overview diagram</li> 
      <li>Timing diagram</li> 
     </ul> 
     <p><strong>Class Diagram </strong><br> <a href="/class-diagram/">Class diagrams</a> are the backbone of almost every object-oriented method, including UML. They describe the static structure of a system. <a href="/class-diagram/">Learn more</a> </p> 
     <p><strong>Package Diagram</strong><br> Package diagrams are a subset of class diagrams, but developers sometimes treat them as a separate technique. Package diagrams organize elements of a system into related groups to minimize dependencies between packages. </p> 
     <img alt="UML Package Diagram" border="0" src="/uml-diagram/img/package-diagram.jpg?bn=1510011096"> 
     <p><strong>Object Diagram</strong><br> Object diagrams describe the static structure of a system at a particular time. They can be used to test class diagrams for accuracy. </p> 
     <img alt="UML Object Diagram" border="0" src="/uml-diagram/img/object-diagram.jpg?bn=1510011096"> 
     <p><strong>Composite Structure Diagram</strong><br> Composite structure diagrams show the internal part of a class. </p> 
     <p><strong>Use Case Diagram </strong><br> <a href="/use-case-diagram/">Use case diagrams</a> model the functionality of a system using actors and use cases. <a href="/use-case-diagram/">Learn more</a> </p> 
     <img alt="UML Use Case Diagram" border="0" src="/uml-diagram/img/use-case-diagram.jpg?bn=1510011096"> 
     <p><strong>Activity Diagram</strong><br> <a href="/activity-diagram/">Activity diagrams</a> illustrate the dynamic nature of a system by modeling the flow of control from activity to activity. An activity represents an operation on some class in the system that results in a change in the state of the system. Typically, activity diagrams are used to model workflow or business processes and internal operation. <a href="/activity-diagram/">Learn more</a> </p> 
     <img alt="UML Activity Diagram" border="0" src="/uml-diagram/img/activity-diagram.jpg?bn=1510011096"> 
     <p><strong>Sequence Diagram</strong><br> <a href="/sequence-diagram/">Sequence diagrams</a> describe interactions among classes in terms of an exchange of messages over time. <a href="/sequence-diagram/">Learn more</a> </p> 
     <img alt="UML Sequence Diagram" border="0" src="/uml-diagram/img/sequence-diagram.jpg?bn=1510011096"> 
     <p><strong>Interaction Overview Diagram</strong><br> Interaction overview diagrams are a combination of activity and sequence diagrams. They model a sequence of actions and let you deconstruct more complex interactions into manageable occurrences. You should use the same notation on interaction overview diagrams that you would see on an activity diagram. </p> 
     <p><strong>Timing Diagram</strong><br> A timing diagram is a type of behavioral or interaction UML diagram that focuses on processes that take place during a specific period of time. They're a special instance of a sequence diagram, except time is shown to increase from left to right instead of top down. </p> 
     <p><strong>Communication Diagram</strong><br> Communication diagrams model the interactions between objects in sequence. They describe both the static structure and the dynamic behavior of a system. In many ways, a communication diagram is a simplified version of a collaboration diagram introduced in UML 2.0. </p> 
     <p><strong>State Diagram</strong><br> <a href="/state-diagram/">Statechart diagrams</a>, now known as state machine diagrams and state diagrams describe the dynamic behavior of a system in response to external stimuli. State diagrams are especially useful in modeling reactive objects whose states are triggered by specific events. <a href="/state-diagram/">Learn more</a> </p> 
     <img alt="UML State Diagram" border="0" src="/uml-diagram/img/state-diagram.jpg?bn=1510011096"> 
     <p><strong>Component Diagram</strong><br> <a href="/component-diagram/">Component diagrams</a> describe the organization of physical software components, including source code, run-time (binary) code, and executables. <a href="/component-diagram/">Learn more</a>. </p> 
     <img alt="UML Component Diagram" border="0" src="/uml-diagram/img/component-diagram.jpg?bn=1510011096"> 
     <p><strong>Deployment Diagram</strong><br> Deployment diagrams depict the physical resources in a system, including nodes, components, and connections. </p> 
     <h2 class="adjustedAnchor" id="UMLSymbols">UML Diagram Symbols</h2> 
     <p>There are many different types of UML diagrams and each has a slightly different symbol set.</p> 
     <p>Class diagrams are perhaps one of the most common UML diagrams used and class diagram symbols center around defining attributes of a class. For example, there are symbols for active classes and interfaces. A class symbol can also be divided to show a class's operations, attributes, and responsibilities.</p> 
     <p>Visibility of any class members are marked by notations of</p> 
     <img alt="UML class members" src="/uml-diagram/img/uml-class-members-notations.png?bn=1510011096"> 
     <p>Lines are also important symbols to denote relationships between components. Generalization and Inheritance are denoted with empty arrowheads. Composition is shown with a filled in diamond. Aggregation is shown with an empty diamond. Dependencies are marked with a dashed line with an arrow. Using &lt;&lt; &gt;&gt; allows you to indicate properties of that dependency. Multiplicity is usually shown with a number at one end of the arrow and a * at the other.</p> 
     <img alt="UML symbols" src="/uml-diagram/img/uml-symbols.png?bn=1510011096"> 
     <p>Package diagrams have symbols defining a package that look like a folder.</p> 
     <img alt="UML package symbol" src="/uml-diagram/img/uml-package-symbol.png?bn=1510011096"> 
     <p>Activity diagrams have symbols for activities, states, including separate symbols for an initial state and a final state. The control flow is usually shown with an arrow and the object flow is shown with a dashed arrow.</p> 
     <img alt="UML initial and final state" src="/uml-diagram/img/uml-initial-final-state.png?bn=1510011096"> 
     <p>Use case diagrams have symbols for actors and use cases.</p> 
     <h2 class="adjustedAnchor" id="WHYUseUML">Why Do We Use UML?</h2> 
     <p>A complex enterprise application with many collaborators will require a solid foundation of planning and clear, concise communication among team members as the project progresses.</p> 
     <p>Visualizing user interactions, processes, and the structure of the system you're trying to build will help save time down the line and make sure everyone on the team is on the same page.</p>
