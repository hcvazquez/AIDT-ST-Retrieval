     <p><em>Editor’s note: This is an updated version of an article originally published here on 21/01/2010.</em></p> 
     <p>It’s been awhile since JavaScript charts and graphs became the go-to industry norm for data visualization. In fact we decided to build our own graphing engine for <a title="Server Monitoring Software" href="http://www.serverdensity.com">Server Density</a> several years ago. That’s because we needed some functionality that was not possible with the Flash charts we used earlier. Plus, it allowed us to customize the experience to better fit our own design.</p> 
     <p>Since then we’ve been revamping the entire engine. Our latest charts take advantage of various modern JS features such as toggling line series, pinning extended info and more.</p> 
     <p><img class="alignnone size-full wp-image-5867" src="https://serverdensity-wpengine.netdna-ssl.com/wp-content/uploads/2010/01/JS1.png" alt="Automatic timezone conversion in JavaScript 1" width="1170" height="709" srcset="https://serverdensity-wpengine.netdna-ssl.com/wp-content/uploads/2010/01/JS1.png 1170w, https://serverdensity-wpengine.netdna-ssl.com/wp-content/uploads/2010/01/JS1-150x91.png 150w, https://serverdensity-wpengine.netdna-ssl.com/wp-content/uploads/2010/01/JS1-300x182.png 300w, https://serverdensity-wpengine.netdna-ssl.com/wp-content/uploads/2010/01/JS1-1024x621.png 1024w, https://serverdensity-wpengine.netdna-ssl.com/wp-content/uploads/2010/01/JS1-600x364.png 600w" sizes="(max-width: 1170px) 100vw, 1170px"></p> 
     <p>Switching to a new graphing engine was no painless journey of course. JS comes with its own challenges, one of which is automatic timezone conversion.</p> 
     <h2>Timezones are a pain</h2> 
     <p>Timezone conversion is one of the issues you should always expect to deal with when building JS applications targeted at clients in varying timezones. Here is what we had to deal with.</p> 
     <p>Our new engine supports user preferences with timezones. We do all the timezone calculations server-side and pass JSON data to the Javascript graphs, with the timestamps for each point already converted.</p> 
     <p>However, it turns out that the JavaScript Date object does its own client-side timezone conversion based on the user’s system timezone settings. This means that if the default date on the graph is 10:00 GMT and your local system timezone is Paris, then JavaScript will automatically change that to 11:00 GMT.</p> 
     <p>This only works when the timestamp passed is in GMT. So it presents a problem when we have already done the timezone conversion server-side, i.e. the conversion will be calculated twice – first on the server, then again on the client.</p> 
     <p>We could allow JavaScript to handle timezones and perform all the conversions. However, this would result in messed up links, because we used data points to redirect the user to the actual snapshots.</p> 
     <p>Snapshots are provided in Unix timestamp format, so even if the JS did the conversion, the snapshot timestamp would still be incorrect. To completely remove the server side conversion and rely solely on JS would require more changes and a lot more JS within the interface.</p> 
     <h2>UTC-based workaround</h2> 
     <p>As such, we modified our getDate function to return the values in UTC—at least it is UTC as far as JS is concerned but in reality we’d have already done the conversion on the server. This effectively disables the JavaScript timezone conversion.</p> 
     <p>The following code snippet converts the Unix timestamp in JavaScript provided by the server into a date representation that we can use to display in the charts:</p> 
     <pre class="language-javascript"><code>getDate: function(timestamp)
{
// Multiply by 1000 because JS works in milliseconds instead of the UNIX seconds
var date = new Date(timestamp * 1000);

var year = date.getUTCFullYear();
var month = date.getUTCMonth() + 1; // getMonth() is zero-indexed, so we'll increment to get the correct month number
var day = date.getUTCDate();
var hours = date.getUTCHours();
var minutes = date.getUTCMinutes();
var seconds = date.getUTCSeconds();

month = (month &lt; 10) ? '0' + month : month;
day = (day &lt; 10) ? '0' + day : day;
hours = (hours &lt; 10) ? '0' + hours : hours;
minutes = (minutes &lt; 10) ? '0' + minutes : minutes;
seconds = (seconds &lt; 10) ? '0' + seconds: seconds;

return year + '-' + month + '-' + day + ' ' + hours + ':' + minutes;
}</code></pre> 
     <p>So this is how we handle timezone with JavaScript for the <a title="Server Monitoring Software" href="http://www.serverdensity.com">Server Density</a> graphing engine. What is your experience with timezones in JavaScript?</p>
