                <div class="jive-rendered-content">
                 <p>Hi Peter,</p>
                 <p style="min-height: 8pt; padding: 0px;">&nbsp;</p>
                 <p>A <em>compare</em> function used in <strong>Array.sort</strong> should always return signed numbers (booleans cannot define a <em>total</em> order).</p>
                 <p>Here are the expected return values of such comparison:</p>
                 <p style="min-height: 8pt; padding: 0px;">&nbsp;</p>
                 <!--[CodeBlockStart:78711437-260d-4b5c-9b68-660538d0e1eb][excluded]-->
                 <pre class="java" name="code">a &lt; b&nbsp;&nbsp;&nbsp; --&gt;&nbsp;&nbsp; ret &lt; 0
a == b&nbsp;&nbsp; --&gt;&nbsp;&nbsp; ret = 0
a &gt; b&nbsp;&nbsp;&nbsp; --&gt;&nbsp;&nbsp; ret &gt; 0
</pre>
                 <!--[CodeBlockEnd:78711437-260d-4b5c-9b68-660538d0e1eb]-->
                 <div style="display:none;"></div>
                 <p style="min-height: 8pt; padding: 0px;">&nbsp;</p>
                 <p>Both your <em>objSort_1</em> and <em>objSort_2</em> functions return booleanswhich are coerced to 1 (true) or 0 (false)so your functions never assert:</p>
                 <p>a &lt; b</p>
                 <p style="min-height: 8pt; padding: 0px;">&nbsp;</p>
                 <p>Now, <em>objSort_1</em> seems to work, by chance, because your a==b and a &gt; b assertions are consistant although you have not a total order. In fact, <em>objSort_1</em> makes no distinction between a &lt; b and a==b, but you don't see that issue because your set does not seem to contain identical items.</p>
                 <p style="min-height: 8pt; padding: 0px;">&nbsp;</p>
                 <p><em>objSort_2</em> is more problematic because <strong>(a.prefix == b.prefix) &amp;&amp; (a.num &gt; b.num)</strong> takes no decision when <strong>a.prefix != b.prefix </strong>which often happens, though!</p>
                 <p style="min-height: 8pt; padding: 0px;">&nbsp;</p>
                 <p>Anyway, you have to provide a full comparison:</p>
                 <p style="min-height: 8pt; padding: 0px;">&nbsp;</p>
                 <!--[CodeBlockStart:c5dadb0f-c8bb-4714-a770-6b93619a129c][excluded]-->
                 <pre class="java" name="code">function objSort_3(a, b)
&nbsp;&nbsp;&nbsp; {
&nbsp;&nbsp;&nbsp; var ax = a.prefix,
&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; bx = b.prefix,
&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; ay = a.num,
&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; by = b.num;

&nbsp;&nbsp;&nbsp; return ( (ax &gt; bx) - (ax &lt; bx) ) ||
&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; ( (ay &gt; by) - (ay &lt; by) );
&nbsp;&nbsp;&nbsp; }
</pre>
                 <!--[CodeBlockEnd:c5dadb0f-c8bb-4714-a770-6b93619a129c]-->
                 <div style="display:none;"></div>
                 <p style="min-height: 8pt; padding: 0px;">&nbsp;</p>
                 <p>Note that <em>objSort_3</em> always returns signed numbers (-1, 0, or +1) because any <strong>M &lt; N</strong> (boolean) is coerced to a number thanks to the subtraction operator.</p>
                 <p>For example, if <em>ax &gt; bx</em>, then (ax &gt; bx) - (ax &lt; bx) evaluates to +1 (TRUE - FALSE).</p>
                 <p>Also, in the case where <em>ax == bx</em>, the previous expression evaluates to 0 (FALSE - FALSE), so the second part (below the || operator) is evaluated.</p>
                 <p style="min-height: 8pt; padding: 0px;">&nbsp;</p>

                 <p>Hi again Peter,</p>
                 <p style="min-height: 8pt; padding: 0px;">&nbsp;</p>
                 <p> As the ECMAScript specification (see ECMA-262 15.4.4.11) makes no assumption&nbsp; about the inner implementation of <strong>Array.sort</strong>, it is possible that ExtendScript uses a <em>comparison algorithm</em> (e.g. a "quick sort") so that a simple "X &lt; Y" predicate still works. [AFAIK several ExtendScript components are based on the C++ Boost library, which sounds to provide the quick sort algorithm Maybe this is the reason.]</p>
                 <p style="min-height: 8pt; padding: 0px;">&nbsp;</p>
                 <p> However, we are not supposed to know that and it is not a good thing to teach that. Indeed, as clearly stated by the ECMA spec, <em>"If comparefn is not undefined, it should be a function that accepts two arguments x and y and returns <strong>a negative value if x &lt; y, zero if x = y, or a positive value if x &gt; y</strong>." </em>Portable JS code should always satisfies this principleeven though we come across <em>many many</em> examples on the Web that violate that rule and use boolean returns, typically: <span style="color: #ff0000;"><strong>function(x, y){ return x &gt; y; }</strong></span></p>
                 <p>I consider such code defective and dangerous anyway. Who knows how ExtendScript will evolve ;-)</p>
                 <p style="min-height: 8pt; padding: 0px;">&nbsp;</p>
                 <p> Now, you're right that it wasn't <em>entirely correct</em> to say <em>"In fact, objSort_1 makes no distinction between a &lt; b and a==b, but you don't see that issue because your set does not seem to contain identical items."</em> Only the first part of my statement is correct! What remains true is that "<em>objSort_1 makes no distinction between a &lt; b and a==b"</em> You can easily check this:</p>
                 <p style="min-height: 8pt; padding: 0px;">&nbsp;</p>
                 <!--[CodeBlockStart:64f73f94-f87f-4622-a61c-b15d397af435][excluded]-->
                 <pre class="java" name="code">function objSort_1 (a, b){ return a.prefix &gt; b.prefix }

var X = {prefix:'a'};
var Y = {prefix:'b'};

alert( objSort_1(X, Y) ); // =&gt; false
alert( objSort_1(X, X) ); // =&gt; false

</pre>
                 <!--[CodeBlockEnd:64f73f94-f87f-4622-a61c-b15d397af435]-->
                 <div style="display:none;"></div>
                 <p style="min-height: 8pt; padding: 0px;">&nbsp;</p>
                 <p>which shows that a &lt; b and a==b leads to the same result (false). Therefore, <em>objSort_1</em> makes no distinction between these two cases.</p>
                 <p style="min-height: 8pt; padding: 0px;">&nbsp;</p>
                 <p>By chance this doesn't impact the resulting sort even with identical items in the set, due to the specific way ExtendScript implements this feature. But you should be aware that this is just a "lucky accident."</p>

                 <!--[CodeBlockStart:25cc6671-1729-4081-9f1b-04d28e640b3d][excluded]-->
                 <pre class="java" name="code">array = [[4,3,2,1], [1,2,3], [1,3,2,4], [5,8,9,7,6,4,8], [4,3,2,1,12] ];

array.sort( function fnComp(a,b){
&nbsp;&nbsp;&nbsp; var aLength = a.length,
&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; bLength = b.length,
&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; minLength = Math.min(aLength, bLength),
&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; i, ai, bi;
&nbsp;&nbsp;&nbsp; 
&nbsp;&nbsp;&nbsp; for( i=0 ; i &lt; minLength ; ++i )
&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; {
&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; ai = a[i];
&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; bi = b[i];
&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; if( ai != bi ) return ai-bi;
&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; }

&nbsp;&nbsp;&nbsp; return aLength - bLength;
&nbsp;&nbsp;&nbsp; });

alert( array.join('\r') );

/*
=&gt;&nbsp;&nbsp;&nbsp; 1,2,3
&nbsp;&nbsp;&nbsp; 1,3,2,4
&nbsp;&nbsp;&nbsp; 4,3,2,1

*/

                 <p>Now I have a "killer trick" for you in the case the compared arrays <strong>only contains positive integers in the range [0..65535]</strong>:</p>
                 <p style="min-height: 8pt; padding: 0px;">&nbsp;</p>
                 <!--[CodeBlockStart:febf81c4-0e3a-4e38-9718-3247a8f5e70b][excluded]-->
                 <pre class="java" name="code">array = [[4,3,2,1], [1,2,3], [1,3,2,4], [5,8,9,7,6,4,8], [4,3,2,1,12] ];

var fChr = String.fromCharCode;

array.sort( function(a,b){
&nbsp;&nbsp;&nbsp; var sa = fChr.apply(null, a),
&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; sb = fChr.apply(null, b);
&nbsp;&nbsp;&nbsp; return (sa &gt; sb) - (sa &lt; sb);
&nbsp;&nbsp;&nbsp; });

alert( array.join('\r') );
</pre>
