                                  <td class="pageText" valign="top" width="50%"><b>Source text - English</b><br>Why WebAssembly is Faster Than asm.js<br> https://hacks.mozilla.org/2017/03/why-webassembly-is-faster-than-asm-js/<br> <br> WebAssembly, a new binary execution format for the Web, is starting to arrive in stable versions of browsers. A major goal of WebAssembly is to be fast. This post gives some technical details about how it achieves that.<br> <br> Of course, “fast” is relative. Compared to JavaScript and other dynamic languages, WebAssembly is fast because it is statically typed and simple to optimize. But WebAssembly is also intended to be as fast as native code. asm.js has already come quite close to that, and WebAssembly narrows the gap further. This post focuses therefore on why WebAssembly is faster than asm.js.<br> <br> Before we start, the usual caveats: Performance is tricky to measure, and has many aspects. Also, in a new technology there are always going to be not-yet-optimized cases. So not every single benchmark will be fast on WebAssembly today. This post describes why WebAssembly should be fast; where it isn’t yet, those are bugs we need to fix.<br> <br> With that out of the way, here is why WebAssembly is fast:<br> <br> 1. Startup<br> <br> WebAssembly is designed to be small to download and fast to parse, so that even large applications start up quickly.<br> <br> It’s actually not that easy to improve on the download size of gzipped minified JavaScript, as it’s already fairly compact when compared with native code. Still, WebAssembly’s binary format can improve on that, by being carefully designed for size in mind (indexes are LEB128s, etc.). It is often around 10–20% smaller (comparing gzipped sizes).<br> <br> WebAssembly improves on parsing in a much bigger way: It can be parsed an order of magnitude faster than JavaScript. This mostly comes down to binary formats being faster to parse, especially ones designed for that. WebAssembly also makes it easy to parse (and optimize) functions in parallel, which helps a lot on multicore machines.<br> <br> Total startup time can include factors other than downloading and parsing, such as the VM fully optimizing the code, or downloading additional data files that are necessary before execution, etc. But downloading and parsing are unavoidable and therefore important to improve upon as much as possible. All the rest can be optimized or mitigated, either in the browser or in the app (for example, fully optimizing the code can be avoided by using a baseline compiler or interpreter for WebAssembly, for the first few frames).</td>
                                  <td class="pageText" valign="top" width="50%"><b>Translation - Spanish</b><br>Por qué WebAssembly es más rápido que asm.js<br> https://hacks.mozilla.org/2017/03/why-webassembly-is-faster-than-asm-js/<br> <br> WebAssembly, un nuevo formato de ejecución binario para la web, está empezando a llegar a las versiones estables de varios navegadores. Comparado con JavaScript y otros lenguajes dinámicos, WebAssembly es rápido porque usa tipado estático y es fácil de optimizar. Pero WebAssembly también tiene la intención de ser tan rápido como el código nativo. asm.js ya se ha acercado a eso, y WebAssembly se acerca aún más. Por lo tanto, esta entrada se centra en por qué WebAssembly es más rápido que asm.js.<br> <br> Antes de que empecemos, las advertencias usuales: El rendimiento es difícil de medir, y tiene muchos aspectos. Además, en una tecnología nueva siempre van a haber casos de funciones que aún no han sido optimizadas. Así que no todos los benchmarks van a ser rápidos en WebAssembly hoy. Esta entrada describe por qué WebAssembly debería ser rápido; donde no lo es aún, esos son errores que tenemos que arreglar.<br> <br> Con eso fuera del camino, diré por qué WebAssembly es rápido:<br> <br> 1. Inicio<br> <br> WebAssembly está diseñado para ser interpretado rápidamente y tener una descarga pequeña, para que hasta las aplicaciones más grandes se inicien con rapidez.<br> <br> En realidad no es tan fácil mejorar el tamaño de la descarga de JavaScript minificado y comprimido con gzip, porque eso ya es bastante compacto comparado con el código nativo. Aún así, el formato binario de WebAssembly puede ser mejor de eso, siendo diseñado cuidadosamente para tener un tamaño más reducido (los índices son LEB128s, etc.). Suele ser 10-20% más pequeño (comparando tamaños con gzip).<br> <br> WebAssembly mejora la interpretación de una forma mucho más grande: Puede ser interpretado mucho más rápido que el JavaScript. Esto principalmente se debe a que los formatos binarios se interpretan más rápido, especialmente si están diseñados para eso. WebAssembly también hace que sea fácil interpretar (y optimizar) funciones en paralelo, lo cual ayuda mucho en máquinas con múltiples núcleos.<br> <br> El tiempo total de inicio puede incluir más factores aparte de descargar e interpretar, como el tiempo que tarda la máquina virtual en optimizar el código completamente, o descargar archivos adicionales de datos que son necesarios antes de la ejecución, etc. Pero descargar e interpretar son dos cosas inevitables, y por eso es importante mejorarlas lo más posible. El resto se puede optimizar o mitigar, sea en el navegador o en la aplicación (por ejemplo, optimizar el código completamente puede ser evitado usando un compilador o intérprete base para WebAssembly, en los primeros cuadros).</td>
