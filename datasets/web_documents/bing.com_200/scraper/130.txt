             <div class="post-text" itemprop="text"> 
              <p>I have posted some versions of this question (or at least similar ones) before. But I once again tried to improve it. </p> 
              <p>The goal is to be able to use this package whether it's for a traditional card-game or a completely different type of cards. </p> 
              <p>I'd like input on best practices with regards to OOP and the overall structure. </p> 
              <p>Here all the interfaces:</p> 
              <p><strong>Card.java</strong></p> 
              <pre><code>package com.tn.deck;


public interface Card&lt;T&gt;  {
    boolean isConsecutive(T other);
    boolean isEquals(T other);
}
</code></pre> 
              <p><strong>Deck.java</strong></p> 
              <pre><code>package com.tn.deck;

import java.util.Comparator;


public interface Deck&lt;T extends Card&gt; {
    void shuffle();
    void sort(Comparator&lt;T&gt; sort);
    &lt;S extends Receivable&lt;T&gt;&gt; void addCardToPlayers(S... players);
}
</code></pre> 
              <p><strong>Hand.java</strong></p> 
              <pre><code>package com.tn.deck;

import java.util.List;


public interface Hand&lt;T extends Card&gt; {
   int calculate();
   void addCardToDeck(int index, List&lt;? super Card&gt; deck);
   boolean isEquals(Hand&lt;T&gt; other);
}
</code></pre> 
              <p><strong>Receivable.java</strong></p> 
              <pre><code>package com.tn.deck;

import java.util.List;


public interface Receivable&lt;T extends Card&gt; {
    void addCards(List&lt;T&gt; cards);
    void addCard(T card);
    void addCard(int index, T card);
}
</code></pre> 
              <p><strong>Sendable.java</strong></p> 
              <pre><code>package com.tn.deck;

import java.util.List;


public interface Sendable&lt;T extends Card&gt; {
    List&lt;T&gt; removeCards(int numberOfCards);
    List&lt;T&gt; removeCards(int... indices);
    T removeCard();
    T removeCard(int index);
}
</code></pre> 
              <p><strong>Playable.java</strong></p> 
              <pre><code>package com.tn.deck;


public interface Playable&lt;T extends Card&gt; extends Sendable&lt;T&gt;, Receivable&lt;T&gt; {
}
</code></pre> 
              <p>The next class is Abstract, without any abstract methods. But I did not want it instantiable, while still being able to inherit from it. </p> 
              <p><strong>AbstractOperation.java</strong></p> 
              <pre><code>package com.tn.deck;

import java.util.ArrayList;
import java.util.Arrays;
import java.util.Collection;
import java.util.List;
import java.util.stream.Collectors;
import java.util.stream.IntStream;


public abstract class AbstractCardOperation&lt;T extends Card&gt; implements Receivable&lt;T&gt;, Sendable&lt;T&gt;  {
    protected List&lt;T&gt; cards;

    public AbstractCardOperation() {
        this.cards = new ArrayList&lt;&gt;();
    }

    public AbstractCardOperation(List&lt;T&gt; cards) {
        this.cards = cards;
    }

    public AbstractCardOperation(List&lt;T&gt; list, int size) {
        this.cards = IntStream.range(0, size)
                .mapToObj(i -&gt; list)
                .flatMap(Collection::stream)
                .collect(Collectors.toList());
    }

    @Override
    public List&lt;T&gt; removeCards(int numberOfCards) {
        return IntStream.range(0, numberOfCards)
                .mapToObj(i -&gt; cards.remove(i))
                .collect(Collectors.toList());
    }

    @Override
    public List&lt;T&gt; removeCards(int... indices) {
        List&lt;T&gt; removedCards = new ArrayList&lt;&gt;();
        Arrays.stream(indices).forEach(index -&gt; removedCards.add(cards.remove(index)));
        return removedCards;
    }

    @Override
    public T removeCard() {
        if (cards.size() &lt; 1) {
            throw new IllegalStateException("Deck is empty");
        }
        return cards.remove(0);
    }

    @Override
    public T removeCard(int index) {
        if (cards.size() - 1 &lt; index) {
            throw new IndexOutOfBoundsException();
        }
        return cards.remove(index);
    }

    @Override
    public void addCards(List&lt;T&gt; cards) {
        this.cards.addAll(cards);
    }

    @Override
    public void addCard(T card) {
        cards.add(card);
    }

    @Override
    public void addCard(int index, T card) {
        cards.add(index, card);
    }

    @Override
    public String toString() {
        return cards.toString();
    }

    public List&lt;T&gt; getCards() {
        return cards;
    }
}
</code></pre> 
              <p><strong>BaseDeck.java</strong></p> 
              <pre><code>package com.tn.deck;

import java.util.Arrays;
import java.util.Collections;
import java.util.Comparator;
import java.util.List;


public class BaseDeck&lt;T extends Card&gt; extends AbstractCardOperation&lt;T&gt; implements Deck&lt;T&gt; {

    public BaseDeck() {
        super();
    }

    public BaseDeck(List&lt;T&gt; deck) {
        super(deck);
    }

    public BaseDeck(List&lt;T&gt; deck, int numberOfDecks) {
        super(deck, numberOfDecks);
    }

    @Override
    public void shuffle() {
        Collections.shuffle(super.cards);
    }

    @Override
    public void sort(Comparator&lt;T&gt; sort) {
        super.cards.sort(sort);
    }

    @Override
    public &lt;S extends Receivable&lt;T&gt;&gt; void addCardToPlayers(S[] players) {
        Arrays.asList(players).forEach(player -&gt; player.addCard(super.cards.remove(0)));
    }


    @SafeVarargs
    public static &lt;T extends Card&gt; void print(Deck&lt;T&gt;... decks) {
        Arrays.stream(decks).forEach(deck -&gt; System.out.println(deck.toString()));
    }

    public int getSize() {
        return super.cards.size();
    }
}
</code></pre> 
              <p><strong>BaseHand.java</strong></p> 
              <pre><code>package com.tn.deck;

import java.util.List;


public class BaseHand&lt;T extends Card&gt; extends AbstractCardOperation&lt;T&gt; implements Hand&lt;T&gt; {

    public BaseHand(List&lt;T&gt; cards) {
        super(cards);
    }

    @Override
    public int calculate() {
        return 0;
    }

    @Override
    public void addCardToDeck(int index, List&lt;? super Card&gt; deck) {
        deck.add(super.cards.remove(index));
    }

    @Override
    public boolean isEquals(Hand&lt;T&gt; other) {
        return super.cards.equals(other);
    }
}
</code></pre>

             <div class="post-text" itemprop="text"> 
              <blockquote> 
               <pre><code>    public void addCardToDeck(int index, List&lt;? super Card&gt; deck) {
        deck.add(super.cards.remove(index));
    }
</code></pre> 
              </blockquote> 
              <p>If this is how the method is supposed to work, consider changing the name from <code>add</code> to <code>move</code>. </p> 
              <pre><code>    public void moveCardToDeck(int index, List&lt;? super Card&gt; deck) {
        deck.add(super.cards.remove(index));
    }
</code></pre> 
              <p>This way it is clearer that the method removes the card from the hand as well as adding it to the deck. </p> 
              <p>You also may want to add some Javadoc to comment this behavior so that people who implement the method understand that this should do both things. </p> 
              <p>There is a school of thought that people shouldn't use extension in Java, only interfaces. The problem with abstract classes is that they confuse the location of implementation. E.g. a method could be defined in <code>DerivedHand</code>, <code>BaseHand</code>, and <code>AbstractCardOperation</code>. As the number of extensions and implementations increase, this only gets worse. Another problem is that you can only chain extensions. There's no multiple inheritance of abstract classes. </p> 
              <p>Anyway, I don't want to pick sides in that particular holy war today. I just didn't want to start using an abstract class before noting that there are arguments against it. In this case, you already have an abstract class, so there isn't much point in not using it for things unless we're going to get rid of it. </p> 
              <p>The name <code>AbstractCardOperation</code> is not quite correct. It doesn't represent a single operation but a number of methods. If it weren't abstract, you might call it <code>CardOperations</code> or <code>CardUtilities</code>. As is, maybe <code>AbstractCardCollection</code> would be better. That would fit with extensions by <code>BaseHand</code> and <code>BaseDeck</code>. </p> 
              <p>Why is <code>deck</code> a <code>List</code> of <code>Card</code> objects? Why not an <code>AbstractCardCollection</code>? To pass a <code>List</code> to the method, you'll have to excavate the <code>cards</code> from your <code>AbstractCardCollection</code> (as your <code>BaseDeck</code> type <code>extends AbstractCardCollection</code>). And what if you are playing a game where cards get passed from player to player? And why isn't the move operation on <code>AbstractCardCollection</code> rather than <code>Hand</code>? </p> 
              <p>Be careful you don't make too many base classes and interfaces. For example, when do you use <code>Sendable</code> separate from <code>Receivable</code> and <code>AbstractCardOperation</code>? </p> 
              <p>It might be useful to implement a remove-only <code>Hand</code> or <code>Deck</code>, but it would be difficult to do here. They couldn't extend <code>BaseHand</code> and <code>BaseDeck</code> or <code>AbstractCardCollection</code>, because all of those have the <code>Receivable</code> operations. </p> 
              <blockquote> 
               <pre><code>    public boolean isEquals(Hand&lt;T&gt; other) {
        return super.cards.equals(other);
    }
</code></pre> 
              </blockquote> 
              <p>Does this even compile? The <code>cards</code> object is a <code>List</code>, while <code>other</code> is a <code>Hand</code>. If it does compile, it is almost certainly going to <code>return false</code> because it is going to go back to <code>Object.equals</code>. It would only <code>return true</code> if the <code>List</code> and the <code>Hand</code> were the same object. </p> 
              <p>Equality doesn't seem like a natural operation for <code>Hand</code>. Yes, there may be times when it is useful, but for many games, it is meaningless. For example, Go Fish. You have a hand, but you would never compare it to another player's hand. The operation there is to check if you have a card of a particular rank. In Blackjack, it's less if two hands are equal and more if one is better than the other. If the player's hand is better than the dealer's, then the player wins. Otherwise the dealer does. </p>

            <td class="answercell"> 
             <div class="post-text" itemprop="text"> 
              <ul> 
               <li><p>Your type <code>Card&lt;T&gt;</code> is generic. You have a <em>Card of T</em>. When I hear that, it doesn't make sense to me. A card is a card. Why do you need the <code>T</code> here? Could you have a <em>Card of Wine</em>, a <em>Card of Deck</em>?</p></li> 
               <li><p>A card in itself cannot be consecutive. Standard playing cards have a <em>rank</em> and a <em>suit</em>, but which ranks are consecutive depends on the game. In Skat, the 10 is ranked between A and K, while in other games it is ranked between J and 9.</p></li> 
               <li><p>In the type <code>Deck</code>, you use the type <code>Card</code>, but without specifying the <em>type</em> of card, i.e. you left out the <code>T</code> in <code>Card&lt;T&gt;</code>.</p></li> 
               <li><p>The method <code>Deck.addCardToPlayers</code> should rather be called <code>deal</code>. It should only deal a single card to a single player.</p></li> 
               <li><p>In <code>Deck.addCardToPlayers</code>, why is the receiving type called <code>S</code> instead of <code>R</code> (receiver) or <code>P</code> (player)?</p></li> 
               <li><p>Not every <code>Hand</code> can be evaluated to a simple <code>int</code>. In Poker, you can do some bit fiddling to map all possible hands into a 32-bit number so that different hands can be compared easily. This is not possible for all games.</p></li> 
               <li><p>A general note: Before creating so many abstract classes and interfaces, you should implement 3 different games using the most concrete types possible. I would start with these types:</p> 
                <ul> 
                 <li><code>enum Suit { CLUBS, DIAMONDS, HEARTS, SPADES }</code></li> 
                 <li><code>enum Rank { ACE, TWO, THREE, ... }</code></li> 
                 <li><code>class Card { Suit suit; Rank rank; }</code></li> 
                 <li><code>class Hand { List&lt;Card&gt; cards; }</code></li> 
                 <li><code>class Deck { List&lt;Card&gt; cards; }</code></li> 
                </ul> <p>See how far you get with these simple definitions. If you need more, add them. Learn about the YAGNI principle.</p> <p>When you have implemented 3 different games, have a look back and see which abstractions you really need. Only make these abstract.</p></li> 
              </ul>
