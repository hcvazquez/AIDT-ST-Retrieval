         <div class="titlepage">
          <div>
           <div>
            <h3 class="title"><a id="_why_is_packaging_needed"></a>7.1.&nbsp;Why is Packaging Needed?</h3>
           </div>
          </div>
         </div>
         <div class="itemizedlist">
          <ul class="itemizedlist" type="disc">
           <li class="listitem"><p class="simpara"> Packaging protects against name collisions. </p>
            <div class="itemizedlist">
             <ul class="itemizedlist" type="circle">
              <li class="listitem"> With thousands of Java libraries available, it’s inevitable that class and interface names will be duplicated. </li>
              <li class="listitem"> Packages allow you to use two different classes or interfaces with the same name. </li>
             </ul>
            </div></li>
           <li class="listitem"><p class="simpara"> Packages hide implementation that spans multiple classes (multi-class encapsulation). </p>
            <div class="itemizedlist">
             <ul class="itemizedlist" type="circle">
              <li class="listitem"> A set of classes might need access to each other, but still be hidden from the outside world. </li>
              <li class="listitem"> Java packages support access levels other than <code class="literal">public</code> and <code class="literal">private</code> to support information sharing and class visibility. </li>
             </ul>
            </div></li>
           <li class="listitem"><p class="simpara"> Packages help organize source code. </p>
            <div class="itemizedlist">
             <ul class="itemizedlist" type="circle">
              <li class="listitem"> Having all source files in the same directory becomes hard to manage as the projects grow. </li>
              <li class="listitem"> Packages allow you to group together related classes and interfaces into common directories. </li>
             </ul>
            </div></li>
          </ul>
         </div>

            <h3 class="title"><a id="_packages_in_java"></a>7.2.&nbsp;Packages in Java</h3>
           </div>
          </div>
         </div>
         <div class="itemizedlist">
          <ul class="itemizedlist" type="disc">
           <li class="listitem"><p class="simpara"> A Java package name consists of a set of name components separated by periods (<code class="literal">.</code>). </p>
            <div class="itemizedlist">
             <ul class="itemizedlist" type="circle">
              <li class="listitem"> Each name component must be a valid Java identifier. </li>
              <li class="listitem"><p class="simpara"> A component name must not start with an upper-case letter. </p>
               <div class="tip" title="Tip" style="margin-left: 0; margin-right: 10%;">
                <table border="0" summary="Tip">
                 <tbody>
                  <tr>
                   <td rowspan="2" align="center" valign="top" width="25"><img alt="[Tip]" src="https://www.protechtraining.com/static/bookshelf/images/xtip.png.pagespeed.ic.jUeiTE9kDZ.png" data-pagespeed-url-hash="414065525" onload="pagespeed.CriticalImages.checkImageForCriticality(this);"></td>
                   <th align="left">Tip</th>
                  </tr>
                  <tr>
                   <td align="left" valign="top"><p>Best practice is for each name component to be all lower-case.</p></td>
                  </tr>
                 </tbody>
                </table>
               </div></li>
             </ul>
            </div></li>
           <li class="listitem"><p class="simpara"> The package name corresponds to a directory structure on disk where the classes are stored. </p>
            <div class="itemizedlist">
             <ul class="itemizedlist" type="circle">
              <li class="listitem"> For example, the class and interface files for <code class="literal">org.xml.sax</code> are located in the directory <code class="literal">org/xml/sax</code>, relative to the directory where Java looks for classes (we’ll discuss this later in this module). </li>
             </ul>
            </div></li>
           <li class="listitem"><p class="simpara"> To add a class or interface to a package: </p>
            <div class="itemizedlist">
             <ul class="itemizedlist" type="circle">
              <li class="listitem"> Add <code class="literal">package </code><span class="emphasis"><em><code class="literal">myPackageName</code></em></span><code class="literal">;</code> as the first Java statement of the source file </li>
              <li class="listitem"> In your development directory, store the source file in a directory structure corresponding to your package name. </li>
             </ul>
            </div></li>
          </ul>
         </div>
         <p></p>
         <div class="tip" title="Tip" style="margin-left: 0; margin-right: 10%;">
          <table border="0" summary="Tip">
           <tbody>
            <tr>
             <td rowspan="2" align="center" valign="top" width="25"><img alt="[Tip]" src="https://www.protechtraining.com/static/bookshelf/images/xtip.png.pagespeed.ic.jUeiTE9kDZ.png" data-pagespeed-url-hash="414065525" onload="pagespeed.CriticalImages.checkImageForCriticality(this);"></td>
             <th align="left">Tip</th>
            </tr>
            <tr>
             <td align="left" valign="top"><p>Java does not require you to store your <code class="literal">.java</code> source files in in package-based directories, but it is a common convention. However, once the source files are compiled, the generated <code class="literal">.class</code> files are (and must be) stored in the package-based directories.</p><p>It’s also common to have separate, parallel directory hierarchies for your <code class="literal">.java</code> and <code class="literal">.class</code> files reflecting the package structure. Having your source files separate from your class files is convenient for maintaining the source code in a source code control system without having to explicitly filter out the generated class files. Having the class files in their own independent directory hierarchy is also useful for generating JAR files, as discussed later in this module.</p><p>Most modern IDEs can create and manage your source and class file directory hierarchies automatically for you.</p></td>
            </tr>
           </tbody>
          </table>
         </div>
         <div class="caution" title="Caution" style="margin-left: 0; margin-right: 10%;">
          <table border="0" summary="Caution">
           <tbody>
            <tr>
             <td rowspan="2" align="center" valign="top" width="25"><img alt="[Caution]" src="https://www.protechtraining.com/static/bookshelf/images/xcaution.png.pagespeed.ic.MODSYbOHxU.png" data-pagespeed-url-hash="618581265" onload="pagespeed.CriticalImages.checkImageForCriticality(this);"></td>
             <th align="left">Caution</th>
            </tr>
            <tr>
             <td align="left" valign="top"><p>If you don’t explicitly specify a package, your classes and interfaces end up in an unnamed packaged, also known as the <span class="emphasis"><em>default package</em></span>. Best practice is not to use the default package for any production code.</p></td>
            </tr>
           </tbody>
          </table>
         </div>
        </div>
        <div class="section" title="7.3.&nbsp;Sub-Packages in Java">
         <div class="titlepage">
          <div>
           <div>
            <h3 class="title"><a id="_sub_packages_in_java"></a>7.3.&nbsp;Sub-Packages in Java</h3>
           </div>
          </div>
         </div>
         <div class="itemizedlist">
          <ul class="itemizedlist" type="disc">
           <li class="listitem"><p class="simpara"> You can construct hierarchical package names, for example <code class="literal">org.xml.sax</code> and <code class="literal">org.xml.sax.helpers</code>. </p>
            <div class="itemizedlist">
             <ul class="itemizedlist" type="circle">
              <li class="listitem"> However, Java treats these as two <span class="strong"><strong>independent packages</strong></span>, one with class files in <code class="literal">org/xml/sax</code> and the other with class files in <code class="literal">org/xml/helpers</code>. </li>
              <li class="listitem"> The “information sharing” features of Java packages do not apply across these independent packages. </li>
              <li class="listitem"> Still, hierarchical package names are useful for grouping together related sets of classes. </li>
             </ul>
            </div></li>
          </ul>
         </div>
         <p></p>
        </div>
        <div class="section" title="7.4.&nbsp;Package Naming Conventions">
         <div class="titlepage">
          <div>
           <div>
            <h3 class="title"><a id="_package_naming_conventions"></a>7.4.&nbsp;Package Naming Conventions</h3>
           </div>
          </div>
         </div>
         <div class="itemizedlist">
          <ul class="itemizedlist" type="disc">
           <li class="listitem"><p class="simpara"> To prevent package name collisions, the convention is for organizations to use their reversed Internet domain names to begin their package names. </p>
            <div class="itemizedlist">
             <ul class="itemizedlist" type="circle">
              <li class="listitem"> For example, <code class="literal">com.marakana</code> </li>
              <li class="listitem"> If the Internet domain name contains an invalid character, such as a hyphen, the convention is to replace the invalid character with an underscore. For example, <code class="literal">com.avia_training</code> </li>
              <li class="listitem"> If a domain name component starts with a digit or consists of a reserved Java keyword, the convention is to add an underscore to the component name. </li>
             </ul>
            </div></li>
           <li class="listitem"><p class="simpara"> Organizations often create sub-packages to reflect business units, product lines, etc. </p>
            <div class="itemizedlist">
             <ul class="itemizedlist" type="circle">
              <li class="listitem"> For example, <code class="literal">com.marakana.android</code> </li>
             </ul>
            </div></li>
           <li class="listitem"> Package names starting with <code class="literal">java.</code> and <code class="literal">javax.</code> are reserved for Java libraries. </li>
          </ul>
         </div>
         <p></p>
         <p>Java does not do any explicit protection of package namespaces. This means that it is possible for an organization ABC to create a class in package <code class="literal">com.xyz</code> even though the domain <code class="literal">xyz.com</code> belongs to an organization called XYZ. However in practice organizations do not abuse each other’s namespaces.</p>
        </div>
        <div class="section" title="7.5.&nbsp;Using Package Members: Qualified Names">
         <div class="titlepage">
          <div>
           <div>
            <h3 class="title"><a id="_using_package_members_qualified_names"></a>7.5.&nbsp;Using Package Members: Qualified Names</h3>
           </div>
          </div>
         </div>
         <div class="itemizedlist">
          <ul class="itemizedlist" type="disc">
           <li class="listitem"><p class="simpara"> The <span class="emphasis"><em>fully-qualified name</em></span> for a class or interface is the package name followed by the class/interface name, separated by a period (<code class="literal">.</code>). </p>
            <div class="itemizedlist">
             <ul class="itemizedlist" type="circle">
              <li class="listitem"> For example, <code class="literal">com.marakana.utils.MyClass</code> </li>
             </ul>
            </div></li>
           <li class="listitem"> Code outside the package can reference public classes and interfaces of a package using their fully-qualified names. </li>
           <li class="listitem"> Code within the package can reference classes and interfaces of the package by their simple names, without package qualification. </li>
           <li class="listitem"><p class="simpara"> To execute a Java application whose class is contained within a package, you must use its fully-qualified name. </p>
            <div class="itemizedlist">
             <ul class="itemizedlist" type="circle">
              <li class="listitem"><p class="simpara"> For example, to run <code class="literal">ShapeDemo.main()</code> in the package <code class="literal">shape</code> you must invoke it as: </p><pre class="literallayout">java shape.ShapeDemo</pre></li>
             </ul>
            </div></li>
          </ul>
         </div>
        </div>
        <div class="section" title="7.6.&nbsp;Importing Package Members">
         <div class="titlepage">
          <div>
           <div>
            <h3 class="title"><a id="_importing_package_members"></a>7.6.&nbsp;Importing Package Members</h3>
           </div>
          </div>
         </div>
         <div class="itemizedlist">
          <ul class="itemizedlist" type="disc">
           <li class="listitem"><p class="simpara"> Instead of using fully-qualified names for classes and interfaces, you can <span class="emphasis"><em>import</em></span> the package member. </p>
            <div class="itemizedlist">
             <ul class="itemizedlist" type="circle">
              <li class="listitem"><p class="simpara"> To import a specific member from a package, include an <code class="literal">import</code> statement specifying the fully-qualified name of that member. For example: </p><pre class="screen">import com.marakana.utils.MyClass;</pre><p class="simpara">You can then use the simple (unqualified) name of the member throughout the rest of the source file.</p></li>
              <li class="listitem"><p class="simpara"> You can import <span class="strong"><strong>all</strong></span> of the members of a package by importing the package with an asterisk (<code class="literal">*</code>) wildcard. For example: </p><pre class="screen">import com.marakana.utils.*;</pre><p class="simpara">You can then use the simple (unqualified) name of all the package members throughout the rest of the source file.</p>
               <div class="important" title="Important" style="margin-left: 0; margin-right: 10%;">
                <table border="0" summary="Important">
                 <tbody>
                  <tr>
                   <td rowspan="2" align="center" valign="top" width="25"><img alt="[Important]" src="https://www.protechtraining.com/static/bookshelf/images/ximportant.png.pagespeed.ic.6hrQF0Txx7.jpg" data-pagespeed-url-hash="2477373160" onload="pagespeed.CriticalImages.checkImageForCriticality(this);"></td>
                   <th align="left">Important</th>
                  </tr>
                  <tr>
                   <td align="left" valign="top"><p>Using the wildcard <code class="literal">import</code> does <span class="strong"><strong>not</strong></span> import any sub-packages or their members.</p></td>
                  </tr>
                 </tbody>
                </table>
               </div></li>
             </ul>
            </div></li>
           <li class="listitem"> The <code class="literal">import</code> statements must appear after any <code class="literal">package</code> statement and before all class/interface definitions in your file. </li>
          </ul>
         </div>
         <div class="note" title="Note" style="margin-left: 0; margin-right: 10%;">
          <table border="0" summary="Note">
           <tbody>
            <tr>
             <td rowspan="2" align="center" valign="top" width="25"><img alt="[Note]" src="https://www.protechtraining.com/static/bookshelf/images/xnote.png.pagespeed.ic.PV1YZjO3DD.jpg" data-pagespeed-url-hash="3147115774" onload="pagespeed.CriticalImages.checkImageForCriticality(this);"></td>
             <th align="left">Note</th>
            </tr>
            <tr>
             <td align="left" valign="top"><p>The <code class="literal">java.lang</code> package is imported automatically into all source files.</p></td>
            </tr>
           </tbody>
          </table>
         </div>
         <p></p>
         <p>In case there are two or more classes with the same name but defined in different imported (or assumed) packages, then those classes must be referenced by their fully-qualified-class-name (FQCN).</p>
         <p>For example, you could define:</p>
         <pre class="literallayout">com.mycompany.calc.AdditionOperation,</pre>
         <p>but someone else could implement:</p>
         <pre class="literallayout">com.othercompany.calculator.AdditionOperation.</pre>
         <p>If you wanted to use both addition operations from a class defined in <code class="literal">com.mycompany.calc</code> package, then doing</p>
         <pre class="literallayout">import com.othercompany.calculator.AdditionOperation;</pre>
         <p>would make AdditionOperation ambiguous. Instead of importing it, you would reference it by its FQCN.</p>
         <div class="caution" title="Caution" style="margin-left: 0; margin-right: 10%;">
          <table border="0" summary="Caution">
           <tbody>
            <tr>
             <td rowspan="2" align="center" valign="top" width="25"><img alt="[Caution]" src="https://www.protechtraining.com/static/bookshelf/images/xcaution.png.pagespeed.ic.MODSYbOHxU.png" data-pagespeed-url-hash="618581265" onload="pagespeed.CriticalImages.checkImageForCriticality(this);"></td>
             <th align="left">Caution</th>
            </tr>
            <tr>
             <td align="left" valign="top"><p>Even though importing an entire package using the asterisk (<code class="literal">*</code>) notation is convenient, it is not generally recommended?—?especially when multiple packages are imported. It makes it harder to track down the classes if you do not know exactly which package they come from. Modern IDEs help with resolving classes (as well as with organizing imports), but you should not assume that everyone will always have an IDE handy.</p></td>
            </tr>
           </tbody>
          </table>
         </div>
        </div>
        <div class="section" title="7.7.&nbsp;Static Imports">
         <div class="titlepage">
          <div>
           <div>
            <h3 class="title"><a id="_static_imports"></a>7.7.&nbsp;Static Imports</h3>
           </div>
          </div>
         </div>
         <div class="itemizedlist">
          <ul class="itemizedlist" type="disc">
           <li class="listitem"><p class="simpara"> Prior to Java 5, you always had to access static fields and methods qualified by their class name. </p>
            <div class="itemizedlist">
             <ul class="itemizedlist" type="circle">
              <li class="listitem"><p class="simpara"> For example: </p><pre class="programlisting">double c= Math.PI * Math.pow(r, 2.0);</pre></li>
             </ul>
            </div></li>
           <li class="listitem"><p class="simpara"> Java 5 introduced <code class="literal">import static</code> to import static fields and methods. </p>
            <div class="itemizedlist">
             <ul class="itemizedlist" type="circle">
              <li class="listitem"><p class="simpara"> For example: </p><pre class="programlisting">import static java.lang.Math.PI;    // A single field
import static java.lang.Math.*;     // All static fields and methods</pre></li>
              <li class="listitem"><p class="simpara"> Once imported, you can use them as if they were defined locally: </p><pre class="programlisting">double c= PI * pow(r, 2.0);</pre></li>
             </ul>
            </div></li>
          </ul>
         </div>
         <p></p>
        </div>
        <div class="section" title="7.8.&nbsp;Access Modifiers and Packages">

          </div>
         </div>

        <div class="section" title="7.9.&nbsp;The Class Path">
         <div class="titlepage">
          <div>
           <div>
            <h3 class="title"><a id="_the_class_path"></a>7.9.&nbsp;The Class Path</h3>
           </div>
          </div>
         </div>
         <div class="itemizedlist">
          <ul class="itemizedlist" type="disc">
           <li class="listitem"> The <span class="emphasis"><em>class path</em></span> is the path that the Java runtime environment searches for classes and other resource files. </li>
           <li class="listitem"> By default, Java looks for classes in only the present working directory (<code class="literal">.</code>). </li>
           <li class="listitem"><p class="simpara"> To access classes at runtime outside the present working directory, there are three options: </p>
            <div class="itemizedlist">
             <ul class="itemizedlist" type="circle">
              <li class="listitem"> Set an environmental variable named <code class="literal">CLASSPATH</code> with a colon-separated list of class directories, JAR files, and/or Zip files. (This should be a semicolon-separated list on Windows.) </li>
              <li class="listitem"> Use use the <code class="literal">-classpath</code> option (<code class="literal">-cp</code> is an alias) with <code class="literal">java</code> and <code class="literal">javac</code> to override the class path with a colon/semicolon-separated list of class directories, JAR files, and/or Zip files. </li>
              <li class="listitem"> Copy classes to <code class="literal">$JAVA_HOME/lib/ext/</code> (this is typically used only for Java extensions). </li>
             </ul>
            </div></li>
          </ul>
         </div>
         <div class="important" title="Important" style="margin-left: 0; margin-right: 10%;">
          <table border="0" summary="Important">
           <tbody>
            <tr>
             <td rowspan="2" align="center" valign="top" width="25"><img alt="[Important]" src="https://www.protechtraining.com/static/bookshelf/images/ximportant.png.pagespeed.ic.6hrQF0Txx7.jpg" data-pagespeed-url-hash="2477373160" onload="pagespeed.CriticalImages.checkImageForCriticality(this);"></td>
             <th align="left">Important</th>
            </tr>
            <tr>
             <td align="left" valign="top"><p>The class path should include only the root directory of a package directory hierarchy, not the individual package directories.</p></td>
            </tr>
           </tbody>
          </table>
         </div>
         <p></p>
         <p>For example, to run <code class="literal">calc.CalculatorDemo</code> located in <code class="literal">/home/me/classes/calc/CalculatorDemo.class</code>, you could do:</p>
         <pre class="literallayout">cd /home/me/classes
java calc.CalculatorDemo</pre>
         <p>or (from any directory):</p>
         <pre class="literallayout">export CLASSPATH=/home/me/classes
java calc.CalculatorDemo</pre>
         <p>or (from any directory):</p>
         <pre class="literallayout">java -classpath /home/me/classes calc.CalculatorDemo</pre>
         <p>or (from any directory):</p>
         <pre class="literallayout">cp -r /home/me/classes/calc $JAVA_HOME/lib/ext/.
java calc.CalculatorDemo</pre>
         <p>It is important to notice the distinction between class path locations and package directories. Package-based directories should never be included within the class path and vice-versa.</p>
         <p>This means that these would not work:</p>
         <pre class="literallayout">java -classpath /home/me/classes/calc CalculatorDemo
java -classpath /home/me classes.calc.CalculatorDemo</pre>
        </div>
        <div class="section" title="7.10.&nbsp;Java Archive (JAR) Files">
         <div class="titlepage">
          <div>
           <div>
            <h3 class="title"><a id="_java_archive_jar_files"></a>7.10.&nbsp;Java Archive (JAR) Files</h3>
           </div>
          </div>
         </div>
         <div class="itemizedlist">
          <ul class="itemizedlist" type="disc">
           <li class="listitem"><p class="simpara"> A Java Archive (JAR) file packages multiple classes in a single compressed file. </p>
            <div class="itemizedlist">
             <ul class="itemizedlist" type="circle">
              <li class="listitem"> Based on the ZIP file format </li>
              <li class="listitem"> Easier and more efficient management of binaries </li>
              <li class="listitem"> Also known as Java library (.jar extension) </li>
              <li class="listitem"> Must preserve package directory structure </li>
             </ul>
            </div></li>
           <li class="listitem"><p class="simpara"> A JAR file can include additional meta-data in a manifest file at the pathname <code class="literal">META-INF/MANIFEST.MF</code>. </p>
            <div class="itemizedlist">
             <ul class="itemizedlist" type="circle">
              <li class="listitem"> The manifest is a text file containing <code class="literal">name: value</code> formatted entries. </li>
              <li class="listitem"><p class="simpara"> For example, you can make an executable JAR by including a <code class="literal">Main-Class</code> entry whose value is the fully-qualified name of a class containing a <code class="literal">main()</code> method. You could then execute the JAR as: </p><pre class="literallayout">java -jar &lt;file&gt;.jar</pre></li>
             </ul>
            </div></li>
          </ul>
         </div>
         <p></p>
         <p>JAR files internally must look like package directories. For example, a class <code class="literal">com.myco.calc.CalculatorDemo</code> would be stored within <code class="literal">calculator.jar</code> file just like it would be stored on disk: <code class="literal">com/myco/calc/CalculatorDemo.class</code>.</p>
         <p>JAR files are referenced in the <code class="literal">CLASSPATH</code> by their actual name, not by the name of the directory in which they are contained.</p>
         <p>To run <code class="literal">com.myco.calc.CalculatorDemo</code> you would do:</p>
         <pre class="literallayout">java -classpath calculator.jar com.myco.calc.CalculatorDemo</pre>
         <p>To make <code class="literal">calculator.jar</code> executable:</p>
         <div class="orderedlist">
          <ol class="orderedlist" type="1">
           <li class="listitem"> Create a file: <code class="literal">META-INF/MANIFEST.MF</code> </li>
           <li class="listitem"><p class="simpara"> Add a line: </p><pre class="literallayout">Main-Class: com.myco.calc.CalculatorDemo</pre></li>
           <li class="listitem"> Include <code class="literal">META-INF/MANIFEST.MF</code> in <code class="literal">calculator.jar</code> </li>
           <li class="listitem"><p class="simpara"> Run with </p><pre class="literallayout">java -jar calculator.jar</pre></li>
          </ol>
         </div>
        </div>
        <div class="section" title="7.11.&nbsp;The jar Command-Line Tool Examples">
         <div class="titlepage">
          <div>
           <div>
            <h3 class="title"><a id="_the_literal_jar_literal_command_line_tool_examples"></a>7.11.&nbsp;The <code class="literal">jar</code> Command-Line Tool Examples</h3>
           </div>
          </div>
         </div>
         <div class="itemizedlist">
          <ul class="itemizedlist" type="disc">
           <li class="listitem"><p class="simpara"> Create a JAR file </p><pre class="literallayout">cd classes; jar -cvf shape.jar shape
jar -cvf shape.jar -C classes shape</pre></li>
           <li class="listitem"><p class="simpara"> View the contents of a JAR file </p><pre class="literallayout">jar -tvf shape.jar</pre></li>
           <li class="listitem"><p class="simpara"> Extract the contents of JAR file </p><pre class="literallayout">jar -xvf shape.jar
jar -xvf shape.jar META-INF/MANIFEST.MF</pre></li>
           <li class="listitem"><p class="simpara"> Update the JAR file (e.g., add to the manifest) </p><pre class="literallayout">jar -uvmf MainClass.txt shape.jar</pre></li>
          </ul>
         </div>
         <p></p>
         <p>You will find that the <code class="literal">jar</code> command has a usage similar to the Unix <code class="literal">tar</code> command:</p>
         <pre class="screen">Usage: jar {ctxui}[vfm0Me] [jar-file] [manifest-file] [entry-point] [-C dir] files ...
Options:
    -c  create new archive
    -t  list table of contents for archive
    -x  extract named (or all) files from archive
    -u  update existing archive
    -v  generate verbose output on standard output
    -f  specify archive file name
    -m  include manifest information from specified manifest file
    -e  specify application entry point for stand-alone application
        bundled into an executable jar file
    -0  store only; use no ZIP compression
    -M  do not create a manifest file for the entries
    -i  generate index information for the specified jar files
    -C  change to the specified directory and include the following file
If any file is a directory then it is processed recursively.
The manifest file name, the archive file name and the entry point name are
specified in the same order as the 'm', 'f' and 'e' flags.

Example 1: to archive two class files into an archive called classes.jar:
       jar cvf classes.jar Foo.class Bar.class
Example 2: use an existing manifest file 'mymanifest' and archive all the
           files in the foo/ directory into 'classes.jar':
       jar cvfm classes.jar mymanifest -C foo/ .</pre>
