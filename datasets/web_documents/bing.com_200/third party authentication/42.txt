       <blockquote> 
        <p>This is the third of a five-part series on building a <a href="https://slack.com/">Slack</a> clone using <a href="https://www.meteor.com/">Meteor</a>. The aim of these tutorials are not just for you to blindly follow instructions, but it's our hope that you'll understand the thought process and reasoning behind the architecture.</p> 
       </blockquote> 
       <p>Right now, everyone is anonymous (everyone is scotch!), and everyone talks on the same channel. Let's do something about that.</p> 
       <h3>User Accounts</h3> 
       <p>As with everything Meteor - there's a package for that. The <code>accounts-base</code> and <code>accounts-password</code> packages provide the capabilities for login, logout, account creation, email validation, password recovery. There's also the <code>accounts-ui</code> and <code>accounts-ui-unstyled</code> which provides you with a login / register form that you can just drop into your application.</p> 
       <p>Using these packages, user data would be stored in the <code>users</code> collection, so you must make sure your collection names do not clash with this.</p> 
       <p>Let's install the packages.</p> 
       <pre><code class="language-bash">
$ meteor add accounts-base accounts-password accounts-ui
</code></pre> 
       <p>And lets drop in the login form template, provided by the <code>accounts-ui</code> package, into our template.</p> 
       <pre><code class="language-javascript">
{{&gt; loginButtons}}
</code></pre> 
       <p>We can see that these packages created a <code>users</code> collection in our database.</p> 
       <p><a href="https://cask.scotch.io/2015/05/meteor-users-collection.png"><img src="https://cask.scotch.io/2015/05/meteor-users-collection-364x250.png" alt="An empty users collection" width="364" height="250" class="alignnone size-medium wp-image-9492"></a></p> 
       <p>And now let's create a dummy user.</p> 
       <p><a href="https://cask.scotch.io/2015/05/meteor-create-account.png"><img src="https://cask.scotch.io/2015/05/meteor-create-account-445x250.png" alt="Creating a dummy user using the UI provided by the accounts-ui package" width="445" height="250" class="alignnone size-medium wp-image-9493"></a></p> 
       <p>And now we look back into our collection, we find an entry.</p> 
       <p><a href="https://cask.scotch.io/2015/05/robomongo-new-user.png"><img src="https://cask.scotch.io/2015/05/robomongo-new-user-364x250.png" alt="The user entry as seen by Robomongo" width="364" height="250" class="alignnone size-medium wp-image-9494"></a></p> 
       <p>And here it is in BSON format.</p> 
       <pre><code class="language-javascript">
{
    "_id": "e6AN2jGt9T9jrcwDt",
    createdAt": ISODate("2015-05-24T15:48:21.547Z"),
    "services": {
        "password": {
            "bcrypt": "$2a$10$6CDufWjV.OD/mQIv0MOxkukXzWgE0UZVGLAUnBF8vRHkjRKl4NoLy"
        },
        "resume": {
            "loginTokens": []
        }
    },
    "emails": [
        {
            "address": "dan@danyll.com",
            "verified": false
        }
    ]
}
</code></pre> 
       <p>Let's go through each field to get a better understanding.</p> 
       <p>The <code>_id</code> field is a unique ID used to identify the document. You can provide this value, but if not, Meteor will generate a random alphanumeric string (e.g. <code>e6AN2jGt9T9jrcwDt</code>). You can retrieve the <code>_id</code> of a user by running <code>Meteor.userId()</code>.</p> 
       <p>The <code>createdAt</code> field stores an <code>ISODate</code> data type of when the user was created. This is not a valid JSON datatype, and is one of the reasons Mongo uses <a href="http://docs.mongodb.org/manual/reference/bson-types/">BSON</a> instead of JSON.</p> 
       <p>The <code>services</code> field contains data that are required for the account service providers. Right now, it is an object with a <code>password</code> field, because we are using the <code>accounts-password</code> package. Later on when we add the <code>accounts-github</code> package, you'll see the <code>github</code> field being written inside the <code>services</code> property. The <code>services.resume</code> property keeps track of all the login sessions the user has.</p> 
       <p>The <code>emails</code> field specifies all the emails that belongs to a user. In the Accounts system, each email can only belong to one user.</p> 
       <h5>Configuring the Accounts UI</h5> 
       <p>At the moment, a user can sign up with just an email and password. But mimicking Slack, we want each user to have a username as well as an email address. This is easily done by configuring <code>Accounts.ui</code>. You may put this inside any client-side code.</p> 
       <pre><code class="language-javascript">
Accounts.ui.config({
    passwordSignupFields: 'USERNAME_AND_EMAIL'
});
</code></pre> 
       <p>And now when we create a new user, we see the <code>username</code> field in the document.</p> 
       <p><a href="https://cask.scotch.io/2015/05/accounts-ui-username.png"><img src="https://cask.scotch.io/2015/05/accounts-ui-username-241x250.png" alt="An additional 'username' field shows up" width="241" height="250" class="alignnone size-medium wp-image-9495"></a></p> 
       <blockquote> 
        <p>Apart from <code>USERNAME_AND_EMAIL</code>, there's also the <code>USERNAME_AND_OPTIONAL_EMAIL</code>, <code>USERNAME_ONLY</code> and <code>EMAIL_ONLY</code> options. <code>EMAIL_ONLY</code> is the default.</p> 
       </blockquote> 
       <p>The <code>accounts-base</code> package will check the <code>username</code> field to ensure it's unique, as well as check the email has not already been used by another user, before inserting into the <code>users</code> collection.</p> 
       <blockquote> 
        <p>The <code>accounts-ui</code> package provides the login form, as well as calling methods to register/login our users. But for production applications, you might want to create your own UI and set your own account management logic. To do that, you must call the <a href="http://docs.meteor.com/#/full/accounts_api">Accounts API</a> and <a href="http://docs.meteor.com/#/full/accounts_passwords">Passwords API</a> yourself. For example, you'd manually call <a href="http://docs.meteor.com/#/full/accounts_createuser"><code>Accounts.createUser()</code></a> to create the user, and customize the creation logic with <a href="http://docs.meteor.com/#/full/accounts_oncreateuser"><code>Accounts.onCreateUser()</code></a></p> 
       </blockquote> 
       <h3>Third-Party Integration</h3> 
       <p>How many usernames and passwords must you keep on top of your head? Even with passwords managers like <a href="https://lastpass.com/">LastPass</a>, keeping track of <em>another</em> username / password set for our Slack clone might deter new users from signing up.</p> 
       <p>So Meteor also provide package such as <code>accounts-google</code>, <code>accounts-facebook</code>, <code>accounts-twitter</code> so you can log in with your existing Google, Facebook and Twitter accounts. One less thing to remember!</p> 
       <h5>GitHub</h5> 
       <p>Each third-party integration is a little different, but instructions should be clear. Here, we'll show you how to integrate with GitHub. Feel free to try it out with other services like Twitter or Facebook!</p> 
       <p>Let's add the <code>accounts-github</code> package.</p> 
       <pre><code class="language-bash">
$ meteor add accounts-github
</code></pre> 
       <p>For your application to use GitHub for logging users in, it must be registered to an account. So go to your application and open up the login UI. Click the red button and follow the instructions set it up.</p> 
       <p><a href="https://cask.scotch.io/2015/05/meteor-github.png"><img src="https://cask.scotch.io/2015/05/meteor-github-445x250.png" alt="meteor-github" width="445" height="250" class="alignnone size-medium wp-image-9496"></a></p> 
       <p><a href="https://cask.scotch.io/2015/05/github-new-oauth.png"><img src="https://cask.scotch.io/2015/05/github-new-oauth-513x250.png" alt="github-new-oauth" width="513" height="250" class="alignnone size-medium wp-image-9497"></a></p> 
       <p>And now we can login using our GitHub account. The data required for this is stored under the <code>github</code> property of <code>services</code>.</p> 
       <pre><code class="language-javascript">
...
"services": {
    "github": {
        "id": 3571481,
        "accessToken": "2D58E69740CAD0FC1721970A1FF6A140A1A76A41",
        "email": "dan@danyll.com",
        "username": "d4nyll"
    }
    ...
},
...
</code></pre> 
       <p>Once the user is logged in, you will also see that reflected in the GitHub Applications tab.</p> 
       <p><a href="https://cask.scotch.io/2015/05/github-slack-oauth.png"><img src="https://cask.scotch.io/2015/05/github-slack-oauth-412x250.png" alt="github-slack-oauth" width="412" height="250" class="alignnone size-medium wp-image-9498"></a></p> 
       <h3>Confirmation Email</h3> 
       <p>When you sign up to most online services these days, you get a confirmation email.</p> 
       <p>We can enable this through <code>Accounts.config</code> (note that it's not <code>Accountsa.ui.config</code>).</p> 
       <p><strong><code>/server/accounts.js</code></strong></p> 
       <pre><code class="language-javascript">
Accounts.config({
    sendVerificationEmail: true
});
</code></pre> 
       <p>This will internally call the <code>Accounts.sendVerificationEmail()</code> on accounts creation.</p> 
       <blockquote> 
        <p>You must call <code>Accounts.config</code> only on the server, since all mail are sent from the server. Specifying it in client code means no emails will be sent.</p> 
       </blockquote> 
       <p>But to send the email, Meteor needs to be hooked up to a mail server. To do that we need the <code>email</code> package provided by the core.</p> 
       <pre><code class="language-bash">
$ meteor add email
</code></pre> 
       <p>The <code>email</code> package reads the <code>MAIL_URL</code> environment variable for the address of the SMTP server. If you use <code>meteor deploy</code>, the app will automatically uses an account provided by <a href="https://www.mailgun.com/">Mailgun</a> to send your emails.</p> 
       <p>If no <code>MAIL_URL</code> variable is set, the mail that <em>would</em> have been sent is output to the console.</p>

       <h6>Mandrill</h6>

       <h3>Updating Our application</h3> 
       <p>Now that we have the idea of users, let's assign each message to the user that sent it. We will store the <code>Meteor.userId</code> to get the ID of the current user.</p> 
       <pre><code class="language-javascript">
Messages.insert({
  text: $('.input-box_text').val(),
  user: Meteor.userId(),
  timestamp: Date.now()
});
</code></pre> 
       <blockquote> 
        <p><code>Meteor.userId()</code> will return the current user's id, and if no user is logged in, returns <code>null</code>.</p> 
       </blockquote> 
       <p>We are storing our users by ID, but when we display it, we'd really want to display their username. So let's create a helper function to do that.</p> 
       <p>Since this function might be used in more than one place, we'd use the <a href="http://docs.meteor.com/#/full/template_registerhelper"><code>Template.registerHelper()</code></a> function to define a helper function which can be used from all templates.</p> 
       <pre><code class="language-javascript">
Template.registerHelper("usernameFromId", function (userId) {
    var user = Meteor.users.findOne({_id: userId});
    if (typeof user === "undefined") {
        return "Anonymous";
    }
    if (typeof user.services.github !== "undefined") {
        return user.services.github.username;
    }
    return user.username;
});
</code></pre> 
       <p>We will also do something similar so the time of the message is stored and displayed.</p> 
       <pre><code class="language-javascript">
Template.registerHelper("timestampToTime", function (timestamp) {
    var date = new Date(timestamp);
    var hours = date.getHours();
    var minutes = "0" + date.getMinutes();
    var seconds = "0" + date.getSeconds();
    return hours + ':' + minutes.substr(minutes.length-2) + ':' + seconds.substr(seconds.length-2);
});
</code></pre> 
       <p>We will use the new properties and helper methods in our <code>message</code> template.</p> 
       <pre><code class="language-markup">
    &lt;div class="message"&gt;
        &lt;a href="" class="message_profile-pic"&gt;&lt;/a&gt;
        &lt;a href="" class="message_username"&gt;{{usernameFromId user}}&lt;/a&gt;
        &lt;span class="message_timestamp"&gt;{{timestampToTime timestamp}}&lt;/span&gt;
        &lt;span class="message_star"&gt;&lt;/span&gt;
        &lt;span class="message_content"&gt;{{text}}&lt;/span&gt;
    &lt;/div&gt;
</code></pre> 
       <p>Now when you log in and post a message, your username and the timestamp when the message was posted is displayed.</p> 
       <p><a href="https://cask.scotch.io/2015/05/meteor-slack-username-time.png"><img src="https://cask.scotch.io/2015/05/meteor-slack-username-time-445x250.png" alt="meteor-slack-username-time" width="445" height="250" class="alignnone size-medium wp-image-9505"></a></p> 
       <h3>Security</h3> 
       <p>Great! Now we have user data - a big responsibility! We would need to keep this information secure.</p> 
       <p>When we created our example Meteor application, the <code>autopublish</code> and <code>insecure</code> packages were automatically added. <code>autopublish</code> makes available all the data from all collections, and <code>insecure</code> allows any user to insert into, update, and/or delete any documents.</p> 
       <p>Why were these two packages included in the first place? Well, imagine I introduced the topic of subscription, publication, allow-deny rules, method calls, all before we were able to send one message! It'd been too much! So Meteor, by design, included those two packages to allow us to start building our application as quickly as possible. But to make our data secure, we must remove both packages.</p> 
       <h5><code>autopublish</code></h5> 
       <p>If we run <code>Meteor.users</code> in the console now, we get the records of all the users. Not great.</p> 
       <pre><code class="language-bash">
$ meteor remove autopublish
</code></pre> 
       <p>You'll see all the messages on screen disappeared, that's because the data from the <code>messages</code> collection is no longer being sent from the server.</p> 
       <p>If we run <code>Meteor.users</code> now, we will only see, at most, the <code>_id</code>, <code>username</code> and <code>profile</code> fields. We won't be able to get other users' data now.</p> 
       <p>But we want our messages to show. So let's publish it now server-side.</p> 
       <p><strong><code>/server/publications.js</code></strong></p> 
       <pre><code class="language-javascript">
Meteor.publish('messages', function () {
    return Messages.find();
});
</code></pre> 
       <p>And now subscribe to the publication client-side.</p> 
       <p><strong>/client/app.js</strong></p> 
       <pre><code class="language-javascript">
Meteor.subscribe('messages');
</code></pre> 
       <p>And now our messages are back! But the usernames have gone. So we must also publish certain fields from our <code>users</code> collection too.</p> 
       <p><strong><code>/server/publications.js</code></strong></p> 
       <pre><code class="language-javascript">
Meteor.publish("allUsernames", function () {
  return Meteor.users.find({}, {fields: {
    "username": 1,
    "services.github.username": 1
  }});
});
</code></pre> 
       <p><strong>/client/app.js</strong></p> 
       <pre><code class="language-javascript">
Meteor.subscribe('allUsernames');
</code></pre> 
       <h5><code>insecure</code></h5> 
       <p>If you open up your browser's console, and you run</p> 
       <pre><code class="language-bash">
Messages.find().fetch()
</code></pre> 
       <p>You can actually get the list of all the messages in the collection. What's more, find the <code>_id</code> of one of the <code>Objects</code> returned and run:</p> 
       <pre><code class="language-bash">
Messages.remove({_id: "tduWf5JMyJbX4w2Qj"})
</code></pre> 
       <p>Where <code>tduWf5JMyJbX4w2Qj</code> is the <code>_id</code> of the <code>Object</code>.</p> 
       <p>The message has been removed! And not just on your local environment, but for everyone! Of course that's a big security hole.</p> 
       <p>Tthe <code>insecure</code> package is just there to help us develop faster by allowing us to alter the server database on the client. Now we are not on training wheels anymore, we can remove it!</p> 
       <pre><code class="language-bash">
$ meteor remove insecure
</code></pre> 
       <p>But now if you try to send a message, this error comes up in the console:</p> 
       <pre><code class="language-bash">
insert failed: Access denied
</code></pre> 
       <p>This is because we have now prevented the user from making changes to the collections. To get control over what actions the user is allowed to perform, we can either use <em>Allow/Deny rules</em> or <em>Meteor methods</em>.</p> 
       <h5>Allow / Deny</h5> 
       <p>We can allow users to interact directly with the collection by setting allow rules.</p> 
       <pre><code class="language-javascript">
Messages.allow({
  insert: function (userId, doc) {
    return true;
  }
});
</code></pre> 
       <p>The <a href="http://docs.meteor.com/#/full/allow"><code>collection.allow()</code></a> method specify which modifications are allowed from the client. Here, we are specifying that all inserts from any users are allowed on the <code>Messages</code> collection, by returning <code>true</code>.</p> 
       <p>All looks innocent, but what we just did allows a user to insert a message masquerading as another user:</p> 
       <pre><code class="language-javascript">
Messages.insert({
  text: $('.input-box_text').val(),
  user: "someoneElsesID",
  timestamp: Date.now()
});
</code></pre> 
       <p>So our allow rule should really check that the <code>userId</code> of the user matches the one specified int he document.</p> 
       <pre><code class="language-javascript">
Messages.allow({
  insert: function (userId, doc) {
    return (userId &amp;&amp; doc.user === userId);
  }
});
</code></pre> 
       <p>As you can see, using allow/deny rules are quick and easy, but are prone to mistakes. Oh, and this also reminds me, I should check that the timestamp given is accurate...but how do we do that? What's to stop someone claim they posted a message predicting the lottery numbers an hour before the draw?</p> 
       <p>You can install the <a href="https://github.com/matb33/meteor-collection-hooks"><code>matb33:collection-hooks</code></a> and do something like this:</p> 
       <pre><code class="language-javascript">
Messages.before.insert(function (userId, doc) {
  doc.timestamp = Date.now();
});
</code></pre> 
       <p>But this involves another package (<a href="https://github.com/meteor/meteor/issues/395">Meteor doesn't support hooks natively</a>). This is why I think using Meteor methods are easier in most cases.</p> 
       <h5>Methods</h5> 
       <p>With Meteor methods, when the client wants to insert into the collection, it asks the server to do it. The server can then do some validation and set its own fields (such as <code>message.timestamp</code>). This saves us having to check whether the timestamp provided by the client is accurate.</p> 
       <p><strong><code>/client/input.js</code></strong></p> 
       <pre><code class="language-javascript">
Meteor.call('newMessage', {text: $('.input-box_text').val()});
</code></pre> 
       <p><strong><code>/server/methods.js</code></strong></p> 
       <pre><code class="language-javascript">
Meteor.methods({
  newMessage: function (message) {
    message.timestamp = Date.now();
    message.user = Meteor.userId();
    Messages.insert(message);
  }
})
</code></pre> 
       <p>All in all, I prefer using Meteor methods over allow/deny rules, mainly for these reasons:</p> 
       <ul> 
        <li>Easier to do complicated validation - you explicitly determine what is allowed or not in the code</li> 
        <li>All the code are in one block - no inserts on the client, setting allow / deny rules on both client and server, and hooks on the server</li> 
        <li>Quite subjective, but I find methods less error-prone</li> 
       </ul> 
       <blockquote> 
        <p>Do you agree / disagree / still not sure? Start a discussion in the comments section!</p> 
       </blockquote> 
       <h6>Stubs</h6> 
       <p>But now, you might think, this defeats the purpose of latency compensation and the database everywhere principles of Meteor, since we must now wait for the server to confirm before the message shows up.</p> 
       <p>If we shut down our application now on the server, users won't see their messages on the message list.</p> 
       <p>But we can define a stub - the equivalent of a Meteor method that gets ran on the client-side. Taken from this <a href="https://stackoverflow.com/a/27947558/3966682">Stack Overflow answer</a> I provided a few months back.</p> 
       <blockquote> 
        <p>When the client calls the method, it will execute the stub method on the client-side, which can quickly return a (predicted) response. When the server comes back with the 'actual' response, it will replace the response generated by the stub and update other elements according.</p> 
       </blockquote> 
       <p>I'd usually create a <code>/client/stubs.js</code> to house all the stub methods, but since there are no secret information we need to hide from the client, we can just move our <code>/server/methods.js</code> to, say, <code>/methods.js</code></p> 
       <p><strong><code>/client/stubs.js</code></strong></p> 
       <pre><code class="language-javascript">
Meteor.methods({
  newMessage: function (message) {
    message.timestamp = Date.now();
    message.user = Meteor.userId();
    Messages.insert(message);
  }
});
</code></pre> 
       <p>Now even when the server is down, the messages will still be rendered immediately on the client. And when the server is back up, the messages are still saved.</p>
