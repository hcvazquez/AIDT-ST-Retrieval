      <p>The <b><a href="/wiki/HTML5" title="HTML5">HTML5</a></b> specification introduced the <b>video element</b> for the purpose of playing videos,<sup id="cite_ref-spec_1-0" class="reference"><a href="#cite_note-spec-1">[1]</a></sup> partially replacing the <a href="/wiki/Object_element" class="mw-redirect" title="Object element">object element</a>. HTML5 video is intended by its creators to become the new standard way to show video on the web, instead of the previous de facto standard of using the proprietary <a href="/wiki/Adobe_Flash" title="Adobe Flash">Adobe Flash</a> plugin, though early adoption was hampered by lack of agreement as to which <a href="/wiki/Video_coding_format" title="Video coding format">video coding formats</a> and <a href="/wiki/Audio_coding_format" title="Audio coding format">audio coding formats</a> should be supported in web browsers.</p>

      <h2><span class="mw-headline" id="History_of_.3Cvideo.3E_element">History of &lt;video&gt; element</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=HTML5_video&amp;action=edit&amp;section=1" title="Edit section: History of <video> element">edit</a><span class="mw-editsection-bracket">]</span></span></h2> 
      <p>The &lt;video&gt; element was proposed by <a href="/wiki/Opera_Software" title="Opera Software">Opera Software</a> in February 2007.<sup id="cite_ref-2" class="reference"><a href="#cite_note-2">[2]</a></sup> Opera also released a preview build that was showcased the same day,<sup id="cite_ref-3" class="reference"><a href="#cite_note-3">[3]</a></sup><sup id="cite_ref-4" class="reference"><a href="#cite_note-4">[4]</a></sup> and a manifesto that called for video to become a first-class citizen of the web.<sup id="cite_ref-5" class="reference"><a href="#cite_note-5">[5]</a></sup></p> 
      <h2><span class="mw-headline" id=".3Cvideo.3E_element_examples">&lt;video&gt; element examples</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=HTML5_video&amp;action=edit&amp;section=2" title="Edit section: <video> element examples">edit</a><span class="mw-editsection-bracket">]</span></span></h2> 
      <p>The following HTML5 code fragment will embed a <a href="/wiki/WebM_Project" class="mw-redirect" title="WebM Project">WebM</a> video into a web page.</p> 
      <div class="mw-highlight mw-content-ltr" dir="ltr"> 
       <pre>
<span class="p">&lt;</span><span class="nt">video</span> <span class="na">src</span><span class="o">=</span><span class="s">"movie.webm"</span> <span class="na">poster</span><span class="o">=</span><span class="s">"movie.jpg"</span> <span class="na">controls</span><span class="p">&gt;</span>
	This is fallback content to display for user agents that do not support the video tag.
<span class="p">&lt;/</span><span class="nt">video</span><span class="p">&gt;</span>
</pre>
      </div> 
      <p>The "controls" attribute enables the browser's own user interface for controlling playback. Alternatively, playback can be controlled with <a href="/wiki/JavaScript" title="JavaScript">JavaScript</a>, which the web designer can use to create a custom user interface. The optional "poster" attribute specifies an image to show in the video's place before playback is started. Its purpose is to be representative of the video.</p> 
      <h3><span class="mw-headline" id="Multiple_sources">Multiple sources</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=HTML5_video&amp;action=edit&amp;section=3" title="Edit section: Multiple sources">edit</a><span class="mw-editsection-bracket">]</span></span></h3> 
      <p>Video format support varies among browsers (see below), so a web page can provide video in multiple formats. For other features, <a href="/wiki/Browser_sniffing" title="Browser sniffing">browser sniffing</a> is used sometimes, which may be error-prone: any web developer's knowledge of browsers will inevitably be incomplete or not up-to-date. The browser in question "knows best" what formats it can use. The "video" element supports fallback through specification of multiple sources. Using any number of &lt;source&gt; elements, as shown below, the browser will choose automatically which file to download. Alternatively, the <a href="/wiki/JavaScript" title="JavaScript">JavaScript</a> canPlayType() function can be used to achieve the same. The "type" attribute specifies the <a href="/wiki/Internet_media_type" class="mw-redirect" title="Internet media type">MIME type</a> and possibly a list of codecs, which helps the browser to determine whether it can decode the file. Even with only one choice, such hints may be necessary to a browser for querying its <a href="/wiki/Multimedia_framework" title="Multimedia framework">multimedia framework</a> for third party codecs.</p> 
      <div class="mw-highlight mw-content-ltr" dir="ltr"> 
       <pre>
<span class="p">&lt;</span><span class="nt">video</span> <span class="na">poster</span><span class="o">=</span><span class="s">"movie.jpg"</span> <span class="na">controls</span><span class="p">&gt;</span>
	<span class="p">&lt;</span><span class="nt">source</span> <span class="na">src</span><span class="o">=</span><span class="s">"movie.webm"</span> <span class="na">type</span><span class="o">=</span><span class="s">'video/webm; codecs="vp8.0, vorbis"'</span><span class="p">&gt;</span>
	<span class="p">&lt;</span><span class="nt">source</span> <span class="na">src</span><span class="o">=</span><span class="s">"movie.ogv"</span> <span class="na">type</span><span class="o">=</span><span class="s">'video/ogg; codecs="theora, vorbis"'</span><span class="p">&gt;</span>
	<span class="p">&lt;</span><span class="nt">source</span> <span class="na">src</span><span class="o">=</span><span class="s">"movie.mp4"</span> <span class="na">type</span><span class="o">=</span><span class="s">'video/mp4; codecs="avc1.4D401E, mp4a.40.2"'</span><span class="p">&gt;</span>
	<span class="p">&lt;</span><span class="nt">p</span><span class="p">&gt;</span>This is fallback content to display for user agents that do not support the video tag.<span class="p">&lt;/</span><span class="nt">p</span><span class="p">&gt;</span>
<span class="p">&lt;/</span><span class="nt">video</span><span class="p">&gt;</span>
</pre>
      </div> 
      <h2><span class="mw-headline" id="Supported_video_and_audio_formats">Supported video and audio formats</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=HTML5_video&amp;action=edit&amp;section=4" title="Edit section: Supported video and audio formats">edit</a><span class="mw-editsection-bracket">]</span></span></h2> 
      <div role="note" class="hatnote navigation-not-searchable">
       See also: 
       <a href="/wiki/Use_of_Ogg_formats_in_HTML5" title="Use of Ogg formats in HTML5">Use of Ogg formats in HTML5</a>
      </div> 
      <p>The HTML5 specification does not specify which video and audio formats browsers should support. User agents are free to support any video formats they feel are appropriate, but content authors cannot assume that any video will be accessible by all complying user agents, since user agents have no minimal set of video and audio formats to support.</p> 
      <p>The HTML5 Working Group considered it desirable to specify at least one video format which all user agents (browsers) should support. The ideal format in this regard would:</p> 
      <ul> 
       <li>Have good compression, good image quality, and low decode processor use.</li> 
       <li>Be royalty-free.</li> 
       <li>In addition to software decoders, a hardware <a href="/wiki/Video_decoder" title="Video decoder">video decoder</a> should exist for the format, as many embedded processors do not have the performance to decode video.</li> 
      </ul> 
      <p>Initially, <a href="/wiki/Ogg" title="Ogg">Ogg</a> <a href="/wiki/Theora" title="Theora">Theora</a> was the recommended standard video format in HTML5, because it was not affected by any known patents. But on December 10, 2007, the HTML5 specification was updated,<sup id="cite_ref-6" class="reference"><a href="#cite_note-6">[6]</a></sup> replacing the reference to concrete formats:</p> 
      <blockquote class="templatequote"> 
       <p>User agents should support Theora video and Vorbis audio, as well as the Ogg container format.</p> 
      </blockquote> 
      <p>with a placeholder:<sup id="cite_ref-7" class="reference"><a href="#cite_note-7">[7]</a></sup></p> 
      <blockquote class="templatequote"> 
       <p>It would be helpful for interoperability if all browsers could support the same codecs. However, there are no known codecs that satisfy all the current players: we need a codec that is known to not require per-unit or per-distributor licensing, that is compatible with the open source development model, that is of sufficient quality as to be usable, and that is not an additional <a href="/wiki/Submarine_patent" title="Submarine patent">submarine patent</a> risk for large companies. This is an ongoing issue and this section will be updated once more information is available.<sup id="cite_ref-8" class="reference"><a href="#cite_note-8">[8]</a></sup></p> 
      </blockquote> 
      <p>The result has been the polarisation of HTML5 video between <a href="/wiki/Technical_standard" title="Technical standard">industry-standard</a>, <a href="/wiki/International_Organization_for_Standardization" title="International Organization for Standardization">ISO-defined</a> but <a href="/wiki/Patented" class="mw-redirect" title="Patented">patented</a> formats, and <a href="/wiki/Free_software" title="Free software">free</a>, <a href="/wiki/Open_software" class="mw-redirect" title="Open software">open</a> formats.</p> 
      <h3><span class="mw-headline" id="Free_formats">Free formats</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=HTML5_video&amp;action=edit&amp;section=5" title="Edit section: Free formats">edit</a><span class="mw-editsection-bracket">]</span></span></h3> 
      <div role="note" class="hatnote navigation-not-searchable">
       See also: 
       <a href="/wiki/Theora" title="Theora">Theora</a> and 
       <a href="/wiki/WebM" title="WebM">WebM</a>
      </div> 
      <p>Although Theora is not affected by known non-free patents, <a href="/wiki/Apple_Inc." title="Apple Inc.">Apple</a><sup id="cite_ref-9" class="reference"><a href="#cite_note-9">[9]</a></sup> has expressed concern about <a href="/wiki/Patent_ambush" title="Patent ambush">unknown patents</a> that might affect it, whose owners might be waiting for a corporation with extensive financial resources to use the format before suing.<sup id="cite_ref-10" class="reference"><a href="#cite_note-10">[10]</a></sup><sup id="cite_ref-apple-ogg_11-0" class="reference"><a href="#cite_note-apple-ogg-11">[11]</a></sup> Formats like <a href="/wiki/H.264/MPEG-4_AVC" title="H.264/MPEG-4 AVC">H.264</a> might also be subject to unknown patents in principle, but they have been deployed much more widely and so it is presumed that any patent-holders would have already made themselves known. <a href="/wiki/Apple_Inc." title="Apple Inc.">Apple</a> has also opposed requiring Ogg format support in the HTML standard (even as a "should" requirement) on the grounds that some devices might support other formats much more easily, and that HTML has historically not required particular formats for anything.<sup id="cite_ref-apple-ogg_11-1" class="reference"><a href="#cite_note-apple-ogg-11">[11]</a></sup></p> 
      <p>Some <a href="/wiki/Web_developer" title="Web developer">web developers</a> criticized the removal of the Ogg formats from the specification.<sup id="cite_ref-12" class="reference"><a href="#cite_note-12">[12]</a></sup> A follow-up discussion also occurred on the W3C questions and answers blog.<sup id="cite_ref-13" class="reference"><a href="#cite_note-13">[13]</a></sup></p> 
      <p><a href="/wiki/Mozilla" title="Mozilla">Mozilla</a> and <a href="/wiki/Opera_Software" title="Opera Software">Opera</a> support only the open formats of <a href="/wiki/Theora" title="Theora">Theora</a> and <a href="/wiki/WebM" title="WebM">WebM</a>. Google stated its intention to remove support for H.264 in 2011, specifically for the HTML5 video tag.<sup id="cite_ref-chromeonvideo_14-0" class="reference"><a href="#cite_note-chromeonvideo-14">[14]</a></sup> Although it has been removed from <a href="/wiki/Chromium_(web_browser)" title="Chromium (web browser)">Chromium</a>, As of November 2016<sup class="plainlinks noprint asof-tag update" style="display:none;"><a class="external text" href="//en.wikipedia.org/w/index.php?title=HTML5_video&amp;action=edit">[update]</a></sup> it has yet to be removed from <a href="/wiki/Google_Chrome" title="Google Chrome">Google Chrome</a> five years later.<sup id="cite_ref-15" class="reference"><a href="#cite_note-15">[15]</a></sup><sup id="cite_ref-16" class="reference"><a href="#cite_note-16">[16]</a></sup></p> 
      <h4><span class="mw-headline" id="MPEG-DASH_Support_via_the_HTML5_Media_Source_Extensions_.28MSE.29">MPEG-DASH Support via the HTML5 Media Source Extensions (MSE)</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=HTML5_video&amp;action=edit&amp;section=6" title="Edit section: MPEG-DASH Support via the HTML5 Media Source Extensions (MSE)">edit</a><span class="mw-editsection-bracket">]</span></span></h4> 
      <div role="note" class="hatnote navigation-not-searchable">
       See also: 
       <a href="/wiki/Dynamic_Adaptive_Streaming_over_HTTP" title="Dynamic Adaptive Streaming over HTTP">Dynamic Adaptive Streaming over HTTP</a>
      </div> 
      <p>The adaptive bitrate streaming standard MPEG-DASH can be used in Web browsers via the <a href="/wiki/Media_Source_Extensions" title="Media Source Extensions">HTML5 Media Source Extensions (MSE)</a><sup id="cite_ref-mse_17-0" class="reference"><a href="#cite_note-mse-17">[17]</a></sup> and JavaScript-based DASH players. Such players are, e.g., the open-source project dash.js<sup id="cite_ref-mse_17-1" class="reference"><a href="#cite_note-mse-17">[17]</a></sup> of the DASH Industry Forum, but there are also products such as bitdash of <a href="/wiki/Bitmovin" title="Bitmovin">bitmovin</a><sup id="cite_ref-bitdash_18-0" class="reference"><a href="#cite_note-bitdash-18">[18]</a></sup> (using HTML5 with JavaScript, but also a Flash-based DASH players for legacy Web browsers not supporting the HTML5 MSE).</p> 
      <h4><span class="mw-headline" id="Google.27s_purchase_of_On2">Google's purchase of On2</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=HTML5_video&amp;action=edit&amp;section=7" title="Edit section: Google's purchase of On2">edit</a><span class="mw-editsection-bracket">]</span></span></h4> 
      <p><a href="/wiki/Google" title="Google">Google</a>'s acquisition of <a href="/wiki/On2_Technologies" title="On2 Technologies">On2</a> in 2010 resulted in its acquisition of the <a href="/wiki/VP8" title="VP8">VP8</a> video format. Google has provided a <a href="/wiki/Royalty-free" title="Royalty-free">royalty-free</a> license to use VP8.<sup id="cite_ref-19" class="reference"><a href="#cite_note-19">[19]</a></sup> Google also started <a href="/wiki/WebM" title="WebM">WebM</a>, which combines the standardized open source VP8 video codec with <a href="/wiki/Vorbis" title="Vorbis">Vorbis</a> audio in a <a href="/wiki/Matroska" title="Matroska">Matroska</a> based container. The opening of VP8 was welcomed by the <a href="/wiki/Free_Software_Foundation" title="Free Software Foundation">Free Software Foundation</a>.<sup id="cite_ref-20" class="reference"><a href="#cite_note-20">[20]</a></sup></p> 
      <p>When Google announced in January 2011 that it would end native support of H.264 in Chrome,<sup id="cite_ref-21" class="reference"><a href="#cite_note-21">[21]</a></sup> criticism came from many quarters including Peter Bright of <a href="/wiki/Ars_Technica" title="Ars Technica">Ars Technica</a><sup id="cite_ref-22" class="reference"><a href="#cite_note-22">[22]</a></sup> and Microsoft web evangelist Tim Sneath, who compared Google's move to declaring <a href="/wiki/Esperanto" title="Esperanto">Esperanto</a> the official language of the United States.<sup id="cite_ref-23" class="reference"><a href="#cite_note-23">[23]</a></sup> However, Haavard Moen of Opera Software strongly criticized the Ars Technica article<sup id="cite_ref-24" class="reference"><a href="#cite_note-24">[24]</a></sup> and Google responded to the reaction by clarifying its intent to promote WebM in its products on the basis of openness.<sup id="cite_ref-chromeonvideo_14-1" class="reference"><a href="#cite_note-chromeonvideo-14">[14]</a></sup></p> 
      <p>After the launch of WebM, Mozilla and Opera have called for the inclusion of VP8 in HTML.<sup id="cite_ref-25" class="reference"><a href="#cite_note-25">[25]</a></sup></p> 
      <p>On March 7, 2013, Google Inc. and <a href="/wiki/MPEG_LA" title="MPEG LA">MPEG LA</a>, LLC announced agreements covering techniques that "may be essential" to VP8, with Google receiving a license from MPEG LA and 11 patent holders, and MPEG LA ending its efforts to form a VP8 patent pool.<sup id="cite_ref-26" class="reference"><a href="#cite_note-26">[26]</a></sup><sup id="cite_ref-27" class="reference"><a href="#cite_note-27">[27]</a></sup><sup id="cite_ref-28" class="reference"><a href="#cite_note-28">[28]</a></sup><sup id="cite_ref-29" class="reference"><a href="#cite_note-29">[29]</a></sup></p> 
      <h3><span class="mw-headline" id="Non-free_formats">Non-free formats</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=HTML5_video&amp;action=edit&amp;section=8" title="Edit section: Non-free formats">edit</a><span class="mw-editsection-bracket">]</span></span></h3> 
      <div role="note" class="hatnote navigation-not-searchable">
       See also: 
       <a href="/wiki/H.264/MPEG-4_AVC" title="H.264/MPEG-4 AVC">H.264/MPEG-4 AVC</a>
      </div> 
      <p><a href="/wiki/H.264/MPEG-4_AVC" title="H.264/MPEG-4 AVC">H.264/MPEG-4 AVC</a> is widely used, and has good speed, compression, hardware decoders, and video quality, but is patent-encumbered.<sup id="cite_ref-30" class="reference"><a href="#cite_note-30">[30]</a></sup> Users of H.264 need licenses either from the individual patent holders, or from the <a href="/wiki/MPEG_LA" title="MPEG LA">MPEG LA</a>, a group of patent holders including Microsoft and Apple, except for some Internet broadcast video uses.<sup id="cite_ref-31" class="reference"><a href="#cite_note-31">[31]</a></sup> H.264 is usually used in the MP4 container format, together with <a href="/wiki/Advanced_Audio_Coding" title="Advanced Audio Coding">Advanced Audio Coding</a> (AAC) audio. AAC is also patented in itself, so users of MP4 will have to license both H.264 and AAC.</p> 
      <p>In June 2009, the WHATWG concluded that no existing format was suitable as a specified requirement.<sup id="cite_ref-32" class="reference"><a href="#cite_note-32">[32]</a></sup></p> 
      <p>Apple still only supports H.264, but Microsoft now supports <a href="/wiki/VP9" title="VP9">VP9</a> and WebM, and has pledged support for <a href="/wiki/AV1" class="mw-redirect" title="AV1">AV1</a>.</p> 
      <h4><span class="mw-headline" id="Cisco_makes_a_licensed_H.264_binary_module_available_for_free">Cisco makes a licensed H.264 binary module available for free</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=HTML5_video&amp;action=edit&amp;section=9" title="Edit section: Cisco makes a licensed H.264 binary module available for free">edit</a><span class="mw-editsection-bracket">]</span></span></h4> 
      <div role="note" class="hatnote navigation-not-searchable">
       Main article: 
       <a href="/wiki/OpenH264" title="OpenH264">OpenH264</a>
      </div> 
      <p>On October 30, 2013, <a href="/wiki/Cisco" class="mw-redirect" title="Cisco">Cisco</a> announced that they were making a binary H.264 module available for download. Cisco will pay the costs of patent licensing for those binary modules <i>when downloaded by the using software while it is being installed</i>, making H.264 free to use in that specific case.<sup id="cite_ref-openh264-faq_33-0" class="reference"><a href="#cite_note-openh264-faq-33">[33]</a></sup></p> 
      <p>In the announcement, Cisco cited its desire of furthering the use of the <a href="/wiki/WebRTC" title="WebRTC">WebRTC</a> project as the reason, since WebRTC's video chat feature will benefit from having a video format supported in all browsers.<sup id="cite_ref-cisco-264-release_34-0" class="reference"><a href="#cite_note-cisco-264-release-34">[34]</a></sup> The H.264 module will be available on "all popular or feasibly supportable platforms, which can be loaded into any application".<sup id="cite_ref-h264_mozilla_35-0" class="reference"><a href="#cite_note-h264_mozilla-35">[35]</a></sup></p> 
      <p>Cisco is also planning to publish source code for those modules under <a href="/wiki/BSD_licenses" title="BSD licenses">BSD license</a>, but without paying the royalties,<sup id="cite_ref-openh264-faq_33-1" class="reference"><a href="#cite_note-openh264-faq-33">[33]</a></sup> so the code will practically be <a href="/wiki/Free_software" title="Free software">free software</a> only in countries without H.264 <a href="/wiki/Software_patent" title="Software patent">software patents</a>, which has already been true about other existing implementations.</p> 
      <p>Also on October 30, 2013, Mozilla's <a href="/wiki/Brendan_Eich" title="Brendan Eich">Brendan Eich</a> announced that Firefox would automatically download Cisco's H.264 module when needed by default. He also noted that the binary module is not a perfect solution, since users do not have full <a href="/wiki/Free_software" title="Free software">free software</a> rights to "modify, recompile, and redistribute without license agreements or fees". Thus Xiph and Mozilla continue the development of <a href="/wiki/Daala" title="Daala">Daala</a>.<sup id="cite_ref-h264_mozilla_35-1" class="reference"><a href="#cite_note-h264_mozilla-35">[35]</a></sup><sup id="cite_ref-monty_36-0" class="reference"><a href="#cite_note-monty-36">[36]</a></sup></p> 
      <p><a href="/wiki/OpenH264" title="OpenH264">OpenH264</a> only supports the <i>baseline</i> profile of H.264, and does not by itself address the need for an AAC decoder. Therefore, it is not considered sufficient for typical MP4 web video, which is typically in the <i>high</i> profile with AAC audio.<sup id="cite_ref-37" class="reference"><a href="#cite_note-37">[37]</a></sup><sup id="cite_ref-38" class="reference"><a href="#cite_note-38">[38]</a></sup><sup id="cite_ref-39" class="reference"><a href="#cite_note-39">[39]</a></sup> However, for use in WebRTC, the omission of AAC was justified in the release announcement: "the standards bodies have aligned on <a href="/wiki/Opus_(codec)" class="mw-redirect" title="Opus (codec)">Opus</a> and <a href="/wiki/G.711" title="G.711">G.711</a> as the common audio codecs for WebRTC".<sup id="cite_ref-cisco-264-release_34-1" class="reference"><a href="#cite_note-cisco-264-release-34">[34]</a></sup> There is doubt as to whether a capped global licensing of AAC, like Cisco's for H.264, is feasible after AAC's licensing bureau removed the price cap shortly after the release of OpenH264.<sup id="cite_ref-40" class="reference"><a href="#cite_note-40">[40]</a></sup></p> 
      <h2><span class="mw-headline" id="Browser_support">Browser support</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=HTML5_video&amp;action=edit&amp;section=10" title="Edit section: Browser support">edit</a><span class="mw-editsection-bracket">]</span></span></h2> 
      <div role="note" class="hatnote navigation-not-searchable">
       Main article: 
       <a href="/wiki/Comparison_of_layout_engines_(HTML5_media)" title="Comparison of layout engines (HTML5 media)">Comparison of layout engines (HTML5 media)</a>
      </div> 
      <p>This table shows which video formats are <i>likely</i> to be supported by a given <a href="/wiki/User_agent" title="User agent">user agent</a>. Most of the browsers listed here use a <a href="/wiki/Multimedia_framework" title="Multimedia framework">multimedia framework</a> for decoding and display of video, instead of incorporating such software components. It is not generally possible to tell the set of formats supported by a multimedia framework without querying it, because that depends on the <a href="/wiki/Operating_system" title="Operating system">operating system</a> and third party codecs.<sup id="cite_ref-phonon-backs_41-0" class="reference"><a href="#cite_note-phonon-backs-41">[41]</a></sup> In these cases, video format support is an attribute of the framework, not the browser (or its layout engine), assuming the browser properly queries its multimedia framework before rejecting unknown video formats. In some cases, the support listed here is not a function of either codecs available within the operating system's underlying media framework, or of codec capabilities built into the browser, but rather could be by a browser add-on that might, for example, bypass the browser's normal HTML parsing of the &lt;video&gt; tag to embed a plug-in based video player.</p> 
      <p>Note that a video file normally contains both video and audio content, each encoded in its own format. The browser has to support both the video and audio formats. See <a href="/wiki/HTML5_audio" class="mw-redirect" title="HTML5 audio">HTML5 audio</a> for a table of which audio formats are supported by each browser.</p> 
      <p>The video format can be specified by <a href="/wiki/Internet_media_type" class="mw-redirect" title="Internet media type">MIME type</a> in HTML (see <a href="#Multiple_sources">example</a>). MIME types are used for querying multimedia frameworks for supported formats.<sup id="cite_ref-phonon-query_42-0" class="reference"><a href="#cite_note-phonon-query-42">[42]</a></sup></p> 
      <p>Of these browsers, only Firefox and Opera employ libraries for built-in decoding. In practice, Internet Explorer and Safari can also guarantee certain format support, because their manufacturers also make their multimedia frameworks. At the other end of the scale, Konqueror has identical format support to Internet Explorer when run on Windows, and Safari when run on Mac, but the selected support here for Konqueror is the typical for GNU/Linux, where Konqueror has most of its users. In general, the format support of browsers is much dictated by conflicting interests of vendors, specifically that <a href="/wiki/Media_Foundation" title="Media Foundation">Media Foundation</a> and <a href="/wiki/QuickTime" title="QuickTime">QuickTime</a> support commercial standards, whereas <a href="/wiki/GStreamer" title="GStreamer">GStreamer</a> and <a href="/wiki/Phonon_(KDE)" class="mw-redirect" title="Phonon (KDE)">Phonon</a> cannot legally support other than free formats by default on the free operating systems that they are intended for.<sup id="cite_ref-gpl-license_43-0" class="reference"><a href="#cite_note-gpl-license-43">[43]</a></sup></p>

      <h2><span class="mw-headline" id="Digital_rights_management_.28Encrypted_Media_Extensions.29">Digital rights management (Encrypted Media Extensions)</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=HTML5_video&amp;action=edit&amp;section=11" title="Edit section: Digital rights management (Encrypted Media Extensions)">edit</a><span class="mw-editsection-bracket">]</span></span></h2> 
      <div role="note" class="hatnote navigation-not-searchable">
       Main article: 
       <a href="/wiki/Encrypted_Media_Extensions" title="Encrypted Media Extensions">Encrypted Media Extensions</a>
      </div> 
      <p>HTML has support for <a href="/wiki/Digital_rights_management" title="Digital rights management">digital rights management</a> (DRM, restricting how content can be used) via the HTML5 <a href="/wiki/Encrypted_Media_Extensions" title="Encrypted Media Extensions">Encrypted Media Extensions</a> (EME). The addition of DRM is controversial because it allows restricting users' freedom to use media restricted by DRM, even where fair use gives users the legal right to do so.<sup id="cite_ref-101" class="reference"><a href="#cite_note-101">[94]</a></sup> A main argument in W3C's approval of EME was that the video content would otherwise be delivered in plugins and apps, and not in the web browser.<sup id="cite_ref-102" class="reference"><a href="#cite_note-102">[95]</a></sup></p> 
      <p>In 2013 <a href="/wiki/Netflix" title="Netflix">Netflix</a> added support for HTML5 video using EME, beside their old delivery method using a <a href="/wiki/Silverlight" class="mw-redirect" title="Silverlight">Silverlight</a> plugin (also with DRM).<sup id="cite_ref-103" class="reference"><a href="#cite_note-103">[96]</a></sup></p> 
      <h2><span class="mw-headline" id="Usage">Usage</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=HTML5_video&amp;action=edit&amp;section=12" title="Edit section: Usage">edit</a><span class="mw-editsection-bracket">]</span></span></h2> 
      <p>In 2010, in the wake of Apple <a href="/wiki/IPad" title="IPad">iPad</a> launch and after <a href="/wiki/Steve_Jobs" title="Steve Jobs">Steve Jobs</a> announced that <a href="/wiki/Apple_and_Adobe_Flash_controversy" class="mw-redirect" title="Apple and Adobe Flash controversy">Apple mobile devices would not support Flash</a>, a number of high-profile sites began to serve H.264 HTML5 video instead of <a href="/wiki/Adobe_Flash" title="Adobe Flash">Adobe Flash</a> for user-agents identifying as iPad.<sup id="cite_ref-104" class="reference"><a href="#cite_note-104">[97]</a></sup> HTML5 video was not as widespread as Flash videos, though there were rollouts of experimental HTML5-based video players from DailyMotion (using Ogg Theora and Vorbis format),<sup id="cite_ref-105" class="reference"><a href="#cite_note-105">[98]</a></sup> YouTube (using the H.264 and WebM formats),<sup id="cite_ref-106" class="reference"><a href="#cite_note-106">[99]</a></sup> and Vimeo (using the H.264 format).<sup id="cite_ref-107" class="reference"><a href="#cite_note-107">[100]</a></sup></p> 
      <p>Support for HTML5 video has been steadily increasing. In June 2013, Netflix added support for HTML5 video.<sup id="cite_ref-108" class="reference"><a href="#cite_note-108">[101]</a></sup> In January 2015, YouTube switched to using HTML5 video instead of Flash by default.<sup id="cite_ref-109" class="reference"><a href="#cite_note-109">[102]</a></sup> In December 2015, Facebook switched from Flash to HTML5 for all video content.<sup id="cite_ref-110" class="reference"><a href="#cite_note-110">[103]</a></sup></p> 
      <p>As of 2016, Flash is still widely installed on desktops, while generally not being supported on mobile devices such as smartphones.<sup id="cite_ref-flashphone_111-0" class="reference"><a href="#cite_note-flashphone-111">[104]</a></sup> The Flash plugin is widely assumed, including by Adobe,<sup id="cite_ref-flashphone_111-1" class="reference"><a href="#cite_note-flashphone-111">[104]</a></sup><sup id="cite_ref-112" class="reference"><a href="#cite_note-112">[105]</a></sup> to be destined to be phased out,<sup id="cite_ref-113" class="reference"><a href="#cite_note-113">[106]</a></sup><sup id="cite_ref-114" class="reference"><a href="#cite_note-114">[107]</a></sup> which will leave HTML5 video as the only widely supported method to play video on the World Wide Web. Chrome,<sup id="cite_ref-115" class="reference"><a href="#cite_note-115">[108]</a></sup><sup id="cite_ref-116" class="reference"><a href="#cite_note-116">[109]</a></sup> Firefox,<sup id="cite_ref-117" class="reference"><a href="#cite_note-117">[110]</a></sup> Safari,<sup id="cite_ref-118" class="reference"><a href="#cite_note-118">[111]</a></sup> and Edge,<sup id="cite_ref-119" class="reference"><a href="#cite_note-119">[112]</a></sup> have plans to make almost all flash content click to play in 2017. The only major browser which does not have announced plans to deprecate Flash is Internet Explorer.<sup id="cite_ref-120" class="reference"><a href="#cite_note-120">[113]</a></sup> Adobe announced on 25 July 2017 that they would be permanently ending development of flash in 2020.<sup id="cite_ref-121" class="reference"><a href="#cite_note-121">[114]</a></sup></p>
