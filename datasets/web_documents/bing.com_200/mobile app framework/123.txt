         <p>Single page web applications - or SPAs, as they are commonly referred to - are quickly becoming the de facto standard for web app development. The fact that a major part of the app runs inside a single web page makes it very interesting and appealing, and the accelerated growth of browser capabilities pushes us closer to the day, when all apps run entirely in the browser. </p> 
         <p>
          <!--more--></p> 
         <p>Technically, most web pages already are SPAs; it's the complexity of a page that differentiates a <em>web page</em> from a <em>web app</em>. In my opinion, a page becomes an app when you incorporate workflows, CRUD operations, and state management around certain tasks. You're working with a SPA when each of these tasks take place on the same page (using AJAX for client/server communication, of course).</p> 
         <p>Let's start with this common understanding, and dive into some of the more important things that should be considered when building SPAs. </p> 
         <hr> 
         <p>There are numerous points to consider before building a new app; to make matters worse, the expansive web development landscape can be intimidating at the outset. I have been in those unsettling shoes, but fortunately, the past few years have brought consensus on the tools and techniques that make the application development experience as enjoyable and productive as possible. </p> 
         <p>Most apps consist of both client and server-side pieces; although this article focuses mostly on the client-side portion of an app, I'll provide a few server-side pointers toward the end of this article.</p> 
         <p>There is a colorful mix of technologies on the client-side, as well as several libraries and practices that enable a productive app development experience. This can be summarized, using the following word cloud.</p> 
         <div class="tutorial_image">
          <img src="https://cdn.tutsplus.com/net/authors/jeremymcpeak/spa-list-cloud.png" data-original-url="http://cdn.tutsplus.com/net.tutsplus.com/authors/jeremymcpeak/spa-list-cloud.png" alt="Important Considerations - checklist" title="" border="0">
         </div> 
         <p>I will expand on each of the points above in the following sections.</p> 
         <hr> 
         <h2>Picking an Application Framework</h2> 
         <p>There are an abundance of frameworks to choose from. Here's but a handful of the most popular:</p> 
         <ul> 
          <li><a href="http://backbonejs.org/">Backbone</a></li> 
          <li><a href="http://canjs.us/">CanJS</a></li> 
          <li><a href="http://spinejs.com/">SpineJS</a></li> 
          <li><a href="http://batmanjs.org/">BatmanJS</a></li> 
          <li><a href="http://emberjs.com/">EmberJS</a></li> 
          <li><a href="http://angularjs.org/">AngularJS</a></li> 
          <li><a href="http://meteor.com/">Meteor</a></li> 
         </ul> 
         <p>Choosing a framework is easily one of the most important choices you will make for your app. Certainly, you'll want to choose the best framework for your team and app. Each of the above frameworks incorporate the MVC design pattern (in some form or another). As such, it's quite common to refer to them as MVC frameworks. If we had to order these frameworks on a scale of complexity, learning curve and feature set, <em>from left to right</em>, it might look like:</p> 
         <div class="tutorial_image">
          <img src="https://cdn.tutsplus.com/net/authors/jeremymcpeak/spa-app-frameworks.png" data-original-url="http://cdn.tutsplus.com/net.tutsplus.com/authors/jeremymcpeak/spa-app-frameworks.png" alt="App Frameworks" title="" border="0">
         </div> 
         <p>Although dissimilar in their implementation and level of sophistication, all the aforementioned frameworks provide some common abstractions, such as: </p> 
         <blockquote class="pullquote">
          <p>Just looking at the past five years, there has been an explosive growth in libraries, tools and practices.</p>
         </blockquote> 
         <ul> 
          <li> <strong>Model</strong>: a wrapper around a JSON data structure with support for property getters/setters and property change notification.</li> 
          <li> <strong>Collection</strong>: a collection of models. Provides notifcations when a model is added, removed, or changed in the collection.</li> 
          <li> <strong>Events</strong>: a standard pattern to subscribe to and publish notifications.</li> 
          <li> <strong>View</strong>: A backing object for a DOM fragment with support for listening to DOM events relative to the DOM fragment. The View has access to the corresponding Model instance. In some frameworks, there is also a <strong>Controller</strong> that orchestrates changes between the View and Model.</li> 
          <li> <strong>Routing</strong>: Navigation within an app via URLs. Relies on the browser history API.</li> 
          <li> <strong>Syncing</strong>: Persisting model changes via Ajax calls.</li> 
         </ul> 
         <p>More advanced frameworks, like CanJS, BatmanJS, EmberJS and AngularJS, expand on these basic features by providing support for automatic data-binding and client-side templates. The templates are data-bound and keep the view in sync with any changes to the model. If you decide to pick an advanced framework, you will certainly get a lot of out-of-the-box features, but it also expects you to build your app in a certain way. </p> 
         <p>Of all the previously listed frameworks, Meteor is the only full-stack framework. It provides tools not only for client-side development, but it also provides you with a server-side piece, via NodeJS, and end-to-end model synchronization, via MongoDB. This means that, when you save a model on the client, it automatically persists in MongoDB. This is a fantastic option, if you run a Node backend and use MongoDB for persistence.</p> 
         <p>Based on the complexity of your app, you should pick the framework that makes you the most productive. There certainly will be a learning curve, but that's a one-time toll you pay for express-lane development. Be sure to carve out some time to evaluate these frameworks, based on a representative use-case. </p> 
         <div class="tip-shortcode"> 
          <p><strong>Note</strong>: If you want to learn more about these frameworks from their creators, listen to <a href="http://www.infoq.com/throne_of_js/">these videos from ThroneJS</a>.</p> 
         </div> 
         <hr> 
         <h2>Client-Side Templates</h2> 
         <p>The most popular JavaScript-based templating systems are <a href="http://underscorejs.org/#template">Underscore templates</a> and <a href="http://handlebarsjs.com/">Handlebars</a>.</p> 
         <blockquote>
          <p>Some of the advanced frameworks from the previous section offer built-in templating systems.</p>
         </blockquote> 
         <p>For example, EmberJS has built-in support for Handlebars. However, you do have to consider a templating engine if you decide to use a lean framework, such as Backbone. Underscore is an excellent starting point, if you have limited templating requirements. Otherwise, Handlebars works great for more advanced projects. It also offers many built-in features for more expressive templates.</p> 
         <p>If you find that you require a large number of client-side templates, you can save some computation time by pre-compiling the templates on the server. Pre-compilation gives you plain JavaScript functions that you invoke to improve the load time of the page. Handlebars supports pre-compilation, making it worth the time and effort to fully explore.</p> 
         <p>ExpressJS users can even use the same templating engine on the client as on the server, giving you the benefit of sharing your templates between both the client and server.</p> 
         <hr> 
         <h2>Modular Development</h2> 
         <blockquote class="pullquote">
          <p>Using a preprocessor requires an extra step in your build process.</p>
         </blockquote> 
         <p>JavaScript code is traditionally added to the page, via the <code>&lt;script /&gt;</code> element. You typically list libraries and other dependencies first, and then list the code that references those dependencies. This style works well, when you only need to include a few files; however, it will quickly become a nightmare to maintain, as you include additional scripts.</p> 
         <p>One solution to this problem is to treat each script file as a <em>Module</em>, and identify it by a name or relative file path. Using these semantics, and with the support of libraries, like <a href="http://requirejs.org/">RequireJS</a> and <a href="https://github.com/substack/node-browserify">Browserify</a>, you can build your app using a module-based system. </p> 
         <p>The module thus becomes a way to identify the functionality within the app. You can organize these modules, using a certain folder structure that groups them based on a particular feature or functionality. Modules help in managing your application's scripts, and it also eliminates global dependencies that must be included with <code>&lt;script /&gt;</code> elements before the application scripts. For libraries that are not AMD compatible, RequireJS offers a <em>shim</em> feature that exposes non-AMD scripts as modules.</p> 
         <blockquote>
          <p>There are currently two types of module-based systems: <a href="http://requirejs.org/docs/whyamd.html">AMD (Asynchronous Module Definition)</a> and <a href="http://www.commonjs.org/">CommonJS</a>.</p>
         </blockquote> 
         <p>In AMD, each module contains a single top-level <code>define()</code> statement that lists all required dependencies, and an export function that exposes the module's functionality. Here's an example:</p> 
         <pre class="brush: js noskimlinks noskimwords">
define([
    // listing out the dependencies (relative paths)
    'features/module/BaseView',
    'utils/formatters'
], function(BaseView, formatters) { // Export function that takes in the dependencies and returns some object

    // do something here

    // An explicit require
    var myModule = require('common/myModule');

    // Object exposing some functionality
    return { ... };
});</pre> 
         <p>CommonJS module names are based on either a relative file path or a built-in module lookup process. There is no <code>define()</code> function in any module, and dependencies are explicitly stated by calls to <code>require()</code>. A module exposes its functionality, via the <code>module.exports</code> object, which each module automatically creates. Here's a CommonJS example:</p> 
         <pre class="brush: js noskimlinks noskimwords">var fs = require('fs'), // standard or built-in modules
    path = require('path'),
    formatters = require('./utils/formatters'); // relative file path as module name

// Export my code
module.exports = { ... };</pre> 
         <p>The CommonJS module style is more prevalent in NodeJS applications, where it makes sense to skip the call to <code>define()</code> call - you are working with a file-system based module lookup. Interestingly, you can do the same in a browser, using Browserify.</p> 
         <hr> 
         <h2>Package Management</h2> 
         <blockquote class="pullquote">
          <p>Performance should be on your mind as you build and add features to your app.</p>
         </blockquote> 
         <p>Most apps have at least one dependency, be it a library or some other third party piece of code. You'll find that you need some way to manage those dependencies as their number increases, and you need to insulate yourself from any breaking changes that newer versions of those dependencies may introduce.</p> 
         <p>Package management identifies all the dependencies in your app with specific names and versions. It gives you greater control over your dependencies, and ensures that everyone on your team is using an identical version of the library. The packages that your app needs are usually listed in a single file that contains a library's version and name. Some of the common package managers for different tech stacks are: </p> 
         <ul> 
          <li>Linux: Aptitude</li> 
          <li>.NET: Nuget</li> 
          <li>PERL: CPAN</li> 
          <li>Ruby: Gems</li> 
          <li>PHP: Composer</li> 
          <li>Node: NPM</li> 
          <li>Java: Maven and Gradle</li> 
         </ul> 
         <p>Although package management is more of a server-side ability, it's gaining popularity in client-side development circles. Twitter introduced <a href="https://github.com/twitter/bower">Bower</a>, a browser package manager similar to NPM for Node. Bower lists the client-side dependencies in <code>component.json</code>, and they are downloaded by running the <code>bower</code> CLI tool. For example, to install jQuery, from the Terminal, you would run:</p> 
         <pre class="brush: bash noskimlinks noskimwords">bower install jquery</pre> 
         <p>The ability to control a project's dependencies makes development more predictable, and provides a clear list of the libraries that an app requires. If you consider consolidating your libraries in the future, doing so will be easier with your package listing file.</p> 
         <hr> 
         <h2>Unit and Integration Testing</h2> 
         <p>It goes without saying that unit testing is a critical part of app development. It ensures that features continue to work as you refactor code, introduce libraries, and make sweeping changes to your app. Without unit tests, it will prove difficult to know when something fails, due to a minor code change. Coupled with end-to-end integration testing, it can be a powerful tool, when making architectural changes.</p> 
         <p>On the client-side, <a href="http://pivotal.github.com/jasmine/">Jasmine</a>, <a href="http://visionmedia.github.com/mocha/">Mocha</a> and <a href="http://qunitjs.com/">Qunit</a> are the most popular testing frameworks. Jasmine and Mocha support a more Behavior-Driven Development (BDD) style, where the tests read like English statements. QUnit, on the other hand, is a more traditional unit testing framework, offering an assertion-style API. </p> 
         <blockquote>
          <p>Jasmine, Mocha or Qunit run tests on a single browser.</p>
         </blockquote> 
         <p>If you want to gather test results from multiple browsers, you can try a tool like <a href="http://vojtajina.github.com/testacular/">Testacular</a> that runs your tests in multiple browsers.</p> 
         <p>To take testing the whole nine yards, you'll likely want to have integration tests in your app, using Selenium and Cucumber/Capybara. <a href="http://cukes.info/">Cucumber</a> allows you to write tests (aka <em>features</em>) in an English-like syntax, called <em>Gherkin</em>, which can even be shared with the business folks. Each test statement in your Cucumber file is backed by executable code that you can write in Ruby, JavaScript or any of the other supported languages. </p> 
         <p>Executing a Cucumber feature file runs your executable code, which in turn tests the app and ensures that all business functionality has been properly implemented. Having an executable feature file is invaluable for a large project, but it might be overkill for smaller projects. It definitely requires a bit of effort to write and maintian these Cucumber scripts, so it really boils down to a team's decision.</p> 
         <hr> 
         <h2>UI Considerations</h2> 
         <blockquote class="pullquote">
          <p>Having a good working knowledge of CSS will help you achieve innovative designs in HTML.</p>
         </blockquote> 
         <p>The UI is my favorite portion of an app; it's one of the things that immediately differentiates your product from the competition. Although apps differ in their purpose and look and feel, there are a few common responsibilities that most apps have. UI design and architecture is a fairly intensive topic, but it's worth mentioning a few design points:</p> 
         <ul> 
          <li> <strong>Form Handling</strong>: use different input controls (numeric inputs, email, date picker, color picker, autocomplete), validations on form submit, highlight errors in form inputs, and propagating server-side errors on the client.</li> 
          <li> <strong>Formatting</strong>: apply custom formats to numbers and other values.</li> 
          <li> <strong>Error Handling</strong>: propagate different kinds of client and server errors. Craft the text for different nuances in errors, maintain an error dictionary and fill placeholders with runtime values. </li> 
          <li> <strong>Alerts and Notifications</strong>: tell the user about important events and activities, and show system messages coming from the server.</li> 
          <li> <strong>Custom Controls</strong>: capture unique interaction patterns in the app as controls that can be reused. Identify the inputs and outputs from the control without coupling with a specific part of the app.</li> 
          <li> <strong>Grid System</strong>: build layouts using a grid system, like Compass Susy, 960gs, CSS Grid. The grid system will also help in creating responsive layout for different form factors.</li> 
          <li> <strong>UI Pattern Library</strong>: get comfortable with common UI patterns. Use <a href="http://www.infragistics.com/products/quince/">Quince</a> for reference.</li> 
          <li> <strong>Layered Graphics</strong>: understand the intricacies of CSS, the box models, floats, positioning, etc. Having a good working knowledge of CSS will help you achieve innovative designs in HTML.</li> 
          <li> <strong>Internationalization</strong>: adapt a site to different locales. Detect the locale using the <code>Accept-Language</code> HTTP header or through a round-trip to gather more info from the client.</li> 
         </ul> 
         <hr> 
         <h2>CSS Preprocessors</h2> 
         <p>CSS is a deceptively simple language that has simple constructs. Interestingly, it can also be very unwieldy to manage, especially if there are many of the same values used among the various selectors and properties. It's not uncommon to reuse a set of colors throughout a CSS file, but doing so introduces repetition, and changing those repeated values increases the potential for human error.</p> 
         <p>CSS preprocessors solve this problem and help to organize, refactor and share common code. Features, such as variables, functions, mixins and partials, make it easy to maintain CSS. For example, you could store the value of a common color within a variable, and then use that variable wherever you want to use its value.</p> 
         <blockquote>
          <p>Using a preprocessor requires an extra step in your build process: you have to generate the final CSS.</p>
         </blockquote> 
         <p>There are, however, tools which auto-compile your files, and you can also find libraries that simplify stylesheet development. <a href="http://sass-lang.com/">SASS</a> and <a href="http://learnboost.github.com/stylus/">Stylus</a> are two popular preprocessors that offer corresponding helper libraries. These libraries also make it easy to build grid-based systems and create a responsive page layout that adapts to different form factors (tablets and phones).</p> 
         <p>Although CSS preprocessors make it easy to build CSS with shared rules, you still have the responsibility of structuring it well, and isolating related rules into their own files. Some principles from <a href="">SMACSS</a> and <a href="http://oocss.org/">OOCSS</a> can serve as a great guide during this process.</p> 
         <blockquote> 
          <p> <a href="https://tutsplus.com/ebook/scalable-and-modular-architecture-for-css/">Scalable and Modular Architecture for CSS</a> is included, as part of a Tuts+ Premium membership. </p> 
         </blockquote> 
         <hr> 
         <h2>Version Control</h2> 
         <p>If you know a hip developer, then you're probably aware that <a href="http://git-scm.com/">Git</a> is the reigning champion of all version control systems (VCS). I won't go into all the details of why Git is superior, but suffice it to say that branching and merging (two very common activities during development) are mostly hassle free. </p> 
         <p>A close parallel to Git, in terms of philosophy, is <a href="http://mercurial.selenic.com/">Mercurial (hg)</a>--although it is not as popular as Git. The next best alternative is the long-standing <a href="http://subversion.tigris.org/">Subversion</a>. The choice of VCS is greatly dependent on your company standards, and, to some extent, your team. However, if you are part of a small task force, Git is easily the preferred option.</p> 
         <hr> 
         <h2>Browser Considerations</h2> 
         <blockquote class="pullquote">
          <p>It goes without saying that unit testing is a critical part of app development.</p>
         </blockquote> 
         <p>There are a variety of browsers that we must support. Libraries, like jQuery and Zepto, already abstract the DOM manipulation API, but there are other differences in JavaScript and CSS, which require extra effort on our parts. The following guidelines can help you manage these differences:</p> 
         <ul> 
          <li>Use a tool, like <a href="https://saucelabs.com/">Sauce Labs</a> or <a href="http://www.browserstack.com/">BrowserStack</a> to test the website on multiple browsers and operating systems.</li> 
          <li>Use polyfills and shims, such as <a href="https://github.com/kriskowal/es5-shim">es5shim</a> and <a href="http://modernizr.com/">Modernizr</a> to detect if the browser supports a given feature before calling the API.</li> 
          <li>Use CSS resets, such as <a href="http://necolas.github.com/normalize.css/">Normalize</a>, <a href="http://www.blueprintcss.org/">Blueprint</a>, and <a href="http://meyerweb.com/eric/tools/css/reset/">Eric Myer's Reset</a> to start with a clean slate look on all browsers.</li> 
          <li>Use vendor prefixes (<code>-webkit-</code>, <code>-moz-</code>, <code>-ms-</code>) on CSS properties to support different rendering engines.</li> 
          <li>Use browser compatibility charts, such as <a href="http://www.findmebyip.com/litmus/">findmebyIP</a> and <a href="http://caniuse.com/">canIuse</a>.</li> 
         </ul> 
         <p>Managing browser differences may involve a bit of trial and error; Google and StackOverflow can be your two best friends, when you find yourself in a browser-induced jam.</p> 
         <hr> 
         <h2>Libraries</h2> 
         <p>There are a few libraries that you might want to consider:</p> 
         <ul> 
          <li> <strong>Visualizations</strong>: <a href="http://omnipotent.net/jquery.sparkline/#s-about">Spark lines</a>, <a href="http://www.highcharts.com/">Highcharts</a>, <a href="http://d3js.org/">D3</a>, <a href="http://tenxer.github.com/xcharts/">xCharts</a>, and <a href="http://raphaeljs.com/">Raphal</a>.</li> 
          <li> <strong>Formatting</strong>: <a href="http://numeraljs.com/">numeraljs</a>, <a href="http://josscrowcroft.github.com/accounting.js/">accountingjs</a>, and <a href="http://momentjs.com/">moment</a>.</li> 
          <li> <strong>Controls</strong>: <a href="http://twitter.github.com/bootstrap/">Bootstrap</a>, <a href="http://jqueryui.com/">jQuery UI</a>, and <a href="http://ivaynberg.github.com/select2/">select2</a>.</li> 
          <li>If you decide to use <strong>BackboneJS</strong>, you can look at <a href="http://marionettejs.com/">Backbone.Marionette</a>, which provides several helper utilities to make your development go faster.</li> 
          <li> <strong>Helpers</strong>: <a href="http://underscorejs.org/">Underscore</a>, <a href="http://sugarjs.com/">Sugar</a>, <a href="https://github.com/kriskowal/es5-shim">es5shim</a>, <a href="http://modernizr.com/">Modernizr</a>, and <a href="http://html5boilerplate.com/">Html5 Boilerplate</a>.</li> 
         </ul> 
         <hr> 
         <h2>Minification</h2> 
         <p>Before deploying your application, it's a good idea to combine all of your scripts into a single file; the same can be said for your CSS. This step is generally referred to as minification, and it aims to reduce the number of HTTP requests and the size of your scripts.</p> 
         <p>You can minify JavaScript and CSS with: <a href="http://requirejs.org/docs/optimization.html">RequireJS optimizer</a>, <a href="https://github.com/mishoo/UglifyJS">UglifyJS</a>, and <a href="http://documentcloud.github.com/jammit/">Jammit</a>. They also combine your images and icons into a single sprite sheet for even more optimization.</p> 
         <div class="tip-shortcode"> 
          <strong>Editor's Note: </strong>I recommend that you use 
          <a href="gruntjs.com">Grunt</a> or 
          <a href="http://yeoman.io">Yeoman</a> (which uses Grunt) to easily build and deploy your applications. 
         </div> 
         <hr> 
         <h2>Tools of the Trade</h2> 
         <blockquote class="pullquote">
          <p>Twitter introduced <a href="https://github.com/twitter/bower">Bower</a>, a browser package manager similar to NPM for Node.</p>
         </blockquote> 
         <p>I would be remiss if I did not mention the tools for building SPAs. The following lists a few:</p> 
         <ul> 
          <li> <strong><a href="http://www.jshint.com/">JsHint</a></strong> to catch lint issues in your JavaScript files. This tool can catch syntactic issues, such as missing semicolons and enforcing a certain code style on the project.</li> 
          <li>Instead of starting a project from scratch, consider a tool, such as <strong><a href="http://yeoman.io/">Yeoman</a></strong> to quickly build the initial scaffolding for the project. It provides built-in support for CSS preprocessors (like SASS, <a href="http://lesscss.org/" target="_blank">Less</a> and Stylus), compiling CoffeeScript files to JavaScript and watching for file changes. It also prepares your app for deployment by minifying and optimizing your assets. Like Yeoman, there are other tools to consider, such as <a href="http://mimosajs.com/">MimosaJS</a> and <a href="http://middlemanapp.com/">Middleman</a>.</li> 
          <li>If you're looking for a make-like tool for JavaScript, look no further than <strong><a href="http://gruntjs.com/">Grunt</a></strong>. It is an extensible build tool that can handle a variety of tasks. Yeoman uses Grunt to handle all of its tasks.</li> 
          <li> <strong><a href="https://github.com/remy/nodemon">Nodemon</a></strong> for auto-starting a Node program each time a file changes. A simliar tool is <a href="https://github.com/nodejitsu/forever">forever</a>.</li> 
          <li> <strong>Code editors</strong>, such as <a href="http://www.sublimetext.com/">Sublime Text</a>, <a href="http://www.vim.org/">Vim</a>, and <a href="http://www.jetbrains.com/webstorm/">JetBrains WebStorm</a>.</li> 
          <li> <strong>Command line</strong> tools ZSH or BASH. Master the shell because it can be very, effective especially when working with tools like Yeoman, Grunt, Bower and NPM.</li> 
          <li> <strong><a href="http://mxcl.github.com/homebrew/">Homebrew</a></strong> is a simple package manager for installing utilities.</li> 
         </ul> 
         <hr> 
         <h2>Performance Considerations</h2> 
         <blockquote class="pullquote">
          <p>CSS preprocessors make it easy to build CSS with shared rules.</p>
         </blockquote> 
         <p>Rather than treating this as an after-thought, performance should be on your mind as you build and add features to your app. If you encounter a performance issue, you should first profile the app. The Webkit inspector offers a built-in profiler that can provide a comprehensive report for CPU, memory and rendering bottlenecks. The profiler helps you isolate the issue, which you can then fix and optimize. Refer to the <a href="https://developers.google.com/chrome-developer-tools/">Chrome Developer Tools</a> for in-depth coverage of the Chrome web inspector.</p> 
         <p>Some common performance improvements include:</p> 
         <ul> 
          <li>Simplify CSS selectors to minimize recalculation and layout costs.</li> 
          <li>Minimize DOM manipulations and remove unnecessary elements.</li> 
          <li>Avoid data bindings when the number of DOM elements run into hundreds.</li> 
          <li>Clean up event handlers in view instances that are no longer needed.</li> 
          <li>Try to generate most of the HTML on the server-side. Once on the client, create the backing view with the existing DOM element.</li> 
          <li>Have region-specific servers for faster turn around.</li> 
          <li>Use CDNs for serving libraries and static assets.</li> 
          <li>Analyze your web page with tools like YSlow and take actions outlined in the report.</li> 
         </ul> 
         <p>The above is only a cursory list. Visit <a href="http://www.html5rocks.com/en/features/performance">Html5Rocks</a> for more comprehensive performance coverage.</p> 
         <hr> 
         <h2>Auditing and Google Analytics</h2> 
         <p>If you plan on tracking your app's usage or gathering audit trails around certain workflows, <a href="http://www.google.com/analytics/">Google Analytics</a> (GA) is probably your best solution. By including a simple GA script on each page with your tracking code, you can gather a variety of your app's metrics. You can also set up goals at the Google Analytics website. This fairly extensive topic is worth investigating, if tracking and auditing is an important concern.</p> 
         <hr> 
         <h2>Keeping up With the Jones</h2> 
         <p>The world of web development changes quickly. Just looking at the past five years, there has been an explosive growth in libraries, tools and practices. The best way to keep tabs on the web's evolution is to subscribe to blogs (like this one), newsletters and just being curious:</p> 
         <ul> 
          <li>Read <a href="http://updates.html5rocks.com/2012/04/Round-up-of-Web-Browser-Internals-Resources">How Browsers Work</a>.</li> 
          <li> <a href="http://www.webplatform.org/">Learn the platform - Web Platform</a>, a project sponsored by the major vendors.</li> 
          <li>Subscribe to <a href="http://net.tutsplus.com/">Nettuts+</a>!</li> 
          <li>Subscribe to <a href="http://html5weekly.com/">Html5 Weekly</a>, <a href="http://javascriptweekly.com/">JavaScript Weekly</a>, and <a href="http://web-design-weekly.com/">Web Design Weekly</a>.</li> 
          <li>Attend conferences, like <a href="http://jsconf.com/">JSConf</a>, <a href="http://html5devconf.com/">Html5DevConf</a>, <a href="http://fluentconf.com/fluent2013">FluentConf</a>, local user groups and conferences.</li> 
          <li>Visit <a href="http://www.html5rocks.com/en/">Html5 Rocks</a>.</li> 
          <li>Explore <a href="https://github.com/languages/JavaScript">GitHub</a>.</li> 
         </ul> 
         <hr> 
         <h2>Operations Management</h2> 
         <p>The client-side, although looking like a large piece of the stack, is actually only half of the equation. The other half is the server, which may also be referred to as operations management. Although beyond the scope of this article, these operations can include:</p> 
         <ul> 
          <li>continuous integration, using build servers such as <a href="http://www.jetbrains.com/teamcity/">TeamCity</a>, <a href="http://jenkins-ci.org/">Jenkins</a>, and <a href="http://hudson-ci.org/">Hudson</a>.</li> 
          <li>persistence, data redundancy, failover and disaster recovery.</li> 
          <li>caching data in-memory and invalidating the cache at regular intervals.</li> 
          <li>handling roles and permissions and validating user requests.</li> 
          <li>scaling under heavy load.</li> 
          <li>security, SSL certificates, and exploit-testing.</li> 
          <li>password management.</li> 
          <li>support, monitoring and reporting tools.</li> 
          <li>deployment and staging.</li> 
         </ul> 
         <hr> 
         <h2>Summary</h2> 
         <p>As you can see, developing an app and bringing it to production involves a variety of modern technologies. We focused primarily on client-side development, but don't forget the server-side portion of the app. Separately, they're useless, but together, you have the necessary layers for a working application.</p> 
         <p>With so much to learn, you wouldn't be alone if you feel overwhelmed. Just stick with it, and don't stop! You'll get there soon enough. </p>
