       <h2>Multiple Activities</h2> 
       <p>Even the simplest applications have more than one functionality. Hence, there is often a need to deal with multiple activities. For example, a game can have two activities: a high scores screen and a game screen. A notepad can have three activities: view a list of notes, read a selected note, and edit a selected or new note.</p> 
       <p>The main activity, as defined in the AndroidManifest XML file, is started when the application is started. This activity can launch another activity, usually after a trigger event. This causes the main activity to pause while the secondary activity is active. When the secondary activity ends, the main activity is brought to the foreground and resumed.</p> 
       <p>To activate a particular component of the application, an intent naming the component explicitly is used. If instead the requirements of an application can be specified by intent filters, an implicit intent can be used. The system then determines the best component or components to use, even if it is in a separate application or native to the OS. Note that unlike other activities, implicit intents that reside in other applications do not need to be declared in the current application's AndroidManifest file.</p> 
       <p>Android utilizes implicit intents as often as possible, providing a powerful framework for modular functionality. When a new component is developed that meets the required implicit intent filter, it can be used in place of an Android internal intent. For example, say a new application for displaying phone contacts is loaded on an Android device. When a user selects a contact, the Android system finds all available activities with the proper intent filter for viewing contacts and asks the user to decide which one should be used.</p> 
       <h3>Recipe: Using Buttons and TextView</h3> 
       <p>To fully demonstrate multiple activities, it is useful to use a trigger event. A button press is introduced here for that purpose. The steps to adding a button to a given layout and assigning an action to a button press are</p> 
       <ol> 
        <li>Put a button in the designated layout XML file: <pre>&lt;Button android:id="@+id/trigger"
   android:layout_width="100dip" android:layout_height="100dip"
   android:text="Press this button" /&gt;
</pre> </li> 
        <li>Declare a button that points to the button ID in the layout file: <pre>Button startButton = (Button) findViewById(R.id.trigger);
</pre> </li> 
        <li>Specify a listener for when the button is clicked: <pre>//setup button listener
startButton.setOnClickListener(new View.OnClickListener() {
    //insert onClick here
});
</pre> </li> 
        <li>Override the <tt>onClick</tt> function for the listener to do the required action: <pre>public void onClick(View view) {
    // do something here

}
</pre> </li> 
       </ol> 
       <p>To show the result of an action, it is useful to change the text on the screen. The steps for defining a text field and changing it programmatically are</p> 
       <ol> 
        <li>Put a text field in the designated layout XML file with an ID. It can also be initialized to some value (here, it can be initialized to the string named "hello" in the strings.xml file): <pre>&lt;TextView android:id="@+id/hello_text"
   android:layout_width="fill_parent"
   android:layout_height="wrap_content"
   android:text="@string/hello"
/&gt;
</pre> </li> 
        <li>Declare a TextView that points to the TextView ID in the layout file: <pre>private TextView tv = (TextView) findViewById(R.id.hello_text);
</pre> </li> 
        <li>If the text needs to be changed, use the setText function: <pre>tv.setText("new text string");
</pre> </li> 
       </ol> 
       <p>These two UI techniques are used in the subsequent recipes in this chapter. A more complete demonstration of UI techniques is covered in Chapter 4.</p> 
       <h3>Recipe: Launching Another Activity from an Event</h3> 
       <p>In this recipe, <tt>MenuScreen</tt> is the main activity as shown in Listing 2.8. It launches the <tt>PlayGame</tt> activity. Here the trigger event is implemented as a button click using the <tt>Button</tt> widget.</p> 
       <p>When a user clicks the button, the <tt>startGame()</tt> function runs; it launches the <tt>PlayGame</tt> activity. When a user clicks the button in the <tt>PlayGame</tt> activity, it calls <tt>finish()</tt> to return control to the calling activity. The steps for launching an activity are</p> 
       <ol> 
        <li>Declare an Intent that points to the activity to be launched.</li> 
        <li>Call startActivity on this intent.</li> 
        <li>Declare the additional activity in the AndroidManifest.</li> 
       </ol> 
       <h4>Listing 2.8. src/com/cookbook/launch_activity/MenuScreen.java</h4> 
       <pre>package com.cookbook.launch_activity;

import android.app.Activity;
import android.content.Intent;
import android.os.Bundle;
import android.view.View;
import android.widget.Button;

public class MenuScreen extends Activity {

    @Override
    public void onCreate(Bundle savedInstanceState) {
        super.onCreate(savedInstanceState);
        setContentView(R.layout.main);

        //setup button listener
        Button startButton = (Button) findViewById(R.id.play_game);
        startButton.setOnClickListener(new View.OnClickListener() {
            public void onClick(View view) {
                startGame();
            }
        });
    }

    private void startGame() {
        <strong>Intent launchGame = new Intent(this, PlayGame.class);</strong>
        <strong>startActivity(launchGame);</strong>
    }
}
</pre> 
       <div class="sidebar"> 
        <p class="title">Provide Current Context in an Anonymous Inner Class</p> 
        <p>Note the additional consideration needed for launching an activity with a Button press, as shown in Listing 2.8. The intent needs a context. However, using the <tt>this</tt> shortcut in the <tt>onClick</tt> function is not properly resolved. Different ways to provide current context in an anonymous inner class are</p> 
        <ul> 
         <li>Use <tt>Context.this</tt> instead of <tt>this.</tt> </li> 
         <li>Use <tt>getApplicationContext()</tt> instead of <tt>this.</tt> </li> 
         <li>Explicitly use the class name <tt>MenuScreen.this</tt>.</li> 
        </ul> 
        <p>Call a function that is declared at the right context level. This is what is used in Listing 2.8: startGame().</p> 
        <p>Each of these methods can usually be interchanged. Utilize the one that works best for the clarity of the situation.</p> 
       </div> 
       <p>The <tt>PlayGame</tt> activity shown in Listing 2.9 is simply a button with a <tt>onClick</tt> listener that calls <tt>finish()</tt> to return control to the main activity. More functionality can be added as needed to this activity, and multiple branches of the code can each lead to their own <tt>finish()</tt> calls.</p>

       <h3>Recipe: Launching an Activity for a Result Using Speech to Text</h3> 
       <p>In this recipe, launching an activity for a result is demonstrated. It also demonstrates how to utilize speech to text functionality from Google's <tt>RecognizerIntent</tt> and print the result to the screen. Here, the trigger event is a button press. It launches the <tt>RecognizerIntent</tt> activity, which does speech recognition on sound from the microphone and converts it into text. When finished, the text is passed back to the calling activity.</p> 
       <p>Upon return, the <tt>onActivityResult()</tt> function is first called with the returned data, and then the <tt>onResume()</tt> function is called to continue the activity as normal. The calling activity can have a problem and not return properly. Therefore, the <tt>resultCode</tt> should always be checked to ensure <tt>RESULT_OK</tt> before continuing to parse the returned data.</p> 
       <p>Note that in general any launched activity that returns data causes the same <tt>onActivityResult()</tt> function to be called. Therefore, a request code is customarily used to distinguish which activity is returning. When the launched activity finishes, it returns control to the calling activity and calls <tt>onActivityResult()</tt> with the same request code.</p> 
       <p>The steps for launching an activity for result are</p> 
       <ol> 
        <li>Call startActivityForResult() with an intent, defining the launched activity and an identifying requestCode.</li> 
        <li>Override the onActivityResult() function to check on the status of the result, check for the expected requestCode, and parse the returned data.</li> 
       </ol> 
       <p>The steps for using <tt>RecognizerIntent</tt> are</p> 
       <ol> 
        <li>Declare an intent with action ACTION_RECOGNIZE_SPEECH.</li> 
        <li>Add any extras to the intent; at least EXTRA_LANGUAGE_MODEL is required. This can be set as either LANGUAGE_MODEL_FREE_FORM or LANGUAGE_MODEL_WEB_SEARCH.</li> 
        <li>The returned data bundle contains a list of strings with possible matches to the original text. Use <tt>data.getStringArrayListExtra</tt> to retrieve this data. This should be cast as an ArrayList for use later.</li> 
       </ol> 
       <p>A <tt>TextView</tt> is used to display the returned text to the screen. The main activity is shown in Listing 2.14.</p> 
       <p>The additional supporting files needed are the <strong>main.xml</strong> and <strong>strings.xml</strong>, which need to define a button and the TextView to hold the result. This is accomplished using Listing 2.10 and 2.12 in the "Launching Another Activity from an Event" recipe. The AndroidManifest needs to declare only the main activity, which is the same as the basic "Creating an Activity" recipe. The RecognizerIntent activity is native to the Android system and does not need to be declared explicitly to be utilized.</p> 
       <h4>Listing 2.14. src/com/cookbook/launch_for_result/RecognizerIntent Example.java</h4> 
       <pre>package com.cookbook.launch_for_result;

       <h3>Recipe: Implementing a List of Choices</h3>

       <h3>Recipe: Using Implicit Intents for Creating an Activity</h3> 
       <p>Implicit intents do not specify an exact component to use. Instead, they specify the functionality required through a filter, and the Android system must determine the best component to utilize. An intent filter can be either an action, data, or a category.</p> 
       <p>The most commonly used intent filter is an action, and the most common action is <tt>ACTION_VIEW</tt>. This mode requires a uniform resource identifier (URI) to be specified and then displays the data to the user. It does the most reasonable action for the given URI. For example, the implicit intents in case 0, 1, and 2 in the following example have the same syntax but produce different results.</p> 
       <p>The steps for launching an activity using an implicit intent are</p> 
       <ol> 
        <li>Declare the intent with the appropriate filter specified (ACTION_VIEW, ACTION_WEB_SEARCH, and so on).</li> 
        <li>Attach any extra information to the intent required to run the activity.</li> 
        <li>Pass this intent to startActivity().</li> 
       </ol> 
       <p>This is shown for multiple intents in Listing 2.15.</p> 
       <h4>Listing 2.15. src/com/cookbook/implicit_intents/ListActivityExample.java</h4> 
       <pre>package com.cookbook.implicit_intents;

       <h3>Recipe: Passing Primitive Data Types Between Activities</h3> 
       <p>Sometimes data needs to be passed to a launched activity. Sometimes a launched activity creates data that needs to be passed back to the calling activity. For example, a final score of a game needs to be returned to a high-scores screen. The different ways to pass information between activities are</p> 
       <ol> 
        <li>Declare the relevant variable in the calling activity (for example, public <tt>int finalScore</tt>) and set it in the launched activity (for example, <tt>CallingActivity.finalScore=score</tt>).</li> 
        <li>Attach extras onto Bundles (demonstrated here).</li> 
        <li>Use Preferences to store data to be retrieved later (covered in Chapter 5, "User Interface Events").</li> 
        <li>Use the SQLite database to store data to be retrieved later (covered in Chapter 9).</li> 
       </ol> 
       <p>A <tt>Bundle</tt> is a mapping from String values to various parcelable types. It can be created by adding extras to an intent. The following example shows data being passed from the main activity to the launched activity, where it is modified and passed back.</p> 
       <p>The variables (in this case, an <tt>integer</tt> and a <tt>String</tt>) are declared in the <tt>StartScreen</tt> activity. When the intent is created to call the <tt>PlayGame</tt> class, these variables are attached to the intent using the <tt>putExtra</tt> method. When the result is returned from the called activity, the variables can be read using the <tt>getExtras</tt> method. These calls are shown in Listing 2.16.</p> 
       <h4>Listing 2.16. src/com/cookbook/passing_data_activities/StartScreen.java</h4> 
       <pre>package com.cookbook.passing_data_activities;

import android.app.Activity;
import android.content.Intent;
import android.os.Bundle;
import android.view.View;
import android.widget.Button;
import android.widget.TextView;

public class StartScreen extends Activity {
    private static final int PLAY_GAME = 1010;
    private TextView tv;
    <strong>private int meaningOfLife = 42;</strong>
    <strong>private String userName = "Douglas Adams";</strong>

    @Override
    public void onCreate(Bundle savedInstanceState) {
        super.onCreate(savedInstanceState);
        setContentView(R.layout.main);
        tv = (TextView) findViewById(R.id.startscreen_text);

        //display initial values
        tv.setText(userName + ":" + meaningOfLife);

        //setup button listener
        Button startButton = (Button) findViewById(R.id.play_game);
        startButton.setOnClickListener(new View.OnClickListener() {
            public void onClick(View view) {
                startGame();
            }
        });
    }

    @Override
    protected void onActivityResult(int requestCode,
            int resultCode, Intent data) {
        if (requestCode == PLAY_GAME &amp;&amp; resultCode == RESULT_OK) {
            <strong>meaningOfLife = data.getExtras().getInt("returnInt");</strong>
            <strong>userName = data.getExtras().getString("userName");</strong>
            //show it has changed
            tv.setText(userName + ":" + meaningOfLife);
        }
        super.onActivityResult(requestCode, resultCode, data);
    }

    private void startGame() {
        Intent launchGame = new Intent(this, PlayGame.class);

        <strong>//passing information to launched activity</strong>
        <strong>launchGame.putExtra("meaningOfLife", meaningOfLife);</strong>
        <strong>launchGame.putExtra("userName", userName);</strong>

        startActivityForResult(launchGame, PLAY_GAME);
    }
}

       <p>The variables passed into the <tt>PlayGame</tt> activity can be read using the <tt>getIntExtra</tt> and <tt>getStringExtra</tt> methods. When the activity finishes and prepares an intent to return, the <tt>putExtra</tt> method can be used to return data back to the calling activity. These calls are shown in Listing 2.17.</p>
