        <p class="MsoNormal">Part of the advantage of ramping up to write Windows Store apps is that you can utilize your existing knowledge of HTML, CSS and Javascript.</p> 
        <p class="MsoNormal">The Windows Library for JavaScript (referred to as <span class="SpellE">WinJS</span>) extends the support for Javascript in Windows Store apps, has a number of helper functions for common app development scenarios. In the article below, well talk about how to get started with <span class="SpellE">WinJS</span> for creating custom types and namespaces, as well as handling and managing event listeners.</p> 
        <h1></h1> 
        <h1>Defining and deriving types with <span class="SpellE">WinJS.Class</span></h1> 
        <p class="MsoNormal">In the sections below, well talk about how you can use <span class="GramE">the <span class="codeChar">define</span></span> and <span class="codeChar">derive</span> methods in the <span class="SpellE">WinJS</span> library to quickly create your own custom types, and specify their property constructors.</p> 
        <h2><span class="SpellE">WinJS.Class.define</span></h2> 
        <p class="MsoNormal">The <span class="SpellE">WinJS.Class.define</span> function is a helper function for defining a JavaScript type. You supply a constructor function, a set of "instance" members (which are defined on the prototype of the type), and a set of "static" members (which are defined on the type itself).</p>

          &nbsp;
         </o:p></p>

        <h1>Organizing your code with <span class="SpellE">WinJS.Namespace</span></h1> 
        <p class="MsoNormal">In addition to creating custom types, you might want to organize these types into your own custom namespace, so that you can reference them conveniently and consistently through your code.</p> 
        <p class="MsoNormal">The <a href="http://msdn.microsoft.com/en-us/library/windows/apps/br212667.aspx"><span class="SpellE"><strong>WinJS.Namespace.define</strong></span></a> function allows you to create your own namespace as a way of organizing your code.</p> 
        <p class="MsoNormal">When you create a type or other element in a namespace, you reference it from outside the namespace by using its qualified name: <span class="SpellE"><em>Namespace.Type</em></span>.</p> 
        <p class="MsoNormal"><strong style="mso-bidi-font-weight: normal;">NOTE</strong>: You should define your elements before you define the namespace, and simply add references to these elements inside the namespace definition. If you define your elements inside the namespace definition, you may try to use <span class="GramE">the <span class="Code2Char"><span style="font-size: 10.0pt; mso-bidi-font-size: 11.0pt; line-height: 115%;">this</span></span></span> keyword to erroneously refer to other members of the namespace.</p>

        <h1>Adding functionality with <span class="SpellE">WinJS</span> <span class="SpellE">mixins</span></h1> 
        <p class="MsoNormal"><span class="SpellE">
          <o:p></o:p><em>Mixins</em></span> are objects that implement a certain class of functionality. For example, in the Windows Library for JavaScript there are <span class="SpellE">mixins</span> that manage events and <span class="SpellE">mixins</span> that handle DOM binding.</p> 
        <p class="MsoNormal"><a style="mso-comment-reference: RL_1; mso-comment-date: 20121206T1958;"></a><span class="SpellE"><span style="mso-comment-continuation: 1;">Mixins</span></span><span style="mso-comment-continuation: 1;"> aren't designed to be used directly, but you can use them to add functionality to your types. <span class="SpellE">Mixins</span> contain implemented functions that can be added to many types. </span></p>

        <h2>Adding <span class="SpellE">DOMEventMixin</span> to a type</h2> 
        <p class="MsoNormal"><a href="http://msdn.microsoft.com/en-us/library/windows/apps/br211693.aspx"><span class="SpellE"><strong>WinJS.Utilities.eventMixin</strong></span></a> is useful for defining events on types that are dealt with completely in JavaScript. It isn't useful when you're working with events that are associated with DOM elements. Windows Library for JavaScript controls wrap and manipulate HTML elements. These controls often contain events that are based on the DOM events raised from their associated DOM elements. For example, when you use a <a href="http://msdn.microsoft.com/en-us/library/windows/apps/br211681.aspx"><span class="SpellE"><strong>DatePicker</strong></span></a>, you find it easier to respond to a change event from the <span class="SpellE"><strong>DatePicker</strong></span> control rather than to track individual change events from the three <strong>select</strong> controls.</p> 
        <p class="MsoNormal">Associating events with the control type rather than the DOM element also protects the code from changes, for example if the implementation of the <a href="http://msdn.microsoft.com/en-us/library/windows/apps/br211681.aspx"><span class="SpellE"><strong>DatePicker</strong></span></a> changes the controls it contains.</p> 
        <p class="MsoNormal">The DOM event model allows for event propagation. DOM events are propagated up the DOM tree and can be handled at any level. This is useful when you have a lot of clickable elements. For example, you can put a single click handler function on the DIV element that contains the elements and then process all the click events from all the contained elements as they bubble up. For more information about DOM event handling, see <a href="http://msdn.microsoft.com/en-us/library/windows/apps/hh967791.aspx">Event capture and event bubbling</a>.</p> 
        <p class="MsoNormal"><a href="http://msdn.microsoft.com/en-us/library/windows/apps/hh700596.aspx"><span class="SpellE"><strong>WinJS.UI.DOMEventMixin</strong></span></a> contains methods that implement events that propagate up through the DOM tree. It has three functions: <a href="http://msdn.microsoft.com/en-us/library/windows/apps/hh700592.aspx"><span class="SpellE"><strong>addEventListener</strong></span></a>, <a href="http://msdn.microsoft.com/en-us/library/windows/apps/hh700598.aspx"><span class="SpellE"><strong>removeEventListener</strong></span></a>, and <a href="http://msdn.microsoft.com/en-us/library/windows/apps/hh700594.aspx"><span class="SpellE"><strong>dispatchEvent</strong></span></a>.</p>

        <h2>Event capture and event bubbling with DOM events</h2> 
        <p class="MsoNormal">In Windows Library for JavaScript there are several <span class="SpellE">addEventListener</span> methods (for example, <a href="http://msdn.microsoft.com/en-us/library/windows/apps/br229799.aspx"><span class="SpellE">WinJS.Application.addEventListener</span></a> and <a href="http://msdn.microsoft.com/en-us/library/windows/apps/br229661.aspx"><span class="SpellE">WinJS.UI.AppBar.addEventListener</span></a>). They all contain a <span class="SpellE"><em>useCapture</em></span> parameter, which is set to <strong>true</strong> if you want event capture or <strong>false</strong> if you want event bubbling. For more information, see <span class="SpellE">see</span> <a href="http://go.microsoft.com/fwlink/p/?LinkId=248283">Event capture</a> and <a href="http://go.microsoft.com/fwlink/p/?LinkId=248284">Event bubbling</a>.</p> 
        <p class="MsoNormal">In the <span class="SpellE"><strong>addEventListener</strong></span> method, if you set the <span class="SpellE"><em>useCapture</em></span> parameter to <strong>true</strong>, the order in which events are handled is top-down. That is, if there are event handlers for the specified event that belong to more than one element in the DOM tree, the handler belonging to the highest element in the tree is called first, and any handlers belonging to child elements are called after that. If you set the <span class="SpellE"><em>useCapture</em></span> parameter to <strong>false</strong>, the handler that belongs to the target element is called first, and any handlers that belong to parent elements are called after that.</p> 
        <p class="MsoNormal">In this example, you create a <a href="http://msdn.microsoft.com/en-us/library/windows/apps/br211681.aspx"><span class="SpellE"><strong>WinJS.UI.DatePicker</strong></span></a> and add the same change handler function to the <span class="SpellE"><span class="Code2Char"><span style="font-size: 10.0pt; mso-bidi-font-size: 11.0pt; line-height: 115%;">WinJS.UI.DatePicker</span></span></span> and the <span class="Code2Char"><span style="font-size: 10.0pt; mso-bidi-font-size: 11.0pt; line-height: 115%;">document</span></span> object. If you set the <span class="SpellE"><em>useCapture</em></span> parameter to <span class="Code2Char"><span style="font-size: 10.0pt; mso-bidi-font-size: 11.0pt; line-height: 115%;">true</span></span>, the handler on the <span class="Code2Char"><span style="font-size: 10.0pt; mso-bidi-font-size: 11.0pt; line-height: 115%;">document</span></span> object is called first, then the handler on the <span class="SpellE"><span class="Code2Char"><span style="font-size: 10.0pt; mso-bidi-font-size: 11.0pt; line-height: 115%;">WinJS.UI.DatePicker</span></span></span> object.</p> 
        <p class="MsoNormal">In an event, <span class="GramE">the <span class="Code2Char"><span style="font-size: 10.0pt; mso-bidi-font-size: 11.0pt; line-height: 115%;">this</span></span></span> keyword refers to <strong style="mso-bidi-font-weight: normal;">the target of the event</strong>, so you should see "change event for [object Document]" and then "change event for [object <span class="SpellE">HTMLDivElement</span>]". If you set the <span class="SpellE"><em>useCapture</em></span> parameter to <span class="Code2Char"><span style="font-size: 10.0pt; mso-bidi-font-size: 11.0pt; line-height: 115%;">false</span></span>, you should see the output for the <span class="SpellE"><span class="Code2Char"><span style="font-size: 10.0pt; mso-bidi-font-size: 11.0pt; line-height: 115%;">DatePicker</span></span></span> handler first, then the output for the document handler.</p>
