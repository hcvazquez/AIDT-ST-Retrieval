        <div class="slds-size--1-of-1 slds-medium-size--12-of-12 slds-large-size--10-of-12"> 
         <h1 class="h1heading postheading">Introducing The LockerService For Lightning Components</h1> 
         <div class="subheading">
          This is an introductory post about a new security feature for Lightning Components called "LockerService (LS)". LockerService uses various technologies and techniques to make Lightning Components secure. It will be available as a critical update as part of Summer '16 and will be enforced in Winter '17. 
         </div> 
         <p class="bodytext boddytext_margin"><em><strong>Note:</strong> As of February 2017, we have updated the information available regarding LockerService. You can find the updated information in <a href="https://developer.salesforce.com/blogs/developer-relations/2017/02/lockerservice-lightning-container-third-party-libraries-lightning-components.html">this blog post</a>.</em></p> 
         <p class="bodytext boddytext_margin">As you know, Trust&nbsp;is Salesforce’s #1 value and product security is a key aspect of Trust.&nbsp;We are constantly searching for ways to improve the security of our products. In that regard, we’ve been working on a set of new security features for Lightning Components called <em>LockerService</em> (LS).</p> 
         <p class="bodytext boddytext_margin">At a high level, LockerService uses various technologies and techniques that are intended to do the following:</p> 
         <p class="bodytext boddytext_margin"><b>Prevent:</b></p> 
         <ul> 
          <li>Components from causing XSS and similar security issues</li> 
          <li>Components from reading other component’s rendered data without any restrictions</li> 
          <li>Components from calling undocumented/private APIs</li> 
         </ul> 
         <p class="bodytext boddytext_margin"><b>Enable:</b></p> 
         <ul> 
          <li>Cool new features like client-side API versioning similar to REST API versioning*</li> 
          <li>Faster security review</li> 
          <li>Better and more secure JS development practices</li> 
          <li>Running 3rd party JS frameworks like React, Angular and so on*</li> 
          <li>Easily adding or removing new security features and policies</li> 
         </ul> 
         <p class="bodytext boddytext_margin"><i>*Some of the features will have limitations and will come during or after Winter 2017 #safeHarbor</i></p> 
         <h2 class="h2heading postheading">Availability</h2> 
         <p class="bodytext boddytext_margin">In Winter ’17, LockerService can be activated or deactivated in your org. Note: To ensure compatibility with LockerService before the Summer ’17 release, test your Lightning components in a sandbox or a Developer Edition org.</p> 
         <p class="bodytext boddytext_margin">In Spring ’17, the existing LockerService critical update will tighten Content Security Policy (CSP) to eliminate the possibility of cross-site scripting attacks. These CSP changes will only be enforced in sandbox and Developer Edition orgs. If you have a sandbox or Developer Edition org with the LockerService critical update activated, the stricter CSP will apply when Spring ’17 rolls out. If you deactivate the LockerService critical update before the Spring ’17 release, the stricter CSP won’t be applied.</p> 
         <p class="bodytext boddytext_margin">In Summer ’17, LockerService will be auto-activated for all orgs with no option to disable.</p> 
         <h3 class="h3heading postheading">Components Installed from Managed Packages</h3> 
         <p class="bodytext boddytext_margin">To control whether LockerService is enforced for components installed from a managed package:</p> 
         <ol> 
          <li>From Setup, enter Lightning Components in the Quick Find box, and then select Lightning Components.</li> 
          <li>Select the Enable LockerService for Managed Packages checkbox to enforce LockerService for components<br> installed from a managed package.<br> <strong>Note:</strong> The checkbox is only visible when the critical update is activated.</li> 
         </ol> 
         <p class="bodytext boddytext_margin">If you deselect the Enable LockerService for Managed Packages checkbox, LockerService is not enforced for<br> components installed from a managed package. Components that you create in your org still run with enforcement of LockerService restrictions.</p> 
         <h2 class="h2heading postheading">Under The Hood</h2> 
         <p class="bodytext boddytext_margin">To understand the technology under the hood, let’s take an example app. &nbsp;The below example Lightning app has four components: a <b>button</b>, a <b>Weather</b> component (that internally has a <b>Map</b> sub-component) and a <b>Finance</b> component.</p> 
         <p class="bodytext boddytext_margin"> <span class="postimagessection"> <img class="postimages" src="http://res.cloudinary.com/hzxejch6p/image/upload/c_scale,h_727,q_100/v1460697095/Screen_Shot_2016-04-14_at_10.11.33_PM_edohpq.png"> </span> </p> 
         <h3 class="h3heading postheading"><b>Before LockerService</b></h3> 
         <p class="bodytext boddytext_margin">Before LockerService, the DOM tree and JS access for the example lightning app would look like the following image.</p> 
         <p class="bodytext boddytext_margin"> <span class="postimagessection"> <img class="postimages" src="http://res.cloudinary.com/hzxejch6p/image/upload/q_100/v1460598564/beforeLocker_qgh3nc.png"> </span> </p> 
         <p class="bodytext boddytext_margin">In this scenario:</p> 
         <ol> 
          <li>Any JS of any component can call any JS functions of any other component as they are all loaded in the DOM</li> 
          <li>Any JS can also call any undocumented/private Lightning APIs</li> 
          <li>Any JS can access real DOM and get rendered data from other components.</li> 
          <li>Components that are not security reviewed can potentially have security issues.</li> 
         </ol> 
         <h3 class="h3heading postheading">After LockerService</h3> 
         <p class="bodytext boddytext_margin">The same lightning app with all the techniques and technologies used in LockerService will look something like the following image.</p> 
         <p class="bodytext boddytext_margin"> <span class="postimagessection"> <img class="postimages" src="http://res.cloudinary.com/hzxejch6p/image/upload/q_100/v1460598684/afterLocker_f5k28d.png"> </span> </p> 
         <h4 class="h4heading postheading">With the LockerService turned&nbsp;ON</h4> 
         <ol> 
          <li>Salesforce-authored components and JS run in “System mode” (similar to the Operating System’s system mode) and have full access to everything.</li> 
          <li>Custom components run in “User mode” and don’t have access to the real Document or real “window” object.</li> 
          <li>Custom components instead gets a custom DOM (“secureDOM”), custom Window and so on.</li> 
          <li>Custom components can access other components’ DOM as long as they are in the same namespace but won’t be able to access other components’ DOM that are in different namespace. For example, JS in the “Weather” component can access DOM of “Map” component, but won’t be able to access DOM elements of the “Finance” or “button” component.</li> 
          <li>In addition, custom components will only be able to access public and documented Lightning JS APIs and won’t be able to access Lightning framework’s “private” APIs.</li> 
          <li>“Use strict” and CSP are enabled and enforced for security of all components.</li> 
         </ol> 
         <p class="bodytext boddytext_margin">In Spring ’17, LockerService tightens CSP to eliminate the possibility of cross-site scripting attacks by disallowing the unsafe-inline and unsafe-eval keywords for inline scripts (script-src). Eliminate use of these keywords in your code, and update third-party libraries to modern versions that don’t depend on unsafe-inline or unsafe-eval. These CSP changes are only enforced in sandbox and Developer Edition orgs.</p> 
         <p class="bodytext boddytext_margin"><b>Note</b><span style="font-weight: 400;">:</span></p> 
         <ul> 
          <li style="font-weight: 400;"><span style="font-weight: 400;">When LockerService is activated in sandbox or Developer Edition orgs, you can’t edit rich-text fields and the fields are grayed out. We’re working on removing that limitation in a future release.</span></li> 
          <li style="font-weight: 400;"><span style="font-weight: 400;">IE11 doesn’t support CSP, so we recommend using other supported browsers for enhanced security.</span></li> 
         </ul> 
         <h3 class="h3heading postheading">Additional Resources</h3> 
         <ol> 
          <li><a href="https://developer.salesforce.com/docs/atlas.en-us.lightning.meta/lightning/security_code.htm?search_text=locker">Lighting Components Developer Guide</a> – More details about LockerService and how it works in Lighting components.</li> 
          <li><a href="http://salesforce.vidyard.com/watch/P6KtQSvpHAaDfuy5extjWK" target="_blank">Lightning LockerService Video</a> – Get detailed guidance from Salesforce staff.</li> 
          <li><a href="https://developer.salesforce.com/docs/atlas.en-us.lightning.meta/lightning/cli_intro.htm" target="_blank">Salesforce Lightning CLI</a> – Use the code review tool to check your code for LockerService compatibility.</li> 
          <li><a href="http://documentation.auraframework.org/lockerApiTest/index.app?aura.mode=DEV" target="_blank">LockerService API Viewer</a> – Check the DOM APIs exposed by LockerService</li> 
         </ol> 
         <h2 class="h2heading postheading">Code Example</h2> 
         <p class="bodytext boddytext_margin">In the below picture, we have a simple <tt>ui:button</tt> that uses a <tt>div</tt> (that’s in “c” namespace) for it’s label. Notice how the console.log differs for window, document and div when LockerService is turned ON. Also notice that when the custom component’s JS tries to “walk the DOM” by doing <tt>div.parentNode.innerHTML</tt> to get the button’s <tt>innerHTML</tt>, it gets “undefined” because <tt>button</tt> is in a different “ui” namespace.</p> 
         <p class="bodytext boddytext_margin"> <span class="postimagessection"> <img class="postimages" src="http://res.cloudinary.com/hzxejch6p/image/upload/q_100/v1460598775/code-sample_vnoag4.png"> </span> </p> 
         <h2 class="h2heading postheading">FAQs</h2> 
         <dl> 
          <dt>
           <strong>What will happen to the existing custom components or those in the Component exchange?</strong>
          </dt> 
          <dd>
           LockerService runs behind the scenes. There is no change to the public and documented APIs. However, LockerService enforces modern JS standards and new security rules. So depending how the component is built, they will need to be upgraded to meet these standards. We’ll be providing tools and articles to make this transition easy.
          </dd> 
         </dl> 
         <p class="bodytext boddytext_margin">&nbsp;</p> 
         <dl> 
          <dt>
           <strong>Can I use 3rd party libraries&nbsp;(React, Angular, etc) if running in Locker?</strong>
          </dt> 
          <dd>
           Yes, provided they are served from Static Resources and approved by security.
          </dd> 
         </dl> 
         <p class="bodytext boddytext_margin">&nbsp;</p> 
         <dl> 
          <dt>
           <strong>Can different components use different versions of a 3rd party library?</strong>
          </dt> 
          <dd>
           Yes, LS provides full JS&nbsp;module level isolation of window, document, etc so your locker (tied to your namespace) is its own world.
          </dd> 
         </dl> 
         <p class="bodytext boddytext_margin">&nbsp;</p> 
         <dl> 
          <dt>
           <strong>Does Locker allow me to access libraries hosted on a CDN (i.e. Google Maps)</strong>
          </dt> 
          <dd>
           It makes it possible for us to open up our Content Security Policy to allow this in Winter 2017 (safe harbor).
          </dd> 
         </dl> 
         <p class="bodytext boddytext_margin">&nbsp;</p> 
         <dl> 
          <dt>
           <strong>Does Locker use an iFrame for sandboxing</strong>
          </dt> 
          <dd>
           No – your code runs directly in the same browser window as the rest of the components in an application.
          </dd> 
         </dl> 
         <p class="bodytext boddytext_margin">&nbsp;</p> 
         <dl> 
          <dt>
           <strong>Are application events sandboxed in a locker?</strong>
          </dt> 
          <dd>
           No, but all events (Aura and DOM) are wrapped in a filtering proxy that only allows your code to see what it has access to. You can continue to use Lightning eventing framework to communicate between components irrespective of their namespace.
          </dd> 
         </dl> 
         <p class="bodytext boddytext_margin">&nbsp;</p> 
         <dl> 
          <dt>
           <strong>Is there a way to access raw DOM elements?</strong>
          </dt> 
          <dd>
           No – you always work with the Secure Virtual DOM – but your code and 3rd party code cannot tell the difference – no change in programming model (e.g. Component.getElement() works the way it does today). You cannot do things we’ve said not to do from day one now – e.g. node.parentNode.parentNode to escape into code you do not own is blocked.
          </dd> 
         </dl> 
         <p class="bodytext boddytext_margin">&nbsp;</p> 
         <dl> 
          <dt>
           <strong>What’s the granularity of a locker? One component per locker? Multiple Components per locker?</strong>
          </dt> 
          <dd>
           Today the granularity is all the components in the same namespace. Future versions of Salesforce are likely to allow both finer grain (e.g. set of components) and courser grain (e.g. set of namespaces owned by an org).
          </dd> 
         </dl> 
         <p class="bodytext boddytext_margin">&nbsp;</p> 
         <p class="bodytext boddytext_margin">Please note that this is just an introductory post. We’ll be releasing tools, blogs, in-depth articles and Trailhead modules in the coming weeks and months before LockerService is made available. Please watch this space for more information.</p>
