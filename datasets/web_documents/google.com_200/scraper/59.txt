          <h1 class="entry-title">Six essential JavaScript libraries</h1> 
         </header> 
         <!-- end .entry-header --> 
         <div class="entry-meta"> 
          <span class="post-category"> posted in <a href="http://jamesknelson.com/category/javascript/" rel="category tag">Javascript</a> </span> 
          <!-- end .post-category --> 
          <span class="post-date"> on <a href="http://jamesknelson.com/six-essential-javascript-libraries/" title="1:13 pm" rel="bookmark"><time class="entry-date" datetime="2012-11-09T23:15:57+00:00">June 5, 2015</time></a> </span> 
          <!-- end .post-date --> 
          <span class="by-author"> by <span class="author vcard"> <a class="url fn n" href="http://jamesknelson.com/author/jamesknelson/" title="View all posts by James K Nelson" rel="author">James K Nelson</a> </span>

         <div class="entry-content clearfix"> 
          <p>Have you ever found yourself searching through the 153,882 packages on npm (as of the time of writing), sure that there must be <em>something</em> which solves your problem, but bewildered by your inability to find it? You’re not alone!</p> 
          <p>The abundance of packages in the JavaScript ecosystem can at times feel more like a curse than a blessing – all the code in the world can’t help you if you don’t know where to find it. However, the great thing about well designed libraries is that you only need a few of them to get you a long way.</p> 
          <p>With that in mind, here are six of the libraries I use to solve problems in my web and node projects. In alphabetical order:</p> 
          <p><span id="more-225"></span></p> 
          <h2>1. <a href="https://github.com/mzabriskie/axios">axios</a></h2> 
          <p>XMLHttpRequest is one of those technologies that I find myself both loving and hating at the same time. Without it, the modern web wouldn’t exist, but the times I’ve been forced to actually use it have made me want to cry.</p> 
          <p>I’m obviously not the only one – there are a plethora of libraries built to try and make XMLHttpRequest easier. <a href="https://visionmedia.github.io/superagent/">Superagent</a> and <a href="https://github.com/request/request">request</a> are two popular ones which lack Promise support, while jQuery’s <code>jQuery.ajax</code> and Angular’s $http service comes close, but with the problem that they’re both only available when you include their monstrous associated frameworks.</p> 
          <p>Enter axios – the XMLHttpRequest library which takes the good parts from Angular’s $http, and throws out everything else. Here is an example of it’s use:</p> 
          <h3>With axios</h3> 
          <pre><code>axios({
    url: 'http://readyourlevel.jamesknelson.com/api/grade',
    method: 'post',
    headers: {'Content-type': 'text/html; charset=UTF-8'},
    data: text
}).then(onGradeResult, onGradeError)
</code></pre> 
          <h3>Without axios</h3> 
          <pre><code>let http = new XMLHttpRequest();
http.open("POST", 'http://readyourlevel.jamesknelson.com/api/grade', true);
http.setRequestHeader('Content-type', 'text/html; charset=UTF-8');
http.onreadystatechange = function() {
    if (http.readyState == 4) {
        if (http.status == 200)
            onGradeResult(JSON.parse(http.responseText));
        else
            onGradeError(http);
    }
};
http.onerror = onGradeError;
http.send(text);
</code></pre> 
          <h2>2. <a href="https://github.com/es-shims/es5-shim">es5-shim</a> and <a href="https://github.com/paulmillr/es6-shim/">es6-shim</a></h2> 
          <p>I’ve lumped these two together, as they do basically the same thing. That is, they make all sorts of ECMAscripty goodness available to you even while hobbled by old-school browsers (without the overhead of an entire build process). No JavaScript developer in their right mind <em>doesn’t</em> want ECMAscripty goodness.</p> 
          <p>Of course, no amount of shimming is going to bring you to the promised land of arrow functions, destructuring and generators – but Set, Map, WeakSet, Promises, etc. are certainly better than nothing.</p> 
          <p>That said, there is one function from es6-shim which makes so many tasks so much easier that it justifies the shim’s use all by itself. Here’s an example from my project <a href="https://github.com/jamesknelson/numbat-ui">numbat-ui</a>.</p> 
          <h3>With es6-shim</h3> 
          <pre><code>var targetProps = Object.assign({}, this.managedCallbacks, other, {disabled: disabled});
</code></pre> 
          <h3>Without es6-shim</h3> 
          <pre><code>var targetProps = {};
for (key in this.managedCallbacks) {
    if (this.managedCallbacks.hasOwnProperty(key)) {
        targetProps[key] = this.managedCallbacks[key];
    }
}
for (key in other) {
    if (other.hasOwnProperty(key)) {
        targetProps[key] = other[key];
    }
}
targetProps.disabled = disabled;
</code></pre> 
          <p>Check out <a href="https://twitter.com/kangax">@kangax</a>‘s amazing <a href="http://kangax.github.io/compat-table/es5/">ES5</a> and <a href="http://kangax.github.io/compat-table/es6/">ES6</a> compatibility tables for details on what es5-shim and es6-shim support, or if you really want everything ES6 has to offer, consider investing in learning <a href="https://babeljs.io/">Babel</a>.</p> 
          <h2>3. <a href="http://highlandjs.org/">Highland</a></h2> 
          <p>ES6 Promises are wonderful and all, but <a href="http://jamesknelson.com/grokking-es6-promises-the-four-functions-you-need-to-avoid-callback-hell/">as awesome as <code>Promise.all</code> is</a>, there are still times when the complexity of your asynchronous code makes you want to pull your hair out.</p> 
          <p>While use of <a href="https://github.com/petkaantonov/bluebird">bluebird</a> and co. can help, they still offer a fundamentally <em>promise-focussed</em> approach, where sometimes what is needed is a completely different abstraction.</p> 
          <p>One example from my recent work was a node.js web scraper for <a href="http://www.aozora.gr.jp/">Aozora Bunko</a>. While the HTTP client I used rightly returns a promise from each HTTP request, looking at the bigger picture shows a flow of data from the original request, through UTF-8 and HTML parsers, through filters, into more HTTP requests, and yet more parsers. What I really needed was to treat my data as a <em>stream</em>.</p> 
          <p>Highland makes working with streams easy, and cleverly works <em>with</em> your promises instead of asking you to throw away your existing friends and only play with <em>it</em>. Here is some relevant code, with highland manifesting itself as <code>_</code>:</p> 
          <h3>With Highland</h3> 
          <pre><code>let pipeline = _.pipeline(
    _.map((data) =&gt; {
        // Make Japanese keys from CSV to English to fit RYG's database
        let result = {};
        KEYS.forEach(([key, newKey], i) =&gt; {
            if (data[key]) result[newKey] = data[key];
        });
        result
    }),
    _.filter((data) =&gt; data.characterType == '?????'),
    _.ratelimit(2, 100),
    _.flatMap((data) =&gt; _(axios({url: data.url, headers: HEADERS}).then(getBookContentHtml)))
);
</code></pre> 
          <h3>Without Highland</h3> 
          <p>I’ll leave this as an exercise for the serious reader. Leave a comment if you manage to get it.</p> 
          <h2>4. <a href="https://lodash.com/">lodash</a></h2> 
          <p>For people who’ve come to JavaScript after some experience with batteries-included languages like Ruby or Python, it is almost inevitable that at some point you’ll have the “there is <em>no way</em> that this function isn’t built in” moment. Or if you’re like me, you’ll never stop having them. E.g.</p> 
          <ul> 
           <li>I wonder what the method to get the last element of an array is? <em>What, it doesn’t exist?!</em></li> 
           <li>Seems I’ve forgotten the method to get the sum of an array’s elements. <em>Nope. I never learned it in the first place because it doesn’t exist.</em></li> 
           <li>Surely javascript has a function to do a deep copy of an object? <em>Nope?! What the fuck were the Netscape gods smoking?!</em></li> 
          </ul> 
          <p>Lodash fixes this – it is JavaScript’s batteries, so to speak. It provides a huge number of helpful functions and a pleasant-to-use interface.</p> 
          <p>Here is an example from the Japanese Grader used at <a href="http://www.readyourgrade.com">Read Your Grade</a> of why you should use lodash:</p> 
          <h3>With lodash</h3> 
          <pre><code>let furiganaSummary = _.chain(kanjiTokens)
  .groupBy('key')
  .mapValues((tokens) =&gt; _.countBy(tokens.map((t) =&gt; t.furiganaType)))
  .value();
</code></pre> 
          <h3>Without lodash</h3> 
          <pre><code>let furiganaSummary = {};
for (let kanjiToken of kanjiTokens) {
    let key = kanjiToken.key;
    if (furiganaSummary[key])
        furiganaSummary[key].push(kanjiToken);
    else
        furiganaSummary[key] = [kanjiToken];
}
for (let key in furiganaSummary) {
    let typeCounts = {};
    for (let kanjiToken of furiganaSummary[key]) {
        let furiganaType = kanjiToken.furiganaType;
        if (typeCounts[furiganaType] === undefined)
            typeCounts[furiganaType] = 1;
        else
            typeCounts[furiganaType] += 1;
    }
    furiganaSummary[key] = typeCounts;
}
</code></pre> 
          <h2>5. <a href="http://momentjs.com/">Moment.js</a></h2> 
          <p>Do you find yourself looking up JavaScript’s date-related functions every time you need to touch one? Maybe you long for the simplicity of Rails’ ActiveSupport where <code>1.day_ago</code> actually is a Date object. Or worse, maybe you just avoid working with dates at all. After all, surely the user can’t find “Fri Jun 05 2015 13:27:39 GMT+0900 (JST)” that much harder to read than “1 minute ago”, right?</p> 
          <p>Moment.js fixes this by providing an incredibly simple interface and a number of powerful functions for working with dates, times and time periods. Here is an example from some accounting software I manage:</p> 
          <h3>With moment.js</h3> 
          <pre><code>let dateString = moment().subtract('month', 1).endOf('month').format("YYYY-MM-DD");
</code></pre> 
          <h3>Without moment.js</h3> 
          <pre><code>let now = new Date();
let month = ((now.getMonth() + 11) % 12) + 1;
let year = month == 12 ? now.getFullYear() - 1 : now.getFullYear();
let leapYear = thisYear % 4 == 0;
let days = [31, leapYear ? 29 : 28, 31, 30, 31, 30, 31, 31, 30, 31, 30, 31];
let paddedMonth = month &lt; 10 ? `0${month}` else month
let parts = `${year}-${paddedMonth}-${days[month - 1]}`;
</code></pre> 
          <h2>6. <a href="http://numeraljs.com/">Numeral.js</a></h2> 
          <p>Almost every project needs to display formatted numbers. Examples include:</p> 
          <ul> 
           <li>formatting currency with the correct number of decimal places</li> 
           <li>truncating long floating point numbers to one or two numbers after the decimal</li> 
           <li>padding numbers in dates (like in the above example)</li> 
          </ul> 
          <p>In most languages, you’d have an inbuilt function to help with this – but not in javascript (notice the theme yet?)! So how do you fix it?</p> 
          <p>Well, you <em>can</em> just format each number manually, but this gets tiring fast. An alternative common solution is to add your own formatting-related helper functions, but this still gets confusing when they start to change between projects, and why spend the time writing your own if you don’t <em>have</em> to?</p> 
          <p>And you don’t! Numeral.js is a simple library which makes this a breeze, and allows you to learn a single API which you can then transfer to any project which your boss/customer/curiosity presents you with. While it may not seem to be that much effort to do this manually in the short term, the benefits from Numeral.js compound over time. Heres an example from my <a href="http://www.readyourgrade.com">Read Your Grade</a> project:</p> 
          <h3>With numeral.js</h3> 
          <pre><code>let formattedNumber = numeral(bookScore).format('0.0');
</code></pre> 
          <h3>Without numeral.js</h3> 
          <pre><code>let formattedNumber = `${parseInt(bookScope)}.${((bookScore % 1) + '')[2]}`;
</code></pre> 
          <p>Sure, they’re both one line – but which one would you rather read/type?</p> 
          <h2>Honorary Mention: <a href="https://github.com/pozadi/kefir">Kefir</a></h2> 
          <p>Kefir is a Functional Reactive Programming tool which can make wiring up user interfaces <em>fun</em> again. I’ve been using it together with Facebook’s <a href="http://facebook.github.io/react/">React</a> while building <a href="https://github.com/jamesknelson/numbat-ui">numbat-ui</a> and thoroughly enjoying it. It’s source is lightweight and easy to read, and the API feels well-designed.</p> 
          <p>So why only an honorary mention? Well, while searching for awesome examples of my usage of Kefir, I couldn’t find any single blocks of code which screamed “wow, this shows just how must time Kefir will save you”. That said, I’m still new to FRP, so this doesn’t mean they’ll never happen. Indeed, my gut feeling is that building UIs with Kefir/React uses such a different thought process to jQuery or Angular that the code which would normally feel painful just doesn’t materialise in the first place.</p> 
          <p>Kefir/React is something I’d like to write on again in the future once I have more experience (and code) with it. Subscribe to my newsletter to make sure you don’t miss out when I do!</p>
