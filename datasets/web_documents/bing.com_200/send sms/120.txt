           All:
           <br> I'm researching the use of Push technology in MIDP 2.0 on the iDEN network to send notifications to our DispatchSuite application that new dispatch data is ready on the server. I am trying *real hard* to avoid reqiring static IPs for our application to work, so I want to use the SMS protocol for the push registration.
           <br> The payload of SMS message we need to send is really a non-issue. All we want to do tell the handset to wake up and use it normal http GET mechanisms to get any waiting messages from our server. So there may be a control bit or two, but size is of no consequence.
           <br> The question is: what is required to send an SMS message over the iDEN nextwork to our J2ME application? 
           <br> Do email messages automatically get transfered into SMS messages. So, would an email message to 
           <a class="snap_shots" href="mailto:8051234567@messaging.nextel.com">8051234567@messaging.nextel.com</a> send using the JavaMail API translate to an SMS message when it gets to the phone? And what is the port that I 
           <a href="http://www.permies.com/t/36936/tnk/" class="api" title="Did you just should on me? A silly thread about the word 'should'" target="_new">should</a> register from the phone?
           <br> Or do I need to use an "SMS broker" such as Simplewire and use the Simplewire SDK.
           <br> Thanks in advance for your knowledge,
           <br> Nancy Lehrer

          <div class="postText ">
           Email probably would not work since the message is delivered to the native Inbox not the AMS -- there is no port number associated with the message. That also rules out toolkits like jSMS.
           <br> Simplewire might work but I have not looked at their APIs for quite a while.
           <br> I think the most promising solution is 
           <a href="http://www.jcp.org/en/jsr/detail?id=212" target="_blank">JSR 212</a>, which defines the server side messaging API. Since I am in the expert group, I can tell you that SMS port is quite likely to be supported. Before the JSR 212 comes out, a similar product I would recommend is the 
           <a href="http://www.forum.nokia.com/nokia_mobile_server" target="_blank" rel="nofollow">Nokia Mobile Server</a>. It goes far beyond SMS messaging. Check it out!The downside of this approach is that the carrier must open their MMSC to you. That could be a real bummer since telecomms are not exactly developer friendly. But you can alsways try.
           <br> Another solution is to borrow the idea of jSMS: For small volmue applications, you can use a phone running WMA as the "gateway" device. It is placed at a always-on location and receives data from a network server. Then it pushes SMS messages (with port numbers) to wake up other handsets in the field. Just some ideas here. 
           <img src="https://www.coderanch.com/images/smilies/3b63d1616c5dfcf29f8a7a031aaa7cad.gif"> Let us know what works for you!

           All:
           <br> I have determined that you need to use an SMS broker such as SimpleWire in order to send an SMS message from a server application to the phone. 
           <br> Some folks set up their own "SMS server" by thethering a phone to their computer and using AT commands...
           <br> The cost to run an SMS message through a broker service varies upon carrier. The carriers charge the broker for sending the message, and the phone user gets charged for receiving the message.
           <br> From SimpleWire, Nextel is the least expensive at around 1.5 cents per message and ATT is the most expensive at 8.5 cents per message.
           <br> I think I will use sockets to notify my devices :0

           Roberto, 
           <br> Here is a little more info I got from SimpleWire after looking through the API and docs for some mention of the port information. Oh and along my journey I found that this bit of information we have all been in search of is the UDH. (UserDefinedHeader) and looks something like
           <br> udh=060504xxxxyyyy where xxxx is the receive port no the phone and yyyy is the outbound port from the phone. (In most cases for pushregistry you only care about xxxx)
           <br> Of course SimpleWires API makes it all a lot easier than encoding this as an SMPP message. Here is those notes that I received earlier today.
           <br> ---------------- message text from email ------------
           <br> The API documentation does not yet have those features explained, but the software does have the features. Make sure you've followed these steps: 
           <br> 1. Verify you have the latest Java SMS SDK from Simplewire. Only the latest version has the features you need. You can download it at 
           <a href="http://www.simplewire.com/developers/sdk/java/" target="_blank" rel="nofollow">http://www.simplewire.com/developers/sdk/java/</a> 
           <br> 2. Navigate to the "Examples" directory included with the software download. Open up the "send_text.java" example code. There are 3 commented out lines. You would need to uncomment those lines to enable sending a text message to a specific port on the phone. 
           <br> Text messages are not sent to a specific port by default. Applications on the phone bind to a specific port, and if no application is bound, then the default handler is the "Text Message Inbox". 
           <br> You won't be able to test this feature without a commercial or demo account from Simplewire. Please follow the directions on our site about how to activate demo credits. You can contact Simplewire sales for information about getting a commercial account. You can contact Simplewire sales at: 
           <br> 
           <a href="http://www.simplewire.com/contact/sales/" target="_blank" rel="nofollow">http://www.simplewire.com/contact/sales/</a> 
           <br> ----------------- end of message ----------
           <br> The lines that are mentioned here are these.....I think.
           <br> //sms.setSourcePort((short)0x0000);
           <br> //sms.setDestPort((short)0x0000);
           <br> --&gt; comments are here with some basic info.
           <br> // sms.setNetworkType(SMS.NETWORK_TYPE_GSM);
           <br> So you uncomment those and tada you should have it working. I'll be
           <br> testing this very soon and will provide info on results.
           <br> Best Wishes
           <br> -Shawn

           Hey Mark here's that update.
           <br> 
           <br> Well as with most things the more effort you put into building it yourself the cheaper 
           <br> you can get it. So having said that here is what I have found.
           <br> 
           <br> SimpleWire - (US) about the easiest to implement, good API, (needs a little work on the docs, but who am I to talk ;-) 
           <br> 
           <br> PhoneCaster - (US) No port assignment support but AWSOME SMS service. 
           <br> 
           <br> SMS2Email - (UK) HTTP based service with port assignment
           <br> 
           <br> RedCoal.com - (UK) HTTP based service port assignement support unknown
           <br> 
           <br> ClickaTel - (S.Africa) Just about every Protocol you can image. Don't know how their docs of the API are but saw some examples. And has Port Assignment.
           <br> 
           <br> SimpleWire appears to be one of the only service providers with a Java API, the rest are a roll your own.
           <br> 
           <br> Depending on your market may influence your decision as well as your budget.
           <br> SimpleWire - simple but pricey...good service. Fast SMS delivery during peak hours (about 10sec), but is based in US market.
           <br> PhoneCaster - also a US based provider and similar response time to simple wire.
           <br> Other provider are not US based, and times can drag up to 20sec for SMS delivery in Peak Hours.
           <br> But read carefully about service areas. It costs more for UK providers to send SMS into the US market. 
           <br> 
           <br> Those are my rough notes. I'm putting some more stuff together but it may be a while before it's
           <br> ready.

           </blockquote>If you are sending messages through a web server, you don't need anything written in J2ME. Most carriers maintain a website for sending messages to their subscribers. For example, this is Sprint's: 
           <a href="http://messaging.sprintpcs.com" target="_blank" rel="nofollow">http://messaging.sprintpcs.com</a>. This is Cingular's: 
           <a href="http://www.cingularme.com/do/public" target="_blank" rel="nofollow">http://www.cingularme.com/do/public</a>. Here is Verizon's: 
           <a href="http://www.vtext.com" target="_blank" rel="nofollow">http://www.vtext.com</a>.
           <br> 
           <br> Another alternative is to use an email-to-SMS gateway. For example, send a message to 
           <a class="snap_shots" href="mailto:8005551212@messaging.sprintpcs.com">8005551212@messaging.sprintpcs.com</a> and Sprint will forward the message as an SMS to 800-555-1212.
           <br> 
           <br> Finally, you could use an email-to-SMS aggregator service like Teleflip. Just send email to 
           <a class="snap_shots" href="mailto:8005551212@teleflip.com">8005551212@teleflip.com</a> and they will forward it as a text message to the appropriate carrier.
