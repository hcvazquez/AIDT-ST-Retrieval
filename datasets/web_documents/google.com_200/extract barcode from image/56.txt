        <div class="entry-content"> 
         <p>There are a number of options if you are looking to implement barcode scanning in your HTML5/WebWorks applications.</p> 
         <p>The original <a href="https://github.com/blackberry/WebWorks-Community-APIs/tree/master/BB10-Cordova/BarcodeScanner">community extension</a> available in GitHub leverages native BlackBerry functionality to scan barcodes. However, it has two caveats:</p> 
         <ul> 
          <li>Data transfer relies on saving/loading images from the filesystem, resulting in a viewfinder (renders camera stream) refresh rate around 2-3FPS.</li> 
          <li>Each render creates a new image from the filesystem, resulting in memory issues when the viewfinder displays for an extended period of time.</li> 
         </ul> 
         <p>Our WebWorks team will soon be integrating BlackBerry 10 support into a cross-platform PhoneGap plugin for barcode scanning, based on the existing community extension. For short-burst scanning, the existing community extension can fill that role.</p> 
         <p>Recently though, I was involved in a project that required extended barcode scanning functionality and, due to the caveats with the original community extension, I opted to explore a purely HTML5 solution to the issue. I want to share my experiences in implementing a solution that would satisfy the following requirements:</p> 
         <ul> 
          <li><strong>Viewfinder: </strong>We need to display <em>what</em> is being scanned.</li> 
          <li><strong>Performance:</strong> Barcode processing should have minimal impact on viewfinder rendering.</li> 
          <li><strong>Continuous Scanning: </strong>The approach should be capable of scanning multiple items in a row without requiring any sort of reset, but also allow for one-offs.</li> 
         </ul>

         <p>There are only a handful of freely distributed barcode scanning libraries out there. After some initial evaluation, I went with <a href="https://github.com/EddieLa/BarcodeReader"><strong>Barcode Reader</strong> by Eddie Larsson</a> for 1D processing and <a href="https://github.com/LazarSoft/jsqrcode/tree/master/src"><strong>jsqrcode</strong> by Lazar Laszlo</a> for 2D processing, which seemed to be the most popular.</p> 
         <p>For this example, we will focus on a <strong>webkit</strong> implementation of <a href="http://dev.w3.org/2011/webrtc/editor/getusermedia.html">getUserMedia</a>. However it’s important to note the following considerations:</p> 
         <ul> 
          <li><strong>iOS</strong> does not currently support getUserMedia</li> 
          <li><strong>Android</strong> supports getUserMedia as of <strong>4.4 </strong>via <strong>Chrome for Android</strong></li> 
         </ul> 
         <p><em>Source: </em><a href="http://caniuse.com/stream"><em>http://caniuse.com/stream</em></a></p> 
         <p>To also target iOS and earlier versions of Android, a Cordova/PhoneGap plugin would be the recommended approach.</p>

         <h2>Performance</h2> 
         <p>Even with optimization, we in the HTML5 world will not enjoy the same 60FPS viewfinder that you get with a native implementation, at least with how things are today. What we can expect is around 15-20FPS. Although this isn’t ideal, it’s good enough for the task at hand.</p> 
         <h2>Performance – Rescaling</h2> 
         <p>We could leverage the data from our&lt;video&gt; element as-is and pass that for barcode processing. However, passing 480×640 pixels (or more) produced long processing time, particularly for the 1D barcode analyzer. I found better performance, with minimal impact to usability, by rescaling the data to 256×256 which is roughly 4.7x fewer pixels to analyze.</p> 
         <p>During re-scaling, it may also be beneficial to maintain the same aspect ratio on the&lt;canvas&gt;as in the originalvideo stream. However, I have not experienced any usability impact so far.</p>

         <h2>Performance – Web Workers</h2> 
         <p>There are a handful of additional issues we need to address.</p> 
         <p>If you look at the chosen barcode libraries you’ll notice that <strong>Barcode Reader</strong> has already been written to function as a Web Worker. This is <strong>critical</strong> because it allows the processing to occur off the main UI thread which would otherwise block operations and make our stream rendering quite choppy.</p> 
         <p>Unfortunately, <strong>jsqrcode</strong> is not designed for this out-of-the-box.</p> 
         <p>The second issue we need to be mindful of is the timing of processing. If both 1D and 2D libraries are constantly processing, our main UI thread can get bombarded by activity, impacting rendering performance. Our goals then are:</p> 
         <ol> 
          <li><strong>Both</strong> libraries should function as Web Workers.</li> 
          <li>We must maintain a rapid rate of processing, but not overwhelm the main UI thread with messages.</li> 
         </ol> 
         <p>With this in mind, the workers were configured as follows.</p> 
         <pre>/* These are made available in the global scope of the application. */</pre> 
         <pre><strong><em>var</em></strong> workers<strong>;</strong></pre> 
         <pre></pre> 
         <pre>// Previously Defined:</pre> 
         <pre>// video, context, vx, vy, vw, vh, cx, cy, cw, ch</pre> 
         <pre></pre> 
         <pre><strong><em>function</em></strong> initWorkers<strong>()</strong> <strong>{</strong></pre> 
         <pre>&nbsp;&nbsp;&nbsp; workers <strong>=</strong> <strong>{</strong></pre> 
         <pre>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; '1D' <strong>:</strong> <strong><em>new</em></strong> Worker<strong>(</strong>'js/BarcodeReader/DecoderWorker.js'<strong>),</strong></pre> 
         <pre>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; '2D' <strong>:</strong> <strong><em>new</em></strong> Worker<strong>(</strong>'js/worker2d.js'<strong>)</strong></pre> 
         <pre>&nbsp;&nbsp;&nbsp; <strong>};</strong></pre> 
         <pre></pre> 
         <pre>&nbsp;&nbsp;&nbsp; workers<strong>[</strong>'1D'<strong>].</strong>onmessage <strong>=</strong> <strong><em>function</em></strong> <strong>(</strong>message<strong>)</strong> <strong>{</strong></pre> 
         <pre>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; <strong><em>if</em></strong> <strong>(</strong>message<strong>.</strong>data<strong>.</strong>success <strong>===</strong> <strong><em>true</em></strong><strong>)</strong> <strong>{</strong></pre> 
         <pre>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; /* Take action here. */</pre> 
         <pre>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; <strong>}</strong></pre> 
         <pre></pre>

         <pre>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; workers<strong>[</strong>'1D'<strong>].</strong>postMessage<strong>(</strong></pre> 
         <pre>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; <strong>{</strong></pre> 
         <pre>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; 'pixels'<strong>:</strong> context<strong>.</strong>getImageData<strong>(</strong>0<strong>,</strong>0<strong>,</strong> cw<strong>,</strong> ch<strong>).</strong>data<strong>,</strong></pre> 
         <pre>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; 'width'<strong>:</strong> cw<strong>,</strong></pre> 
         <pre>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; 'height'<strong>:</strong> ch<strong>,</strong></pre> 
         <pre>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; 'cmd'<strong>:</strong>'normal'</pre> 
         <pre>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; <strong>}</strong></pre> 
         <pre>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; <strong>);</strong></pre> 
         <pre>&nbsp;&nbsp;&nbsp; <strong>};</strong></pre> 
         <pre><strong>}</strong></pre> 
         <p>First, we define a workers object with <strong>1D </strong>and <strong>2D</strong> Workers. The <strong>1D</strong> Worker leverages DecoderWorker.js, which is provided with the Barcode Reader library. The <strong>2D</strong> worker leverages a <strong>custom JavaScript </strong>file on which we’ll go into detail shortly.</p> 
         <p>Next, we configure the onmessage functions of each worker very similarly.</p> 
         <ul> 
          <li>When a message is received, check whether a barcode was successfully decoded. If so, we can take specific actions (i.e. display the barcode value, etc.).</li> 
          <li>Following that, we make a call tocontext<strong>.</strong>drawImagewhich copies our &nbsp;&lt;video&gt; stream data to our&lt;canvas&gt; element.</li> 
          <li>Finally, we call the <strong>alternate</strong> worker to process the new data.</li> 
         </ul> 
         <p>In this manner, we are flipping between the <strong>1D</strong> and <strong>2D</strong> workers. This prevents us from relying on methods such assetTimeoutto maintain processing and avoids the case where processing is happening to slowly (or too quickly) based on an arbitrary time delay.</p> 
         <p>With this approach, we simply process as soon as the previous processing is complete. By alternating between <strong>1D </strong>and <strong>2D</strong>, we’re also ensuring that the main UI thread isn’t getting constantly bombarded by the quicker library.</p> 
         <p>By leveraging workers in this way (and scaling the image through the canvas) I was able to eliminate almost all rendering lag in the&lt;video&gt;.</p> 
         <p>The piece that we haven’t yet discussed is how to convert the <strong>2D</strong> library into something that is worker-friendly. It actually ends up being quite simple and this is what my <strong>worker2d.js</strong> file looks like:</p> 
         <pre>/*global self, importScripts, qrcode */</pre> 
         <pre></pre>

         <pre>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; <strong>}</strong> <strong><em>catch</em></strong> <strong>(</strong>e<strong>)</strong> <strong>{</strong></pre> 
         <pre>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; success <strong>=</strong> <strong><em>false</em></strong><strong>;</strong></pre> 
         <pre>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; result <strong>=</strong> e<strong>.</strong>toString<strong>();</strong></pre> 
         <pre>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; <strong>}</strong></pre> 
         <pre></pre> 
         <pre>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; self<strong>.</strong>postMessage<strong>({</strong></pre> 
         <pre>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; 'result'<strong>:</strong> result<strong>,</strong></pre> 
         <pre>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; 'success'<strong>:</strong> success</pre>

         <p>As you can see, the <strong>jsqrcode</strong> library only requires four (4) things:</p> 
         <ol> 
          <li>The imagedata, passed in from our main UI thread.</li> 
          <li>Thewidth of theimagedata, matches our canvas dimensions.</li> 
          <li>Theheight of theimagedata, matches our canvas dimensions.</li> 
          <li>A call toqrcode<strong>.</strong>process.</li> 
         </ol> 
         <p>With this, the library will be able to process off the main UI thread and we can simply return the success/failure of the decoding attempt.</p> 
         <h2>Continuous Scanning</h2> 
         <p>In the previous section, I discussed how I leveraged the <strong>1D </strong>and <strong>2D </strong>workers to alternate. With that setup, we already have <em>continuous</em> scanning since we were alternating workers on both success and failures.</p> 
         <p>What I didn’t show yet was how to actually <strong>start</strong> that scanning cycle.</p> 
         <pre>/* These are made available in the global scope of the application. */</pre> 
         <pre><strong><em>var</em></strong> scanningTimeout<strong>;</strong></pre> 
         <pre></pre> 
         <pre>// Previously Defined:</pre> 
         <pre>// video, context, workers, vx, vy, vw, vh, cx, cy, cw, ch</pre> 
         <pre></pre> 
         <pre><strong><em>function</em></strong> startScanning<strong>(</strong>timeout<strong>)</strong> <strong>{</strong></pre> 
         <pre>&nbsp;&nbsp;&nbsp; scanningTimeout <strong>=</strong> window<strong>.</strong>setTimeout<strong>(</strong><strong><em>function</em></strong> <strong>()</strong> <strong>{</strong></pre> 
         <pre>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; context<strong>.</strong>drawImage<strong>(</strong>video<strong>,</strong> vx<strong>,</strong> vy<strong>,</strong> vw<strong>,</strong> vh<strong>,</strong> cx<strong>,</strong> cy<strong>,</strong> cw<strong>,</strong> ch<strong>);</strong></pre> 
         <pre>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; workers<strong>[</strong>'2D'<strong>].</strong>postMessage<strong>(</strong>context<strong>.</strong>getImageData<strong>(</strong>0<strong>,</strong>0<strong>,</strong> cw<strong>,</strong> ch<strong>));</strong></pre> 
         <pre>&nbsp;&nbsp;&nbsp; <strong>},</strong> timeout <strong>||</strong> 0<strong>);</strong></pre> 
         <pre><strong>}</strong></pre> 
         <p>We can callstartScanningwithin our application to kick off the scanning cycle. I chose to start with the <strong>2D </strong>worker as it appeared to be returning more quickly than the <strong>1D</strong> worker, but in reality the choice is arbitrary.</p> 
         <p>There’s also the ability to define atimeoutvalue which will delay the start of the scan. I found this particularly useful following a <strong>successful</strong> decode. Chances are the person holding the device isn’t immediately scanning another item so I used this to introduce a <strong>two second</strong> delay between scans. The scanningTimeout variable allows us to <strong>cancel</strong> the start should the user do something in those two seconds that would require scanning to stop, like navigate to another page.</p> 
         <p>And finally, we have the same process inside thesetTimeout, where we are rendering the currentvideo data to the canvas and then allowing our <strong>2D</strong> worker to process that data, thus beginning the scanning cycle.</p>
