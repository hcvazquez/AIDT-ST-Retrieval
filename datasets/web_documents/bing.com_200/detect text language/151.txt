             <h3 class="post-title entry-title"> <a href="http://illegalargumentexception.blogspot.com.ar/2009/05/java-rough-guide-to-character-encoding.html">Java: a rough guide to character encoding</a> </h3> 
             <div class="post-header-line-1"></div> 
             <div class="post-body entry-content"> 
              <p>It can be tricky figuring out the difference between character handling code that works and code that just <i>appears to work</i> because testing did not encounter cases that exposed bugs. This is a post about some of the pitfalls of character handling in Java.</p> 
              <a name="more"></a> 
              <p>Topics:</p> 
              <ul> 
               <li><a href="#javaencoding_sourcefiles">Unicode in source files</a></li> 
               <li><a href="#javaencoding_javadatatypes">Unicode and Java data types</a></li> 
               <li><a href="#javaencoding_strnlen">How long is a (piece of) String?</a></li> 
               <li><a href="#javaencoding_encodings">Encodings</a></li> 
               <li><a href="#javaencoding_streams">Stream encoding</a></li> 
               <li><a href="#javaencoding_lossyconversions">Lossy conversions</a></li> 
               <li><a href="#javaencoding_boms">Unicode byte order marks</a></li> 
               <li><a href="#javaencoding_autodetect">Automatic detection of encoding</a></li> 
               <li><a href="#javaencoding_stringclass">The potential pitfalls of encoding and decoding using the String class</a></li> 
               <li><a href="#javaencoding_notes">Notes</a></li> 
              </ul> 
              <p><em>I wrote a <a href="http://illegalargumentexception.blogspot.com/2009/04/i18n-unicode-at-windows-command-prompt.html#charsets_unicode">little bit</a> about <a href="http://www.unicode.org/standard/WhatIsUnicode.html">Unicode</a> before.</em></p> 
              <p><em>This post might be exhausting, but it isn't <span style="font-style: normal">exhaustive</span>.</em></p> 
              <a name="javaencoding_sourcefiles"></a> 
              <h4>Unicode in source files</h4> 
              <p>Java source files include support for Unicode. There are two common mechanisms for writing code that includes a range of Unicode characters.</p> 
              <p>One choice is to encode the source files as Unicode, write the characters and inform the compiler at compile time. <code>javac</code> provides the <code>-encoding &lt;encoding&gt;</code> option for this. The downside is that anyone who uses the source file needs to be aware of this. Otherwise, attempting to compile the file on a system that uses a different default encoding will produce unpredictable results. The bad thing is that it might <i>appear</i> to work.</p> 
              <p>Code saved as UTF-8, as might be written on an Ubuntu machine:</p> 
              <pre><font color="#000000"><font color="#660066"><strong>public</strong></font> <font color="#990099"><strong>class</strong></font> PrintCopyright <font color="#000000">{</font>
  <font color="#660066"><strong>public</strong></font> <font color="#660066"><strong>static</strong></font> <font color="#990099"><strong>void</strong></font> <font color="#990033"><strong>main</strong></font>(String[] args) <font color="#000000">{</font>
    System.out.<font color="#990033"><strong>println</strong></font>(<font color="#0033cc">"</font><font color="#0033cc">©</font><font color="#0033cc"> </font><font color="#0033cc">Acme,</font><font color="#0033cc"> </font><font color="#0033cc">Inc.</font><font color="#0033cc">"</font>);
  <font color="#000000">}</font>
<font color="#000000">}</font>
</font></pre> 
              <p>1. Compiling the code using the correct encoding:</p> 
              <pre>
javac -encoding UTF-8 PrintCopyright.java
</pre> 
              <p>2. Simulating a straight compile on Western-locale Windows:</p> 
              <pre>
javac -encoding Cp1252 PrintCopyright.java
</pre> 
              <p>These compiler settings will produce different outputs; only the first one is correct.</p> 
              <p><em>Note: the JDK 1.6 <code>javac</code> compiler will not compile a UTF-8 source file starting with a byte order mark, failing with the error <code>illegal character: \65279</code>. Byte order marks are discussed further down the page.</em></p> 
              <p>The other way to handle Unicode in source files is to only use characters that are encoded with the same values in many character encodings and replace other characters with Unicode escape sequences. In the amended source, the copyright sign <code>©</code> is replaced with the escape sequence <code>\u00A9</code>:</p> 
              <pre><font color="#000000"><font color="#660066"><strong>public</strong></font> <font color="#990099"><strong>class</strong></font> PrintCopyright <font color="#000000">{</font>
  <font color="#660066"><strong>public</strong></font> <font color="#660066"><strong>static</strong></font> <font color="#990099"><strong>void</strong></font> <font color="#990033"><strong>main</strong></font>(String[] args) <font color="#000000">{</font>
    System.out.<font color="#990033"><strong>println</strong></font>(<font color="#0033cc">"</font><font color="#0033cc">\u00A9</font><font color="#0033cc"> </font><font color="#0033cc">Acme,</font><font color="#0033cc"> </font><font color="#0033cc">Inc.</font><font color="#0033cc">"</font>);
  <font color="#000000">}</font>
<font color="#000000">}</font>
</font></pre> 
              <p>The amended source, saved as UTF-8, will produce the same output whether processed as UTF-8 or Cp1252. The characters used have the same binary values in both encodings.</p> 
              <p>These escape sequences apply to the whole file, not just <code>String</code>/<code>char</code> literals. This can produce some funny edge cases. Try compiling this code:</p> 
              <pre><font color="#000000"><font color="#660066"><strong>public</strong></font> <font color="#990099"><strong>class</strong></font> WillNotCompile <font color="#000000">{</font>
  <font color="#006666">//</font><font color="#006666"> </font><font color="#006666">the</font><font color="#006666"> </font><font color="#006666">path</font><font color="#006666"> </font><font color="#006666">c:\udir</font><font color="#006666"> </font><font color="#006666">is</font><font color="#006666"> </font><font color="#006666">interpreted</font><font color="#006666"> </font><font color="#006666">as</font><font color="#006666"> </font><font color="#006666">an</font><font color="#006666"> </font><font color="#006666">escape</font><font color="#006666"> </font><font color="#006666">sequence</font>
  <font color="#006666">//</font><font color="#006666"> </font><font color="#006666">the</font><font color="#006666"> </font><font color="#006666">escape</font><font color="#006666"> </font><font color="#006666">\u000A</font><font color="#006666"> </font><font color="#006666">is</font><font color="#006666"> </font><font color="#006666">replaced</font><font color="#006666"> </font><font color="#006666">with</font><font color="#006666"> </font><font color="#006666">a</font><font color="#006666"> </font><font color="#006666">new</font><font color="#006666"> </font><font color="#006666">line</font>
<font color="#000000">}</font>
</font></pre> 
              <p>Unicode support is detailed in the <a href="http://java.sun.com/docs/books/jls/third_edition/html/lexical.html">Lexical Structure chapter</a> of the Java Language Specification.</p> 
              <a name="javaencoding_javadatatypes"></a>

              <p>The table above shows some of the things we have to look out for.</p> 
              <p><strong>1. Stored characters can take up an inconsistent number of bytes.</strong> A UTF-8 encoded character might take between one (LATIN_CAPITAL_LETTER_A) and four (MATHEMATICAL_FRAKTUR_CAPITAL_G) bytes. <a href="http://en.wikipedia.org/wiki/Variable-width_encoding">Variable width encoding</a> has implications for reading into and decoding from byte arrays.</p> 
              <p><strong>2. Characters can be represented in <a href="http://en.wikipedia.org/wiki/Duplicate_characters_in_Unicode">multiple forms</a>.</strong> As can be seen with e-acute (<code>é</code>), sometimes there is more than one way to store and render a grapheme. Sometimes they can be formed using combining sequences (as in the e-acute example); sometimes there are similar characters (Greek mu ? vs Mathematical micro µ). This may have implications for sorting, expression matching and capitalisation. It might raise compatibility issues for translating data between encodings. You can normalise strings using the <a href="http://java.sun.com/javase/6/docs/api/java/text/Normalizer.html">Normalizer</a> class, but be aware of any <a href="http://unicode.org/faq/normalization.html">gotchas</a>.</p> 
              <p><strong>3. Not all code points can be stored in a <code>char</code>.</strong> The MATHEMATICAL_FRAKTUR_CAPITAL_G example lies in the supplementary range of characters and cannot be stored in 16 bits. It must be represented by two sequential <code>char</code> values, neither of which is meaningful by itself. The <a href="http://java.sun.com/javase/6/docs/api/java/lang/Character.html">Character</a> class provides methods for working with 32-bit code points.</p>

              <h4>How long is a (piece of) String?</h4> 
              <p>So, if a "character" can span multiple <code>char</code> values, we need to know how that affects our ability to calculate string lengths.</p> 
              <ul> 
               <li><a href="http://java.sun.com/javase/6/docs/api/java/lang/String.html#length()">String.length()</a> returns the number of <code>char</code>s in the String.</li> 
               <li><a href="http://java.sun.com/javase/6/docs/api/java/lang/String.html#codePointCount(int,%20int)">String.codePointCount(int, int)</a> returns the number of Unicode <strong>code points</strong> in the String.</li> 
               <li><a href="http://java.sun.com/javase/6/docs/api/java/text/BreakIterator.html#getCharacterInstance(java.util.Locale)">BreakIterator.getCharacterInstance()</a> can be used to count the number of <strong>graphemes</strong> in a String.</li> 
              </ul> 
              <p>The <code>char</code> count is likely to be most useful for low level data operations; the grapheme count is the one you would use to do your own line wrapping in a GUI. You can find out if a particular font can render your string using the <a href="http://java.sun.com/javase/6/docs/api/java/awt/Font.html#canDisplayUpTo(java.lang.String)">Font class</a>.</p>

              <h4>Encodings</h4>

Most encodings can represent only a subset of the characters supported by Unicode. Java strings use <code>UTF-16</code>.</p>

              <p>If a standard Java API converts between <code>byte</code> and <code>char</code> data, there is a high probability that it is using the default encoding. Examples of such calls are: <a href="http://java.sun.com/javase/6/docs/api/java/lang/String.html#String(byte[])">String(byte[])</a>; <a href="http://java.sun.com/javase/6/docs/api/java/lang/String.html#getBytes()">String.getBytes()</a>; <a href="http://java.sun.com/javase/6/docs/api/java/io/InputStreamReader.html#InputStreamReader(java.io.InputStream)">InputStreamReader(InputStream)</a>; <a href="http://java.sun.com/javase/6/docs/api/java/io/OutputStreamWriter.html#OutputStreamWriter(java.io.OutputStream)">OutputStreamWriter(OutputStream)</a>; anything in <a href="http://java.sun.com/javase/6/docs/api/java/io/FileReader.html">FileReader</a> or <a href="http://java.sun.com/javase/6/docs/api/java/io/FileWriter.html">FileWriter</a>.</p> 
              <p>The problem with using these calls is that it is that you cannot predict whether data written on one machine will be read correctly on another, even if you use the same application. An English Windows PC will use <code>Windows-1252</code>; a Russian Windows PC will use <code>Windows-1251</code>; an Ubuntu machine will use <code>UTF-8</code>. <strong>Prefer encoding methods like <a href="http://java.sun.com/javase/6/docs/api/java/io/OutputStreamWriter.html#OutputStreamWriter(java.io.OutputStream,%20java.nio.charset.Charset)">OutputStreamWriter(OutputStream, Charset)</a> that let you set the encoding explicitly.</strong> Fall back on the default encoding only when you have no choice.</p> 
              <p>Portability is not the only problem with relying on the default encoding. As will be shown, using the old 8-bit encodings that many systems use for compatibility reasons can also result in data loss.</p> 
              <p><em>Encoding operations transform data. Do not try to pass "binary" data through character methods. A Java <code>char</code> is not a C <code>char</code>.</em></p> 
              <p><em>UPDATE (Dec 2009): A quick note on the <code>file.encoding</code> system property, which you may see mentioned in Java encoding discussions. Do not set or rely on this property. <a href="http://bugs.sun.com/view_bug.do?bug_id=4163515">Sun says</a>:</em></p> 
              <blockquote>
               The "file.encoding" property is not required by the J2SE platform specification; it's an internal detail of Sun's implementations and should not be examined or modified by user code. It's also intended to be read-only; it's technically impossible to support the setting of this property to arbitrary values on the command line or at any other time during program execution.
              </blockquote>

              <p>The flaw in this code is that characters in the stream can span reads into the byte buffer. The character U+00A3 (£) becomes corrupted as one half of it ends up in the tail end of the buffer in one pass and the other half ends up at the start of the buffer in the next pass. Neither value is meaningful by itself. Because a UTF-8 encoded character may be between one and four bytes in length, there is no buffer size that will always read whole characters. The entire data would need to be read into a larger byte buffer and decoded in one go. It is safer to use an <a href="http://java.sun.com/javase/6/docs/api/java/io/InputStreamReader.html">InputStreamReader</a>.</p>

              <p>The problem here is that the <code>UTF-16</code> encoding scheme used adds a byte order mark to encoded data. Every time a string is written, another BOM is added, littering the content with unwanted data. When the data is decoded, extra characters end up in the text. All text would need to be concatenated and encoded in one go. It is better to use an <a href="http://java.sun.com/javase/6/docs/api/java/io/OutputStreamWriter.html#OutputStreamWriter(java.io.OutputStream,%20java.nio.charset.Charset)">OutputStreamWriter</a>.</p> 
              <p>There are encodings for which these operations are safe, but you would be tying your method implementations to specific encodings rather than making them general. <strong>Use string class encode/decode methods only on whole data.</strong></p> 
              <a name="javaencoding_notes"></a> 
              <h4>Notes</h4> 
              <p><code>System.out</code> may not always print the results you expect. I <a href="http://illegalargumentexception.blogspot.com/2009/04/i18n-unicode-at-windows-command-prompt.html#charsets_javaconsole">posted about this</a> in terms of the Windows console, but Windows is not the only system with this sort of behaviour.</p> 
              <p>If you feel you are on rocky ground with character encodings, <a href="http://www.bobzblog.com/">Bob Balaban</a> has a good presentation: <a href="http://www.bobzblog.com/tuxedoguy.nsf/dx/multilanguage.pdf/$file/multilanguage.pdf">Multi-Language Character Sets: What They Are, How to Use Them</a> (2001; PDF) [<a href="http://www.lotusgeek.com/sapphireoak/lotusgeekblog.nsf/DOWNLOADS/2BCEBB5206735AA28525711D006DB392/$FILE/Multi_Lang_Char_Sets.pdf">alt link</a>]. He <a href="http://www.lotusgeek.com/LotusGeekBlog.nsf/D6Plinks/ROLR-6M9RAX">worked on character issues</a> for more than a little while. Chances are that you are not a <a href="http://en.wikipedia.org/wiki/IBM_Lotus_Notes">Lotus Notes</a> developer, so you can skip some of the specifics.</p> 
              <p>The <a href="http://site.icu-project.org/">ICU4J</a> library is a good place to turn if you you're trying to find Unicode (and other I18N) <a href="http://site.icu-project.org/charts/comparison">features not included</a> in the Java standard library.</p> 
              <p>Examples were developed using Java 6.</p>

                           <p id="bc_0_5MC" class="comment-content">Code <i>points</i> are unique writing system constructs identified and supported by the Unicode consortium. Consider accented characters like é. In European, Latin-based languages the number of accents (diacritics) available on a letter is usually one at a time. Other writing systems are more complex. So Unicode must allow the building of visible graphemes using more than one diacritic - (e.g. the letter, followed by the accent). There are too many combinations to represent every form. This leads to the situation where there is more than one way to describe é in code points.<br><br>Code <i>points</i> are an abstraction: define a writing system element; give it a number.<br><br>Everything must be represented in memory somehow in a computer system. These use cases allocate units of memory (8/16/32 bits) - code <i>units</i> to represent code points - and are called Unicode Transformation Formats. In UTF-8, a code <i>point</i> is 1, 2, 3 or 4 code <i>units</i>. In UTF-16 a code point is 1 or 2 code units. In UTF-32 a code point is a code unit.<br><br>Java strings are UTF-16 so there can be a difference between the number of code units and the number of code points.<br><br>I encourage reading the <a href="http://www.unicode.org/versions/latest/" rel="nofollow">latest Unicode specification</a> for more information.<br><br>I will leave the code as it is - I hope it describes how to measure the differences clearly enough.</p>
