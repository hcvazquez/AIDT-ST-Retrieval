      <h3 id="linux">Linux</h3> 
      <p>Under Linux, Realm’s package repositories are managed through <a href="https://packagecloud.io/">PackageCloud</a>.       <p>For information on deploying Realm Object Server on Amazon EC2, Azure, or Digital Ocean, read the documentation on <a href="/docs/get-started/installation/deployment">Cloud Deployment</a>.</p> 
      <h2 id="running-the-server">Running the Server</h2> 
      <ul class="nav nav-tabs getting-started" role="tablist"> 
       <li class="active"><a href="#rhel6-administer" role="tab" data-toggle="tab">RHEL/CentOS 6</a></li> 
       <li><a href="#rhel7-ubuntu-administer" role="tab" data-toggle="tab">RHEL/CentOS 7 &amp; Ubuntu</a></li> 
       <li><a href="#mac-administer" role="tab" data-toggle="tab">macOS</a></li> 
      </ul> 
      <div class="tab-content getting-started"> 
       <div class="tab-pane active" id="rhel6-administer"> 
        <p>The Realm Object Server service uses standard <code>service</code> commands:</p> 
        <pre class="highlight shell"><code>sudo service realm-object-server status
sudo service realm-object-server start
sudo service realm-object-server stop
sudo service realm-object-server restart</code></pre> 
       </div> 
       <div class="tab-pane" id="rhel7-ubuntu-administer">

        <pre class="highlight shell"><code>sudo systemctl status realm-object-server
sudo systemctl start realm-object-server
sudo systemctl stop realm-object-server
sudo systemctl restart realm-object-server</code></pre> 
       </div> 
       <div class="tab-pane" id="mac-administer"> 
        <p>To start the server double-click on the <code>start-object-server.command</code> file in the realm-mobile-platform directory you originally downloaded.</p> 
        <p>To stop the macOS version of the server, simply press CTRL-C in the terminal window that was opened by the <code>start-object-server.command</code>.</p>

      <h2 id="the-realm-dashboard">The Realm Dashboard</h2> 
      <p><img src="../../assets/img/docs/ros-rmp/Realm-Dashboard-main.png" alt="The Realm Dashboard" class="img-responsive img-rounded center-block"></p> 
      <p>The Dashboard is a browser-based administration application for the Realm Object Server that provides the following:</p> 
      <ul> 
       <li><strong>Dashboard:</strong> System status, including real-time displays of data rate in/out, bytes in/out, open connections, and open Realms</li> 
       <li><strong>Realms:</strong> Paths, permissions, and owners of Realms synced to this Object Server, with the ability to “drill down” and display the models and contents of individual Realms</li> 
       <li><strong>Users:</strong> User information and management for this Object Server, including granting and removing administrative privileges</li> 
       <li><strong>Functions:</strong> Management and entry of <a href="#realm-functions">Realm Functions</a></li> 
       <li><strong>Logs:</strong> System logs for the Object Server, with selectable detail level</li> 
      </ul> 
      <p>To access the Dashboard on the same machine at the default port (<code>9080</code>), open a new browser window and go to <a href="http://localhost:9080/">http://localhost:9080/</a>. If your server is not your local machine, replace <code>localhost</code> with its hostname or IP address.</p> 
      <p>The first time you access the Dashboard, you will be prompted to create an admin user by supplying an email address and password. After registering, you can log in using those credentials.</p> 
      <h2 id="the-realm-browser">The Realm Browser</h2> 
      <p>The Realm Browser allows developers to observe and edit Realms. The Realm Browser can connect to either macOS or Linux instances of the Realm Object Server. <em>Note: the Realm Browser is currently only available for macOS.</em></p> 
      <div class="card download-card"> 
       <p class="button"> <a class="btn btn-lg btn-indigo" href="https://itunes.apple.com/us/app/realm-browser/id1007457278?mt=12">Download Realm Browser</a> </p> 
       <p>You can also <a href="https://github.com/realm/realm-browser-osx/releases/">download the source on GitHub</a>.</p> 
      </div> 
      <p>You can use the <code>Connect to Object Server</code> option in Realm Browser in one of two ways:</p> 
      <ol> 
       <li>Specifying the username/email address and password of an administrator account</li> 
       <li>Providing the admin token generated by the Realm Object Server</li> 
      </ol> 
      <p>The admin token is stored in a text file on the server. Under Linux, view the token with:</p> 
      <pre class="highlight shell"><code>cat /etc/realm/admin_token.base64</code></pre> 
      <p>On macOS, the token file is in the <code>realm-object-server</code> folder, inside the Realm Mobile Platform folder. Navigate to that folder and view the token:</p> 
      <pre class="highlight shell"><code><span class="nb">cd </span>path-to/realm-mobile-platform
cat realm-object-server/admin_token.base64</code></pre>

      <h2 id="configuring-the-server">Configuring the Server</h2> 
      <p>Realm Object Server includes a configuration file in YAML format, <code>configuration.yml</code>, that comes with sensible defaults for a development environment. The configuration file can be edited to customize the installation and make it production-ready.</p> 
      <p>The default location of the configuration file depends on your platform:</p> 
      <ul> 
       <li><strong>Linux:</strong> <code>/etc/realm/configuration.yml</code></li> 
       <li><strong>macOS:</strong> <code>realm-object-server/object-server</code> inside the Realm Mobile Platform folder</li> 
      </ul> 
      <p>Paths in the configuration file can be either absolute or relative. If relative, they are resolved according to the current working directory. <strong>We recommend always using absolute paths to avoid confusion.</strong></p> 
      <h3 id="mandatory-settings">Mandatory Settings</h3> 
      <p>We recommend specifying these in <code>configuration.yml</code>, but they can also be given as command line arguments.</p> 
      <p>The mandatory configuration defines:</p> 
      <ol> 
       <li>An existing directory where Realm Object Server will store all its files,</li> 
       <li>Paths to a key pair used for securing access to Realm Object Server to only allow authenticated clients to access it. The keys given must be in PEM format and they must be a matching pair.</li> 
      </ol> 
      <table> 
       <thead> 
        <tr> 
         <th>Configuration key</th> 
         <th>CLI argument</th> 
         <th>Description</th> 
        </tr> 
       </thead> 
       <tbody> 
        <tr> 
         <td><code>storage.root_path</code></td> 
         <td><code>--root</code></td> 
         <td>Path to the directory where ROS will store all data files</td> 
        </tr> 
        <tr> 
         <td><code>auth.public_key_path</code></td> 
         <td><code>--public-key</code></td> 
         <td>Path to the public key (stored in PEM format) used to authenticate client tokens</td> 
        </tr> 
        <tr> 
         <td><code>auth.private_key_path</code></td> 
         <td><code>--private-key</code></td> 
         <td>Path to the private key (stored in PEM format) used to authenticate client tokens</td> 
        </tr> 
       </tbody> 
      </table> 
      <div class="alert alert-warning"> 
       <p>The Realm Object Server <em>must</em> have read/write access to the directory pointed to by the <code>storage.root_path</code> setting (or the <code>--root</code> CLI argument). If you change this setting, make sure the <code>realm</code> user has access to the new storage directory (i.e., <code>chown -R realm:realm /new/storage/path</code>).</p> 
      </div> 
      <h3 id="network">Network</h3> 
      <p>In its default configuration Realm Object Server runs internal <a href="#backend-services">backend services</a> as well as a <a href="#proxy-module">proxy module</a>. The backend services are only reachable from the host running the server, and all traffic to them are passed through the proxy module which listen on all interfaces. The following diagram describes the default configuration:</p> 
      <div class="row flex center"> 
       <div class="col-xs-12"> 
        <p><img class="hidden-xs" src="../../assets/svg/docs/ros-proxy.svg" alt="Proxy config diagram"> <img class="visible-xs center-block" src="../../assets/svg/docs/ros-proxy-mobile.svg" alt="Proxy config diagram"></p> 
       </div> 
      </div> 
      <h4 id="backend-services">Backend Services</h4> 
      <p>Realm Object Server internally starts two backend services. The <code>sync</code> service is responsible for core synchronization functionality, and listens on port <code>27800</code>. The <code>http</code> service handles requests for authentication and the dashboard, and listens on port <code>27080</code>. By default, the services only listen on the host’s IPv4 loopback address (<code>127.0.0.1</code>), and requests are routed through the <a href="#proxy-module">proxy module</a>.</p> 
      <p>Should you wish to disable the proxy module and let backend services listen to all IPv4 and IPv6 interfaces, this can be achieved by setting the appropriate <code>listen_address</code> to <code>::</code>. Similarly, the listening ports can be changed by modifying the appropriate <code>listen_port</code>.</p>

       <thead>

         <th>Description</th> 
        </tr>

      <h4 id="proxy-module">Proxy Module</h4> 
      <p>The included reverse proxy module dispatches all incoming traffic to the appropriate backend service. This proxy is capable of handling both HTTP, WebSocket, HTTPS and Secure WebSocket traffic (but exclusively traffic destined for the Realm Object Server, this is not a general purpose proxy). In the default configuration, the HTTP proxy listens on all IPv4 and IPv6 interfaces on port <code>9080</code>, but the HTTPS proxy is disabled. Both variants of the proxy can be enabled simultaneously.</p> 
      <div class="alert alert-warning"> 
       <p>It is recommended that you keep the Sync and HTTP services listening on <code>localhost</code> (default) and simply use the proxy for external access.</p> 
      </div> 
      <p>If you wish, you can disable the included proxy module and replace it with a standalone proxy installation, or expose the backend services on the network for direct client access.</p> 
      <table> 
       <thead> 
        <tr>

         <th>Description</th> 
        </tr>

        </tr> 
        <tr> 
         <td><code>proxy.http.listen_port</code></td> 
         <td><code>9080</code></td> 
         <td>Supply an alternative port for the HTTP Proxy</td> 
        </tr> 
        <tr>

         <td><code>false</code></td>

      <p>If you wish to enable the HTTPS Proxy, you must provide certificate and private key files in PEM format by setting the <code>proxy.https.certificate_path</code> and <code>proxy.https.private_key_path</code> options. The certificate cannot be self-signed.</p> 
      <div class="alert alert-warning"> 
       <p>You must also choose a port number above 1024, as the Realm Object Server does not run as root. It is recommended to use the default port (9443).</p> 
      </div> 
      <p>If you would like to be able to connect to Realm Object Server on a port lower than 1024, such as the default HTTPS port 443, you can forward traffic to the port that Realm Object Server is listening on:</p> 
      <pre class="highlight shell"><code>sudo iptables -A PREROUTING -t nat -p tcp --dport 443 -j REDIRECT --to-port 9443</code></pre> 
      <p>Here is an example of a working proxy configuration with HTTPS:</p> 
      <pre class="highlight yaml"><code><span class="s">proxy</span><span class="pi">:</span>

    <span class="s">enable</span><span class="pi">:</span> <span class="s">true</span>

  <span class="s">http</span><span class="pi">:</span>
    <span class="s">enable</span><span class="pi">:</span> <span class="s">false</span></code></pre>

      <h2 id="realm-functions">Realm Functions</h2> 
      <div class="alert alert-warning"> 
       <p>Realm Functions are a beta release. If you have any issues, please report them <a href="https://github.com/realm/realm-mobile-platform/issues/new">on Github</a>.</p> 
      </div> 
      <p>Realm Functions provide a way to add “serverless” logic that responds to changes in Realms. Using the <a href="#the-realm-dashboard">Dashboard</a>, you can create, edit, start, stop, and delete Functions that respond to changes on all Realms or changes on Realms that match specific patterns (e.g., any user’s Realm that contains <code>/myrealm/</code> in its path).</p> 
      <p><img src="../../assets/img/docs/ros-rmp/realm-functions.png" alt="Realm Functions management screen" class="img-responsive img-rounded center-block"></p> 
      <p>Realm Functions use the <a href="/docs/realm-object-server/pe-ee/#event-handling">Event Handling</a> capability of the Realm Object Server. The Functions themselves are Node.js functions that will be called by the Realm Object Server’s global listening API and passed <code>changeEvent</code> objects.</p> 
      <div class="alert alert-info"> 
       <p>Beta Realm Functions are available in the Developer Edition of the Realm Mobile Platform, but you will be limited to three <em>active</em> functions with DE. You can create as many functions as you wish, but only three may be running at any given time. This limitation is removed on the Professional and Enterprise Editions. (The traditional server-side Event Handling capabilities of the Realm Mobile Platform Realm Functions are available only on Professional and Enterprise Editions.) Other limitations may apply in the final release for the Developer Edition.</p> 
      </div> 
      <p>To create a function, you’ll need to provide three things to the Realm Object Server on the <strong>Create New Function</strong> form:</p> 
      <ol> 
       <li>The name of your function script, as it will appear on the Dashboard.</li> 
       <li>A regular expression specifying the Realm(s) to monitor.</li> 
       <li>The function body.</li> 
      </ol> 
      <h3 id="specifying-realm-url-path-regular-expressions">Specifying Realm URL Path Regular Expressions</h3> 
      <p>The <a href="http://www.regular-expressions.info">regular expression</a> (“regex”) for a Function defines which Realms send change notifications to the function. Only Realms whose <a href="/docs/data-model/#opening-a-realm">Realm URL</a> is matched by the regex will trigger the function. A full regular expression tutorial is beyond the scope of this documentation, but here are a few practical examples:</p> 
      <ul> 
       <li>Match all Realms: <code>.*</code> (e.g., match zero or more of anything!)</li> 
       <li>Match a public “catalog” Realm: <code>^/catalog$</code></li> 
       <li>Match any user-owned “settings” Realm: <code>^/([0-9a-f]+)/settings$</code></li> 
      </ul> 
      <p>The default regex, <code>^/([0-9a-f]+)/myrealm$</code>, matches any user-owned Realm named “myrealm,” restricting the first path segment (the user ID) to hexadecimal characters only (e.g., the digits <code>0-9</code> and the letters <code>a-f</code>).</p> 
      <h3 id="writing-functions">Writing Functions</h3> 
      <p>For the function body, the editing screen starts out with the following snippet of skeleton code:</p> 
      <pre class="highlight javascript"><code><span class="nx">console</span><span class="p">.</span><span class="nx">log</span><span class="p">(</span><span class="s2">"Starting function"</span><span class="p">);</span>
<span class="c1">// add your initialization code here</span>

<span class="nx">module</span><span class="p">.</span><span class="nx">exports</span> <span class="o">=</span> <span class="kd">function</span><span class="p">(</span><span class="nx">changeEvent</span><span class="p">)</span> <span class="p">{</span>
	<span class="c1">// event handler code goes here</span>

	<span class="nx">console</span><span class="p">.</span><span class="nx">log</span><span class="p">(</span><span class="s2">"Changes in realm at:"</span><span class="p">,</span> <span class="nx">changeEvent</span><span class="p">.</span><span class="nx">path</span><span class="p">);</span>

	<span class="kd">var</span> <span class="nx">realm</span> <span class="o">=</span> <span class="nx">changeEvent</span><span class="p">.</span><span class="nx">realm</span><span class="p">;</span>
	
	<span class="k">for</span> <span class="p">(</span><span class="kd">var</span> <span class="nx">className</span> <span class="k">in</span> <span class="nx">changeEvent</span><span class="p">.</span><span class="nx">changes</span><span class="p">)</span> <span class="p">{</span>
	    <span class="kd">var</span> <span class="nx">changes</span> <span class="o">=</span> <span class="nx">changeEvent</span><span class="p">.</span><span class="nx">changes</span><span class="p">[</span><span class="nx">className</span><span class="p">];</span>
	    <span class="kd">var</span> <span class="nx">objects</span> <span class="o">=</span> <span class="nx">realm</span><span class="p">.</span><span class="nx">objects</span><span class="p">(</span><span class="nx">className</span><span class="p">);</span>
	    
	    <span class="nx">console</span><span class="p">.</span><span class="nx">log</span><span class="p">(</span><span class="s2">"Changes in Model:"</span><span class="p">,</span> <span class="nx">className</span><span class="p">);</span>
	    <span class="k">for</span> <span class="p">(</span><span class="kd">let</span> <span class="nx">pos</span> <span class="nx">of</span> <span class="nx">changes</span><span class="p">.</span><span class="nx">insertions</span><span class="p">)</span> <span class="p">{</span>
	        <span class="nx">console</span><span class="p">.</span><span class="nx">log</span><span class="p">(</span><span class="s2">"- object inserted at position "</span><span class="p">,</span> <span class="nx">pos</span><span class="p">,</span> <span class="s2">" : "</span><span class="p">,</span> <span class="nx">objects</span><span class="p">[</span><span class="nx">pos</span><span class="p">]);</span>
	    <span class="p">}</span>
	    <span class="k">for</span> <span class="p">(</span><span class="kd">let</span> <span class="nx">pos</span> <span class="nx">of</span> <span class="nx">changes</span><span class="p">.</span><span class="nx">modifications</span><span class="p">)</span> <span class="p">{</span>
	        <span class="nx">console</span><span class="p">.</span><span class="nx">log</span><span class="p">(</span><span class="s2">"- object modified at position "</span><span class="p">,</span> <span class="nx">pos</span><span class="p">,</span> <span class="s2">" : "</span><span class="p">,</span> <span class="nx">objects</span><span class="p">[</span><span class="nx">pos</span><span class="p">]);</span>
	    <span class="p">}</span>
	    <span class="k">for</span> <span class="p">(</span><span class="kd">let</span> <span class="nx">pos</span> <span class="nx">of</span> <span class="nx">changes</span><span class="p">.</span><span class="nx">deletions</span><span class="p">)</span> <span class="p">{</span>
	        <span class="nx">console</span><span class="p">.</span><span class="nx">log</span><span class="p">(</span><span class="s2">"- object deleted at position "</span><span class="p">,</span> <span class="nx">pos</span><span class="p">);</span>
	    <span class="p">}</span>
	<span class="p">}</span>
	<span class="nx">console</span><span class="p">.</span><span class="nx">log</span><span class="p">(</span><span class="s2">""</span><span class="p">);</span>
<span class="p">};</span></code></pre> 
      <p>This function is your <em>event handler,</em> which will be executed by Node.js on the object server. The event handler receives a <code>changeEvent</code> object that has four keys:</p> 
      <ul> 
       <li><code>path</code>: The path of the changed Realm</li> 
       <li><code>realm</code>: the changed Realm itself</li> 
       <li><code>oldRealm</code>: the changed Realm in its <em>old</em> state, before the changes were applied</li> 
       <li><code>changes</code>: an object containing a hash map of the Realm’s changed objects</li> 
      </ul> 
      <p>The <code>changes</code> object itself has a more complicated structure: it’s a series of key/value pairs, the keys of which are the names of objects, and the values of which are <em>another</em> object with key/value pairs listing insertions, deletions, and modifications to those objects. The values of those keys are index values into the Realm. Here’s the overall structure of the <code>changeEvent</code> object:</p> 
      <pre class="highlight plaintext"><code>{
  path: "realms://server/user/realm",
  realm: &lt;realm object&gt;,
  oldRealm: &lt;realm object&gt;,
  changes: {
    objectType1: {
      insertions: [ a, b, c, ...],
      deletions: [ a, b, c, ...],
      modifications: [ a, b, c, ...]
    },
    objectType2: {
      insertions: [ a, b, c, ...],
      deletions: [ a, b, c, ...],
      modifications: [ a, b, c, ...]
    }
  }
}</code></pre> 
      <p>The sample snippet shows how to access all of these keys in practice. For a more complex example, check out the <a href="/docs/realm-object-server/pe-ee/#event-handling">Event Handling</a> documentation in the Professional and Enterprise Edition section; you’ll see the code for the event handler function is essentially the same. (In the code example there, the <code>NOTIFIER_PATH</code> variable is how the Realm URL path regex is specified.)</p>

      <h2 id="access-control">Access Control</h2> 
      <p>Realm Object Server includes access control mechanisms to restrict which users can sync which Realm files. In order to grant a user access to a Realm, Realm Object Server <em>authenticates</em> the user to verify their identity, then <em>authorizes</em> that the user has the correct permissions to access the Realm.</p> 
      <ul> 
       <li>Realm Object Server uses <a href="#authentication">configurable authentication providers</a> to validate user credentials and authenticate users. A client connects to the server using some user credentials, and these are given to an appropriate provider for validation. If the credentials are valid, the user is granted access to Realm Object Server. A new user account is created if the credentials are not coupled to an existing account.</li> 
       <li>After the client is granted access as a user, it can request read or write access to a specific path identifying a Realm. If the user has the requested access permissions for the given path, the user is <a href="#authorization">authorized</a> to access the Realm, and normal sync operations can begin.</li> 
      </ul> 
      <h3 id="authentication">Authentication</h3> 
      <p>Authentication providers allow developers to utilize third-party authentication mechanisms to control access to Realm apps that go beyond Realm’s own username/password mechanism. Realm supports Facebook, Google, and Apple’s iCloud.</p> 
      <p>These third-party providers currently operate at the single-server level, and will need to be set up on each server that needs to make use of a given authentication mechanism. The setup is done by providing a set of key-value pairs that identify the authentication mechanism and the required keys/secrets.</p> 
      <p>This information goes into the <code>auth.providers</code> section of the Realm <code>configuration.yml</code> file.</p> 
      <p>Below are examples that demonstrate how to integrate each of these third-party authentication providers and where applicable links to the developer pages for those services. Complete examples of all authentication providers can be found in the Realm Object Server <code>configuration.yml</code> file.</p>

      <h3 id="custom-authentication">Custom Authentication</h3> 
      <p>The Realm Object Server supports the ability to use external authentication providers. This allows users to authenticate users against legacy databases or APIs, or integrate with providers that are not supported out-of-the-box by the Realm Object Server. This section of the manual will walk you through the process of writing and setting up a third-party authentication provider.</p> 
      <p>The basics of authentication in the Realm Object Server is simple: when a client tries to login, it provides credentials to the server. The server checks those credentials, typically against an external server or API, and grants rights to the client based on the response.</p> 
      <p>Custom authentication providers can be installed anywhere. By default, the server is configured to look in <code>/usr/local/lib/realm/auth/providers</code>. An authentication provider takes the shape of a Node.js script which uses one of our development APIs.</p> 
      <h4 id="writing-an-authentication-provider">Writing an authentication provider</h4> 
      <div class="alert alert-warning"> 
       <p>Any NPM dependencies your authentication provider relies on must be installed before starting the Realm Object Server. If your custom authentication provider fails with error messages similar to “Expected implementation for [provider name] to be a function, but got object,” check to make sure all your dependencies are available!</p> 
      </div> 
      <p>Create the directory indicated above if it doesn’t exist yet, and create a file with a <code>.js</code> extension. The filename itself can be anything you want, although we recommend using something significant for the type of authentication provider you are implementing (e.g., <code>github.js</code>). In this example, we will create the <code>FooAuth</code> sample provider, which asks an external server to verify the token given by the client.</p> 
      <pre class="highlight javascript"><code><span class="c1">// fooauth.js</span>

<span class="cm">/** * This will be called when the server is started. * * It should return the constructor for the authentication provider. * * @param {object} deps - The dependencies passed from the running server * to this implementation. * @param {function} deps.BaseAuthProvider - the base class to use * @param {object} deps.problem - a set of exceptions to throw on failure * @param {object} deps.models - the models of the admin-Realm * @returns {function} */</span>
<span class="nx">module</span><span class="p">.</span><span class="nx">exports</span> <span class="o">=</span> <span class="kd">function</span><span class="p">(</span><span class="nx">deps</span><span class="p">)</span> <span class="p">{</span>
  <span class="k">return</span> <span class="kr">class</span> <span class="nx">FooAuthProvider</span> <span class="kr">extends</span> <span class="nx">deps</span><span class="p">.</span><span class="nx">BaseAuthProvider</span> <span class="p">{</span>

    <span class="c1">// return name of this custom authentication provider</span>
    <span class="kr">static</span> <span class="nx">get</span> <span class="nx">name</span><span class="p">()</span> <span class="p">{</span>
      <span class="k">return</span> <span class="s1">'custom/fooauth'</span><span class="p">;</span>
    <span class="p">}</span>

    <span class="c1">// ensure required default options are set (optional function)</span>
    <span class="kr">static</span> <span class="nx">get</span> <span class="nx">defaultOptions</span><span class="p">()</span> <span class="p">{</span>
      <span class="k">return</span> <span class="p">{</span>
        <span class="na">server</span><span class="p">:</span> <span class="s1">'https://my.auth.server.example'</span><span class="p">,</span>
      <span class="p">}</span>
    <span class="p">}</span>

    <span class="nx">constructor</span><span class="p">(</span><span class="nx">name</span><span class="p">,</span> <span class="nx">options</span><span class="p">,</span> <span class="nx">requestPromise</span><span class="p">)</span> <span class="p">{</span>
      <span class="kr">super</span><span class="p">(</span><span class="nx">name</span><span class="p">,</span> <span class="nx">options</span><span class="p">,</span> <span class="nx">requestPromise</span><span class="p">);</span>

      <span class="k">this</span><span class="p">.</span><span class="nx">httpMethod</span> <span class="o">=</span> <span class="s1">'GET'</span><span class="p">;</span>
    <span class="p">}</span>

    <span class="c1">// perform the authentication verification</span>
    <span class="nx">verifyIdentifier</span><span class="p">(</span><span class="nx">req</span><span class="p">)</span> <span class="p">{</span>
      <span class="c1">// The token submitted by the client</span>
      <span class="kr">const</span> <span class="nx">token</span> <span class="o">=</span> <span class="nx">req</span><span class="p">.</span><span class="nx">body</span><span class="p">.</span><span class="nx">data</span><span class="p">;</span>

      <span class="c1">// options for the HTTP request</span>
      <span class="kr">const</span> <span class="nx">httpOptions</span> <span class="o">=</span> <span class="p">{</span>
        <span class="na">uri</span><span class="p">:</span> <span class="err">`</span><span class="nx">$</span><span class="p">{</span><span class="k">this</span><span class="p">.</span><span class="nx">options</span><span class="p">.</span><span class="nx">server</span><span class="p">}</span><span class="sr">/api/</span><span class="nx">auth</span><span class="p">?</span><span class="nx">token</span><span class="o">=</span><span class="nx">$</span><span class="p">{</span><span class="nx">token</span><span class="p">}</span><span class="err">`</span><span class="p">,</span>
        <span class="nx">method</span><span class="p">:</span> <span class="k">this</span><span class="p">.</span><span class="nx">httpMethod</span><span class="p">,</span>
        <span class="na">json</span><span class="p">:</span> <span class="kc">true</span><span class="p">,</span>
      <span class="p">};</span>

      <span class="c1">// make request to external provider and return result</span>
      <span class="k">return</span> <span class="k">this</span><span class="p">.</span><span class="nx">request</span><span class="p">(</span><span class="nx">httpOptions</span><span class="p">)</span>
        <span class="p">.</span><span class="k">catch</span><span class="p">((</span><span class="nx">err</span><span class="p">)</span> <span class="o">=&gt;</span> <span class="p">{</span>
          <span class="c1">// Please see src/node/services/problem/http_problem.js</span>
          <span class="c1">// for other error cases</span>
          <span class="k">throw</span> <span class="k">new</span> <span class="nx">deps</span><span class="p">.</span><span class="nx">problem</span><span class="p">.</span><span class="nx">HttpProblem</span><span class="p">.</span><span class="nx">Unauthorized</span><span class="p">({</span>
            <span class="na">detail</span><span class="p">:</span> <span class="err">`</span><span class="nx">Something</span> <span class="nx">bad</span> <span class="na">happened</span><span class="p">:</span> <span class="nx">$</span><span class="p">{</span><span class="nx">err</span><span class="p">.</span><span class="nx">toString</span><span class="p">()}</span><span class="err">`</span><span class="p">,</span>
          <span class="p">});</span>
        <span class="p">})</span>
        <span class="p">.</span><span class="nx">then</span><span class="p">((</span><span class="nx">result</span><span class="p">)</span> <span class="o">=&gt;</span> <span class="p">{</span>
          <span class="c1">// assume user ID value is in the `userId` JSON key returned</span>
          <span class="k">return</span> <span class="nx">result</span><span class="p">.</span><span class="nx">userId</span><span class="p">;</span>
        <span class="p">});</span>
    <span class="p">}</span>
  <span class="p">};</span>
<span class="p">}</span></code></pre> 
      <p>We begin by defining a new class for our authentication provider, <code>FooAuthProvider</code>, which extends the <code>BaseAuthProvider</code> class. This gives us access to a promise-based HTTP request. Let’s walk through the methods our provider must define.</p> 
      <ul> 
       <li> <p>The <code>name()</code> function simply returns the name of the authentication provider. It must begin with <code>custom/</code>, to avoid potential name conflicts between first-party and third-party providers.</p> <p>You can use this name to define options in the <code>configuration.yml</code> file for the Realm Object Server.</p> <pre class="highlight plaintext"><code>custom/fooauth:
  server: 'https://newserver.example'</code></pre> <p>These options become available to your provider under <code>this.options</code> (e.g., <code>this.options.server</code>).</p> </li> 
       <li> <p>The <code>defaultOptions()</code> function is optional. It returns default values for any configuration file variables your code requires, in case they’re not defined.</p> </li> 
       <li> <p>The <code>constructor</code> receives the name, options, and the request Promise. It can set instance variables, verify configuration file values, and perform any other initialization tasks.</p> </li> 
       <li> <p>The <code>verifyIdentifier()</code> function is the heart of your provider. It’s called with the HTTP request, and it should return a unique user ID that will be used by Realm to either create a new user on the Realm Object Server or log in an existing user, depending on whether the user is already in the system or not.</p> <p>In this example, we fill in <code>httpOptions</code> using the <code>server</code> configuration variable and the <code>httpMethod</code> variable defined in the constructor, then make an HTTP request to our provider with that information. On an error, we throw an exception from <code>deps.problem</code>; on success, we return the user ID.</p> </li> 
      </ul> 
      <p>For a real world example, see our AWS Cognito provider on Github:</p> 
      <ul> 
       <li><a href="https://gist.github.com/ianpward/5207ee4cb6cd0526d7738fc95255497d">cognito.js</a> (Server side)</li> 
       <li><a href="https://github.com/realm-demos/realm-loginkit/blob/master/RealmLoginKit%20Apple/RealmLoginKit/Models/AuthenticationProviders/AWSCognitoAuthenticationProvider.swift">AWSCognitoAuthenticationProvider.swift</a> (Part of <a href="https://github.com/realm-demos/realm-loginkit">realm-loginkit</a>)</li> 
      </ul> 
      <h4 id="configuring-the-authentication-provider">Configuring the authentication provider</h4> 
      <p>Now that custom authentication provider has been written, the configuration needs to be updated to tell the server to load it. This is done by adding a few lines to the global configuration file:</p> 
      <pre class="highlight yaml"><code><span class="c1"># configuration.yml</span>
<span class="s">auth</span><span class="pi">:</span>
  <span class="s">providers</span><span class="pi">:</span>
    <span class="c1"># The name of a custom authentication provider must start with `custom/` to</span>
    <span class="c1"># differentiate it from ROS-vendored authentication providers.</span>
    <span class="s">custom/fooauth</span><span class="pi">:</span>
      <span class="c1"># This refers to the `fooauth.js` file.</span>
      <span class="s">implementation</span><span class="pi">:</span> <span class="s">fooauth</span>
      <span class="c1"># This is an option, as defined in the custom authentication provider.</span>
      <span class="s">auth_server</span><span class="pi">:</span> <span class="s1">'</span><span class="s">https://another.server.example'</span></code></pre> 
      <p>By simply adding this configuration, the ROS will automatically load the JS file and enable the authentication provider.</p> 
      <h4 id="using-the-custom-provider-on-the-client">Using the custom provider on the client</h4> 
      <p>All the client SDKs support custom authentication out of the box. It is only a matter of sending the correct information using the right login function. For more details, consult the documentation for your language bindings, and find “Authentication” under the “Sync” heading.</p> 
      <h3 id="authorization">Authorization</h3> 
      <p>Each Realm managed by the Realm Object Server has <em>access permissions</em> that can be set on it; permissions are the fundamental authorization control for the server. Permissions are set on each Realm using three boolean flags:</p> 
      <ul> 
       <li>The <code>mayRead</code> flag indicates the user can read from the Realm.</li> 
       <li>The <code>mayWrite</code> flag indicates the user can write to the Realm.</li> 
       <li>The <code>mayManage</code> flag indicates the user can change permissions on the Realm for other users.</li> 
      </ul> 
      <p>Permissions for a Realm can be set on a <em>default</em> basis and a <em>per-user</em> basis. When a user requests access for a Realm, first the Realm Object Server checks to see if there are per-user permissions set for that user on that Realm. If there are no per-user permissions set for that user, the default permissions for the Realm are used. For example, a Realm might have <code>mayRead</code> set true by default, with individual users being granted <code>mayWrite</code> permissions.</p> 
      <p>By default, a Realm is exclusive to its owner: the owner has all permissions on it, and no other user has <em>any</em> permissions for it. Other users must be explicitly granted access.</p> 
      <div class="alert alert-warning"> 
       <p><b>Write-only</b> permissions (i.e., <code>mayWrite</code> set without <code>mayRead</code>) are not currently supported.</p> 
      </div> 
      <p>Note that admin users are always granted all permissions to all Realms on the Realm Object Server.</p> 
      <h4 id="the-management-realm">The Management Realm</h4> 
      <p>All access management operations are performed by writing special permission change objects to a <strong>Management Realm.</strong> This is a special synchronized Realm; objects created in it are effectively requests to the server to change default and per-user permissions on other Realms.</p> 
      <p>Consult the documentation for the binding you’re using to learn how to access the Management Realm. Generally, it’s retrieved by calling a method on a user object (e.g., <code>SyncUser.managementRealm()</code> for Swift).</p> 
      <h4 id="permission-change-objects">Permission Change Objects</h4> 
      <p>A user adds <code>PermissionChange</code> objects to the Management Realm to change permissions; both per-user permissions and default Realm permissions are set the same way. The user must have the <code>mayManage</code> permission for the Realm they’re trying to manage to make modifications.</p> 
      <p>There are five fields in <code>PermissionChange</code> objects. The first two are required; the permission fields are all optional:</p> 
      <ul> 
       <li><code>realmUrl</code> (string): the URL of the Realm, or <code>*</code> for all Realms owned by the requesting user</li> 
       <li><code>userId</code> (string): the identify of the user whose permissions are being modified, or <code>*</code> for the Realm default permissions</li> 
       <li><code>mayRead</code> (boolean): read access to the Realm</li> 
       <li><code>mayWrite</code> (boolean): write access to the Realm</li> 
       <li><code>mayManage</code> (boolean): management permission for the Realm</li> 
      </ul> 
      <p>If one of the <code>may*</code> flags is not set in the <code>PermissionChange</code> object, it will remain unchanged. Per-user permissions always take higher priority than default permissions, so changing the default permissions for a Realm will not remove permissions already set on individual users.</p> 
      <p>The <code>PermissionChange</code> object will be modified by the Realm Object Server after it attempts to apply the permissions. If the permission change succeeds, the <code>statusCode</code> field will be populated and set to <code>0</code>, and a description may be added in the <code>statusMessage</code> field. If an error occurs, <code>statusCode</code> will be set to a value greater than <code>0</code>, and the error message will be in <code>statusMessage</code>.</p> 
      <p>For the specifics of how to create and use <code>PermissionChange</code> objects, please consult the documentation for your client binding.</p> 
      <h4 id="permission-offer-and-response">Permission Offer and Response</h4> 
      <p>The <code>PermissionOffer</code> and <code>PermissionOfferResponse</code> mechanism allows flexible sharing management between users. A <code>PermissionOffer</code> object functions similarly to a <code>PermissionChange</code> object, but creates a token that can be sent to another user to grant them permissions. The steps are, roughly:</p> 
      <ol> 
       <li>Create the <code>PermissionOffer</code> object and add it to the Management Realm.</li> 
       <li>Wait for the server to process the object and populate a <code>token</code> property on it.</li> 
       <li>Send the <code>token</code> to another user via an external method: email, chat, your own API, etc.</li> 
       <li>The receiving user creates a <code>PermissionOfferResponse</code> object with the token and adds it to their Management Realm.</li> 
       <li>Wait for the server to process the object and return a successful status.</li> 
       <li>Use the <code>realmUrl</code> property now populated on the <code>PermissionOfferResponse</code> object for syncing with that Realm.</li> 
      </ol> 
      <p>Consult the documentation for your client binding for details.</p> 
      <h4 id="the-admin-realm">The Admin Realm</h4> 
      <p>Ultimately, changes made via the Management Realm are made to a single global, authoritative store containing the complete set of permissions for all Realm files and users controlled by a Realm Object Server instance. This Admin Realm is <em>only</em> modifiable by the Object Server itself, and is not accessible to Realm Mobile Database applications.</p> 
      <h2 id="high-availability">High Availability</h2> 
      <p>One of the unique aspects of Realm Mobile Platform is that it is designed to be <strong>offline-first.</strong> Architecturally, clients retain a local copy of data from the server and read or write directly to their copy. The client SDK then sends the changes asynchronously when network connectivity is available and Realm Object Server automatically integrates these changes in a deterministic manner. Such a system is inherently highly available from the perspective of the client. Reads and writes can always occur locally irrelevant of the network conditions or server operations.</p> 
      <p>Realm is currently working on a clustering solution that supports automatic failover. This functionality will be available in the Enterprise Edition. To learn more and get early access, please <a href="https://www2.realm.io/enterprise-pricing">contact us</a>.</p> 
      <h2 id="backup">Backup</h2> 
      <p>The Realm Object Server provides one or two backup systems, depending on your Edition.</p> 
      <ul> 
       <li>The <strong>manual backup</strong> system is included in all versions of Realm Mobile Platform via a command line utility. It can be triggered during server operations to create a copy of the data in the Object Server.</li> 
       <li>The <strong>continuous backup</strong> system, available only in the Enterprise Edition, is a backup server running alongside the main process in the Realm Object Server. It continuously watches for changes across all synchronized Realms and sends changes to one or more backup clients.</li> 
      </ul> 
      <p>Both systems create a directory of Realms from which the Realm Object Server can be restarted after a server crash. The backed up data includes the user Realms, all user account information, and all other metadata used by the Realm Object Server. Both manual and continuous backups can be made from a running Realm Object Server without taking it offline.</p> 
      <p>The following documentation applies to the Manual Backup feature. For more about Continous Backup, read <a href="/docs/realm-object-server/pe-ee/#continous-backup">Continuous Backup</a> in the Enterprise Edition documentation.</p> 
      <p>The manual backup is a console command that backs up a running instance of the Realm Object Server. This command can be executed on a server without shutting it down, any number of times, in order to create an “at rest” backup that can be stored on long-term storage for safekeeping. In order to be as agnostic as possible regarding the way the backup will be persisted, the command simply creates a directory structure containing everything the server needs to get started again in the event of a disk failure.</p> 
      <p>It is recommended that the resulting directory is compressed and sent to an off-site location, such as Amazon S3 or Online C14.</p> 
      <p>Because Realms can be modified during the backup process, the backup command uses Realm’s transaction features to take a consistent snapshot of each Realm. However, since the server is running continuously, the backed up Realms do not represent the state of the server at one particular instant in time. A Realm added to the server while a backup is in progress might be completely left out of the backup. Such a Realm will be included in the next backup.</p> 
      <p>To run <code>realm-backup</code> at the command line, type:</p> 
      <pre class="highlight shell"><code>realm-backup SOURCE TARGET</code></pre> 
      <ul> 
       <li><code>SOURCE</code> is the data directory of the Realm Object Server (typically configured in <code>/etc/realm/configuration.yml</code> under <code>storage.root_path</code>).</li> 
       <li><code>TARGET</code> is the directory where the backup files will be placed. This directory must be empty or absent when the backup starts for safety reasons.</li> 
      </ul> 
      <p>After the backup command completes, <code>TARGET</code> will be a directory with the same sub directory structure as <code>SOURCE</code> and a backup of all individual Realms.</p> 
      <h3 id="server-recovery-from-a-backup">Server Recovery From A Backup</h3> 
      <p>If the data of a Realm Object Server is lost or otherwise corrupted, a new Realm Object Server can be restarted with the backed up data. This is done simply by stopping the server and copying the latest backup directory (e.g., the <code>TARGET</code> directory in the manual backup procedure, or the <code>storage.root_path</code> directory specified by a continuous backup client) into the Realm Object Server’s data directory (e.g., the <code>SOURCE</code> directory in the manual backup procedure). After the backup has been <em>fully copied</em> into the Realm Object Server’s data directory, the server may be restarted.</p> 
      <h3 id="client-recovery-from-a-backup">Client Recovery From A Backup</h3> 
      <p>Any data added to the Object Server <em>after</em> its most recent backup will be lost when the server is restored from that backup. Since Realm Mobile Database clients communicate with the Realm Object Server continuously, though, it’s possible for them to have been updated with that newer data that’s no longer on the server.</p> 
      <p>In the case of such an inconsistency, the Realm Object Server will send an error message to the client, and refuse to synchronize data. The error messages relating to synchronization inconsistencies are:</p> 
      <ul> 
       <li>207 “Bad server file identifier (INTENT)”</li> 
       <li>208 “Bad client file identifier (IDENT)”</li> 
       <li>209 “Bad server version (IDENT, UPLOAD)”</li> 
       <li>211 “Diverging histories (IDENT)”</li> 
      </ul> 
      <p>If the client receives one of these error messages, the only way to continue synchronization from the server is to start over: erase all the data in the local copy of the Realm that received the error, and re-sync with the server. This way, the client will have all the data (and <em>only</em> the data) on the Realm Object Server and will be in a consistent state.</p> 
      <p>Note that if this happens, the client will lose any data that only existed on its local, inconsistent copy of the synchronized Realm (i.e., data that was added to it between the time the Realm Object Server crashed and was restored from its backup). The best way to minimize this potential data loss is to use the continuous backup system. If you can’t do that, run the manual backup system frequently.</p> 
      <h2 id="troubleshooting">Troubleshooting</h2>

sudo service iptables save</code></pre>

      <h3 id="configuration-errors">Configuration Errors</h3> 
      <p>Generally the default settings will work well for most installations. Most settings inside the Realm Object Server configuration file are commented out (the system has pre-programmed default values) and show examples of how they can be to customized. If the configuration file has been customized and there is a problem, error messages will be written to the default log location (in the terminal window under macOS or <code>/var/log/realm-object-server.log</code> on Linux systems).</p> 
      <p>If the Realm Object Server will not start, an error in the configuration file is usually the cause. To debug the config file, you can use the built-in checker by starting a terminal and using:</p> 
      <pre class="highlight shell"><code>realm-object-server --check-configuration /etc/realm/configuration.yml</code></pre> 
      <p>If running under macOS, replace the configuration file path with the path to your <code>configuration.yml</code>.</p> 
      <p>When an error occurs, the server’s parser will print a detailed error message stating the problem and possibly highlighting the line the error occurred on. Once all errors are corrected, the server will start; stop the server by pressing <code>^C</code> and then start it normally by rebooting the system or restarting the Realm Object Server using <code>systemd</code> on Linux or the <code>start-object-server.command</code> on macOS.</p> 
      <p>Error messages are usually self-explanatory, but here are some potential hints:</p> 
      <ul> 
       <li>To be able to run Realm Object Server, the following settings are mandatory: 
        <ul> 
         <li>Storage root directory: Where Realm Object Server should store all its files. Please set <code>storage.root_path</code> or give the <code>--root</code> CLI argument. The directory must exist before ROS can start.</li> 
         <li>Authentication key pair: For securing access to Realm Object Server to only authenticated clients. Please set <code>auth.private_key_path</code> or give the <code>--private-key</code> CLI argument, and please set <code>auth.public_key_path</code> or give the <code>--public-key</code> CLI argument. The keys given must be in PEM format and they must be a matching pair.</li> 
        </ul> </li> 
       <li> <p>The configuration for listening ports must not be overlapping, meaning that <code>listen_port</code> values must be unique in the file. Additionally, the ports must not be bound in another process.</p> </li> 
       <li>The HTTPS proxy will only start if given paths to a valid certificate and private key in <code>proxy.https.certificate_path</code> and <code>proxy.https.private_key_path</code>. The certificate and private key must be in PEM format and they must be a matching pair. The certificate cannot be self-signed.</li> 
      </ul> 
      <h3 id="operational-errors">Operational Errors</h3> 
      <p>Occasionally it can be useful for debugging purposes to check the Realm Object Server logs - which are in the terminal window on macOS or <code>/var/log/realm-object-server.log</code> on Linux systems. Realm Object Server produces two specific classes of error and warning diagnostics that may be useful to system admins and developers.</p> 
      <h3 id="session-specific-errors">Session Specific Errors</h3> 
      <ul> 
       <li> <p>204 “Illegal Realm path (BIND)” Indicates that the Realm path is not valid for the user.</p> </li> 
       <li> <p>207 “Bad server file identifier (IDENT)” Indicates that the local Realm specifies a link to a server-side Realm that does not exist. This is most likely because the server state has been completely reset.</p> </li> 
       <li> <p>211 “Diverging histories (IDENT)” Indicates that the local Realm specifies a server version that does not exists. This is most likely because the server state has been partially reset (for example because a backup was restored).</p> </li> 
      </ul> 
      <h3 id="client-level-errors">Client Level Errors</h3> 
      <ul> 
       <li> <p>105 “Wrong protocol version (CLIENT)” The client and the server use different versions of the sync protocol due to a mismatch in upgrading.</p> </li> 
       <li> <p>108 “Client file bound in other session (IDENT)” Indicates that multiple sync sessions for the same client-side Realm file overlap in time.</p> </li> 
       <li> <p>203 “Bad user authentication (BIND, REFRESH)” Indicates that the server has produced a bad token, or that the SDK has done something wrong.</p> </li> 
       <li> <p>206 “Permission denied (BIND, REFRESH)” Indicates that the user does not have permission to access the Realm at the given path.</p> </li> 
      </ul> 
      <h2 id="conflict-resolution">Conflict Resolution</h2> 
      <p>One of the defining features of mobile is the fact that you can never count on being online. Loss of connectivity is a fact of life, and so are slow networks and choppy connections. But people still expect their apps to work!</p> 
      <p>This means that you may end up having two or more users making changes to the same piece of data independently, creating conflicts. Note that this can happen even with perfect connectivity as the latency of communicating between the phone and the server may be slow enough that they can end up creating conflicting changes at the same time.</p> 
      <p>What Realm does in this case is that it merges the changes after the application of specific rules that ensure that both sides always end up converging to the same result even, though they may have applied the changes in different order.</p> 
      <p>This means that you no longer have the kind of perfect consistency that you could have in a traditional database, what you have now is rather what is termed “strong eventual consistency”. The tradeoff is that you have to be aware of the rules to ensure the consistent result you want, but the upside is that by following a few rules you can have devices working entirely offline and still converging on meaningful results when they meet.</p> 
      <p>At a very high level the rules are as follows:</p> 
      <ul> 
       <li> <p><strong>Deletes always win.</strong> If one side deletes an object it will always stay deleted, even if the other side has made changes to it later on.</p> </li> 
       <li> <p><strong>Last update wins.</strong> If two sides update the same property, the value will end up as the last updated.</p> </li> 
       <li> <p><strong>Inserts in lists are ordered by time.</strong> If two items are inserted at the same position, the item that was inserted first will end up before the other item. This means that if both sides append items to the end of a list they will end up in order of insertion time.</p> </li> 
      </ul> 
      <h3 id="primary-keys">Primary Keys</h3> 
      <p>A <em>primary key</em> is a property whose value uniquely identifies an object in a Realm (just like a primary key in a conventional relational database is a field that uniquely identifies a row in a table). Primary keys aren’t required by Realm, but you can enable them on any object type. Add a property to a Realm model class that you’d like to use as the primary key (such as <code>id</code>), and then let Realm know that property is the primary key. The method for doing that is dependent on the language you’re using; in Cocoa, you override the <code>primaryKey()</code> class method, whereas Java and .NET use annotations. (Consult the documentation for your language binding for more details.)</p> 
      <p>Once a Realm model class has a primary key, Realm will ensure that no other object can be added to the Realm with the same key value. You can <em>update</em> the existing object; in fact, you can update only a subset of properties on a specified object without fetching a copy of the object from the Realm. Again, consult the documentation for your language binding for specifics.</p> 
      <p>For more information, read the <a href="https://realm.io/news/realm-primary-keys-tutorial/">Primary Keys Tutorial</a>.</p> 
      <h3 id="strings">Strings</h3> 
      <div class="alert alert-warning"> 
       <p><em>Note:</em> Strings are not exposed in client APIs yet.</p> 
      </div> 
      <p>Strings are special in that you can see them both as scalar values and as lists of characters. This means that you can set the string to a new string (replacing the entire string) or you can <em>edit</em> the string. If multiple users are editing the same string, you want conflicts to be handled at the character level (similar to the experience you would have in something like Google docs).</p> 
      <h3 id="counters">Counters</h3> 
      <div class="alert alert-warning"> 
       <p><em>Note:</em> Counters are not exposed in client APIs yet.</p> 
      </div> 
      <p>Using integers for counting is also a special case. The way that most programming languages would implement an increment operation (like <code>v += 1</code>), is to read the value, increment the result, and then store it back. This will obviously not work if you have multiple parties doing it simultaneously (they may both read <code>10</code>, increment it to <code>11</code>, and when it merges you would get the result of <code>11</code> rather than the intended <code>12</code>).</p> 
      <p>To support this common use case we offer a way to express the <em>intent</em> that you are incrementing (or decrementing) the value, giving enough hints to the merge that it can reach the correct result. Just as with the strings above, it gives you the choice of updating the entire value, or editing it in a way that conveys more meaning, and allow you to get more precise control of the conflict resolution.</p> 
      <h3 id="custom-conflict-resolution">Custom Conflict Resolution</h3> 
      <p>The standard way to do custom conflict resolution is to change the value into a list. Then each side can add its updates to the list and apply any conflict resolution rules it wants directly in the data model.</p> 
      <p>You can use this technique to implement max, min, first write wins, last write wins and pretty much any kind of resolution you can think of.</p>
