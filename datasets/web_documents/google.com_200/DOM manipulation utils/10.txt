          <h4><a href="https://www.sitepoint.com/series/native-javascript-equivalents-of-jquery-methods/" class="series-2921" title="Native JavaScript Equivalents of jQuery Methods">Native JavaScript Equivalents of jQuery Methods</a></h4>
          <ul class="serieslist-ul">
           <li class="serieslist-li-current">Native JavaScript Equivalents of jQuery Methods: the DOM and Forms</li>
           <li class="serieslist-li"><a href="https://www.sitepoint.com/jquery-vs-raw-javascript-3-events-ajax/" title="Native JavaScript Equivalents of jQuery Methods: Events, Ajax and Utilities">Native JavaScript Equivalents of jQuery Methods: Events, Ajax and Utilities</a></li>

         <p>The debate is still raging on my recent <a href="/do-you-really-need-jquery/"><em>Do You Really Need jQuery?</em></a> article but, in summary, there are two reasons why using jQuery is a good idea:</p> 
         <ol>
          <li>you need to support IE6/7/8 (remember you can’t migrate to jQuery 2.0), or</li> 
          <li>without jQuery, you’d spend longer writing a jQuery-like library than developing your application.</li> 
         </ol>
         <p>For everything else, be pragmatic. jQuery is a 270Kb generic library. You are unlikely to require all the functionality it provides and, even if you omit certain modules, it remains a sizable quantity of code. You may load the 30Kb minified version from a CDN but the browser must halt processing and parse the code on every page before doing anything else.</p> 
         <p>This is the first in a series of articles showing native JavaScript equivalents of common jQuery methods. While you might wish to wrap some of these in shorter alias-like functions, you certainly don’t need to create your own jQuery-like libraries.</p> 
         <h2>DOM Selectors</h2> 
         <p>jQuery permits DOM node selection using CSS selector syntax, e.g.</p> 
         <pre><code>// find all paragraphs with the class "summary" in the article with ID "first"
var n = $("article#first p.summary");
</code></pre> 
         <p>The native equivalent:</p> 
         <pre><code>var n = document.querySelectorAll("article#first p.summary");</code></pre> 
         <p><code>document.querySelectorAll</code> is implemented in all modern browsers and IE8 (although that only supports CSS2.1 selectors). jQuery has additional support for more advanced selectors but, for the most part, it’ll be running <code>document.querySelectorAll</code> inside the <code>$()</code> wrapper.</p> 
         <p>Native JavaScript also provides four alternatives which will almost certainly be faster than querySelectorAll if you can use them:</p> 
         <ol>
          <li><code>document.querySelector(selector)</code> — fetches the first matching node only</li> 
          <li><code>document.getElementById(idname)</code> — fetches a single node by its ID name</li> 
          <li><code>document.getElementsByTagName(tagname)</code> — fetches nodes matching an element (e.g. h1, p, strong, etc).</li> 
          <li><code>document.getElementsByClassName(class)</code> — fetches nodes with a specific class name</li> 
         </ol>
         <p>The getElementsByTagName and getElementsByClassName methods can also be applied to single nodes to limit the result to descendants only, e.g.</p> 
         <pre><code>var n = document.getElementById("first");
var p = n.getElementsByTagName("p");</code></pre> 
         <p>Let’s do some testing. I wrote a small selection of scripts to retrieve all the comment nodes from my <a href="/do-you-really-need-jquery/"><em>Do You Really Need jQuery?</em></a> article 10,000 times. The result:</p>

         <p>I can’t claim strict laboratory conditions and it doesn’t reflect real-world usage but, in this situation, native JavaScript was up to 60x faster. It also illustrates that fetching nodes by ID, tag or class will normally be preferable to <code>querySelectorAll</code>.</p> 
         <h2>DOM Manipulation</h2> 
         <p>jQuery offers several methods for adding further content to the DOM, e.g.</p> 
         <pre><code>$("#container").append("&lt;p&gt;more content&lt;/p&gt;");</code></pre> 
         <p>Beneath the surface, jQuery uses the native innerHTML method, e.g.</p> 
         <pre><code>document.getElementById("container").innerHTML += "&lt;p&gt;more content&lt;/p&gt;";</code></pre> 
         <p>You can also use DOM building techniques. These are safer but rarely faster than innerHTML:</p> 
         <pre><code>var p = document.createElement("p");
p.appendChild(document.createTextNode("more content");
document.getElementById("container").appendChild(p);</code></pre> 
         <p>We can also remove all child nodes in jQuery:</p> 
         <pre><code>$("#container").empty();</code></pre> 
         <p>The native equivalent using innerHTML:</p> 
         <pre><code>document.getElementById("container").innerHTML = null;</code></pre> 
         <p>or a small function:</p> 
         <pre><code>var c = document.getElementById("container");
while (c.lastChild) c.removeChild(c.lastChild);</code></pre> 
         <p>Finally, we could remove the whole element from the DOM in jQuery:</p> 
         <pre><code>$("#container").remove();</code></pre> 
         <p>or native JavaScript:</p> 
         <pre><code>var c = document.getElementById("container");
c.parentNode.removeChild(c);</code></pre>

         <h2>Scalable Vector Graphics</h2> 
         <p>The core jQuery library has been developed to work with the current document. SVGs also have a DOM, but jQuery does not offer direct manipulation of these objects because it’s normally necessary to use methods such as <code>createElementNS</code> and <code>getAttributeNS</code>. It can be made to work and several <a href="http://keith-wood.name/svg.html">plug-ins are available</a>, but it will be more efficient to roll your own code or use a specialized SVG library such as <a href="http://raphaeljs.com/">Raphaël</a> or <a href="http://svgjs.com/">svg.js</a>.</p> 
         <h2>HTML5 Forms</h2> 
         <p>Even the most basic web application will have a form or two. You should always validate user data server-side but, ideally, you’ll supplement it with client-side validation to capture errors before the form is submitted.</p> 
         <p>Client-side validation is straight-forward:</p> 
         <ol>
          <li>You run a function when the form is submitted.</li> 
          <li>If any issues are encountered, you halt the submission and show an error.</li> 
         </ol>
         <p>You can use jQuery. You can use native JavaScript. Which should you choose? <em>Neither</em>.</p> 
         <p>HTML5 has built-in support for various common input types such as emails, telephones, URLs, numbers, times, dates, colors and custom fields based on regular expressions. For example, if you want to force the user to enter an email address, use:</p> 
         <pre><code>&lt;input type="email" name="email" required="required" /&gt;</code></pre> 
         <p>There’s no need for additional JavaScript or jQuery code unless you require a little more sophistication such as comparing two or more fields or showing custom error messages.</p> 
         <p>Older browsers (including IE9 and below) do not understand the new types and will revert to standard textual input fields. Those users will fall back to server-side validation; not a great experience but you can <a href="https://github.com/Modernizr/Modernizr/wiki/HTML5-Cross-Browser-Polyfills#web-forms">apply a shim</a> or hope those people see the light and upgrade.</p>
