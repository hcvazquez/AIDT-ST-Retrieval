      <div class="article__content content column"> 
       <p>Spring Boot makes it fun and <em>easy</em> to build rich Java webapps. It allows you to rapidly develop, test, run and deploy Spring applications. Plus with over 100 starters, Spring Boot provides a huge amount of out-of-the-box functionality that traditionally you had to build yourself.</p> 
       <p>While Spring Boot makes it easier to build Java applications, authentication and authorization protocols can still be one of the biggest pain points for any application developer. Good news though! This 15-minute tutorial shows you how to build a fully-operational Spring Boot webapp that protects user access to restricted paths with Spring Security and <a href="https://stormpath.com/">Stormpath</a>. </p> 
       <p>You never have to build auth again with Stormpath's <a href="https://docs.stormpath.com/java/spring-boot-web/quickstart.html">Spring Boot integration</a>, which offers out-of-the-box identity management that you can implement in minutes. You'll see how easy it is to roll out a simple Spring Boot web application, with a complete user registration and login system, with these features:</p> 
       <ul>
        <li>Login and registration pages</li> 
        <li>Password reset workflows</li> 
        <li>Access control based on Group membership</li> 
        <li>Additional configurable features from the <a href="https://github.com/stormpath/stormpath-sdk-java">Stormpath Java library</a>, including API authentication, SSO, social login, and more</li> 
       </ul>
       <p>This demo uses the <a href="https://github.com/stormpath/stormpath-sdk-java/tree/master/extensions/spring/boot/stormpath-default-spring-boot-starter"><code>stormpath-default-spring-boot-starter</code></a>. The modular design of this starter encompasses Spring Boot 1.3.6 and Spring Security 4.1.1 as well as Spring Boot WebMVC, and the Thymeleaf templating engine. I will be using my Mac, the Terminal app, and the IntelliJ IDE.</p> 
       <p>Grab the code for this tutorial <a href="https://github.com/stormpath/spring-boot-spring-security-tutorial">here</a>, and follow along! </p> 
       <p>Also, throughout this post you can see the example code in action by clicking on the Deploy to Heroku button. All you need to do is register for a free <a href="https://heroku.com">Heroku</a> account.</p> 
       <h2>Get Stormpath</h2> 
       <p>Stormpath is an Identity Management API that enables you to:</p> 
       <ul>
        <li>Authenticate and authorize your users</li> 
        <li>Store data about your users</li> 
        <li>Perform password and social based login</li> 
        <li>Send password reset messages</li> 
        <li>Issue API keys for API-based web apps</li> 
        <li>And much more! Check out detailed <a href="https://docs.stormpath.com/home/">Product Documentation</a></li> 
       </ul>
       <p>In short, Stormpath makes user account management a lot easier, more secure, and more scalable.</p> 
       <p>The first step in this tutorial is to register for a free developer account <a href="https://api.stormpath.com/register">here</a>, if you don't already have one! </p> 
       <h2>Set up Your Spring Boot Project</h2> 
       <p>Whether you are a Maven <a href="https://www.urbandictionary.com/define.php?term=maven">maven</a> or Gradle great, getting your project setup is a snap.</p> 
       <p>For Maven, start with this <code>pom.xml</code> file:</p> 
       <pre><code class="language-xml">&lt;project xmlns="http://maven.apache.org/POM/4.0.0" xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance" xsi:schemaLocation="http://maven.apache.org/POM/4.0.0 http://maven.apache.org/maven-v4_0_0.xsd"&gt;

    &lt;modelVersion&gt;4.0.0&lt;/modelVersion&gt;

    &lt;groupId&gt;com.stormpath.sample&lt;/groupId&gt;

    &lt;version&gt;0.1.0&lt;/version&gt;
    &lt;name&gt;Spring Boot Spring Security Stormpath Tutorial&lt;/name&gt;
    &lt;description&gt;A simple Spring Boot Web MVC application with Spring Security and out-of-the-box login and self-service screens!&lt;/description&gt;

    &lt;dependencies&gt;
        &lt;dependency&gt;

        &lt;/dependency&gt;
    &lt;/dependencies&gt;

    &lt;build&gt;

            &lt;plugin&gt;
                &lt;groupId&gt;org.springframework.boot&lt;/groupId&gt;

                &lt;/executions&gt;
            &lt;/plugin&gt;
        &lt;/plugins&gt;
    &lt;/build&gt;

&lt;/project&gt;</code></pre> 
       <p>Or, here's a <code>build.gradle</code> file, if you prefer:</p> 
       <pre><code class="language-javascript">buildscript {
    repositories {
        mavenCentral()
    }
    dependencies {
        classpath("org.springframework.boot:spring-boot-gradle-plugin:1.3.6.RELEASE")
    }
}

apply plugin: 'java'
apply plugin: 'maven'

    mavenCentral()
}

    compile group: 'com.stormpath.spring', name: 'stormpath-default-spring-boot-starter', version:'1.0.RC9.2'
}</code></pre> 
       <p>You may notice that we're working with a single dependency, regardless of whether you're using Maven or Gradle: <code>stormpath-default-spring-boot-starter</code>. </p> 
       <p>That one dependency gives you all of the Spring Boot, Spring Security and Stormpath magic at once, which we'll demonstrate as we progress through the tutorial.</p> 
       <h2>Gather Your API Credentials and Application href</h2> 
       <p>The connection between your webapp and Stormpath is secured with an API key pair which your webapp will use when it communicates with Stormpath. Download your API key pair (it'll be the <code>apiKey.properties</code> file) from the <a href="https://api.stormpath.com">Admin Console</a>.</p> 
       <p>Also in the Admin Console, you'll want to get the href for your default Stormpath Application. In Stormpath, an Application object is used to link your web app to your user stores inside Stormpath. All new developer accounts have an app called "My Application." Click on "Applications" in the Admin Console, then click on "My Application." On that page you will see the href for the Application. Copy this -- we will need it later.</p> 
       <h2>Write the Spring Boot Application</h2> 
       <p>The code for this section can be found in the <a href="https://github.com/stormpath/spring-boot-spring-security-tutorial/tree/02-LockedDown">LockedDown</a> tag of the code repo. <a href="https://heroku.com/deploy?template=https://github.com/stormpath/spring-boot-spring-security-tutorial/tree/02-LockedDown"><img style="margin:0px;" alt="Deploy" src="https://www.herokucdn.com/deploy/button.png"></a></p> 
       <p>We need three small Java classes and an html template to fire up the first version of our webapp. Let's get to it!</p> 
       <h3>Spring Boot Application Entry Point</h3> 
       <p>All Spring Boot applications have an entry point that works just like an ordinary Java program, <code>main</code> method and everything.</p> 
       <p>Here's <code>Application.java</code>:</p> 
       <pre><code class="language-java">@SpringBootApplication
public class Application  {
    public static void main(String[] args) {
        SpringApplication.run(Application.class, args);
    }
}</code></pre> 
       <p>It's just six little lines of code, including the <code>@SpringBootApplication</code> annotation that kicks off the party.</p> 
       <h3>Spring Security Configuration</h3> 
       <p>Spring Security's default is to lock down your entire application. While this conforms to industry-standard best security practices, it's not terribly useful by itself. Additionally, we need to hook Spring Security and Stormpath together. That brings us to our <code>SpringSecurityWebAppConfig.java</code>:</p> 
       <pre><code class="language-java">@Configuration
public class SpringSecurityWebAppConfig extends WebSecurityConfigurerAdapter {
    @Override
    protected void configure(HttpSecurity http) throws Exception {
        http.apply(stormpath());
    }
}</code></pre> 
       <p>The <code>@Configuration</code> annotation causes Spring Boot to instantiate this class as a configuration and <code>.apply(stormpath())</code> hooks all of the Stormpath authentication and authorization workflows into Spring Security.</p> 
       <p>Because there is no further configuration in the <code>configure</code> method, we maintain the best practice behavior of having everything locked down. However, instead of the default Spring Security authentication flows, we will see the default Stormpath flows. Attempting to browse to any path in the application will result in a redirect to the Stormpath <code>login</code> page.</p> 
       <p>Yep, you saw it happen. A one-line method call and we've got security!</p> 
       <h3>Tie It All Together With Spring WebMVC</h3> 
       <p>Our security configuration above ensures that all paths in the application will be secured.</p> 
       <p>A Controller determines how requested paths get directed to display which templates.</p> 
       <p>Here's our <code>HelloController.java</code>:</p> 
       <pre><code class="language-java">@Controller
public class HelloController {
    @RequestMapping("/")
    String home() {
        return "home";
    }
}</code></pre> 
       <p>The <code>@Controller</code> annotation signals Spring Boot that this is a controller. We have one path defined on line 3, <code>/</code>. Line 5 returns the Thymeleaf template named <code>home</code>. Welcome to MVC routing!</p> 
       <h3>Bring Us <code>home.html</code></h3> 
       <p>By default, the Thymeleaf templating engine will look for templates returned from controllers in a folder called <code>templates</code> in your classpath. It will also look for the default extension of <code>.html</code>.</p> 
       <p>When when the controller we just built returns <code>"home"</code>, Thymeleaf will find the template in <code>resources/templates/home.html</code>.</p> 
       <p>Let's take a look at the <code>home.html</code> file:</p> 
       <pre><code class="language-html">&lt;html xmlns:th="http://www.thymeleaf.org"&gt;
    &lt;head&gt;
        &lt;!--/*/ &lt;th:block th:include="fragments/head :: head"/&gt; /*/--&gt;
    &lt;/head&gt;
    &lt;body&gt;
        &lt;div class="container-fluid"&gt;
            &lt;div class="row"&gt;
                &lt;div class="box col-md-6 col-md-offset-3"&gt;
                    &lt;div class="stormpath-header"&gt;
                        &lt;img src="http://stormpath.com/images/template/logo-nav.png"/&gt;
                    &lt;/div&gt;

                    &lt;h1 th:inline="text"&gt;Hello, [[${account.fullName}]]!&lt;/h1&gt;
                    &lt;a th:href="@{/logout}" class="btn btn-danger"&gt;Logout&lt;/a&gt;
                &lt;/div&gt;
            &lt;/div&gt;
        &lt;/div&gt;
    &lt;/body&gt;
&lt;/html&gt;</code></pre> 
       <p>Line 1 sets up the <code>th</code> namespace for Thymeleaf.</p> 
       <p>Line 3 looks like an html/xml comment. However, this is a directive that Thymeleaf picks up on to include a fragment in this template. The fragment is found at: <code>resources/templates/fragments/head.html</code> and contains all the setup needed to hook in <a href="https://getbootstrap.com/">Bootstrap</a> styling for our views.</p> 
       <p>Lines 13 and 14 are where the action happens. Since every pathway in our application is locked down, we know that we can only access this page post-login. Part of the Stormpath magic is that once logged in, an <code>account</code> object is always in scope to your views. Line 13 shows the logged in user's full name. Line 14 provides a link to log out when clicked.</p> 
       <h3>Fire up Your Spring Boot Application</h3> 
       <p>So that's 1 Stormpath account, 15 lines of Java code, and 19 lines of html template code (3 of which are significant) to bring us to the point of a fully functional Spring Boot WebMVC app protected by Spring Security and backed by Stormpath.</p> 
       <p>Wow.</p> 
       <p>If you stored your <code>apiKey.properties</code> file from before in the standard location: <code>~/.stormpath/apiKey.properties</code> and if you have only the default Stormpath Application that was created for you, no other configuration is necessary to start up the application.</p> 
       <p>Here's the Maven way:</p> 
       <pre><code class="language-bash">mvn clean package
mvn spring-boot:run</code></pre> 
       <p>Note: The <code>spring-boot-maven-plugin</code> also creates an uber-jar due to the presence of the <code>repackage</code> execution. You can exercise the same code by just running Java:</p> 
       <pre><code class="language-bash">mvn clean package
java -jar target/*.jar</code></pre> 
       <p>And, here's the Gradle way:</p> 
       <pre><code class="language-bash">gradle clean build
java -jar build/libs/*.jar</code></pre> 
       <p>You can browse to <a href="http://localhost:8080/"></a><a href="http://localhost:8080/">http://localhost:8080/</a> and see it in action:</p> 
       <p><img data-original="http://stormpath.com/wp-content/uploads/2016/04/spring-boot-spring-security-basic.gif" alt="spring security basic"></p> 
       <p>But wait! What if you stored your <code>apiKey.properties</code> file somewhere else or you have other Stormpath Applications defined? Totally not a problem!</p> 
       <p>Remember that Application href you saved earlier? Go grab it.</p> 
       <p>Maven:</p> 
       <pre><code class="language-bash">mvn clean package
STORMPATH_API_KEY_FILE=~/.stormpath/apiKey.properties \
STORMPATH_APPLICATION_HREF=https://api.stormpath.com/v1/applications/4YHCurbtcaFC4TKKsd3AYQ \
mvn spring-boot:run</code></pre> 
       <p>Gradle:</p> 
       <pre><code class="language-bash">gradle clean build
STORMPATH_API_KEY_FILE=~/.stormpath/apiKey.properties \
STORMPATH_APPLICATION_HREF=https://api.stormpath.com/v1/applications/4YHCurbtcaFC4TKKsd3AYQ \
java -jar build/libs/*.jar</code></pre> 
       <p>By adding <code>STORMPATH_API_KEY_FILE</code> and <code>STORMPATH_APPLICATION_HREF</code> environment variables to the command line, we can easily tell our app where to find the API key pairs and which Stormpath Application to use.</p> 
       <p>The <a href="https://docs.stormpath.com/java/">Stormpath Java SDK</a> has an extremely flexible configuration mechanism. We will see more of that below when we get to restricting access to your application by Group membership.</p> 
       <h2>Refined Access Control With Spring Security</h2> 
       <p>The code for this section can be found under the <a href="https://github.com/stormpath/spring-boot-spring-security-tutorial/tree/03-BasicAccessControl"><code>BasicAccessControl</code></a> tag of the code repo. </p> 
       <p><a href="https://heroku.com/deploy?template=https://github.com/stormpath/spring-boot-spring-security-tutorial/tree/03-BasicAccessControl"><img style="margin:0px;" alt="Deploy" src="https://www.herokucdn.com/deploy/button.png"></a></p> 
       <p>So, in the last section we created a webapp that was locked up tight. Every path, including <code>/</code>, required login.</p> 
       <p>But, maybe you need a publicly accessible home page. Or some areas of the site that any authenticated user can access and other areas restricted to members based on their Group.</p> 
       <p>Stormpath makes those kinds of fine-grained controls possible, <em>and simple</em>.</p> 
       <h3>Spring Security: Your Bouncer at the Door</h3> 
       <p>We'll start by allowing public access to the home page. Users will still have to authenticate to access any another page. Remember our empty <code>SpringSecurityWebAppConfig.java</code> from before? We are going to add a little something to it now:</p> 
       <pre><code class="language-java">@Configuration
public class SpringSecurityWebAppConfig extends WebSecurityConfigurerAdapter {
    @Override
    protected void configure(HttpSecurity http) throws Exception {
        http
            .apply(stormpath()).and()
            .authorizeRequests()
            .antMatchers("/").permitAll();
    }
}</code></pre> 
       <p><a href="http://projects.spring.io/spring-security/">Spring Security provides</a> a <a href="https://en.wikipedia.org/wiki/Fluent_interface">fluent interface</a> for providing access rules.</p> 
       <p>On Lines 5 - 8 above, we are building a rule set for how Spring Security will allow access to our application.</p> 
       <p>You might state it like this in plain English:</p> 
       <pre><code>Permit anyone to go to the front door
Ensure that they've authenticated for anything else</code></pre> 
       <p>The rules we are specifying take precedence before the default behavior of locking everything down.</p> 
       <p>Let's update our <code>home.html</code> template as well:</p> 
       <pre><code class="language-html">&lt;html xmlns:th="http://www.thymeleaf.org"&gt;
    &lt;head&gt;
        &lt;!--/*/ &lt;th:block th:include="fragments/head :: head"/&gt; /*/--&gt;
    &lt;/head&gt;
    &lt;body&gt;
        &lt;div class="container-fluid"&gt;
            &lt;div class="row"&gt;
                &lt;div class="box col-md-6 col-md-offset-3"&gt;
                    &lt;div class="stormpath-header"&gt;
                        &lt;img src="http://stormpath.com/images/template/logo-nav.png"/&gt;
                    &lt;/div&gt;

                    &lt;!--/* displayed if account IS NOT null, indicating that the user IS logged in */--&gt;
                    &lt;div th:if="${account}"&gt;
                        &lt;h1 th:inline="text"&gt;Hello, [[${account.fullName}]]!&lt;/h1&gt;
                        &lt;a href="/restricted" class="btn btn-primary"&gt;Restricted&lt;/a&gt;
                        &lt;a th:href="@{/logout}" class="btn btn-danger"&gt;Logout&lt;/a&gt;
                    &lt;/div&gt;

                    &lt;!--/* displayed if account IS null, indicating that the user IS NOT logged in */--&gt;
                    &lt;div th:unless="${account}"&gt;
                        &lt;h1&gt;Who are you?&lt;/h1&gt;
                        &lt;a href="/restricted" class="btn btn-primary"&gt;Restricted&lt;/a&gt;
                    &lt;/div&gt;
                &lt;/div&gt;
            &lt;/div&gt;
        &lt;/div&gt;
    &lt;/body&gt;
&lt;/html&gt;</code></pre> 
       <p>Notice how we now have two distinct sections. The first starts on line 13 and is displayed if the user is logged in. The second section starting on line 20 is displayed if the user is not logged in.</p> 
       <p>Here you're seeing Thymeleaf templates in action! Thymeleaf provides very powerful controls for conditionally showing parts of a template.</p> 
       <p>Before we make any additional changes, let's pause and start up the app.</p> 
       <p>When you browse to <a href="http://localhost:8080"></a><a href="http://localhost:8080">http://localhost:8080</a>, you should see the unauthenticated version of the <code>home</code> template.</p> 
       <p><img data-original="https://stormpath.com/wp-content/uploads/2015/11/restricted.png" alt="restricted"></p> 
       <p>Click the <code>Restricted</code> button, and you'll be redirected to the <code>login</code> form, as expected. After you authenticate, you'll end up at a <code>404</code> page, because we haven't defined the restricted page yet. </p> 
       <p>Defining the restricted page is as easy as adding a route in our controller and creating a template to show. Here's the updated <code>HelloController.java</code>:</p> 
       <pre><code class="language-java">@Controller
public class HelloController {

    @RequestMapping("/")
    String home() {
        return "home";
    }

    @RequestMapping("/restricted")
    String restricted() {
        return "restricted";
    }
}</code></pre> 
       <p>And, here's a new <code>restricted.html</code> template:</p> 
       <pre><code class="language-html">&lt;html xmlns:th="http://www.thymeleaf.org"&gt;
    &lt;head&gt;
        &lt;!--/*/ &lt;th:block th:include="fragments/head :: head"/&gt; /*/--&gt;
    &lt;/head&gt;
    &lt;body&gt;
        &lt;div class="container-fluid"&gt;
            &lt;div class="row"&gt;
                &lt;div class="box col-md-6 col-md-offset-3"&gt;
                    &lt;div class="stormpath-header"&gt;
                        &lt;img src="http://stormpath.com/images/template/logo-nav.png"/&gt;
                    &lt;/div&gt;

                    &lt;h1 th:inline="text"&gt;[[${account.fullName}]], You are allowed here.&lt;/h1&gt;
                    &lt;a href="/" class="btn btn-primary"&gt;Go Home&lt;/a&gt;

                &lt;/div&gt;
            &lt;/div&gt;
        &lt;/div&gt;
    &lt;/body&gt;
&lt;/html&gt;</code></pre> 
       <p>Notice how we re-use the <code>head</code> fragment to provide Bootstrap styling for this template.</p> 
       <p>Re-start the app again, and you will get the full experience of the home page changing depending on whether or not you are logged in.</p> 
       <p><img data-original="http://stormpath.com/wp-content/uploads/2016/04/semi-restricted.gif" alt="semi restricted"></p> 
       <h2>Spring Security Access Control By Group Membership</h2> 
       <p>The code for this section can be found in the <a href="https://github.com/stormpath/spring-boot-spring-security-tutorial/tree/04-GroupAccessControl">GroupAccessControl</a> tag of the code repo. </p> 
       <p><a href="https://heroku.com/deploy?template=https://github.com/stormpath/spring-boot-spring-security-tutorial/tree/04-GroupAccessControl"><img style="margin:0px;" alt="Deploy" src="https://www.herokucdn.com/deploy/button.png"></a></p> 
       <p>Spring Security provides a set of annotations and a rich expression language for controlling access to methods in your application. Among the most commonly used Spring Security annotations is <code>@PreAuthorize</code>. And, among the most commonly used <a href="http://docs.spring.io/spring/docs/current/spring-framework-reference/html/expressions.html">SpringEL</a> expressions is <code>hasRole</code>.</p> 
       <p>We integrate with this mechanism, connecting Stormpath Groups to Spring Security roles.</p> 
       <p>Let's break down this code by adding a new service that restricts access by Group membership. Here's <code>AdminService</code>:</p> 
       <pre><code class="language-java">@Service
public class AdminService {
    @PreAuthorize("hasRole(@roles.ADMIN)")
    public boolean ensureAdmin() {
        return true;
    }
}</code></pre> 
       <p>Line 3 above is the key. The annotation along with the <a href="http://docs.spring.io/spring/docs/current/spring-framework-reference/html/expressions.html">SpringEL</a> expression could be stated in plain English as:</p> 
       <pre><code>Before this method is even entered,
check to see that user is authenticated and
is a member of the ADMIN group</code></pre> 
       <p>The <code>check to see that user is authenticated</code> part of this may not be obvious. What's going on is that a <code>@PreAuthorize</code> check can only be done on an authenticated user. Spring Security is smart enough to check that the user is logged in before checking to see if they are a member of the specified group.</p> 
       <p>Let's dig in to the Spring Expression Language above. Where is Spring Security looking for <code>@roles.ADMIN</code>? The <code>@</code> symbol is special - it identifies a Java bean, in this case named <code>roles</code>. Defined inside that bean we expect to find a constant named <code>ADMIN</code>.</p> 
       <p>Ultimately, <code>hasRole</code> needs to be checking against a unique href representing a Stormpath Group. So, our <code>ADMIN</code> constant needs to be a Java <code>string</code> that holds the href to our Stormpath Group used for admin.</p> 
       <p>To complete this configuration and to make it awesomely dynamic, we need a new class called <code>Roles.java</code>:</p> 
       <pre><code class="language-java">@Component
public class Roles {
    public final String ADMIN;

    @Autowired
    public Roles(Environment env) {
        ADMIN = env.getProperty("stormpath.authorized.group.admin");
    }
}</code></pre> 
       <p>These 9 lines are so amazingly powerful, I'm geeking out over here! Are you seeing it yet?</p> 
       <p>By annotating this class with <code>@Component</code> on line 1, Spring will instantiate it and expose it as a bean. Guess what the name of the bean is? Spring will take the name of the class and camel-case it to derive the bean name by default. So, the bean name is <code>roles</code>. Sweet!</p> 
       <p>The <code>@Autowired</code> annotation on line 5 causes the Spring <code>Environment</code> object to be passed into the constructor. Inside the constructor, we have our only opportunity to set <code>ADMIN</code> since it's declared <code>final</code> - a requirement to be able to use it inside the <code>hasRoles</code> clause.</p> 
       <p>The last piece of the puzzle utilizes some Spring configuration magic. Notice that we are setting the value of <code>ADMIN</code> to whatever the environment property named <code>stormpath.authorized.group.admin</code> is set to. This is standard Spring. If you have a property in your <code>application.properties</code> file with this name, it will be available in the Spring Environment.</p> 
       <p>Spring also supports the ability to set this as a system environment variable, alleviating the need to have the value - a Stormpath Group href in this case - hardcoded anywhere in your application.</p> 
       <p>Typically, system environment variables are <code>ALL_CAPS_WITH_WORDS_SEPARATED_BY_UNDERSCORES</code>. Spring automatically converts these system variables into the <code>lowercase.dotted.notation</code>.</p> 
       <p>Dig this:</p> 
       <pre><code class="language-bash">STORMPATH_AUTHORIZED_GROUP_ADMIN=https://api.stormpath.com/v1/groups/1wcsYMUZhzytUH5GmdaFXC \
java -jar build/libs/spring-boot-spring-security-tutorial-0.1.0.jar</code></pre> 
       <p>Behind the scenes, Spring will convert the <code>STORMPATH_AUTHORIZED_GROUP_ADMIN</code> system environment variable to a Spring env variable named <code>stormpath.authorized.group.admin</code>. That will get picked up by our code above.</p> 
       <p>Now, we need to wire the <code>AdminService</code> to our Controller. Here are the relevant parts of our updated <code>HelloController.java</code>:</p> 
       <pre><code class="language-java">@Controller
public class HelloController {

    @Autowired
    AdminService adminService;

    ...

    @RequestMapping("/admin")
    String admin() {
        adminService.ensureAdmin();
        return "admin";
    }
}</code></pre> 
       <p><code>AdminService</code> is Autowired in on lines 4 &amp; 5. Notice on line 11, we are calling the <code>adminService.ensureAdmin</code> method. If the logged in user is NOT in the <code>ADMIN</code> group, a <code>403</code> (forbidden) response will be generated.</p> 
       <p>The last bit of code housekeeping to do here is to create an <code>admin.html</code> template. In the <a href="https://github.com/stormpath/spring-boot-spring-security-tutorial/tree/04-GroupAccessControl">code</a> that is associated with this post, there's a simple <code>admin.html</code> template that shows a nicely formatted message confirming that you are, indeed, an admin.</p> 
       <p>Now, to see this in action, you'll need to do a little bit of Stormpath housekeeping in the <a href="https://api.stormpath.com">admin console</a>.</p> 
       <p>Here are the steps:</p> 
       <ol>
        <li> <p>Create a new Application</p> <p><img data-original="http://stormpath.com/wp-content/uploads/2016/04/applications-home.png" alt="applications home"></p> <p><img data-original="http://stormpath.com/wp-content/uploads/2016/04/create-application.png" alt="create application"></p> </li> 
        <li> <p>Create a new Group called "admin" for the Application</p> <p><img data-original="http://stormpath.com/wp-content/uploads/2016/04/create-group.png" alt="groups home"></p> <p><img data-original="http://stormpath.com/wp-content/uploads/2016/04/create-group-2.png" alt="create group"></p> </li> 
        <li> <p>Create a new Account in the admin Group</p> <p><img data-original="http://stormpath.com/wp-content/uploads/2016/04/create-account-in-group.png" alt="accounts home"></p> <p><img data-original="http://stormpath.com/wp-content/uploads/2016/04/create-account-in-group-2.png" alt="create account"></p> <p><img data-original="http://stormpath.com/wp-content/uploads/2016/04/create-account-in-group-3.png" alt="accounts home 2"></p> </li> 
        <li> <p>Create another new Account, but NOT in the admin Group</p> <p><img data-original="http://stormpath.com/wp-content/uploads/2016/04/create-account.png" alt="accounts home"></p> <p><img data-original="http://stormpath.com/wp-content/uploads/2016/04/create-account-2.png" alt="create account"></p> </li> 
       </ol>
       <p>In the <a href="https://github.com/stormpath/spring-boot-spring-security-tutorial/tree/03-GroupAccessControl">code</a> for this post, I've also included a handy error page so that if you are not in the admin Group you get a nicely formatted page rather than the default <code>403</code> page.</p> 
       <p>Now we're going to start up the application again, only this time we'll use the hrefs found in the <a href="https://api.stormpath.com">Admin Console</a> for the new Application and Group you just created.</p> 
       <p>With Maven:</p> 
       <pre><code class="language-bash">mvn clean package
STORMPATH_API_KEY_FILE=~/.stormpath/apiKey.properties \
STORMPATH_APPLICATION_HREF=https://api.stormpath.com/v1/applications/4YHCurbtcaFC4TKKsd3AYQ \
STORMPATH_AUTHORIZED_GROUP_ADMIN=https://api.stormpath.com/v1/groups/1wcsYMUZhzytUH5GmdaFXC \
mvn spring-boot:run</code></pre> 
       <p>With Gradle:</p> 
       <pre><code class="language-bash">gradle clean build
STORMPATH_API_KEY_FILE=~/.stormpath/apiKey.properties \
STORMPATH_APPLICATION_HREF=https://api.stormpath.com/v1/applications/4YHCurbtcaFC4TKKsd3AYQ \
STORMPATH_AUTHORIZED_GROUP_ADMIN=https://api.stormpath.com/v1/groups/1wcsYMUZhzytUH5GmdaFXC \
java -jar build/libs/spring-boot-spring-security-tutorial-0.1.0.jar</code></pre> 
       <p>Browse to the <code>/admin</code> page.</p> 
       <p>If you log in as the user you created in the Stormpath <code>admin</code> Group (micah+admin@stormpath.com in my case), you will have access to the admin page. If you log in as the user you created that's NOT in the Stormpath <code>admin</code> Group (micah+user@stormpath.com in my case), you will get the forbidden page. </p> 
       <p>No matter who you log in as, you will have access to the <code>/restricted</code> page.</p> 
       <p><img data-original="http://stormpath.com/wp-content/uploads/2016/04/admin-group.gif" alt="admin group"></p> 
       <h2>Wrapping Up</h2> 
       <p>We've covered a lot of ground here, haven't we? Let's review:</p> 
       <ul>
        <li>We saw how you can protect individual pages as well as protect methods based on membership in a Group.</li> 
        <li>We saw how you can <em>apply</em> Stormpath in your class that extends <code>WebSecurityConfigurerAdapter</code> to define access controls for different paths as well as make use of the Spring Security <code>@PreAuthorize</code> annotation for a finer grain of control.</li> 
        <li>We saw how the Stormpath Java SDK manages environment configuration properties to provide a high degree of flexibility for your application without having to hardcode anything.</li> 
       </ul>
       <p>There are <strong>a ton</strong> of additional features in the Java SDK and the Spring Boot integration that I didn't demonstrate in this tutorial. You can <a href="http://docs.stormpath.com/java/spring-boot-web/tutorial.html">walk through a more in-depth Stormpath tutorial here</a>.</p> 
       <p>If you have questions or want to learn more catch me on Twitter <a href="twitter.com/afitnerd">@afitnerd</a> ot drop a line over to <a href="mailto:support@stormpath.com">support</a> anytime.</p>
