      <p>In <a href="/wiki/Computing" title="Computing">computing</a>, <b>JavaScript Object Notation</b> or <b>JSON</b> (<span class="nowrap"><span class="IPA nopopups noexcerpt"><a href="/wiki/Help:IPA/English" title="Help:IPA/English">/<span style="border-bottom:1px dotted"><span title="/?/: primary stress follows">?</span><span title="/d?/: 'j' in 'jam'">d?</span><span title="/e?/: 'a' in 'face'">e?</span><span title="'s' in 'sigh'">s</span><span title="/?n/: 'on' in 'button'">?n</span></span>/</a></span></span> <a href="/wiki/Help:Pronunciation_respelling_key" title="Help:Pronunciation respelling key"><small title="English pronunciation respelling"><i><b>JAY</b>-s?n</i></small></a>),<sup id="cite_ref-Pronunciation_1-0" class="reference"><a href="#cite_note-Pronunciation-1">[1]</a></sup> is an <a href="/wiki/Open_standard" title="Open standard">open-standard</a> <a href="/wiki/File_format" title="File format">file format</a> that uses <a href="/wiki/Human-readable_medium" title="Human-readable medium">human-readable</a> text to transmit data objects consisting of <a href="/wiki/Attribute%E2%80%93value_pair" title="Attribute–value pair">attribute–value pairs</a> and <a href="/wiki/Array_data_type" title="Array data type">array data types</a> (or any other <a href="/wiki/Serialization" title="Serialization">serializable</a> value). It is a very common data format used for <a href="/wiki/Asynchronous_I/O" title="Asynchronous I/O">asynchronous</a> browser–server communication, including as a replacement for <a href="/wiki/XML" title="XML">XML</a> in some <a href="/wiki/Ajax_(programming)" title="Ajax (programming)">AJAX</a>-style systems.<sup id="cite_ref-2" class="reference"><a href="#cite_note-2">[2]</a></sup></p> 
      <p>JSON is a <a href="/wiki/Language-independent_specification" title="Language-independent specification">language-independent</a> data format. It was derived from <a href="/wiki/JavaScript" title="JavaScript">JavaScript</a>, but as of 2017<sup class="plainlinks noprint asof-tag update" style="display:none;"><a class="external text" href="//en.wikipedia.org/w/index.php?title=JSON&amp;action=edit">[update]</a></sup> many <a href="/wiki/Programming_language" title="Programming language">programming languages</a> include code to generate and <a href="/wiki/Parsing" title="Parsing">parse</a> JSON-format data. The official Internet <a href="/wiki/Media_type" title="Media type">media type</a> for JSON is <code>application/json</code>. JSON filenames use the extension <code>.json</code>.</p> 
      <p><a href="/wiki/Douglas_Crockford" title="Douglas Crockford">Douglas Crockford</a> originally specified the JSON format in the early 2000s; two competing standards, <a href="/wiki/Request_for_Comments_(identifier)" class="mw-redirect" title="Request for Comments (identifier)">RFC</a>&nbsp;<a rel="nofollow" class="external text" href="https://tools.ietf.org/html/rfc7159">7159</a> and <a rel="nofollow" class="external text" href="https://www.ecma-international.org/publications/standards/Ecma-404.htm">ECMA-404</a>, defined it in 2013. The ECMA standard describes only the allowed syntax, whereas the RFC covers some security and interoperability considerations.<sup id="cite_ref-3" class="reference"><a href="#cite_note-3">[3]</a></sup></p> 
      <p>A restricted profile of JSON, known as <b>I-JSON</b> (short for "Internet JSON"), seeks to overcome some of the interoperability problems with JSON. It is defined in <a href="/wiki/Request_for_Comments_(identifier)" class="mw-redirect" title="Request for Comments (identifier)">RFC</a>&nbsp;<a rel="nofollow" class="external text" href="https://tools.ietf.org/html/rfc7493">7493</a>.<sup id="cite_ref-RFC7493_4-0" class="reference"><a href="#cite_note-RFC7493-4">[4]</a></sup>.</p>

      <h2><span class="mw-headline" id="History">History</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=JSON&amp;action=edit&amp;section=1" title="Edit section: History">edit</a><span class="mw-editsection-bracket">]</span></span></h2> 
      <div class="thumb tright"> 
       <div class="thumbinner" style="width:222px;">
        <a href="/wiki/File:Douglas_Crockford.jpg" class="image"><img alt="" src="//upload.wikimedia.org/wikipedia/commons/thumb/e/ef/Douglas_Crockford.jpg/220px-Douglas_Crockford.jpg" width="220" height="293" class="thumbimage" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/e/ef/Douglas_Crockford.jpg/330px-Douglas_Crockford.jpg 1.5x, //upload.wikimedia.org/wikipedia/commons/thumb/e/ef/Douglas_Crockford.jpg/440px-Douglas_Crockford.jpg 2x" data-file-width="1920" data-file-height="2560"></a> 
        <div class="thumbcaption"> 
         <div class="magnify">
          <a href="/wiki/File:Douglas_Crockford.jpg" class="internal" title="Enlarge"></a>
         </div> Douglas Crockford first specified JSON. Image from 2007.
        </div>

      <p>JSON grew out of a need for <a href="/wiki/State_(computer_science)" title="State (computer science)">stateful</a>, real-time server-to-browser communication protocol without using browser plugins such as <a href="/wiki/Adobe_Flash" title="Adobe Flash">Flash</a> or <a href="/wiki/Java_(programming_language)" title="Java (programming language)">Java</a> applets, the dominant methods used in the early 2000s.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (December 2016)">citation needed</span></a></i>]</sup></p> 
      <p><a href="/wiki/Douglas_Crockford" title="Douglas Crockford">Douglas Crockford</a> first specified<sup id="cite_ref-5" class="reference"><a href="#cite_note-5">[5]</a></sup> and popularized the JSON format. The acronym originated at State Software (originally named Veil Networks, Inc.), a company co-founded by Crockford, <a href="/wiki/Randy_Farmer" title="Randy Farmer">Randy Farmer</a> (left to work at 3DO), Greg Macdonald, <a href="/wiki/Chip_Morningstar" title="Chip Morningstar">Chip Morningstar</a>, Robert F. Napiltonia and Dominik Zynis in March 2001. State Software was funded by Tesla Ventures with $1.8 Million in October 2011, and tried to trademark the word "State". The co-founders agreed to build a system that used standard browser capabilities and provided an <a href="/wiki/Abstraction_layer" title="Abstraction layer">abstraction layer</a> for Web developers to create stateful Web applications that had a persistent duplex connection to a Web server by holding two <a href="/wiki/Hypertext_Transfer_Protocol" title="Hypertext Transfer Protocol">HTTP</a> connections open and recycling them before standard browser time-outs if no further data were exchanged. The co-founders had a round-table discussion and voted whether to call the data format JSML or JSON, as well as under what <a href="/wiki/Software_license" title="Software license">license</a> type to make it available. Crockford, being inspired by the words of then President Bush, should also be credited with coming up with the "evil-doers" JSON license ("The Software shall be used for Good, not Evil.") in order to <a href="/wiki/Open-source_model" title="Open-source model">open-source</a> the JSON libraries, but force (<a href="/wiki/Copyright_troll" title="Copyright troll">troll</a>) corporate lawyers, or those who are overly pedantic, to seek to pay for a license from State. Morningstar developed the idea for the State Application Framework at State Software.<sup id="cite_ref-Morningstar_6-0" class="reference"><a href="#cite_note-Morningstar-6">[6]</a></sup><sup id="cite_ref-State_Software_7-0" class="reference"><a href="#cite_note-State_Software-7">[7]</a></sup> On the other hand, this clause led to <a href="/wiki/License_compatibility" title="License compatibility">license compatibility</a> problems of the JSON license with other <a href="/wiki/Open-source_license" title="Open-source license">open-source licenses</a>.<sup id="cite_ref-8" class="reference"><a href="#cite_note-8">[8]</a></sup></p> 
      <p>A precursor to the JSON libraries was used in a children's digital asset trading game project named <a href="/wiki/Cartoon_Orbit" title="Cartoon Orbit">Cartoon Orbit</a> at Communities.com (the State co-founders had all worked at this company previously) for Cartoon Network, which used a browser side plug-in with a proprietary messaging format to manipulate <a href="/wiki/Dynamic_HTML" title="Dynamic HTML">DHTML</a> elements (this system is also owned by 3DO). Upon discovery of early <a href="/wiki/Ajax_(programming)" title="Ajax (programming)">Ajax</a> capabilities, digiGroups, Noosh, and others used frames to pass information into the user browsers' visual field without refreshing a Web application's visual context, realizing real-time rich Web applications using only the standard HTTP, HTML and JavaScript capabilities of Netscape 4.0.5+ and IE 5+. Crockford then found that JavaScript could be used as an object-based messaging format for such a system. The system was sold to <a href="/wiki/Sun_Microsystems" title="Sun Microsystems">Sun Microsystems</a>, <a href="/wiki/Amazon.com" class="mw-redirect" title="Amazon.com">Amazon.com</a> and <a href="/wiki/Electronic_Data_Systems" title="Electronic Data Systems">EDS</a>. The <a rel="nofollow" class="external text" href="http://json.org/">JSON.org</a> Web site was launched in 2002. In December 2005, <a href="/wiki/Yahoo!" title="Yahoo!">Yahoo!</a> began offering some of its <a href="/wiki/Web_service" title="Web service">Web services</a> in JSON.<sup id="cite_ref-yahoo_9-0" class="reference"><a href="#cite_note-yahoo-9">[9]</a></sup></p> 
      <p>JSON was originally intended to be a subset of the <a href="/wiki/JavaScript" title="JavaScript">JavaScript</a> scripting language (specifically, Standard <a href="/wiki/Ecma_International" title="Ecma International">ECMA</a>-262 3rd Edition—December 1999<sup id="cite_ref-10" class="reference"><a href="#cite_note-10">[10]</a></sup><sup class="noprint Inline-Template" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Verifiability" title="Wikipedia:Verifiability"><span title="Quotation needed from source to verify. (January 2017)">need quotation to verify</span></a></i>]</sup>) and is commonly used with Javascript, but it is a <a href="/wiki/Language-independent_specification" title="Language-independent specification">language-independent</a> data format. Code for <a href="/wiki/Parsing" title="Parsing">parsing</a> and generating JSON data is readily available in many <a href="/wiki/Programming_languages" class="mw-redirect" title="Programming languages">programming languages</a>. JSON's Web site lists JSON <a href="/wiki/Language_binding" title="Language binding">libraries</a> by language.</p> 
      <p>Though JSON was originally advertised and believed to be a strict subset of JavaScript and ECMAScript,<sup id="cite_ref-11" class="reference"><a href="#cite_note-11">[11]</a></sup><sup class="noprint Inline-Template" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Verifiability" title="Wikipedia:Verifiability"><span title="The material near this tag failed verification of its source citation(s). (January 2017)">not in citation given</span></a></i>]</sup> it inadvertently allows some unescaped characters in strings that are illegal in JavaScript and ECMAScript strings. See <a href="#Data_portability_issues">Data portability issues</a> below.</p> 
      <p>JSON itself became an ECMA international standard in 2013 as the <i>ECMA-404 standard</i>.<sup id="cite_ref-ecma2013_12-0" class="reference"><a href="#cite_note-ecma2013-12">[12]</a></sup> In the same year <a href="/wiki/Request_for_Comments_(identifier)" class="mw-redirect" title="Request for Comments (identifier)">RFC</a>&nbsp;<a rel="nofollow" class="external text" href="https://tools.ietf.org/html/rfc7158">7158</a> used ECMA-404 as reference. In 2014 <a href="/wiki/Request_for_Comments_(identifier)" class="mw-redirect" title="Request for Comments (identifier)">RFC</a>&nbsp;<a rel="nofollow" class="external text" href="https://tools.ietf.org/html/rfc7159">7159</a> became the main reference for JSON's internet uses (ex. MIME application/json), and obsoletes <a href="/wiki/Request_for_Comments_(identifier)" class="mw-redirect" title="Request for Comments (identifier)">RFC</a>&nbsp;<a rel="nofollow" class="external text" href="https://tools.ietf.org/html/rfc4627">4627</a> and <a href="/wiki/Request_for_Comments_(identifier)" class="mw-redirect" title="Request for Comments (identifier)">RFC</a>&nbsp;<a rel="nofollow" class="external text" href="https://tools.ietf.org/html/rfc7158">7158</a> (but preserving ECMA-262 and ECMA-404 as main references).</p> 
      <h2><span id="Data_types,_syntax_and_example"></span><span class="mw-headline" id="Data_types.2C_syntax_and_example">Data types, syntax and example</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=JSON&amp;action=edit&amp;section=2" title="Edit section: Data types, syntax and example">edit</a><span class="mw-editsection-bracket">]</span></span></h2> 
      <p>JSON's basic data types are:</p> 
      <ul> 
       <li>Number: a signed decimal number that may contain a fractional part and may use exponential <a href="/wiki/E_notation" class="mw-redirect" title="E notation">E notation</a>, but cannot include non-numbers such as <a href="/wiki/NaN" title="NaN">NaN</a>. The format makes no distinction between integer and floating-point. JavaScript uses a <a href="/wiki/Double-precision_floating-point_format" title="Double-precision floating-point format">double-precision floating-point format</a> for all its numeric values, but other languages implementing JSON may encode numbers differently.</li> 
       <li><a href="/wiki/String_(computer_science)" title="String (computer science)">String</a>: a sequence of zero or more <a href="/wiki/Unicode" title="Unicode">Unicode</a> characters. Strings are delimited with double-quotation marks and support a backslash <a href="/wiki/Escape_character" title="Escape character">escaping</a> syntax.</li> 
       <li><a href="/wiki/Boolean_datatype" class="mw-redirect" title="Boolean datatype">Boolean</a>: either of the values <code>true</code> or <code>false</code></li> 
       <li><a href="/wiki/Array_data_structure" title="Array data structure">Array</a>: an <a href="/wiki/List_(abstract_data_type)" title="List (abstract data type)">ordered list</a> of zero or more values, each of which may be of any type. Arrays use <a href="/wiki/Square_bracket" class="mw-redirect" title="Square bracket">square bracket</a> notation and elements are comma-separated.</li> 
       <li>Object: an unordered collection of <a href="/wiki/Attribute%E2%80%93value_pair" title="Attribute–value pair">name–value pairs</a> where the names (also called keys) are strings. Since objects are intended to represent <a href="/wiki/Associative_array" title="Associative array">associative arrays</a>,<sup id="cite_ref-ecma2013_12-1" class="reference"><a href="#cite_note-ecma2013-12">[12]</a></sup> it is recommended, though not required,<sup id="cite_ref-rfc7519_13-0" class="reference"><a href="#cite_note-rfc7519-13">[13]</a></sup> that each key is unique within an object. Objects are delimited with <a href="/wiki/Braces_(punctuation)#Braces" class="mw-redirect" title="Braces (punctuation)">curly brackets</a> and use commas to separate each pair, while within each pair the colon ':' character separates the key or name from its value.</li> 
       <li><code><a href="/wiki/Nullable_type" title="Nullable type">null</a></code>: An empty value, using the word <code>null</code></li> 
      </ul> 
      <p>Limited <a href="/wiki/Whitespace_character" title="Whitespace character">whitespace</a> is allowed and ignored around or between syntactic elements (values and punctuation, but not within a string value). Only four specific characters are considered whitespace for this purpose: space, horizontal tab, line feed, and carriage return. In particular, the <a href="/wiki/Byte_order_mark" title="Byte order mark">byte order mark</a> must not be generated by a conforming implementation (though it may be accepted when parsing JSON). JSON does not provide syntax for comments.</p> 
      <p>Early versions of JSON (such as specified by <a href="/wiki/Request_for_Comments_(identifier)" class="mw-redirect" title="Request for Comments (identifier)">RFC</a>&nbsp;<a rel="nofollow" class="external text" href="https://tools.ietf.org/html/rfc4627">4627</a>) required that a valid JSON "document" must consist of only an object or an array type, which could contain other types within them.</p>

      <h3><span class="mw-headline" id="Data_portability_issues">Data portability issues</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=JSON&amp;action=edit&amp;section=4" title="Edit section: Data portability issues">edit</a><span class="mw-editsection-bracket">]</span></span></h3> 
      <p>Although Douglas Crockford originally asserted that JSON is a strict subset of JavaScript, his specification actually allows valid JSON documents that are invalid JavaScript. Specifically, JSON allows the <a href="/wiki/Unicode#Newlines" title="Unicode">Unicode line terminators</a> <span class="nowrap"><span style="font-family: monospace, monospace;">U+2028</span></span> <span class="smallcaps" style="font-variant: small-caps; font-size: smaller;">LINE SEPARATOR</span> and <span class="nowrap"><span style="font-family: monospace, monospace;">U+2029</span></span> <span class="smallcaps" style="font-variant: small-caps; font-size: smaller;">PARAGRAPH SEPARATOR</span> to appear unescaped in quoted strings, while JavaScript does not.<sup id="cite_ref-json-2028_14-0" class="reference"><a href="#cite_note-json-2028-14">[14]</a></sup> This is a consequence of JSON disallowing only "control characters". For maximum portability, these characters should be backslash-escaped. This subtlety is important when generating <a href="/wiki/JSONP" title="JSONP">JSONP</a>.</p> 
      <p>JSON documents can be encoded in <a href="/wiki/UTF-8" title="UTF-8">UTF-8</a>, <a href="/wiki/UTF-16" title="UTF-16">UTF-16</a> or <a href="/wiki/UTF-32" title="UTF-32">UTF-32</a>, the default encoding being UTF-8<sup id="cite_ref-rfc7519_13-1" class="reference"><a href="#cite_note-rfc7519-13">[13]</a></sup> (I-JSON only supports UTF-8). These encodings support the full Unicode character set, including those characters outside the <a href="/wiki/Plane_(Unicode)#Basic_Multilingual_Plane" title="Plane (Unicode)">Basic Multilingual Plane</a> (U+10000 to U+10FFFF). However, if escaped, those characters must be written using <a href="/wiki/UTF-16" title="UTF-16">UTF-16</a> surrogate pairs, a detail missed by some JSON parsers. For example, to include the <a href="/wiki/Emoji" title="Emoji">Emoji</a> character <span class="nowrap"><span style="font-family: monospace, monospace;">U+1F602</span></span> <span style="font-size:125%">?</span> <span class="smallcaps" style="font-variant: small-caps; font-size: smaller;">FACE WITH TEARS OF JOY</span> in JSON:</p> 
      <div class="mw-highlight mw-content-ltr" dir="ltr"> 
       <pre>
<span class="p">{</span> <span class="s2">"face"</span><span class="o">:</span> <span class="s2">"?"</span> <span class="p">}</span>
<span class="c1">// or</span>
<span class="p">{</span> <span class="s2">"face"</span><span class="o">:</span> <span class="s2">"\uD83D\uDE02"</span> <span class="p">}</span>
</pre>
      </div> 
      <p>Numbers in JSON are agnostic with regard to their representation within programming languages. No differentiation is made between an integer and floating-point value: some implementations may treat <code>42</code>, <code>42.0</code>, and <code>4.2E+1</code> as the same number while others may not. Furthermore, no requirements are made regarding implementation issues such as <a href="/wiki/Arithmetic_overflow" class="mw-redirect" title="Arithmetic overflow">overflow</a>, <a href="/wiki/Arithmetic_underflow" title="Arithmetic underflow">underflow</a>, loss of precision, or rounding. Additionally, JSON says nothing about the treatment of <a href="/wiki/Signed_zeros" class="mw-redirect" title="Signed zeros">signed zeros</a>: whether <code>0.0</code> is distinct from <code>-0.0</code>. Most implementations that use the <a href="/wiki/IEEE_754" title="IEEE 754">IEEE 754</a> floating-point standard, including JavaScript, preserve signed zeros; but not all JSON implementations may do so.</p> 
      <h3><span class="mw-headline" id="Using_JSON_in_JavaScript">Using JSON in JavaScript</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=JSON&amp;action=edit&amp;section=5" title="Edit section: Using JSON in JavaScript">edit</a><span class="mw-editsection-bracket">]</span></span></h3> 
      <p>As of 2017, all major browsers support at least the fifth edition <a href="/wiki/ECMAScript" title="ECMAScript">ECMAScript</a> which provides<sup id="cite_ref-15" class="reference"><a href="#cite_note-15">[15]</a></sup> a safe and fast method of decoding JSON:</p> 
      <div class="mw-highlight mw-content-ltr" dir="ltr"> 
       <pre>
<span class="kd">var</span> <span class="nx">p</span> <span class="o">=</span> <span class="nx">JSON</span><span class="p">.</span><span class="nx">parse</span><span class="p">(</span><span class="nx">json_string</span><span class="p">);</span>
</pre>
      </div> 
      <h3><span class="mw-headline" id="Unsupported_native_data_types">Unsupported native data types</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=JSON&amp;action=edit&amp;section=6" title="Edit section: Unsupported native data types">edit</a><span class="mw-editsection-bracket">]</span></span></h3> 
      <p><a href="/wiki/JavaScript_syntax" title="JavaScript syntax">JavaScript syntax</a> defines several native data types that are not included in the JSON standard:<sup id="cite_ref-rfc7519_13-2" class="reference"><a href="#cite_note-rfc7519-13">[13]</a></sup> Map, Set, Date, Error, Regular Expression, Function, Promise, and <code>undefined</code>.<sup id="cite_ref-undefinedType_16-0" class="reference"><a href="#cite_note-undefinedType-16">[note 1]</a></sup> These JavaScript data types must be represented by some other data format, with the programs on both ends agreeing on how to convert between the types. As of 2011<sup class="plainlinks noprint asof-tag update" style="display:none;"><a class="external text" href="//en.wikipedia.org/w/index.php?title=JSON&amp;action=edit">[update]</a></sup>, there are some de facto standards, <i>e.g.</i>, converting from Date to String, but none universally recognized.<sup id="cite_ref-17" class="reference"><a href="#cite_note-17">[16]</a></sup><sup id="cite_ref-18" class="reference"><a href="#cite_note-18">[17]</a></sup> Other languages may have a different set of native types that must be <a href="/wiki/Serialization" title="Serialization">serialized</a> carefully to deal with this type of conversion.</p> 
      <h2><span class="mw-headline" id="Schema_and_metadata">Schema and metadata</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=JSON&amp;action=edit&amp;section=7" title="Edit section: Schema and metadata">edit</a><span class="mw-editsection-bracket">]</span></span></h2> 
      <h3><span class="mw-headline" id="JSON_Schema">JSON Schema</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=JSON&amp;action=edit&amp;section=8" title="Edit section: JSON Schema">edit</a><span class="mw-editsection-bracket">]</span></span></h3> 
      <p><span id="Schema"></span> JSON Schema<sup id="cite_ref-19" class="reference"><a href="#cite_note-19">[18]</a></sup> specifies a JSON-based format to define the structure of JSON data for validation, documentation, and interaction control. It provides a contract for the JSON data required by a given application, and how that data can be modified.</p> 
      <p>JSON Schema is based on the concepts from <a href="/wiki/XML_Schema_(W3C)" title="XML Schema (W3C)">XML Schema</a> (XSD), but is JSON-based. As in XSD, the same serialization/deserialization tools can be used both for the schema and data; and is self-describing. It is described in an <a href="/wiki/Internet_Draft" title="Internet Draft">Internet Draft</a> currently in its 6th draft, which was released on April 15, 2017<sup id="cite_ref-20" class="reference"><a href="#cite_note-20">[19]</a></sup>. Draft 4 expired on August 4, 2013,<sup id="cite_ref-21" class="reference"><a href="#cite_note-21">[20]</a></sup> but continued to be used in the lapse of more than 3 years between its expiration and the release of Draft 5. There are several validators available for different programming languages,<sup id="cite_ref-22" class="reference"><a href="#cite_note-22">[21]</a></sup> each with varying levels of conformance.</p> 
      <p>There is no standard file extension, but some have suggested <code>.schema.json</code>.<sup id="cite_ref-23" class="reference"><a href="#cite_note-23">[22]</a></sup></p> 
      <p>Example JSON Schema (draft 4):</p> 
      <div class="mw-highlight mw-content-ltr" dir="ltr"> 
       <pre>
<span class="p">{</span>
  <span class="nt">"$schema"</span><span class="p">:</span> <span class="s2">"http://json-schema.org/schema#"</span><span class="p">,</span>
  <span class="nt">"title"</span><span class="p">:</span> <span class="s2">"Product"</span><span class="p">,</span>
  <span class="nt">"type"</span><span class="p">:</span> <span class="s2">"object"</span><span class="p">,</span>
  <span class="nt">"required"</span><span class="p">:</span> <span class="p">[</span><span class="s2">"id"</span><span class="p">,</span> <span class="s2">"name"</span><span class="p">,</span> <span class="s2">"price"</span><span class="p">],</span>
  <span class="nt">"properties"</span><span class="p">:</span> <span class="p">{</span>
    <span class="nt">"id"</span><span class="p">:</span> <span class="p">{</span>
      <span class="nt">"type"</span><span class="p">:</span> <span class="s2">"number"</span><span class="p">,</span>
      <span class="nt">"description"</span><span class="p">:</span> <span class="s2">"Product identifier"</span>
    <span class="p">},</span>
    <span class="nt">"name"</span><span class="p">:</span> <span class="p">{</span>
      <span class="nt">"type"</span><span class="p">:</span> <span class="s2">"string"</span><span class="p">,</span>
      <span class="nt">"description"</span><span class="p">:</span> <span class="s2">"Name of the product"</span>
    <span class="p">},</span>
    <span class="nt">"price"</span><span class="p">:</span> <span class="p">{</span>
      <span class="nt">"type"</span><span class="p">:</span> <span class="s2">"number"</span><span class="p">,</span>
      <span class="nt">"minimum"</span><span class="p">:</span> <span class="mi">0</span>
    <span class="p">},</span>
    <span class="nt">"tags"</span><span class="p">:</span> <span class="p">{</span>
      <span class="nt">"type"</span><span class="p">:</span> <span class="s2">"array"</span><span class="p">,</span>
      <span class="nt">"items"</span><span class="p">:</span> <span class="p">{</span>
        <span class="nt">"type"</span><span class="p">:</span> <span class="s2">"string"</span>
      <span class="p">}</span>
    <span class="p">},</span>
    <span class="nt">"stock"</span><span class="p">:</span> <span class="p">{</span>
      <span class="nt">"type"</span><span class="p">:</span> <span class="s2">"object"</span><span class="p">,</span>
      <span class="nt">"properties"</span><span class="p">:</span> <span class="p">{</span>
        <span class="nt">"warehouse"</span><span class="p">:</span> <span class="p">{</span>
          <span class="nt">"type"</span><span class="p">:</span> <span class="s2">"number"</span>
        <span class="p">},</span>
        <span class="nt">"retail"</span><span class="p">:</span> <span class="p">{</span>
          <span class="nt">"type"</span><span class="p">:</span> <span class="s2">"number"</span>
        <span class="p">}</span>
      <span class="p">}</span>
    <span class="p">}</span>
  <span class="p">}</span>
<span class="p">}</span>
</pre>
      </div> 
      <p>The JSON Schema above can be used to test the validity of the JSON code below:</p> 
      <div class="mw-highlight mw-content-ltr" dir="ltr">

      <h2><span class="mw-headline" id="MIME_type">MIME type</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=JSON&amp;action=edit&amp;section=9" title="Edit section: MIME type">edit</a><span class="mw-editsection-bracket">]</span></span></h2> 
      <p>The official <a href="/wiki/MIME_type" class="mw-redirect" title="MIME type">MIME type</a> for JSON text is "<code>application/json</code>",<sup id="cite_ref-24" class="reference"><a href="#cite_note-24">[23]</a></sup> and most modern implementations have adopted this.</p> 
      <p>The (unofficial) MIME type "<code>text/json</code>" or the content-type "<code>text/javascript</code>" also get legacy support by many service providers, browsers, servers, web applications, libraries, frameworks, and APIs. Notable examples include the Google Search API,<sup id="cite_ref-faraday-stack-pull-2_25-0" class="reference"><a href="#cite_note-faraday-stack-pull-2-25">[24]</a></sup> Yahoo!,<sup id="cite_ref-faraday-stack-pull-2_25-1" class="reference"><a href="#cite_note-faraday-stack-pull-2-25">[24]</a></sup><sup id="cite_ref-26" class="reference"><a href="#cite_note-26">[25]</a></sup> Flickr,<sup id="cite_ref-faraday-stack-pull-2_25-2" class="reference"><a href="#cite_note-faraday-stack-pull-2-25">[24]</a></sup> Facebook API,<sup id="cite_ref-27" class="reference"><a href="#cite_note-27">[26]</a></sup> <a href="/wiki/Lift_(web_framework)" title="Lift (web framework)">Lift framework</a>,<sup id="cite_ref-28" class="reference"><a href="#cite_note-28">[27]</a></sup> Dojo Toolkit 0.4,<sup id="cite_ref-29" class="reference"><a href="#cite_note-29">[28]</a></sup> etc.</p> 
      <h2><span class="mw-headline" id="Applications">Applications</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=JSON&amp;action=edit&amp;section=10" title="Edit section: Applications">edit</a><span class="mw-editsection-bracket">]</span></span></h2> 
      <h3><span class="mw-headline" id="JSON-RPC">JSON-RPC</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=JSON&amp;action=edit&amp;section=11" title="Edit section: JSON-RPC">edit</a><span class="mw-editsection-bracket">]</span></span></h3> 
      <p><a href="/wiki/JSON-RPC" title="JSON-RPC">JSON-RPC</a> is a <a href="/wiki/Remote_procedure_call" title="Remote procedure call">remote procedure call</a> (RPC) protocol built on JSON, as a replacement for <a href="/wiki/XML-RPC" title="XML-RPC">XML-RPC</a> or <a href="/wiki/SOAP" title="SOAP">SOAP</a>. It is a simple protocol that defines only a handful of data types and commands. JSON-RPC lets a system send notifications (information to the server that does not require a response) and multiple calls to the server that can be answered out of order. Example of a JSON-RPC 2.0 request and response using positional parameters.</p> 
      <div class="mw-highlight mw-content-ltr" dir="ltr"> 
       <pre>
<span class="o">--&gt;</span> <span class="p">{</span><span class="s2">"jsonrpc"</span><span class="o">:</span> <span class="s2">"2.0"</span><span class="p">,</span> <span class="s2">"method"</span><span class="o">:</span> <span class="s2">"subtract"</span><span class="p">,</span> <span class="s2">"params"</span><span class="o">:</span> <span class="p">[</span><span class="mi">42</span><span class="p">,</span> <span class="mi">23</span><span class="p">],</span> <span class="s2">"id"</span><span class="o">:</span> <span class="mi">1</span><span class="p">}</span>
<span class="o">&lt;--</span> <span class="p">{</span><span class="s2">"jsonrpc"</span><span class="o">:</span> <span class="s2">"2.0"</span><span class="p">,</span> <span class="s2">"result"</span><span class="o">:</span> <span class="mi">19</span><span class="p">,</span> <span class="s2">"id"</span><span class="o">:</span> <span class="mi">1</span><span class="p">}</span>
</pre>
      </div> 
      <h3><span class="mw-headline" id="AJAJ">AJAJ</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=JSON&amp;action=edit&amp;section=12" title="Edit section: AJAJ">edit</a><span class="mw-editsection-bracket">]</span></span></h3> 
      <p><a href="/wiki/Asynchronous_JavaScript_and_JSON" class="mw-redirect" title="Asynchronous JavaScript and JSON">Asynchronous JavaScript and JSON</a> (or AJAJ) refers to the same <a href="/wiki/Dynamic_web_page" title="Dynamic web page">dynamic web page</a> methodology as <a href="/wiki/Ajax_(programming)" title="Ajax (programming)">Ajax</a>, but instead of <a href="/wiki/XML" title="XML">XML</a>, JSON is the data format. AJAJ is a web development technique that provides for the ability of a <a href="/wiki/Webpage" class="mw-redirect" title="Webpage">webpage</a> to request new data after it has loaded into the <a href="/wiki/Web_browser" title="Web browser">web browser</a>. Typically it renders new data from the server in response to user actions on that webpage. For example, what the user types into a <a href="/wiki/Text_box" title="Text box">search box</a>, <a href="/wiki/Client-side_scripting" class="mw-redirect" title="Client-side scripting">client-side code</a> then sends to the server, which immediately responds with a <a href="/wiki/Drop-down_list" title="Drop-down list">drop-down list</a> of matching <a href="/wiki/Database" title="Database">database</a> items.</p> 
      <p>The following JavaScript code is an example of a <a href="/wiki/Client-side" title="Client-side">client</a> using <a href="/wiki/XMLHttpRequest" title="XMLHttpRequest">XMLHttpRequest</a> to request data in JSON format from a server. (The <a href="/wiki/Server-side" title="Server-side">server-side</a> programming is omitted; it must be set up to service requests to the <code>url</code> containing a JSON-formatted string.)</p> 
      <div class="mw-highlight mw-content-ltr" dir="ltr"> 
       <pre>
<span class="kd">var</span> <span class="nx">my_JSON_object</span><span class="p">;</span>
<span class="kd">var</span> <span class="nx">http_request</span> <span class="o">=</span> <span class="k">new</span> <span class="nx">XMLHttpRequest</span><span class="p">();</span>
<span class="nx">http_request</span><span class="p">.</span><span class="nx">open</span><span class="p">(</span><span class="s2">"GET"</span><span class="p">,</span> <span class="nx">url</span><span class="p">,</span> <span class="kc">true</span><span class="p">);</span>
<span class="nx">http_request</span><span class="p">.</span><span class="nx">responseType</span> <span class="o">=</span> <span class="s2">"json"</span><span class="p">;</span>
<span class="nx">http_request</span><span class="p">.</span><span class="nx">onreadystatechange</span> <span class="o">=</span> <span class="kd">function</span> <span class="p">()</span> <span class="p">{</span>
  <span class="kd">var</span> <span class="nx">done</span> <span class="o">=</span> <span class="mi">4</span><span class="p">,</span> <span class="nx">ok</span> <span class="o">=</span> <span class="mi">200</span><span class="p">;</span>
  <span class="k">if</span> <span class="p">(</span><span class="nx">http_request</span><span class="p">.</span><span class="nx">readyState</span> <span class="o">===</span> <span class="nx">done</span> <span class="o">&amp;&amp;</span> <span class="nx">http_request</span><span class="p">.</span><span class="nx">status</span> <span class="o">===</span> <span class="nx">ok</span><span class="p">)</span> <span class="p">{</span>
    <span class="nx">my_JSON_object</span> <span class="o">=</span> <span class="nx">http_request</span><span class="p">.</span><span class="nx">response</span><span class="p">;</span>
  <span class="p">}</span>
<span class="p">};</span>
<span class="nx">http_request</span><span class="p">.</span><span class="nx">send</span><span class="p">(</span><span class="kc">null</span><span class="p">);</span>

      <h2><span class="mw-headline" id="Security_considerations">Security considerations</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=JSON&amp;action=edit&amp;section=13" title="Edit section: Security considerations">edit</a><span class="mw-editsection-bracket">]</span></span></h2> 
      <p>JSON is intended as a <a href="/wiki/Data_serialization" class="mw-redirect" title="Data serialization">data serialization</a> format. However, its design as a non-strict subset of JavaScript can lead to the misconception that it is safe to pass JSON strings to the JavaScript <code class="mw-highlight" id="" style="" dir="ltr"><span class="nb">eval</span><span class="p">()</span></code> function. This is not safe, due to the fact that certain valid JSON strings are actually not valid JavaScript code.<sup id="cite_ref-Magnus_Holm_30-0" class="reference"><a href="#cite_note-Magnus_Holm-30">[29]</a></sup></p> 
      <p>To avoid the many pitfalls caused by executing arbitrary code from the internet, a new function, <code class="mw-highlight" id="" style="" dir="ltr"><span class="nx">JSON</span><span class="p">.</span><span class="nx">parse</span><span class="p">()</span></code> was first added to the fifth edition of ECMAScript<sup id="cite_ref-31" class="reference"><a href="#cite_note-31">[30]</a></sup>, which as of 2017 is supported by all major browsers. For non-supported browsers, an <a rel="nofollow" class="external text" href="https://github.com/douglascrockford/JSON-js/blob/master/json2.js">API-compatible JavaScript library</a> is provided by <a href="/wiki/Douglas_Crockford" title="Douglas Crockford">Douglas Crockford</a>.</p> 
      <h2><span class="mw-headline" id="Vulnerabilities_in_specific_JSON_parsers">Vulnerabilities in specific JSON parsers</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=JSON&amp;action=edit&amp;section=14" title="Edit section: Vulnerabilities in specific JSON parsers">edit</a><span class="mw-editsection-bracket">]</span></span></h2> 
      <p>Various JSON parser implementations have suffered from <a href="/wiki/Denial-of-service_attack" title="Denial-of-service attack">denial-of-service attack</a> and <a href="/wiki/Mass_assignment_vulnerability" title="Mass assignment vulnerability">mass assignment vulnerability</a>.<sup id="cite_ref-32" class="reference"><a href="#cite_note-32">[31]</a></sup><sup id="cite_ref-33" class="reference"><a href="#cite_note-33">[32]</a></sup></p> 
      <h2><span class="mw-headline" id="Object_references">Object references</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=JSON&amp;action=edit&amp;section=15" title="Edit section: Object references">edit</a><span class="mw-editsection-bracket">]</span></span></h2> 
      <p>The JSON standard does not support object <a href="/wiki/Reference_(computer_science)" title="Reference (computer science)">references</a>, but an <a href="/wiki/IETF" class="mw-redirect" title="IETF">IETF</a> draft standard for JSON-based object references exists.<sup id="cite_ref-34" class="reference"><a href="#cite_note-34">[33]</a></sup> The <a href="/wiki/Dojo_Toolkit" title="Dojo Toolkit">Dojo Toolkit</a> supports object references using standard JSON; specifically, the <code>dojox.json.ref</code> module provides support for several forms of referencing including <a href="/wiki/Circular_reference" title="Circular reference">circular</a>, multiple, inter-message, and <a href="/wiki/Lazy_evaluation" title="Lazy evaluation">lazy</a> referencing.<sup id="cite_ref-35" class="reference"><a href="#cite_note-35">[34]</a></sup><sup id="cite_ref-36" class="reference"><a href="#cite_note-36">[35]</a></sup><sup id="cite_ref-37" class="reference"><a href="#cite_note-37">[36]</a></sup> Alternatively, non-standard solutions exist such as the use of Mozilla JavaScript Sharp Variables. However this functionality became obsolete with JavaScript 1.8.5 and was removed in Firefox version 12.<sup id="cite_ref-38" class="reference"><a href="#cite_note-38">[37]</a></sup></p> 
      <h2><span class="mw-headline" id="Comparison_with_other_formats">Comparison with other formats</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=JSON&amp;action=edit&amp;section=16" title="Edit section: Comparison with other formats">edit</a><span class="mw-editsection-bracket">]</span></span></h2> 
      <div role="note" class="hatnote navigation-not-searchable">
       See also: 
       <a href="/wiki/Comparison_of_data_serialization_formats" title="Comparison of data serialization formats">Comparison of data serialization formats</a>
      </div> 
      <p>JSON is promoted as a low-overhead alternative to XML as both of these formats have widespread support for creation, reading, and decoding in the real-world situations where they are commonly used.<sup id="cite_ref-39" class="reference"><a href="#cite_note-39">[38]</a></sup> Apart from XML, examples could include <a href="/wiki/OGDL" title="OGDL">OGDL</a>, <a href="/wiki/YAML" title="YAML">YAML</a> and <a href="/wiki/Comma-separated_values" title="Comma-separated values">CSV</a>. Also, <a href="/wiki/Protocol_Buffers" title="Protocol Buffers">Google Protocol Buffers</a> can fill this role, although it is not a data interchange language.</p> 
      <h3><span class="mw-headline" id="YAML">YAML</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=JSON&amp;action=edit&amp;section=17" title="Edit section: YAML">edit</a><span class="mw-editsection-bracket">]</span></span></h3> 
      <p><a href="/wiki/YAML" title="YAML">YAML</a> version 1.2 is a superset of JSON; prior versions were "not strictly compatible". For example, escaping a slash (/) with a backslash (\) is valid in JSON, but was not valid in YAML. (This is common practice when injecting JSON into HTML to protect against <a href="/wiki/Cross-site_scripting" title="Cross-site scripting">cross-site scripting attacks</a>.) Nonetheless, many YAML parsers can natively parse the output from many JSON encoders.<sup id="cite_ref-YAML_Version_1.2_40-0" class="reference"><a href="#cite_note-YAML_Version_1.2-40">[39]</a></sup></p> 
      <h3><span class="mw-headline" id="XML">XML</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=JSON&amp;action=edit&amp;section=18" title="Edit section: XML">edit</a><span class="mw-editsection-bracket">]</span></span></h3> 
      <p><a href="/wiki/XML" title="XML">XML</a> has been used to describe structured data and to serialize objects. Various XML-based protocols exist to represent the same kind of data structures as JSON for the same kind of data interchange purposes. Data can be encoded in XML several ways. The most expansive form using tag pairs results in a much larger representation than JSON, but if data is stored in attributes and 'short tag' form where the closing tag is replaced with '/&gt;', the representation is often about the same size as JSON or just a little larger. If the data is compressed using an algorithm like <a href="/wiki/Gzip" title="Gzip">gzip</a>, there is little difference because compression is good at saving space when a pattern is repeated.</p> 
      <p>XML also has the concept of <a href="/wiki/XML_schema" title="XML schema">schema</a>. This permits strong typing, user-defined types, predefined tags, and formal structure, allowing for formal validation of an XML stream in a portable way. There is, however, an <a href="/wiki/IETF" class="mw-redirect" title="IETF">IETF</a> draft proposal for a schema system for JSON.<sup id="cite_ref-41" class="reference"><a href="#cite_note-41">[40]</a></sup></p> 
      <p>XML supports comments, but JSON does not.<sup id="cite_ref-Saternos45_42-0" class="reference"><a href="#cite_note-Saternos45-42">[41]</a></sup></p>

      <p>The XML encoding <i>may</i> therefore be comparable in length to the equivalent JSON encoding. A wide range of XML processing technologies exist, from the <a href="/wiki/Document_Object_Model" title="Document Object Model">Document Object Model</a> to <a href="/wiki/XPath" title="XPath">XPath</a> and <a href="/wiki/XSLT" title="XSLT">XSLT</a>. XML can also be styled for immediate display using <a href="/wiki/Cascading_Style_Sheets" title="Cascading Style Sheets">CSS</a>. <a href="/wiki/XHTML" title="XHTML">XHTML</a> is a form of XML so that elements can be passed in this form ready for direct insertion into webpages using client-side scripting.</p>

      <h2><span class="mw-headline" id="Notes">Notes</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=JSON&amp;action=edit&amp;section=24" title="Edit section: Notes">edit</a><span class="mw-editsection-bracket">]</span></span></h2> 
      <div class="reflist" style="list-style-type: decimal;"> 
       <div class="mw-references-wrap"> 
        <ol class="references"> 
         <li id="cite_note-undefinedType-16"><span class="mw-cite-backlink"><b><a href="#cite_ref-undefinedType_16-0">^</a></b></span> <span class="reference-text">The <code>undefined</code> type was left out of the JSON standard, and one finds suggestions that <code>null</code> be used instead. In fact, the current standard says that for a sparse array such as:</span> 
          <div class="mw-highlight mw-content-ltr" dir="ltr"> 
           <pre>
<span class="reference-text"> <span class="kd">var</span> <span class="nx">v</span> <span class="o">=</span> <span class="p">[</span><span class="mi">0</span><span class="p">];</span>
 <span class="nx">v</span><span class="p">[</span><span class="mi">3</span><span class="p">]</span> <span class="o">=</span> <span class="mi">3</span><span class="p">;</span>
</span>
</pre>
          </div> <p>which behaves in JavaScript as if it were:</p> 
          <div class="mw-highlight mw-content-ltr" dir="ltr"> 
           <pre>
 <span class="kd">var</span> <span class="nx">vx</span> <span class="o">=</span> <span class="p">[</span><span class="mi">0</span><span class="p">,</span> <span class="kc">undefined</span><span class="p">,</span> <span class="kc">undefined</span><span class="p">,</span> <span class="mi">3</span><span class="p">];</span>
</pre>
          </div> <p>with the <code>undefined</code> entries being only implicit rather than explicit, should translate to JSON as if it were:</p> 
          <div class="mw-highlight mw-content-ltr" dir="ltr"> 
           <pre>
 <span class="kd">var</span> <span class="nx">vx</span> <span class="o">=</span> <span class="p">[</span><span class="mi">0</span><span class="p">,</span> <span class="kc">null</span><span class="p">,</span> <span class="kc">null</span><span class="p">,</span> <span class="mi">3</span><span class="p">];</span>
</pre>
          </div> <p>with explicit <code>null</code> fillers for the undefined entries.</p> <p>Furthermore, in JavaScript <code>{a: undefined}</code> often behaves the same as <code>{}</code>. Both translate as <code>"{}"</code> in JSON. However <code>undefined</code> as an explicit property value does have use in JavaScript inheritance situations such as:</p> 
          <div class="mw-highlight mw-content-ltr" dir="ltr"> 
           <pre>
 <span class="kd">var</span> <span class="nx">x</span> <span class="o">=</span> <span class="p">{</span><span class="nx">a</span><span class="o">:</span> <span class="mi">1</span><span class="p">};</span>
 <span class="kd">var</span> <span class="nx">xi</span> <span class="o">=</span> <span class="nb">Object</span><span class="p">.</span><span class="nx">create</span><span class="p">(</span><span class="nx">x</span><span class="p">);</span>
 <span class="nx">xi</span><span class="p">.</span><span class="nx">a</span> <span class="o">=</span> <span class="kc">undefined</span><span class="p">;</span>
</pre>
          </div> <p>where the inheritance of <code>x</code>'s property <code>a</code> is overridden in <code>xi</code> and makes it pretty much behave as if nothing was inherited. <code>JSON.stringify</code> itself ignores inherited values - it only translates the enumerable own properties as given by <code>Object.keys(y)</code>. The default stringification, while not encoding inheritance, can (except for <code>undefined</code> values) encode enough of an object to reconstruct it in an environment that knows what inheritance it should have. To encode JavaScript objects that contain explicit <code>undefined</code> values a convention for representing <code>undefined</code> must be established, such as mapping it to the string <code>"UNDEFINED"</code>. One can then pass <code>JSON.stringify</code> the optional <code>replacer</code> argument to translate with this convention:</p> 
          <div class="mw-highlight mw-content-ltr" dir="ltr"> 
           <pre>
 <span class="kd">var</span> <span class="nx">y</span> <span class="o">=</span> <span class="p">{</span><span class="nx">a</span><span class="o">:</span> <span class="kc">undefined</span><span class="p">};</span>
 <span class="kd">var</span> <span class="nx">ys</span> <span class="o">=</span> <span class="nx">JSON</span><span class="p">.</span><span class="nx">stringify</span><span class="p">(</span><span class="nx">y</span><span class="p">,</span>
  <span class="kd">function</span> <span class="p">(</span><span class="nx">k</span><span class="p">,</span> <span class="nx">v</span><span class="p">){</span><span class="k">return</span> <span class="p">(</span><span class="nx">v</span> <span class="o">===</span> <span class="kc">undefined</span><span class="p">)</span> <span class="o">?</span> <span class="s2">"UNDEFINED"</span> <span class="o">:</span> <span class="nx">v</span><span class="p">});</span>
</pre>
          </div> <p>Converting this JSON back into JavaScript is not as straightforward. While <code>JSON.parse</code> can take an optional <code>reviver</code> argument that is, essentially, the inverse of a <code>replacer</code>, it can't be used in this situation. If that function returns <code>undefined</code>, the <code>JSON.parse</code> logic interprets this to mean to not define a property rather than define one with a <code>undefined</code> value. Instead one has to explicitly post process the result from <code>JSON.parse</code> replacing each <code>"UNDEFINED"</code> with <code>undefined</code>.</p> </li> 
        </ol> 
       </div>
