       </svg></a>jsdom</h1> 
     <p>jsdom is a pure-JavaScript implementation of many web standards, notably the WHATWG <a href="https://dom.spec.whatwg.org/">DOM</a> and <a href="https://html.spec.whatwg.org/multipage/">HTML</a> Standards, for use with Node.js. In general, the goal of the project is to emulate enough of a subset of a web browser to be useful for testing and scraping real-world web applications.</p> 
     <p>The latest versions of jsdom require Node.js v6 or newer. (Versions of jsdom below v10 still work with Node.js v4, but are unsupported.)</p> 
     <p>As of v10, jsdom has a new API (documented below). The old API is still supported for now; <a href="https://github.com/tmpvar/jsdom/blob/HEAD/lib/old-api.md">see its documentation</a> for details.</p>

       </svg></a>Loading subresources</h3> 
     <p>By default, jsdom will not load any subresources such as scripts, stylesheets, images, or iframes. If you'd like jsdom to load such resources, you can pass the <code>resources: "usable"</code> option, which will load all usable resources. Those are:</p> 
     <ul> 
      <li>Frames and iframes, via <code>&lt;frame&gt;</code> and <code>&lt;iframe&gt;</code></li> 
      <li>Stylesheets, via <code>&lt;link rel="stylesheet"&gt;</code></li> 
      <li>Scripts, via <code>&lt;script&gt;</code>, but only if <code>runScripts: "dangerously"</code> is also set</li> 
      <li>Images, via <code>&lt;img&gt;</code>, but only if the <code>canvas</code> (or <code>canvas-prebuilt</code>) npm package is also installed (see "Canvas Support" below)</li> 
     </ul> 
     <p>In the future we plan to offer more customization of resource loading via this option, but for now the default and the <code>"usable"</code> option are the two modes offered.</p>

     <ul> 
      <li>The <code>url</code> and <code>contentType</code> options cannot be provided.</li> 
      <li>The <code>referrer</code> option is used as the HTTP <code>Referer</code> request header of the initial request.</li> 
      <li>The <code>userAgent</code> option is used as the HTTP <code>User-Agent</code> request header of any requests.</li>

     </ul>

     <p>jsdom includes support for using the <a href="https://www.npmjs.com/package/canvas"><code>canvas</code></a> or <a href="https://npmjs.org/package/canvas-prebuilt"><code>canvas-prebuilt</code></a> package to extend any <code>&lt;canvas&gt;</code> elements with the canvas API. To make this work, you need to include <code>canvas</code> as a dependency in your project, as a peer of <code>jsdom</code>. If jsdom can find the <code>canvas</code> package, it will use it, but if it's not present, then <code>&lt;canvas&gt;</code> elements will behave like <code>&lt;div&gt;</code>s.</p> 
     <h3><a id="user-content-encoding-sniffing" class="deep-link" href="#encoding-sniffing" aria-hidden="true">
       <svg aria-hidden="true" class="deep-link-icon" height="16" version="1.1" viewbox="0 0 16 16" width="16">
        <path d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path>
       </svg></a>Encoding sniffing</h3> 
     <p>In addition to supplying a string, the <code>JSDOM</code> constructor can also be supplied binary data, in the form of a Node.js <a href="https://nodejs.org/docs/latest/api/buffer.html"><code>Buffer</code></a> or a standard JavaScript binary data type like <code>ArrayBuffer</code>, <code>Uint8Array</code>, <code>DataView</code>, etc. When this is done, jsdom will <a href="https://html.spec.whatwg.org/multipage/syntax.html#encoding-sniffing-algorithm">sniff the encoding</a> from the supplied bytes, scanning for <code>&lt;meta charset&gt;</code> tags just like a browser does.</p> 
     <p>This encoding sniffing also applies to <code>JSDOM.fromFile()</code> and <code>JSDOM.fromURL()</code>. In the latter case, just as in a browser, any <code>Content-Type</code> headers sent with the response will take priority.</p> 
     <p>Note that in many cases supplying bytes in this fashion can be better than supplying a string. For example, if you attempt to use Node.js's <code>buffer.toString("utf-8")</code> API, Node.js will not strip any leading BOMs. If you then give this string to jsdom, it will interpret it verbatim, leaving the BOM intact. But jsdom's binary data decoding code will strip leading BOMs, just like a browser; in such cases, supplying <code>buffer</code> directly will give the desired result.</p> 
     <h3><a id="user-content-closing-down-a-jsdom" class="deep-link" href="#closing-down-a-jsdom" aria-hidden="true">
       <svg aria-hidden="true" class="deep-link-icon" height="16" version="1.1" viewbox="0 0 16 16" width="16">
        <path d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path>
       </svg></a>Closing down a jsdom</h3> 
     <p>Timers in the jsdom (set by <code>window.setTimeout()</code> or <code>window.setInterval()</code>) will, by definition, execute code in the future in the context of the window. Since there is no way to execute code in the future without keeping the process alive, outstanding jsdom timers will keep your Node.js process alive. Similarly, since there is no way to execute code in the context of an object without keeping that object alive, outstanding jsdom timers will prevent garbage collection of the window on which they are scheduled.</p> 
     <p>If you want to be sure to shut down a jsdom window, use <code>window.close()</code>, which will terminate all running timers (and also remove any event listeners on the window and document).</p> 
     <h3><a id="user-content-running-jsdom-inside-a-web-browser" class="deep-link" href="#running-jsdom-inside-a-web-browser" aria-hidden="true">
       <svg aria-hidden="true" class="deep-link-icon" height="16" version="1.1" viewbox="0 0 16 16" width="16">
        <path d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path>
       </svg></a>Running jsdom inside a web browser</h3> 
     <p>jsdom has some support for being run inside a web browser, using <a href="https://browserify.org/">browserify</a>. That is, inside a web browser, you can use a browserified jsdom to create an entirely self-contained set of plain JavaScript objects which look and act much like the browser's existing DOM objects, while being entirely independent of them. "Virtual DOM", indeed!</p> 
     <p>jsdom's primary target is still Node.js, and so we use language features that are only present in recent Node.js versions (namely, Node.js v6+). Thus, older browsers will likely not work. (Even transpilation will not help much: we plan to use <code>Proxy</code>s extensively throughout the course of jsdom v10.x.)</p> 
     <p>Notably, jsdom works well inside a web worker. The original contributor, <a href="https://github.com/lawnsea/">@lawnsea</a>, who made this possible, has <a href="https://pdfs.semanticscholar.org/47f0/6bb6607a975500a30e9e52d7c9fbc0034e27.pdf">published a paper</a> about his project which uses this capability.</p> 
     <p>Not everything works perfectly when running jsdom inside a web browser. Sometimes that is because of fundamental limitations (such as not having filesystem access), but sometimes it is simply because we haven't spent enough time making the appropriate small tweaks. Bug reports are certainly welcome.</p>

       </svg></a>Asynchronous script loading</h3> 
     <p>People often have trouble with asynchronous script loading when using jsdom. Many pages loads scripts asynchronously, but there is no way to tell when they're done doing so, and thus when it's a good time to run your code and inspect the resulting DOM structure. This is a fundamental limitation; we cannot predict what scripts on the web page will do, and so cannot tell you when they are done loading more scripts.</p> 
     <p>This can be worked around in a few ways. The best way, if you control the page in question, is to use whatever mechanisms are given by the script loader to detect when loading is done. For example, if you're using a module loader like RequireJS, the code could look like:</p>

        <path d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path>
       </svg></a>Missing features in the new API</h3> 
     <p>Compared to the old jsdom API from v9.x and before, the new API is noticably missing fine-grained control of resource loads. Previous versions of jsdom allowed you to set options that were used when making requests (both for the initial request, in the old equivalent of <code>JSDOM.fromURL()</code>, and for subresource requests). They also allowed you to control which subresources were requested and applied to the main document, so that you could e.g. download stylesheets but not scripts. Finally, they provided a customizable resource loader that let you intercept any outgoing request and fulfill it with a completely synthetic response.</p> 
     <p>None of these features are yet in the new jsdom API, although we are hoping to add them back soon! This requires a decent amount of behind-the-scenes work to implement in a reasonable way, unfortunately.</p> 
     <p>In the meantime, please feel free to use the old jsdom API to get access to this functionality. It is supported and maintained, although it will not be getting new features. The documentation is found in <a href="https://github.com/tmpvar/jsdom/blob/HEAD/lib/old-api.md">lib/old-api.md</a>.</p> 
     <h3><a id="user-content-unimplemented-parts-of-the-web-platform" class="deep-link" href="#unimplemented-parts-of-the-web-platform" aria-hidden="true">
       <svg aria-hidden="true" class="deep-link-icon" height="16" version="1.1" viewbox="0 0 16 16" width="16">
        <path d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path>
       </svg></a>Unimplemented parts of the web platform</h3> 
     <p>Although we enjoy adding new features to jsdom and keeping it up to date with the latest web specs, it has many missing APIs. Please feel free to file an issue for anything missing, but we're a small and busy team, so a pull request might work even better.</p> 
     <p>Beyond just features that we haven't gotten to yet, there are two major features that are currently outside the scope of jsdom. These are:</p> 
     <ul> 
      <li><strong>Navigation</strong>: the ability to change the global object, and all other objects, when clicking a link or assigning <code>location.href</code> or similar.</li> 
      <li><strong>Layout</strong>: the ability to calculate where elements will be visually laid out as a result of CSS, which impacts methods like <code>getBoundingClientRects()</code> or properties like <code>offsetTop</code>.</li> 
     </ul> 
     <p>Currently jsdom has dummy behaviors for some aspects of these features, such as sending a "not implemented" <code>"jsdomError"</code> to the virtual console for navigation, or returning zeros for many layout-related properties. Often you can work around these limitations in your code, e.g. by creating new <code>JSDOM</code> instances for each page you "navigate" to during a crawl, or using <code>Object.defineProperty()</code> to change what various layout-related getters and methods return.</p> 
     <p>Note that other tools in the same space, such as PhantomJS, do support these features. On the wiki, we have a more complete writeup about <a href="https://github.com/tmpvar/jsdom/wiki/jsdom-vs.-PhantomJS">jsdom vs. PhantomJS</a>.</p>
