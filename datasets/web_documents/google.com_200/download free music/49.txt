         <h2>52% of All JavaScript npm Packages Could Have Been Hacked via Weak Credentials</h2> 
         <div class="cz-news-story-title-section"> 
          <div class="cz-news-title-left-area">
            By 
           <h6><a rel="author" href="https://www.bleepingcomputer.com/author/catalin-cimpanu/" class="author"><span itemprop="author" itemscope itemtype="http://schema.org/Person"><span itemprop="name">Catalin Cimpanu</span></span></a></h6> 
          </div> 
          <meta itemprop="articleSection" content="Security"> 
          <div class="cz-news-title-right-area"> 
           <ul> 
            <li class="cz-news-date">June 27, 2017</li> 
            <li class="cz-news-time">07:25 AM</li> 
            <li class="cz-news-comment"><a href="#comment_form">0</a></li> 
           </ul> 
          </div> 
         </div> 
         <div class="articleBody"> 
          <p style="text-align:center"><img alt="npm" height="475" src="https://www.bleepstatic.com/content/posts/2017/06/27/npm.png" width="1250"></p> 
          <p>Tens of thousands of developers using weak credentials to secure their npm accounts inadvertently put more than half of the npm packages (JavaScript libraries and tools) at risk of getting hijacked and used to deploy malicious code to legitimate applications that use them in their build process.</p> 
          <p>npm Inc, the company that runs the npm package manager, has addressed the issue at the start of June by triggering <a href="http://blog.npmjs.org/post/161515829950/credentials-resets" rel="nofollow" target="_blank">password reset operations</a> for all affected users.</p> 
          <p>Initially, there was a lot of confusion about npm Inc's actions, and many believed the organization might have been breached. It was only over the last weekend when we discovered the real reason behind the massive npm account password resets that took place at the start of the month.</p> 
          <h2>Node.js project member at the heart of it all</h2> 
          <p>Behind it all was <a href="https://github.com/ChALkeR" rel="nofollow" target="_blank">Nikita&nbsp;Andreevich Skovoroda</a>, a Moscow-based developer and member of the <a href="https://github.com/nodejs/node#current-project-team-members" rel="nofollow" target="_blank">Node.js CTC (Core Technical Committee)</a>.</p> 
          <p>Speaking to Bleeping Computer, the developer said he performed a series of checks for leaked passwords since May this year, and forwarded the results to npm Inc, who then opted to reset the passwords of all affected users.</p> 
          <p>Skovoroda's interest in exposing leaked passwords didn't come out of the blue. The developer has been warning npm users since 2015 when he first started noticing careless developers leaving passwords for their npm accounts in their code or other places.</p> 
          <p>"I was originally working on some other thing: a crude code scanner over all of npm packages to check how frequent and where are some Node.js APIs being used, for the needs of [Node.js] project," Skovoroda told Bleeping Computer in a private conversation about his initial <a href="https://github.com/ChALkeR/notes/blob/master/Do-not-underestimate-credentials-leaks.md" rel="nofollow" target="_blank">2015 scan</a>.</p> 
          <p>"Back then, I noticed packaged credentials in some npm packages," he added. "That was [an] issue I reported to npm Inc over email back then."</p> 
          <p>The result of his initial report was that npm Inc added a feature to scan for user credentials in npm packages and revoke them.</p> 
          <h2>Researcher finds npm accounts secured with weak credentials</h2> 
          <p>"This [latest] check was started at the beginning of May on the Node.js collaborator summit [...] mostly in free time, with a tool to collect the published credentials from GitHub and check them against npm automatically," Skovoroda says.</p> 
          <p>The developer later improved his scanner to check if npm accounts used common passwords such as "123456," "password," or others; and later to check for cleartext passwords leaked in public breaches such as the ones from Adobe, Last.fm, and others.</p> 
          <p>"I was mostly doing this at a slow pace in my free time until the scan over leaked passwords started giving me lots of valid accounts," Skovoroda says, "I didn't expect that to happen originally."</p> 
          <p>When he finished, the developer says he forwarded the results of his scan to npm's team, who reset passwords for all affected accounts.</p> 
          <h2>Scan results were bad. Really bad.</h2> 
          <p>If a malicious actor would have scanned and discovered these issues before Skovoroda, the attacker could have gained direct "publish access" to 66,876 npm packages, representing nearly 13% of the entire JavaScript npm ecosystem.</p> 
          <p>This is a huge issues because gone are the days when developers load JavaScript libraries by hand in their code. Nowadays, developers primarily use npm to load the libraries they need using npm's dependency management system.</p> 
          <p>This automatic dependency management system means that when a developer loads npm package A, he also loads its dependencies, npm packages B, C, D, E, F, G, etc..</p> 
          <p>While Skovoroda discovered credentials that granted him direct publish access to only 13% of npm packages, through dependencies, an attacker would have been able to spread his malicious code to about 52% of the entire npm ecosystem.</p> 
          <p>Whenever the owner of an indirectly affected package rebuilt his project — for an update or bugfix release — he would have loaded the tainted npm packages into his project, and granted the attacker a means to reach his users.</p> 
          <p>Thanks to Skovoroda's forward thinking, npm Inc was alerted and has now implemented extra checks that will automatically ask users to change their passwords if they use simplistic or previously exposed credentials.</p> 
          <h2>15,495 npm accounts found using weak passwords</h2> 
          <p>Below are some of the other <a href="https://github.com/ChALkeR/notes/blob/master/Gathering-weak-npm-credentials.md" rel="nofollow" target="_blank">results of Skovoroda's scan</a>, some of which readers will find very interesting.</p> 
          <ul> 
           <li>In total, I found <strong>15568 valid credentials for 15495 accounts</strong> since this May.</li> 
           <li>Of those, 15343 accounts have published something (I was targeting only those for everything but npm credentials leaks). The total number of such accounts on npm was 125665, so that gives us <strong>12% of accounts with leaked or weak credentials</strong>.</li> 
           <li>The total number of directly affected packages was <strong>66876 — 13% of the ecosystem</strong>.</li> 
           <li>The total percentage of indirectly affected packages is estimated to be about <strong>52% of the ecosystem</strong> — that is, including packages affected by dependencies.</li> 
           <li>I obtained accounts of <strong>4 users from the top-20 list</strong>.</li> 
           <li>Of the affected accounts, <strong>40 users had more than 10 million downloads/month (each)</strong>. <em>For comparison, <code>express</code> package has 13 million downloads/month atm.</em> <strong>13 users had more than 50 million downloads/month</strong>.</li> 
           <li>One of the passwords with access to publish <a href="https://www.npmjs.com/package/koa" rel="nofollow">koa</a> <strong>was literally «<code>password</code>»</strong>.</li> 
           <li>One of the users directly controlling more than 20 million downloads/month chose to improve their previously revoked leaked password by adding a <code>!</code> to it at the end.</li> 
           <li>One of those 4 users from the top-20 list set their password back to the leaked one shortly after it was reset (so it got reset again).</li> 
           <li>At least one password was significantly inappropriate — to the extent that one wouldn't want that to be linked to them online and could be publicly blamed in that case (i.e. not just a swearword). <a href="https://medium.com/@malcomvetter/offensive-passwords-451371ccd02e" rel="nofollow">Don't use offensive passwords</a> — those could (and in this case were) leaked to the public in cleartext.</li> 
           <li><strong>662 users had password «<code>123456</code>», 168 — «<code>123</code>», 115 — «<code>password</code>»</strong>.</li> 
           <li><strong>1409 users (1%) used their username as their password</strong>, in its original form, without any modifications.</li> 
           <li><strong>10% of users reused their leaked passwords</strong>: 9.7% — directly, and 0.6% — with very minor modifications.</li> 
           <li>Total downloads/month of the unique packages which I got myself publish access to was 1 946 302 172, that's <strong>20% of the total number of d/m</strong> directly.</li> 
          </ul> 
          <p>The frailty of the npm JavaScript ecosystem has been discussed before, last year, when an angry developer pulled his project offline and indirectly <a href="http://news.softpedia.com/news/annoyed-developer-brings-down-thousands-of-javascript-node-js-projects-502075.shtml" rel="nofollow" target="_blank">brought down thousands of other JS tools</a> that had included his package in their dependencies.</p> 
          <p>Similarly, a team of researchers detailed the possibility of <a href="http://news.softpedia.com/news/node-js-package-manager-vulnerable-to-malicious-worm-packages-502216.shtml" rel="nofollow" target="_blank">an npm worm</a> that would also use npm dependencies to spread and infect other npm packages.<br> &nbsp;<br> <span style="font-size:12px"><em>Image credits: npm, Inc., Bleeping Computer</em></span></p>

function cz_strip_tags(input, allowed) {
  allowed = (((allowed || '') + '')
    .toLowerCase()
    .match(/<[a-z][a-z0-9]*>/g) || [])
    .join(''); // making sure the allowed arg is a string containing only tags in lowercase (<a><b><c>)
  var tags = /<\/?([a-z][a-z0-9]*)\b[^>]*>/gi,
    commentsAndPhpTags = /<!--[\s\S]*?-->|<\?(?:php)?[\s\S]*?\?>/gi;
  return input.replace(commentsAndPhpTags, '')
    .replace(tags, function($0, $1) {
      return allowed.indexOf('<' + $1.toLowerCase() + '>') > -1 ? $0 : '';
    });
}
function cz_br2nl(str) {
	var regex = /<br\s*[\/]?>/gi;
    //var pure_str = str.replace(regex,"\n");
    var pure_str = str.replace(regex,"");
    return cz_strip_tags(pure_str,'');
}
