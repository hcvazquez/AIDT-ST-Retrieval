       <div class="term" termid="CM.Glossary.ResponseTimeMonitoring"> 
        <span class="termTitle">Response Time Monitoring</span> 
        <div class="termContent"> 
         <span class="shortTitle">Response time monitoring</span> 
         <span class="shortText">Response time monitoring is testing how fast is your site.</span> 
         <div class="fullText"> 
          <p>Response time monitoring is testing how fast is your site. It might be accessible, but bad performance can still make it unattractive for the clients. The monitoring of the response time allows you to check how the site acts as the time goes on. Do any of the scheduled tasks, like backup, updates etc affect the performance? Does the site works well for the whole world, or some regions of interest could meet a confusing delay? </p> 
          <p><a href="http://www.host-tracker.com">HostTracker</a> provides a possibility to check the reaction of your site. Select <b>Response time check</b> on the main screen:</p> 
          <a href="//www.host-tracker.com/Resource/RespTime0" style="clear: left; margin-bottom: 1em; margin-right: 1em;"> <img alt="Create ContentCheck Task" height="315" width="640" src="//www.host-tracker.com/Resource/RespTime0"></a> 
          <p>Then, select the <b>largest</b> satisfactory response time for the site. Is this value is exceeded, you receive a notification. Also, all the statistics is stored in the check log, and response graph is generated. You may check this data at anytime.</p> 
          <a href="//www.host-tracker.com/Resource/RespTime1" style="clear: left; margin-bottom: 1em; margin-right: 1em;"> <img alt="Create ContentCheck Task" height="315" width="640" src="//www.host-tracker.com/Resource/RespTime1"></a> 
          <p>NOTE! Response time depends on the distance between the datacenter of your site and our checking server. It might be large enough for a remote country. To decrease this effect, select a checking region in advanced options.</p> 
          <a href="//www.host-tracker.com/Resource/RegionSelect" style="clear: left; margin-bottom: 1em; margin-right: 1em;"> <img alt="Create ContentCheck Task" height="315" width="640" src="//www.host-tracker.com/Resource/RegionSelect"></a>

       <div class="term" termid="CM.Blog.Shellshock"> 
        <span class="blogTitle">Shellshock vulnerability online check</span> 
        <div class="blogContent"> 
         <span class="articleDate">2014-06-12</span>
         <div class="shortText">
          <p><img alt="" src="https://www.host-tracker.com/en/Resource/d41b664f-9e11-e711-9173-0003ff7328cc" style="float:left; height:166px; margin:5px; width:250px">Considering the recently discovered Shellshock vulnerability, HostTracker has created a tool for testing it.</p> 
          <div class="clear"></div>
         </div>
         <div class="fullText">
          <p><a href="//www.host-tracker.com/Resource/Shellshock0"><img alt="Check your server for vulnerability" src="//www.host-tracker.com/Resource/Shellshock0" style="height:315px; width:640px"></a></p> 
          <p><b>How does it work?</b></p> 
          <p>It is developed for a Linux server with a web server installed on it. The algorithm is very simple. We consequently generate 4 http requests:</p> 
          <ul> 
           <li>1. Ordinary request</li> 
           <li>2. The request tries, using vulneratility, post a "harmful" cookie which causes 2-seconds delay in respond to <b>our special</b> <a href="http://www.host-tracker.com/Glossary/HTTP">http request</a>.</li> 
           <li>3. The request tries, using vulneratility, post a "harmful" cookie which causes 4-seconds delay in respond to <b>our special</b> http request.</li> 
           <li>4. Same as #3</li> 
          </ul> 
          <p><img alt="" src="https://www.host-tracker.com/en/Resource/c819a921-1f12-e711-9173-0003ff73517a" style="height:198px; width:700px"></p> 
          <p><b>How to understand the result?</b></p> 
          <p>We compare response time for all 4 requests. Three situation are possible:</p> 
          <ul> 
           <li>1. Vulnerability found. We may affirm that if the difference in responses is about 2 seconds for requests without cookie and with 2-second-delay cookie, as well as for requests with 2 and 4-second delay cookie. It means that our request was able to use the vulnerability and set these cookies.</li> 
           <li>2. Vulnerability not found. All the requests have about the same response time. The cookies, likely, were not installed because there is no vulnerability.</li> 
           <li>3. Uncertain situation. If the response time differs widely, without coincidence with preset by cookies delay, we can not say for sure. It could be if the server is under high load. To check this, we use two requests with same cookies (#3 and #4). If the response time for two same checks varies, we make a conclusion that the response time is not affected by cookies. At least, not only by them. So in this case our method can not detect vulnerability</li> 
          </ul> 
          <p><b>Safety of checks</b></p> 
          <p>Our test can not damage your server. The risk consists of appearance of one extra-cookie, which is used only for our requests and can not affect normal work-flow of your site.</p>
