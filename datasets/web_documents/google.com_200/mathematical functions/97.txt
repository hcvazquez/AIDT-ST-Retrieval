   <h2 id="types">Types</h2> 
   <p><a name="types--primitives"></a><a name="1.1"></a> - <a href="#types--primitives">1.1</a> <strong>Primitives</strong>: When you access a primitive type you work directly on its value.</p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>+ `string`
+ `number`
+ `boolean`
+ `null`
+ `undefined`

```javascript
const foo = 1;
let bar = foo;

bar = 9;

console.log(foo, bar); // =&gt; 1, 9
```
</code></pre> 
   </div> 
   <p><a name="types--complex"></a><a name="1.2"></a> - <a href="#types--complex">1.2</a> <strong>Complex</strong>: When you access a complex type you work on a reference to its value.</p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>+ `object`
+ `array`
+ `function`

```javascript
const foo = [1, 2];
const bar = foo;

bar[0] = 9;

console.log(foo[0], bar[0]); // =&gt; 9, 9
```
</code></pre> 
   </div> 
   <p><strong><a href="#table-of-contents">? back to top</a></strong></p> 
   <h2 id="references">References</h2> 
   <p><a name="references--prefer-const"></a><a name="2.1"></a> - <a href="#references--prefer-const">2.1</a> Use <code class="highlighter-rouge">const</code> for all of your references; avoid using <code class="highlighter-rouge">var</code>. eslint: <a href="http://eslint.org/docs/rules/prefer-const.html"><code class="highlighter-rouge">prefer-const</code></a>, <a href="http://eslint.org/docs/rules/no-const-assign.html"><code class="highlighter-rouge">no-const-assign</code></a></p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>&gt; Why? This ensures that you can't reassign your references, which can lead to bugs and difficult to comprehend code.

```javascript
// bad
var a = 1;
var b = 2;

// good
const a = 1;
const b = 2;
```
</code></pre> 
   </div> 
   <p><a name="references--disallow-var"></a><a name="2.2"></a> - <a href="#references--disallow-var">2.2</a> If you must reassign references, use <code class="highlighter-rouge">let</code> instead of <code class="highlighter-rouge">var</code>. eslint: <a href="http://eslint.org/docs/rules/no-var.html"><code class="highlighter-rouge">no-var</code></a> jscs: <a href="http://jscs.info/rule/disallowVar"><code class="highlighter-rouge">disallowVar</code></a></p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>&gt; Why? `let` is block-scoped rather than function-scoped like `var`.

```javascript
// bad
var count = 1;
if (true) {
  count += 1;
}

// good, use the let.
let count = 1;
if (true) {
  count += 1;
}
```
</code></pre> 
   </div> 
   <p><a name="references--block-scope"></a><a name="2.3"></a> - <a href="#references--block-scope">2.3</a> Note that both <code class="highlighter-rouge">let</code> and <code class="highlighter-rouge">const</code> are block-scoped.</p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>```javascript
// const and let only exist in the blocks they are defined in.
{
  let a = 1;
  const b = 1;
}
console.log(a); // ReferenceError
console.log(b); // ReferenceError
```
</code></pre> 
   </div> 
   <p><strong><a href="#table-of-contents">? back to top</a></strong></p> 
   <h2 id="objects">Objects</h2> 
   <p><a name="objects--no-new"></a><a name="3.1"></a> - <a href="#objects--no-new">3.1</a> Use the literal syntax for object creation. eslint: <a href="http://eslint.org/docs/rules/no-new-object.html"><code class="highlighter-rouge">no-new-object</code></a></p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>```javascript
// bad
const item = new Object();

// good
const item = {};
```
</code></pre> 
   </div> 
   <p><a name="es6-computed-properties"></a><a name="3.4"></a> - <a href="#es6-computed-properties">3.2</a> Use computed property names when creating objects with dynamic property names.</p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>&gt; Why? They allow you to define all the properties of an object in one place.

```javascript

function getKey(k) {
  return `a key named ${k}`;
}

// bad
const obj = {
  id: 5,
  name: 'San Francisco',
};
obj[getKey('enabled')] = true;

// good
const obj = {
  id: 5,
  name: 'San Francisco',
  [getKey('enabled')]: true,
};
```
</code></pre> 
   </div> 
   <p><a name="es6-object-shorthand"></a><a name="3.5"></a> - <a href="#es6-object-shorthand">3.3</a> Use object method shorthand. eslint: <a href="http://eslint.org/docs/rules/object-shorthand.html"><code class="highlighter-rouge">object-shorthand</code></a> jscs: <a href="http://jscs.info/rule/requireEnhancedObjectLiterals"><code class="highlighter-rouge">requireEnhancedObjectLiterals</code></a></p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>```javascript
// bad
const atom = {
  value: 1,

  addValue: function (value) {
    return atom.value + value;
  },
};

// good
const atom = {
  value: 1,

  addValue(value) {
    return atom.value + value;
  },
};
```
</code></pre> 
   </div> 
   <p><a name="es6-object-concise"></a><a name="3.6"></a> - <a href="#es6-object-concise">3.4</a> Use property value shorthand. eslint: <a href="http://eslint.org/docs/rules/object-shorthand.html"><code class="highlighter-rouge">object-shorthand</code></a> jscs: <a href="http://jscs.info/rule/requireEnhancedObjectLiterals"><code class="highlighter-rouge">requireEnhancedObjectLiterals</code></a></p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>&gt; Why? It is shorter to write and descriptive.

```javascript
const lukeSkywalker = 'Luke Skywalker';

// bad
const obj = {
  lukeSkywalker: lukeSkywalker,
};

// good
const obj = {
  lukeSkywalker,
};
```
</code></pre> 
   </div> 
   <p><a name="objects--grouped-shorthand"></a><a name="3.7"></a> - <a href="#objects--grouped-shorthand">3.5</a> Group your shorthand properties at the beginning of your object declaration.</p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>&gt; Why? It's easier to tell which properties are using the shorthand.

```javascript
const anakinSkywalker = 'Anakin Skywalker';
const lukeSkywalker = 'Luke Skywalker';

// bad
const obj = {
  episodeOne: 1,
  twoJediWalkIntoACantina: 2,
  lukeSkywalker,
  episodeThree: 3,
  mayTheFourth: 4,
  anakinSkywalker,
};

// good
const obj = {
  lukeSkywalker,
  anakinSkywalker,
  episodeOne: 1,
  twoJediWalkIntoACantina: 2,
  episodeThree: 3,
  mayTheFourth: 4,
};
```
</code></pre> 
   </div> 
   <p><a name="objects--quoted-props"></a><a name="3.8"></a> - <a href="#objects--quoted-props">3.6</a> Only quote properties that are invalid identifiers. eslint: <a href="http://eslint.org/docs/rules/quote-props.html"><code class="highlighter-rouge">quote-props</code></a> jscs: <a href="http://jscs.info/rule/disallowQuotedKeysInObjects"><code class="highlighter-rouge">disallowQuotedKeysInObjects</code></a></p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>&gt; Why? In general we consider it subjectively easier to read. It improves syntax highlighting, and is also more easily optimized by many JS engines.

```javascript
// bad
const bad = {
  'foo': 3,
  'bar': 4,
  'data-blah': 5,
};

// good
const good = {
  foo: 3,
  bar: 4,
  'data-blah': 5,
};
```
</code></pre> 
   </div> 
   <p><a name="objects--prototype-builtins"></a> - <a href="#objects--prototype-builtins">3.7</a> Do not call <code class="highlighter-rouge">Object.prototype</code> methods directly, such as <code class="highlighter-rouge">hasOwnProperty</code>, <code class="highlighter-rouge">propertyIsEnumerable</code>, and <code class="highlighter-rouge">isPrototypeOf</code>.</p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>&gt; Why? These methods may be shadowed by properties on the object in question - consider `{ hasOwnProperty: false }` - or, the object may be a null object (`Object.create(null)`).

```javascript
// bad
console.log(object.hasOwnProperty(key));

// good
console.log(Object.prototype.hasOwnProperty.call(object, key));

// best
const has = Object.prototype.hasOwnProperty; // cache the lookup once, in module scope.
/* or */
import has from 'has';
// ...
console.log(has.call(object, key));
```
</code></pre> 
   </div> 
   <p><a name="objects--rest-spread"></a> - <a href="#objects--rest-spread">3.8</a> Prefer the object spread operator over <a href="https://developer.mozilla.org/en/docs/Web/JavaScript/Reference/Global_Objects/Object/assign"><code class="highlighter-rouge">Object.assign</code></a> to shallow-copy objects. Use the object rest operator to get a new object with certain properties omitted.</p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>```javascript
// very bad
const original = { a: 1, b: 2 };
const copy = Object.assign(original, { c: 3 }); // this mutates `original` ?_?
delete copy.a; // so does this

// bad
const original = { a: 1, b: 2 };
const copy = Object.assign({}, original, { c: 3 }); // copy =&gt; { a: 1, b: 2, c: 3 }

// good
const original = { a: 1, b: 2 };
const copy = { ...original, c: 3 }; // copy =&gt; { a: 1, b: 2, c: 3 }

const { a, ...noA } = copy; // noA =&gt; { b: 2, c: 3 }
```
</code></pre> 
   </div> 
   <p><strong><a href="#table-of-contents">? back to top</a></strong></p> 
   <h2 id="arrays">Arrays</h2> 
   <p><a name="arrays--literals"></a><a name="4.1"></a> - <a href="#arrays--literals">4.1</a> Use the literal syntax for array creation. eslint: <a href="http://eslint.org/docs/rules/no-array-constructor.html"><code class="highlighter-rouge">no-array-constructor</code></a></p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>```javascript
// bad
const items = new Array();

// good
const items = [];
```
</code></pre> 
   </div> 
   <p><a name="arrays--push"></a><a name="4.2"></a> - <a href="#arrays--push">4.2</a> Use <a href="https://developer.mozilla.org/en/docs/Web/JavaScript/Reference/Global_Objects/Array/push">Array#push</a> instead of direct assignment to add items to an array.</p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>```javascript
const someStack = [];

// bad
someStack[someStack.length] = 'abracadabra';

// good
someStack.push('abracadabra');
```
</code></pre> 
   </div> 
   <p><a name="es6-array-spreads"></a><a name="4.3"></a> - <a href="#es6-array-spreads">4.3</a> Use array spreads <code class="highlighter-rouge">...</code> to copy arrays.</p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>```javascript
// bad
const len = items.length;
const itemsCopy = [];
let i;

for (i = 0; i &lt; len; i += 1) {
  itemsCopy[i] = items[i];
}

// good
const itemsCopy = [...items];
```
</code></pre> 
   </div> 
   <p><a name="arrays--from"></a><a name="4.4"></a> - <a href="#arrays--from">4.4</a> To convert an array-like object to an array, use <a href="https://developer.mozilla.org/en/docs/Web/JavaScript/Reference/Global_Objects/Array/from">Array.from</a>.</p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>```javascript
const foo = document.querySelectorAll('.foo');
const nodes = Array.from(foo);
```
</code></pre> 
   </div> 
   <p><a name="arrays--callback-return"></a><a name="4.5"></a> - <a href="#arrays--callback-return">4.5</a> Use return statements in array method callbacks. It’s ok to omit the return if the function body consists of a single statement following <a href="#8.2">8.2</a>. eslint: <a href="http://eslint.org/docs/rules/array-callback-return"><code class="highlighter-rouge">array-callback-return</code></a></p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>```javascript
// good
[1, 2, 3].map((x) =&gt; {
  const y = x + 1;
  return x * y;
});

// good
[1, 2, 3].map(x =&gt; x + 1);

// bad
const flat = {};
[[0, 1], [2, 3], [4, 5]].reduce((memo, item, index) =&gt; {
  const flatten = memo.concat(item);
  flat[index] = flatten;
});

// good
const flat = {};
[[0, 1], [2, 3], [4, 5]].reduce((memo, item, index) =&gt; {
  const flatten = memo.concat(item);
  flat[index] = flatten;
  return flatten;
});

// bad
inbox.filter((msg) =&gt; {
  const { subject, author } = msg;
  if (subject === 'Mockingbird') {
    return author === 'Harper Lee';
  } else {
    return false;
  }
});

// good
inbox.filter((msg) =&gt; {
  const { subject, author } = msg;
  if (subject === 'Mockingbird') {
    return author === 'Harper Lee';
  }

  return false;
});
```
</code></pre> 
   </div>

   <h2 id="destructuring">Destructuring</h2> 
   <p><a name="destructuring--object"></a><a name="5.1"></a> - <a href="#destructuring--object">5.1</a> Use object destructuring when accessing and using multiple properties of an object. jscs: <a href="http://jscs.info/rule/requireObjectDestructuring"><code class="highlighter-rouge">requireObjectDestructuring</code></a></p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>&gt; Why? Destructuring saves you from creating temporary references for those properties.

```javascript
// bad
function getFullName(user) {
  const firstName = user.firstName;
  const lastName = user.lastName;

  return `${firstName} ${lastName}`;
}

// good
function getFullName(user) {
  const { firstName, lastName } = user;
  return `${firstName} ${lastName}`;
}

// best
function getFullName({ firstName, lastName }) {
  return `${firstName} ${lastName}`;
}
```
</code></pre> 
   </div> 
   <p><a name="destructuring--array"></a><a name="5.2"></a> - <a href="#destructuring--array">5.2</a> Use array destructuring. jscs: <a href="http://jscs.info/rule/requireArrayDestructuring"><code class="highlighter-rouge">requireArrayDestructuring</code></a></p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>```javascript
const arr = [1, 2, 3, 4];

// bad
const first = arr[0];
const second = arr[1];

// good
const [first, second] = arr;
```
</code></pre> 
   </div> 
   <p><a name="destructuring--object-over-array"></a><a name="5.3"></a> - <a href="#destructuring--object-over-array">5.3</a> Use object destructuring for multiple return values, not array destructuring. jscs: <a href="http://jscs.info/rule/disallowArrayDestructuringReturn"><code class="highlighter-rouge">disallowArrayDestructuringReturn</code></a></p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>&gt; Why? You can add new properties over time or change the order of things without breaking call sites.

```javascript
// bad
function processInput(input) {
  // then a miracle occurs
  return [left, right, top, bottom];
}

// the caller needs to think about the order of return data
const [left, __, top] = processInput(input);

// good
function processInput(input) {
  // then a miracle occurs
  return { left, right, top, bottom };
}

// the caller selects only the data they need
const { left, top } = processInput(input);
```
</code></pre> 
   </div> 
   <p><strong><a href="#table-of-contents">? back to top</a></strong></p> 
   <h2 id="strings">Strings</h2> 
   <p><a name="strings--quotes"></a><a name="6.1"></a> - <a href="#strings--quotes">6.1</a> Use single quotes <code class="highlighter-rouge">''</code> for strings. eslint: <a href="http://eslint.org/docs/rules/quotes.html"><code class="highlighter-rouge">quotes</code></a> jscs: <a href="http://jscs.info/rule/validateQuoteMarks"><code class="highlighter-rouge">validateQuoteMarks</code></a></p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>```javascript
// bad
const name = "Capt. Janeway";

// bad - template literals should contain interpolation or newlines
const name = `Capt. Janeway`;

// good
const name = 'Capt. Janeway';
```
</code></pre> 
   </div> 
   <p><a name="strings--line-length"></a><a name="6.2"></a> - <a href="#strings--line-length">6.2</a> Strings that cause the line to go over 100 characters should not be written across multiple lines using string concatenation.</p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>&gt; Why? Broken strings are painful to work with and make code less searchable.

```javascript
// bad
const errorMessage = 'This is a super long error that was thrown because \
of Batman. When you stop to think about how Batman had anything to do \
with this, you would get nowhere \
fast.';

// bad
const errorMessage = 'This is a super long error that was thrown because ' +
  'of Batman. When you stop to think about how Batman had anything to do ' +
  'with this, you would get nowhere fast.';

// good
const errorMessage = 'This is a super long error that was thrown because of Batman. When you stop to think about how Batman had anything to do with this, you would get nowhere fast.';
```
</code></pre> 
   </div> 
   <p><a name="es6-template-literals"></a><a name="6.4"></a> - <a href="#es6-template-literals">6.3</a> When programmatically building up strings, use template strings instead of concatenation. eslint: <a href="http://eslint.org/docs/rules/prefer-template.html"><code class="highlighter-rouge">prefer-template</code></a> <a href="http://eslint.org/docs/rules/template-curly-spacing"><code class="highlighter-rouge">template-curly-spacing</code></a> jscs: <a href="http://jscs.info/rule/requireTemplateStrings"><code class="highlighter-rouge">requireTemplateStrings</code></a></p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>&gt; Why? Template strings give you a readable, concise syntax with proper newlines and string interpolation features.

```javascript
// bad
function sayHi(name) {
  return 'How are you, ' + name + '?';
}

// bad
function sayHi(name) {
  return ['How are you, ', name, '?'].join();
}

// bad
function sayHi(name) {
  return `How are you, ${ name }?`;
}

// good
function sayHi(name) {
  return `How are you, ${name}?`;
}
```
</code></pre> 
   </div> 
   <p><a name="strings--eval"></a><a name="6.5"></a> - <a href="#strings--eval">6.4</a> Never use <code class="highlighter-rouge">eval()</code> on a string, it opens too many vulnerabilities.</p> 
   <p><a name="strings--escaping"></a> - <a href="#strings--escaping">6.5</a> Do not unnecessarily escape characters in strings. eslint: <a href="http://eslint.org/docs/rules/no-useless-escape"><code class="highlighter-rouge">no-useless-escape</code></a></p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>&gt; Why? Backslashes harm readability, thus they should only be present when necessary.

```javascript
// bad
const foo = '\'this\' \i\s \"quoted\"';

// good
const foo = '\'this\' is "quoted"';
const foo = `my name is '${name}'`;
```
</code></pre> 
   </div> 
   <p><strong><a href="#table-of-contents">? back to top</a></strong></p> 
   <h2 id="functions">Functions</h2> 
   <p><a name="functions--declarations"></a><a name="7.1"></a> - <a href="#functions--declarations">7.1</a> Use named function expressions instead of function declarations. eslint: <a href="http://eslint.org/docs/rules/func-style"><code class="highlighter-rouge">func-style</code></a> jscs: <a href="http://jscs.info/rule/disallowFunctionDeclarations"><code class="highlighter-rouge">disallowFunctionDeclarations</code></a></p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>&gt; Why? Function declarations are hoisted, which means that it’s easy - too easy - to reference the function before it is defined in the file. This harms readability and maintainability. If you find that a function’s definition is large or complex enough that it is interfering with understanding the rest of the file, then perhaps it’s time to extract it to its own module! Don’t forget to name the expression - anonymous functions can make it harder to locate the problem in an Error's call stack. ([Discussion](https://github.com/airbnb/javascript/issues/794))

```javascript
// bad
const foo = function () {
  // ...
};

// bad
function foo() {
  // ...
}

// good
const foo = function bar() {
  // ...
};
```
</code></pre> 
   </div> 
   <p><a name="functions--iife"></a><a name="7.2"></a> - <a href="#functions--iife">7.2</a> Wrap immediately invoked function expressions in parentheses. eslint: <a href="http://eslint.org/docs/rules/wrap-iife.html"><code class="highlighter-rouge">wrap-iife</code></a> jscs: <a href="http://jscs.info/rule/requireParenthesesAroundIIFE"><code class="highlighter-rouge">requireParenthesesAroundIIFE</code></a></p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>&gt; Why? An immediately invoked function expression is a single unit - wrapping both it, and its invocation parens, in parens, cleanly expresses this. Note that in a world with modules everywhere, you almost never need an IIFE.

```javascript
// immediately-invoked function expression (IIFE)
(function () {
  console.log('Welcome to the Internet. Please follow me.');
}());
```
</code></pre> 
   </div> 
   <p><a name="functions--in-blocks"></a><a name="7.3"></a> - <a href="#functions--in-blocks">7.3</a> Never declare a function in a non-function block (if, while, etc). Assign the function to a variable instead. Browsers will allow you to do it, but they all interpret it differently, which is bad news bears. eslint: <a href="http://eslint.org/docs/rules/no-loop-func.html"><code class="highlighter-rouge">no-loop-func</code></a></p> 
   <p><a name="functions--note-on-blocks"></a><a name="7.4"></a> - <a href="#functions--note-on-blocks">7.4</a> <strong>Note:</strong> ECMA-262 defines a <code class="highlighter-rouge">block</code> as a list of statements. A function declaration is not a statement. <a href="http://www.ecma-international.org/publications/files/ECMA-ST/Ecma-262.pdf#page=97">Read ECMA-262’s note on this issue</a>.</p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>```javascript
// bad
if (currentUser) {
  function test() {
    console.log('Nope.');
  }
}

// good
let test;
if (currentUser) {
  test = () =&gt; {
    console.log('Yup.');
  };
}
```
</code></pre> 
   </div> 
   <p><a name="functions--arguments-shadow"></a><a name="7.5"></a> - <a href="#functions--arguments-shadow">7.5</a> Never name a parameter <code class="highlighter-rouge">arguments</code>. This will take precedence over the <code class="highlighter-rouge">arguments</code> object that is given to every function scope.</p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>```javascript
// bad
function foo(name, options, arguments) {
  // ...
}

// good
function foo(name, options, args) {
  // ...
}
```
</code></pre> 
   </div> 
   <p><a name="es6-rest"></a><a name="7.6"></a> - <a href="#es6-rest">7.6</a> Never use <code class="highlighter-rouge">arguments</code>, opt to use rest syntax <code class="highlighter-rouge">...</code> instead. eslint: <a href="http://eslint.org/docs/rules/prefer-rest-params"><code class="highlighter-rouge">prefer-rest-params</code></a></p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>&gt; Why? `...` is explicit about which arguments you want pulled. Plus, rest arguments are a real Array, and not merely Array-like like `arguments`.

```javascript
// bad
function concatenateAll() {
  const args = Array.prototype.slice.call(arguments);
  return args.join('');
}

// good
function concatenateAll(...args) {
  return args.join('');
}
```
</code></pre> 
   </div> 
   <p><a name="es6-default-parameters"></a><a name="7.7"></a> - <a href="#es6-default-parameters">7.7</a> Use default parameter syntax rather than mutating function arguments.</p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>```javascript
// really bad
function handleThings(opts) {
  // No! We shouldn't mutate function arguments.
  // Double bad: if opts is falsy it'll be set to an object which may
  // be what you want but it can introduce subtle bugs.
  opts = opts || {};
  // ...
}

// still bad
function handleThings(opts) {
  if (opts === void 0) {
    opts = {};
  }
  // ...
}

// good
function handleThings(opts = {}) {
  // ...
}
```
</code></pre> 
   </div> 
   <p><a name="functions--default-side-effects"></a><a name="7.8"></a> - <a href="#functions--default-side-effects">7.8</a> Avoid side effects with default parameters.</p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>&gt; Why? They are confusing to reason about.

```javascript
var b = 1;
// bad
function count(a = b++) {
  console.log(a);
}
count();  // 1
count();  // 2
count(3); // 3
count();  // 3
```
</code></pre> 
   </div> 
   <p><a name="functions--defaults-last"></a><a name="7.9"></a> - <a href="#functions--defaults-last">7.9</a> Always put default parameters last.</p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>```javascript
// bad
function handleThings(opts = {}, name) {
  // ...
}

// good
function handleThings(name, opts = {}) {
  // ...
}
```
</code></pre> 
   </div> 
   <p><a name="functions--constructor"></a><a name="7.10"></a> - <a href="#functions--constructor">7.10</a> Never use the Function constructor to create a new function. eslint: <a href="http://eslint.org/docs/rules/no-new-func"><code class="highlighter-rouge">no-new-func</code></a></p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>&gt; Why? Creating a function in this way evaluates a string similarly to eval(), which opens vulnerabilities.

```javascript
// bad
var add = new Function('a', 'b', 'return a + b');

// still bad
var subtract = Function('a', 'b', 'return a - b');
```
</code></pre> 
   </div> 
   <p><a name="functions--signature-spacing"></a><a name="7.11"></a> - <a href="#functions--signature-spacing">7.11</a> Spacing in a function signature. eslint: <a href="http://eslint.org/docs/rules/space-before-function-paren"><code class="highlighter-rouge">space-before-function-paren</code></a> <a href="http://eslint.org/docs/rules/space-before-blocks"><code class="highlighter-rouge">space-before-blocks</code></a></p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>&gt; Why? Consistency is good, and you shouldn’t have to add or remove a space when adding or removing a name.

```javascript
// bad
const f = function(){};
const g = function (){};
const h = function() {};

// good
const x = function () {};
const y = function a() {};
```
</code></pre> 
   </div> 
   <p><a name="functions--mutate-params"></a><a name="7.12"></a> - <a href="#functions--mutate-params">7.12</a> Never mutate parameters. eslint: <a href="http://eslint.org/docs/rules/no-param-reassign.html"><code class="highlighter-rouge">no-param-reassign</code></a></p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>&gt; Why? Manipulating objects passed in as parameters can cause unwanted variable side effects in the original caller.

```javascript
// bad
function f1(obj) {
  obj.key = 1;
}

// good
function f2(obj) {
  const key = Object.prototype.hasOwnProperty.call(obj, 'key') ? obj.key : 1;
}
```
</code></pre> 
   </div> 
   <p><a name="functions--reassign-params"></a><a name="7.13"></a> - <a href="#functions--reassign-params">7.13</a> Never reassign parameters. eslint: <a href="http://eslint.org/docs/rules/no-param-reassign.html"><code class="highlighter-rouge">no-param-reassign</code></a></p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>&gt; Why? Reassigning parameters can lead to unexpected behavior, especially when accessing the `arguments` object. It can also cause optimization issues, especially in V8.

```javascript
// bad
function f1(a) {
  a = 1;
  // ...
}

function f2(a) {
  if (!a) { a = 1; }
  // ...
}

// good
function f3(a) {
  const b = a || 1;
  // ...
}

function f4(a = 1) {
  // ...
}
```
</code></pre> 
   </div> 
   <p><a name="functions--spread-vs-apply"></a><a name="7.14"></a> - <a href="#functions--spread-vs-apply">7.14</a> Prefer the use of the spread operator <code class="highlighter-rouge">...</code> to call variadic functions. eslint: <a href="http://eslint.org/docs/rules/prefer-spread"><code class="highlighter-rouge">prefer-spread</code></a></p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>&gt; Why? It's cleaner, you don't need to supply a context, and you can not easily compose `new` with `apply`.

```javascript
// bad
const x = [1, 2, 3, 4, 5];
console.log.apply(console, x);

// good
const x = [1, 2, 3, 4, 5];
console.log(...x);

// bad
new (Function.prototype.bind.apply(Date, [null, 2016, 8, 5]));

// good
new Date(...[2016, 8, 5]);
```
</code></pre> 
   </div> 
   <p><a name="functions--signature-invocation-indentation"></a> - <a href="#functions--signature-invocation-indentation">7.15</a> Functions with multiline signatures, or invocations, should be indented just like every other multiline list in this guide: with each item on a line by itself, with a trailing comma on the last item.</p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>```javascript
// bad
function foo(bar,
             baz,
             quux) {
  // ...
}

// good
function foo(
  bar,
  baz,
  quux,
) {
  // ...
}

// bad
console.log(foo,
  bar,
  baz);

// good
console.log(
  foo,
  bar,
  baz,
);
```
</code></pre> 
   </div> 
   <p><strong><a href="#table-of-contents">? back to top</a></strong></p> 
   <h2 id="arrow-functions">Arrow Functions</h2> 
   <p><a name="arrows--use-them"></a><a name="8.1"></a> - <a href="#arrows--use-them">8.1</a> When you must use function expressions (as when passing an anonymous function), use arrow function notation. eslint: <a href="http://eslint.org/docs/rules/prefer-arrow-callback.html"><code class="highlighter-rouge">prefer-arrow-callback</code></a>, <a href="http://eslint.org/docs/rules/arrow-spacing.html"><code class="highlighter-rouge">arrow-spacing</code></a> jscs: <a href="http://jscs.info/rule/requireArrowFunctions"><code class="highlighter-rouge">requireArrowFunctions</code></a></p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>&gt; Why? It creates a version of the function that executes in the context of `this`, which is usually what you want, and is a more concise syntax.

&gt; Why not? If you have a fairly complicated function, you might move that logic out into its own function declaration.

```javascript
// bad
[1, 2, 3].map(function (x) {
  const y = x + 1;
  return x * y;
});

// good
[1, 2, 3].map((x) =&gt; {
  const y = x + 1;
  return x * y;
});
```
</code></pre> 
   </div> 
   <p><a name="arrows--implicit-return"></a><a name="8.2"></a> - <a href="#arrows--implicit-return">8.2</a> If the function body consists of a single expression, omit the braces and use the implicit return. Otherwise, keep the braces and use a <code class="highlighter-rouge">return</code> statement. eslint: <a href="http://eslint.org/docs/rules/arrow-parens.html"><code class="highlighter-rouge">arrow-parens</code></a>, <a href="http://eslint.org/docs/rules/arrow-body-style.html"><code class="highlighter-rouge">arrow-body-style</code></a> jscs: <a href="http://jscs.info/rule/disallowParenthesesAroundArrowParam"><code class="highlighter-rouge">disallowParenthesesAroundArrowParam</code></a>, <a href="http://jscs.info/rule/requireShorthandArrowFunctions"><code class="highlighter-rouge">requireShorthandArrowFunctions</code></a></p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>&gt; Why? Syntactic sugar. It reads well when multiple functions are chained together.

```javascript
// bad
[1, 2, 3].map(number =&gt; {
  const nextNumber = number + 1;
  `A string containing the ${nextNumber}.`;
});

// good
[1, 2, 3].map(number =&gt; `A string containing the ${number}.`);

// good
[1, 2, 3].map((number) =&gt; {
  const nextNumber = number + 1;
  return `A string containing the ${nextNumber}.`;
});

// good
[1, 2, 3].map((number, index) =&gt; ({
  [index]: number,
}));
```
</code></pre> 
   </div> 
   <p><a name="arrows--paren-wrap"></a><a name="8.3"></a> - <a href="#arrows--paren-wrap">8.3</a> In case the expression spans over multiple lines, wrap it in parentheses for better readability.</p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>&gt; Why? It shows clearly where the function starts and ends.

```javascript
// bad
['get', 'post', 'put'].map(httpMethod =&gt; Object.prototype.hasOwnProperty.call(
    httpMagicObjectWithAVeryLongName,
    httpMethod,
  )
);

// good
['get', 'post', 'put'].map(httpMethod =&gt; (
  Object.prototype.hasOwnProperty.call(
    httpMagicObjectWithAVeryLongName,
    httpMethod,
  )
));
```
</code></pre> 
   </div> 
   <p><a name="arrows--one-arg-parens"></a><a name="8.4"></a> - <a href="#arrows--one-arg-parens">8.4</a> If your function takes a single argument and doesn’t use braces, omit the parentheses. Otherwise, always include parentheses around arguments. eslint: <a href="http://eslint.org/docs/rules/arrow-parens.html"><code class="highlighter-rouge">arrow-parens</code></a> jscs: <a href="http://jscs.info/rule/disallowParenthesesAroundArrowParam"><code class="highlighter-rouge">disallowParenthesesAroundArrowParam</code></a></p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>&gt; Why? Less visual clutter.

```javascript
// bad
[1, 2, 3].map((x) =&gt; x * x);

// good
[1, 2, 3].map(x =&gt; x * x);

// good
[1, 2, 3].map(number =&gt; (
  `A long string with the ${number}. It’s so long that we don’t want it to take up space on the .map line!`
));

// bad
[1, 2, 3].map(x =&gt; {
  const y = x + 1;
  return x * y;
});

// good
[1, 2, 3].map((x) =&gt; {
  const y = x + 1;
  return x * y;
});
```
</code></pre> 
   </div> 
   <p><a name="arrows--confusing"></a><a name="8.5"></a> - <a href="#arrows--confusing">8.5</a> Avoid confusing arrow function syntax (<code class="highlighter-rouge">=&gt;</code>) with comparison operators (<code class="highlighter-rouge">&lt;=</code>, <code class="highlighter-rouge">&gt;=</code>). eslint: <a href="http://eslint.org/docs/rules/no-confusing-arrow"><code class="highlighter-rouge">no-confusing-arrow</code></a></p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>```javascript
// bad
const itemHeight = item =&gt; item.height &gt; 256 ? item.largeSize : item.smallSize;

// bad
const itemHeight = (item) =&gt; item.height &gt; 256 ? item.largeSize : item.smallSize;

// good
const itemHeight = item =&gt; (item.height &gt; 256 ? item.largeSize : item.smallSize);

// good
const itemHeight = (item) =&gt; {
  const { height, largeSize, smallSize } = item;
  return height &gt; 256 ? largeSize : smallSize;
};
```
</code></pre> 
   </div> 
   <p><strong><a href="#table-of-contents">? back to top</a></strong></p> 
   <h2 id="classes--constructors">Classes &amp; Constructors</h2> 
   <p><a name="constructors--use-class"></a><a name="9.1"></a> - <a href="#constructors--use-class">9.1</a> Always use <code class="highlighter-rouge">class</code>. Avoid manipulating <code class="highlighter-rouge">prototype</code> directly.</p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>&gt; Why? `class` syntax is more concise and easier to reason about.

```javascript
// bad
function Queue(contents = []) {
  this.queue = [...contents];
}
Queue.prototype.pop = function () {
  const value = this.queue[0];
  this.queue.splice(0, 1);
  return value;
};


// good
class Queue {
  constructor(contents = []) {
    this.queue = [...contents];
  }
  pop() {
    const value = this.queue[0];
    this.queue.splice(0, 1);
    return value;
  }
}
```
</code></pre> 
   </div> 
   <p><a name="constructors--extends"></a><a name="9.2"></a> - <a href="#constructors--extends">9.2</a> Use <code class="highlighter-rouge">extends</code> for inheritance.</p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>&gt; Why? It is a built-in way to inherit prototype functionality without breaking `instanceof`.

```javascript
// bad
const inherits = require('inherits');
function PeekableQueue(contents) {
  Queue.apply(this, contents);
}
inherits(PeekableQueue, Queue);
PeekableQueue.prototype.peek = function () {
  return this.queue[0];
};

// good
class PeekableQueue extends Queue {
  peek() {
    return this.queue[0];
  }
}
```
</code></pre> 
   </div> 
   <p><a name="constructors--chaining"></a><a name="9.3"></a> - <a href="#constructors--chaining">9.3</a> Methods can return <code class="highlighter-rouge">this</code> to help with method chaining.</p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>```javascript
// bad
Jedi.prototype.jump = function () {
  this.jumping = true;
  return true;
};

Jedi.prototype.setHeight = function (height) {
  this.height = height;
};

const luke = new Jedi();
luke.jump(); // =&gt; true
luke.setHeight(20); // =&gt; undefined

// good
class Jedi {
  jump() {
    this.jumping = true;
    return this;
  }

  setHeight(height) {
    this.height = height;
    return this;
  }
}

const luke = new Jedi();

luke.jump()
  .setHeight(20);
```
</code></pre> 
   </div> 
   <p><a name="constructors--tostring"></a><a name="9.4"></a> - <a href="#constructors--tostring">9.4</a> It’s okay to write a custom toString() method, just make sure it works successfully and causes no side effects.</p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>```javascript
class Jedi {
  constructor(options = {}) {
    this.name = options.name || 'no name';
  }

  getName() {
    return this.name;
  }

  toString() {
    return `Jedi - ${this.getName()}`;
  }
}
```
</code></pre> 
   </div> 
   <p><a name="constructors--no-useless"></a><a name="9.5"></a> - <a href="#constructors--no-useless">9.5</a> Classes have a default constructor if one is not specified. An empty constructor function or one that just delegates to a parent class is unnecessary. eslint: <a href="http://eslint.org/docs/rules/no-useless-constructor"><code class="highlighter-rouge">no-useless-constructor</code></a></p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>```javascript
// bad
class Jedi {
  constructor() {}

  getName() {
    return this.name;
  }
}

// bad
class Rey extends Jedi {
  constructor(...args) {
    super(...args);
  }
}

// good
class Rey extends Jedi {
  constructor(...args) {
    super(...args);
    this.name = 'Rey';
  }
}
```
</code></pre> 
   </div> 
   <p><a name="classes--no-duplicate-members"></a> - <a href="#classes--no-duplicate-members">9.6</a> Avoid duplicate class members. eslint: <a href="http://eslint.org/docs/rules/no-dupe-class-members"><code class="highlighter-rouge">no-dupe-class-members</code></a></p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>&gt; Why? Duplicate class member declarations will silently prefer the last one - having duplicates is almost certainly a bug.

```javascript
// bad
class Foo {
  bar() { return 1; }
  bar() { return 2; }
}

// good
class Foo {
  bar() { return 1; }
}

// good
class Foo {
  bar() { return 2; }
}
```
</code></pre> 
   </div>

   <h2 id="modules">Modules</h2> 
   <p><a name="modules--use-them"></a><a name="10.1"></a> - <a href="#modules--use-them">10.1</a> Always use modules (<code class="highlighter-rouge">import</code>/<code class="highlighter-rouge">export</code>) over a non-standard module system. You can always transpile to your preferred module system.</p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>&gt; Why? Modules are the future, let's start using the future now.

```javascript
// bad
const AirbnbStyleGuide = require('./AirbnbStyleGuide');
module.exports = AirbnbStyleGuide.es6;

// ok
import AirbnbStyleGuide from './AirbnbStyleGuide';
export default AirbnbStyleGuide.es6;

// best
import { es6 } from './AirbnbStyleGuide';
export default es6;
```
</code></pre> 
   </div> 
   <p><a name="modules--no-wildcard"></a><a name="10.2"></a> - <a href="#modules--no-wildcard">10.2</a> Do not use wildcard imports.</p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>&gt; Why? This makes sure you have a single default export.

```javascript
// bad
import * as AirbnbStyleGuide from './AirbnbStyleGuide';

// good
import AirbnbStyleGuide from './AirbnbStyleGuide';
```
</code></pre> 
   </div> 
   <p><a name="modules--no-export-from-import"></a><a name="10.3"></a> - <a href="#modules--no-export-from-import">10.3</a> And do not export directly from an import.</p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>&gt; Why? Although the one-liner is concise, having one clear way to import and one clear way to export makes things consistent.

```javascript
// bad
// filename es6.js
export { es6 as default } from './AirbnbStyleGuide';

// good
// filename es6.js
import { es6 } from './AirbnbStyleGuide';
export default es6;
```
</code></pre> 
   </div> 
   <p><a name="modules--no-duplicate-imports"></a> - <a href="#modules--no-duplicate-imports">10.4</a> Only import from a path in one place. eslint: <a href="http://eslint.org/docs/rules/no-duplicate-imports"><code class="highlighter-rouge">no-duplicate-imports</code></a> &gt; Why? Having multiple lines that import from the same path can make code harder to maintain.</p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>```javascript
// bad

// … some other imports … //
import { named1, named2 } from 'foo';

// good
import foo, { named1, named2 } from 'foo';

// good
import foo, {
  named1,
  named2,
} from 'foo';
```
</code></pre> 
   </div> 
   <p><a name="modules--no-mutable-exports"></a> - <a href="#modules--no-mutable-exports">10.5</a> Do not export mutable bindings. eslint: <a href="https://github.com/benmosher/eslint-plugin-import/blob/master/docs/rules/no-mutable-exports.md"><code class="highlighter-rouge">import/no-mutable-exports</code></a> &gt; Why? Mutation should be avoided in general, but in particular when exporting mutable bindings. While this technique may be needed for some special cases, in general, only constant references should be exported.</p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>```javascript
// bad

// good

export { foo };
```
</code></pre> 
   </div> 
   <p><a name="modules--prefer-default-export"></a> - <a href="#modules--prefer-default-export">10.6</a> In modules with a single export, prefer default export over named export. eslint: <a href="https://github.com/benmosher/eslint-plugin-import/blob/master/docs/rules/prefer-default-export.md"><code class="highlighter-rouge">import/prefer-default-export</code></a></p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>```javascript
// bad

// good

```
</code></pre> 
   </div> 
   <p><a name="modules--imports-first"></a> - <a href="#modules--imports-first">10.7</a> Put all <code class="highlighter-rouge">import</code>s above non-import statements. eslint: <a href="https://github.com/benmosher/eslint-plugin-import/blob/master/docs/rules/first.md"><code class="highlighter-rouge">import/first</code></a> &gt; Why? Since <code class="highlighter-rouge">import</code>s are hoisted, keeping them all at the top prevents surprising behavior.</p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>```javascript
// bad
import foo from 'foo';
foo.init();

import bar from 'bar';

// good
import foo from 'foo';
import bar from 'bar';

foo.init();
```
</code></pre> 
   </div> 
   <p><a name="modules--multiline-imports-over-newlines"></a> - <a href="#modules--multiline-imports-over-newlines">10.8</a> Multiline imports should be indented just like multiline array and object literals.</p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>&gt; Why? The curly braces follow the same indentation rules as every other curly brace block in the style guide, as do the trailing commas.

```javascript
// bad

// good
import {

```

Since using Webpack syntax in the imports couples the code to a module bundler. Prefer using the loader syntax in <code class="highlighter-rouge">webpack.config.js</code>.</p> 
   <div class="highlighter-rouge">

// good

```
</code></pre>

   <h2 id="iterators-and-generators">Iterators and Generators</h2> 
   <p><a name="iterators--nope"></a><a name="11.1"></a> - <a href="#iterators--nope">11.1</a> Don’t use iterators. Prefer JavaScript’s higher-order functions instead of loops like <code class="highlighter-rouge">for-in</code> or <code class="highlighter-rouge">for-of</code>. eslint: <a href="http://eslint.org/docs/rules/no-iterator.html"><code class="highlighter-rouge">no-iterator</code></a> <a href="http://eslint.org/docs/rules/no-restricted-syntax"><code class="highlighter-rouge">no-restricted-syntax</code></a></p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>&gt; Why? This enforces our immutable rule. Dealing with pure functions that return values is easier to reason about than side effects.

&gt; Use `map()` / `every()` / `filter()` / `find()` / `findIndex()` / `reduce()` / `some()` / ... to iterate over arrays, and `Object.keys()` / `Object.values()` / `Object.entries()` to produce arrays so you can iterate over objects.

```javascript
const numbers = [1, 2, 3, 4, 5];

// bad
let sum = 0;
for (let num of numbers) {
  sum += num;
}
sum === 15;

// good
let sum = 0;
numbers.forEach(num =&gt; sum += num);
sum === 15;

// best (use the functional force)
const sum = numbers.reduce((total, num) =&gt; total + num, 0);
sum === 15;

// bad
const increasedByOne = [];
for (let i = 0; i &lt; numbers.length; i++) {
  increasedByOne.push(numbers[i] + 1);
}

// good
const increasedByOne = [];
numbers.forEach(num =&gt; increasedByOne.push(num + 1));

// best (keeping it functional)
const increasedByOne = numbers.map(num =&gt; num + 1);
```
</code></pre> 
   </div> 
   <p><a name="generators--nope"></a><a name="11.2"></a> - <a href="#generators--nope">11.2</a> Don’t use generators for now.</p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>&gt; Why? They don't transpile well to ES5.
</code></pre> 
   </div> 
   <p><a name="generators--spacing"></a> - <a href="#generators--spacing">11.3</a> If you must use generators, or if you disregard <a href="#generators--nope">our advice</a>, make sure their function signature is spaced properly. eslint: <a href="http://eslint.org/docs/rules/generator-star-spacing"><code class="highlighter-rouge">generator-star-spacing</code></a></p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>&gt; Why? `function` and `*` are part of the same conceptual keyword - `*` is not a modifier for `function`, `function*` is a unique construct, different from `function`.

```javascript

  // ...
}

  // ...
};

// bad
const baz = function *() {
  // ...
};

// bad

  // ...
};

// bad

  // ...
}

// bad
function *foo() {
  // ...
}

// very bad
function
*
foo() {
  // ...
}

// very bad
const wat = function
*
() {
  // ...
};

// good

  // ...
}

// good

  // ...
};
```
</code></pre>

   <h2 id="variables">Variables</h2> 
   <p><a name="variables--const"></a><a name="13.1"></a> - <a href="#variables--const">13.1</a> Always use <code class="highlighter-rouge">const</code> to declare variables. Not doing so will result in global variables. We want to avoid polluting the global namespace. Captain Planet warned us of that. eslint: <a href="http://eslint.org/docs/rules/no-undef"><code class="highlighter-rouge">no-undef</code></a> <a href="http://eslint.org/docs/rules/prefer-const"><code class="highlighter-rouge">prefer-const</code></a></p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>```javascript
// bad
superPower = new SuperPower();

// good
const superPower = new SuperPower();
```
</code></pre> 
   </div> 
   <p><a name="variables--one-const"></a><a name="13.2"></a> - <a href="#variables--one-const">13.2</a> Use one <code class="highlighter-rouge">const</code> declaration per variable. eslint: <a href="http://eslint.org/docs/rules/one-var.html"><code class="highlighter-rouge">one-var</code></a> jscs: <a href="http://jscs.info/rule/disallowMultipleVarDecl"><code class="highlighter-rouge">disallowMultipleVarDecl</code></a></p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>&gt; Why? It's easier to add new variable declarations this way, and you never have to worry about swapping out a `;` for a `,` or introducing punctuation-only diffs. You can also step through each declaration with the debugger, instead of jumping through all of them at once.

```javascript
// bad
const items = getItems(),
    goSportsTeam = true,
    dragonball = 'z';

// bad
// (compare to above, and try to spot the mistake)
const items = getItems(),
    goSportsTeam = true;
    dragonball = 'z';

// good
const items = getItems();
const goSportsTeam = true;
const dragonball = 'z';
```
</code></pre> 
   </div> 
   <p><a name="variables--const-let-group"></a><a name="13.3"></a> - <a href="#variables--const-let-group">13.3</a> Group all your <code class="highlighter-rouge">const</code>s and then group all your <code class="highlighter-rouge">let</code>s.</p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>&gt; Why? This is helpful when later on you might need to assign a variable depending on one of the previous assigned variables.

```javascript
// bad
let i, len, dragonball,
    items = getItems(),
    goSportsTeam = true;

// bad
let i;
const items = getItems();
let dragonball;
const goSportsTeam = true;
let len;

// good
const goSportsTeam = true;
const items = getItems();
let dragonball;
let i;
let length;
```
</code></pre> 
   </div> 
   <p><a name="variables--define-where-used"></a><a name="13.4"></a> - <a href="#variables--define-where-used">13.4</a> Assign variables where you need them, but place them in a reasonable place.</p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>&gt; Why? `let` and `const` are block scoped and not function scoped.

```javascript
// bad - unnecessary function call
function checkName(hasName) {
  const name = getName();

  if (hasName === 'test') {
    return false;
  }

  if (name === 'test') {
    this.setName('');
    return false;
  }

  return name;
}

// good
function checkName(hasName) {
  if (hasName === 'test') {
    return false;
  }

  const name = getName();

  if (name === 'test') {
    this.setName('');
    return false;
  }

  return name;
}
```   &lt;a name="variables--no-chain-assignment"&gt;&lt;/a&gt;&lt;a name="13.5"&gt;&lt;/a&gt;   - [13.5](#variables--no-chain-assignment) Don't chain variable assignments.

&gt; Why? Chaining variable assignments creates implicit global variables.

```javascript
// bad
(function example() {
  // JavaScript interprets this as
  // let a = ( b = ( c = 1 ) );
  // The let keyword only applies to variable a; variables b and c become
  // global variables.
  let a = b = c = 1;
}());

console.log(a); // undefined
console.log(b); // 1
console.log(c); // 1

// good
(function example() {
  let a = 1;
  let b = a;
  let c = a;
}());

console.log(a); // undefined
console.log(b); // undefined
console.log(c); // undefined

// the same applies for `const`
```
</code></pre> 
   </div> 
   <p><a name="variables--unary-increment-decrement"></a><a name="13.6"></a> - <a href="#variables--unary-increment-decrement">13.6</a> Avoid using unary increments and decrements (++, –). eslint <a href="http://eslint.org/docs/rules/no-plusplus"><code class="highlighter-rouge">no-plusplus</code></a></p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>&gt; Why? Per the eslint documentation, unary increment and decrement statements are subject to automatic semicolon insertion and can cause silent errors with incrementing or decrementing values within an application. It is also more expressive to mutate your values with statements like `num += 1` instead of `num++` or `num ++`. Disallowing unary increment and decrement statements also prevents you from pre-incrementing/pre-decrementing values unintentionally which can also cause unexpected behavior in your programs.

```javascript
// bad

let array = [1, 2, 3];
let num = 1;
num++;
--num;

let sum = 0;
let truthyCount = 0;
for (let i = 0; i &lt; array.length; i++) {
  let value = array[i];
  sum += value;
  if (value) {
    truthyCount++;
  }
}

// good

let array = [1, 2, 3];
let num = 1;
num += 1;
num -= 1;

const sum = array.reduce((a, b) =&gt; a + b, 0);
const truthyCount = array.filter(Boolean).length;
```
</code></pre> 
   </div>

   <h2 id="hoisting">Hoisting</h2> 
   <p><a name="hoisting--about"></a><a name="14.1"></a> - <a href="#hoisting--about">14.1</a> <code class="highlighter-rouge">var</code> declarations get hoisted to the top of their scope, their assignment does not. <code class="highlighter-rouge">const</code> and <code class="highlighter-rouge">let</code> declarations are blessed with a new concept called <a href="https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Statements/let#Temporal_dead_zone_and_errors_with_let">Temporal Dead Zones (TDZ)</a>. It’s important to know why <a href="http://es-discourse.com/t/why-typeof-is-no-longer-safe/15">typeof is no longer safe</a>.</p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>```javascript
// we know this wouldn't work (assuming there
// is no notDefined global variable)
function example() {
  console.log(notDefined); // =&gt; throws a ReferenceError
}

// creating a variable declaration after you
// reference the variable will work due to
// variable hoisting. Note: the assignment
// value of `true` is not hoisted.
function example() {
  console.log(declaredButNotAssigned); // =&gt; undefined
  var declaredButNotAssigned = true;
}

// the interpreter is hoisting the variable
// declaration to the top of the scope,
// which means our example could be rewritten as:
function example() {
  let declaredButNotAssigned;
  console.log(declaredButNotAssigned); // =&gt; undefined
  declaredButNotAssigned = true;
}

// using const and let
function example() {
  console.log(declaredButNotAssigned); // =&gt; throws a ReferenceError
  console.log(typeof declaredButNotAssigned); // =&gt; throws a ReferenceError
  const declaredButNotAssigned = true;
}
```
</code></pre> 
   </div> 
   <p><a name="hoisting--anon-expressions"></a><a name="14.2"></a> - <a href="#hoisting--anon-expressions">14.2</a> Anonymous function expressions hoist their variable name, but not the function assignment.</p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>```javascript
function example() {
  console.log(anonymous); // =&gt; undefined

  anonymous(); // =&gt; TypeError anonymous is not a function

  var anonymous = function () {
    console.log('anonymous function expression');
  };
}
```
</code></pre> 
   </div> 
   <p><a name="hoisting--named-expresions"></a><a name="14.3"></a> - <a href="#hoisting--named-expresions">14.3</a> Named function expressions hoist the variable name, not the function name or the function body.</p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>```javascript
function example() {
  console.log(named); // =&gt; undefined

  named(); // =&gt; TypeError named is not a function

  superPower(); // =&gt; ReferenceError superPower is not defined

  var named = function superPower() {
    console.log('Flying');
  };
}

// the same is true when the function name
// is the same as the variable name.
function example() {
  console.log(named); // =&gt; undefined

  named(); // =&gt; TypeError named is not a function

  var named = function named() {
    console.log('named');
  };
}
```
</code></pre> 
   </div> 
   <p><a name="hoisting--declarations"></a><a name="14.4"></a> - <a href="#hoisting--declarations">14.4</a> Function declarations hoist their name and the function body.</p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>```javascript
function example() {
  superPower(); // =&gt; Flying

  function superPower() {
    console.log('Flying');
  }
}
```
</code></pre> 
   </div> 
   <ul> 
    <li>For more information refer to <a href="http://www.adequatelygood.com/2010/2/JavaScript-Scoping-and-Hoisting/">JavaScript Scoping &amp; Hoisting</a> by <a href="http://www.adequatelygood.com/">Ben Cherry</a>.</li> 
   </ul>

   <h2 id="comparison-operators--equality">Comparison Operators &amp; Equality</h2> 
   <p><a name="comparison--eqeqeq"></a><a name="15.1"></a> - <a href="#comparison--eqeqeq">15.1</a> Use <code class="highlighter-rouge">===</code> and <code class="highlighter-rouge">!==</code> over <code class="highlighter-rouge">==</code> and <code class="highlighter-rouge">!=</code>. eslint: <a href="http://eslint.org/docs/rules/eqeqeq.html"><code class="highlighter-rouge">eqeqeq</code></a></p> 
   <p><a name="comparison--if"></a><a name="15.2"></a> - <a href="#comparison--if">15.2</a> Conditional statements such as the <code class="highlighter-rouge">if</code> statement evaluate their expression using coercion with the <code class="highlighter-rouge">ToBoolean</code> abstract method and always follow these simple rules:</p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>+ **Objects** evaluate to **true**
+ **Undefined** evaluates to **false**
+ **Null** evaluates to **false**
+ **Booleans** evaluate to **the value of the boolean**
+ **Numbers** evaluate to **false** if **+0, -0, or NaN**, otherwise **true**
+ **Strings** evaluate to **false** if an empty string `''`, otherwise **true**

```javascript
if ([0] &amp;&amp; []) {
  // true
  // an array (even an empty one) is an object, objects will evaluate to true
}
```
</code></pre> 
   </div> 
   <p><a name="comparison--shortcuts"></a><a name="15.3"></a> - <a href="#comparison--shortcuts">15.3</a> Use shortcuts for booleans, but explicit comparisons for strings and numbers.</p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>```javascript
// bad
if (isValid === true) {
  // ...
}

// good
if (isValid) {
  // ...
}

// bad
if (name) {
  // ...
}

// good
if (name !== '') {
  // ...
}

// bad
if (collection.length) {
  // ...
}

// good
if (collection.length &gt; 0) {
  // ...
}
```
</code></pre> 
   </div> 
   <p><a name="comparison--moreinfo"></a><a name="15.4"></a> - <a href="#comparison--moreinfo">15.4</a> For more information see <a href="https://javascriptweblog.wordpress.com/2011/02/07/truth-equality-and-javascript/#more-2108">Truth Equality and JavaScript</a> by Angus Croll.</p> 
   <p><a name="comparison--switch-blocks"></a><a name="15.5"></a> - <a href="#comparison--switch-blocks">15.5</a> Use braces to create blocks in <code class="highlighter-rouge">case</code> and <code class="highlighter-rouge">default</code> clauses that contain lexical declarations (e.g. <code class="highlighter-rouge">let</code>, <code class="highlighter-rouge">const</code>, <code class="highlighter-rouge">function</code>, and <code class="highlighter-rouge">class</code>).</p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>&gt; Why? Lexical declarations are visible in the entire `switch` block but only get initialized when assigned, which only happens when its `case` is reached. This causes problems when multiple `case` clauses attempt to define the same thing.

eslint rules: [`no-case-declarations`](http://eslint.org/docs/rules/no-case-declarations.html).

```javascript
// bad
switch (foo) {
  case 1:
    let x = 1;
    break;
  case 2:
    const y = 2;
    break;
  case 3:
    function f() {
      // ...
    }
    break;
  default:
    class C {}
}

// good
switch (foo) {
  case 1: {
    let x = 1;
    break;
  }
  case 2: {
    const y = 2;
    break;
  }
  case 3: {
    function f() {
      // ...
    }
    break;
  }
  case 4:
    bar();
    break;
  default: {
    class C {}
  }
}
```
</code></pre> 
   </div> 
   <p><a name="comparison--nested-ternaries"></a><a name="15.6"></a> - <a href="#comparison--nested-ternaries">15.6</a> Ternaries should not be nested and generally be single line expressions.</p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>eslint rules: [`no-nested-ternary`](http://eslint.org/docs/rules/no-nested-ternary.html).

```javascript
// bad
const foo = maybe1 &gt; maybe2
  ? "bar"
  : value1 &gt; value2 ? "baz" : null;

// better
const maybeNull = value1 &gt; value2 ? 'baz' : null;

const foo = maybe1 &gt; maybe2
  ? 'bar'
  : maybeNull;

// best
const maybeNull = value1 &gt; value2 ? 'baz' : null;

const foo = maybe1 &gt; maybe2 ? 'bar' : maybeNull;
```
</code></pre> 
   </div> 
   <p><a name="comparison--unneeded-ternary"></a><a name="15.7"></a> - <a href="#comparison--unneeded-ternary">15.7</a> Avoid unneeded ternary statements.</p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>eslint rules: [`no-unneeded-ternary`](http://eslint.org/docs/rules/no-unneeded-ternary.html).

```javascript
// bad
const foo = a ? a : b;
const bar = c ? true : false;
const baz = c ? false : true;

// good
const foo = a || b;
const bar = !!c;
const baz = !c;
```
</code></pre> 
   </div>

   <h2 id="blocks">Blocks</h2> 
   <p><a name="blocks--braces"></a><a name="16.1"></a> - <a href="#blocks--braces">16.1</a> Use braces with all multi-line blocks.</p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>```javascript
// bad
if (test)

   <p><a name="coercion--numbers"></a><a name="21.3"></a> - <a href="#coercion--numbers">21.3</a> Numbers: Use <code class="highlighter-rouge">Number</code> for type casting and <code class="highlighter-rouge">parseInt</code> always with a radix for parsing strings. eslint: <a href="http://eslint.org/docs/rules/radix"><code class="highlighter-rouge">radix</code></a></p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>```javascript
const inputValue = '4';

// bad
const val = new Number(inputValue);

// bad
const val = +inputValue;

// bad
const val = inputValue &gt;&gt; 0;

// bad
const val = parseInt(inputValue);

// good
const val = Number(inputValue);

// good
const val = parseInt(inputValue, 10);
```
</code></pre> 
   </div> 
   <p><a name="coercion--comment-deviations"></a><a name="21.4"></a> - <a href="#coercion--comment-deviations">21.4</a> If for whatever reason you are doing something wild and <code class="highlighter-rouge">parseInt</code> is your bottleneck and need to use Bitshift for <a href="https://jsperf.com/coercion-vs-casting/3">performance reasons</a>, leave a comment explaining why and what you’re doing.</p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>```javascript
// good
/**
 * parseInt was the reason my code was slow.
 * Bitshifting the String to coerce it to a
 * Number made it a lot faster.
 */
const val = inputValue &gt;&gt; 0;
```
</code></pre> 
   </div> 
   <p><a name="coercion--bitwise"></a><a name="21.5"></a> - <a href="#coercion--bitwise">21.5</a> <strong>Note:</strong> Be careful when using bitshift operations. Numbers are represented as <a href="https://es5.github.io/#x4.3.19">64-bit values</a>, but bitshift operations always return a 32-bit integer (<a href="https://es5.github.io/#x11.7">source</a>). Bitshift can lead to unexpected behavior for integer values larger than 32 bits. <a href="https://github.com/airbnb/javascript/issues/109">Discussion</a>. Largest signed 32-bit Int is 2,147,483,647:</p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>```javascript
2147483647 &gt;&gt; 0; // =&gt; 2147483647
2147483648 &gt;&gt; 0; // =&gt; -2147483648
2147483649 &gt;&gt; 0; // =&gt; -2147483647
```
</code></pre> 
   </div> 
   <p><a name="coercion--booleans"></a><a name="21.6"></a> - <a href="#coercion--booleans">21.6</a> Booleans:</p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>```javascript
const age = 0;

// bad
const hasAge = new Boolean(age);

// good
const hasAge = Boolean(age);

// best
const hasAge = !!age;
```
</code></pre> 
   </div> 
   <p><strong><a href="#table-of-contents">? back to top</a></strong></p> 
   <h2 id="naming-conventions">Naming Conventions</h2> 
   <p><a name="naming--descriptive"></a><a name="22.1"></a> - <a href="#naming--descriptive">22.1</a> Avoid single letter names. Be descriptive with your naming. eslint: <a href="http://eslint.org/docs/rules/id-length"><code class="highlighter-rouge">id-length</code></a></p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>```javascript
// bad
function q() {
  // ...
}

// good
function query() {
  // ...
}
```
</code></pre> 
   </div> 
   <p><a name="naming--camelCase"></a><a name="22.2"></a> - <a href="#naming--camelCase">22.2</a> Use camelCase when naming objects, functions, and instances. eslint: <a href="http://eslint.org/docs/rules/camelcase.html"><code class="highlighter-rouge">camelcase</code></a> jscs: <a href="http://jscs.info/rule/requireCamelCaseOrUpperCaseIdentifiers"><code class="highlighter-rouge">requireCamelCaseOrUpperCaseIdentifiers</code></a></p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>```javascript
// bad
const OBJEcttsssss = {};
const this_is_my_object = {};
function c() {}

// good
const thisIsMyObject = {};
function thisIsMyFunction() {}
```
</code></pre> 
   </div> 
   <p><a name="naming--PascalCase"></a><a name="22.3"></a> - <a href="#naming--PascalCase">22.3</a> Use PascalCase only when naming constructors or classes. eslint: <a href="http://eslint.org/docs/rules/new-cap.html"><code class="highlighter-rouge">new-cap</code></a> jscs: <a href="http://jscs.info/rule/requireCapitalizedConstructors"><code class="highlighter-rouge">requireCapitalizedConstructors</code></a></p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>```javascript
// bad
function user(options) {
  this.name = options.name;
}

const bad = new user({
  name: 'nope',
});

// good
class User {
  constructor(options) {
    this.name = options.name;
  }
}

const good = new User({
  name: 'yup',
});
```
</code></pre> 
   </div> 
   <p><a name="naming--leading-underscore"></a><a name="22.4"></a> - <a href="#naming--leading-underscore">22.4</a> Do not use trailing or leading underscores. eslint: <a href="http://eslint.org/docs/rules/no-underscore-dangle.html"><code class="highlighter-rouge">no-underscore-dangle</code></a> jscs: <a href="http://jscs.info/rule/disallowDanglingUnderscores"><code class="highlighter-rouge">disallowDanglingUnderscores</code></a></p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>&gt; Why? JavaScript does not have the concept of privacy in terms of properties or methods. Although a leading underscore is a common convention to mean “private”, in fact, these properties are fully public, and as such, are part of your public API contract. This convention might lead developers to wrongly think that a change won't count as breaking, or that tests aren't needed. tl;dr: if you want something to be “private”, it must not be observably present.

```javascript
// bad
this.__firstName__ = 'Panda';
this.firstName_ = 'Panda';
this._firstName = 'Panda';

// good
this.firstName = 'Panda';
```
</code></pre> 
   </div> 
   <p><a name="naming--self-this"></a><a name="22.5"></a> - <a href="#naming--self-this">22.5</a> Don’t save references to <code class="highlighter-rouge">this</code>. Use arrow functions or <a href="https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/Function/bind">Function#bind</a>. jscs: <a href="http://jscs.info/rule/disallowNodeTypes"><code class="highlighter-rouge">disallowNodeTypes</code></a></p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>```javascript
// bad
function foo() {
  const self = this;
  return function () {
    console.log(self);
  };
}

// bad
function foo() {
  const that = this;
  return function () {
    console.log(that);
  };
}

// good
function foo() {
  return () =&gt; {
    console.log(this);
  };
}
```
</code></pre> 
   </div> 
   <p><a name="naming--filename-matches-export"></a><a name="22.6"></a> - <a href="#naming--filename-matches-export">22.6</a> A base filename should exactly match the name of its default export.</p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>```javascript
// file 1 contents
class CheckBox {
  // ...
}
export default CheckBox;

// file 2 contents
export default function fortyTwo() { return 42; }

// file 3 contents
export default function insideDirectory() {}

// in some other file
// bad
import CheckBox from './checkBox'; // PascalCase import/export, camelCase filename
import FortyTwo from './FortyTwo'; // PascalCase import/filename, camelCase export
import InsideDirectory from './InsideDirectory'; // PascalCase import/filename, camelCase export

// bad
import CheckBox from './check_box'; // PascalCase import/export, snake_case filename
import forty_two from './forty_two'; // snake_case import/filename, camelCase export
import inside_directory from './inside_directory'; // snake_case import, camelCase export
import index from './inside_directory/index'; // requiring the index file explicitly
import insideDirectory from './insideDirectory/index'; // requiring the index file explicitly

// good
import CheckBox from './CheckBox'; // PascalCase export/import/filename
import fortyTwo from './fortyTwo'; // camelCase export/import/filename
import insideDirectory from './insideDirectory'; // camelCase export/import/directory name/implicit "index"
// ^ supports both insideDirectory.js and insideDirectory/index.js
```
</code></pre> 
   </div> 
   <p><a name="naming--camelCase-default-export"></a><a name="22.7"></a> - <a href="#naming--camelCase-default-export">22.7</a> Use camelCase when you export-default a function. Your filename should be identical to your function’s name.</p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>```javascript
function makeStyleGuide() {
  // ...
}

export default makeStyleGuide;
```
</code></pre> 
   </div> 
   <p><a name="naming--PascalCase-singleton"></a><a name="22.8"></a> - <a href="#naming--PascalCase-singleton">22.8</a> Use PascalCase when you export a constructor / class / singleton / function library / bare object.</p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>```javascript
const AirbnbStyleGuide = {
  es6: {
  },
};

export default AirbnbStyleGuide;
```
</code></pre> 
   </div> 
   <p><a name="naming--Acronyms-and-Initialisms"></a> - <a href="#naming--Acronyms-and-Initialisms">22.9</a> Acronyms and initialisms should always be all capitalized, or all lowercased.</p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>&gt; Why? Names are for readability, not to appease a computer algorithm.

```javascript
// bad
import SmsContainer from './containers/SmsContainer';

// bad
const HttpRequests = [
  // ...
];

// good
import SMSContainer from './containers/SMSContainer';

// good
const HTTPRequests = [
  // ...
];

// best
import TextMessageContainer from './containers/TextMessageContainer';

// best
const Requests = [
  // ...
];
```
</code></pre> 
   </div> 
   <p><strong><a href="#table-of-contents">? back to top</a></strong></p> 
   <h2 id="accessors">Accessors</h2> 
   <p><a name="accessors--not-required"></a><a name="23.1"></a> - <a href="#accessors--not-required">23.1</a> Accessor functions for properties are not required.</p> 
   <p><a name="accessors--no-getters-setters"></a><a name="23.2"></a> - <a href="#accessors--no-getters-setters">23.2</a> Do not use JavaScript getters/setters as they cause unexpected side effects and are harder to test, maintain, and reason about. Instead, if you do make accessor functions, use getVal() and setVal(‘hello’).</p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>```javascript
// bad
class Dragon {
  get age() {
    // ...
  }

  set age(value) {
    // ...
  }
}

// good
class Dragon {
  getAge() {
    // ...
  }

  setAge(value) {
    // ...
  }
}
```
</code></pre> 
   </div> 
   <p><a name="accessors--boolean-prefix"></a><a name="23.3"></a> - <a href="#accessors--boolean-prefix">23.3</a> If the property/method is a <code class="highlighter-rouge">boolean</code>, use <code class="highlighter-rouge">isVal()</code> or <code class="highlighter-rouge">hasVal()</code>.</p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>```javascript
// bad
if (!dragon.age()) {
  return false;
}

// good
if (!dragon.hasAge()) {
  return false;
}
```
</code></pre> 
   </div> 
   <p><a name="accessors--consistent"></a><a name="23.4"></a> - <a href="#accessors--consistent">23.4</a> It’s okay to create get() and set() functions, but be consistent.</p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>```javascript
class Jedi {
  constructor(options = {}) {
    const lightsaber = options.lightsaber || 'blue';
    this.set('lightsaber', lightsaber);
  }

  set(key, val) {
    this[key] = val;
  }

  get(key) {
    return this[key];
  }
}
```
</code></pre> 
   </div> 
   <p><strong><a href="#table-of-contents">? back to top</a></strong></p> 
   <h2 id="events">Events</h2> 
   <p><a name="events--hash"></a><a name="24.1"></a> - <a href="#events--hash">24.1</a> When attaching data payloads to events (whether DOM events or something more proprietary like Backbone events), pass a hash instead of a raw value. This allows a subsequent contributor to add more data to the event payload without finding and updating every handler for the event. For example, instead of:</p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>```javascript
// bad
$(this).trigger('listingUpdated', listing.id);

// ...

$(this).on('listingUpdated', (e, listingId) =&gt; {
  // do something with listingId
});
```

prefer:

```javascript
// good
$(this).trigger('listingUpdated', { listingId: listing.id });

// ...

$(this).on('listingUpdated', (e, data) =&gt; {
  // do something with data.listingId
});
```
</code></pre> 
   </div> 
   <p><strong><a href="#table-of-contents">? back to top</a></strong></p> 
   <h2 id="jquery">jQuery</h2> 
   <p><a name="jquery--dollar-prefix"></a><a name="25.1"></a> - <a href="#jquery--dollar-prefix">25.1</a> Prefix jQuery object variables with a <code class="highlighter-rouge">$</code>. jscs: <a href="http://jscs.info/rule/requireDollarBeforejQueryAssignment"><code class="highlighter-rouge">requireDollarBeforejQueryAssignment</code></a></p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>```javascript
// bad
const sidebar = $('.sidebar');

// good
const $sidebar = $('.sidebar');

// good
const $sidebarBtn = $('.sidebar-btn');
```
</code></pre> 
   </div> 
   <p><a name="jquery--cache"></a><a name="25.2"></a> - <a href="#jquery--cache">25.2</a> Cache jQuery lookups.</p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>```javascript
// bad
function setSidebar() {
  $('.sidebar').hide();

  // ...

  $('.sidebar').css({
    'background-color': 'pink',
  });
}

// good
function setSidebar() {
  const $sidebar = $('.sidebar');
  $sidebar.hide();

  // ...

  $sidebar.css({
    'background-color': 'pink',
  });
}
```
</code></pre> 
   </div> 
   <p><a name="jquery--queries"></a><a name="25.3"></a> - <a href="#jquery--queries">25.3</a> For DOM queries use Cascading <code class="highlighter-rouge">$('.sidebar ul')</code> or parent &gt; child <code class="highlighter-rouge">$('.sidebar &gt; ul')</code>. <a href="http://jsperf.com/jquery-find-vs-context-sel/16">jsPerf</a></p> 
   <p><a name="jquery--find"></a><a name="25.4"></a> - <a href="#jquery--find">25.4</a> Use <code class="highlighter-rouge">find</code> with scoped jQuery object queries.</p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>```javascript
// bad
$('ul', '.sidebar').hide();

// bad
$('.sidebar').find('ul').hide();

// good
$('.sidebar ul').hide();

// good
$('.sidebar &gt; ul').hide();

// good
$sidebar.find('ul').hide();
```
</code></pre> 
   </div>

   <h2 id="testing">Testing</h2> 
   <p><a name="testing--yup"></a><a name="28.1"></a> - <a href="#testing--yup">28.1</a> <strong>Yup.</strong></p> 
   <div class="highlighter-rouge"> 
    <pre class="highlight"><code>```javascript
function foo() {
  return true;
}
```
</code></pre> 
   </div> 
   <p><a name="testing--for-real"></a><a name="28.2"></a> - <a href="#testing--for-real">28.2</a> <strong>No, but seriously</strong>: - Whichever testing framework you use, you should be writing tests! - Strive to write many small pure functions, and minimize where mutations occur. - Be cautious about stubs and mocks - they can make your tests more brittle. - We primarily use <a href="https://www.npmjs.com/package/mocha"><code class="highlighter-rouge">mocha</code></a> at Airbnb. <a href="https://www.npmjs.com/package/tape"><code class="highlighter-rouge">tape</code></a> is also used occasionally for small, separate modules. - 100% test coverage is a good goal to strive for, even if it’s not always practical to reach it. - Whenever you fix a bug, <em>write a regression test</em>. A bug fixed without a regression test is almost certainly going to break again in the future.</p>
