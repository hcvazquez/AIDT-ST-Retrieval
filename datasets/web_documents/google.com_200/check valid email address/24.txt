      <h1 class="p-name"> Form Validation Part 2: The Constraint Validation API&nbsp;(JavaScript) </h1> 
     </header> 
     <p class="author-byline"> <span class="author-and-date"> By <a href="https://css-tricks.com/author/chrisferdinandi/"> <span class="p-author"> Chris Ferdinandi </span> </a> <span class="slashes">On</span> <time datetime="2017-06-27" class="dt-published"> June 27, 2017 </time> </span> <span class="byline-tags"> 
       <svg class="icon-tag" width="12px" height="12px">
        <use xlink:href="#icon-tag"></use>
       </svg><a href="https://css-tricks.com/tag/form-validation/" rel="tag">form validation</a>, <a href="https://css-tricks.com/tag/forms/" rel="tag">forms</a>, <a href="https://css-tricks.com/tag/ux/" rel="tag">ux</a> </span> </p> 
     <div class="article-content"> 
      <div class="module module-ad"> 
       <div id="bsap_1279518" class="bsarocks bsap_3469a2a501a9e18091036aa0c89f9dcb"></div> 
      </div> 
      <p>In my last article, I showed you how to use native browser form validation through a combination of semantic input types (for example, <code>&lt;input type="email"&gt;</code>) and validation attributes (such as <code>required</code> and <code>pattern</code>).</p> 
      <p>While incredibly easy and super lightweight, this approach does have a few shortcomings.</p> 
      <ol> 
       <li>You can style fields that have errors on them with the <code>:invalid</code> pseudo-selector, but you can't style the error messages themselves.</li> 
       <li>Behavior is also inconsistent across browsers.</li> 
      </ol> 
      <p>User studies from <a href="https://baymard.com/blog/inline-form-validation">Christian Holst</a> and <a href="https://alistapart.com/article/inline-validation-in-web-forms">Luke Wroblewski</a> (separately) found that displaying an error when the user leaves a field, and keeping that error persistent until the issue is fixed, provided the best and fastest user experience. </p> 
      <p>Unfortunately, none of the browsers natively behave this way. However, there is a way to get this behavior without depending on a large JavaScript form validation library.</p> 
      <p><span id="more-256019"></span></p> 
      <div class="article-series"> 
       <h4>Article Series:</h4> 
       <ol> 
        <li><a href="https://css-tricks.com/form-validation-part-1-constraint-validation-html/">Constraint Validation in HTML</a></li> 
        <li>The Constraint Validation API in JavaScript (You are here!)</li> 
        <li><a href="https://css-tricks.com/form-validation-part-3-validity-state-api-polyfill/">A Validity State API Polyfill</a></li> 
        <li><a href="https://css-tricks.com/form-validation-part-4-validating-mailchimp-subscribe-form/">Validating the MailChimp Subscribe Form</a></li> 
       </ol> 
      </div> 
      <h3>The Constraint Validation API</h3> 
      <p>In addition to HTML attributes, browser-native constraint validation also provides a JavaScript API we can use to customize our form validation behavior.</p> 
      <p>There are a few different methods the API exposes, but the most powerful, Validity State, allows us to use the browser's own field validation algorithms in our scripts instead of writing our own.</p> 
      <p>In this article, I'm going to show you how to use Validity State to customize the behavior, appearance, and content of your form validation error messages.</p> 
      <h3>Validity State</h3> 
      <p>The <code>validity</code> property provides a set of information about a form field, in the form of boolean (<code>true</code>/<code>false</code>) values.</p> 
      <pre rel="JavaScript"><code class="language-javascript">var myField = document.querySelector('input[type="text"]');
var validityState = myField.validity;</code></pre> 
      <p>The returned object contains the following properties:</p> 
      <ul> 
       <li><code>valid</code> - Is <code>true</code> when the field passes validation.</li> 
       <li><code>valueMissing</code> - Is <code>true</code> when the field is empty but required.</li> 
       <li><code>typeMismatch</code> - Is <code>true</code> when the field <code>type</code> is <code>email</code> or <code>url</code> but the entered <code>value</code> is not the correct type.</li> 
       <li><code>tooShort</code> - Is <code>true</code> when the field contains a <code>minLength</code> attribute and the entered <code>value</code> is shorter than that length.</li> 
       <li><code>tooLong</code> - Is <code>true</code> when the field contains a <code>maxLength</code> attribute and the entered <code>value</code> is longer than that length.</li> 
       <li><code>patternMismatch</code> - Is <code>true</code> when the field contains a <code>pattern</code> attribute and the entered <code>value</code> does not match the pattern.</li> 
       <li><code>badInput</code> - Is <code>true</code> when the input <code>type</code> is <code>number</code> and the entered <code>value</code> is not a number.</li> 
       <li><code>stepMismatch</code> - Is <code>true</code> when the field has a <code>step</code> attribute and the entered <code>value</code> does not adhere to the step values.</li> 
       <li><code>rangeOverflow</code> - Is <code>true</code> when the field has a <code>max</code> attribute and the entered number <code>value</code> is greater than the max.</li> 
       <li><code>rangeUnderflow</code> - Is <code>true</code> when the field has a <code>min</code> attribute and the entered number <code>value</code> is lower than the min.</li> 
      </ul> 
      <p>By using the <code>validity</code> property in conjunction with our input types and HTML validation attributes, we can build a robust form validation script that provides a great user experience with a relatively small amount of JavaScript.</p> 
      <p>Let's get to it!</p> 
      <h3>Disable native form validation</h3> 
      <p>Since we're writing our validation script, we want to disable the native browser validation by adding the <code>novalidate</code> attribute to our forms. We can still use the Constraint Validation API — we just want to prevent the native error messages from displaying.</p> 
      <p>As a best practice, we should add this attribute with JavaScript so that if our script has an error or fails to load, the native browser form validation will still work.</p> 
      <pre rel="JavaScript"><code class="language-javascript">// Add the novalidate attribute when the JS loads
var forms = document.querySelectorAll('form');
for (var i = 0; i &lt; forms.length; i++) {
    forms[i].setAttribute('novalidate', true);
}</code></pre> 
      <p>There may be some forms that you don't want to validate (for example, a search form that shows up on every page). Rather than apply our validation script to all forms, let's apply it just to forms that have the <code>.validate</code> class.</p> 
      <pre rel="JavaScript"><code class="language-javascript">// Add the novalidate attribute when the JS loads
var forms = document.querySelectorAll('.validate');
for (var i = 0; i &lt; forms.length; i++) {
    forms[i].setAttribute('novalidate', true);
}</code></pre> 
      <p data-height="501" data-theme-id="1" data-slug-hash="yXYNNM" data-default-tab="result,js" data-user="cferdinandi" data-embed-version="2" data-pen-title="Form Validation: Add `novalidate` programatically" class="codepen">See the Pen <a href="https://codepen.io/cferdinandi/pen/yXYNNM/">Form Validation: Add `novalidate` programatically</a> by Chris Ferdinandi (<a href="https://codepen.io/cferdinandi">@cferdinandi</a>) on <a href="https://codepen.io">CodePen</a>.</p> 
      <h3>Check validity when the user leaves the field</h3> 
      <p>Whenever a user leaves a field, we want to check if it's valid. To do this, we'll setup an event listener.</p> 
      <p>Rather than add a listener to every form field, we'll use a technique called event bubbling (or event propagation) to listen for all <code>blur</code> events.</p> 
      <pre rel="JavaScript"><code class="language-javascript">// Listen to all blur events
document.addEventListener('blur', function (event) {
    // Do something on blur...
}, true);</code></pre> 
      <p>You'll note that the last argument in <code>addEventListener</code> is set to <code>true</code>. This argument is called <code>useCapture</code>, and it's normally set to <code>false</code>. The <code>blur</code> event doesn't bubble the way events like <code>click</code> do. Setting this argument to true allows us to capture all <code>blur</code> events rather than only those that happen directly on the element we're listening to.</p> 
      <p>Next, we want to make sure that the blurred element was a field in a form with the <code>.validate</code> class. We can get the blurred element using <code>event.target</code>, and get it's parent form by calling <code>event.target.form</code>. Then we'll use <code>classList</code> to check if the form has the validation class or not.</p> 
      <p>If it does, we can check the field validity.</p> 
      <pre rel="JavaScript"><code class="language-javascript">// Listen to all blur events
document.addEventListener('blur', function (event) {

    // Only run if the field is in a form to be validated
    if (!event.target.form.classList.contains('validate')) return;

    // Validate the field
    var error = event.target.validity;
    console.log(error);

}, true);</code></pre> 
      <p>If <code>error</code> is <code>true</code>, the field is valid. Otherwise, there's an error.</p> 
      <p data-height="458" data-theme-id="1" data-slug-hash="yXYYyq" data-default-tab="js,result" data-user="cferdinandi" data-embed-version="2" data-pen-title="Form Validation: Validate On Blur" class="codepen">See the Pen <a href="https://codepen.io/cferdinandi/pen/yXYYyq/">Form Validation: Validate On Blur</a> by Chris Ferdinandi (<a href="https://codepen.io/cferdinandi">@cferdinandi</a>) on <a href="https://codepen.io">CodePen</a>.</p> 
      <h3>Getting the error</h3> 
      <p>Once we know there's an error, it's helpful to know what the error actually is. We can use the other Validity State properties to get that information.</p> 
      <p>Since we need to check each property, the code for this can get a bit long. Let's setup a separate function for this and pass our field into it.</p> 
      <pre rel="JavaScript"><code class="language-javascript">// Validate the field
var hasError = function (field) {
    // Get the error
};

// Listen to all blur events
document.addEventListner('blur', function (event) {

    // Only run if the field is in a form to be validated
    if (!event.target.form.classList.contains('validate')) return;

    // Validate the field
    var error = hasError(event.target);

}, true);</code></pre> 
      <p>There are a few field types we want to ignore: fields that are disabled, <code>file</code> and <code>reset</code> inputs, and <code>submit</code> inputs and buttons. If a field isn't one of those, let's get it's validity.</p> 
      <pre rel="JavaScript"><code class="language-javascript">// Validate the field
var hasError = function (field) {

    // Don't validate submits, buttons, file and reset inputs, and disabled fields
    if (field.disabled || field.type === 'file' || field.type === 'reset' || field.type === 'submit' || field.type === 'button') return;

    // Get validity
    var validity = field.validity;

};</code></pre> 
      <p>If there's no error, we'll return <code>null</code>. Otherwise, we'll check each of the Validity State properties until we find the error. </p> 
      <p>When we find the match, we'll return a string with the error. If none of the properties are <code>true</code> but <code>validity</code> is false, we'll return a generic "catchall" error message (I can't imagine a scenario where this happens, but it's good to plan for the unexpected).</p> 
      <pre rel="JavaScript"><code class="language-javascript">// Validate the field
var hasError = function (field) {

    // Don't validate submits, buttons, file and reset inputs, and disabled fields
    if (field.disabled || field.type === 'file' || field.type === 'reset' || field.type === 'submit' || field.type === 'button') return;

    // Get validity
    var validity = field.validity;

    // If valid, return null
    if (validity.valid) return;

    // If field is required and empty
    if (validity.valueMissing) return 'Please fill out this field.';

    // If not the right type
    if (validity.typeMismatch) return 'Please use the correct input type.';

    // If too short
    if (validity.tooShort) return 'Please lengthen this text.';

    // If too long
    if (validity.tooLong) return 'Please shorten this text.';

    // If number input isn't a number
    if (validity.badInput) return 'Please enter a number.';

    // If a number value doesn't match the step interval
    if (validity.stepMismatch) return 'Please select a valid value.';

    // If a number field is over the max
    if (validity.rangeOverflow) return 'Please select a smaller value.';

    // If a number field is below the min
    if (validity.rangeUnderflow) return 'Please select a larger value.';

    // If pattern doesn't match
    if (validity.patternMismatch) return 'Please match the requested format.';

    // If all else fails, return a generic catchall error
    return 'The value you entered for this field is invalid.';

};</code></pre> 
      <p>This is a good start, but we can do some additional parsing to make a few of our errors more useful. For <code>typeMismatch</code>, we can check if it's supposed to be an <code>email</code> or <code>url</code> and customize the error accordingly.</p> 
      <pre rel="JavaScript"><code class="language-javascript">// If not the right type
if (validity.typeMismatch) {

    // Email
    if (field.type === 'email') return 'Please enter an email address.';

    // URL
    if (field.type === 'url') return 'Please enter a URL.';

}</code></pre> 
      <p>If the field value is too long or too short, we can find out both how long or short it's supposed to be and how long or short it actually is. We can then include that information in the error.</p> 
      <pre rel="JavaScript"><code class="language-javascript">// If too short
if (validity.tooShort) return 'Please lengthen this text to ' + field.getAttribute('minLength') + ' characters or more. You are currently using ' + field.value.length + ' characters.';

// If too long
if (validity.tooLong) return 'Please short this text to no more than ' + field.getAttribute('maxLength') + ' characters. You are currently using ' + field.value.length + ' characters.';</code></pre> 
      <p>If a number field is over or below the allowed range, we can include that minimum or maximum allowed value in our error.</p> 
      <pre rel="JavaScript"><code class="language-javascript">// If a number field is over the max
if (validity.rangeOverflow) return 'Please select a value that is no more than ' + field.getAttribute('max') + '.';

// If a number field is below the min
if (validity.rangeUnderflow) return 'Please select a value that is no less than ' + field.getAttribute('min') + '.';</code></pre> 
      <p>And if there is a <code>pattern</code> mismatch and the field has a <code>title</code>, we can use that as our error, just like the native browser behavior.</p> 
      <pre rel="JavaScript"><code class="language-javascript">// If pattern doesn't match
if (validity.patternMismatch) {

    // If pattern info is included, return custom error
    if (field.hasAttribute('title')) return field.getAttribute('title');

    // Otherwise, generic error
    return 'Please match the requested format.';

}</code></pre> 
      <p>Here's the complete code for our <code>hasError()</code> function.</p> 
      <pre rel="JavaScript"><code class="language-javascript">// Validate the field
var hasError = function (field) {

    // Don't validate submits, buttons, file and reset inputs, and disabled fields
    if (field.disabled || field.type === 'file' || field.type === 'reset' || field.type === 'submit' || field.type === 'button') return;

    // Get validity
    var validity = field.validity;

    // If valid, return null
    if (validity.valid) return;

    // If field is required and empty
    if (validity.valueMissing) return 'Please fill out this field.';

    // If not the right type
    if (validity.typeMismatch) {

        // Email
        if (field.type === 'email') return 'Please enter an email address.';

        // URL
        if (field.type === 'url') return 'Please enter a URL.';

    }

    // If too short
    if (validity.tooShort) return 'Please lengthen this text to ' + field.getAttribute('minLength') + ' characters or more. You are currently using ' + field.value.length + ' characters.';

    // If too long
    if (validity.tooLong) return 'Please shorten this text to no more than ' + field.getAttribute('maxLength') + ' characters. You are currently using ' + field.value.length + ' characters.';

    // If number input isn't a number
    if (validity.badInput) return 'Please enter a number.';

    // If a number value doesn't match the step interval
    if (validity.stepMismatch) return 'Please select a valid value.';

    // If a number field is over the max
    if (validity.rangeOverflow) return 'Please select a value that is no more than ' + field.getAttribute('max') + '.';

    // If a number field is below the min
    if (validity.rangeUnderflow) return 'Please select a value that is no less than ' + field.getAttribute('min') + '.';

    // If pattern doesn't match
    if (validity.patternMismatch) {

        // If pattern info is included, return custom error
        if (field.hasAttribute('title')) return field.getAttribute('title');

        // Otherwise, generic error
        return 'Please match the requested format.';

    }

    // If all else fails, return a generic catchall error
    return 'The value you entered for this field is invalid.';

};</code></pre> 
      <p>Try it yourself in the pen below.</p> 
      <p data-height="494" data-theme-id="1" data-slug-hash="qjOOZB" data-default-tab="js,result" data-user="cferdinandi" data-embed-version="2" data-pen-title="Form Validation: Get the Error" class="codepen">See the Pen <a href="https://codepen.io/cferdinandi/pen/qjOOZB/">Form Validation: Get the Error</a> by Chris Ferdinandi (<a href="https://codepen.io/cferdinandi">@cferdinandi</a>) on <a href="https://codepen.io">CodePen</a>.</p> 
      <h3>Show an error message</h3> 
      <p>Once we get our error, we can display it below the field. We'll create a <code>showError()</code> function to handle this, and pass in our field and the error. Then, we'll call it in our event listener.</p> 
      <pre rel="JavaScript"><code class="language-javascript">// Show the error message
var showError = function (field, error) {
    // Show the error message...
};

// Listen to all blur events
document.addEventListener('blur', function (event) {

    // Only run if the field is in a form to be validated
    if (!event.target.form.classList.contains('validate')) return;

    // Validate the field
    var error = hasError(event.target);

    // If there's an error, show it
    if (error) {
        showError(event.target, error);
    }

}, true);</code></pre> 
      <p>In our <code>showError</code> function, we're going to do a few things:</p> 
      <ol> 
       <li>We'll add a class to the field with the error so that we can style it.</li> 
       <li>If an error message already exists, we'll update it with new text.</li> 
       <li>Otherwise, we'll create a message and inject it into the DOM immediately after the field.</li> 
      </ol> 
      <p>We'll also use the field ID to create a unique ID for the message so we can find it again later (falling back to the field <code>name</code> in case there's no ID).</p> 
      <pre rel="JavaScript"><code class="language-javascript">var showError = function (field, error) {

    // Add error class to field
    field.classList.add('error');

    // Get field id or name
    var id = field.id || field.name;
    if (!id) return;

    // Check if error message field already exists
    // If not, create one
    var message = field.form.querySelector('.error-message#error-for-' + id );
    if (!message) {
        message = document.createElement('div');
        message.className = 'error-message';
        message.id = 'error-for-' + id;
        field.parentNode.insertBefore( message, field.nextSibling );
    }

    // Update error message
    message.innerHTML = error;

    // Show error message
    message.style.display = 'block';
    message.style.visibility = 'visible';

};</code></pre> 
      <p>To make sure that screen readers and other assistive technology know that our error message is associated with our field, we also need to add the <code>aria-describedby</code> role.</p> 
      <pre rel="JavaScript"><code class="language-javascript">var showError = function (field, error) {

    // Add error class to field
    field.classList.add('error');

    // Get field id or name
    var id = field.id || field.name;
    if (!id) return;

    // Check if error message field already exists
    // If not, create one
    var message = field.form.querySelector('.error-message#error-for-' + id );

    // Update error message

    // Show error message
    message.style.display = 'block';
    message.style.visibility = 'visible';

};</code></pre> 
      <h3>Style the error message</h3> 
      <p>We can use the <code>.error</code> and <code>.error-message</code> classes to style our form field and error message.</p> 
      <p>As a simple example, you may want to display a red border around fields with an error, and make the error message red and italicized.</p> 
      <pre rel="CSS"><code class="language-css">.error {
  border-color: red;
}

.error-message {
  color: red;
  font-style: italic;
}</code></pre> 
      <p data-height="489" data-theme-id="1" data-slug-hash="RgWWRo" data-default-tab="css,result" data-user="cferdinandi" data-embed-version="2" data-pen-title="Form Validation: Display the Error" class="codepen">See the Pen <a href="https://codepen.io/cferdinandi/pen/RgWWRo/">Form Validation: Display the Error</a> by Chris Ferdinandi (<a href="https://codepen.io/cferdinandi">@cferdinandi</a>) on <a href="https://codepen.io">CodePen</a>.</p> 
      <h3>Hide an error message</h3> 
      <p>Once we show an error, your visitor will (hopefully) fix it. Once the field validates, we need to remove the error message. Let's create another function, <code>removeError()</code>, and pass in the field. We'll call this function from event listener as well.</p> 
      <pre rel="JavaScript"><code class="language-javascript">// Remove the error message
var removeError = function (field) {
    // Remove the error message...
};

// Listen to all blur events
document.addEventListener('blur', function (event) {

    // Only run if the field is in a form to be validated
    if (!event.target.form.classList.contains('validate')) return;

    // Validate the field
    var error = event.target.validity;

    // If there's an error, show it
    if (error) {
        showError(event.target, error);
        return;
    }

    // Otherwise, remove any existing error message
    removeError(event.target);

}, true);</code></pre> 
      <p>In <code>removeError()</code>, we want to:</p> 
      <ol> 
       <li>Remove the error class from our field.</li> 
       <li>Remove the <code>aria-describedby</code> role from the field.</li> 
       <li>Hide any visible error messages in the DOM.</li> 
      </ol> 
      <p>Because we could have multiple forms on a page, and there's a chance those forms might have fields with the same name or ID (even though that's invalid, it happens), we're going to limit our search for the error message with <code>querySelector</code> the form our field is in rather than the entire document.</p> 
      <pre rel="JavaScript"><code class="language-javascript">// Remove the error message
var removeError = function (field) {

    // Remove error class to field
    field.classList.remove('error');

    // Remove ARIA role from the field
    field.removeAttribute('aria-describedby');

    // Get field id or name
    var id = field.id || field.name;
    if (!id) return;

    // Check if an error message is in the DOM
    var message = field.form.querySelector('.error-message#error-for-' + id + '');
    if (!message) return;

    // If so, hide it
    message.innerHTML = '';
    message.style.display = 'none';
    message.style.visibility = 'hidden';

};</code></pre> 
      <p data-height="559" data-theme-id="1" data-slug-hash="JJYYpY" data-default-tab="js,result" data-user="cferdinandi" data-embed-version="2" data-pen-title="Form Validation: Remove the Error After It's Fixed" class="codepen">See the Pen <a href="https://codepen.io/cferdinandi/pen/JJYYpY/">Form Validation: Remove the Error After It's Fixed</a> by Chris Ferdinandi (<a href="https://codepen.io/cferdinandi">@cferdinandi</a>) on <a href="https://codepen.io">CodePen</a>.</p> 
      <p><script async src="https://production-assets.codepen.io/assets/embed/ei.js"></script></p> 
      <p>If the field is a radio button or checkbox, we need to change how we add our error message to the DOM.</p> 
      <p>The field label often comes after the field, or wraps it entirely, for these types of inputs. Additionally, if the radio button is part of a group, we want the error to appear after the group rather than just the radio button.</p> 
      <p data-height="537" data-theme-id="0" data-slug-hash="eRpprw" data-default-tab="css,result" data-user="cferdinandi" data-embed-version="2" data-pen-title="Form Validation: Issues with Radio Buttons &amp; Checkboxes" class="codepen">See the Pen <a href="https://codepen.io/cferdinandi/pen/eRpprw/">Form Validation: Issues with Radio Buttons &amp; Checkboxes</a> by Chris Ferdinandi (<a href="https://codepen.io/cferdinandi">@cferdinandi</a>) on <a href="https://codepen.io">CodePen</a>.</p> 
      <p>First, we need to modify our <code>showError()</code> method. If the field <code>type</code> is <code>radio</code> <em>and</em> it has a <code>name</code>, we want get all radio buttons with that same <code>name</code> (ie. all other radio buttons in the group) and reset our <code>field</code> variable to the last one in the group.</p> 
      <pre rel="JavaScript"><code class="language-javascript">// Show the error message
var showError = function (field, error) {

    // Add error class to field
    field.classList.add('error');

    // If the field is a radio button and part of a group, error all and get the last item in the group
    if (field.type === 'radio' &amp;&amp; field.name) {
        var group = document.getElementsByName(field.name);
        if (group.length &gt; 0) {
            for (var i = 0; i &lt; group.length; i++) {
                // Only check fields in current form
                if (group[i].form !== field.form) continue;
                group[i].classList.add('error');
            }
            field = group[group.length - 1];
        }
    }

    ...

};</code></pre> 
      <p>When we go to inject our message into the DOM, we first want to check if the field type is <code>radio</code> or <code>checkbox</code>. If so, we want to get the field label and inject our message after it instead of after the field itself.</p> 
      <pre rel="JavaScript"><code class="language-javascript">// Show the error message
var showError = function (field, error) {

    ...

    // Check if error message field already exists
    // If not, create one
    var message = field.form.querySelector('.error-message#error-for-' + id );
    if (!message) {
        message = document.createElement('div');
        message.className = 'error-message';
        message.id = 'error-for-' + id;

        // If the field is a radio button or checkbox, insert error after the label
        var label;
        if (field.type === 'radio' || field.type ==='checkbox') {
            label = field.form.querySelector('label[for="' + id + '"]') || field.parentNode;
            if (label) {
                label.parentNode.insertBefore( message, label.nextSibling );
            }
        }

        // Otherwise, insert it after the field
        if (!label) {
            field.parentNode.insertBefore( message, field.nextSibling );
        }
    }

    ...

};</code></pre> 
      <p>When we go to remove the error, we similarly need to check if the field is a radio button that's part of a group, and if so, use the last radio button in that group to get the ID of our error message.</p> 
      <pre rel="JavaScript"><code class="language-javascript">// Remove the error message

    // Remove error class to field
    field.classList.remove('error');

    // If the field is a radio button and part of a group, remove error from all and get the last item in the group

      <p data-height="972" data-theme-id="1" data-slug-hash="dRYYgN" data-default-tab="js,result" data-user="cferdinandi" data-embed-version="2" data-pen-title="Form Validation: Fixing Radio Buttons &amp; Checkboxes" class="codepen">See the Pen <a href="https://codepen.io/cferdinandi/pen/dRYYgN/">Form Validation: Fixing Radio Buttons &amp; Checkboxes</a> by Chris Ferdinandi (<a href="https://codepen.io/cferdinandi">@cferdinandi</a>) on <a href="https://codepen.io">CodePen</a>.</p> 
      <h3>Checking all fields on submit</h3> 
      <p>When a visitor submits our form, we should first validate every field in the form and display error messages on any invalid fields. We should also bring the first field with an error into focus so that the visitor can immediately take action to correct it.</p> 
      <p>We'll do this by adding a listener for the <code>submit</code> event.</p> 
      <pre rel="JavaScript"><code class="language-javascript">// Check all fields on submit
document.addEventListener('submit', function (event) {
    // Validate all fields...
}, false);</code></pre> 
      <p>If the form has the <code>.validate</code> class, we'll get every field, loop through each one, and check for errors. We'll store the first invalid field we find to a variable and bring it into focus when we're done. If no errors are found, the form can submit normally.</p> 
      <pre rel="JavaScript"><code class="language-javascript">// Check all fields on submit
document.addEventListener('submit', function (event) {

    // Only run on forms flagged for validation
    if (!event.target.classList.contains('validate')) return;

    // Get all of the form elements
    var fields = event.target.elements;

    // Validate each field
    // Store the first field with an error to a variable so we can bring it into focus later
    var error, hasErrors;
    for (var i = 0; i &lt; fields.length; i++) {
        error = hasError(fields[i]);
        if (error) {
            showError(fields[i], error);
            if (!hasErrors) {
                hasErrors = fields[i];
            }
        }
    }

    // If there are errrors, don't submit form and focus on first element with error
    if (hasErrors) {
        event.preventDefault();
        hasErrors.focus();
    }

    // Otherwise, let the form submit normally
    // You could also bolt in an Ajax form submit process here

}, false);</code></pre> 
      <p data-height="531" data-theme-id="1" data-slug-hash="eRppxO" data-default-tab="js,result" data-user="cferdinandi" data-embed-version="2" data-pen-title="Form Validation: Validate on Submit" class="codepen">See the Pen <a href="https://codepen.io/cferdinandi/pen/eRppxO/">Form Validation: Validate on Submit</a> by Chris Ferdinandi (<a href="https://codepen.io/cferdinandi">@cferdinandi</a>) on <a href="https://codepen.io">CodePen</a>.</p> 
      <h3>Tying it all together</h3> 
      <p>Our finished script weight just 6kb (2.7kb minified). You can <a href="https://github.com/cferdinandi/validate">download a plugin version on GitHub</a>.</p> 
      <p>It works in all modern browsers and provides support IE support back to IE10. But, there are some browser gotchas…</p> 
      <ol> 
       <li>Because we can't have nice things, <a href="https://quirksmode.org/dom/forms/index.html#link2">not every browser supports every Validity State property</a>.</li> 
       <li>Internet Explorer is, of course, the main violator, though Edge does lack support for <code>tooLong</code> even though IE10+ supports it. Go figure.</li> 
      </ol> 
      <p>Here's the good news: with a lightweight polyfill (5kb, 2.7kb minified) we can extend our browser support all the way back to IE9, and add missing properties to partially supporting browsers, without having to touch any of our core code.</p> 
      <p>There is <em>one</em> exception to the IE9 support: radio buttons. IE9 doesn't support CSS3 selectors (like <code>[name="' + field.name + '"]</code>). We use that to make sure at least one radio button has been selected within a group. IE9 will always return an error.</p>
