           <br>the jCifs package offers the jcifs.http.NtlmHttpFilter which, when installed 
           <br>as a servlet filter, attempts authenticating throw the NTLM protocol. The 
           <br>identity of the authenticated user is propagated in the attributes of the 
           <br>request, so that the servlet or a further filter can eventually assign a 
           <br>role to the user, if Grinder needs it. 
           <br>
           <br>The setup of the filter is a bit tricky and may require some compromise 
           <br>among features, but then it is used to work as expected.

           <br>the jCifs package offers the jcifs.http.NtlmHttpFilter which, when installed 
           <br>as a servlet filter, attempts authenticating throw the NTLM protocol. The 
           <br>identity of the authenticated user is propagated in the attributes of the 
           <br>request, so that the servlet or a further filter can eventually assign a 
           <br>role to the user, if Grinder needs it. 
           <br>
           <br>The setup of the filter is a bit tricky and may require some compromise 
           <br>among features, but then it is used to work as expected.

So every time I am getting response code "proxy authorization 407". So I am trying to write a code by using 3rd party package so that Grinder will support NTLM authentication. 
           <br>
           <br>It sounds like you want to implement the client side of NTLM proxy 
           <br>authentication in this tool that you're using. 
           <br>
           <br>Proxy authentication is slightly different from authenticating with an 
           <br>HTTP server directly. I don't have a proxy server that supports NTLM 
           <br>(although I do need an exemplary one to implement NTLM proxy 
           <br>authentication elsewhere so if someone knows about NTLM proxies please 
           <br>let me know) so I cannot comment on the details but I think you 
           <br>basically just want to use jcifs.smb.client.NtlmContext.initSecContext 
           <br>with the jcifs.util.Base64 class in a loop to emit and consume tokens 
           <br>between the client and the proxy server. Once that completes 
           <br>succcessfully, the original request will go through. 
           <br>
           <br>However, you must first understand that NTLM is a multi-request 
           <br>process and figure out how that will integrate with this tool that 
           <br>you're using. That is very unclear. You have some research to do 
           <br>regarding the protocol and the tool. WireShark will be required for 
           <br>that.

&lt;<a href="/user/SendEmail.jtp?type=node&amp;node=2515223&amp;i=1" target="_top" rel="nofollow" link="external">[hidden email]</a>&gt; wrote:
&gt; I am posting this mail again because my exchange server gave some error during post for first time. If you have received this mail, then kindly ignore this and sorry for spamming your mail box.
&gt;
&gt; Hi,
&gt;
&gt; &nbsp; &nbsp; &nbsp; &nbsp;I am trying to replay back my Grinder script through network proxy authorization type NTLM. But the Grinder tool does not support NTLM authentication. So every time I am getting response code "proxy authorization 407". So I am trying to write a code by using 3rd party package so that Grinder will support NTLM authentication.

It sounds like you want to implement the client side of NTLM proxy
authentication in this tool that you're using.

Proxy authentication is slightly different from authenticating with an
HTTP server directly. I don't have a proxy server that supports NTLM
(although I do need an exemplary one to implement NTLM proxy
authentication elsewhere so if someone knows about NTLM proxies please
let me know) so I cannot comment on the details but I think you
basically just want to use jcifs.smb.client.NtlmContext.initSecContext
with the jcifs.util.Base64 class in a loop to emit and consume tokens
between the client and the proxy server. Once that completes
succcessfully, the original request will go through.

However, you must first understand that NTLM is a multi-request
process and figure out how that will integrate with this tool that
you're using. That is very unclear. You have some research to do
regarding the protocol and the tool. WireShark will be required for
that.

            Just for the record, I am also very interested in the same subject. 
           <br>Basically my issue is the following one, which might overlap yours and 
           <br>the OP's to some extent. 
           <br>
           <br>We have developed a browser applet in Java, which allows the user to 
           <br>select a series of files/directories locally on his workstation, and 
           <br>send them to a webserver as POSTs (one POST per file). &nbsp;The applet is 
           <br>smart enough to pick up proxy settings from the browser (if any), and 
           <br>apply them to its own POSTs if needed. 
           <br>For the POST part, we used the Apache Commons HTTPClient 3.x library. 
           <br>The POSTS are multipart/form-data, because in addition to the file 
           <br>itself, there are some additional parameters being posted each time. 
           <br>Generally, it works fine. 
           <br>
           <br>However, we have a problem when the POSTs have to go through a corporate 
           <br>proxy which requires NTLM authentication. 
           <br>The Apache HTTPClient library does not handle that. 
           <br>Sun's java.net.URL class handles this fine, and seems to "magically" do 
           <br>whatever's needed, with GETs and also with basic POSTs (url-encoded). 
           <br>But java.net.URL does not handle "multipart/form-data" posts, and it 
           <br>seems quite complicated to add this capability from the outside. 
           <br>So for the moment, we're stuck, and conjuring up many swearwords to the 
           <br>attention of whomever invented proxies with NTLM authentication. 
           <br>
           <br>One frustrating part in all this, is that the browser in which this 
           <br>applet runs, has no problem at all authenticating to the proxy and 
           <br>sending POSTs through it (to submit its html &lt;forms&gt; e.g.). But the 
           <br>applet running in the browser does. 
           <br>
           <br>André

            If I understand it correctly, Sun has a licence from Microsoft to use the NTLM algorithms, but only on Windows machines. So Sun's Windows JVM will work with NTLM-based proxy servers but JVMs on non-Windows platforms won't. (That was why I had to search out a proxy server which did know how to use NTLM on my behalf.) 
           <br>
           <br>The natural place to fix your problem would be inside the Apache HttpClient library, but you and the HttpClient committers won't be able to get your hands on the official NTLM documents or on a licence to implement NTLM authentication, I don't think.

           <br>Just for the record, I am also very interested in the same subject. 
           <br>Basically my issue is the following one, which might overlap yours and the OP's to some extent. 
           <br>
           <br>We have developed a browser applet in Java, which allows the user to select a series of files/directories locally on his workstation, and send them to a webserver as POSTs (one POST per file). &nbsp;The applet is smart enough to pick up proxy settings from the browser (if any), and apply them to its own POSTs if needed. 
           <br>For the POST part, we used the Apache Commons HTTPClient 3.x library. 
           <br>The POSTS are multipart/form-data, because in addition to the file itself, there are some additional parameters being posted each time. 
           <br>Generally, it works fine. 
           <br>
           <br>However, we have a problem when the POSTs have to go through a corporate proxy which requires NTLM authentication. 
           <br>The Apache HTTPClient library does not handle that. 
           <br>Sun's java.net.URL class handles this fine, and seems to "magically" do whatever's needed, with GETs and also with basic POSTs (url-encoded). 
           <br>But java.net.URL does not handle "multipart/form-data" posts, and it seems quite complicated to add this capability from the outside. 
           <br>So for the moment, we're stuck, and conjuring up many swearwords to the attention of whomever invented proxies with NTLM authentication. 
           <br>
           <br>One frustrating part in all this, is that the browser in which this applet runs, has no problem at all authenticating to the proxy and sending POSTs through it (to submit its html &lt;forms&gt; e.g.). But the applet running in the browser does.

           <br>Proxy authentication is slightly different from authenticating with an 
           <br>HTTP server directly. I don't have a proxy server that supports NTLM 
           <br>(although I do need an exemplary one to implement NTLM proxy 
           <br>authentication elsewhere so if someone knows about NTLM proxies please 
           <br>let me know) so I cannot comment on the details but I think you 
           <br>basically just want to use jcifs.smb.client.NtlmContext.initSecContext 
           <br>with the jcifs.util.Base64 class in a loop to emit and consume tokens 
           <br>between the client and the proxy server. Once that completes 
           <br>succcessfully, the original request will go through. 
           <br>
           <br>However, you must first understand that NTLM is a multi-request 
           <br>process and figure out how that will integrate with this tool that 
           <br>you're using. That is very unclear. You have some research to do 
           <br>regarding the protocol and the tool. WireShark will be required for 
           <br>that.

           <br>Hi Michael, 
           <br>
           <br>the MS Proxy was a server package which was meant to do what is supposed a 
           <br>proxy does, plus the NTLM transparent authentication. As usual in many proxy 
           <br>servers, it allowed to establish user and group policies and rules in 
           <br>accessing outside web resources through it.

           <br>product: you wouldn't get a copy even signing a MAPS agreement with MS, in 
           <br>example. To my knowledge, the version of the latest MS Proxy package is 2.0. 
           <br>I installed it once not early than five years ago. 
           <br>
           <br>Its "silent" withdrawal by MS is probably due to the fact that it had a very 
           <br>limited market impact from its very first release: most of the companies 
           <br>which could afford buying it had already move to xDSL lines dropping the old 
           <br>ISDN and POTS ones in order to connect to Internet, thereby reducing the 
           <br>appeal of MS Proxy. Also, the other benefits the MS Proxy could give (access 
           <br>control, logging, security) where interesting only for middle to big 
           <br>organizations, and they could however rely in other similar (open source) 
           <br>products (read: squid) which were also more powerful (squid, in example, can 
           <br>be plugged to an AV in order to block viruses and the like). Finally, squid 
           <br>implemented NTLM authentication well before the MS Proxy 2.0 version, 
           <br>eventually removing the last remaining "plus" of MS Proxy.

            <br>&gt;&gt; In particular I'm curious as to what Microsoft's 
            <br>&gt;&gt; proxy software is. I've tried to search around on their website but I 
            <br>&gt;&gt; just don't know enough about it to get a toe hold. Then again I 
            <br>&gt;&gt; haven't spent much time on it. 
            <br>&gt; 
            <br>&gt; Hi Michael, 
            <br>&gt; 
            <br>&gt; the MS Proxy was a server package which was meant to do what is supposed a 
            <br>&gt; proxy does, plus the NTLM transparent authentication. As usual in many proxy 
            <br>&gt; servers, it allowed to establish user and group policies and rules in 
            <br>&gt; accessing outside web resources through it. 
            <br>&gt; 
            <br>&gt; You were unsuccessful in finding more docs about it because it is an old 
            <br>&gt; product: you wouldn't get a copy even signing a MAPS agreement with MS, in 
            <br>&gt; example. To my knowledge, the version of the latest MS Proxy package is 2.0. 
            <br>&gt; I installed it once not early than five years ago. 
            <br>&gt; 
            <br>&gt; Its "silent" withdrawal by MS is probably due to the fact that it had a very 
            <br>&gt; limited market impact from its very first release: most of the companies 
            <br>&gt; which could afford buying it had already move to xDSL lines dropping the old 
            <br>&gt; ISDN and POTS ones in order to connect to Internet, thereby reducing the 
            <br>&gt; appeal of MS Proxy. Also, the other benefits the MS Proxy could give (access 
            <br>&gt; control, logging, security) where interesting only for middle to big 
            <br>&gt; organizations, and they could however rely in other similar (open source) 
            <br>&gt; products (read: squid) which were also more powerful (squid, in example, can 
            <br>&gt; be plugged to an AV in order to block viruses and the like). Finally, squid 
            <br>&gt; implemented NTLM authentication well before the MS Proxy 2.0 version, 
            <br>&gt; eventually removing the last remaining "plus" of MS Proxy. 
           </div>
           <br>Very interesting. 
           <br>
           <br>So does anyone even really using NTLM proxy authentication? 
           <br>
           <br>Does Squid even support NTLMv2? I would imaging it cannot unless it 
           <br>interfaced with Samba. 
           <br>
           <br>It sounds like NTLM proxy authentication is not as common as I first thought.

           </div>
           <br>Yes, they are: squid can adopt NTLM to authenticate proxy users. 
           <br>
           <br>In the full-fledged config, you usually run winbind in the same server 
           <br>hosting squid, and configure squid to use the wb_ntlmauth application (an 
           <br>external squid helper devoted to this). 
           <br>
           <br>
           <br>&gt; Does Squid even support NTLMv2? I would imaging it cannot unless it 
           <br>&gt; interfaced with Samba. 
           <br>
           <br>My last experience (which was two years ago in migrating the old MS Proxy to 
           <br>squid ;) ), says "no: only NTLMv1". 
           <br>
           <br>I see, however, that some development had been done in this area so I'm not 
           <br>sure about this. 
           <br>
           <br>
           <br>&gt; It sounds like NTLM proxy authentication is not as common as I first 
           <br>&gt; thought. 
           <br>
           <br>Well, I think that most of the companies running a proxy use the NTLM 
           <br>authentication. The fact is that web proxies are not that common. Many are 
           <br>the reasons, among which the fact that a proxy is one more thing to maintain 
           <br>and tune in a company. You really have the need to have your systems as safe 
           <br>as possible: actually most of the appeal of a WEB Proxy is to avoid people 
           <br>in a company to accidentally download stuff which could hurt the whole 
           <br>network. Then management likes also the fact they can control the time 
           <br>people can freely surf and can even limit surfing of sites not related to 
           <br>work. Finally they like the fact that they can track the sites visited by 
           <br>people, as a further way to know what happened when a virus eventually 
           <br>entered in the network (and possibly yellowing at the responsible). Please 
           <br>note that occasionally people don't like to have a proxy and can manage to 
           <br>dismiss it. This is due to the fact that many sites play (accidentally) 
           <br>dirty through a proxy and to the fact that most of the time management likes 
           <br>to enforce high security levels in jscripts and the like, often resulting in 
           <br>a frustrating user experience on the sites people so smoothly access from 
           <br>home. I think that if the company installing the proxy doesn't really know 
           <br>what they want to do with it, they'll dismiss it in a couple of months... 
           <br>
           <br>Transparent web proxy caching was probably used by some small provider, but 
           <br>it used not to be authenticated and actually the most part of internet 
           <br>traffic is due to p2p, not http... 
           <br>
           <br>In summary, few companies work on web proxies because there is a limited 
           <br>market need and you need to have the right channels to enter into it: proxy 
           <br>demanding orgs are mostly middle and big ones... 
           <br>
