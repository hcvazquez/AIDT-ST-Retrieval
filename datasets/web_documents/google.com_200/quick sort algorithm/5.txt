      <p><b>Quicksort</b> (sometimes called <b>partition-exchange sort</b>) is an <a href="/wiki/Algorithm_efficiency" class="mw-redirect" title="Algorithm efficiency">efficient</a> <a href="/wiki/Sorting_algorithm" title="Sorting algorithm">sorting algorithm</a>, serving as a systematic method for placing the elements of an <a href="/wiki/Array_data_structure" title="Array data structure">array</a> in order. Developed by <a href="/wiki/Tony_Hoare" title="Tony Hoare">Tony Hoare</a> in 1959<sup id="cite_ref-1" class="reference"><a href="#cite_note-1">[1]</a></sup> and published in 1961,<sup id="cite_ref-alg64_2-0" class="reference"><a href="#cite_note-alg64-2">[2]</a></sup> it is still a commonly used algorithm for sorting. When implemented well, it can be about two or three times faster than its main competitors, <a href="/wiki/Merge_sort" title="Merge sort">merge sort</a> and <a href="/wiki/Heapsort" title="Heapsort">heapsort</a>.<sup id="cite_ref-skiena_3-0" class="reference"><a href="#cite_note-skiena-3">[3]</a></sup><sup class="noprint Inline-Template" style="white-space:nowrap;">[<i><a href="/wiki/Category:Articles_contradicting_other_articles" title="Category:Articles contradicting other articles"><span title="This text contradicts material at the section 'Relation to other algorithms' on this page. (July 2017)">contradictory</span></a></i>]</sup><sup class="noprint Inline-Template" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Accuracy_dispute#Disputed_statement" title="Wikipedia:Accuracy dispute"><span title="The material near this tag is possibly inaccurate or nonfactual. (July 2017)">dubious</span></a> <span class="metadata">– <a href="/wiki/Talk:Quicksort#Dubious" title="Talk:Quicksort">discuss</a></span></i>]</sup></p> 
      <p>Quicksort is a <a href="/wiki/Comparison_sort" title="Comparison sort">comparison sort</a>, meaning that it can sort items of any type for which a "less-than" relation (formally, a <a href="/wiki/Total_order" title="Total order">total order</a>) is defined. In efficient implementations it is not a <a href="/wiki/Stable_sort" class="mw-redirect" title="Stable sort">stable sort</a>, meaning that the relative order of equal sort items is not preserved. Quicksort can operate <a href="/wiki/In-place_algorithm" title="In-place algorithm">in-place</a> on an array, requiring small additional amounts of <a href="/wiki/Main_memory" class="mw-redirect" title="Main memory">memory</a> to perform the sorting.</p> 
      <p><a href="/wiki/Analysis_of_algorithms" title="Analysis of algorithms">Mathematical analysis</a> of quicksort shows that, <a href="/wiki/Best,_worst_and_average_case" title="Best, worst and average case">on average</a>, the algorithm takes <a href="/wiki/Big_O_notation" title="Big O notation">O</a>(<i>n</i>&nbsp;log&nbsp;<i>n</i>) comparisons to sort <i>n</i> items. In the <a href="/wiki/Best,_worst_and_average_case" title="Best, worst and average case">worst case</a>, it makes O(<i>n</i><sup>2</sup>) comparisons, though this behavior is rare.</p>

      <h2><span class="mw-headline" id="History">History</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Quicksort&amp;action=edit&amp;section=1" title="Edit section: History">edit</a><span class="mw-editsection-bracket">]</span></span></h2> 
      <p>The quicksort algorithm was developed in 1959 by <a href="/wiki/Tony_Hoare" title="Tony Hoare">Tony Hoare</a> while in the <a href="/wiki/Soviet_Union" title="Soviet Union">Soviet Union</a>, as a visiting student at <a href="/wiki/Moscow_State_University" title="Moscow State University">Moscow State University</a>. At that time, Hoare worked in a project on <a href="/wiki/Machine_translation" title="Machine translation">machine translation</a> for the <a href="/wiki/National_Physical_Laboratory,_UK" class="mw-redirect" title="National Physical Laboratory, UK">National Physical Laboratory</a>. As a part of the translation process, he needed to sort the words of Russian sentences prior to looking them up in a Russian-English dictionary that was already sorted in alphabetic order on <a href="/wiki/Magnetic_tape_data_storage" title="Magnetic tape data storage">magnetic tape</a>.<sup id="cite_ref-4" class="reference"><a href="#cite_note-4">[4]</a></sup> After recognizing that his first idea, <a href="/wiki/Insertion_sort" title="Insertion sort">insertion sort</a>, would be slow, he quickly came up with a new idea that was Quicksort. He wrote a program in Mercury <a href="/wiki/Autocode" title="Autocode">Autocode</a> for the partition but couldn't write the program to account for the list of unsorted segments. On return to England, he was asked to write code for <a href="/wiki/Shellsort" title="Shellsort">Shellsort</a> as part of his new job. Hoare mentioned to his boss that he knew of a faster algorithm and his boss bet sixpence that he didn't. His boss ultimately accepted that he had lost the bet. Later, Hoare learned about <a href="/wiki/ALGOL" title="ALGOL">ALGOL</a> and its ability to do recursion that enabled him to publish the code in <a href="/wiki/Association_for_Computing_Machinery" title="Association for Computing Machinery">ACM</a>.<sup id="cite_ref-alg64_2-1" class="reference"><a href="#cite_note-alg64-2">[2]</a></sup><sup id="cite_ref-5" class="reference"><a href="#cite_note-5">[5]</a></sup><sup class="noprint Inline-Template" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Verifiability#Self-published_sources" title="Wikipedia:Verifiability"><span title="The material near this tag may rely on a self-published source. (July 2015)">self-published source?</span></a></i>]</sup></p> 
      <p>Quicksort gained widespread adoption, appearing, for example, in <a href="/wiki/Unix" title="Unix">Unix</a> as the default library sort subroutine. Hence, it lent its name to the <a href="/wiki/C_standard_library" title="C standard library">C standard library</a> subroutine <tt><a href="/wiki/Qsort" title="Qsort">qsort</a></tt><sup id="cite_ref-engineering_6-0" class="reference"><a href="#cite_note-engineering-6">[6]</a></sup> and in the reference implementation of <a href="/wiki/Java_(programming_language)" title="Java (programming language)">Java</a>.</p> 
      <p><a href="/wiki/Robert_Sedgewick_(computer_scientist)" title="Robert Sedgewick (computer scientist)">Robert Sedgewick</a>'s Ph.D. thesis in 1975 is considered a milestone in the study of Quicksort where he resolved many open problems related to the analysis of various pivot selection schemes including <a href="/wiki/Samplesort" title="Samplesort">Samplesort</a>, adaptive partitioning by Van Emden<sup id="cite_ref-7" class="reference"><a href="#cite_note-7">[7]</a></sup> as well as derivation of expected number of comparisons and swaps.<sup id="cite_ref-engineering_6-1" class="reference"><a href="#cite_note-engineering-6">[6]</a></sup> Bentley and McIlroy incorporated various improvements for use in programming libraries, including a technique to deal with equal elements and a pivot scheme known as <i>pseudomedian of nine,</i> where a sample of nine elements is divided into groups of three and then the median of the three medians from three groups is chosen.<sup id="cite_ref-engineering_6-2" class="reference"><a href="#cite_note-engineering-6">[6]</a></sup> <a href="/wiki/Jon_Bentley_(computer_scientist)" title="Jon Bentley (computer scientist)">Jon Bentley</a> described another simpler and compact partitioning scheme in his book <i><a href="/wiki/Jon_Bentley_(computer_scientist)" title="Jon Bentley (computer scientist)">Programming Pearls</a></i> that he attributed to Nico Lomuto. Later Bentley wrote that he used Hoare's version for years but never really understood it but Lomuto's version was simple enough to prove correct.<sup id="cite_ref-8" class="reference"><a href="#cite_note-8">[8]</a></sup> Bentley described Quicksort as the "most beautiful code I had ever written" in the same essay. Lomuto's partition scheme was also popularized by the textbook <i><a href="/wiki/Introduction_to_Algorithms" title="Introduction to Algorithms">Introduction to Algorithms</a></i> although it is inferior to Hoare's scheme because it does three times more swaps on average and degrades to <span class="texhtml"><i>O</i>(<i>n</i><sup>2</sup>)</span> runtime when all elements are equal.<sup id="cite_ref-:1_9-0" class="reference"><a href="#cite_note-:1-9">[9]</a></sup><sup class="noprint Inline-Template" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Verifiability#Self-published_sources" title="Wikipedia:Verifiability"><span title="The material near this tag may rely on a self-published source. (August 2015)">self-published source?</span></a></i>]</sup></p> 
      <p>In 2009, Vladimir Yaroslavskiy proposed the new dual pivot Quicksort implementation.<sup id="cite_ref-:0_10-0" class="reference"><a href="#cite_note-:0-10">[10]</a></sup> In the Java core library mailing lists, he initiated a discussion claiming his new algorithm to be superior to the runtime library’s sorting method, which was at that time based on the widely used and carefully tuned variant of classic Quicksort by Bentley and McIlroy.<sup id="cite_ref-11" class="reference"><a href="#cite_note-11">[11]</a></sup> Yaroslavskiy’s Quicksort has been chosen as the new default sorting algorithm in Oracle’s Java 7 runtime library after extensive empirical performance tests.<sup id="cite_ref-12" class="reference"><a href="#cite_note-12">[12]</a></sup></p> 
      <h2><span class="mw-headline" id="Algorithm">Algorithm</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Quicksort&amp;action=edit&amp;section=2" title="Edit section: Algorithm">edit</a><span class="mw-editsection-bracket">]</span></span></h2> 
      <div class="thumb tright"> 
       <div class="thumbinner" style="width:202px;">
        <a href="/wiki/File:Quicksort-diagram.svg" class="image"><img alt="" src="//upload.wikimedia.org/wikipedia/commons/thumb/a/af/Quicksort-diagram.svg/200px-Quicksort-diagram.svg.png" width="200" height="400" class="thumbimage" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/a/af/Quicksort-diagram.svg/300px-Quicksort-diagram.svg.png 1.5x, //upload.wikimedia.org/wikipedia/commons/thumb/a/af/Quicksort-diagram.svg/400px-Quicksort-diagram.svg.png 2x" data-file-width="400" data-file-height="800"></a> 
        <div class="thumbcaption"> 
         <div class="magnify">
          <a href="/wiki/File:Quicksort-diagram.svg" class="internal" title="Enlarge"></a>
         </div> Full example of quicksort on a random set of numbers. The shaded element is the pivot. It is always chosen as the last element of the partition. However, always choosing the last element in the partition as the pivot in this way results in poor performance (
         <span class="texhtml"><i>O</i>(<i>n</i>²)</span>) on 
         <i>already sorted</i> arrays, or arrays of identical elements. Since sub-arrays of sorted / identical elements crop up a lot towards the end of a sorting procedure on a large set, versions of the quicksort algorithm that choose the pivot as the middle element run much more quickly than the algorithm described in this diagram on large sets of numbers.
        </div> 
       </div> 
      </div> 
      <p>Quicksort is a <a href="/wiki/Divide_and_conquer_algorithm" title="Divide and conquer algorithm">divide and conquer algorithm</a>. Quicksort first divides a large array into two smaller sub-arrays: the low elements and the high elements. Quicksort can then recursively sort the sub-arrays.</p> 
      <p>The steps are:</p> 
      <ol> 
       <li>Pick an element, called a <i>pivot</i>, from the array.</li> 
       <li><i>Partitioning</i>: reorder the array so that all elements with values less than the pivot come before the pivot, while all elements with values greater than the pivot come after it (equal values can go either way). After this partitioning, the pivot is in its final position. This is called the <i>partition</i> operation.</li> 
       <li><a href="/wiki/Recursion_(computer_science)" title="Recursion (computer science)">Recursively</a> apply the above steps to the sub-array of elements with smaller values and separately to the sub-array of elements with greater values.</li> 
      </ol> 
      <p>The base case of the recursion is arrays of size zero or one, which never need to be sorted.</p> 
      <p>The pivot selection and partitioning steps can be done in several different ways; the choice of specific implementation schemes greatly affects the algorithm's performance.</p> 
      <h3><span class="mw-headline" id="Lomuto_partition_scheme">Lomuto partition scheme</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Quicksort&amp;action=edit&amp;section=3" title="Edit section: Lomuto partition scheme">edit</a><span class="mw-editsection-bracket">]</span></span></h3> 
      <p>This scheme is attributed to Nico Lomuto and popularized by Bentley in his book <i>Programming Pearls</i><sup id="cite_ref-:3_13-0" class="reference"><a href="#cite_note-:3-13">[13]</a></sup> and Cormen <i>et al.</i> in their book <i><a href="/wiki/Introduction_to_Algorithms" title="Introduction to Algorithms">Introduction to Algorithms</a></i>.<sup id="cite_ref-:2_14-0" class="reference"><a href="#cite_note-:2-14">[14]</a></sup> This scheme chooses a pivot that is typically the last element in the array. The algorithm maintains index <span style="font-family: monospace, monospace;">i</span> as it scans the array using another index <span style="font-family: monospace, monospace;">j</span> such that the elements <span style="font-family: monospace, monospace;">lo</span> through <span style="font-family: monospace, monospace;">i</span> (inclusive) are less than or equal to the pivot, and the elements <span style="font-family: monospace, monospace;">i+1</span> through <span style="font-family: monospace, monospace;">j-1</span> (inclusive) are greater than the pivot. As this scheme is more compact and easy to understand, it is frequently used in introductory material, although it is less efficient than Hoare's original scheme.<sup id="cite_ref-15" class="reference"><a href="#cite_note-15">[15]</a></sup> This scheme degrades to <span class="texhtml"><i>O</i>(<i>n</i><sup>2</sup>)</span> when the array is already sorted as well as when the array has all equal elements.<sup id="cite_ref-:1_9-1" class="reference"><a href="#cite_note-:1-9">[9]</a></sup> There have been various variants proposed to boost performance including various ways to select pivot, deal with equal elements, use other sorting algorithms such as <a href="/wiki/Insertion_sort" title="Insertion sort">Insertion sort</a> for small arrays and so on. In <a href="/wiki/Pseudocode" title="Pseudocode">pseudocode</a>, a quicksort that sorts elements <span style="font-family: monospace, monospace;">lo</span> through <span style="font-family: monospace, monospace;">hi</span> (inclusive) of an array <span class="texhtml mvar" style="font-style:italic;">A</span> can be expressed as:<sup id="cite_ref-:2_14-1" class="reference"><a href="#cite_note-:2-14">[14]</a></sup></p> 
      <pre>
<b>algorithm</b> quicksort(A, lo, hi) <b>is</b>
    <b>if</b> lo &lt; hi <b>then</b>
        p := partition(A, lo, hi)
        quicksort(A, lo, p – 1)
        quicksort(A, p + 1, hi)

<b>algorithm</b> partition(A, lo, hi) <b>is</b>
    pivot := A[hi]
    i := lo - 1    
    <b>for</b> j := lo <b>to</b> hi - 1 <b>do</b>
        <b>if</b> A[j] &lt; pivot <b>then</b>
            i := i + 1
            swap A[i] with A[j]
    <b>if</b> A[hi] &lt; A[i + 1] <b>then</b>
        swap A[i + 1] with A[hi]
    <b>return</b> i + 1
</pre> 
      <p>Sorting the entire array is accomplished by <span style="font-family: monospace, monospace;">quicksort(A, 0, length(A) - 1)</span>.</p> 
      <h3><span class="mw-headline" id="Hoare_partition_scheme">Hoare partition scheme</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Quicksort&amp;action=edit&amp;section=4" title="Edit section: Hoare partition scheme">edit</a><span class="mw-editsection-bracket">]</span></span></h3> 
      <p>The original partition scheme described by C.A.R. Hoare uses two indices that start at the ends of the array being partitioned, then move toward each other, until they detect an inversion: a pair of elements, one greater than or equal to the pivot, one lesser or equal, that are in the wrong order relative to each other. The inverted elements are then swapped.<sup id="cite_ref-16" class="reference"><a href="#cite_note-16">[16]</a></sup> When the indices meet, the algorithm stops and returns the final index. There are many variants of this algorithm, for example, selecting pivot from <tt>A[hi]</tt> instead of <tt>A[lo]</tt>. Hoare's scheme is more efficient than Lomuto's partition scheme because it does three times fewer swaps on average, and it creates efficient partitions even when all values are equal.<sup id="cite_ref-:1_9-2" class="reference"><a href="#cite_note-:1-9">[9]</a></sup><sup class="noprint Inline-Template" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Verifiability#Self-published_sources" title="Wikipedia:Verifiability"><span title="The material near this tag may rely on a self-published source. (August 2015)">self-published source?</span></a></i>]</sup> Like Lomuto's partition scheme, Hoare partitioning also causes Quicksort to degrade to <span class="texhtml"><i>O</i>(<i>n</i><sup>2</sup>)</span> when the input array is already sorted; it also doesn't produce a stable sort. Note that in this scheme, the pivot's final location is not necessarily at the index that was returned, and the next two segments that the main algorithm recurs on are <span style="font-family: monospace, monospace;">(lo..p)</span> and <span style="font-family: monospace, monospace;">(p+1..hi)</span> as opposed to <span style="font-family: monospace, monospace;">(lo..p-1)</span> and <span style="font-family: monospace, monospace;">(p+1..hi)</span> as in Lomuto's scheme. However, the partitioning algorithm guarantees <span class="nowrap"><span style="font-family: monospace, monospace;">lo ? p &lt; hi</span></span> which implies both resulting partitions are non-empty, hence there's no risk of infinite recursion. In <a href="/wiki/Pseudocode" title="Pseudocode">pseudocode</a>,<sup id="cite_ref-:2_14-2" class="reference"><a href="#cite_note-:2-14">[14]</a></sup></p> 
      <pre>
<b>algorithm</b> quicksort(A, lo, hi) <b>is</b>
    <b>if</b> lo &lt; hi <b>then</b>
        p := partition(A, lo, hi)
        quicksort(A, lo, p)
        quicksort(A, p + 1, hi)

<b>algorithm</b> partition(A, lo, hi) <b>is</b>
    pivot := A[lo]
    i := lo - 1
    j := hi + 1
    <b>loop forever</b>
        <b>do</b>
            i := i + 1
        <b>while</b> A[i] &lt; pivot

        <b>do</b>
            j := j - 1
        <b>while</b> A[j] &gt; pivot

        <b>if</b> i &gt;= j <b>then</b>
            <b>return</b> j

        swap A[i] with A[j]
</pre> 
      <p>The entire array is sorted by <span style="font-family: monospace, monospace;">quicksort(A, 0, length(A)-1)</span>.</p> 
      <h3><span class="mw-headline" id="Implementation_issues">Implementation issues</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Quicksort&amp;action=edit&amp;section=5" title="Edit section: Implementation issues">edit</a><span class="mw-editsection-bracket">]</span></span></h3> 
      <h4><span class="mw-headline" id="Choice_of_pivot">Choice of pivot</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Quicksort&amp;action=edit&amp;section=6" title="Edit section: Choice of pivot">edit</a><span class="mw-editsection-bracket">]</span></span></h4> 
      <p>In the very early versions of quicksort, the leftmost element of the partition would often be chosen as the pivot element. Unfortunately, this causes worst-case behavior on already sorted arrays, which is a rather common use-case. The problem was easily solved by choosing either a random index for the pivot, choosing the middle index of the partition or (especially for longer partitions) choosing the <a href="/wiki/Median" title="Median">median</a> of the first, middle and last element of the partition for the pivot (as recommended by <a href="/wiki/Robert_Sedgewick_(computer_scientist)" title="Robert Sedgewick (computer scientist)">Sedgewick</a>).<sup id="cite_ref-sedgewickBook_17-0" class="reference"><a href="#cite_note-sedgewickBook-17">[17]</a></sup> This "median-of-three" rule counters the case of sorted (or reverse-sorted) input, and gives a better estimate of the optimal pivot (the true median) than selecting any single element, when no information about the ordering of the input is known.</p> 
      <p>Specifically, the expected number of comparisons needed to sort <span class="texhtml mvar" style="font-style:italic;">n</span> elements (see <a href="#Analysis_of_randomized_quicksort">§&nbsp;Analysis of randomized quicksort</a>) with random pivot selection is <span class="texhtml">1.386 <i>n</i> log <i>n</i></span>. Median-of-three pivoting brings this down to <span class="texhtml"><a href="/wiki/Binomial_coefficient" title="Binomial coefficient"><i>C</i></a><sub><i>n</i>, 2</sub> ? 1.188 <i>n</i> log <i>n</i></span>, at the expense of a three-percent increase in the expected number of swaps.<sup id="cite_ref-engineering_6-3" class="reference"><a href="#cite_note-engineering-6">[6]</a></sup> An even stronger pivoting rule, for larger arrays, is to pick the <a href="/wiki/Ninther" class="mw-redirect" title="Ninther">ninther</a>, a recursive median-of-three (Mo3), defined as<sup id="cite_ref-engineering_6-4" class="reference"><a href="#cite_note-engineering-6">[6]</a></sup></p> 
      <dl> 
       <dd>
        <span class="texhtml">ninther(<i>a</i>) = median(Mo3(first ? of <i>a</i>), Mo3(middle ? of <i>a</i>), Mo3(final ? of <i>a</i>))</span>
       </dd> 
      </dl> 
      <p>Selecting a pivot element is also complicated by the existence of <a href="/wiki/Integer_overflow" title="Integer overflow">integer overflow</a>. If the boundary indices of the subarray being sorted are sufficiently large, the naïve expression for the middle index, <span class="texhtml">(<i>lo</i> + <i>hi</i>)/2</span>, will cause overflow and provide an invalid pivot index. This can be overcome by using, for example, <span class="texhtml"><i>lo</i> + (<i>hi</i>?<i>lo</i>)/2</span> to index the middle element, at the cost of more complex arithmetic. Similar issues arise in some other methods of selecting the pivot element.</p> 
      <h4><span class="mw-headline" id="Repeated_elements">Repeated elements</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Quicksort&amp;action=edit&amp;section=7" title="Edit section: Repeated elements">edit</a><span class="mw-editsection-bracket">]</span></span></h4> 
      <p>With a partitioning algorithm such as the ones described above (even with one that chooses good pivot values), quicksort exhibits poor performance for inputs that contain many repeated elements. The problem is clearly apparent when all the input elements are equal: at each recursion, the left partition is empty (no input values are less than the pivot), and the right partition has only decreased by one element (the pivot is removed). Consequently, the algorithm takes <a href="/wiki/Quadratic_time" class="mw-redirect" title="Quadratic time">quadratic time</a> to sort an array of equal values.</p> 
      <p>To solve this problem (sometimes called the <a href="/wiki/Dutch_national_flag_problem" title="Dutch national flag problem">Dutch national flag problem</a><sup id="cite_ref-engineering_6-5" class="reference"><a href="#cite_note-engineering-6">[6]</a></sup>), an alternative linear-time partition routine can be used that separates the values into three groups: values less than the pivot, values equal to the pivot, and values greater than the pivot. (Bentley and McIlroy call this a "fat partition" and note that it was already implemented in the <span style="font-family: monospace, monospace;"><a href="/wiki/Qsort" title="Qsort">qsort</a></span> of <a href="/wiki/Version_7_Unix" title="Version 7 Unix">Version 7 Unix</a>.<sup id="cite_ref-engineering_6-6" class="reference"><a href="#cite_note-engineering-6">[6]</a></sup>) The values equal to the pivot are already sorted, so only the less-than and greater-than partitions need to be recursively sorted. In pseudocode, the quicksort algorithm becomes</p> 
      <pre>
<b>algorithm</b> quicksort(A, lo, hi) <b>is</b>
    <b>if</b> lo &lt; hi <b>then</b>
        p := pivot(A, lo, hi)
        left, right := partition(A, p, lo, hi)  <i>// note: multiple return values</i>
        quicksort(A, lo, left - 1)
        quicksort(A, right + 1, hi)
</pre> 
      <p>The <code>partition</code> algorithm returns indices to the first ('leftmost') and to the last ('rightmost') item of the middle partition. Every item of the partition is equal to <code>p</code> and is therefore sorted. Consequently the items of the partition need not be included in the recursive calls to <code>quicksort</code>.</p> 
      <p>The best case for the algorithm now occurs when all elements are equal (or are chosen from a small set of <span class="texhtml"><i>k</i> ? <i>n</i></span> elements). In the case of all equal elements, the modified quicksort will perform only two recursive calls on empty subarrays and thus finish in linear time (assuming the <code>partition</code> subroutine takes no longer than linear time).</p> 
      <h4><span class="mw-headline" id="Optimizations">Optimizations</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Quicksort&amp;action=edit&amp;section=8" title="Edit section: Optimizations">edit</a><span class="mw-editsection-bracket">]</span></span></h4> 
      <p>Two other important optimizations, also suggested by Sedgewick and widely used in practice are:<sup id="cite_ref-glibc_qsort_18-0" class="reference"><a href="#cite_note-glibc_qsort-18">[18]</a></sup><sup id="cite_ref-19" class="reference"><a href="#cite_note-19">[19]</a></sup></p> 
      <ul> 
       <li>To make sure at most <span class="texhtml"><i>O</i>(log <i>n</i>)</span> space is used, <a href="https://en.wiktionary.org/wiki/recurse" class="extiw" title="wikt:recurse">recurse</a> first into the smaller side of the partition, then use a <a href="/wiki/Tail_call" title="Tail call">tail call</a> to recurse into the other.</li> 
       <li>When the number of elements is below some threshold (perhaps ten elements), switch to a non-recursive sorting algorithm such as <a href="/wiki/Insertion_sort" title="Insertion sort">insertion sort</a> that performs fewer swaps, comparisons or other operations on such small arrays. The ideal 'threshold' will vary based on the details of the specific implementation.</li> 
       <li>An older variant of the previous optimization: when the number of elements is less than the threshold <span class="texhtml mvar" style="font-style:italic;">k</span>, simply stop; then after the whole array has been processed, perform insertion sort on it. Stopping the recursion early leaves the array <span class="texhtml mvar" style="font-style:italic;">k</span>-sorted, meaning that each element is at most <span class="texhtml mvar" style="font-style:italic;">k</span> positions away from its final sorted position. In this case, insertion sort takes <span class="texhtml"><i>O</i>(<i>kn</i>)</span> time to finish the sort, which is linear if <span class="texhtml mvar" style="font-style:italic;">k</span> is a constant.<sup id="cite_ref-sedgewickQsortPaper_20-0" class="reference"><a href="#cite_note-sedgewickQsortPaper-20">[20]</a></sup><sup id="cite_ref-:3_13-1" class="reference"><a href="#cite_note-:3-13">[13]</a></sup><sup class="reference" style="white-space:nowrap;">:117</sup> Compared to the "many small sorts" optimization, this version may execute fewer instructions, but it makes suboptimal use of the <a href="/wiki/Cache_memory" class="mw-redirect" title="Cache memory">cache memories</a> in modern computers.<sup id="cite_ref-LaMarca1999_21-0" class="reference"><a href="#cite_note-LaMarca1999-21">[21]</a></sup></li> 
      </ul> 
      <h4><span class="mw-headline" id="Parallelization">Parallelization</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Quicksort&amp;action=edit&amp;section=9" title="Edit section: Parallelization">edit</a><span class="mw-editsection-bracket">]</span></span></h4> 
      <p>Quicksort's divide-and-conquer formulation makes it amenable to <a href="/wiki/Parallel_algorithm" title="Parallel algorithm">parallelization</a> using <a href="/wiki/Task_parallelism" title="Task parallelism">task parallelism</a>. The partitioning step is accomplished through the use of a <a href="/wiki/Prefix_sum" title="Prefix sum">parallel prefix sum</a> algorithm to compute an index for each array element in its section of the partitioned array.<sup id="cite_ref-22" class="reference"><a href="#cite_note-22">[22]</a></sup><sup id="cite_ref-23" class="reference"><a href="#cite_note-23">[23]</a></sup> Given an array of size <span class="texhtml mvar" style="font-style:italic;">n</span>, the partitioning step performs <span class="texhtml">O(<i>n</i>)</span> work in <span class="texhtml"><i>O</i>(log <i>n</i>)</span> time and requires <span class="texhtml">O(<i>n</i>)</span> additional scratch space. After the array has been partitioned, the two partitions can be sorted recursively in parallel. Assuming an ideal choice of pivots, parallel quicksort sorts an array of size <span class="texhtml mvar" style="font-style:italic;">n</span> in <span class="texhtml">O(<i>n</i> log <i>n</i>)</span> work in <span class="texhtml">O(log² <i>n</i>)</span> time using <span class="texhtml">O(<i>n</i>)</span> additional space.</p> 
      <p>Quicksort has some disadvantages when compared to alternative sorting algorithms, like <a href="/wiki/Merge_sort" title="Merge sort">merge sort</a>, which complicate its efficient parallelization. The depth of quicksort's divide-and-conquer tree directly impacts the algorithm's scalability, and this depth is highly dependent on the algorithm's choice of pivot. Additionally, it is difficult to parallelize the partitioning step efficiently in-place. The use of scratch space simplifies the partitioning step, but increases the algorithm's memory footprint and constant overheads.</p> 
      <p>Other more sophisticated parallel sorting algorithms can achieve even better time bounds.<sup id="cite_ref-24" class="reference"><a href="#cite_note-24">[24]</a></sup> For example, in 1991 David Powers described a parallelized quicksort (and a related <a href="/wiki/Radix_sort" title="Radix sort">radix sort</a>) that can operate in <span class="texhtml"><i>O</i>(log <i>n</i>)</span> time on a CRCW <a href="/wiki/Parallel_Random_Access_Machine" class="mw-redirect" title="Parallel Random Access Machine">PRAM</a> with <span class="texhtml mvar" style="font-style:italic;">n</span> processors by performing partitioning implicitly.<sup id="cite_ref-25" class="reference"><a href="#cite_note-25">[25]</a></sup></p> 
      <h2><span class="mw-headline" id="Formal_analysis">Formal analysis</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Quicksort&amp;action=edit&amp;section=10" title="Edit section: Formal analysis">edit</a><span class="mw-editsection-bracket">]</span></span></h2> 
      <h3><span class="mw-headline" id="Worst-case_analysis">Worst-case analysis</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Quicksort&amp;action=edit&amp;section=11" title="Edit section: Worst-case analysis">edit</a><span class="mw-editsection-bracket">]</span></span></h3> 
      <p>The most unbalanced partition occurs when the pivot divides the list into two sublists of sizes <span class="texhtml">0</span> and <span class="texhtml"><i>n</i> ? 1</span>. This may occur if the pivot happens to be the smallest or largest element in the list, or in some implementations (e.g., the Lomuto partition scheme as described above) when all the elements are equal.</p>

      <h3><span class="mw-headline" id="Best-case_analysis">Best-case analysis</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Quicksort&amp;action=edit&amp;section=12" title="Edit section: Best-case analysis">edit</a><span class="mw-editsection-bracket">]</span></span></h3> 
      <p>In the most balanced case, each time we perform a partition we divide the list into two nearly equal pieces. This means each recursive call processes a list of half the size. Consequently, we can make only <span class="texhtml">log<sub>2</sub> <i>n</i></span> nested calls before we reach a list of size 1. This means that the depth of the <a href="/wiki/Call_stack" title="Call stack">call tree</a> is <span class="texhtml">log<sub>2</sub> <i>n</i></span>. But no two calls at the same level of the call tree process the same part of the original list; thus, each level of calls needs only <span class="texhtml"><i>O</i>(<i>n</i>)</span> time all together (each call has some constant overhead, but since there are only <span class="texhtml"><i>O</i>(<i>n</i>)</span> calls at each level, this is subsumed in the <span class="texhtml"><i>O</i>(<i>n</i>)</span> factor). The result is that the algorithm uses only <span class="texhtml"><i>O</i>(<i>n</i> log <i>n</i>)</span> time.</p> 
      <h3><span class="mw-headline" id="Average-case_analysis">Average-case analysis</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Quicksort&amp;action=edit&amp;section=13" title="Edit section: Average-case analysis">edit</a><span class="mw-editsection-bracket">]</span></span></h3> 
      <p>To sort an array of <span class="texhtml mvar" style="font-style:italic;">n</span> distinct elements, quicksort takes <span class="texhtml"><i>O</i>(<i>n</i> log <i>n</i>)</span> time in expectation, averaged over all <span class="texhtml"><i>n</i>!</span> permutations of <span class="texhtml mvar" style="font-style:italic;">n</span> elements with <a href="/wiki/Uniform_distribution_(discrete)" class="mw-redirect" title="Uniform distribution (discrete)">equal probability</a>. We list here three common proofs to this claim providing different insights into quicksort's workings.</p> 
      <h4><span class="mw-headline" id="Using_percentiles">Using percentiles</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Quicksort&amp;action=edit&amp;section=14" title="Edit section: Using percentiles">edit</a><span class="mw-editsection-bracket">]</span></span></h4>

      <p>When the input is a random permutation, the pivot has a random rank, and so it is not guaranteed to be in the middle 50 percent. However, when we start from a random permutation, in each recursive call the pivot has a random rank in its list, and so it is in the middle 50 percent about half the time. That is good enough. Imagine that you flip a coin: heads means that the rank of the pivot is in the middle 50 percent, tail means that it isn't. Imagine that you are flipping a coin over and over until you get <span class="texhtml mvar" style="font-style:italic;">k</span> heads. Although this could take a long time, on average only <span class="texhtml">2<i>k</i></span> flips are required, and the chance that you won't get <span class="texhtml mvar" style="font-style:italic;">k</span> heads after <span class="texhtml">100<i>k</i></span> flips is highly improbable (this can be made rigorous using Chernoff bounds). Note that the algorithm does not have to verify that the pivot is in the middle half—if we hit it any constant fraction of the times, that is enough for the desired complexity.</p>

      <p>Solving the recurrence gives <span class="texhtml"><i>C</i>(<i>n</i>) = 2<i>n</i> ln <i>n</i> ? 1.39<i>n</i> log? <i>n</i></span>.</p> 
      <p>This means that, on average, quicksort performs only about 39% worse than in its best case. In this sense, it is closer to the best case than the worst case. Also note that a <a href="/wiki/Comparison_sort" title="Comparison sort">comparison sort</a> cannot use less than <span class="texhtml">log?(<i>n</i>!)</span> comparisons on average to sort <span class="texhtml mvar" style="font-style:italic;">n</span> items (as <a href="/wiki/Comparison_sort#Lower_bound_for_the_average_number_of_comparisons" title="Comparison sort">explained in the article Comparison sort</a>) and in case of large <span class="texhtml mvar" style="font-style:italic;">n</span>, <a href="/wiki/Stirling%27s_approximation" title="Stirling's approximation">Stirling's approximation</a> yields <span class="texhtml">log?(<i>n</i>!) ? <i>n</i>(log? <i>n</i> ? log? <i>e</i>)</span>, so quicksort is not much worse than an ideal comparison sort. This fast average runtime is another reason for quicksort's practical dominance over other sorting algorithms.</p>

      <h3><span class="mw-headline" id="Space_complexity">Space complexity</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Quicksort&amp;action=edit&amp;section=17" title="Edit section: Space complexity">edit</a><span class="mw-editsection-bracket">]</span></span></h3> 
      <p>The space used by quicksort depends on the version used.</p> 
      <p>The in-place version of quicksort has a space complexity of <span class="texhtml"><i>O</i>(log <i>n</i>)</span>, even in the worst case, when it is carefully implemented using the following strategies:</p> 
      <ul> 
       <li>in-place partitioning is used. This unstable partition requires <span class="texhtml"><i>O</i>(1)</span> space.</li> 
       <li>After partitioning, the partition with the fewest elements is (recursively) sorted first, requiring at most <span class="texhtml"><i>O</i>(log <i>n</i>)</span> space. Then the other partition is sorted using <a href="/wiki/Tail_recursion" class="mw-redirect" title="Tail recursion">tail recursion</a> or iteration, which doesn't add to the call stack. This idea, as discussed above, was described by <a href="/wiki/Robert_Sedgewick_(computer_scientist)" title="Robert Sedgewick (computer scientist)">R. Sedgewick</a>, and keeps the stack depth bounded by <span class="texhtml"><i>O</i>(log <i>n</i>)</span>.<sup id="cite_ref-sedgewickBook_17-1" class="reference"><a href="#cite_note-sedgewickBook-17">[17]</a></sup><sup id="cite_ref-sedgewickQsortPaper_20-1" class="reference"><a href="#cite_note-sedgewickQsortPaper-20">[20]</a></sup></li> 
      </ul> 
      <p>Quicksort with in-place and unstable partitioning uses only constant additional space before making any recursive call. Quicksort must store a constant amount of information for each nested recursive call. Since the best case makes at most <span class="texhtml"><i>O</i>(log <i>n</i>)</span> nested recursive calls, it uses <span class="texhtml"><i>O</i>(log <i>n</i>)</span> space. However, without Sedgewick's trick to limit the recursive calls, in the worst case quicksort could make <span class="texhtml"><i>O</i>(<i>n</i>)</span> nested recursive calls and need <span class="texhtml"><i>O</i>(<i>n</i>)</span> auxiliary space.</p> 
      <p>From a bit complexity viewpoint, variables such as <i>lo</i> and <i>hi</i> do not use constant space; it takes <span class="texhtml"><i>O</i>(log <i>n</i>)</span> bits to index into a list of <span class="texhtml mvar" style="font-style:italic;">n</span> items. Because there are such variables in every stack frame, quicksort using Sedgewick's trick requires <span class="texhtml"><i>O</i>((log <i>n</i>)²)</span> bits of space. This space requirement isn't too terrible, though, since if the list contained distinct elements, it would need at least <span class="texhtml"><i>O</i>(<i>n</i> log <i>n</i>)</span> bits of space.</p> 
      <p>Another, less common, not-in-place, version of quicksort uses <span class="texhtml"><i>O</i>(<i>n</i>)</span> space for working storage and can implement a stable sort. The working storage allows the input array to be easily partitioned in a stable manner and then copied back to the input array for successive recursive calls. Sedgewick's optimization is still appropriate.</p> 
      <h2><span class="mw-headline" id="Relation_to_other_algorithms">Relation to other algorithms</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Quicksort&amp;action=edit&amp;section=18" title="Edit section: Relation to other algorithms">edit</a><span class="mw-editsection-bracket">]</span></span></h2> 
      <p>Quicksort is a space-optimized version of the <a href="/wiki/Binary_tree_sort" class="mw-redirect" title="Binary tree sort">binary tree sort</a>. Instead of inserting items sequentially into an explicit tree, quicksort organizes them concurrently into a tree that is implied by the recursive calls. The algorithms make exactly the same comparisons, but in a different order. An often desirable property of a <a href="/wiki/Sorting_algorithm" title="Sorting algorithm">sorting algorithm</a> is stability – that is the order of elements that compare equal is not changed, allowing controlling order of multikey tables (e.g. directory or folder listings) in a natural way. This property is hard to maintain for in situ (or in place) quicksort (that uses only constant additional space for pointers and buffers, and <span class="nowrap"><i>O</i>(log <i>n</i>)</span> additional space for the management of explicit or implicit recursion). For variant quicksorts involving extra memory due to representations using pointers (e.g. lists or trees) or files (effectively lists), it is trivial to maintain stability. The more complex, or disk-bound, data structures tend to increase time cost, in general making increasing use of virtual memory or disk.</p> 
      <p>The most direct competitor of quicksort is <a href="/wiki/Heapsort" title="Heapsort">heapsort</a>. Heapsort's running time is <span class="nowrap"><i>O</i>(<i>n</i> log <i>n</i>)</span>, but heapsort's average running time is usually considered slower than in-place quicksort. This result is debatable; some publications indicate the opposite.<sup id="cite_ref-26" class="reference"><a href="#cite_note-26">[26]</a></sup><sup id="cite_ref-27" class="reference"><a href="#cite_note-27">[27]</a></sup> <a href="/wiki/Introsort" title="Introsort">Introsort</a> is a variant of quicksort that switches to heapsort when a bad case is detected to avoid quicksort's worst-case running time.</p> 
      <p>Quicksort also competes with <a href="/wiki/Mergesort" class="mw-redirect" title="Mergesort">mergesort</a>, another <span class="nowrap"><i>O</i>(<i>n</i> log <i>n</i>)</span> sorting algorithm. Mergesort is a <a href="/wiki/Stable_sort" class="mw-redirect" title="Stable sort">stable sort</a>, unlike standard in-place quicksort and heapsort, and can be easily adapted to operate on <a href="/wiki/Linked_list" title="Linked list">linked lists</a> and very large lists stored on slow-to-access media such as <a href="/wiki/Disk_storage" title="Disk storage">disk storage</a> or <a href="/wiki/Network_attached_storage" class="mw-redirect" title="Network attached storage">network attached storage</a>. Although quicksort can be implemented as a stable sort using linked lists, it will often suffer from poor pivot choices without random access. The main disadvantage of mergesort is that, when operating on arrays, efficient implementations require <i>O</i>(<i>n</i>) auxiliary space, whereas the variant of quicksort with in-place partitioning and tail recursion uses only <span class="nowrap"><i>O</i>(log <i>n</i>)</span> space. (Note that when operating on linked lists, mergesort only requires a small, constant amount of auxiliary storage.)</p> 
      <p><a href="/wiki/Bucket_sort" title="Bucket sort">Bucket sort</a> with two buckets is very similar to quicksort; the pivot in this case is effectively the value in the middle of the value range, which does well on average for uniformly distributed inputs.</p> 
      <h3><span class="mw-headline" id="Selection-based_pivoting">Selection-based pivoting</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Quicksort&amp;action=edit&amp;section=19" title="Edit section: Selection-based pivoting">edit</a><span class="mw-editsection-bracket">]</span></span></h3> 
      <p>A <a href="/wiki/Selection_algorithm" title="Selection algorithm">selection algorithm</a> chooses the <i>k</i>th smallest of a list of numbers; this is an easier problem in general than sorting. One simple but effective selection algorithm works nearly in the same manner as quicksort, and is accordingly known as <a href="/wiki/Quickselect" title="Quickselect">quickselect</a>. The difference is that instead of making recursive calls on both sublists, it only makes a single tail-recursive call on the sublist that contains the desired element. This change lowers the average complexity to linear or <i>O</i>(<i>n</i>) time, which is optimal for selection, but the sorting algorithm is still <i>O</i>(<i>n</i><sup>2</sup>).</p> 
      <p>A variant of quickselect, the <a href="/wiki/Median_of_medians" title="Median of medians">median of medians</a> algorithm, chooses pivots more carefully, ensuring that the pivots are near the middle of the data (between the 30th and 70th percentiles), and thus has guaranteed linear time – <i>O</i>(<i>n</i>). This same pivot strategy can be used to construct a variant of quicksort (median of medians quicksort) with <span class="nowrap"><i>O</i>(<i>n</i> log <i>n</i>)</span> time. However, the overhead of choosing the pivot is significant, so this is generally not used in practice.</p> 
      <p>More abstractly, given an <i>O</i>(<i>n</i>) selection algorithm, one can use it to find the ideal pivot (the median) at every step of quicksort and thus produce a sorting algorithm with <span class="nowrap"><i>O</i>(<i>n</i> log <i>n</i>)</span> running time. Practical implementations this variant are considerably slower on average, but they are of theoretical interest because they show an optimal selection algorithm can yield an optimal sorting algorithm.</p> 
      <h3><span class="mw-headline" id="Variants">Variants</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Quicksort&amp;action=edit&amp;section=20" title="Edit section: Variants">edit</a><span class="mw-editsection-bracket">]</span></span></h3> 
      <dl> 
       <dt>
        Multi-pivot quicksort
       </dt> 
       <dd>
        Instead of partitioning into two subarrays using a single pivot, multi-pivot quicksort (also multiquicksort
        <sup id="cite_ref-LaMarca1999_21-1" class="reference"><a href="#cite_note-LaMarca1999-21">[21]</a></sup>) partitions its input into some 
        <span class="texhtml mvar" style="font-style:italic;">s</span> number of subarrays using 
        <span class="texhtml"><i>s</i> ? 1</span> pivots. While the dual-pivot case (
        <span class="texhtml"><i>s</i> = 3</span>) was considered by Sedgewick and others already in the mid-1970s, the resulting algorithms were not faster in practice than the "classical" quicksort.
        <sup id="cite_ref-28" class="reference"><a href="#cite_note-28">[28]</a></sup> A 1999 assessment of a multiquicksort with a variable number of pivots, tuned to make efficient use of processor caches, found it to increase the instruction count by some 20%, but simulation results suggested that it would be more efficient on very large inputs.
        <sup id="cite_ref-LaMarca1999_21-2" class="reference"><a href="#cite_note-LaMarca1999-21">[21]</a></sup> A version of dual-pivot quicksort developed by Yaroslavskiy in 2009
        <sup id="cite_ref-:0_10-1" class="reference"><a href="#cite_note-:0-10">[10]</a></sup> turned out to be fast enough to warrant implementation in 
        <a href="/wiki/Java_version_history#Java_SE_7_.28July_28.2C_2011.29" title="Java version history">Java 7</a>, as the standard algorithm to sort arrays of 
        <a href="/wiki/Primitive_data_type" title="Primitive data type">primitives</a> (sorting arrays of 
        <a href="/wiki/Object_(computer_science)" title="Object (computer science)">objects</a> is done using 
        <a href="/wiki/Timsort" title="Timsort">Timsort</a>).
        <sup id="cite_ref-29" class="reference"><a href="#cite_note-29">[29]</a></sup> The performance benefit of this algorithm was subsequently found to be mostly related to cache performance, and experimental results indicate that the three-pivot variant may perform even better on modern machines.
        <sup id="cite_ref-30" class="reference"><a href="#cite_note-30">[30]</a></sup>
       </dd> 
      </dl> 
      <dl> 
       <dt>
        External quicksort
       </dt> 
       <dd>
        The same as regular quicksort except the pivot is replaced by a buffer. First, read the M/2 first and last elements into the buffer and sort them. Read the next element from the beginning or end to balance writing. If the next element is less than the least of the buffer, write it to available space at the beginning. If greater than the greatest, write it to the end. Otherwise write the greatest or least of the buffer, and put the next element in the buffer. Keep the maximum lower and minimum upper keys written to avoid resorting middle elements that are in order. When done, write the buffer. Recursively sort the smaller partition, and loop to sort the remaining partition. This is a kind of three-way quicksort in which the middle partition (buffer) represents a sorted subarray of elements that are 
        <i>approximately</i> equal to the pivot.
       </dd> 
      </dl> 
      <dl> 
       <dt>
        Three-way radix quicksort
       </dt> 
      </dl> 
      <div role="note" class="hatnote navigation-not-searchable">
       Main article: 
       <a href="/wiki/Multi-key_quicksort" title="Multi-key quicksort">Multi-key quicksort</a>
      </div> 
      <dl> 
       <dd>
        This algorithm is a combination of 
        <a href="/wiki/Radix_sort" title="Radix sort">radix sort</a> and quicksort. Pick an element from the array (the pivot) and consider the first character (key) of the string (multikey). Partition the remaining elements into three sets: those whose corresponding character is less than, equal to, and greater than the pivot's character. Recursively sort the "less than" and "greater than" partitions on the same character. Recursively sort the "equal to" partition by the next character (key). Given we sort using bytes or words of length W bits, the best case is O(KN) and the worst case O(2
        <sup>K</sup>N) or at least O(N
        <sup>2</sup>) as for standard quicksort, given for unique keys N&lt;2
        <sup>K</sup>, and K is a hidden constant in all standard 
        <a href="/wiki/Comparison_sort" title="Comparison sort">comparison sort</a> algorithms including quicksort. This is a kind of three-way quicksort in which the middle partition represents a (trivially) sorted subarray of elements that are 
        <i>exactly</i> equal to the pivot.
       </dd> 
      </dl> 
      <dl> 
       <dt>
        Quick radix sort
       </dt> 
       <dd>
        Also developed by Powers as an o(K) parallel 
        <a href="/wiki/Parallel_Random_Access_Machine" class="mw-redirect" title="Parallel Random Access Machine">PRAM</a> algorithm. This is again a combination of 
        <a href="/wiki/Radix_sort" title="Radix sort">radix sort</a> and quicksort but the quicksort left/right partition decision is made on successive bits of the key, and is thus O(KN) for N K-bit keys. Note that all 
        <a href="/wiki/Comparison_sort" title="Comparison sort">comparison sort</a> algorithms effectively assume an ideal K of O(logN) as if k is smaller we can sort in O(N) using a hash table or 
        <a href="/wiki/Integer_sorting" title="Integer sorting">integer sorting</a>, and if K &gt;&gt; logN but elements are unique within O(logN) bits, the remaining bits will not be looked at by either quicksort or quick radix sort, and otherwise all comparison sorting algorithms will also have the same overhead of looking through O(K) relatively useless bits but quick radix sort will avoid the worst case O(N
        <sup>2</sup>) behaviours of standard quicksort and radix quicksort, and will be faster even in the best case of those comparison algorithms under these conditions of uniqueprefix(K) &gt;&gt; logN. See Powers 
        <sup id="cite_ref-31" class="reference"><a href="#cite_note-31">[31]</a></sup> for further discussion of the hidden overheads in comparison, radix and parallel sorting.
       </dd> 
      </dl> 
      <dl> 
       <dt>
        Partial and incremental quicksort
       </dt> 
      </dl> 
      <div role="note" class="hatnote navigation-not-searchable">
       Main article: 
       <a href="/wiki/Partial_sorting" title="Partial sorting">Partial sorting</a>
      </div> 
      <dl> 
       <dd>
        Several variants of quicksort exist that separate the 
        <span class="texhtml mvar" style="font-style:italic;">k</span> smallest or largest elements from the rest of the input.
       </dd> 
      </dl>
