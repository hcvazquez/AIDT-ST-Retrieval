       <a class="jump_point" id="Getting_Started"></a> 
       <h2 class="">Getting Started</h2> 
       <a class="jump_point" id="Basics"></a> 
       <h4>Basics</h4> 
       <p> Tracker provides an API that can be used to create, retrieve, update, and delete virtually all of the resources stored and manipulated by the system. This interface incorporates several elements of the REST tradition, making access to these actions consistent and intuitive. </p> 
       <p> In a nutshell: collections are accessed via a URL ending in a plural noun such as '.../stories', and to target an individual resource instance, append its numerical id (.../stories/85). To tell the API whether you are viewing, updating, or deleting the resource, make use of the "HTTP verbs," GET, PUT, PATCH, and DELETE. To create a new resource, send a POST request to the collection's URL (.../stories). The API's response will be the content of the requested resource(s) or the data from the as-updated resource. </p> 
       <p> This is an entirely new version of the API, expanded to cover much more of Tracker's functionality, using JSON as its primary encoding with some endpoints being able to encode responses in CSV format. The <a href="/help/api?version=v3">documentation of API v3</a> remains available, though that API version is deprecated. </p> 
       <p> The Pivotal Tracker API allows client software to retrieve and operate on a user's data by making <a href="http://en.wikipedia.org/wiki/Hypertext_Transfer_Protocol">HTTP</a> requests to transfer <a href="http://en.wikipedia.org/wiki/Json">JSON</a>-encapsulated data. </p>
       <p> With a few exceptions, requests have to be authenticated with credentials for a specific Tracker user, on whose behalf the request is being made. Credentials are currently a single per-user <a href="/help/articles/api_token">API Token</a> which users obtain from their personal <a href="/profile">Profile page</a>. </p>
       <p> Because the Tracker API is accessible by anything that can make HTTP requests, not only can API client software be written in almost any available programming language, but people can interact with it directly through any tool that's capable of making requests and receiving responses. The Tracker API documentation contains many examples written as command lines using the popular and widely-available utility <a href="http://en.wikipedia.org/wiki/CURL">curl</a>. </p>
       <p> In addition to software that runs under the user's OS or on a server somewhere, the Pivotal Tracker API supports requests originating from JavaScript executing within the user's browser via <a href="/help/api#Cross_Origin_Resource_Sharing">Cross-Origin Resource Sharing</a> (<a href="http://en.wikipedia.org/wiki/Cross-origin_resource_sharing">CORS</a>). Requests made by non-Tracker API clients from within a browser use the same authentication mechanism (user API tokens) as requests originating from anywhere else. </p> 
       <a class="jump_point" id="cURL_Examples"></a> 
       <h4>cURL Examples</h4> 
       <p> This section contains several example operations that you can try immediately using <code>curl</code>. The only prerequisites are your Pivotal Tracker API token, that you have access to a project in Tracker, and that you know the ID number of that project. When viewing a project in Tracker on the web, the project's ID number is contained in the browser address bar. </p> 
       <p> Each example shows the <code>curl</code> command to create the request, and shows the server response when you click the associated "View Response" button. The response data is intended to be typical of a realistic example, rather than comprehensive. Most responses can vary significantly based on the specific content of the project, see <a href="#Response_Customization">Response Customization</a>. </p>

       <a class="jump_point" id="Notes_on_cURL_Examples"></a> 
       <h5>Notes on cURL Examples</h5> 
       <p> In most of the request examples, above and in the rest of these documentation, the actual <code>curl</code> command is preceded by <a href="http://www.gnu.org/software/bash/manual/bashref.html#index-export">shell exports</a> that set variables that are incorporated into the <code>curl</code> request. The export statements shown in the examples contain either constants consistent with the example or place-holders intended to indicate what a valid value would be. However, if you define the same variables on your system, but with data that matches your authentication, project and other IDs, etc., then you should be able to execute the <code>curl</code> commands directly without modification, simply by copying and pasting. The lines for initializing such variables with realistic values should look like the following (obviously the API token value shown is invalid). </p> 
       <pre class="example_box">
  export TOKEN=8cef9ce9f959c4348742c1595f340836
  export PROJECT_ID=99
</pre> 
       <p> Most of the examples show authentication using an API token passed through the HTTP request header <code>X-TrackerToken</code>. This, and other request headers that are not automatically included by the <code>curl</code> command are shown in the command line as arguments of the <code>-H</code> option. However, there are a number of request headers that <code>curl</code> does include automatically. If you find yourself having trouble reproducing the operation of one of these examples using another command or inside of a program, don't forget that the Tracker API may be expecting/requiring request headers that <code>curl</code> provides but that are not part of another system by default. </p>
       <p> One case of this is the <a href="/help/api/rest/v5#projects_project_id_uploads">/projects/{project_id}/uploads</a> endpoint, where the example relies on <code>curl</code> creating a <code>Content-type</code> header that matches the multi-part request body that it generates, both of which will have to be done by any API client that uses this endpoint. </p>

       <a class="jump_point" id="The_Tracker_Data_Model"></a> 
       <h2 class="">The Tracker Data Model</h2> 
       <p> Pivotal Tracker stores user data using a number of special-purpose resource types, each of which has a particular set of attributes, including its relationship to the other resources. The <a href="/help/api/rest/v5#Resources">Resources</a> section provides a detailed reference of each type of resource. This section lists the most-used resources. </p> 
       <p> All resources in Tracker have either a unique integer ID or (for the <a href="/help/api/rest/v5#iteration_resource">iteration</a> and <a href="/help/api/rest/v5#iteration_override_resource">iteration_override</a> models) a <code>number</code> that identifies them within the scope of their project. The relationships between resources are represented by attributes on one resource that contain a, or a list of, the ID(s) or number(s) of the associated resource. Some associations are represented by attributes on both resources, referencing each other, but others are defined by attributes on only one end of the relation. </p> 
       <p> The major Pivotal Tracker API resource types are: </p> 
       <dl> 
        <dt>
         <a href="/help/api/rest/v5#story_resource">story</a>
        </dt> 
        <dd>
         <p> Stories are the heart of Tracker. For a software development team, stories are the description of what the people will be able to do with your system once the work is completed. For other uses of Tracker, stories are the basic "work to do" item. Virtually all of the other resources can be viewed simply as meta-data about stories, information about individual stories or information about how the stories are grouped. </p>
        </dd> 
        <dt>
         <a href="/help/api/rest/v5#project_resource">project</a>
        </dt> 
        <dd>
         <p> In Pivotal Tracker, a project represents the combination of a group of stories (work to do) and the team that will be working on them (the people who are "members" of the project). Projects also contain the information about how the work has proceeded in the past and a best guess about how it is expected to unfold in the near future. </p>
        </dd> 
        <dt>
         <a href="/help/api/rest/v5#label_resource">label</a>
        </dt> 
        <dd>
         <p> Labels are used to mark stories that are associated, share a common attribute, or are all members of the same <a href="/help/api/rest/v5#epic_resource">epic</a>. The project contains the full set of available labels, and labels have to be created within the project before they can be applied to a story or an epic. (Some API endpoints automatically create labels as needed, when requested to apply a not-yet-existing label to another resource, so that clients do not have to perform the label create operation separately.) </p>
         <p> Stories and epics are associated with the labels applied to them through attributes on the stories/epics (only), with a <code>label_ids</code> array on each <code>story</code> and a <code>label_id</code> on each <code>epic</code>. </p>
        </dd> 
        <dt>
         <a href="/help/api/rest/v5#task_resource">task</a>
        </dt> 
        <dd>
         <p> Each story may have a list of tasks. Each story's tasks are unique to that story, they are not shared as labels are. </p>
        </dd> 
        <dt>
         <a href="/help/api/rest/v5#comment_resource">comment</a>
        </dt> 
        <dd>
         <p> Each story may also have a list of comments. Like tasks, the comments are owned (contained) by a single story. Unlike tasks, comments can contain (be associated with) other resources, representing attachments made to the comment. </p>
        </dd> 
        <dt>
         <a href="/help/api/rest/v5#file_attachment_resource">file_attachment</a>
        </dt> 
        <dd>
         <p> When a file is uploaded to Tracker and attached to a story by creating a comment, a <code>file_attachment</code> resource is created to contain the metadata describing the attachment, and fetching the resource provides the client with the URL necessary for downloading the attached file itself. </p>
        </dd> 
        <dt>
         <a href="/help/api/rest/v5#google_attachment_resource">google_attachment</a>
        </dt> 
        <dd>
         <p> When a Google Drive file is attached to a comment in Tracker, a <code>google_attachment</code> resource is created to provide a representation of it in the Pivotal Tracker system. </p>
        </dd> 
        <dt>
         <a href="/help/api/rest/v5#epic_resource">epic</a>
        </dt> 
        <dd>
         <p> Epics are an organization mechanism for stories, intended to provide a container for all of the stories associated with an overall system feature or product initiative. Each epic is associated with a single <a href="/help/api/rest/v5#label_resource">label</a>, and all of the stories marked with that label are contained by the epic. The project contains a list of all its epics, each epic references its label, and the client can request all of the stories that are marked with a particular label. </p>
        </dd> 
        <dt>
         <a href="/help/api/rest/v5#person_resource">person</a>
        </dt> 
        <dd>
         <p> A <code>person</code> resource represents an individual Tracker login and profile. The same person information is available (through the <a href="/help/api/rest/v5#project_membership_resource">project_membership</a> resource) from each project that the same person is a member of. </p>
        </dd> 
        <dt>
         <a href="/help/api/rest/v5#project_membership_resource">project_membership</a>
        </dt> 
        <dd>
         <p> A <code>project_membership</code> represents the relationship between an individual <a href="/help/api/rest/v5#person_resource">person</a> and one <a href="/help/api/rest/v5#project_resource">project</a> that they are a member of. </p>
        </dd> 
        <dt>
         <a href="/help/api/rest/v5#iteration_resource">iteration</a>
        </dt> 
        <dd>
         <p> An <code>iteration</code> is a period of time over the course of work on a project. Iterations can be either in the past or in the future. Each iteration has a <code>number</code>, but there is not a fixed relationship between an iteration number and the iteration's start and end dates. For example, iteration number 1 is always the first iteration in a <a href="/help/api/rest/v5#project_resource">project</a>, even if something causes the project's start date to change occurs. And the number of iterations in the project (and therefore the largest valid iteration number) will vary over time. In particular, iterations are created implicitly as stories are moved from the Icebox to the Backlog, so that every non-Icebox story can be placed into an iteration that has a valid number. </p>
        </dd> 
        <dt>
         <a href="/help/api/rest/v5#iteration_override_resource">iteration_override</a>
        </dt> 
        <dd>
         <p> The <code>iteration_override</code> resource is used to manipulate configuration attributes of the underlying <code>iteration</code>, associated by having a matching <code>number</code>. While <code>iteration</code>s exist in a contiguous, fully-populated set, <code>iteration_override</code> resources exist only for iteration numbers that have had their configuration attributes set to a non-default value. </p>
        </dd> 
        <dt>
         <a href="/help/api/rest/v5#integration_resource">integration</a>
        </dt> 
        <dd>
         <p> An <code>integration</code> resource contains the data that is necessary for Tracker to exchange information with a third-party system. They are typically created in the Pivotal Tracker web user interface by selecting the <b>INTEGRATIONS</b> tab in a project's settings pages, and can be used through the API to get the integration's list of <a href="/help/api/rest/v5#external_story_resource">external_story</a> resources, see <a href="/help/api/rest/v5#projects_project_id_integrations_integration_id_stories">/projects/{project_id}/integrations/{integration_id}/stories</a>. These are the story-like entries that are shown in the integration panels that can be opened within the Tracker web user interface, and used to create Pivotal Tracker stories from data imported from the third-party system. </p>
        </dd> 
        <dt>
         <a href="/help/api/rest/v5#account_resource">account</a>
        </dt> 
        <dd>
         <p> <code>account</code>s are resources that serve as containers for a group of related projects, and provide for that group of projects to all be covered by the same paid subscription to Pivotal Tracker. </p>
        </dd> 
       </dl> 
      </section> 
      <section> 
       <a class="jump_point" id="The_Tracker_API"></a> 
       <h2 class="">The Tracker API</h2> 
       <p> This section describes the basic conventions of the Tracker API, including the supported fundamental data types, URL structure, and data encoding. Lower-level details are addressed under <a href="#Response_Customization">Response Customization</a>, below. </p> 
       <a class="jump_point" id="Tracker_API_Conventions"></a> 
       <h4>Tracker API Conventions</h4> 
       <p> Each response will contain data that describes one of the API <a href="/help/api/rest/v5#Resources">Resources</a>, or an array of resource data structures. The keys that can occur within a resource structure are given in the matching section of this documentation. However, new keys may be added at any time without an increase in the Tracker API version as long as no existing keys are modified, so clients must be capable of ignoring any resource attribute that they are unable to process. </p> 
       <a class="jump_point" id="API_Data_Types"></a> 
       <h5>API Data Types</h5> 
       <dl> 
        <dt> 
         <a class="jump_point" id="boolean"></a> boolean 
        </dt> 
        <dd> 
         <p> The value is either true or false. In JSON-encoded requests and responses, a boolean value is represented with the bare symbols <code>true</code> and <code>false</code>. Attributes that are boolean are never "empty," so they will not be dropped from server responses like other attributes can be. </p> 
        </dd> 
        <dt> 
         <a class="jump_point" id="string"></a> string 
        </dt> 
        <dd> 
         <p> This type of value is a string of text characters. Tracker expects/uses UTF-8 charset strings, and values of this type support up to encoding with up to three bytes per character. Most string-type parameters have a maximum character count. The length limit is represented in the <a href="/help/api/rest/v5#Resources">Resources</a> reference by a number, enclosed in square brackets, added to the end of the type name. For example, <code>string[120]</code> indicates a string with a maximum length of 120 characters. In JSON-encoded requests and responses, strings must be enclosed in quotation marks and employ conventional JSON (JavaScript) character escaping and encoding. </p> 
        </dd> 
        <dt> 
         <a class="jump_point" id="extended_string"></a> extended string 
        </dt> 
        <dd> 
         <p> Similar to <code>string</code> values, but supports characters with four-byte UTF-8 encodings, such as emoji. </p> 
        </dd> 
        <dt> 
         <a class="jump_point" id="enumerated_string"></a> enumerated string 
        </dt> 
        <dd> 
         <p> Enumerated strings are like strings, except that the content of the string is constrained to a predefined set of possible values. Each parameter or attribute that has the type "enumerated string" is accompanied by a "valid enumeration values" list that shows all of the legal values for that particular enumerated string. Values must be exactly as given, including matching case and without any padding/white space. </p> 
        </dd> 
        <dt> 
         <a class="jump_point" id="int"></a> int 
        </dt> 
        <dd> 
         <p> An integer value. Integers are decimal, and some may be zero and/or negative. Most integer parameters/attributes must be 1 or larger. The requirements for a particular integer value should be clear from context, and the API will return error messages if not. </p> 
        </dd> 
        <dt> 
         <a class="jump_point" id="float"></a> float 
        </dt> 
        <dd> 
         <p> A real number. Encoding must conform to the JSON syntax. </p> 
        </dd> 
        <dt> 
         <a class="jump_point" id="datetime"></a> datetime 
        </dt> 
        <dd> 
         <p> A representation of a particular date and time. In general, <code>datetime</code> values can be expressed in one of two ways. They may be a string encoded according to the <a href="http://en.wikipedia.org/wiki/ISO_8601">ISO 8601 standard</a>, like <code>"2013-04-30T04:25:15Z"</code>. Or they will be an integer number of milliseconds since the beginning of the epoch, like <code>1367296015000</code>. When supplying a parameter value that is a datetime, the client may use either format and Tracker will determine which is being used dynamically. By default, datetime values included in a response will be ISO 8601 strings. There are a handful of responses in which resources have <code>datetime</code> attributes which are only able to be formatted in one of these ways. This fact is noted in the descriptions of those particular attributes. For the majority, to cause dates to be formatted as millisecond offsets instead, supply the parameter <code>date_format=millis</code> in the query string of the request URL. </p> 
        </dd> 
        <dt> 
         <a class="jump_point" id="datetime_interval"></a> datetime_interval 
        </dt> 
        <dd> 
         <p> A representation of a particular date and time range. It is string encoded using the <a href="http://en.wikipedia.org/wiki/ISO_8601">ISO 8601 interval standard</a>, like <code>"2013-04-29T04:00:00Z/2013-04-30T04:00:00Z"</code>. </p> 
        </dd> 
        <dt> 
         <a class="jump_point" id="date"></a> date 
        </dt> 
        <dd> 
         <p> <code>date</code> values are encoded according to the <a href="http://en.wikipedia.org/wiki/ISO_8601">ISO 8601 standard</a>, like <code>"2013-04-30"</code>. <code>date</code>-type attributes do not have a time incorporated in them like <code>datetime</code> attributes do. </p> 
        </dd> 
        <dt> 
         <a class="jump_point" id="scalar"></a> scalar 
        </dt> 
        <dd> 
         <p> This indicates that the value is type-checked as being one of the preceding types, but it may be any one of those types at different types. For example, a resource attribute whose type is <code>List[scalar]</code> can contain a mix of differently-typed values. The actual type of each value can be determined interactively or may be established by contention for a particular API request or use. </p> 
        </dd> 
        <dt> 
         <a class="jump_point" id="object"></a> object 
        </dt> 
        <dd> 
         <p> This indicates that the value is an un-type-checked nested hash data structure. For both request parameters and response attributes listed as being <code>object</code> type, their expected/possible content should be clear from context. </p> 
        </dd> 
       </dl> 
       <p> The same set of types is used for the parameters that may be supplied in API requests, and the parameters to individual <a href="/help/api/command/c2#Commands">Commands</a>. </p> 
       <a class="jump_point" id="Request_Parameter_Encoding"></a> 
       <h5>Request Parameter Encoding</h5> 
       <p> Parameters are supplied as parts of Pivotal Tracker API requests in a number of ways: </p> 
       <ul> 
        <li> As components of the request URL's path. These are marked as <code>path</code> parameters in the endpoint reference, and are typically ID numbers or other integers. </li>
        <li> In the request URL's query string (the portion following the question mark, <code>?</code>). These are marked as <code>query</code> parameters in the endpoint reference. Parameters that are expected to be passed in the query string cannot be passed in the request body (and <i>vice versa</i>). </li>
        <li> In the body of non-GET requests. These are marked as <code>body</code> parameters in the endpoint reference. When a body is provided with a Tracker API request, it must be encoded using JSON unless a specific endpoint description specifies another encoding. </li>
        <li> In custom headers in the HTTP request itself. These are not listed as parameters of individual requests, but are common across requests and described under <a href="/help/api#Common_HTTP_Extension_Request_Headers">Common HTTP Extension Request Headers</a>. </li> 
       </ul> 
       <a class="jump_point" id="Nested_Resources_as_Parameters"></a> 
       <p> Body parameters can be simple scalars, arrays, nested resource structures, or arrays of structures. Arrays, either of integer ID values or of nested structures are used to represent associations between one resource and several others. </p> 
       <p> When a resource structure is used as the value of a parameter, the client has some flexibility in choosing what attributes to include in the nested resource structure it creates. If the client intends to reference an existing resource in the system, then it must supply sufficient information to uniquely identify the existing resource (the <code>id</code> value is usually easiest/best, but things like the <code>name</code> attribute for labels are also unique identifiers). Any additional information that is included must match the existing values of the sub-resource. If the client intends to implicitly create a new resource, it must not include the <code>id</code> parameter (the server will be assigning a new <code>id</code>) and it must supply any attributes for which the server cannot supply a default value. </p> 
       <a class="jump_point" id="Warning_Content_Type_application_json_will_be_required"></a> 
       <p class="warning_callout"> <i>'Content-Type' will be required, and required to be 'application/json'.</i> In order to ensure forward compatibility with API extensions that allow request bodies to be encoded in a format other than JSON, we will be imposing an error check that requires clients to explicitly specify the format of the request body by including the <code>Content-Type</code> header in the request. </p> 
       <a class="jump_point" id="Common_Request_Parameters"></a> 
       <h5>Common Request Parameters</h5> 
       <a class="jump_point" id="token_Parameter_for_Authentication"></a> 
       <h6>token Parameter for Authentication</h6> 
       <p> Requests that are authenticated (see <a href="/help/api#Authenticating_Using_API_Tokens">Authenticating Using API Tokens</a>) have to pass the API token for the user on whose behalf the client is operating to the server with each request. The preferred method is to send in API token n the HTTP request as the value of the <code>X-TrackerToken</code> header. However, it is also possible to use the query parameter <code>token</code> in the URL of a request to pass the API token value. </p> 
       <a class="jump_point" id="envelope_Parameter"></a> 
       <h6>envelope Parameter</h6> 
       <p> The Tracker API can return responses in two different formats: just the raw resource data with any metadata passed in HTTP response headers (the default), or the resource data embedded as a single element of an 'envelope' structure that contains any metadata in the response body. </p>
       <p> This is controlled by the <code>envelope</code> parameter being included in the request URL's query string. Requests with no <code>envelope</code> parameter or <code>envelope=false</code> return the default format. To embed the response resources in the envelope, the client should make its request with <code>envelope=true</code>. The structure of the response envelope is described in <a href="/help/api#Envelope_Return_Structure">Envelope Return Structure</a>, below. </p> 
       <a class="jump_point" id="method_Parameter_for_Request_Method_Simulation"></a> 
       <h6>_method Parameter for Request Method Simulation</h6> 
       <p> To use all of the operations available in the Pivotal Tracker API, clients must be able to generate the HTTP methods GET, POST, PUT, PATCH, and DELETE. However, some older network infrastructure can only generate GET and POST. If you are implementing a client in such an environment, PUT, PATCH, and DELETE requests can be simulated by making POST requests that include the <i>query</i> parameter <code>_method</code>. The permissible values for <code>_method</code> are "PUT", "PATCH", and "DELETE" only. Clients that are capable of generating correct request methods should not use this parameter. </p>
       <p> <code>_method</code> cannot be used as a POST-body parameter. Supplying a non-empty POST body is inappropriate for DELETE operations, whether performed using the actual DELETE method or simulating it by performing a POST request with <code>_method=DELETE</code>. </p>
       <p> Note that while the Tracker API supports clients which use the HTTP PATCH method to update resources, it does so by making PUT and PATCH synonymous. While it is technically more correct to use PATCH when updating a subset of the attributes of a resource, and use PUT only in the case where the entire content of the resource is replaced, Tracker supports partial-update through PUT as well. Where the reference description of an individual endpoint references PUT, either PUT or PATCH may be used. </p>
       <p> The API also accepts HTTP requests using the OPTIONS method to support in-browser JavaScript clients using <a href="/help/api#Cross_Origin_Resource_Sharing">Cross-Origin Resource Sharing</a>. OPTIONS is another method that isn't universally supported. However, the <code>_method</code> parameter cannot be used to simulation an OPTIONS request. Because CORS is a browser-to-server protocol, that isn't meaningful to non-browser clients, there is no value in clients that are not CORS-capable browsers issuing requests using OPTIONS. </p> 
       <a class="jump_point" id="Common_HTTP_Extension_Request_Headers"></a> 
       <h5>Common HTTP Extension Request Headers</h5> 
       <a class="jump_point" id="X_TrackerToken_Header_for_Authentication"></a> 
       <h6>X-TrackerToken Header for Authentication</h6> 
       <p> Requests that are authenticated (see <a href="/help/api#Authenticating_Using_API_Tokens">Authenticating Using API Tokens</a>) have to pass the API token (for the user on whose behalf the client is operating) to the server with each request. The preferred method is to send in API token in the HTTP request as the value of the <code>X-TrackerToken</code> header. If this request header is not used, the <a href="/help/api#token">token</a> request parameter must be. </p> 
       <a class="jump_point" id="X_Tracker_Warn_Unless_Project_Version_Is_Header"></a> 
       <h6>X-Tracker-Warn-Unless-Project-Version-Is Header</h6> 
       <p> When used, the value passed with this header should be the integer project version for the Tracker project resource referenced by the request. The <a href="/help/api/rest/v5#project_resource">project</a> version is increased each time a change is made to the content of the project, and the version number can be used by a client to keep track of whether the data stored on the server has changed since the client has most recently retrieved information. The version number can be obtained from any request that provides information from the project. </p>
       <p> This header will be ignored if it is included in a request that does not identify a single project, <i>i.e.</i>, only requests that somehow include a <code>project_id</code> parameter will observe <code>X-Tracker-Warn-Unless-Project-Version-Is</code>. </p>
       <p> For endpoints that support it, when Tracker receives a request with the <code>X-Tracker-Warn-Unless-Project-Version-Is</code> header, it will either include the header <code>X-Tracker-Warning-Project-Was-Stale</code> in the HTTP response, or it will include the key <code>warning_project_was_stale</code> in the response envelope, depending on whether <code>envelope=true</code> is included in the request's query parameters, see <a href="/help/api#Envelope_Return_Structure">Envelope Return Structure</a>. </p>
       <p> During request processing, Tracker compares the project version number passed in as the header's value with the selected project's <em>current</em> version. If the two match, the value returned (in the response header or envelope) will be <code>false</code> (no warning), and if the version numbers don't match, it will be <code>true</code>. This allows a client that is keeping a local copy of some or all of the project's data to know whether it's store remains valid, or if it might be out of date relative to what's stored on the server (and presumably also in other clients). A client could always poll to determine this, by making one of the GET requests that returns a project version number. However, polling is expensive and wasteful, especially if the client doesn't have to be assured that it is up to date prior to making other requests. </p> 
       <p> There are a handful of endpoints which are project-specific but do not support the <code>X-Tracker-Warn-Unless-Project-Version-Is</code> header. These include <a href="/help/api/rest/v5#source_commits">/source_commits</a>. </p>

       <a class="jump_point" id="Response_Structure"></a> 
       <h5>Response Structure</h5> 
       <p> Responses from the Pivotal Tracker API share a number of structural details and conventions, explained in this section. In addition, there are a number ways in which the client can request that the responses be customized. These are described under <a href="/help/api#Response_Customization">Response Customization</a>, below. </p> 
       <a class="jump_point" id="Default_Return_Structure"></a> 
       <h6>Default Return Structure</h6> 
       <p> The Pivotal Tracker API can return the results of a request to the client either as the raw requested information (by default) or wrapped in a JSON envelope structure, see the section <a href="/help/api#Envelope_Return_Structure">Envelope Return Structure</a> just below. </p>
       <p> The response to each API request will conform to the specified one of the structures documented under <a href="/help/api/rest/v5#Resources">Resources</a> if the request was successful, or to the structure described under <a href="/help/api#Error_Responses">Error Responses</a> if an error occurred or was detected. </p>
       <p> When the client doesn't request that the response be wrapped in an envelope, metadata associated with the client/server transport, or which characterizes the response data, is returned to the client in custom HTTP response headers. </p> 
       <p> The optional response headers that may be found in a Pivotal Tracker API response are: </p> 
       <dl> 
        <dt>
         <code>X-Tracker-Project-Version</code>
        </dt> 
        <dd>
         <p> This is included when the response contains information that is specific to a single Pivotal Tracker project. The value associated with this key is identical to the <code>version</code> attribute of the <a href="/help/api/rest/v5#project_resource">project</a> resource: an integer that indicates how many changes had been made to the project prior to when the data in the response was captured. The project version number can be used to determine if the responses from successive API requests reflect a consistent view of the project, or if it was changed by some other client in between requests. Project version is also used to allow a number of clients to synchronize their views of the shared project. </p>
        </dd> 
        <dt>
         <code>X-Tracker-Downtime-Start</code> 
         <code>X-Tracker-Downtime-Finish</code>
        </dt> 
        <dd>
         <p> This is included when there is an upcoming scheduled maintenance downtime for the Pivotal Tracker service that will affect service of API requests. A client can use this to inform users of the upcoming availability. The value is a hash that indicates the planned start and end times of the maintenance window, see the example below. </p>
        </dd> 
        <dt>
         <code>X-Tracker-Deprecation-Warning</code>
        </dt> 
        <dd>
         <p> This will be in the future if the request that has generated the response, or an attribute contained in the response, has been deprecated. The value will contain an English string. The value may begin with a datetime value encoded as an ISO 8601 string (only). If the datetime is present, it will be separated from the string by a pipe character (<code>|</code>), and indicate the time after which the deprecated behavior/endpoint/etc. may be removed from the system. </p>
        </dd> 
        <dt> 
         <code>X-Tracker-Pagination-Limit</code> 
         <code>X-Tracker-Pagination-Offset</code> 
         <code>X-Tracker-Pagination-Total</code> 
         <code>X-Tracker-Pagination-Returned</code> 
        </dt> 
        <dd>
         <p> Some combination of these keys will be present when the response contains a paginated list, rather than a representation of a single resource. They inform the client of what portion of the available set of items (the logical list selected by the request) has actually been returned. The client can affect what portion of the list is returned by an individual request using the standard pagination parameters. For details about the meaning of each of these headers' values and the pagination parameters see <a href="#Paginating_List_Responses">Paginating List Responses</a>. </p>
        </dd> 
       </dl> 
       <a class="jump_point" id="Envelope_Return_Structure"></a> 
       <h6>Envelope Return Structure</h6> 
       <p> All responses from the Pivotal Tracker API may be wrapped in a common top-level structure, a standard "envelope" around the data that is otherwise returned from the particular request that was made. This section describes the envelope structure and gives examples of various successful responses. The way error information is encoded, and is encapsulated in the envelope structure is described in <a href="#Error_Responses">Error Responses</a>, below. </p>
       <p> The following is an example of an API response that is carrying (minimal) data from a single Pivotal Tracker story: </p> 
       <code class="json example_box"> { "http_status": "200", "project_version": 42, "data": { "id": 501, "name": "Install cargo elevator platforms in docking bays." } } </code> 
       <p> This shows a typical envelope structure, but there are a number of optional attributes in the envelope structure, including some of what's shown above. A successful response will always have: </p> 
       <ul> 
        <li> A <code>data</code> key, under which is the structure that would ordinarily be the entire server response. In the example above, the hash under <code>data</code> contains attributes from a <code>story</code> resource. </li> 
        <li> <code>http_status</code>: The value of this key is the same as the content of the HTTP response's numeric status string. It is provided here for the convenience of client's that find it difficult to access the response's status. </li> 
       </ul> 
       <p> In addition, the value from each of the Pivotal Tracker HTTP response extension headers listed above will be moved to an entry in the response envelope. Note that, in the envelope structure, several of the values are consolidated together into nested hashes, rather than being a collection of scalars as the HTTP header mechanism necessitates. The mapping is as follows: </p> 
       <ul> 
        <li><code>X-Tracker-Project-Version</code>'s value goes to <code>"project_version":</code></li> 
        <li><code>X-Tracker-Downtime-Start</code>'s value goes to <code>"downtime": { "start": }</code></li> 
        <li><code>X-Tracker-Downtime-Finish</code>'s value goes to <code>"downtime": { "finish": }</code></li> 
        <li><code>X-Tracker-Deprecation-Warning</code>'s value goes to <code>deprecation_warning</code></li> 
        <li><code>X-Tracker-Pagination-Limit</code>'s value goes to <code>"pagination": { "limit": }</code></li> 
        <li><code>X-Tracker-Pagination-Offset</code>'s value goes to <code>"pagination": { "offset": }</code></li> 
        <li><code>X-Tracker-Pagination-Total</code>'s value goes to <code>"pagination": { "total": }</code></li> 
        <li><code>X-Tracker-Pagination-Returned</code>'s value goes to <code>"pagination": { "returned": }</code></li> 
       </ul> 
       <p> This example API response shows some of the optional entries described above. (Assume this was generated by a request with <code>?fields=id&amp;date_format=millis</code>.) </p> 
       <code class="json example_box"> { "http_status": "200", "project_version": 420, "data": [ { "id": 1790 }, { "id": 1941 }, { "id": 5203 }, { "id": 11078 } ], "downtime": { "start": 1361473200000, "finish": 1361473590000 }, "pagination": { "total": 4, "limit": 50, "offset": 0, "returned": 4 } } </code> 
       <a class="jump_point" id="Successful_Responses"></a> 
       <h5>Successful Responses</h5> 
       <p> When a request to the Tracker API is successfully processed, the status returned will be in the <code>2xx</code> or <code>3xx</code> ranges, consistent with the HTTP protocol. Note that this includes <code>204</code> (No Content). Several requests do not currently have a defined response structure, and do not produce a response body when successful (unless the client has requested the response envelope). This includes the responses to DELETE-method requests, as well as some others. For example, note how the description of DELETE-method requests to the stories endpoint (<a href="/help/api/rest/v5#projects_project_id_stories_story_id_">/projects/{project_id}/stories/{story_id}</a>), does not include a statement about what resource is included in the response. </p> 
       <a class="jump_point" id="Warning_POSTs_for_creation_will_return_201"></a> 
       <p class="warning_callout"> Currently POST operations which create new resources on the Tracker server return 200. They will be updated to return 201 (Created). </p> 
       <a class="jump_point" id="Error_Responses"></a> 
       <h5>Error Responses</h5> 
       <p> If an error occurs while processing an API request, Pivotal Tracker will return a hash of information regarding the type of failure, it's cause, and possibly how it can be corrected by the user (different data input) or the API client developer (different structure or sequence). The error response hash will either be the entire response or will be nested within the response envelope if the <code>envelope</code> parameter was used to request one. When an error is returned within the response envelope structure, it will be stored under the key <code>data</code>, where a successful response's content is also located. For example, this is an error-producing request without the response envelope: </p> 
       <div class="overview_example"> 
        <div class="body"> 
         <div class="curl"> 
          <div class="example_block command">

       <p> This same error structure is used for all error cases. There are currently seven keys defined, the first three of which will always be returned: </p> 
       <dl> 
        <dt>
         <code>kind</code>
        </dt> 
        <dd>
         <p> Error hashes, like other response resources, include a <code>kind</code> attribute to identify their type, independent of the context in which they're returned. Unlike other API responses, because the <code>fields</code> parameter cannot be replied to error responses, <code>kind</code> will always be included. </p>
        </dd> 
        <dt>
         <code>code</code>
        </dt> 
        <dd>

        </dd> 
        <dt>
         <code>error</code>
        </dt> 
        <dd>
         <p> This is an English string describing the category of the error which occurred. There is a one-to-one match between the <code>code</code> value and the <code>error</code> string for a particular error. <code>error</code> is always included in an error response. </p>
        </dd> 
        <dt>
         <code>requirement</code>
        </dt> 
        <dd>
         <p> An English string describing the constraint on the API that wasn't met by the current request. </p>
        </dd> 
        <dt>
         <code>general_problem</code>
        </dt> 
        <dd>
         <p> An English string with a broader/alternative description of the problem. </p>
        </dd> 
        <dt>
         <code>possible_fix</code>
        </dt> 
        <dd>
         <p> An English string with hints to the API client developer. Consider this a micro-FAQ answer specific to the particular error case. Very unlikely to be suitable for direct presentation to a user. </p>
        </dd> 
        <dt>
         <code>validation_errors</code>
        </dt> 
        <dd>

Each validation error hash contains a <code>field</code> key with the name of a request parameter (resource attribute), and a <code>problem</code> key with an English string describing the error(s) detected in the value of that parameter. </p>
        </dd> 
       </dl>

       <a class="jump_point" id="Request_Authentication_and_CORS"></a> 
       <h4>Request Authentication and CORS</h4> 
       <p> Most requests to the Pivotal Tracker API must be authenticated, using credentials that belong to a specific Tracker user login and profile. Clients should be constructed to obtain credentials from the user on whose behalf it is performing operations, so that correct attribution of changes is made. Some clients may be created with the intent of performing maintenance or monitoring or automatic updates on behalf of the entire team. In this case, it is recommended that a separate Tracker user account be created for exclusive use of the API client, rather than it being set up with the credentials of an existing person who is a member of the target project. </p>
       <p> Requests that do not require authentication are (1) read operations (primarily GET requests) that are (2) performed on <a href="/help/articles/using_public_projects">Public projects</a>. The information from a Public project can be read by anyone on the Internet, whether they have a Pivotal Tracker login or not. This is the case in the Tracker web UI and through API requests. </p>

       <a class="jump_point" id="Cross_Origin_Resource_Sharing"></a> 
       <h5>Cross-Origin Resource Sharing</h5> 
       <p> <a href="http://en.wikipedia.org/wiki/Cross-origin_resource_sharing">CORS</a> is a capability that applies only to Tracker API clients that are written to execute inside a web browser that is subjecting them to the <a hre="http://en.wikipedia.org/wiki/Same_origin_policy">Same Origin Policy</a>. </p>
       <p> In order to allow for Tracker API clients that are JavaScript programs executing within the user's browser, Pivotal Tracker supports Cross-Origin Resource Sharing when accessing API endpoint URLs. Tracker supports the user's browser sending "pre-flight" requests using the HTTP OPTIONS method, and it returns <code>Access-Control-</code>... headers that give the browser permission to make requests to the API. </p>
       <p> While CORS allows JavaScript clients to access the Tracker API from within a browser, the client still must have the API token for a particular Pivotal Tracker user in order to make most requests (<em>all</em> requests that access the data of a private project). Because Tracker API tokens are a means of single-factor authentication, it is very important that they not be used or communicated in a way that would allow them to be copied by third parties. As with previous versions of the Tracker API, ensuring the security with which user API tokens are obtained and (if necessary) stored by client software is the responsibility of that software's authors. However, the fact that the source and runtime environment of a JavaScript API client are inherently public creates new opportunities for token-handling to be compromised. </p>
       <p> A JavaScript client that obtained the API token from the user anew on each load/execution could provide a great deal of security, since the token would only ever be loaded in runtime environments that the user had control over. On the other hand, while hard-coding an API token into the source of a client is dangerous practice when the client runs on a team server or a user's desktop, it is completely unacceptable for a JavaScript client. Sending the API client software to the user's browser distributes the token used far and wide. Publishing the API token for a user that is a Viewer on a project (has read-only access) is the security equivalent of making the project public. Distributing the token for a collaborator could allow anyone anywhere to do anything within the project that they want. </p>
       <p> It is almost as dangerous to have the user API token to be used transmitted from the server to the client outside of the client source. Even if the transmission of the token is secure, anyone with access to the browser could obtain the token value and then re-use it again later in a different application. </p>
       <p> We also strongly discourage the creation of sample JavaScript client code which is intended to have the user API token added to it by the person running the code. This kind of sample can easily be incorporated into larger, more widely distributed pieces of code without the dangerous handling of the user API tokens being corrected. </p>
       <p> Note that Tracker's current configuration prevents the browser from sending authentication parameters along with CORS requests. This doesn't affect sending of the API token (either in the header or as a query parameter), but it does prohibit the browser from sending credentials through Basic auth, which has the effect of prohibiting a client that relies on CORS from using the <a href="/help/api/rest/v5#me">/me</a> endpoint to obtain a user's API token. </p> 
       <a class="jump_point" id="Warning_CORS_will_require_OAuth_once_available"></a> 
       <p class="warning_callout"> <i>CORS will be limited to OAuth-using clients.</i> At some point in the future, we will be adding <a href="http://oauth.net/">OAuth</a> support to version 5 of the Pivotal Tracker API as an alternative to using API tokens for user authentication. Once this occurs, we will be restricting the availability of CORS to OAuth-using clients and prohibiting the use of API tokens for authenticating CORS-using requests <em>without increasing the API version number</em>. There will of course be an announcement prior to the removal of support for tokens+CORS, but it is recommended that CORS only be used in clients that have relatively short deployment cycles so that a client version updated with OAuth support can be released between the incorporation of OAuth support and the removal of tokens+CORS support, in order to avoid an interruption in client access to the Tracker API. </p> 
       <a class="jump_point" id="Request_Authorization"></a> 
       <h4>Request Authorization</h4> 
       <p> Each request made to the Pivotal Tracker API is authorized based on the result of authentication. If the authentication credentials provided are invalid, the request will fail prior to authorization. If no authentication credentials are provided, authorization is based on whether the information requested is available to an anonymous visitor. Otherwise, authorization is based on the authenticated Tracker user's relationship to the resource(s) referenced by the request. </p>
       <p> Each Tracker user profile can be associated with any number of projects and any number of accounts. Whether a particular request can the authorized depends on which resources the request references, and the user's relationship to those resources through the user's project(s) and account(s). API requests conform to the same authorization rules implemented by the Tracker web site. In general: </p> 
       <ul> 
        <li> A person who is a Viewer on a project can make GET requests to fetch resources in that project, but cannot make requests that would modify it. </li>
        <li> Only a person who is an Owner of a project can modify the project settings or modify its integrations. </li>
        <li> An anonymous request with no authentication can only be a GET, and then only for resources in a public project. That is, a project which has its <code>public</code> attribute equal to true. </li> 
       </ul> 
       <p> In addition to the general rules that apply broadly to all resources that are contained in a project, there are some less-obvious constraints that apply to specific resources and attributes: </p> 
       <ul> 
        <li> The <code>email</code> attribute of person resources is filtered out of responses, even when the client has requested it to be included, based on specific authorization criteria, independent of whether the underlying request is authorized. There are a number of detailed rules regarding when the <code>email</code> attribute will be populated, but in short it will only be available to requests made using authentication for a user that would be able to view the email address through the Pivotal Tracker web site. A client application designed to work on behalf of arbitrary users must accommodate not receiving <code>email</code> in the responses it receives. </li>
        <li> A POST request to the <a href="/help/api/rest/v5#projects">/projects</a> endpoint will be authorized only for users who have the "project creator" permission on the account whose ID is included in the request parameters. </li> 
       </ul> 
       <a class="jump_point" id="API_Versioning"></a> 
       <h4>API Versioning</h4> 
       <p> The current stable version of the Pivotal Tracker API is "<code>v5</code>". There is one deprecated API version still available "<code>v3</code>", which is documented <a href="/help/api?version=v3">here</a>. </p>
       <p> This section lays out general policies on ensuring a client's ongoing compatibility with the Tracker API in general, and with the current recommended API version (<code>v5</code>) in particular. We will be adding new capabilities to the API without increasing the version number while those capabilities are compatible with the existing functionality and encodings. We are also introducing a standing <code>edge</code> API version that will be used to introduce and obtain feedback on new features that are incompatible with the current API version, that we intend to incorporate in the next stable API version. The determination of what API behaviors are incompatible and therefore have to be introduced in <code>edge</code> or a new numbered version is made based on the content of this section. The use of <code>edge</code> and what it currently contains is covered in its own reference page, see <a href="/help/api/rest/edge">REST API Edge</a>. </p> 
       <a class="jump_point" id="Meaning_of_Stability_in_the_API_Specification"></a> 
       <h5>Meaning of "Stability" in the API Specification</h5> 
       <p> The behavior of API v5 will be held stable until and unless it is announced to be deprecated, and it will remain supported for several months after such time. </p>
       <p> However, for the purposes of the Tracker API, "stable" means that no published endpoint, resource, or attribute will be removed or redefined. We intend to continue adding endpoints, resources, and attributes to existing resources, incrementally expanding the definition of v5 in forward compatible ways. The balance of this section lists potential variations in the responses generated or requests accepted by the API, which ones would be considered an incompatible change in the API, and which ones an API client must be written to anticipate and accommodate. </p> 
       <a class="jump_point" id="Transport_and_Encoding"></a> 
       <h6>Transport and Encoding</h6> 
       <p> An API client should be tolerant of changes in the way the HTTP protocol is used by the Tracker server, as long as transport remains compliant with the standard. In particular: </p> 
       <ul> 
        <li>Changes in the types of transport encodings available</li> 
        <li>Changes in the content of cache-control headers returned by Tracker</li> 
        <li>Introduction of new extension headers in HTTP responses</li> 
        <li>Accepting and operating on HTTP headers in requests that begin with <code>X-Tracker</code></li> 
       </ul> 
       <p> An API client should be tolerant of changes in the formatting of responses from the API, as long as responses remain compliant to the <a href="http://www.json.org/">JSON standard</a>. In particular, the following can all vary within a JSON-encoded response without having any impact on the data structure it is intended to convey, and therefore without constituting a change to the definition of the API: </p> 
       <ul> 
        <li>The amount and location of white-space that is outside of a string constant or symbol</li> 
        <li>The order of keys within the text representation of a hash</li> 
        <li>The types of quotation marks used</li> 
        <li>The escaping of characters within a string constant</li> 
        <li>Variations in the inclusion of optional characters in value encodings, for example including or excluding ".0" at the end of floating-point constants</li> 
       </ul> 
       <p> This list is mentioned in particular because more than one client for API v3 was written to "parse" the responses using regular expressions that were dependent on the order of items in the XML-encoded response, including within semantically-unordered parts of the data. The clients built this way had frequent problems that occurred in response to minor changes in Tracker's infrastructure, and authors of v5 API clients are strongly advised to use an available, "full" JSON parser for their client rather than building something custom. </p> 
       <a class="jump_point" id="Data_and_Operations"></a> 
       <h6>Data and Operations</h6> 
       <p> Expansion of available data. </p>
       <p> Future API rules that limit the range of a value. </p>
       <p> Restrictions on writeability may be relaxed. </p>
       <p> Addition of new resource types implies the possibility for new values to occur in some fields. For example, The <code>primary_resources</code> and <code>changes</code> attributes of <a href="/help/api/rest/v5#activity_resource">activity</a> structures are untyped arrays of object hashes. Each of these may reference any of the resources available in the API, including ones not originally defined when the API was released. This means that clients should be prepared to discover hashes which identify themselves as matching resource types that it isn't aware of. </p> 
       <a class="jump_point" id="Cases_to_Be_Cautious_Of"></a> 
       <h6>Cases to Be Cautious Of</h6> 
       <ul> 
        <li> ISO 8601-encoded <a href="/help/api#datetime">datetime</a> values. The current server-side implementation returns most <code>datetime</code> values most of the time in the UTC time zone, denoted with the encoded time string ending in "Z". However, clients should be prepared to parse any legal ISO 8601 date/time value, including ones that are returned in a time zone other than UTC. </li> 
        <li> The <code>point_scale</code> attribute of <a href="/help/api/rest/v5#project_resource">project</a> is a string, not an enumeration. The client should be prepared to handle any comma-delimited list of decimal-encoded floating-point numbers as a <code>point_scale</code>. In addition, the set of point scales that are "built in" to Tracker may change at any time. The client should rely on the value of the <code>point_scale_is_custom</code> flag if it needs to distinguish built-in versus user-provided point scales, and not have its own list of what the built-in point scale values are. </li> 
        <li> The <code>estimate</code> attribute of <a href="/help/api/rest/v5#story_resource">story</a>. Note that the type of this attribute is <a href="/help/api#float">float</a>, even though currently only whole numbers are ever used (because Tracker doesn't currently permit point-scale strings to be specified with fractional values). <code>estimate</code> is explicitly defined as a <code>float</code> to permit future support of fractional values in a project's point scale. Clients will be expected to handle this when it occurs. </li> 
        <li> The <code>message</code> string in <a href="/help/api/rest/v5#activity_resource">activity</a> is an English string intended to be user-presentable, which summarizes the action that the <code>activity</code> represents. There are no guarantees as to the format or content of this string over time, except that the content of the <code>highlight</code> from the same <code>activity</code> record will be a substring of <code>message</code>. If you want to extract information from an <code>activity</code>, you should parse the object structure and not the content of the <code>message</code> string. </li> 
        <li> Note that the <code>kind</code> of <a href="/help/api/rest/v5#activity_resource">activity</a> structures is special, and rather than being the fixed string <code>"activity"</code>, it is an "open enumeration" of strings that end in ...<code>_activity</code>. Because the initial portion of each <code>activity</code>'s <code>kind</code> is a string made up of a resource name and a type of operation performed on that resource, it is possible for clients to encounter <code>activity</code> structures that reference operations and/or resources of which it is not aware. For purposes of displaying activity to users, the client may ignore such structures, or display the <code>message</code> string from them, or attempt to build a message based on the portions of the structure that it does recognize. For purposes of updating the client's local copy of project data (if any), it should attempt to process the <code>changes</code> array of all <code>activity</code> structures received, as it is very likely it will find changes to resources it does recognize even in a type of <code>activity</code> that it doesn't recognize. </li> 
       </ul> 
       <a class="jump_point" id="Interpreting_Maturity_Notes"></a> 
       <h5>Interpreting Maturity Notes</h5> 
       <p> The reference documentation for the API endpoints (URLs and request methods), and the underlying resource data structures manipulated by the API, includes a small fraction of items which are marked <b>unimplemented</b>, <b>experimental</b>, <b>beta</b>, or <b>deprecated</b>. These are warnings about the degree of support and stability that the marked items (entire endpoints/resources, or individual parameters or attributes). Items without one of these maturity indicators are "normal": they're fully supported, expected to behave reliably, and intended to be stable through the entire lifespan of the API version that includes them. Note endpoints/resources marked as experimental or beta may change and/or have different rate limits than "normal" endpoints. </p> 
      </section> 
      <section> 
       <a class="jump_point" id="Response_Customization"></a> 
       <h2 class="">Response Customization</h2> 
       <a class="jump_point" id="Default_Response_Content"></a> 
       <h4>Default Response Content</h4> 
       <a class="jump_point" id="Inclusion_of_Attributes"></a> 
       <h5>Inclusion of Attributes</h5> 
       <p> When making a request, the client can explicitly determine which attributes of each resource should be returned by sending the <code>fields</code> parameter. This parameter and its effects are described in <a href="#Selecting_Attributes_to_Include_in_Responses">Selecting Attributes to Include in Responses</a>. </p>
       <p> There are some attributes that are always included in the representation of a resource when a response is generated, regardless of the <code>fields</code> parameter. In the <a href="/help/api/rest/v5#Resources">Resources</a> section, the description of one of these attributes will include the sentence <b>This field is always returned.</b> For other attributes, most are included by default when <code>fields</code> isn't used. Some, however, are defined and available, but only included in a response when specified with <code>fields</code>. The description of an attribute that is not put into responses by default will include the sentence <b>This field is excluded by default.</b> </p>
       <p> Also regardless of the <code>fields</code> parameter, scalar attributes whose values are "empty" (missing, undefined) are not included in the response. For example, if the following were returned from a request with <code>fields=name,description</code> </p> 
       <code class="json example_box"> [ { "id": 10, "name": "Establish ice-surface transport landing area", "description": "We have to ensure that we've got enough cleared and smoothed ice to land all our transports" }, { "id": 12, "name": "Reassemble main power generator for base" } ] </code> 
       <p> it would indicate that the second story's description is empty. This does not apply to attributes whose values are arrays: empty arrays and nested structures are always explicitly included unless the field is omitted due to the <code>fields</code> parameter. For example, the following might be returned by a GET of <code>/projects?fields=label_ids,story_ids,epic_ids</code> </p> 
       <code class="json example_box"> [ { "id": 5, "label_ids": [1992, 2011], "story_ids": [42, 973, 55, 1414, 1415, 1418, 828], "epic_ids": [] }, { "id": 7, "label_ids": [], "story_ids": [], "epic_ids": [] }, { "id": 15, "label_ids": [749, 1141, 1238], "story_ids": [], "epic_ids": [3, 5, 9] } ] </code> 
       <p> In this example, the authenticated user is associated with three projects. The first project contains several stories and labels, but no epics. The second project is essentially empty, and the third project may be in the planning phaseseveral epics but no stories yet. </p>
       <p> Note that empty/missing is defined as <code>null</code> for all scalar resource attribute types. This is unimportant for interpreting resources in responses (where they won't be included), but for operations used to change an attribute from some other value to empty must include the attribute name with a value of <code>null</code>. When a new resource object is created by an API operation (a POST to a resource endpoint) the response contains the complete (non-<code>null</code>) content of the object, including any default values supplied by the server. </p> 
       <a class="jump_point" id="Representation_of_Associations_Between_Resources"></a> 
       <h5>Representation of Associations Between Resources</h5> 
       <p> Many of the attributes of resources are simple values as described above. Others represent the relationship between resources. For example, the <a href="/help/api/rest/v5#epic_resource">epic</a> resource contains several attributes that express relationships: <code>project_id</code>, <code>label_id</code>, <code>after_id</code>, <code>before_id</code>, and <code>comment_ids</code>. </p>
       <p> Of these, <code>project_id</code> is a reference to a resource that <em>contains</em> the epic, and potentially other epics as well. <code>before_id</code> and <code>after_id</code> are also special, because they are associations to other instances of the same type of resource. These are always returned as numeric IDs (including nested copies could lead to a request that should return a single <code>epic</code> actually returning multiple copies of every epic in the project). </p>
       <p> For the other association attributes, they may be included in responses (or provided as parameters) either as an integer ID or as a nested structure containing attributes of the associated type of resource. Whether an association attribute is an integer (or array of integers) or it is a structure (or array of structures) is determined by the attribute name that is used. Most resource attributes whose name ends in "<code>_id</code>" or "<code>_ids</code>" implies the presence of an alternative, structure-containing attribute that has the same base name but without the _id suffix. (There are a few exceptions, attributes which are strings that contain IDs used by external systems with which Tracker integrates. These do not behave like association attributes, and are marked in the individual resource descriptions.) </p>
       <p> Continuing to use epic as an example, the implied attribute names for complete resources are: <code>label</code>, <code>project</code> and <code>comments</code>. Note that <code>comments</code> is plural, because <code>comment_ids</code> attribute is plural. </p> 
       <a class="jump_point" id="Response_Controlling_Parameters"></a> 
       <h4>Response-Controlling Parameters</h4> 
       <p> This section describes standard parameters that the client can include as part of any request. These parameters are listed/described here only. They apply to all API requests, but for clarity their definition is not repeated again in the description of each individual request. </p>
       <p> Each successful request causes the server to generate a response containing either a single data structure or an array of structures. The schema for these structures (the type of object they encode) is indicated in the description of each request. Each structure is a composition of scalar values, arrays, and/or nested sub-structures. Each possible response data structure has a default composition, the particular elements that will be included in the response if the requesting client doesn't include any parameters to affect the response format. </p>
       <p> In general, the default is to: </p> 
       <ul> 
        <li>exclude all scalar values that do not have a defined value,</li> 
        <li>include all other scalar values that can occur in a structure,</li> 
        <li>exclude all association collections (associations whose values are arrays),</li> 
        <li>return any collections that are included as arrays of nested resource hashes rather than IDs, and</li> 
        <li>return scalar associations as integer IDs.</li> 
       </ul> 
       <p> Responses from individual API requests can differ from this general case, intended to make responses to request without configuration parameters fit the most common use cases. Each is noted in the description of the particular field(s) in the <a href="/help/api/rest/v5#Resources">Resources</a> section. </p> 
       <a class="jump_point" id="Selecting_Attributes_to_Include_in_Responses"></a> 
       <h5>Selecting Attributes to Include in Responses</h5> 
       <p> Control of what fields are included in responses, and whether an association's content is expressed as a resource ID (or <code>number</code>) or as a nested resource hash, is accomplished through the <code>fields</code> parameter. This query parameter is accepted by almost every API request, and its value controls the content of the response. However, it cannot be used to control responses in CSV format. When "the default" of what is included in a response is discussed, this is the API's behavior when no <code>fields</code> parameter is sent by the client. </p>
       <p> There are several different aspects of response generation that can be affected by the <code>fields</code> parameter. Each one of the following subsections addresses one of these aspects, starting from the simplest use of <code>fields</code> and continuing through how all the different types of behavior interact. </p> 
       <a id="fields"></a> 
       <a class="jump_point" id="fields_for_Single_Resources"></a> 
       <h6>fields for Single Resources</h6> 
       <p> If the <code>fields</code> parameter is sent with a request, only those resource attributes whose names are specified by the comma-separated list that is the value <code>fields</code> parameter will be included in the response. </p>
       <p> For example, without <code>fields</code> the request <code>GET /services/v5/projects/1/stories/42/tasks</code> could produce the following reply: </p> 
       <code class="json example_box"> [ { "id": 10, "kind": "task", "story_id": 42, "description": "Refuel X-wing fighters", "complete": false, "position": 1, "created_at": "2013-03-19T05:51:00Z", "updated_at": "2013-03-19T05:51:00Z" }, { "id": 12, "kind": "task", "story_id": 42, "description": "Run diagnostics on all flightline R2 droids", "complete": false, "position": 2, "created_at": "2013-03-19T05:51:25Z", "updated_at": "2013-03-19T05:51:25Z" } ] </code> 
       <p> If the request had instead been executed using the <code>fields</code> parameter, like <code>GET /services/v5/projects/1/stories/42/tasks?fields=description,complete</code>, then the response would have been: </p> 
       <code class="json example_box"> [ { "id": 10, "description": "Refuel X-wing fighters", "complete": false }, { "id": 12, "description": "Run diagnostics on all flightline R2 droids", "complete": false } ] </code> 
       <p> Note that even though it was not included in the <code>fields</code> list, the <code>id</code> element is included in the response structure for each task. This is because some elements, like <code>id</code>, are included in responses regardless of the content of the <code>fields</code> parameter. These "always returned" fields are indicated in the description of each attribute in the <a href="/help/api/rest/v5#Resources">Resources</a> section. </p> 
       <a class="jump_point" id="fields_for_Controlling_Associations"></a> 
       <h6>fields for Controlling Associations</h6> 
       <p> Just as scalar resource attribute names can be listed in the value of the <code>fields</code> parameter to select what should be included in server responses, including the names of associations will cause them to be included in the response. </p>
       <p> For example, the request <code>GET /services/v5/projects/1/stories/42</code> could produce the following reply: </p> 
       <code class="json example_box"> { "id": 42, "kind": "story", "project_id": 1, "name": "Prepare fighter wing to assault Death Star", "description": "Get Red, Gold, and Blue squadrons prepped to fly ASAP. Includes getting the new astromech droids from Commenor integrated.", "story_type": "feature", "current_state": "unstarted", "estimate": 2, "requested_by_id": 421, "labels": [], "url": "https://www.pivotaltracker.com/projects/1/stories/42", "created_at": "2013-03-19T05:30:13Z", "updated_at": "2013-03-19T05:30:13Z" } </code> 
       <p> This example reflects the default set of response fields for the <code>story</code> resource. Several of the story attributes that are not excluded by default are not present in this example, illustrating the fact that "empty" attributes are omitted from responses regardless of whether they're ordinarily included by default or have been explicitly requested through the <code>fields</code> parameter. With this same data, the request <code>GET /services/v5/projects/1/stories/42?fields=name,description,story_type,requested_by,owned_by,label_ids,comments,tasks</code> could produce the following reply: </p> 
       <code class="json example_box"> { "id": 42, "name": "Prepare fighter wing to assault Death Star", "description": "Get Red, Gold, and Blue squadrons prepped to fly ASAP. Includes getting the new astromech droids from Commenor integrated.", "story_type": "feature", "requested_by": { "id": 421, "kind": "person", "name": "General Jan Dodonna", "initials": "JD", "email": "dodonna@yavin4.alliance-secure" }, "label_ids": [], "comments": [], "tasks": [ { "id": 10, "kind": "task", "story_id": 42, "description": "Refuel X-wing fighters", "complete": false, "position": 1, "created_at": "2013-03-19T05:51:00Z", "updated_at": "2013-03-19T05:51:00Z" }, { "id": 12, "kind": "task", "story_id": 42, "description": "Run diagnostics on all flightline R2 droids", "complete": false, "position": 2, "created_at": "2013-03-19T05:51:25Z", "updated_at": "2013-03-19T05:51:25Z" } ] } </code> 
       <p>There are several things to note in this example:</p> 
       <ul> 
        <li> Only the scalar fields included in the <code>fields</code> parameter's list are in the response. Others that were included in the default response are omitted. </li> 
        <li> <code>requested_by</code> has replaced the default <code>requested_by_id</code>, and now contains a nested resource structure containing the information for the person. </li> 
        <li> <code>owned_by</code> is not present, just as <code>owned_by_id</code> wasn't present in the original example because (presumably) the story has no owner. Because <code>owned_by</code> is a scalar association, it is omitted when empty, whether the requested expression of the association is an ID or a nested structure. </li> 
        <li> <code>label_ids</code> has replaced the default <code>labels</code> because it was explicitly requested, and is still present in the result because it is a collection association, and array-valued fields are always included, even when empty. </li> 
        <li> The nested person and task structures follow their own specific defaults. So, specifically, they all contain the <code>kind</code> field, even though <code>kind</code> was not included in the <code>fields</code> list and there is no <code>kind</code> field in the top-level story structure. </li> 
       </ul> 
       <a class="jump_point" id="fields_Starting_From_the_default"></a> 
       <h6>fields Starting From the :default</h6> 
       <p> In cases where the client wants to fetch most or all of the attributes of a resource plus one or more of the available attributes that are not included in responses by default, the value strings of the <code>fields</code> parameter can become quite long. To help address this, the string <code>:default</code> can be included in a <code>fields</code> parameter as if it were an attribute name, and it instructs the server to include all of the resource attributes in the current response that would normally be included by default. For example, the request <code>GET /services/v5/projects/1/epics/5</code> could produce this response: </p> 
       <code class="json example_box"> { "kind": "epic", "id": 5, "project_id": 99, "name": "Build-out of Emperor's Suite", "label": { "kind": "label", "id": 19, "project_id": 99, "name": "suite", "created_at": "2013-03-19T05:30:13Z", "updated_at": "2013-03-19T05:30:13Z" }, "created_at": "2013-03-19T05:30:13Z", "updated_at": "2013-03-19T05:30:13Z" } </code> 
       <p> To obtain this same response but also include the list of the epic's comment IDs, the following could be used as the value for the <code>fields</code> parameter: </p> 
       <pre class="example_box">
  kind,project_id,name,label,created_at,updated_at,comment_ids
</pre> 
       <p> This is relatively long, and can be much longer for resources with more attributes. Using the <code>:default</code> short-hand, it can be replaced by </p> 
       <pre class="example_box">
  :default,comment_ids
</pre> 
       <p> So that the request <code>GET /services/v5/projects/1/epics/5?fields=:default,comment_ids</code> could produce this response: </p> 
       <code class="json example_box"> { "kind": "epic", "id": 5, "project_id": 99, "name": "Build-out of Emperor's Suite", "comment_ids": [ 23, 28, 34, 67, 72, 97, 100, 104, 113, 126, 129, 137 ], "label": { "kind": "label", "id": 19, "project_id": 99, "name": "suite", "created_at": "2013-03-19T05:30:13Z", "updated_at": "2013-03-19T05:30:13Z" }, "created_at": "2013-03-19T05:30:13Z", "updated_at": "2013-03-19T05:30:13Z" } </code> 
       <p> It is possible to specify an attribute name along with <code>:default</code> that over-rides a default attribute. For example, by default the <code>label_id</code> attribute of an epic is included in results as <code>label</code> with a nested structure representing the label resource's content. To cause epics to be returned by the API with the label as only an ID, the fields value of <code>:default,label_id</code> could be used. A request with that value for <code>fields</code> could produce the following response: </p> 
       <code class="json example_box"> { "kind": "epic", "id": 5, "project_id": 99, "name": "Build-out of Emperor's Suite", "label_id": 19, "created_at": "2013-03-19T05:30:13Z", "updated_at": "2013-03-19T05:30:13Z" } </code> 
       <a class="jump_point" id="fields_for_Selecting_Content_of_Nested_Resources"></a> 
       <h6>fields for Selecting Content of Nested Resources</h6> 
       <p> An association (collection or not) will be filled in with a nested structure when its non-<code>_id</code> name is in the fields list. If only the association's name is present, then the nested structure will conform to that particular resource type's response defaults, as shown in the example above. The fields included in a nested structure can be controlled by following the association field's name with a pair of parentheses surrounding the field list <em>for the nested structure</em>. The general form of this kind of value to the <code>fields</code> parameter is: </p> 
       <pre class="example_box">
  scalar,scalar,association(subfield,subfield),scalar
</pre> 
       <p> In this example, the "scalar" and "association" entries control the population of the top-level resource type being returned by the request. The "subfield" entries are the names of attributes (scalar or association) of the type of resource referenced by the top-level resource's association. Any association name can be followed by a parenthesized list of sub-resource field names, even if it is already inside the parenthesized list for another association. </p>
       <p> More concretely, the request </p> 
       <code class="example_box"> GET /services/v5/projects/1/epics/4?fields=name,description,label(name),comments(text,file_attachments,google_attachments(title)) </code> 
       <p> could produce the following reply: </p> 
       <code class="json example_box"> { "id": 4, "name": "Operation Skyhook", "description": "Obtain plans for and intelligence about the Empire's new weapon station.", "label": { "id": 24, "name": "skyhook" }, "comments": [ { "id": 1433, "text": "Here's the list of our current Bothan contacts. Eye's Only.", "file_attachments": [], "google_attachments": [{ "id": 93, "title": "Bothan Humint Latest" }] }, { "id": 4242, "text": "Death Star plans in hand! See attached.", "file_attachments": [{ "id": 126, "kind": "file_attachment", "filename": "station.vrml", "size": 67832388065637, "width": 4096, "height": 4096, "uploader_id": 88, "thumbnail_url": "https://s3.amazonaws.com/prod.tracker2/resource/1622777/thumb.gif?AWSAccessKeyId=AKIAIKWOAN6H4H3QMJ6Q&amp;Expires=1363677792&amp;Signature=Q6BZTAUtJXsiCQ9R%2BkcHoL%2BfSUk%3D", "thumbnailable": true, "uploaded": true, "download_url": "https://www.pivotaltracker.com/resource/download/1622777", "big_url": "https://s3.amazonaws.com/prod.tracker2/resource/1622777/bigthumb.gif?AWSAccessKeyId=AKIAIKWOAN6H4H3QMJ6Q&amp;Expires=1363677792&amp;Signature=Q6BZTAUtJXsiCQ9R%2BkcHoL%2BfSUk%3D", "created_at": "2013-03-19T06:56:19Z", "content_type": "model/vrml7" }], "google_attachments": [] } ] } </code> 
       <p> The <code>:default</code> short-hand can be used at any level of the attribute selection within nested resources. Also, it doesn't matter what position within a comma-separated list of attribute names the request </p> 
       <code class="example_box"> GET /services/v5/projects/1/epics/4?fields=name,description,comments(file_attachment_ids,:default) </code> 
       <p> could produce the following reply: </p>

       <a class="jump_point" id="Selecting_the_Date_Time_Format"></a> 
       <h5>Selecting the Date/Time Format</h5> 
       <p> Several fields of different resources contain date/time values. These can be encoded either in a string formatted according to the ISO 8601 standard (<a href="http://www.iso.org/iso/catalogue_detail?csnumber=40874">Representation of dates and times</a>) or as an integer count of milliseconds elapsed since the epoch, as is commonly used by the JavaScript Date object (as in <code><a href="https://developer.mozilla.org/en-US/docs/JavaScript/Reference/Global_Objects/Date">new Date(value)</a></code>). </p>
       <p> When sending date/time values to Tracker in an API request, the client can supply them in either format. Tracker will determine which is being presented by examining the value, and interpret it accordingly. This parameter will work with responses in the CSV format. </p>
       <p> Response date/time values are in the ISO 8601 by default. If the client wants to receive date/times in milliseconds, the following parameter should be added to the request's query string: </p> 
       <pre class="example_box">
  ?date_format=millis
</pre> 
       <p> The <code>date_format</code> parameter applies to the entirety of a responsethere is no way to request that some date fields be formatted in milliseconds and others as ISO 8601. </p> 
       <a class="jump_point" id="Paginating_List_Responses"></a> 
       <h5>Paginating List Responses</h5> 
       <p> Many of the API's GET requests that return an array of resource structures support pagination, so that the request can be repeated multiple times with all but the pagination parameters the same, and obtain a different subset of the overall available result set for each response. When a request supports pagination of the results, it is noted in the description of the operation below. The <a href="#Envelope_Return_Structure">Envelope Return Structure</a> section describes the <code>pagination</code> structure that is returned with each result from a paginated endpoint, and which contains information about the subset of the possible results that is contained in the current response. It is possible to use these parameters when requesting a response in CSV format. </p>
       <p> The two request query parameters that control pagination are: </p> 
       <ul> 
        <li> <code>offset</code>  An integer, that gives the number of items between the first item returned by default (no <code>offset</code> provided) and the first item that is desired in the response to the current request. If no <code>offset</code> parameter is supplied for a page-able request, the default offset used will be 0. Most page-able requests return the first available item with <code>offset=0</code>. However, there are a couple of endpoints where the URL used already implies a subset of a larger possible result space. In this case, negative offset values are acceptable to specify that the results should include items that come before the default first item. </li> 
        <li> <code>limit</code>  An integer, that gives the maximum number of items to include in the response to the current request. If no <code>limit</code> parameter is supplied for a paginated request, a default limit value will be used. If the <code>limit</code> parameter value supplied is larger than the maximum <code>limit</code> currently available through the endpoint, the <code>limit</code> used is capped at the maximum, and the value used, rather than the value from the request, will be reflected in the <code>pagination.limit</code> (or <code>X-Tracker-Pagination-Limit</code>) value in the response. <br> Each paginated operation has its own independent default and maximum limit values, which will change from time to time, as the result of server updates or dynamic conditions. </li> 
       </ul> 
       <p> The <code>pagination</code> structure included in the response from a page-able request includes the information necessary for the client to generate requests for the next or previous group of items, and to know how many items make up the total possible data set. </p>
       <p> For example, the request: </p> 
       <pre class="example_box">
  GET projects/1/stories?offset=25&amp;limit=300&amp;envelope=true
</pre> 
       <p> Could generate the response: </p> 
       <code class="json example_box"> { "http_status": "200", "project_version": 42, "pagination": { "total": 1543, "limit": 300, "offset": 25, "returned": 298 }, "data": [] } </code> 
       <p> The <code>offset</code> value in the response will always be either the parameter value from the request or the default provided by the server. If there was no <code>limit</code> parameter provided in the request, then the <code>limit</code> value in the response will be the default limit currently available for the request operation performed. If a <code>limit</code> parameter was included in the request, then the <code>limit</code> value in the response will be the minimum of it and the maximum allowed limit. </p>
       <p> The <code>total</code> value in the response is the number of available item positions <em>at the current time</em> that correspond to the other (the non-pagination) request parameters provided. Depending on the endpoint, it is completely possible for the value of the <code>total</code> to change between subsequent requests, even those made very close together in time. The value of <code>returned</code> in the response gives the number of entries in the data array within the response envelope (<code>stories</code> in the example above). </p>
       <p> There are several reasons why the value of <code>returned</code> may be smaller than that of <code>limit</code>. First, if the <code>offset</code> is large enough that there are fewer result items remaining than the <code>limit</code>, then <code>returned</code> will reflect the actual number of items in the response. For example, assume that the <code>total</code> in a response is the same as shown above, 1543, and that the request's <code>offset</code> was 1500 and the <code>limit</code> was 150. In that circumstance, <code>returned</code> could be 43. </p>
       <p> There is a second reason that <code>returned</code> might be less than <code>limit</code> (or less than <code>total</code> in the case where <code>total</code> is also less than <code>limit</code>). Occasionally, due to the way Tracker indexes information, there will be item locations counted in <code>total</code> that do not actually contain items. For example, with a <code>total</code> of 1543, and the request's <code>offset</code> equal to 100 and <code>limit</code> equal to 150, it is reasonable to expect that <code>returned</code> would be equal to 150. And most of the time it will be. But occasionally it might be 149 or 137. If this case occurs, the client should still add the current <code>offset</code> and <code>limit</code> values in order to generate the <code>offset</code> parameter for the next request if the desired behavior is to obtain successive non-overlapping pages of data. </p>
       <p> Third, it is possible for the underlying data that is being requested to change between one request and another using pagination to request a different portion of the same results. At a minimum, clients should read the values of <code>total</code> and <code>limit</code> provided in each response and not simply assume that the values returned in the first request of a set remain fixed. Depending on the client's requirements for precision in data use/presentation, it may not need to do any other processing. However, if the client has to be guaranteed to access all of the data items that match the request, it may need to be capable of re-fetching some pages or overlapping pages fetched in order to detect/compensate for data changes. </p> 
       <a class="jump_point" id="CSV_Response_Format"></a> 
       <h6>CSV Response Format</h6>

All responses in this format are the structured the same way, with headers setting the content type and disposition, and comma separated text values in the body. Datetimes are returned in user's local time zone in the <code>MM/DD/YYYY HH:MM:SS</code> format. Alternative date/time formats, the envelope option and field selection are not supported. </p>

       <h2 class="">Activity Information</h2> 
       <p> The Tracker API can provide information about the sequence of changes made to the content of projects by returning data formatted as <a href="/help/api/rest/v5#activity_resource">activity</a> structures. There are API endpoints to retrieve recent <code>activity</code> within a particular scope, 
        <!--
<code>activity</code> is returned as a component of a processed command from endpoints like
intradoc_link('/commands', 'v5') -%> and
intradoc_link('/project_stale_commands/{project_id}/{project_version}', 'v5') -%>
    --> and Pivotal Tracker can POST JSON <code>activity</code> structures to the Internet URL of your choice using <a href="/help/articles/activity_webhook">web hooks</a>. </p> 
       <a class="jump_point" id="The_Activity_Endpoints"></a> 
       <h4>The Activity Endpoints</h4> 
       <p> There are four different <a href="/help/api/rest/v5#Activity">Activity</a> endpoints. Their parameters, pagination, and responses (arrays of <a href="/help/api/rest/v5#activity_resource">activity</a> resources) are all common. The difference between them is that each implicitly filters the activity selected for the response by different criteria. </p>
       <p> Information from all of these endpoints is returned in <em>reverse chronologic order</em>, so that the first activity in a response indicates the most recently made modification to the project, and increasing values of the <code>offset</code> pagination parameter select older and older portions of the project's history. Clients using this endpoint should take into account that other clients, including users working in the Pivotal Tracker web application, can be continuously making changes that cause new activity to be added to the <em>beginning</em> of the responses from this endpoint. Not only should it be expected that, over time, the responses to identical activity requests will change, but the a client which is paging through multiple activity requests and wants to guarantee that it processes all activity exactly once should pay attention to the project <code>version</code> numbers of activity it has already handled, as the same activity structures can occur at the end of one request and at the beginning of a subsequent request for the next page of activity if updates have been made to the data on the server between those the requests. </p> 
       <dl> 
        <dt>
         Project Activity
        </dt> 
        <dd>
          The endpoint 
         <a href="/help/api/rest/v5#projects_project_id_activity">/projects/{project_id}/activity</a> returns all of the activity (over the selected period) that affected the project whose ID is included in the request path, regardless of who took the actions that caused the activity or what resources within the project were modified. This endpoint returns information equivalent to the "Project History" panel in Pivotal Tracker's native web UI. 
        </dd> 
        <dt>
         Story Activity
        </dt> 
        <dd>
          The endpoint 
         <a href="/help/api/rest/v5#projects_project_id_stories_story_id_activity">/projects/{project_id}/stories/{story_id}/activity</a> returns all of the activity (over the selected period) that affected the single story identified by the request path. This will always be a subset of the activity for the story's project over the same period, and is equivalent to the content of a story-specific History panel in Pivotal Tracker. 
        </dd> 
        <dt>
         Epic Activity
        </dt> 
        <dd>
          The endpoint 
         <a href="/help/api/rest/v5#projects_project_id_epics_epic_id_activity">/projects/{project_id}/epics/{epic_id}/activity</a> returns all of the activity (over the selected period) that affected the single epic identified by the request path. This will always be a subset of the activity for the story's project over the same period, and is equivalent to the content of a story-specific History panel in Pivotal Tracker. Note that the activities returned by this endpoint are ones that affected the epic directly. This endpoint does not return the collective activity for all of the stories contained in an epic. 
        </dd> 
        <dt>
         User Activity
        </dt> 
        <dd>
          The endpoint 
         <a href="/help/api/rest/v5#my_activity">/my/activity</a> returns all of the activity (over the selected period) that was instigated by the user whose authentication credentials were used to make the request. Unlike the three other activity endpoints, this endpoint can return activity from a variety of projects. In this case, activity structures continue to be returned in reverse chronologic order over all, regardless of the relative 
         <code>version</code> number sequences in the different projects. 
        </dd> 
       </dl> 
       <p> See the references for these endpoints for examples of <code>activity</code> items. </p>

       <a class="jump_point" id="Default_Message_and_Metadata"></a> 
       <h5>Default Message and Metadata</h5> 
       <p> Each <code>activity</code> structure contains an English-language <code>message</code> string composed by the server which is intended to be displayable to the user. Clients may choose to simply present this string to the user as-is. (Note that it should only ever be presented to the user. The actual content of the messages we generate for the same types of activity may change over time, and parsing the message string to obtain data about the activity is strongly discouraged.) </p>
       <p> If the client wishes, it can use the <code>highlight</code> string as a guide to how the message may be formatted. The value of <code>highlight</code> will always be a substring of the <code>message</code> and can be used to divide the message into three parts (before the <code>highlight</code>, the <code>highlight</code> itself, and after). Many messages correspond to a user-verb-object structure, with the verb being contained in the <code>highlight</code>. The Pivotal Tracker web interface displays the <code>highlight</code> in an empasis font, relative to the rest of the <code>message</code>, and whether highlight identifies the verb in the message or not, it will correspond to the portion of the message that we believe deserves accenting relative to the rest of it. </p>
       <p> The other consideration when using the default <code>message</code> is that it may be phrased to assume the context in which it is displayed. For example, some messages use indefinite phrases as objects, such as "this feature" for an activity that describes a change to a feature-type <a href="/help/api/rest/v5#story_resource">story</a>. The resource(s) that any object phrase references are those included in the <code>primary_resources</code> array (more on this attribute below). It is recommended that a client using the default <code>message</code> display identifying information from the <code>primary_resources</code> proximal to the message. </p>
       <p> The time at which the operation which is represented by an activity item was performed is in the <code>occurred_at</code> attribute of an <code>activity</code>. </p>
       <p> The <code>primary_resources</code> array contains a list of the identifying information for the resource(s) that were updated by the operation. It is an array because activity can result from user operations that are performed on a number of resources simultaneously. For example, when multiple stories are moved (reprioritized) as a group, one hash occurs in <code>primary_resources</code> for each story affected. In addition to identifying the context that might be referenced by the default <code>message</code> string, <code>primary_resources</code> can be used for grouping. In the "Project History" panel in Pivotal Tracker, operations performed on the same "thing" at around the same time are displayed under a banner identifying the "thing" and with sub-items organized chronologically instead of in reverse (the overall organization of the different History panels). The <code>primary_resources</code> array is what is used to identify the "thing" (resource or group of resources) that have been acted on for grouping purposes. </p>
       <p> What resource(s) appear in <code>primary_resources</code> depend on both the type of activity and the kind of resources. In particular, resources that are in the object hierarchy below stories and epics (<a href="/help/api/rest/v5#comment_resource">comment</a>s, <a href="/help/api/rest/v5#task_resource">task</a>s, <a href="/help/api/rest/v5#follower_resource">follower</a>s, etc.) do not appear in <code>primary_resources</code>. Instead, the story or epic that they belong to will be listed. Most other activity lists the same type of resources in the <code>primary_resources</code> array as are named in the activity's <code>kind</code>, but resources that are children of <a href="/help/api/rest/v5#project_resource">project</a> <em>may</em> have the project that contains them as the entry in <code>primary_resources</code>. </p>
       <p> A hash in the <code>primary_resources</code> array will have the following keys: </p> 
       <ul> 
        <li> <code>kind</code>  This reflects the <code>kind</code> attribute of the resource that is being referenced by this <code>primary_resources</code> entry. </li> 
        <li> <code>id</code> or <code>number</code>  This is the unique identifier for the particular resource being referenced. As always, <a href="/help/api/rest/v5#iteration_resource">iteration</a> and <a href="/help/api/rest/v5#iteration_override_resource">iteration_override</a> have <code>number</code> while other persistent resources have <code>id</code>. </li> 
        <li> An identifying string  In most cases there is a string-type value in the hash in <code>primary_resources</code>. The content of this key will be ellipsified if the matching attribute from the referenced resource is too long. The key for this string will match the "main" string attribute in the type of resource being represented. For example, the key is <code>name</code> for stories, epics, projects and other resources with a <code>name</code> attribute. The key is <code>description</code> for tasks and <code>text</code> for comments. The resources for which no identifying string is available include things like <code>iteration</code> and <code>iteration_override</code>. </li> 
        <li> Sometimes <code>story_type</code>  when the resource is a story, its hash in <code>primary_resources</code> will contain this additional key. </li> 
        <li> Sometimes <code>url</code>  when the resource is a story or an epic, its hash in <code>primary_resources</code> will contain this additional key. </li> 
       </ul> 
       <p> Note that the "identifying information" values in <code>primary_resources</code> reflect <em>current</em> information. For example, if a comment had been added to a story five months ago, fetching the <code>activity</code> for that occurrence immediately afterward would give you the story name that was present at that time. But if the story's name had been edited a month ago, and now you fetch the <code>activity</code> for that same change, the response will contain the story's current name, and not the value that the story name had at the time the comment was added. This is intended to ensure that users looking at activity see reference information that matches what they'll see elsewhere in Tracker at the moment. </p> 
       <a class="jump_point" id="Creating_a_Message"></a> 
       <h5>Creating a Message</h5> 
       <p> A client might want to make its own message string rather than use the default message provided in the <code>activity</code>. Perhaps you want to eliminate the indefinite references in some Tracker messages ("this story"), or want to create a message with language localization. The <code>activity</code> contains all of the information necessary to create the default message strings and the context in which they are displayed in the various History panels in the Tracker app. </p>
       <p> The phrasing of messages, choice of words (like the verb in the default message), etc. will usually require the <code>kind</code> of the <code>activity</code> to be parsed by the client. While new words may be introduced as new <code>activity</code> structures are added to the API over time, the existing vocabulary of activity <code>kind</code>s is a stable part of the API specification. Thus, the client can rely on <code>story_update_activity</code> to always be identified in the same way, so that it can create a stable mapping to the fixed language in its user representation of <code>activity</code>. </p>
       <p> As mentioned above, the information needed to identify and describe to the user the resource(s) that were affected by the action being described is present in the <code>primary_resources</code> array. </p>
       <p> In addition, accurately describing an <code>activity</code> to the user frequently requires information about resources other than just the "primary". For example, if the <code>activity</code> represents a user's clicking the check box on a story's task to mark it completed, the <code>primary_resources</code> array will contain the identifying information about the story. However, the information identifying the task that was modified is included in the <code>changes</code> array along with the information about exactly what was modified. The exact structure is in <a href="/help/api#History_Details">History Details</a>, next. </p>
       <p> Similarly, you may need to examine the exact resource attributes updated in order to make very specific messages. For example, a <code>story_update_activity</code> that represents a user changing just a story's state from <code>"started"</code> to <code>"finished"</code> contains a message that says the user "finished" the story, rather than a more generic sentence about editing or updating the story. To generate messages this specific, you'll have to look at the presence (and absence) of specific attributes in a particular resource's entry in the <code>changes</code> array. </p> 
       <a class="jump_point" id="History_Details"></a> 
       <h5>History Details</h5> 
       <p> When Pivotal Tracker displays information in one of its History panels, it not only includes the <code>message</code> strings from activity, and groups multiple activities together based on matching <code>primary_resources</code>, but it also includes an expandable "details" section for some kinds of history entries. The information in these sections comes from the <code>changes</code> array in <code>activity</code>. </p>
       <p> The <code>changes</code> array contains one or more hashes, each of which contains identifying information about the resource, a list of all the changed attributes' new values, and a list of those attributes' values from immediately before the change. For example (for reference, a matching default message string is "Darth Vader started this feature"): </p> 
       <code class="json example_box"> { "changes": [{ "kind": "story", "id": 556, "name": "Interrogate Leia Organa", "story_type": "feature", "change_type": "update", "original_values": { "after_id": 557, "before_id": 555, "current_state": "unscheduled" }, "new_values": { "after_id": 559, "before_id": 566, "current_state": "started" } }] } </code> 
       <p> <code>kind</code>, <code>id</code> (or <code>number</code>), <code>name</code> (identifying string), and <code>story_type</code> are all present and populated identically to the matching hash entries present in a <code>primary_resources</code> hash, described above. (<code>url</code> is not included in the identifying information in the <code>changes</code> array.) </p>
       <p> <code>change_type</code> identifies how the resource has been modified. The possible values for this string are <code>create</code>, <code>update</code>, and <code>delete</code>. The set of possible values for <code>change_type</code> may be extended without a change in the Tracker API version number. Clients should be prepared to receive a hash that contains a <code>change_type</code> value that they do not know how to process, and should gracefully ignore such hashes. </p>
       <p> <code>new_values</code> is a hash. The keys of the hash are a subset of the attributes defined for the type of resource identified by the <code>changes</code> entry's <code>kind</code>. In this example, the single entry in <code>changes</code> represents a change to a story, so the keys that can occur in the <code>new_values</code> hash are those that are defined for a <a href="/help/api/rest/v5#story_resource">story</a>, the types of the values of each key conform to the <code>story</code> resource's schema, etc. There will not be a <code>new_values</code> entry in the case where <code>change_type</code> is <code>"delete"</code>. </p>
       <p> The values in <code>new_values</code> will remain invariant over time. Unlike the resource-identifying information in an <code>activity</code> in which <em>current</em> resource information is returned, these values are always the values that the activity actually produced, even though many of the fields modified may have been re-modified subsequently. </p>
       <p> Note that, in some cases, multiple keys that are mutually exclusive in other contexts where the resource's values might occur or be returned <em>will</em> be included in <code>new_values</code>. For example, when the labels on a story have been updated, both a <code>labels</code> key and a <code>label_ids</code> key will be present. Unlike the other cases in which a resource association may be represented by a non-<code>_id</code> attribute name (where the content of the attribute would be a complete nested resource), this key naming convention in <code>activity</code> will contain a string (or an array of strings, as in the <code>labels</code> case) that is the "identifying string" of the associated resource (<code>label.name</code> for example). This denormalization of the data is intended to provide clients the ability to present meaningful information to the user about a modified association without the client having to look up/fetch (all) of the resources referenced. </p>
       <p> As this information is captured in <code>new_values</code>, it represents the resource's identifying strings at the time the activity occurred, and not necessarily the current value at the time the <code>activity</code> structure is returned. If a client absolutely has to return the current name of a resource, it must dereference the resource ID also included in the <code>new_values</code> to obtain the resource's current state. On the other hand, if the resource has been deleted since the activity occurred, then this denormalization provides at least some user-meaningful information. </p>
       <p> The <code>original_values</code> is the mirror-image of <code>new_values</code> and contains the values that the changed resource attributes had prior to the change being made. There will only be an <code>original_values</code> entry in the case where <code>change_type</code> is <code>"update"</code>. </p> 
       <a class="jump_point" id="Local_Updates_from_Changes"></a> 
       <h5>Local Updates from Changes</h5> 
       <p> Some clients will be most efficiently implemented if they operate on the same basic set of information retrieved from Pivotal Tracker for a sustained period of time, and are capable of updating their locally-maintained copies of Tracker data in place, without re-fetching all of their data from Tracker periodically or when something indicates to them that data may have changed. </p>
       <p> Such a client can fetch <code>activity</code> hashes and use the various <code>new_values</code> hashes in their <code>changes</code> as <em>deltas</em> that allow them to incrementally update their locally-stored data in place in order to stay in sync with what's stored by the Tracker server and seen by users viewing the Tracker web application. Because of Tracker's business rules, it is possible for seemingly simple client requests to have side effects that affect a number of otherwise-unrelated objects. Processing <code>activity.changes</code> allows a client to benefit from all of the server's logic to perform system-wide updates, as all changes carried out by the server will be represented by some <code>new_values</code> applied to one or more (possibly unanticipated) resources. </p>
       <p> Basically, "all" the client has to do is process the entries from each <code>activity</code>'s <code>changes</code> in the order that those activities have occurred. This means that in those cases where a GET from an activity endpoint returns multiple items, the <em>last</em> entry in the response array should be the first one processed, because the activity endpoints return data in reverse-chronologic order. </p>
       <p> What merging the <code>new_values</code> into a client's representation of a particular resource entails is, of course, client specific. However, there are a number of easily-predicted side effects of some changes that will not be communicated by the server in <code>changes</code> because of the impact that would have on the size of the activity items. These are: </p> 
       <ul> 
        <li> Changes in <code>task.position</code>. When a task is moved up within the list of tasks, that implicitly means that all of the other tasks that are below its new position are implicitly moved down. If your client is storing the <code>position</code> attribute of <code>task</code>s explicitly, rather than (for example) keeping tasks in an array and determining <code>position</code> dynamically when needed based on a task's array index, then it is the client's responsibility to handle each change in <code>task.position</code> that occurs in <code>changes</code> by updating all necessary other <code>position</code> values of tasks on the same story. </li> 
        <li> Similarly, while changes to the <code>before_id</code> and <code>after_id</code> values for a story or epic (which are used as command parameters, although not currently returned as attributes of the resource from GET operations, which is why they are marked as Unimplemented in the <a href="/help/api/rest/v5#story_resource">story</a> resource's reference description) <em>are</em> returned in <code>activity</code> for operations which have caused the resource to move relative to its siblings. However, the implicit changes to other stories/epics that were not otherwise modified by the operation are not included in <code>changes</code>. For example, in the case where a single story is moved from one position in the Backlog to another, only that story will occur in the <code>changes</code> array. However, if your client is storing its copies of story resource data in a doubly-linked list with explicit and stored <code>before_id</code> and <code>after_id</code> values, then four other stories will also have to be updated: the two stories that the moved story used to be between, and the two stories that are around the position that the moved story went to. The client is responsible for performing these updates if necessary. </li> 
        <li> Changes to <code>iteration</code> start, end, and stories list. A number of changes to <a href="/help/api/rest/v5#story_resource">story</a>'s and <a href="/help/api/rest/v5#iteration_override_resource">iteration_override</a>s can cause the arrangement of iterations in the project to change. If your client displays or otherwise needs to track which story is in which iteration, you will need to update or refetch the iteration information following one of these changes. </li> 
        <li> Changes to associations between resources are represented as <code>changes</code> to only one of the resources involved. For example, when a comment is added to a story (created), the activity's <code>changes</code> array will contain an entry for both the story and the comment. The entry for the comment will be a <code>"create"</code> change that lists the <code>story_id</code> of the story that it was added to. This is the entry that indicates the change in the association between the two resources. The entry for the story is present to communicate the new value for the story's <code>updated_at</code> attribute, but the <code>comment_ids</code> attribute never shows up in a <code>changes</code> list. If the client keeps an explicit copy of this array (rather than translating story comment arrays it receives into a more direct link between its internal resource objects), then it is responsible for making updates to it as comment resources that reference the story are created and deleted. </li> 
       </ul> 
       <a class="jump_point" id="Data_Integrity_Over_Time"></a> 
       <h5>Data Integrity Over Time</h5> 
       <p> Tracker currently makes the last six month's worth of data changes available through the activity endpoints. In almost all cases, the above statements about the content of an <code>activity</code> structure are correct. In some exceptional cases (more likely for activity from older operations) not all of the data will be available to completely fill in the activity structure. In particular: </p> 
       <ul> 
        <li> In most cases, Tracker has "identifying strings" available even for resources that have been deleted by users. But in some cases, this isn't true. When this occurs, the API will return place-holder strings, for example <code>"*deleted*"</code>, as the content of an identifying string. The exact content of a place-holder string is not necessarily stable over time, and it is recommended that clients allow them to pass through to user presentation, rather than trying to detect and handle them. </li> 
        <li> <code>original_values</code>: Again, in most cases Tracker has the complete set of pre-existing values for updated attributes. But not in all. Clients should be prepared to handle <code>activity</code> items that are missing keys from an <code>original_values</code> hash (cases where a key that occurs in <code>new_values</code> doesn't occur in <code>original_values</code>), and to handle updates in which the entire <code>original_values</code> hash is missing. </li> 
        <li> In some cases for labels that have been deleted, Tracker has been able to retain the denormalized name that the label had at the time the original operation was performed, but not the <code>id</code> value that the deleted label used to have. When returning <code>activity</code> for (the few) cases where this occurred, Tracker will fill in the missing <code>id</code> values with <em>-1</em>. Clients should be prepared to handle this in a way that is appropriate for them. (Of course, if the client uses only the strings and never tries to look up a resource referenced by <code>activity</code> structures, then it would never look at the various resource <code>id</code>s, and never actually notice this inconsistency. </li> 
       </ul> 
      </section>

       <a class="jump_point" id="Reports_and_Analytics"></a> 
       <h2 class="">Reports and Analytics</h2> 
       <p> The Tracker API provides daily history and analytical information at the project, release and iteration level, and story progress for a given time frame. You can use this information to analyze project trends and produce reports of project and story progress. </p> 
       <p> All of the endpoints and resources highlighted below, either aggregate or expose data based on the basic building blocks of <a href="/help/api/rest/v5#activity_resource">activity</a>, <a href="/help/api/rest/v5#story_transition_resource">story_transition</a>s, and daily <a href="/help/api/rest/v5#project_snapshot_resource">project_snapshot</a>s. Tracker currently makes the last six month's worth of reports and analytical data available. This may be increased in the future. </p> 
       <p> In an effort to make to make reports and analytics data available to everyone, all of the new endpoints support the <a href="/help/api#CSV_Response_Format">CSV Response Format</a> as well as the default JSON encoding. </p> 
       <a class="jump_point" id="Analytical_Endpoints_and_Resources"></a> 
       <h4>Analytical Endpoints and Resources</h4> 
       <p> These endpoints enable API users to analyze project trends, find bottlenecks, and opportunities for process improvement. The data can be accessed through a combination of iteration and story level analytical endpoints and resources. </p> 
       <p> Story transition data can be accessed directly or through aggregate cycle time data at the iteration or story level. Tracker defines cycle time as the total time spent in each state from the time a story was started until it's been accepted. </p> 
       <dl> 
        <dt>
         Iteration Analytics
        </dt> 
        <dd>
         The 
         <a href="/help/api/rest/v5#analytics_resource">analytics</a> resource provides cycle time, rejection rate, the number of new bugs created, and the number of stories accepted. It can be optionally returned as part of the 
         <a href="/help/api/rest/v5#iteration_resource">iteration</a> resource. 
        </dd> 
        <dt>
         Story Cycle Time Details
        </dt> 
        <dd>
          The 
         <a href="/help/api/rest/v5#cycle_time_details_resource">cycle_time_details</a> resource provides detailed information around time spent and the number of times a story cycled through each state. This resource can be optionally returned as part of the 
         <a href="/help/api/rest/v5#story_resource">story</a> resource or by the endpoint 
         <a href="/help/api/rest/v5#projects_project_id_iterations_iteration_number_analytics_cycle_time_details">/projects/{project_id}/iterations/{iteration_number}/analytics/cycle_time_details</a> for all of the stories in a given iteration. 
        </dd> 
        <dt>
         Story Transitions
        </dt> 
        <dd>
          The 
         <a href="/help/api/rest/v5#projects_project_id_story_transitions">/projects/{project_id}/story_transitions</a> and 
         <a href="/help/api/rest/v5#projects_project_id_stories_story_id_transitions">/projects/{project_id}/stories/{story_id}/transitions</a> endpoints give you access to all of the recorded stories' transitions. A story transition is a time based event describing when a story transitions to a new state. This is the raw data used to calculate the cycle time details for each story. 
        </dd> 
       </dl> 
       <p>&nbsp;</p> 
       <a class="jump_point" id="Historical_Endpoints_and_Resources"></a> 
       <h4>Historical Endpoints and Resources</h4> 
       <p> The <a href="/help/api/rest/v5#Project_History">Project History</a> endpoints provide a means to retrieve a daily history of story points, counts by state, per day information about story state, estimate and position in current, backlog, and the icebox. All history endpoints represent the state of the project at midnight in the project's timezone. To focus in on a specific feature or category of stories, you can filter history endpoints by label. </p> 
       <dl> 
        <dt>
         Project History
        </dt> 
        <dd>
          The endpoint 
         <a href="/help/api/rest/v5#projects_project_id_history_days">/projects/{project_id}/history/days</a> returns daily history (over the selected period) for the entire project. If no start date or end date is provided, the project's most recent history is returned. 
        </dd> 
        <dt>
         Release History
        </dt> 
        <dd>
          The endpoint 
         <a href="/help/api/rest/v5#projects_project_id_history_releases_id_days">/projects/{project_id}/history/releases/{id}/days</a> returns daily history of the release id specified in the path. To find releases in your project, use the 
         <a href="/help/api/rest/v5#Stories">Stories</a> endpoint. Filter your search for unstarted releases by using the with_story_type=release and with_state=unstarted parameters. 
        </dd> 
        <dt>
         Iteration History
        </dt> 
        <dd>
          The endpoint 
         <a href="/help/api/rest/v5#projects_project_id_history_iterations_iteration_number_days">/projects/{project_id}/history/iterations/{iteration_number}/days</a> returns the daily history of the iteration number specified in the path. 
        </dd> 
        <dt>
         Project Snapshots
        </dt> 
        <dd>
          The endpoint 
         <a href="/help/api/rest/v5#projects_project_id_history_snapshots">/projects/{project_id}/history/snapshots</a> returns a list of 
         <a href="/help/api/rest/v5#project_snapshot_resource">project_snapshot</a>s that provide per day information about story state, estimate and position in current, backlog, and the icebox. 
        </dd> 
       </dl>

       <h2 class="">Special Endpoint Usage</h2> 
       <p> This section provides tutorial information for a number of different API operations whose semantics are more complicated than the straight-forward resource CRUD operations performed by the balance of the RESTy endpoints. </p> 
       <a class="jump_point" id="Manipulating_Attachments_on_Comments"></a> 
       <h4>Manipulating Attachments on Comments</h4> 
       <p> When comments (for stories or epics) are created or updated, they can have associated attachments, either files uploaded to Tracker or "files" stored within Google Apps. In both cases, the data must already exist before it can be associated with a comment being created. This section describes the sequence of operations used to create or access data for attachments, and how to create new comments that reference them. </p> 
       <a class="jump_point" id="Uploading_File_Attachments"></a> 
       <h5>Uploading File Attachments</h5> 
       <p> Before being attached to a comment, a file must be uploaded to the Tracker server, which creates a new <a href="/help/api/rest/v5#file_attachment_resource">file_attachment</a> resource. After it has been created, the information describing the <code>file_attachment</code> can be included in an operation that creates a new comment referencing it. </p>
       <p> File attachments are uploaded to Tracker by POST'ing to the <a href="/help/api/rest/v5#projects_project_id_uploads">/projects/{project_id}/uploads</a> endpoint. Unlike almost all other API endpoints, the request body POSTed to <code>/projects/{project_id}/uploads</code> is not expected to be JSON-encoded. Instead, it should be formatted and sent with the MIME type <code>multipart/form-data</code>. The following example shows a <code>curl</code> command that uploads a file from the local filesystem to Tracker. </p>

       <a class="jump_point" id="Downloading_File_Attachments"></a> 
       <h5>Downloading File Attachments</h5> 
       <p> To download a file that was attached to a comment, the client needs the unique ID of the attachment, which is available in the <code>file_attachment_ids</code> list in a <a href="/help/api/rest/v5#comment_resource">comment</a> resource. (Which the client can obtain from one of the <code>.../comments</code> endpoints or as a nested resource in another endpoint's result.) </p>
       <p> The request path (endpoint) used to download attachment files is unversioned, and not under the <code>/services/</code> URL space, but is still part of the Tracker API. GET requests may be made to the <a href="/help/api/rest/v5#ROOT_file_attachments_file_attachment_id_download">https://www.pivotaltracker.com/file_attachments/{file_attachment_id}/download</a> endpoint to retrieve the attachment file content. </p>
       <p> The download endpoint is consistent with other API endpoints for authentication and authorization, and in the case of failures will return a JSON error information hash. There are two primary differences between performing a GET request to this endpoint and the other API operations available: </p> 
       <ul> 
        <li> The body of a successful response will be the originally-uploaded content of the attachment, with the associated MIME type. It will not be a JSON-encoded data structure as is returned by (almost all) of the other API requests. The attachment content will not be nested inside of a JSON response envelope structure. </li>
        <li> The response to this request may be an HTTP redirect (status 303 with a <code>Location:</code> response header) indicating that the actual data can be obtained from a different server. The client should not store the redirect URLs. These are temporary and will become invalid a while after they are issued. (This is done to ensure the security of attachment data. The API token of a Tracker user who has been removed from a project will no longer be authorized to obtain attachment data from this endpoint, directly or through redirect. Ensuring that the redirect URLs become invalid over time prevents authorization from being circumvented by people storing them.) </li> 
       </ul> 
       <a class="jump_point" id="Accessing_a_Google_Attachment"></a> 
       <h5>Accessing a Google Attachment</h5> 
       <p> When a client obtains a <code>google_attachment</code> resource from Tracker, using the information in it to access the Google Apps document itself can be accomplished using one or more of the Google Apps APIs. Note that separate authentication and authorization is required to access Google Apps directlywhile a Tracker user profile can be linked to a Google Apps domain user, Tracker and Google Apps have separate API client authentication, and a Tracker API token will not provide access to Google Apps, even though Tracker ensures that the sharing on the Google Apps document will allow the client's user access to it. </p>
       <p> A JavaScript client executing in a browser can delegate the access to Google Apps to the browser itself by navigating to (or opening a new tab/window) the URL provided in the <code>alternate_link</code> attribute of the <code>google_attachment</code> resource. </p> 
       <a class="jump_point" id="Tracker_Updates_in_SCM_Post_Commit_Hooks"></a> 
       <h4>Tracker Updates in SCM Post-Commit Hooks</h4> 
       <a class="jump_point" id="Using_the_source_commits_Endpoint"></a> 
       <h5>Using the /source_commits Endpoint</h5> 
       <p> The Tracker API supports integration with post-commit hooks of Source Control Management (SCM) systems such as Git, Subversion, etc. When a commit is made to the SCM, a trigger can make a request to the Tracker API to add a story comment with the commit ID, author and message. It can also optionally change story state. </p>
       <p> The request is POSTed to the <a href="/help/api/rest/v5#source_commits">/source_commits</a> endpoint. For this request, the POST body should contain a single parameter, <code>source_commit</code>, which must be a hash whose structure matches the <a href="/help/api/rest/v5#source_commit_resource">source_commit</a> resource. In this structure, the <code>message</code> string is required, and optionally <code>commit_id</code>, <code>url</code>, and/or <code>author</code> may be supplied. The content of the <code>message</code> string is parsed by Tracker to determine what story or stories are affected by the commit, and what, if anything, the API request should do to those stories. </p>
       <p> Like other API requests, POSTs to <code>/source_commits</code> must be authenticated and authorized. Authentication operates like other API endpoints, which implies that the actions taken by the <code>/source_commits</code> request are assumed to be on behalf of a specific, known Tracker user. While not mandatory, it is common practice to create a Tracker login (Profile) and API Token that will be used specifically for <code>/source_commits</code> requests (or for all your automated/integration API requests). Only those actions permissible to the user authenticated for the request can be performed by the API for <code>/source_commits</code>. Therefore, you will likely want to add your source-control user to each Tracker project that might be affected by a post-commit hook request. </p>
       <p> To associate an SCM commit with specific Tracker stories, Tracker looks for a special syntax in the commit <code>message</code> string to indicate one or more story IDs and (optionally) a state change for the story. Your commit message should contain one pair of square brackets, containing one or more sets of a hash character followed immediately by a story ID. </p>

       <p> If an included story was not already started (it was in the "not started" state), an update to that story from <code>/source_commits</code> that doesn't contain any other state-change information will automatically start the story. </p>
       <p> To automatically finish a story by using a commit message, include "fixed", "completed", or "finished" in the square brackets in addition to the story ID.        <p> In some environments, code that is committed is automatically deployed. For this situation, use the keyword "delivers" and feature stories will be put in the <code>delivered</code> state. </p> 
       <a class="jump_point" id="Subversion_Example"></a> 
       <h5>Subversion Example</h5> 
       <p> The following is an example shell script that could be a Subversion post-commit hook script (don't forget to make it executable with <code>chmod a+x [script-name]</code>): </p> 
       <pre class="example_box">
  #!/bin/sh
  set -e

  REPO="$1"
  REV="$2"
  TOKEN="$3"

  AUTHOR=`/usr/bin/svnlook author $REPO -r $REV`
  MESSAGE=`/usr/bin/svnlook log $REPO -r $REV`
  URL="http://example.com/the_url_to_show_this_revision_in_a_web_browser"

  # You must HTML-escape the JSON post data
  MESSAGE=${MESSAGE//&amp;/&amp;amp;}
  MESSAGE=${MESSAGE//&lt;/&amp;lt;}
  MESSAGE=${MESSAGE//&gt;/&amp;gt;}
  # This is just a simple example.  You also need to handle the AUTHOR, as well as quotes, backticks, etc...
  # If you have a better one, please let us know, and we'll add it to the list of Tracker 3rd Party Tools
  # at http://pivotaltracker.com/integrations.

  RESPONSE=`curl -H "X-TrackerToken: $TOKEN" -X POST -H "Content-type: application/json" \
      -d "{\"source_commit\":{\"message\":\"$MESSAGE\", \"author\":\"$AUTHOR\", \"commit_id\":\"$REV\", \"url\":\"$URL\"}}" \
  https://www.pivotaltracker.com/services/v5/source_commits`

  echo $RESPONSE &gt;&gt; /tmp/tracker_post_commit.log
</pre> 
       <p>For more details, see the Subversion Book section on <a href="http://svnbook.red-bean.com/en/1.5/svn-book.html#svn.reposadmin.create.hooks">Implementing Repository Hooks</a>. </p>

       <a class="jump_point" id="Manipulating_Labels_on_Stories_as_Hashes"></a> 
       <h4>Manipulating Labels on Stories as Hashes</h4> 
       <p> As a type of resource associated with a story, the set of labels on a particular story is represented by default as a list of <code>label_ids</code>. For responses returned from the Pivotal Tracker API, whether information on associated labels is included in a story at all, and how it is represented is determined by the defaults (see <a href="/help/api#Default_Response_Content">Default Response Content</a> and <a href="/help/api/rest/v5#story_resource">story</a>) and by the <code>fields</code> parameter that can be provided to most API requests (see <a href="/help/api#Response_Controlling_Parameters">Response-Controlling Parameters</a>). </p>
       <p> When an API request's parameters include the information that defines a story, it will generally allow the set of associated labels to be specified. The common behavior for the API is to accept either an array of <code>label_ids</code> or an array of <code>labels</code>nested resource structures (hashes), each of which represents a label with which the story is associated. The general rules for using nested resources in parameters in an association are given <a href="#Nested Resources as Parameters">above</a>. This section presents concrete examples of creating and updating stories using nested resources instead of IDs to specify the story's labels. </p>
       <p> A new story is created by POSTing to the <a href="/help/api/rest/v5#projects_project_id_stories">/projects/{project_id}/stories</a> endpoint.        <h4>Using the GET-Request Aggregator</h4> 
       <p> In addition to the RESTful API endpoints described in the version-specific reference sections, Tracker supports a request aggregation endpoint that allows a client to bundle a group of GET requests into a single server operation. Note that this is intended primarily to improve server response time and minimize the number of network connections the client has to maintain. Logically, individual requests contained in the group sent to the aggregator are processed as if they were a set of independent requests received essentially simultaneously. </p>
       <p> The aggregator is used by <em>POST</em>ing a JSON data structure containing an array of the URLs that the client wants to GET from. The aggregator will then perform independent GET requests to Tracker for each of those URLs, and compile the result into a JSON data structure containing a hash where each key is the URL fetched from and each value is the JSON response received from fetching that URL. This endpoint does not aggregate responses in the CSV format. </p>
