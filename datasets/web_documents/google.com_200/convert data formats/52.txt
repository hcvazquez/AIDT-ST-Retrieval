          <h1>JavaScript Data Types and Variables: Chapter 2 - Learning Javascript</h1> by 
          <a href="/pub/au/90">Shelley Powers</a>
          <br> 
          <p>Variables<a class="indexterm" name="ch02-datatypes001"></a> <a class="indexterm" name="ch02-variables001"></a> in JavaScript are basically named buckets of data, a way of creating a reference to that data—regardless of whether the data is a string, number, boolean, array, or other object—so that you can access the same data again and again. More importantly, you can use variables to persist data from one process to another. For instance, your JavaScript application can store the value of a form element in a variable, and manipulate that value without having to actually manipulate the form element itself.</p> 
          <div class="bookpromo"> 
           <a href="http://oreilly.com/catalog/9780596521875/"><img src="http://covers.oreilly.com/images/9780596521875/thumb.gif" alt="JavaScript: The Good Parts book cover"></a> 
           <p>This excerpt is from <strong><a href="http://oreilly.com/catalog/9780596521875/">Learning JavaScript, Second Edition</a></strong>. You'll learn everything from primitive data types to complex features, including JavaScript elements involved with Ajax and dynamic page effects. By the end of the book, you'll be able to work with even the most sophisticated libraries and web applications.</p> 
           <a href="http://oreilly.com/catalog/9780596521875/"><img src="http://oreillynet.com/images/buyitnow-button.png" alt="buy button"></a> 
          </div> 
          <div class="clear"></div> 
          <div class="chapter" lang="en"> 
           <p>The variable’s <span class="emphasis"><em>data type</em></span> is the JavaScript scripting engine’s interpretation of the type of data that variable is currently holding. A string variable holds a string; a number variable holds a number value, and so on. However, unlike many other languages, in JavaScript, the same variable can hold different types of data, all within the same <span class="keep-together">application.</span> This is a concept known by the terms<a class="indexterm" name="I_indexterm2_d1e2137"></a><a class="indexterm" name="I_indexterm2_d1e2140"></a> <span class="emphasis"><em>loose typing</em></span> and <span class="emphasis"><em>dynamic typing</em></span>, both of which mean that a JavaScript variable can hold different data types at different times depending on context.</p> 
           <!--ONJava MPU Ad --> 
           <!-- me --> 
           <p>With a loosely typed language, you don’t have to declare ahead of time that a variable will be a string or a number or a boolean, as the data type is actually determined while the application is being processed. If you start out with a string variable and then want to use it as a number, that’s perfectly fine, as long as the string actually contains something that resembles a number and not something such as an email address. If you later want to treat it as a string again, that’s fine, too.</p>
           <p>The forgiving nature of loose typing can end up generating problems. If you try to add two numbers together, but the JavaScript engine interprets the variable holding one of them as a string data type, you end up with an odd string, rather than the sum you were expecting. Context is everything when it comes to variables and data types with <span class="keep-together">JavaScript.</span></p>
           <p>This chapter covers the JavaScript <a class="indexterm" name="I_indexterm2_d1e2158"></a><a class="indexterm" name="I_indexterm2_d1e2161"></a><a class="indexterm" name="I_indexterm2_d1e2164"></a><a class="indexterm" name="I_indexterm2_d1e2167"></a>primitive data types of <code class="literal">string</code>, <code class="literal">boolean</code>, and <code class="literal">number</code>, as well as the built-in functions for modifying values of these types. In addition, we’ll look at two special data types in JavaScript, <code class="literal">null</code> and <code class="literal">undefined</code>, toward the end of the chapter. Along the way, we’ll explore escape sequences in strings and take a brief look at Unicode. The chapter also delves into the topic of variables, including what makes valid and meaningful variable identifiers.</p>
           <div class="sect1" lang="en">
            <div class="titlepage">
             <div>
              <div>
               <h2 class="title" style="clear: both"><a name="identifying_variables"></a>Identifying Variables</h2>
              </div>
             </div>
            </div>
            <p>JavaScript<a class="indexterm" name="ch02-varidentifvying"></a> variables have an identifier, scope, and a specific data type. Because the language is loosely typed, the rest, as they say, is subject to change without notice.</p>
            <p>Variables in JavaScript are much like those in any other language; you use them to hold values in such a way that the values can be explicitly accessed in different places in the code. Each has an identifier that is unique to the scope of use (more on this later), consisting of any combination of letters, digits, underscores, and dollar signs. An identifier doesn’t have a required format, other than it must begin with a character, dollar sign, or underscore:</p>
            <a name="I_programlisting2_d1e2200"></a>
            <pre class="programlisting">_variableidentifier
__variableidentifier
variableIdentifier
$variable_identifier
var-ident</pre>
            <p>Starting with JavaScript 1.5, you can also use<a class="indexterm" name="I_indexterm2_d1e2204"></a> Unicode letters (such as <code class="literal">?</code>) and digits, as well as escape sequences (such as <code class="literal">\u0009</code>) in variable identifiers. The following are also valid variable identifiers for JavaScript:</p>
            <a name="I_programlisting2_d1e2216"></a>
            <pre class="programlisting">_?valid
T\u0009</pre>
            <p>Use special characters with caution, though, as some tools such as debuggers may have difficulty with them.</p>
            <p>JavaScript is case-sensitive, <a class="indexterm" name="I_indexterm2_d1e2222"></a>which means it treats upper- and lowercase characters differently. For instance, JavaScript sees the following two variable identifiers as separate variables:</p>
            <a name="I_programlisting2_d1e2226"></a>
            <pre class="programlisting">stringVariable
stringvariable</pre>

Other keywords<a class="indexterm" name="I_indexterm2_d1e2232"></a> may be added over time, as new versions of JavaScript (well, technically, ECMAScript) are released.</p>

              <div>
               <div>
                <h3 class="title"><a name="naming_guidelines"></a>Naming Guidelines</h3>
               </div>
              </div>
             </div>
             <p>Apart from the variable naming restrictions covered in the preceding section, you can use any identifier for variables and functions within code, but several naming <span class="keep-together">practices—</span>many inherited from Java and other programming languages—can make the code easier to follow and maintain.</p>
             <p>First, use meaningful words rather than something you’ve thrown together quickly. For instance, the variable identifier <code class="literal">interestRate</code> is more descriptive than the variable <code class="literal">intRt</code> or even <code class="literal">ir</code>. The latter two names are too cryptic and too difficult to understand, even within a given context.</p>
             <p>You can also provide a data type clue as part of the name, using something such as the following, which is a string, holding a first name:</p>
             <a name="I_programlisting2_d1e2701"></a>
             <pre class="programlisting">var strFirstName = "Shelley";</pre>
             <p>This type of naming convention—using the data type as part of the variable name—is known<a class="indexterm" name="I_indexterm2_d1e2705"></a> as <span class="emphasis"><em>Hungarian notation</em></span>, and is especially popular in Windows development. As such, you’ll most likely see it used within the older JScript applications created for Internet Explorer, but less often in more modern JavaScript development.</p>
             <p>Another naming convention is to use a plural for collections of items:</p>
             <a name="I_programlisting2_d1e2714"></a>
             <pre class="programlisting">var customerNames = new Array();</pre>
             <p>Typically, variables are not capitalized because capitalization is usually reserved for objects such as <code class="literal">String</code>:</p>
             <a name="I_programlisting2_d1e2721"></a>
             <pre class="programlisting">var firstName = String("Shelley");</pre>
             <p>Reserving capitalization for objects makes them easier to differentiate from simple <span class="keep-together">variables.</span></p>
             <p>Functions and variables frequently start with lowercase letters, and incorporate a verb representing what the function is doing. It’s pretty easy to guess what the following function is doing:</p>
             <a name="I_programlisting2_d1e2730"></a>
             <pre class="programlisting">function validateNameInRegister(firstName,lastName) ...</pre>
             <p>Many times, variables and functions have one or more words concatenated into a unique identifier, following a format popularized in other languages and frequently referred to<a class="indexterm" name="I_indexterm2_d1e2734"></a> as <span class="emphasis"><em>CamelCase</em></span> because of the up-down nature of the name, like a camel’s humps:</p>
             <a name="I_programlisting2_d1e2741"></a>
             <pre class="programlisting">validateName
firstName</pre>
             <p>The CamelCase naming format makes the variable much more readable, though dashes or underscores between the variable “words” work as well:</p>
             <a name="I_programlisting2_d1e2745"></a>
             <pre class="programlisting">validate-name
first_name</pre>
             <p>The newer JavaScript libraries invariably use CamelCase notation, which I also prefer for my own applications.</p>
             <p>Though you can use a dollar sign, number, or underscore to begin a variable, your best bet is to start with a letter. Unnecessary use of unexpected characters in variable names can make the code harder to read and follow, especially for newer JavaScript developers. However, if you’ve looked at some of the newer JavaScript libraries and examples, you might have noticed some odd-looking variable names. The popular Ajax-based <span class="keep-together">Prototype</span> JavaScript library is a strong influence in this regard—so much so that I think of the rise of new naming conventions as the “Prototype effect.”</p>
             <p>The following is an example of this effect:</p>
             <a name="I_programlisting2_d1e2756"></a>
             <pre class="programlisting">var _break = someval;</pre>
             <p>The underscore<a class="indexterm" name="I_indexterm2_d1e2760"></a><a class="indexterm" name="I_indexterm2_d1e2763"></a> is used in these libraries to signal a variable that’s an object’s private data member, a concept I’ll cover in Chapter?13, <i>Creating Custom JavaScript Objects</i>. Another interesting naming variation that<a class="indexterm" name="I_indexterm2_d1e2769"></a> Prototype has introduced is the following, which uses<a class="indexterm" name="I_indexterm2_d1e2775"></a><a class="indexterm" name="I_indexterm2_d1e2780"></a> the dollar sign (<code class="literal">$</code>) to name a function that returns a reference to a page element:</p>
             <a name="I_programlisting2_d1e2791"></a>
             <pre class="programlisting">$('test').invokeSomeMethod();</pre>
             <p>The use of the underscore or dollar sign doesn’t change the behavior of the variable, even though such usage is relatively new. It’s just another way of naming something.</p>
             <div class="note" style="margin-left: 0.5in; margin-right: 0.5in;">
              <h3 class="title"><a name="note-11"></a>Note</h3>
              <p>You can find the Prototype JavaScript library at <a href="http://www.prototypejs.org/" target="_top">http://www.prototypejs.org/</a>.</p>
             </div>
             <p>Aside from the few JavaScript naming restrictions, nothing is mandatory or magical about the naming conventions I’ve outlined. They help to make JavaScript easier to read and debug.</p>
            </div>
           </div>
           <div class="sect1" lang="en">
            <div class="titlepage">
             <div>
              <div>
               <h2 class="title" style="clear: both"><a name="primitive_types"></a>Primitive Types</h2>
              </div>
             </div>
            </div>
            <p>JavaScript<a class="indexterm" name="I_indexterm2_d1e2807"></a><a class="indexterm" name="I_indexterm2_d1e2808"></a> is a trim language, with just enough functionality to do the job—no more, no less. However, as I’ve said before, it is a confusing language in some respects.</p>
            <p>For instance, JavaScript has just three primitive data<a class="indexterm" name="I_indexterm2_d1e2814"></a><a class="indexterm" name="I_indexterm2_d1e2817"></a><a class="indexterm" name="I_indexterm2_d1e2820"></a> types: <code class="literal">string</code>, <code class="literal">numeric</code>, and <code class="literal">boolean</code>. Each is differentiated from the others by the type of value it contains: string, numeric, and boolean, respectively. However, JavaScript also has built-in objects known as <code class="literal">String</code>, <code class="literal">Number</code>, and <code class="literal">Boolean</code>. These<a class="indexterm" name="I_indexterm2_d1e2843"></a><a class="indexterm" name="I_indexterm2_d1e2846"></a><a class="indexterm" name="I_indexterm2_d1e2849"></a> would seem to be very different from each other: the first three are types of primitive values, whereas the latter three are objects, each one with its own built-in properties and methods.</p>
            <p>In actuality, the two are connected. The <code class="literal">String</code> object <span class="emphasis"><em>wraps</em></span> the string primitive—just as the <code class="literal">Number</code> and <code class="literal">Boolean</code> objects wrap their individual primitive types—when using the primitive type like an object. When you create a simple string variable in JavaScript, and then use one of the <code class="literal">String</code> methods, JavaScript implicitly wraps the string primitive in a <code class="literal">String</code> object, processes the <code class="literal">String</code> object property or method call, and then discards the object. In the following code snippet, when the method <code class="literal">toUpperCase</code> is<a class="indexterm" name="I_indexterm2_d1e2880"></a> called on <code class="literal">firstName</code>, an object is created to wrap the string and then process the method call before the object is discarded:</p>
            <a name="I_programlisting2_d1e2887"></a>
            <pre class="programlisting">var firstName = "Shelley";
var cappedName = firstName.toUpperCase();</pre>
            <p>For all intents and purposes, <code class="literal">firstName</code> looks like an object, and it acts like an object when calling <code class="literal">toUpperCase</code>, but it is a primitive. If I call another <code class="literal">String</code> object method, the same thing will happen again: a <code class="literal">String</code> object is created and then wraps the primitive, processes the method call, and discards the object. As you can imagine, if you’re going to be treating a string like an object, you’d be better off creating it as an object. At the same time, if all you need is a simple string to print a message or hold a value, you don’t need all the functionality that accompanies an object, so creating a string primitive is the better option.</p>
            <p>Rather than continue to mix primitive data types and objects in a confusing mishmash that wanders from primitive to object and back again, in the next three sections, we’ll look at each of the primitive data types—how they’re created and manipulated, and how you can convert values of one type to other type. In Chapter?4, <i>The JavaScript Objects</i>, I’ll cover the data objects, their methods, and their properties.</p>
            <div class="note" style="margin-left: 0.5in; margin-right: 0.5in;">
             <h3 class="title"><a name="note-12"></a>Note</h3>
             <p>When I use the word <span class="emphasis"><em>wrap</em></span> in the book, I’m talking about an object that typically encloses a simpler item, such as a <code class="literal">String</code> object wrapping a string primitive.</p>
            </div>
           </div>
           <div class="sect1" lang="en">
            <div class="titlepage">
             <div>
              <div>
               <h2 class="title" style="clear: both"><a name="the_string_data_type"></a>The String Data Type</h2>
              </div>
             </div>
            </div>
            <p>Because JavaScript is<a class="indexterm" name="ch02-stringdatatype"></a> a loosely typed language, nothing differentiates a string variable from a variable that’s a number or a boolean, other than the literal value assigned to the string variable when it’s initialized and the context of its use.</p>
            <p>A string literal is a sequence of characters delimited by single<a class="indexterm" name="I_indexterm2_d1e2927"></a><a class="indexterm" name="I_indexterm2_d1e2930"></a><a class="indexterm" name="I_indexterm2_d1e2933"></a><a class="indexterm" name="I_indexterm2_d1e2936"></a> or double quotes:</p>
            <a name="I_programlisting2_d1e2940"></a>
            <pre class="programlisting">var strString = "This is a string";
var anotherString= 'But this is also a string';</pre>
            <p>No rule states which type of quote you use, except that the ending quote character must be the same as the beginning one. You can include any variation of characters in the string:</p>
            <a name="I_programlisting2_d1e2944"></a>
            <pre class="programlisting">var thirdString = "This is 1 string.";
var stringFour = "This is--another string.";
var stringAsNumber = "543";</pre>
            <p>The last example of a string contains a number, but because it’s surrounded by quotes, JavaScript creates the variable as a string.</p>
            <p>A string can also include quotes. You can use single and double quotes interchangeably if you need to include a quote within a quoted string. All you have to do is be <span class="keep-together">consistent—</span>if the string contains a single quote, use double quotes around the string; the same is true with the double quote. For example:</p>
            <a name="I_programlisting2_d1e2953"></a>
            <pre class="programlisting">var string_value = "This is a 'string' with a quote."</pre>
            <p>or:</p>
            <a name="I_programlisting2_d1e2957"></a>
            <pre class="programlisting">var string_value = 'This is a "string" with a quote.'</pre>
            <p>The empty string is a special case: you’d commonly use it to initialize a string variable when it’s defined. The following are examples of empty strings:</p>
            <a name="I_programlisting2_d1e2962"></a>
            <pre class="programlisting">var string_value = '';
var anotherStringValue = "";</pre>
            <p>Which quote character you use makes no difference to the JavaScript engine. What’s more important is to use one or the other consistently in your applications.</p>
            <div class="sect2" lang="en">
             <div class="titlepage">
              <div>
               <div>
                <h3 class="title"><a name="string_escape_sequences"></a>String Escape Sequences</h3>
               </div>
              </div>
             </div>
             <p>Not all characters<a class="indexterm" name="I_indexterm2_d1e2971"></a><a class="indexterm" name="I_indexterm2_d1e2976"></a> are treated equally within a string in JavaScript. A string can also contain an <span class="emphasis"><em>escape sequence</em></span>, such as <code class="literal">\n</code> for<a class="indexterm" name="I_indexterm2_d1e2986"></a><a class="indexterm" name="I_indexterm2_d1e2989"></a> the end-of-line terminator. An escape <span class="keep-together">sequence</span> is a pattern in which certain characters are encoded in certain ways in order to include them within a string.</p>
             <p>The following snippet of code assigns a string literal containing a line-terminator escape sequence to a variable. When the string is used in a dialog window, the escape sequence, <code class="literal">\n</code>, is interpreted literally, and a newline is published:</p>
             <a name="I_programlisting2_d1e3002"></a>
             <pre class="programlisting">var string_value = "This is the first line\nThis is the second line";</pre>
             <p>This results in:</p>
             <a name="I_programlisting2_d1e3006"></a>
             <pre class="programlisting">This is the first line
This is the second line</pre>
             <p>You can also use <a class="indexterm" name="I_indexterm2_d1e3010"></a><a class="indexterm" name="I_indexterm2_d1e3013"></a>the backslash to denote that the quote in the string is meant to be taken as a literal character, not as an end-of-string terminator:</p>
             <a name="I_programlisting2_d1e3017"></a>
             <pre class="programlisting">var string_value = "This is a \"string\" with a quote."</pre>
             <p>By using the backslash with quotes, you can include single and double quotes within a string.</p>
             <p>To include a backslash in a string, use two backslashes in a row:</p>
             <a name="I_programlisting2_d1e3023"></a>
             <pre class="programlisting">var string_value = "This is a \\string\\ with a backslash."</pre>
             <p>The result of this line of code is a string with two backslashes, one on each side of the word “string”.</p>
             <p>You can also include <a class="indexterm" name="I_indexterm2_d1e3030"></a>Unicode<a class="indexterm" name="I_indexterm2_d1e3036"></a> characters in a string by preceding the four-digit hexadecimal value of the character with <code class="literal">\u</code>. For instance, the following outputs the Chinese (simplified) ideogram for “love”:</p>
             <a name="I_programlisting2_d1e3043"></a>
             <pre class="programlisting">document.writeln("\u7231");</pre>
             <p>What displays is somewhat browser-dependent; however, most of the more commonly used browsers now have adequate Unicode support.</p>
             <div class="note" style="margin-left: 0.5in; margin-right: 0.5in;">
              <h3 class="title"><a name="note-13"></a>Note</h3>
              <p>You can learn more about Unicode, and access relevant charts, at <a href="http://www.unicode.org/" target="_top">http://www.unicode.org/</a>.</p>
             </div>
            </div>
            <div class="sect2" lang="en">
             <div class="titlepage">
              <div>
               <div>
                <h3 class="title"><a name="string_encoding"></a>String Encoding</h3>
               </div>
              </div>
             </div>
             <p>Using the<a class="indexterm" name="I_indexterm2_d1e3057"></a> backslash to escape characters is helpful when you’re using<a class="indexterm" name="I_indexterm2_d1e3063"></a> ASCII characters that are normally control characters within a string. However, the backslash can’t do anything with characters that are not ASCII; nor can it do anything if you want to make an entire string safe for HTML processing, which is necessary for Ajax-based applications (I’ll touch on this at the end of the book).</p>
             <p>You use the <code class="literal">encodeURI</code> and <code class="literal">encodeURIComponent</code> methods<a class="indexterm" name="I_indexterm2_d1e3075"></a><a class="indexterm" name="I_indexterm2_d1e3078"></a> to escape, or more properly, to <span class="emphasis"><em>encode</em></span> entire strings, converting ASCII and non-ASCII characters to URI<a class="indexterm" name="I_indexterm2_d1e3085"></a> <span class="keep-together">encoding</span>, which you can use in links and Ajax applications.</p>
             <div class="note" style="margin-left: 0.5in; margin-right: 0.5in;">
              <h3 class="title"><a name="note-14"></a>Note</h3>
              <p>URI stands for Uniform Resource Identifier, an example of which is a web page URL. An example of URI encoding is ISO Latin-1 (also known as ISO 8859-1).</p>
             </div>
             <p>The <code class="literal">encodeURI</code> makes an assumption that the string is a URI such as “<a href="http://oreilly.com" target="_top">http://oreilly.com</a>”, and reserves the following characters:</p>
             <a name="I_programlisting2_d1e3103"></a>
             <pre class="programlisting">; , / ? : @ &amp; = + $</pre>
             <p>Alphanumeric characters and the following punctuation are also not encoded:</p>
             <a name="I_programlisting2_d1e3107"></a>
             <pre class="programlisting">- _ . ! ~ * ' ( )</pre>
             <p>The page fragment <a class="indexterm" name="I_indexterm2_d1e3111"></a><a class="indexterm" name="I_indexterm2_d1e3114"></a><a class="indexterm" name="I_indexterm2_d1e3117"></a>symbol (<code class="literal">#</code>) is also not encoded.</p>
             <p>The <code class="literal">encodeURIComponent</code>, however, encodes all characters except the alphanumeric and punctuation characters listed earlier. It assumes that the string being encoded is being used as a parameter to a URI, and therefore characters that are normally part of a URI, such as the following, are encoded:</p>
             <a name="I_programlisting2_d1e3129"></a>
             <pre class="programlisting"># &amp; + =</pre>
             <p>Both functions also have their opposite <a class="indexterm" name="I_indexterm2_d1e3134"></a>member: <code class="literal">decodeURI</code>, to decode the <code class="literal">encodeURI</code> encoded string, and <code class="literal">decodeURIComponent</code>, to decode the <code class="literal">encodeURIComponent</code> string.</p>
             <p><a href="#uri_encoding_two_strings_using_the_javas" title="Example?2.1.?URI encoding two strings using the JavaScript encodeURI and encodeURIComponent methods">Example?2.1, “URI encoding two strings using the JavaScript encodeURI and encodeURIComponent methods”</a> shows a web page that uses all four functions to encode and then decode two strings, all of which are then printed out to the current web page using <code class="literal">document.writeln</code>.</p>
             <div class="example">
              <a name="uri_encoding_two_strings_using_the_javas"></a>
              <p class="title"><b>Example?2.1.?URI encoding two strings using the JavaScript encodeURI and encodeURIComponent methods</b></p>
              <div class="example-contents">
               <pre class="programlisting">&lt;!DOCTYPE html PUBLIC "-//W3C//DTD XHTML 1.1//EN"
"http://www.w3.org/TR/xhtml11/DTD/xhtml11.dtd"&gt;
&lt;html xmlns="http://www.w3.org/1999/xhtml" xml:lang="en"&gt;

    var sOne =

    document.write(sOutputDecoded);

}
//]]&gt;

   &lt;p&gt;&lt;/p&gt;

&lt;/html&gt;</pre>
              </div>
             </div>

             <p>These are all demonstrations of how to explicitly create a string variable, and variations of string literals that incorporate special characters. You can also convert the values within a specific variable from other data types, depending on the context.</p>

              <div>
               <div>
                <h3 class="title"><a name="converting_to_strings"></a>Converting to Strings</h3>
               </div>
              </div>
             </div>
             <p>You can <a class="indexterm" name="ch02-stringconverting"></a>convert other data types, such as numbers and booleans, to a string; typically, the scripting engine will do the conversion automatically, based on the context. As an example, if a numeric or boolean variable is passed to a function that expects a string, the value is implicitly converted to a string first, before the value is processed:</p>
             <a name="I_programlisting2_d1e3185"></a>
             <pre class="programlisting">var num_value = 35.00;
alert(num_value); // expects a string</pre>
             <p>In addition, when the plus sign (<code class="literal">+</code>) is used with two variables in an assignment statement, and one value is a string and the other a <a class="indexterm" name="I_indexterm2_d1e3192"></a><a class="indexterm" name="I_indexterm2_d1e3197"></a>number, the number is converted to a string and then the two strings are concatenated:</p>
             <a name="I_programlisting2_d1e3203"></a>
             <pre class="programlisting">var num_value = 35.00;
var string_value = "This is a number:" + num_value;</pre>
             <p>When the conversion from number to string occurs depends on when the JavaScript scripting engine encounters the string. For instance, if the string is the first in a sequence of values, all of the numbers that follow are treated as strings:</p>
             <a name="I_programlisting2_d1e3207"></a>
             <pre class="programlisting">var strValue = "4" + 3 + 1; // becomes "431"
var strValueTwo = 4 + 3 + "1"; // becomes 71</pre>
             <p>However, if you use operators other than <code class="literal">+</code>, the opposite type of conversion is <span class="keep-together">applied—</span>the string is converted to a number:</p>
             <a name="I_programlisting2_d1e3217"></a>
             <pre class="programlisting">var firstResult = "35" - 3; // subtraction is applied, resulting in 32
var secondResult = 30 / "3"; // division is applied, resulting in 10
var thirdResult = "3" * 3; // multiplication is applied, resulting in 9</pre>
             <p>This implicit conversion with its dependency on both operator and position demonstrates more fully the danger of loose typing: the values you end up with can vary widely, depending on something as simple as where in the sequence of operations you introduce a new data type, or what type of operator you use.</p>
             <div class="note" style="margin-left: 0.5in; margin-right: 0.5in;">
              <h3 class="title"><a name="note-15"></a>Note</h3>
              <p>I cover the addition and other operators demonstrated in this chapter, as well as others available in JavaScript, more fully in Chapter?3, <i>Operators and Statements</i>.</p>
             </div>
             <p>Rather than depend fully on happenstance data type conversion, you can explicitly convert a variable to a string using the <code class="literal">String</code> global function.<a class="indexterm" name="I_indexterm2_d1e3232"></a> If the value being <span class="keep-together">converted</span> is a boolean, the resultant string is a text <span class="keep-together">representation</span> of the boolean value: <code class="literal">"true"</code> for true; <code class="literal">"false"</code> for false. For numbers, the string is, again, a string representation of the number, such as <code class="literal">"–123.06"</code> for –123.06, depending on the number of digits and the precision (the placement of the decimal point).                <h2 class="title" style="clear: both"><a name="the_number_data_type"></a>The Number Data Type</h2>

            <p>Number<a class="indexterm" name="ch02-numberdatatype"></a> data types in JavaScript are<a class="indexterm" name="I_indexterm2_d1e3551"></a> floating-point numbers, but they may or may not have a fractional component. If they don’t have a decimal point or fractional component, they’re treated as integers—base-10 whole numbers in a range of –2<sup>53</sup> to 2<sup>53</sup>.</p>
            <p>The following are valid integers:</p>
            <a name="I_programlisting2_d1e3563"></a>
            <pre class="programlisting">var negativeNumber = -1000;
var zero = 0;
var fourDigits = 2534;</pre>
            <p>The floating-point representation has a decimal, with a decimal component to the right. You also can represent the number as an exponent, using scientific notation. All of the following are valid floating-point numbers:</p>
            <a name="I_programlisting2_d1e3567"></a>
            <pre class="programlisting">var someFloat = 0.3555
var anotherNumber = 144.006;
var negDecimal = -2.3;
var lastNum = 19.5e-2 //which is equivalent to .195
var zeroDecimal = 12.0;</pre>
            <p>Though larger numbers are supported, some functions can work only with numbers in a range of –2e31 to 2e31 (–2,147,483,648 to 2,147,483,648); as such, you should limit your number use to this range.</p>
            <p>Two special numbers exist: positive and negative<a class="indexterm" name="I_indexterm2_d1e3573"></a><a class="indexterm" name="I_indexterm2_d1e3576"></a> infinity. In JavaScript, they are represented by <code class="literal">Infinity</code> and <code class="literal">-Infinity</code>, respectively. A positive infinity is returned <span class="keep-together">whenever</span> a math overflow occurs in a JavaScript application. A negative infinity is returned when a number occurs that is smaller than the minimum value supported in JavaScript.</p>
            <p>In addition to base-10 representation, you can use<a class="indexterm" name="I_indexterm2_d1e3591"></a><a class="indexterm" name="I_indexterm2_d1e3594"></a> octal and hexadecimal notation with JavaScript numbers, though octal is newer and may be confused for hexadecimal with older browsers. A hexadecimal number begins with a zero, followed by an <code class="literal">x</code>:</p>
            <a name="I_programlisting2_d1e3601"></a>
            <pre class="programlisting">var firstHex = -0xCCFF;</pre>
            <p>An octal value begins with a zero, but there is no leading <code class="literal">x</code>:</p>
            <a name="I_programlisting2_d1e3609"></a>
            <pre class="programlisting">var firstOct = 0526;</pre>
            <p>What’s interesting with both the octal and decimal representations is that if you convert either to a string using the <code class="literal">String</code> function (covered earlier),<a class="indexterm" name="I_indexterm2_d1e3616"></a> the scripting engine first converts the number to its base-10 (decimal) representation and then converts the result to a string. So, a string conversion of <code class="literal">firstOct</code> is <code class="literal">"342"</code>, which is the string conversion of the decimal conversion of the octal number 0526.</p>
            <p>In the preceding two sections, I demonstrated how to convert other types to a string or a boolean. In this section, I’m going to demonstrate two different functions you can use to convert a string to a number: <code class="literal">parseInt</code> and <code class="literal">parseFloat</code>. The <code class="literal">parseInt</code> function<a class="indexterm" name="I_indexterm2_d1e3637"></a><a class="indexterm" name="I_indexterm2_d1e3640"></a> returns the integer portion of a number in a string, regardless of whether the string is formatted as an integer or as a floating-point number. The <code class="literal">parseFloat</code> function returns the floating-point value until a character that is not a sign (<code class="literal">+</code> or <code class="literal">-</code>), decimal, number, or exponent is reached.</p>
            <p><a href="#converting_strings_to_numbers_using_eith" title="Example?2.3.?Converting strings to numbers using either parseInt or parseFloat">Example?2.3, “Converting strings to numbers using either parseInt or parseFloat”</a> passes three strings containing numeric values to either <code class="literal">parseInt</code> or <code class="literal">parseFloat</code>, and writes the values to the page.</p>

               <h2 class="title" style="clear: both"><a name="the_null_and_undefined_variables"></a>The null and undefined Variables</h2>
              </div>
             </div>
            </div>
            <p>Nowhere<a class="indexterm" name="I_indexterm2_d1e3844"></a><a class="indexterm" name="I_indexterm2_d1e3849"></a><a class="indexterm" name="I_indexterm2_d1e3850"></a><a class="indexterm" name="I_indexterm2_d1e3853"></a> in JavaScript is the line between literals, simple data types, and objects blurred more than it is when you’re looking at two variables that represent nonexistence or incomplete existence: <code class="literal">null</code> and <code class="literal">undefined</code>.</p>
            <p>A <code class="literal">null</code> variable is one that you have defined and to which you have assigned <span class="emphasis"><em>null</em></span> as its value. The following is an example of a <code class="literal">null</code> variable:</p>
            <a name="I_programlisting2_d1e3874"></a>
            <pre class="programlisting">var nullString = null;</pre>
            <p>If you have declared but not initialized the variable, it is considered undefined:</p>
            <a name="I_programlisting2_d1e3878"></a>
            <pre class="programlisting">var undefString;</pre>
            <p>A variable is not null and is not undefined when you declare it and give it an initial value:</p>
            <a name="I_programlisting2_d1e3882"></a>
            <pre class="programlisting">var sValue = "";</pre>
            <p>When you’re using several JavaScript libraries and fairly complex code it’s not unusual for a variable not to be set; if you try to use the variable in an expression, you can get an adverse result—usually a JavaScript error. One approach to testing variables if you’re unsure of their state is to use the variable in a conditional test, such as the following:</p>
            <a name="I_programlisting2_d1e3886"></a>
            <pre class="programlisting">if (sValue) ... // if not null and initialized, true; otherwise false</pre>
            <p>We’ll look at <a class="indexterm" name="I_indexterm2_d1e3890"></a>conditional statements in Chapter?3, <i>Operators and Statements</i>, but for now, know that the expression consisting of just the variable <code class="literal">sValue</code> evaluates to <code class="literal">true</code> if <code class="literal">sValue</code> has been declared and initialized; otherwise, the result of the expression is <code class="literal">false</code>:</p>
            <a name="I_programlisting2_d1e3912"></a>
            <pre class="programlisting">if (unknownVariable)// false, variable is not declared or assigned a value

if (undefinedString) // false, as variable has not been given a value

if (nullString) // variable has been defined and given a value, but that value is
null and so the result is false

if (sValue) // true if a variable is both defined and given a value (including
empty string)</pre>
            <p>Using the <code class="literal">null</code> keyword, you can specifically test to see whether a value is null:</p>
            <a name="I_programlisting2_d1e3919"></a>
            <pre class="programlisting">if (sValue == null)</pre>
            <p>In JavaScript, a variable is undefined, even if it is declared, until it is initialized. A variable can be undeclared but initialized, in which case it is not null and is not undefined. However, in this instance, it’s considered a global variable, and as discussed earlier, not specifically declaring variables with <code class="literal">var</code> causes problems more often than not.</p>
            <div class="note" style="margin-left: 0.5in; margin-right: 0.5in;">
             <h3 class="title">Note</h3>
             <p>In some of the code snippets, comments that begin with <code class="literal">//</code> wrap to the next line because of the page width, not because the comments are <span class="keep-together">actually</span> multiple lines.</p>
            </div>
            <p>Though not related to existence, a third unique value is related to the type of a<a class="indexterm" name="I_indexterm2_d1e3937"></a> variable: <code class="literal">NaN</code>, or Not a Number. If a string or boolean variable cannot be coerced into a number, it’s considered <code class="literal">NaN</code> and is treated accordingly:</p>
            <a name="I_programlisting2_d1e3949"></a>
            <pre class="programlisting">var nValue = 1.0;
if (nValue == 'one' ) // false, the second operand is NaN</pre>
            <p>You can specifically test whether a variable is <code class="literal">NaN</code> with the <code class="literal">isNaN</code> function:</p>
            <a name="I_programlisting2_d1e3959"></a>
            <pre class="programlisting">if (isNaN(sValue)) // if string cannot be implicitly converted into number, return
true</pre>
            <p>By its very nature, a null value is <code class="literal">NaN</code>.</p>
            <div class="note" style="margin-left: 0.5in; margin-right: 0.5in;">
             <h3 class="title"><a name="note-17"></a>Note</h3>
             <p>Author and respected technologist<a class="indexterm" name="I_indexterm2_d1e3969"></a> Simon Willison gave an excellent talk at O’Reilly’s 2006 ETech conference, titled “A (Re)-Introduction to JavaScript.” You can view his slides at his website, <a href="http://simon.incutio.com/" target="_top">http://simon.incutio.com/</a>. The whole presentation is a very worthwhile read, but my favorite is the following line:</p>
             <p>“In other words, zero, null, NaN, and the empty string are inherently <code class="literal">false</code>; everything else is inherently <code class="literal">true</code>.”</p>
            </div>
            <p>For the most part, JavaScript developers create code in such a way that we know a variable is going to be defined ahead of time and/or given a value. In most instances, we don’t explicitly test to see whether a variable is set, and if so, whether it’s assigned a value.</p>
            <p>However, when using large and complex JavaScript libraries, and applications that can incorporate web service responses, it becomes increasingly important to test variables that originate and/or are set outside our control. It is also increasingly important to be aware of how null and undefined variables behave when accessed in the application.</p>
           </div>
           <div class="sect1" lang="en">
            <div class="titlepage">
             <div>
              <div>
               <h2 class="title" style="clear: both"><a name="constants_named_but_not_variables"></a>Constants: Named but Not Variables</h2>
              </div>
             </div>
            </div>
            <p>Sometimes<a class="indexterm" name="I_indexterm2_d1e3993"></a> you’ll want to define a value once, and then have it treated as a read-only value from that time forward. You use the keyword <code class="literal">const</code> to create a JavaScript <span class="keep-together">constant:</span></p>
            <a name="I_programlisting2_d1e4002"></a>
            <pre class="programlisting">const CURRENT_MONTH = 3.5;</pre>
            <p>The constant can be of any value, and because it can’t be assigned or reassigned a value at a later time, it’s initialized to its constant value when defined.</p>
            <p>Just as with variables, a JavaScript constant has global and local scope. I use constants at a global level, primarily because they contain a value I want to be accessible (and unchanged) by a JavaScript block. Also, notice how the entire constant name is in uppercase. This isn’t required, but it is a fairly standard naming convention, making <span class="keep-together">constants</span> easier to spot in code.</p>
           </div>
           <div class="sect1" lang="en">
            <div class="titlepage">
             <div>
              <div>
               <h2 class="title" style="clear: both"><a name="questions"></a>Test Your Knowledge: Quiz</h2>
              </div>
             </div>
            </div>
            <div class="orderedlist">
             <ol type="1">
              <li><p>Of the following identifiers, which are valid, which are not, and why?</p><a name="I_programlisting2_d1e4018"></a><pre class="programlisting">$someVariable
_someVariable
1Variable
some_variable
som?variable
function
some*variable</pre></li>
              <li><p>Convert the following identifiers to CamelCase notation:</p><a name="I_programlisting2_d1e4023"></a><pre class="programlisting">var some_month;
function theMonth // function to return current month
current-month // a constant
var summer_month; // an array of summer months
MyLibrary-afunction // a function from a JavaScript package</pre></li>
              <li><p>Is the following string literal valid? If not, how would you fix it?</p><a name="I_programlisting2_d1e4028"></a><pre class="programlisting">var someString = 'Who once said, "Only two things are infinite, the universe and
human stupidity, and I'm not sure about the former."'</pre></li>
              <li><p>Given a number, 432.54, what JavaScript function(s) returns the integer component of the number, and then finds the hexadecimal and octal conversions?</p></li>
              <li><p>You’re creating a JavaScript function in a library that other applications can use. A parameter, <code class="literal">someMonth</code>, is passed to the function. How would you determine whether it’s null or undefined?</p></li>
             </ol>
            </div>
           </div>
           <div class="sect1" lang="en">
            <div class="titlepage">
             <div>
              <div>
               <h2 class="title" style="clear: both"><a name="answers"></a>Test Your Knowledge: Answers</h2>
              </div>
             </div>
            </div>
            <div class="orderedlist">
             <ol type="1">
              <li><p>The following are valid:</p><a name="I_programlisting2_d1e4046"></a><pre class="programlisting">$someVariable
_someVariable
some_variable
som?variable</pre><p>The following are not valid:</p><a name="I_programlisting2_d1e4050"></a><pre class="programlisting">1Variable does not start with a valid character
function is a reserved word
some*variable uses an invalid character</pre></li>
              <li><p>The identifiers are converted as follows:</p><a name="I_programlisting2_d1e4055"></a><pre class="programlisting">var someMonth
function getCurrentMonth
CURRENT_MONTH
summerMonths
myLibraryFunction</pre></li>
              <li><p>The string is not valid. To make it valid, either use single quotes only within the double quote, or escape the double quotes:</p><a name="I_programlisting2_d1e4060"></a><pre class="programlisting">var someString = "Who once said, 'Only two things are infinite, the universe and
human stupidity, and I'm not sure about the former'";</pre><p>or:</p><a name="I_programlisting2_d1e4064"></a><pre class="programlisting">var someString = 'Who once said, "Only two things are infinite, the universe and
human stupidity, and I\'m not sure about the former"';</pre><p>Don’t forget to escape the apostrophes in contractions.</p></li>
              <li><p>The following code would work:</p><a name="I_programlisting2_d1e4071"></a><pre class="programlisting">var fltNumber = 432.54;
var intNumber = parseInt(fltNumber);
var octNumber = intNumber.toString(8);
var hexNumber = intNumber.toString(16);</pre></li>
              <li><p>This is a trick question. Passing a variable that’s not declared or defined to a function or object method results in a JavaScript error, so your function will not have to test the parameter.</p><p>Use the following to test to see whether a variable has been set elsewhere in code<a class="indexterm" name="I_indexterm2_d1e4078"></a><a class="indexterm" name="I_indexterm2_d1e4079"></a>:</p><a name="I_programlisting2_d1e4081"></a><pre class="programlisting">if (a) {
   ...
}</pre></li>
             </ol>
            </div>
           </div>
