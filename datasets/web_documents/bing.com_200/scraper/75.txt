       <p> Why reinvent the wheel? That cliche applies to software development where some developers frequently rewrite the same code for different programs. Two disadvantages with that approach are: </p>
       <ol>
        <li>It wastes time</li>
        <li>It introduces the potential for bugs in debugged code</li>
       </ol>
       <p> As an alternative to rewriting the same code, many software development environments provide a library tool that organizes frequently used code. Once developers finish debugging some reusable code, they use the tool to store that code in a <em>library</em>one or more files that contain frequently used code for use in various programs. During program-building, the compiler or library tool accesses the library to connect the program's library-referenced code to the program. </p>
       <p> Libraries are fundamental to Java. They allow, in part, the JVM's classloader to locate classfiles. (I will explore classloaders in a future article.) For that reason, Java's libraries are commonly known as <em>class libraries.</em> However, Java refers to class libraries as <em>packages.</em> </p>
       <p> This article explores packages; I show you how to create packages of classes and interfaces, how to import (that is, bring into a program) packaged classes and interfaces, how to move packages on the hard drive, and how to use jar files to encapsulate packages. </p>

       <h2>What are packages?</h2>
       <p> A <em>package</em> is a collection of classes and interfaces. Each package has its own name and organizes its top-level (that is, nonnested) classes and interfaces into a separate <em>namespace,</em> or name collection. Although same-named classes and interfaces cannot appear in the same package, they can appear in different packages because a separate namespace assigns to each package. </p>
       <p> From an implementation perspective, equating a package with a directory proves helpful, as does equating a package's classes and interfaces with a directory's classfiles. Keep in mind other approachessuch as the use of databasesto implementing packages, so do not get into the habit of always equating packages with directories. But because many JVMs use directories to implement packages, this article equates packages with directories. The Java 2 SDK organizes its vast collection of classes and interfaces into a tree-like hierarchy of packages within packages, which is equivalent to directories within directories. That hierarchy allows Sun Microsystems to easily distribute (and you to easily work with) those classes and interfaces. Examples of Java's packages include: </p>
       <ul>
        <li><code><strong>java.lang</strong>:</code> A collection of language-related classes, such as <code>Object</code> and <code>String</code>, organized in the <code>java</code> package's <code>lang</code> subpackage</li>
        <li><code><strong>java.lang.ref</strong>:</code> A collection of reference-related language classes, such as <code>SoftReference</code> and <code>ReferenceQueue</code>, organized in the <code>ref</code> sub-subpackage of the <code>java</code> package's <code>lang</code> subpackage</li>
        <li><code><strong>javax.swing</strong>:</code> A collection of Swing-related component classes, such as <code>JButton</code>, and interfaces, such as <code>ButtonModel</code>, organized in the <code>javax</code> package's <code>swing</code> subpackage</li>
       </ul>
       <p> Period characters separate package names. For example, in <code>javax.swing</code>, a period character separates package name <code>javax</code> from subpackage name <code>swing</code>. A period character is the platform-independent equivalent of forward slash characters (<code>/</code>), backslash characters (<code>\</code>), or other characters to separate directory names in a directory-based package implementation, database branches in a hierarchical database-based package implementation, and so on. </p>
       <div class="table-wrapper">
        <table border="1" cellpadding="3" cellspacing="3" table="table">

       <h2>Create a package of classes and interfaces</h2>
       <p> Every source file's classes and interfaces organize into a package. In the <code>package</code> directive's absence, those classes and interfaces belong to the unnamed package (the directory the JVM regards as the current directorythe directory where a Java program begins its execution via the Windows <code>java.exe</code>, or OS-equivalent, programand contains no subpackages). But if the <code>package</code> directive appears in a source file, that directive names the package for those classes and interfaces. Use the following syntax to specify a <code>package</code> directive in source code: </p>
       <aside class="nativo-promo tablet desktop" id=""> 
       </aside>
       <pre class="prettyprint">'package' <em>packageName</em> [ '.' <em>subpackageName</em> ... ] ';'
</pre>
       <p> A <code>package</code> directive begins with the <code>package</code> keyword. An identifier that names a package, <em><code>packageName</code>,</em> immediately follows. If classes and interfaces are to appear in a subpackage (at some level) within <em><code>packageName</code></em>, one or more period-separated <em><code>subpackageName</code></em> identifiers appear after <em><code>packageName</code></em>. The following code fragment presents a pair of <code>package</code> directives: </p>
       <pre class="prettyprint">package game;
package game.devices;
</pre>
       <p> The first <code>package</code> directive identifies a package named <code>game</code>. All classes and interfaces appearing in that directive's source file organize in the <code>game</code> package. The second <code>package</code> directive identifies a subpackage named <code>devices</code>, which resides in a package named <code>game</code>. All classes and interfaces appearing in that directive's source file organize in the <code>game</code> package's <code>devices</code> subpackage. If a JVM implementation maps package names to directory names, <code>game.devices</code> maps to a <code>game\devices</code> directory hierarchy under Windows and a <code>game/devices</code> directory hierarchy under Linux or Solaris. </p>

       <h2>Import a package's classes and interfaces</h2>
       <p> Once you have a package, you will want to import classes and/or interfacesactually, class and/or interface namesfrom that package to your program, so it can use those classes and/or interfaces.        <p> According to <code>Usetestpkg1</code>'s output, the <code>main()</code> method's thread successfully accesses <code>testpkg.A</code>'s <code>y</code> field and calls the <code>returny()</code> method. Furthermore, the output shows a successful implementation of the <code>testpkg.A.StartStop</code> inner interface. </p>
       <p> For <code>Usetestpkg1</code>, prefixing <code>testpkg.</code> to <code>A</code> in three places doesn't seem a big deal. But who wants to specify a fully qualified package name prefix in a hundred places? Fortunately, Java supplies the <code>import</code> directive to import a package's public reference type name(s), so you do not have to enter fully qualified package name prefixes. Express an <code>import</code> directive in source code via the following syntax: </p>
       <pre class="prettyprint">'import' <em>packageName</em> [ '.' <em>subpackageName</em> ... ] '.' ( <em>referencetypeName</em> | '*' ) ';'
</pre>
       <p> An <code>import</code> directive consists of the <code>import</code> keyword immediately followed by an identifier that names a package, <em><code>packageName</code></em>. An optional list of <code>subpackageName</code> identifiers follows to identify the appropriate subpackage (if necessary). The directive concludes with either a <em><code>referencetypeName</code></em> identifier that identifies a specific class or interface from the package, or an asterisk (<code>*</code>) character. If <em><code>referencetypeName</code></em> appears, the directive is a <em>single-type <code>import</code></em> directive. If an asterisk character appears, the directive is a <em>type-on-demand <code>import</code></em> directive. </p>
       <div class="table-wrapper">
        <table border="1" cellpadding="3" cellspacing="3" table="table">
         <tbody>
          <tr>
           <th bgcolor="#c00000">Caution </th>
          </tr>
          <tr>
           <td>As with the <code>package</code> directive, <code>import</code> directives must appear before any other code, with three exceptions: a <code>package</code> directive, other <code>import</code> directives, or comments.</td>
          </tr>
         </tbody>
        </table>
       </div>
       <p> The single-type <code>import</code> directive imports the name of a single public reference type from a package, as the following code fragment demonstrates: </p>
       <pre class="prettyprint">import java.util.Date;
</pre>
       <p> The previous single-type <code>import</code> directive imports class name <code>Date</code> into source code. As a result, you specify <code>Date</code> instead of <code>java.util.Date</code> in each place that class name appears in source code. For example, when creating a <code>Date</code> object, specify <code>Date d = new Date ();</code> instead of <code>java.util.Date d = new java.util.Date ();</code>. </p>
       <p> Exercise care with single-type <code>import</code> directives. If the compiler detects a single-type <code>import</code> directive that specifies a reference type name also declared in a source file, the compiler reports an error, as the following code fragment demonstrates: </p>
       <pre class="prettyprint">import java.util.Date;
class Date {}
</pre>
       <p> The compiler regards the code fragment as an attempt to introduce two reference types with the same <code>Date</code> name: </p>
